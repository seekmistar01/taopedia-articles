# Drand Time-Lock Encryption

Drand time-lock encryption is the timing mechanism Bittensor uses around Commit Reveal to delay
access to fresh validator weights. The goal is not to hide weights forever. It is to keep recently
submitted weights concealed until the reveal time, so public weight information is less useful for
copying. The [Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal) page explains
the delayed reveal flow, and the
[glossary](https://docs.learnbittensor.org/resources/glossary#drandtime-lock-encryption) defines
Drand time-lock encryption as data that can only be decrypted after a future Drand round.

## Why delayed access matters

Validator weights become part of Bittensor's consensus process, so freshly visible weights can be
valuable information. Weight copying happens when a validator copies or derives from public weights
instead of independently evaluating miners. Delaying access to fresh weights reduces that
free-riding window. The
[Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)
source explains the copying risk, while
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal) explains the concealment
period.

## Time-lock idea

Time-lock encryption means encrypted data is not meant to be decryptable until a future time or
round. In the Bittensor documentation, Drand provides the timing source for that delayed access. The
useful Taopedia-level idea is simple: the reveal depends on a later Drand round rather than on
someone choosing to reveal early. The
[glossary](https://docs.learnbittensor.org/resources/glossary#drandtime-lock-encryption) describes
Drand as a randomness beacon used for time-lock encryption.

## Commit Reveal use

In Commit Reveal, validator weights are committed in concealed form and revealed later for the
consensus process. Drand time-lock encryption supports that delayed reveal by making the concealed
weights unavailable before the target reveal round. The
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal) source describes the
commit-and-reveal flow and connects it to copied-weight protection.

## What it does not decide

Drand time-lock encryption does not evaluate miners, choose validator weights, or allocate
emissions. It controls timing and visibility around concealed weight information. Consensus still
depends on validator evaluations and the later aggregation process. The
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) page explains how validator
weights feed consensus outcomes, while
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal) explains the delayed
visibility layer.

## Relation to nearby concepts

## Relationship to Multiple Mechanisms

Drand time-lock encryption is usually discussed for one incentive mechanism at a time, but subnets
can also run multiple mechanisms in parallel. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-matrix) notes that each
mechanism has its own weight matrix for independent consensus calculations.

For readers, Drand time-lock encryption still delays access to concealed validator weights inside a
subnet, but the mechanism context matters. Concealed weights for one mechanism should be read
separately when a subnet uses more than one evaluation system.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),

## Relationship to Commit Reveal

Drand time-lock encryption and Commit Reveal address related but different parts of delayed
validator weight visibility. Commit Reveal names the subnet mechanism that conceals fresh validator
weights, while Drand time-lock encryption provides the timing source that prevents decryption before
the target reveal round. The [Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)
documentation describes Drand as the reveal timing mechanism, and the
[Glossary: Drand/time-lock encryption](https://docs.learnbittensor.org/resources/glossary#drandtime-lock-encryption)
defines time-lock encryption using Drand rounds.

For readers, Commit Reveal names the weight-concealment mechanism, while Drand time-lock encryption
names the timing layer that enforces delayed access to concealed weights.

References: [Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal),
[Glossary: Drand/time-lock encryption](https://docs.learnbittensor.org/resources/glossary#drandtime-lock-encryption)

## Relationship to Weight Copying

Drand time-lock encryption and weight copying address related but different parts of
validator-weight protection. Weight copying names the problem where validators reuse visible weight
signals instead of evaluating miners independently, while Drand time-lock encryption supports
delayed access to fresh weights inside Commit Reveal so copied signals are less useful. The
[The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)
documentation describes the copying risk, and the
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal) documentation connects
delayed reveal to copied-weight protection.

For readers, weight copying names the behavior delayed visibility is designed to reduce, while Drand
time-lock encryption names the timing mechanism behind that delay.

References:
[The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)

## Relationship to Validator Weights

Drand time-lock encryption and validator weights address related but different parts of subnet
weight processing. Validator weights are the evaluation signals validators submit for miners, while
Drand time-lock encryption controls when concealed forms of those signals can be decrypted inside
Commit Reveal. The
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)
describes the per-validator evaluation inputs, and the
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal) documentation describes
concealed weight submission and delayed reveal.

For readers, validator weights name the evaluation content, while Drand time-lock encryption names
the timing layer around when concealed weight information becomes available.

References:
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)

## Relationship to Yuma Consensus

Drand time-lock encryption and Yuma Consensus address related but different parts of Bittensor
consensus flow. Drand time-lock encryption controls timing and visibility around concealed validator
weights before reveal, while Yuma Consensus aggregates revealed validator evaluations into miner
incentives and validator dividends. The
[Glossary: Drand/time-lock encryption](https://docs.learnbittensor.org/resources/glossary#drandtime-lock-encryption)
describes delayed access to encrypted data, and
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) describes how validator
weight signals become consensus outcomes after they are available.

For readers, Drand time-lock encryption names a visibility and timing layer, while Yuma Consensus
names the aggregation step that uses revealed validator signals.

References:
[Glossary: Drand/time-lock encryption](https://docs.learnbittensor.org/resources/glossary#drandtime-lock-encryption),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)
