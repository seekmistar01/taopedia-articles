# Extrinsics

Extrinsics are submitted actions that ask the Bittensor chain to perform a runtime operation
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

The term belongs to the action side of Subtensor interaction. It should be read separately from
chain reads, which observe public state without requesting a change.

## Submitted Action Context

An extrinsic is a submitted request for the runtime to process. The Subtensor extrinsics reference
lists runtime methods exposed as submitted actions
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

The method name identifies the action family. Fees, selected network, and surrounding chain state
give the submission its context
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

## Read Versus Submit Boundary

Transaction-fee documentation separates free chain reads from state-changing actions. That
distinction is the main boundary for extrinsics: reading public chain state observes the network,
while submitting an extrinsic asks the runtime to process a change
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

This boundary matters because observation and submission answer different questions. A read can show
state; an extrinsic requests state processing.

Use storage or inspection references when the claim is about reading state. Use the extrinsics
reference when the claim is about submitting a runtime action
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

## Method Reference Scope

The Subtensor extrinsics reference can identify supported submitted actions, but it is not a full
procedure guide for every interface, condition, or network context that may affect a submission
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

For readers, the method reference is strongest as vocabulary support. Claims about success, failure,
or resulting state need the relevant event, storage, error, fee, or mechanism source.

That keeps method names tied to what was requested. Whether the request produced records, changed
state, or returned a rejection belongs with events, storage, and error references
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

## Submission and Outcome Context

An extrinsic describes what was submitted to the runtime. The outcome may involve changed storage,
emitted records, fees, or an error when the runtime rejects the request
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

Those surfaces answer different questions. The extrinsic identifies the requested action; events,
storage, and errors provide surrounding outcome evidence.

For example, a submitted method can name the requested action, an event can record runtime activity,
and a storage entry can expose state after processing
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

## Network Scope

Extrinsics are interpreted inside a selected Bittensor network environment. Bittensor network
documentation separates mainnet, testnet, and localnet as network contexts
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

The same method name can appear across environments while belonging to different chain histories.
Network context should stay attached when precision matters.

Examples should keep that environment label attached. A localnet or testnet submission can
illustrate extrinsic vocabulary, but it should not be treated as evidence for another network
environment ([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

## Reader Boundary

Extrinsics should be read as submitted-action vocabulary. The term is not an operator guide,
interface tutorial, or proof that a submitted action succeeded
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

The stable article-level point is that extrinsics ask the Subtensor runtime to process a submitted
operation, while separate evidence shows what happened afterward.

It is also not a synonym for every chain interaction. Read-only inspection belongs with chain-read
or storage vocabulary, while extrinsics belong with submitted runtime actions
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).
