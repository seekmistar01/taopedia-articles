# Subnet Liquidity Pools

Subnet liquidity pools are the reserve context for subnet tokenomics in Bittensor. Each pool
connects a TAO reserve with the subnet's alpha-token reserve.

Pool context matters when reading staking, unstaking, slippage, emission, and protocol-alpha claims
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Understanding Slippage](https://docs.learnbittensor.org/learn/slippage)).

The term should be read as reserve vocabulary. It names the pool structure and the reserve
relationship, not a static report of exact balances or a complete explanation of every tokenomics
mechanism connected to the pool.

## Reserve Context

A subnet pool has a TAO reserve and an alpha reserve. The TAO side connects the subnet to
network-level TAO, while the alpha side connects the pool to the subnet-specific token context
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Those reserves should be kept separate from alpha outstanding. Pool reserves describe what sits in
the pool, while alpha outstanding describes alpha outside that reserve context. Mixing those terms
can make a subnet-token claim sound more precise than the source supports.

## Constant Product Context

The slippage documentation describes Bittensor subnet pools as constant product automated market
maker pools. In that model, staking and unstaking pass through paired reserve math rather than a
fixed one-sided conversion
([Understanding Slippage](https://docs.learnbittensor.org/learn/slippage)).

That is why pool claims should name both reserve sides together. A change involving one side of the
pool is interpreted through the paired TAO and alpha reserve relationship.

## Staking and Unstaking Context

When TAO is staked into a subnet, the staking flow passes through the subnet pool reserve context
and produces alpha-denominated stake. When stake is removed, alpha-denominated stake is converted
back through the same reserve system
([Staking and Delegation](https://docs.learnbittensor.org/staking-and-delegation/delegation),
[Glossary: Unstaking](https://docs.learnbittensor.org/resources/glossary#unstaking)).

This keeps staking and unstaking connected without making them identical. Staking begins from
network-level TAO and creates subnet alpha stake, while unstaking moves in the opposite direction
through the pool context.

## Slippage Boundary

Slippage is the difference between a simple expected conversion and the result produced by the pool
context used for the action
([Understanding Slippage](https://docs.learnbittensor.org/learn/slippage)). For Taopedia, the
important point is conceptual: pool reserves affect the resulting TAO or alpha amount.

Exact results need the source values and context being used. A static article should not treat one
example as a general pool rule, because the pool relationship is part of the calculation.

## Emissions and Reserve Injection

Bittensor emission documentation connects subnet pools to reserve injection. Emission flow can add
TAO into subnet pool context, while the surrounding mechanism also separates TAO reserve, alpha
reserve, and alpha-outstanding concepts
([Emission](https://docs.learnbittensor.org/learn/emissions),
[Emission: TAO reserve injection](https://docs.learnbittensor.org/learn/emissions#tao-reserve-injection)).

This means pool liquidity and reward distribution are related, but they are not interchangeable. The
pool names reserve structure; the emission documentation explains how protocol processes can feed
that structure.

## Protocol Alpha Boundary

Protocol alpha is related to subnet pools but not the same term. The glossary defines protocol alpha
as protocol-owned alpha cached per subnet, while the emission documentation describes reserve
injection as the process that can generate that protocol-owned alpha context
([Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha),
[Emission: TAO reserve injection](https://docs.learnbittensor.org/learn/emissions#tao-reserve-injection)).

A subnet liquidity pool names the paired reserve structure. Protocol alpha names a protocol-held
alpha balance produced inside the broader pool and emission system. The two terms answer different
questions even when the same mechanism connects them.

## Coinbase Context

Coinbase is another adjacent term. The coinbase documentation describes a per-block protocol process
that drives TAO emission, pending-emission accounting, and epoch-boundary activity
([Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)).

The pool is the reserve structure that can be affected by emission flow. Coinbase is a process that
can update that structure over time. Keeping those terms separate prevents the pool from being read
as the process itself.

## Reader Boundary

Subnet liquidity pools should be read as reserve and conversion vocabulary. They help explain why
TAO reserves, alpha reserves, staking, unstaking, slippage, emissions, and protocol alpha need
separate wording
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Understanding Slippage](https://docs.learnbittensor.org/learn/slippage)).

The stable article-level point is that a subnet liquidity pool names the paired TAO and alpha
reserve context for a subnet. Claims about exact values, timing, or examples need the specific
source that reports those values.
