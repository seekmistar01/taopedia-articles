Handshake58 is a public project associated with Bittensor Subnet 58. The live Finney identity
queried with `btcli subnets get-identity --netuid 58 --network finney` currently reports the subnet
name as Pending and does not list a GitHub source, subnet URL, or description. The public
Handshake58 repositories, however, explicitly describe Handshake58 and HS58-subnet as Bittensor
Subnet 58 sources.

Handshake58's public materials describe a marketplace and monitoring layer for AI service providers.
Agents can discover providers, while the subnet-side oracle checks whether listed provider endpoints
are reachable and reliable.

## What Handshake58 Provides

Handshake58 provides an AI provider directory connected to a subnet oracle. The main HS58 repository
describes provider discovery, DRAIN and MPP request flows, and a marketplace that syncs provider
scores from Bittensor. The HS58-subnet repository narrows the subnet mechanism to provider
monitoring: miners probe service endpoints, and validators compare reports to consensus before
setting weights.

The subnet source describes support for multiple provider types. For article purposes, the common
abstraction is an HTTP service endpoint that can be checked for reachability, response status, and
latency. This makes the subnet's core commodity reliable provider monitoring rather than model
generation itself.

## Miner and Validator Roles

Miners act as neutral monitors. They receive provider-probe tasks, perform HTTP checks against the
target service, and report reachability, status, and latency to validators.

Validators coordinate and score those reports. The HS58-subnet README says validators fetch provider
lists from a marketplace registry, deterministically choose probe targets from the epoch block hash,
send the same probe tasks to active miners, and score each miner against consensus. Its published
scoring formula weights reachability match, status match, and latency-band agreement before
smoothing scores and setting subnet weights.

Those weights then enter [[Yuma Consensus]], while the broader role split follows the usual
[[Mining and Validating]] pattern.

## On-Chain Identity

Readers can reproduce the current identity fields with the Bittensor CLI command documented in the
[btcli reference](https://docs.learnbittensor.org/btcli/#btcli-subnets-get-identity):
`btcli subnets get-identity --netuid 58 --network finney`.

At the time checked, that Finney identity reports subnet name Pending, with no GitHub source, subnet
URL, or description. This article therefore cites the public Handshake58 repositories for mechanism
details and treats the live identity fields as pending.
