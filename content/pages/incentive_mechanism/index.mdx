# Incentive Mechanism

An incentive mechanism is the rule context that connects work inside a Bittensor subnet to
evaluation and reward outcomes. It describes how miner work, validator evaluation, and incentives
fit together
([Glossary: Incentive Mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism)).

The term belongs to subnet design because each subnet defines the work it wants and the way that
work is evaluated
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## Subnet Context

A subnet is the market context, while an incentive mechanism is the rule context inside that market.
The subnet asks for a kind of digital work, and the mechanism gives validators a way to judge that
work ([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

This distinction keeps the article from treating the subnet and the mechanism as the same thing. A
subnet can be identified by its broader market, while its incentive mechanism explains how work is
evaluated within that market
([Glossary: Incentive Mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism)).

## Miner and Validator Roles

The mechanism shapes both miner and validator behavior. Miners try to produce work the subnet
values, while validators measure that work under the subnet's evaluation rules
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

For readers, the important relationship is direct: miner output is judged through validator
evaluation, and that evaluation is interpreted through the incentive mechanism
([Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner)).

## Emission Context

Incentive mechanisms matter because evaluation feeds reward outcomes. Validator rankings and weight
signals connect subnet work to the emission path
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

That does not make an incentive mechanism a fixed payout table. It is the evaluation context that
links work quality, validator assessment, and incentive outcomes
([Glossary: Incentives](https://docs.learnbittensor.org/resources/glossary#incentives)).

## Yuma Consensus Handoff

An incentive mechanism explains how validators evaluate work inside a subnet. Yuma Consensus is the
shared aggregation step that turns validator weight signals into subnet outcomes
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

This handoff lets subnets specialize their evaluation rules while still using a common consensus
layer for miner incentives and validator-side outcomes
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## Multiple Mechanism Context

Some subnets can use more than one incentive mechanism. In that case, each mechanism should be read
as its own evaluation context rather than blended into a single undifferentiated score
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

The stable concept does not change: an incentive mechanism is still the rule context connecting
work, evaluation, and rewards for the relevant mechanism
([Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)).

## Reader Boundary

An incentive mechanism should be read as a subnet evaluation concept. It is not the subnet itself,
not a miner role, and not the final consensus aggregation step
([Glossary: Incentive Mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism)).

The stable article-level point is that the mechanism connects what miners produce, how validators
evaluate it, and how the subnet's incentive outcomes are shaped
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For incentive
mechanisms, that sequence changes how readers should interpret examples of miner work evaluation and
reward outcomes.

In localnet, an incentive mechanism can be developed and tested in an isolated environment. Localnet
miner evaluation results and incentive outcomes reflect local chain settings and do not represent
production subnet state.

On testnet, an incentive mechanism can be run in a shared, non-production network. Testnet
evaluation outcomes and incentive distributions are separate from mainnet subnet state.

On mainnet, an incentive mechanism is the live rule context connecting real miner work, validator
evaluation, and reward outcomes in production Bittensor subnet operations. The
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
documentation describes the mechanism design that applies on the production network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. An incentive mechanism example from one environment should
not be read as representing miner evaluation or reward outcomes in another environment.
