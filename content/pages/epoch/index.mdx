# Epoch

An epoch is the consensus period inside a Bittensor subnet
([Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)). It is tied to subnet
tempo and marks the timing window in which subnet consensus work is executed.

## Consensus Period

Epoch is timing vocabulary for subnet consensus. It does not name the subnet task, miner output, or
a single validator's evaluation. It names the period in which the subnet's consensus mechanism
processes the relevant validator signals
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Epoch is a boundary term for timing: it tells the reader when consensus work is grouped, not what
useful work the subnet is designed to evaluate.

## Tempo Relationship

Epoch and tempo are related but distinct timing terms. Tempo is the cadence that determines how
frequently epochs occur; epoch is the consensus period produced by that cadence
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

This distinction keeps interval language clear. Tempo names the cadence setting; epoch names the
resulting period in which consensus runs.

## Validator Signal Role

Epochs matter because Bittensor consensus aggregates validator weight signals into subnet outcomes.
In [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus), those weights are
processed over validator stake during the epoch window.

An epoch is not a standalone quality score. It is the timing period for processing consensus inputs,
and its outcomes depend on the validator weight set submitted within it.

## Subnet Mechanism Scope

Each subnet defines useful work and evaluation standards through its incentive mechanism. The
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
documentation places task design, protocol, and scoring inside subnet design. Epoch belongs to that
setting as the timing context for a subnet's consensus period.

Epoch claims should therefore stay scoped to the subnet mechanism being discussed. The period has
meaning because it belongs to a specific subnet's consensus path, not as a universal timing unit.

## Multiple Mechanisms

When a subnet uses multiple incentive mechanisms, epoch language still needs mechanism context. The
[Multiple Incentive Mechanisms](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)
documentation explains that each mechanism can carry independent consensus calculations.

Epoch remains a consensus-period term, but the relevant mechanism determines which evaluation path
applies during that period.

## Reader Boundary

Epoch is the right term when the focus is a subnet consensus period. If the focus is cadence,
[tempo](https://docs.learnbittensor.org/resources/glossary#tempo) is more precise. If the focus is
aggregation logic, [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) is more
precise. If the focus is evaluation signals themselves, validator weights are more precise.

These boundaries keep epoch focused on consensus timing, not a subnet task description, a validator
ranking, or a complete account of reward outcomes.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For epochs, that
sequence changes how readers should interpret examples of subnet consensus period timing and
validator weight submission outcomes.

In localnet, epoch mechanics can be tested in an isolated environment. Localnet epoch timing and
Yuma Consensus round outcomes reflect local chain settings and do not represent production subnet
consensus state.

On testnet, epoch behavior can be observed in a shared, non-production network. Testnet epoch timing
and consensus outcomes are separate from mainnet subnet state.

On mainnet, an epoch is the live consensus period during which subnet validators submit weights that
are aggregated by Yuma Consensus in production Bittensor subnet operations. The
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) documentation describes the
epoch mechanics that apply on the production network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. An epoch example from one environment should not be read
as representing consensus timing or reward outcomes in another environment.

## Timing Boundary

Epoch language should keep timing and mechanism context attached. Bittensor references use epochs
around recurring chain processes, and network documentation distinguishes observations by selected
chain environment ([Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

An epoch is not a universal wall-clock promise. Its practical meaning depends on the network,
subnet, and mechanism being discussed.
