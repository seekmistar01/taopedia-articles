# Subnet Validator

A subnet validator is the evaluation role inside a Bittensor subnet. Validators measure miner work
and set weights based on miner output
([Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

The term belongs to subnet incentive vocabulary. It names the role that evaluates work, not the
miner role or the weight signal by itself.

In plain language, subnet validator answers: who scores the miners in this subnet? A validator
observes miner outputs against the subnet's standard and submits weights that reflect that
evaluation. Those weights then feed into Yuma Consensus to produce rank and trust outcomes
([Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator)).

For a reader, the distinction from subnet miner matters most when reading weight or rank data. A
subnet miner produces work; a subnet validator evaluates it. Subnet validator also differs from
blockchain validator — the two roles operate in different contexts within Bittensor
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## Evaluation Role

Subnet validators operate relative to the standards of the subnet they serve. Different subnets can
pursue different goals, so useful evaluation depends on the subnet's task and incentive design
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

This keeps validator vocabulary scoped to the subnet's definition of useful miner output. A
validator is not evaluating one universal task across every subnet.

The role is therefore subnet-specific. A validator on one subnet measures the work that subnet asks
for, while another subnet can define a different task, protocol, scoring model, or incentive
mechanism.

## Weight Signal Role

The glossary defines subnet validators around evaluating miner performance and setting weights from
miner output. Those weights then feed the subnet consensus process
([Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Weights are therefore evaluation signals, not the miner work itself. The subnet validator is the
role that produces those signals after measuring miner output against subnet standards.

Validator evaluation produces the weight signals that enter the Yuma Consensus process for the
subnet epoch. The consensus step aggregates those signals into miner incentives and validator
dividends, making evaluation the measurement layer between task execution and emission allocation
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

A weight signal names a validator's measurement of miner output. Final emission outcomes come later,
after validator signals are processed through consensus.

## Miners Produce Work Validators Evaluate

Subnet miners and subnet validators form the core work-and-evaluation relationship inside a subnet.
Miners produce work for the subnet, and validators measure that work
([Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner),
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator)).

Subnet miner names the producing role; subnet validator names the evaluating role. A subnet
validator mention does not by itself identify which miner produced a given output, because miner
vocabulary names the producing role rather than the evaluating role
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task)).

## Root Validators Evaluate Staking Not Miner Work

Subnet validator and root validator share validator vocabulary but operate in different settings.
The [Root Subnet](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero) has no
miner work to evaluate, while a subnet validator evaluates miner work inside an ordinary subnet.

Root validator belongs to the root staking path; subnet validator belongs to the miner-evaluation
path inside a subnet. In this article, subnet validator refers to the role that evaluates miner
output for a subnet incentive mechanism.

## Incentive Mechanism Scope

Validator evaluation is part of the subnet's incentive mechanism. The task defines what work is
requested, the protocol shapes miner-validator interaction, and the evaluation method gives
validators a way to judge results
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

When a subnet uses multiple mechanisms, validators evaluate miners separately for each mechanism.
The mechanism label stays attached to any claim about validator work because a weight signal in one
mechanism is a different signal from one in another, even on the same subnet
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For subnet
validators on a subnet such as netuid 1, that sequence changes how readers should interpret weight
submission examples and evaluation outcomes.

In localnet, validator role mechanics can be demonstrated in isolation. Local validator examples
reflect local chain configuration rather than production subnet performance.

On testnet, validator evaluation can be observed in a shared, non-production network. Testnet weight
outcomes are separate from mainnet subnet state.

On mainnet, subnet validator is the live evaluation role inside production subnet markets on the
connected Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. Validator evaluation examples from one environment should
not be read as evidence for production subnet performance in another environment.

## Registration Places the Validator in the Participant Set

[Validating: Validator registration](https://docs.learnbittensor.org/validators#validator-registration)
describes registering with a subnet to receive a UID before a validator can take part in evaluation
work. [Register](https://docs.learnbittensor.org/resources/glossary#register) vocabulary names that
subnet-entry step; subnet validator names the evaluation role that follows once entry succeeds
([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot)).

Validator reading therefore starts after registration. Registration places the hotkey into the
participant set that can evaluate miners and submit weights.

## Validator Permit Gates Weight Submission

The
[Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit)
describes a flag required for setting weights and participating in consensus, awarded to the top
neurons by [stake weight](https://docs.learnbittensor.org/resources/glossary#stake-weight) on the
subnet.

Registration places a validator on a subnet; the permit answers whether that neuron may submit
weights that count in aggregation. Subnet validator vocabulary names the evaluation role; permit
vocabulary names whether that role currently carries validation rights inside the subnet.

## Activity Cutoff Times Weight Submission Each Epoch

Subnet [hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) include an
activity cutoff measured in blocks within each
[epoch](https://docs.learnbittensor.org/resources/glossary#epoch). A validator must submit weights
within the first activity-cutoff blocks of an epoch or wait until the next epoch to participate
([Subnet Hyperparameters: Activity Cutoff](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#activitycutoff)).

That timing rule affects which evaluation signals enter the consensus pass. Subnet validators must
act inside the early epoch window for their weights to count in the current round.

## Validator Dividends Pair Bonds With Miner Incentives

[Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions)
documentation calculates each validator's share from its
[validator-miner bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)
weighted by each bonded miner's incentive outcome. Subnet validator vocabulary names the evaluation
role; [dividends](https://docs.learnbittensor.org/resources/glossary#emission) name the
validator-side emission result that follows from bonds and miner-side incentives in the same
consensus flow.

Weights feed rank and miner incentives first; bonds and dividends translate ongoing validator-miner
relationships into validator-side returns.

## Reader Boundary

Subnet validator names the evaluation role inside a subnet. It explains evaluation vocabulary rather
than procedure details, validator lists, ranking reports, or claims about a specific validator's
performance
([Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

A subnet validator evaluates miner work inside a Bittensor subnet and expresses that evaluation as
weights for the subnet consensus path.

## Relationship to Pruning Score

A subnet validator and pruning score meet at deregistration eligibility. A subnet validator is a
neuron that evaluates miners and submits weights in a subnet, while pruning score is the per-neuron
value that ranks which UID is removed first when the subnet is full, so a low-scoring validator
outside its immunity window can be displaced like any other low scorer
([Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator),
[Glossary: Pruning Score](https://docs.learnbittensor.org/resources/glossary#pruning-score),
[Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)).

- **Subnet validator** — a neuron that evaluates miners and submits weights in a subnet.
- **Pruning score** — the removal-ranking value that can deregister a low-scoring neuron, validators
  included.

## Relationship to Stake Weight

A subnet validator and stake weight describe validation as the role and the influence it carries. A
subnet validator is the participant that measures miner work and submits weights, while stake weight
is the stake-derived value that scales how much each validator's submitted weights count when
consensus is computed, so the role does the evaluating and stake weight decides how heavily that
evaluation lands
([Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator),
[Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

- **Subnet validator** — the role that evaluates miners and submits weights.
- **Stake weight** — the stake-derived value that scales how much those weights count.
