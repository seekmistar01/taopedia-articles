# Validator Weights

Validator weights are evaluation signals submitted by subnet validators after judging miner work in
a subnet. They are inputs that Yuma Consensus aggregates into miner incentives and validator
dividends ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The term is subnet-specific. A weight only has meaning inside the incentive mechanism and evaluation
criteria of the subnet where it was produced
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## Evaluation Signal Context

A validator weight is not a generic reputation score. It is a subnet-specific evaluation of miner
work under the rules of the subnet's incentive mechanism
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

That means the same numeric-looking signal can mean different things in different subnets. The
subnet task, scoring method, and mechanism design give the weight its interpretation.

## From Evaluation to Consensus

Validators submit weight vectors that express their relative evaluation of miners. Yuma Consensus
combines those submitted vectors into a broader consensus result rather than treating one
validator's evaluation as final on its own
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emissions](https://docs.learnbittensor.org/learn/emissions)).

This aggregation step is why validator weights matter to rewards. They are evaluation inputs that
help determine how incentives are allocated inside the subnet.

## Consensus Score Boundary

Validator weights and consensus score are related but distinct. Validator weights are the submitted
evaluation inputs, while consensus score is the stake-weighted threshold used to filter outlier
submissions before final rank calculations
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

For readers, this means a validator weight should not be read as the consensus score itself.

## Weight Matrix Boundary

Validator weights and the weight matrix describe the same data at different scopes. A weight vector
is one validator's submitted evaluation signal, while the weight matrix is the subnet-level
collection of submitted vectors used as input to consensus
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

That boundary keeps individual submissions separate from the organized consensus input.

## Quality and Timing Context

Validator weights are most useful when they reflect independent evaluation. Weight copying weakens
that signal by relying on visible weight information instead of fresh evaluation
([The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

Commit Reveal can conceal fresh weight submissions until a later reveal period, reducing the
usefulness of copying newly submitted signals.

## Multiple-Mechanism Context

When a subnet uses multiple incentive mechanisms, validator weights must be read inside the
mechanism context that produced them. A weight for one mechanism does not automatically describe
miner performance under another mechanism
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)).

For readers, validator weights still mean subnet-specific evaluation signals. Multiple mechanisms
only add another boundary: which mechanism's evaluation path produced the signal.

## Reader Boundary

Validator weights should be read as consensus inputs, not as universal miner ratings or cross-subnet
comparisons. A weight claim should stay tied to the subnet, incentive mechanism, and Yuma Consensus
context that give it meaning
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

Tempo and epoch describe cadence, Commit Reveal describes delayed visibility, and validator weights
describe the evaluation signals that consensus uses within that cadence.
