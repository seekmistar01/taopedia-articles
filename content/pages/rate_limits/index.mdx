# Rate Limits

Rate limits are Bittensor rules that pace repeated chain actions. The official
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits) documentation describes them
as block-based cooldown periods in Subtensor. When a rate-limited action succeeds, another attempt
at the same kind of action must wait for the required number of blocks before it can succeed again.

## Why Rate Limits Exist

Rate limits help prevent spam, protect network stability, and keep access to shared chain resources
orderly. They are not a judgment about whether an action is useful; they are timing rules around how
often certain successful actions may be repeated. The official Bittensor rate-limit reference frames
the feature in terms of stability, fairness, and cooldown windows.

## Block-Based Cooldowns

Bittensor rate limits are measured in blocks. A cooldown begins after the relevant action succeeds.
If the action is attempted again before enough blocks have passed, the chain can reject the repeat
attempt. The same official page notes that unsuccessful attempts may be retried, which means the
cooldown is tied to successful rate-limited activity rather than every failed attempt.

## Network and Subnet Scope

The rate-limit reference groups limits into network-wide and subnet-specific areas. Some limits
apply broadly across the network, while others apply within a subnet context. The important
Taopedia-level distinction is scope: a reader should ask which action is being paced and whether the
rule is global or tied to a particular subnet setting.

## Relationship to Errors

Rate limits often appear to readers as rejection messages when an action is attempted too soon.
Those messages should be read as timing feedback from the chain layer, not as a complete explanation
of the surrounding action. The
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor) reference lists
standard Subtensor error names, while the
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits) page explains the cooldown
rules themselves.

## Careful Interpretation

Rate-limit values can differ by action and context, and readers should avoid generalizing one
cooldown to every Bittensor activity. A statement about a rate limit should identify the action, the
unit, and the source. For current values and exact rule names, the official Bittensor rate-limit
documentation and linked Subtensor references are the appropriate sources.

## Relationship to Subtensor Error Codes

Rate limits and Subtensor error codes address related but different sides of chain rejection
feedback. [Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits) explain block-based
cooldown rules for repeated actions, while
[Subtensor Error Codes](https://docs.learnbittensor.org/errors) groups the failure messages readers
may see when Subtensor rejects an action.

For readers, rate limits explain timing feedback for too-soon repeats, while Subtensor error codes
name the broader rejection vocabulary that includes staking, subnet, and wallet-related conditions.

References: [Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors),
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor)

## Relationship to Multiple Mechanisms

Rate limits pace repeated chain actions in Subtensor. Some limits apply within a subnet context. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism) notes that each
subnet has one or more incentive mechanisms.

For readers, rate limits still name chain-layer cooldown rules. A subnet may run more than one
incentive mechanism, but rate limits belong to action pacing rather than to per-mechanism miner
evaluation.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Incentive Mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism)
