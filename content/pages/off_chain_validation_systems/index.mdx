# Off-Chain Validation Systems

[Off-chain validation systems](https://docs.learnbittensor.org/subnets/understanding-subnets) are
the subnet-specific evaluation logic that validators run before Bittensor consensus uses their
weight signals. They sit inside subnet incentive mechanisms, where miner work is measured outside
the chain and then represented through validator rankings.

## Subnet Context

In Bittensor, a [subnet](https://docs.learnbittensor.org/subnets/understanding-subnets) is an
incentive-based marketplace with miners that produce a digital commodity and validators that measure
miner work for quality. That makes validation a subnet-specific activity rather than a single
universal scoring rule shared by every subnet.

## Incentive-Mechanism Role

A subnet's
[incentive mechanism](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism) defines
how validators evaluate miner work. The same source describes incentive mechanisms as scoring models
that guide miner behavior and validator evaluation, so the off-chain validation system is the part
of a subnet that turns task performance into validator scores.

## From Scores to Chain Input

The
[incentive-mechanism documentation](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
describes validators periodically computing weight vectors for miners and transmitting those vectors
to the blockchain. The chain then forms weight matrices from those rankings as input for Yuma
Consensus.

## Boundary with On-Chain Consensus

[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) runs on-chain and computes
emissions from validators' rankings of miner performance. Off-chain validation systems are the
subnet-side evaluation layer before that consensus step; they are not the on-chain emissions
algorithm itself.

## Scope of the Term

[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) describes a
subnet incentive mechanism as unique to the subnet and maintained off-chain by the subnet creator.
The term therefore refers to the concept of subnet-specific evaluation logic, not to a live subnet's
current validator behavior or a repository-specific guide.
