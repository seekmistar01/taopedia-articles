# Public Subtensor

Public Subtensor refers to publicly reachable access to the Bittensor chain layer. It is an access
context, not a separate blockchain.

The
[Glossary: Public Subtensor](https://docs.learnbittensor.org/resources/glossary#public-subtensor)
entry defines the term as publicly reachable access to Subtensor. The
[Subtensor Nodes](https://docs.learnbittensor.org/subtensor-nodes) documentation gives operational
context for connecting to the Subtensor chain layer.

## Public Access Versus Local Development

Public Subtensor is different from private local development context. The
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)
describes a local blockchain as private, isolated, and used for development and testing. That local
context can demonstrate chain behavior, but it is not the same as publicly reachable Subtensor
access.

The distinction is useful when reading examples. A local run can show how a mechanism behaves in an
isolated environment, while Public Subtensor describes access to shared chain context. Those sources
should not be collapsed into one evidence category.

## Access Versus Mechanism

Public Subtensor names an access context, not a protocol mechanism. Through public access, an
observer can inspect chain state exposed by Subtensor, but the access path is not itself the
mechanism that creates that state.

For example, the
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
documentation describes a per-block protocol process inside Subtensor. Public Subtensor may expose
state affected by that process, but it does not produce emissions, trigger consensus, or replace the
mechanism documentation.

## Signing Boundary

Public access does not make every chain action permissionless. Public Subtensor can expose readable
chain state, while submitted extrinsics still depend on the action's required signer and authority
([Bittensor CLI: Permissions Guide](https://docs.learnbittensor.org/btcli/btcli-permissions),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

That keeps public access separate from signing authority. A reader can use a public chain context to
observe data, but changing state still belongs to the relevant wallet, role, fee, and extrinsic
requirements.

## Distinction from Mainchain

Mainchain names Bittensor's primary production network, while Public Subtensor names publicly
reachable access to the Subtensor chain layer — related terms with different scope
([Glossary: Mainchain](https://docs.learnbittensor.org/resources/glossary#mainchain),
[Glossary: Public Subtensor](https://docs.learnbittensor.org/resources/glossary#public-subtensor)).

The public access surface is not the same as the production network it connects to
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

## Distinction from Chain Reads

Public Subtensor names where the chain is reached publicly; a chain read names the read-only
inspection performed through an available access path
([Subtensor Nodes](https://docs.learnbittensor.org/subtensor-nodes),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Subtensor storage documentation describes the readable chain surfaces exposed through that layer,
separate from the access route itself.

## Access Evidence Layers

Public Subtensor examples are easiest to read when the access route, query surface, chain context,
and signing requirement stay separate. The public endpoint explains how the chain is reached, while
storage, events, extrinsics, and network references explain what kind of evidence the reader is
using ([Subtensor Nodes](https://docs.learnbittensor.org/subtensor-nodes),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

| Layer             | What it answers                            | What it does not prove              |
| ----------------- | ------------------------------------------ | ----------------------------------- |
| Public access     | Whether the chain can be reached publicly  | That every action is permissionless |
| Storage query     | Which readable state surface was inspected | Why that state changed              |
| Event record      | What runtime activity was emitted          | The full history of the action      |
| Extrinsic surface | What runtime action was submitted          | That the signer had authority       |
| Network context   | Which environment supplied the observation | A claim about every environment     |
| Block context     | Where an observation sits in chain history | Future endpoint availability        |

This layering keeps a public endpoint from becoming a source for every claim around it. A storage
read through Public Subtensor can support a statement about observed chain state, but a mechanism
claim still needs the relevant mechanism or runtime reference
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

Signed actions need an additional check. Submitting an extrinsic through a public access path does
not remove the key, role, balance, or fee requirements attached to the action
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Bittensor CLI: Permissions Guide](https://docs.learnbittensor.org/btcli/btcli-permissions)).

Network and block context finish the evidence chain. The same query shape can be used against
mainnet, testnet, or localnet, but the result belongs to the environment and block where it was
observed ([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

## Evidence Boundary

The
[Public Subtensor glossary entry](https://docs.learnbittensor.org/resources/glossary#public-subtensor)
supports claims about public chain access. It does not provide endpoint recommendations, connection
procedures, node-management guidance, or private account diagnosis.

That boundary keeps the article conceptual. The
[Public Subtensor glossary entry](https://docs.learnbittensor.org/resources/glossary#public-subtensor)
supports public-access claims, while mechanism claims should still use the mechanism's own
documentation and identify the relevant network context.

## Public Access Differs From Signing Authority

The [Bittensor CLI: Permissions Guide](https://docs.learnbittensor.org/btcli/btcli-permissions)
separates permissionless chain reads from actions that require wallet keys. Public Subtensor
vocabulary therefore covers observation and query surfaces, not authorization to move balances or
set weights.

Readers inspecting public chain data still need the appropriate coldkey or hotkey material to submit
state-changing extrinsics.

## Subtensor Nodes Name the Connection Layer

[Subtensor Nodes](https://docs.learnbittensor.org/subtensor-nodes) documentation describes how
participants reach Bittensor chain infrastructure. Public Subtensor sits in that connection context
as shared access to chain state rather than as a subnet participant role
([Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor)).

Network environment still matters: public access on testnet does not describe mainnet state.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Public
Subtensor, that sequence determines which network's shared chain access is being described.

In localnet, chain access is isolated rather than public. Localnet observations reflect private
local chain activity and do not represent publicly reachable Bittensor chain state.

On testnet, publicly reachable access to the testnet chain layer is available. Testnet public chain
access exposes testnet state, which is separate from mainnet chain state.

On mainnet, Public Subtensor is the live, publicly reachable access to the production Bittensor
chain layer. The [Subtensor Nodes](https://docs.learnbittensor.org/subtensor-nodes) documentation
describes the connection context for reaching the production network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. Public chain access on one network should not be read as
representative of chain state on another network.

## Distinction from Subtensor

Public Subtensor and Subtensor name an access surface and the chain it reaches. Subtensor is
Bittensor's layer-one blockchain and system of record that stores state and processes extrinsics,
while Public Subtensor is publicly reachable access to that chain layer, so Subtensor is the chain
itself and Public Subtensor is one way of reaching it rather than a separate network or runtime
([Glossary: Public Subtensor](https://docs.learnbittensor.org/resources/glossary#public-subtensor),
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor)).

- **Subtensor** — the layer-one blockchain and system of record.
- **Public Subtensor** — publicly reachable access to that chain layer.

The public-access part of the term matters. Public Subtensor describes access to Subtensor through a
publicly reachable network context, while Subtensor itself names the Bittensor blockchain layer. The
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor) describes that
chain layer as Bittensor's system of record.

This distinction keeps the concept narrow. Public Subtensor does not rename the chain, create a new
runtime, or describe every node-deployment detail. It identifies the public-access context for
reaching the Subtensor chain layer.

## Distinction from Yuma Consensus

Public Subtensor names the publicly reachable chain endpoint and network access layer. Yuma
Consensus is the on-chain mechanism that converts validator weight submissions into emission shares
each tempo and is not network-endpoint vocabulary
([Public Subtensor](https://docs.learnbittensor.org/resources/glossary#public-subtensor),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Public Subtensor** — publicly reachable chain endpoint for network access.
- **Yuma Consensus** — on-chain settlement that turns validator weights into emission shares. Public
  Subtensor names chain connectivity; Yuma processes validator weights into rewards
  ([Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions)).

## Distinction from Network

Public Subtensor should be read with network context attached. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet as different environments. Public access can expose shared
network state, but the selected environment determines what the observed data can support.

A public-access example from one environment should not be treated as a universal Bittensor fact.
The relevant network, endpoint context, queried state, and block context determine how strong a
claim can be.

## Reader Boundary

Public Subtensor should not be read as private key access, wallet custody, or a guarantee about any
particular endpoint operator. It names publicly reachable chain-layer access vocabulary
([Glossary: Public Subtensor](https://docs.learnbittensor.org/resources/glossary#public-subtensor)).
