# Regenerating a Key

Regenerating a key refers to recovering wallet key material from recovery information. In Bittensor
wallet context, documentation treats key material and recovery phrases as security-sensitive.

## Wallet Recovery Context

The term belongs to wallet recovery and key custody. Recovery information can restore access to
wallet keys, so it should be protected with the same care as other sensitive wallet material.

## Recovery Versus Subnet Participation

Regenerating a key restores wallet key material from recovery information; it does not by itself
recreate subnet UID slots, validator permits, or on-chain registration state. The glossary defines
regenerating a key as recreating a lost or deleted coldkey or hotkey using the associated mnemonic,
which keeps the operation inside wallet recovery vocabulary rather than subnet entry vocabulary.

Readers should not treat key regeneration as a substitute for registration or delegation setup on a
subnet. Restored hotkey files can sign operational work again, but subnet participation still
depends on the registration and permit context described in mining and validating documentation.
Live chain state determines what a restored key can operate after recovery completes. Registration
commands and permit tables belong in those operational references rather than in recovery prose.

## Regeneration Versus Rotation Operations

Regenerating a key recreates lost key files from saved recovery material. Hotkey swap and coldkey
swap, by contrast, are deliberate on-chain rotation or identity migration operations while keys
still exist or while a controlled migration is authorized. The recovery path answers missing local
key material; swap operations answer compromised or migrated on-chain identity under documented
rules.

Mixing the terms can confuse incident response. A deleted hotkey file may call for regeneration from
a mnemonic, while a suspected hotkey leak may call for hotkey swap under the security documentation
without treating the coldkey as lost. Operational steps, fees, and rate limits for swap paths belong
in wallet and security references rather than in this concept article. Recovery restores local
signing files; swap migrations change on-chain key identity under separate documented rules.

## Regeneration Reproduces the Same Key

Regeneration recovers the original key rather than producing a new one. The
[Glossary: Regenerating a Key](https://docs.learnbittensor.org/resources/glossary#regenerating-a-key)
defines it as recreating a lost or deleted coldkey or hotkey from the associated mnemonic, so the
same mnemonic maps back to the same key material each time.

For readers, that is why regeneration restores the original
[SS58 address](https://docs.learnbittensor.org/resources/glossary#ss58-encoded) instead of a
different one. A recovered coldkey or hotkey holds the same on-chain identity it had before, which
is also why anyone with the mnemonic can reproduce that key and the words must be protected like the
key itself ([Glossary: Mnemonic](https://docs.learnbittensor.org/resources/glossary#mnemonic)).

## Distinction from Seed Phrase Security

Seed phrase security names handling practices for recovery words. Regenerating a key names the
restore action those words enable
([Handle your Seed Phrase/Mnemonic Securely](https://docs.learnbittensor.org/keys/handle-seed-phrase),
[Glossary: Regenerating a Key](https://docs.learnbittensor.org/resources/glossary#regenerating-a-key)).

## Distinction from Mnemonic

A mnemonic is the recovery phrase itself. Regenerating a key is the procedure that uses it to
rebuild a lost coldkey or hotkey
([Glossary: Mnemonic](https://docs.learnbittensor.org/resources/glossary#mnemonic),
[Glossary: Regenerating a Key](https://docs.learnbittensor.org/resources/glossary#regenerating-a-key)).

## Distinction from Coldkeys

A coldkey is the custody key that controls TAO and hotkeys. Regenerating a key can restore that
coldkey from its mnemonic when local files are lost
([Glossary: Coldkey](https://docs.learnbittensor.org/resources/glossary#coldkey),
[Glossary: Regenerating a Key](https://docs.learnbittensor.org/resources/glossary#regenerating-a-key)).

## Distinction from Local Wallet

Regenerating a key is the restore action recovery words enable for a specific wallet key. A local
wallet is the on-disk wallet file structure those keys are stored in after creation
([Glossary: Regenerating a Key](https://docs.learnbittensor.org/resources/glossary#regenerating-a-key),
[Glossary: Local Wallet](https://docs.learnbittensor.org/resources/glossary#local-wallet)).

- **Regenerating a key** — restore action from recovery material.
- **Local wallet** — on-disk wallet file that holds generated keys.

Restore vocabulary names the recovery action; local-wallet vocabulary names the storage container
([Working with Keys](https://docs.learnbittensor.org/keys/working-with-keys)).

## Distinction from Public Key

A public key is the openly shareable component of a wallet keypair, used to verify signatures and
identify an account, while regenerating a key is the procedure that rebuilds a lost coldkey or
hotkey from its mnemonic
([Glossary: Public Key](https://docs.learnbittensor.org/resources/glossary#public-key)).

- **Regenerating a key** — the recovery action that reconstructs lost key material.
- **Public key** — the shareable component that recovery reproduces unchanged, so others verify the
  same identity afterward.

## Distinction from Private Key

A private key is the secret component that authorizes transactions and signs operations, whereas
regenerating a key is the recovery step that reproduces that secret from the mnemonic when local
files are gone
([Glossary: Private Key](https://docs.learnbittensor.org/resources/glossary#private-key)).

- **Regenerating a key** — the action that restores access when key files are lost.
- **Private key** — the secret signing component that restoration brings back, which is why the
  mnemonic deserves the same protection.

## Distinction from Yuma Consensus

Regenerating a key is the process of recreating a lost or deleted coldkey or hotkey from its
associated mnemonic. It names wallet recovery vocabulary, not the on-chain step that turns validator
weight submissions into emission shares
([Glossary: Regenerating a Key](https://docs.learnbittensor.org/resources/glossary#regenerating-a-key),
[Working with Keys](https://docs.learnbittensor.org/keys/working-with-keys)).

Yuma Consensus is the on-chain process that converts submitted validator weights into emission
shares after evaluation completes. Key regeneration restores signing material offline; it does not
recreate subnet UID slots or validator permits by itself
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers should keep key-recovery vocabulary separate from the settlement step. Regenerating a key
answers how wallet keys are restored from a mnemonic; Yuma Consensus answers how validator weights
become emission shares in a subnet
([Glossary: Regenerating a Key](https://docs.learnbittensor.org/resources/glossary#regenerating-a-key),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Regenerating a key** — recreating coldkey or hotkey material from a mnemonic.
- **Yuma Consensus** — on-chain weight aggregation that settles emissions.

## Distinction from Coldkey and Hotkey

The
[Glossary: Regenerating a Key](https://docs.learnbittensor.org/resources/glossary#regenerating-a-key)
defines regenerating a key as recreating a lost or deleted coldkey or hotkey using the associated
mnemonic. The recovery path applies to wallet key material tied to that mnemonic, not to subnet
registration slots or validator permissions.

Those roles carry different authority after recovery: coldkeys control balances, stake authority,
and hotkey management, while hotkeys carry operational authority for subnet roles. Regenerating one
key type restores that role's signing material rather than every other key in the wallet hierarchy,
and the mnemonic must match the specific key being restored
([Glossary: Regenerating a Key](https://docs.learnbittensor.org/resources/glossary#regenerating-a-key),
[Glossary: Coldkey](https://docs.learnbittensor.org/resources/glossary#coldkey)).

## Reader Boundary

This page defines the term. It does not ask readers to enter a recovery phrase, expose keys, or
follow a specific recovery action. The
[Working with Keys](https://docs.learnbittensor.org/keys/working-with-keys) documentation is the
better source for current security-sensitive wallet actions.

## Distinction from Seed Phrase

Regenerating a key and a seed phrase relate as a recovery action and the input it consumes. A seed
phrase is the human-readable word sequence that can recover wallet key material, while regenerating
a key is the act of recovering that key material from such recovery information
([Glossary: Regenerating a Key](https://docs.learnbittensor.org/resources/glossary#regenerating-a-key),
[Glossary: Seed Phrase](https://docs.learnbittensor.org/resources/glossary#seed-phrase),
[Glossary: Mnemonic](https://docs.learnbittensor.org/resources/glossary#mnemonic)).

- **Seed phrase** — the word sequence that can recover wallet key material.
- **Regenerating a key** — the act of recovering key material from that information.
