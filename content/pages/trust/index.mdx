# Trust

Trust is a Bittensor consensus metric for the miner side of evaluation. It describes how much of a
miner's original validator support remains after consensus clipping
([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

The term belongs to Yuma Consensus vocabulary because it is read after validator weight signals are
filtered and aggregated ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Consensus Clipping Context

Validators submit weight signals after evaluating miner work. Consensus filtering can reduce outlier
support before the final miner-side result is read
([Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

Trust describes how much miner support remains after that filtering. It is therefore about
consensus-adjusted support, not a separate miner label outside the consensus process
([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

## Rank Context

Rank and trust are connected but not identical. Rank names the consensus-adjusted miner result,
while trust describes how much support was preserved between pre-rank and final rank
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

For readers, rank says where the miner landed after aggregation; trust says how much clipping
changed the path to that result
([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

## Pre-Rank Relationship

Trust is easiest to read as a comparison between the miner's pre-rank and final rank. Pre-rank is
the support pattern before consensus filtering, while rank is read after outlier weights have been
clipped by the consensus process
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

That comparison keeps the metric from being mistaken for raw popularity. A miner can receive initial
validator support, but trust describes how much of that support remains once consensus filtering has
affected the final rank
([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

The useful reading is therefore relational. Trust does not replace rank; it explains how strongly
the final rank stayed aligned with the pre-clipping support represented by validator weights
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Miner-Side Interpretation

Trust belongs to the evaluated miner side of the consensus path. Validator weights enter consensus
filtering, and the resulting rank is read for the miner being evaluated
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

In that path, trust helps distinguish two different questions. Rank asks what final miner-side
result remains after aggregation. Trust asks how much the clipping step changed the support that fed
into that result ([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

This makes trust useful when comparing consensus effects across miner evaluations inside the same
subnet context. The metric points to the effect of clipping on miner support; it does not identify
which validator created each weight or summarize the validator's own alignment with consensus
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)).

## Same-Context Comparison

Trust comparisons are most meaningful when the compared values come from the same subnet consensus
context. Yuma Consensus operates over validator weights inside a subnet, so miner-side trust values
are clearest when read against the consensus process that produced them
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Across different subnets, the task, validator set, incentive mechanism, and weight pattern can
differ. Trust still has the same definition, but the surrounding consensus setting can change what a
particular value is telling the reader
([Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

## Validator Trust Boundary

Trust is read from the evaluated miner's side of the process. Validator trust is read from the
validator side, describing how much of a validator's submitted signal remains after filtering
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)).

Keeping the two terms separate avoids treating trust as a universal quality score for every role in
a subnet ([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

## Consensus Score Context

Consensus score is the agreement threshold used during clipping. Trust is an outcome metric after
that filtering affects miner-side support
([Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

The two concepts are related because trust depends on how much validator support survives the
consensus threshold ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Reader Boundary

Trust should be read inside subnet consensus interpretation. It is not an operating instruction, a
staking guide, or a promise about future miner results
([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

The stable article-level point is that trust explains how consensus clipping changes miner-side
support between pre-rank and final rank
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).
