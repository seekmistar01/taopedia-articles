# Token Bridging

Token bridging is the Bittensor capability that connects TAO across Substrate and EVM account
contexts on the same Bittensor blockchain. The official
[Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao) documentation frames it
as two related ideas: moving the same TAO between SS58 and H160 account formats, and representing
staked TAO as vTAO inside the EVM layer.

In Taopedia terms, token bridging names a connectivity concept, not a wallet type or a subnet role.
It belongs alongside the H160 address concept because bridging is what makes TAO usable inside
Bittensor's EVM environment.

## Why Token Bridging Exists

TAO is the native token of the Bittensor network and is normally held in Substrate-style SS58
wallets. The [Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao)
documentation explains that Bittensor also has an EVM layer, and participants need a way to use the
same TAO inside that layer without changing the asset itself.

For readers, token bridging is best understood as the link between two account contexts for one
token. The value does not change when TAO moves between contexts — only the account format and the
set of available operations change.

References: [Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao),
[Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets)

## What Token Bridging Does

Token bridging moves TAO between SS58 and H160 account formats so the same token can participate in
both the Substrate side and the EVM side of the Bittensor chain. The
[Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao) documentation describes
this as transferring the same token between account formats, where SS58 wallets access
Substrate-side operations and H160 wallets access EVM-side operations.

The second part of token bridging is vTAO, the EVM representation of staked TAO. The documentation
describes vTAO as representing TAO that has been staked, where the vTAO balance a holder holds stays
the same while the underlying staked TAO grows with staking rewards over time.

For readers, token bridging is the concept that keeps TAO usable across both account contexts and
across the staked representation that the EVM layer uses.

References: [Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao)

## Account and Signing Boundary

Token bridging crosses account-format context as well as token context. On the native side, TAO is
associated with Substrate-style SS58 wallet addresses. On the EVM side, TAO is used through
Ethereum-style H160 addresses. The
[Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao) documentation frames
bridging as the path between those account formats, and the
[H160 conversion](https://docs.learnbittensor.org/evm-tutorials/convert-h160-to-ss58) documentation
explains the relationship between H160 and SS58 address representations.

This does not make the two wallet contexts identical. A Bittensor wallet signs native Subtensor
actions through the Bittensor wallet model, while an EVM wallet signs EVM-layer interactions for the
H160 account context. Bridging connects TAO across those contexts, but the surrounding wallet
software, address format, and signing surface remain important parts of the reader's interpretation.

The boundary also keeps Bittensor EVM scoped to Bittensor. The token-bridging documentation notes
that Bittensor EVM smart contracts execute on the Bittensor blockchain rather than on Ethereum.
Moving TAO into H160 context therefore means using the Bittensor EVM side, not sending TAO to a
separate Ethereum network.

Likewise, address conversion context does not transfer coldkey authority by itself. H160 and SS58
formats help readers identify which account context is being referenced, while signing authority
still comes from the wallet material used for that side of the interaction.

For readers, the useful split is asset, account, and execution location. TAO is the asset being
connected, SS58 and H160 are the account formats, and Bittensor remains the chain environment where
the native and EVM contexts meet. Confusing those layers can make a bridge example sound like an
asset change, a wallet migration, or an Ethereum transfer when the official documentation is
describing Bittensor-side bridging.

References: [Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao),
[H160 Address](https://docs.learnbittensor.org/evm-tutorials/convert-h160-to-ss58),
[Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials),
[Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets)

## Distinction from vTAO

Token bridging and vTAO name a connectivity topic and an EVM token. The
[Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao) documentation covers
moving TAO between SS58 and H160 account contexts on Bittensor and using vTAO as a bridge token to
other EVM chains, while vTAO is the EVM-side representation of staked TAO on Bittensor whose backing
balance grows with staking rewards
([Glossary: TAO](https://docs.learnbittensor.org/resources/glossary#tao-tau)).

- **Token bridging** — SS58/H160 movement on Bittensor and vTAO bridging to other EVM chains.
- **vTAO** — the EVM representation of staked TAO on Bittensor EVM.

## Distinction from Transfer

Token bridging and a transfer both move value, but the documentation frames them differently. An
ordinary transfer moves a balance between accounts in one wallet context, while token bridging can
move TAO between SS58 and H160 on Bittensor or move vTAO across EVM chains
([Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao),
[Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets)).

- **Transfer** — a balance movement between accounts in one wallet context.
- **Token bridging** — cross-format movement on Bittensor or vTAO movement across EVM chains.

## Distinction from H160 Address

Token bridging and the H160 address are related because H160 is the destination account format for
moving TAO into the EVM context. The
[Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao) documentation describes
moving TAO into Ethereum-style H160 wallets to use it in the EVM layer, and the
[H160 address](https://docs.learnbittensor.org/evm-tutorials/convert-h160-to-ss58) documentation
describes the format itself.

- **H160 address** — the EVM-side account format that can hold bridged TAO.
- **Token bridging** — the capability of moving TAO into that format.

SS58 and H160 are therefore account views on one network: bridging names a cross-format path on the
same Bittensor chain rather than a transfer to an external blockchain, so the format labels describe
how the asset is represented and signed, not which network settles the balance
([Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao),
[Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials)).

## Distinction from Staking

Token bridging and staking are related because vTAO, the EVM representation used in bridging, is a
staked form of TAO. The
[Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation)
describes staking as committing TAO to support a validator and participate in rewards, and the
[Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao) documentation describes
vTAO as representing staked TAO inside the EVM layer.

- **Staking** — committing TAO to support a validator and earn rewards.
- **Token bridging** — representing that staked TAO in the EVM context through vTAO.

## Distinction from Emission

Token bridging and emission are related because the staked TAO behind vTAO grows with staking
rewards over time. The [Emissions](https://docs.learnbittensor.org/learn/emissions) overview
describes how staked TAO earns rewards, and the
[Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao) documentation notes that
the underlying TAO represented by vTAO increases with those rewards.

- **Emission** — the reward flow that grows staked TAO.
- **Token bridging** — the EVM-side representation that tracks that growth through vTAO.

## Distinction from Bittensor EVM Smart Contracts

Token bridging is what makes TAO usable inside Bittensor's EVM smart-contract layer. The
[Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao) documentation notes that
Bittensor EVM smart contracts are executed solely on the Bittensor blockchain, not on the Ethereum
blockchain.

- **Token bridging** — moves TAO into the H160 account context on Bittensor.
- **Bittensor EVM smart contracts** — the on-chain EVM layer where bridged TAO can interact.

## Distinction from Yuma Consensus

Token bridging connects TAO between Substrate SS58 and EVM H160 account contexts on the Bittensor
chain. It names cross-format connectivity vocabulary, not the on-chain step that turns validator
weight submissions into emission shares
([Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao)).

Yuma Consensus is the on-chain process that converts submitted validator weights into emission
shares after evaluation completes on subnets. Bridging moves TAO between account formats; Yuma
Consensus settles validator weight outcomes and emission shares
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers should keep bridging vocabulary separate from the settlement step. Token bridging answers
how TAO moves between SS58 and EVM contexts; Yuma Consensus answers how validator weights become
emission shares in a subnet
([Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Token bridging** — connectivity between Substrate and EVM TAO account formats.
- **Yuma Consensus** — on-chain weight aggregation that settles emissions.

## Reader Boundary

This article defines token bridging and vTAO as concepts. It does not describe bridging steps,
wallet setup, exchange rates, tool-specific instructions, or supported chain lists. Readers should
use the official Bittensor EVM documentation for operational detail and current parameters.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For token
bridging, that sequence changes how readers should interpret evidence about bridge behavior and vTAO
conversion outcomes.

In localnet, token bridging mechanics can be tested in an isolated environment. Localnet bridge
behavior and conversion outcomes reflect local chain configuration and do not represent production
EVM bridge state.

On testnet, token bridging can be observed in a shared, non-production network. Testnet bridge
interactions and vTAO conversion outcomes are separate from mainnet production bridge state.

On mainnet, token bridging is a live operation that moves TAO representation between the Subtensor
layer and the Bittensor EVM environment. The
[Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao) documentation describes
the bridge mechanics that apply on the production network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A token bridging example from one environment should not
be read as representing bridge parameters or conversion rates in another environment.

## Bridging Moves TAO Between SS58 and H160 Contexts

The [Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao) documentation
describes moving the same TAO between Substrate-style SS58 wallets and Ethereum-style H160 wallets
on the Bittensor chain. Token bridging therefore names a cross-format path on one network, not a
transfer to a separate blockchain.

Readers should keep SS58 and H160 as account-context labels. Bridging connects native Bittensor
wallet operations with EVM wallet operations for the same TAO asset
([Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials)).

## The Bridged Asset Remains TAO

Official bridging material frames token bridging as letting participants use the same TAO inside the
EVM layer without changing the asset itself
([Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao)). Bridging vocabulary
should therefore stay on the TAO side of tokenomics, separate from subnet
[alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha) tokens discussed in
subnet markets.

A move into H160 context changes how the balance is represented and which wallet software signs, not
which network token is being held.

## Bridging Moves TAO, Not Subnet Alpha Inventory

Official bridging documentation frames vTAO as staked TAO represented for EVM contracts on
Bittensor's own chain. Subnet alpha tokens remain subnet-local assets traded and staked through
subnet liquidity pools rather than through the EVM bridge path
([Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao),
[Understanding Subnets: Liquidity Pools](https://docs.learnbittensor.org/subnets/understanding-subnets#liquidity-pools),
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)).

For readers, token bridging vocabulary should stay on the TAO and vTAO side of tokenomics. Alpha
positions created by staking into a subnet pool are a separate custody and market context from
moving staked TAO into an H160 balance
([Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation),
[Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials)).

Keeping that boundary clear prevents bridge examples from being read as instructions for subnet
alpha transfers or subnet-pool arbitrage
([Understanding Subnets: Price/Rate of Alpha Tokens](https://docs.learnbittensor.org/subnets/understanding-subnets#pricerate-of-alpha-tokens)).

## vTAO Tracks Staked TAO That Continues Accruing Rewards

Bridging documentation describes vTAO as the EVM representation of staked TAO on Bittensor's own
chain. The underlying staked TAO behind vTAO can grow as staking rewards accrue, so the bridged
balance is tied to emission and delegation mechanics rather than to a static one-time deposit
([Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao),
[Emissions](https://docs.learnbittensor.org/learn/emissions),
[Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation)).

For readers, token bridging therefore changes account context and signing software, not the reward
path of the staked asset. Emission vocabulary still applies to the staked TAO backing vTAO, while
bridge vocabulary names how that position is represented for EVM contracts
([Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials),
[Glossary: TAO](https://docs.learnbittensor.org/resources/glossary#tao-tau)).

Subnet alpha positions remain outside that bridge path and continue to use pool and staking
vocabulary on their home subnet
([Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha),
[Understanding Subnets: Liquidity Pools](https://docs.learnbittensor.org/subnets/understanding-subnets#liquidity-pools)).

## Bridge Direction Does Not Change Underlying Stake Rewards

Bridging documentation describes vTAO as the EVM-side representation of staked TAO that can keep
accruing staking rewards on Bittensor's own chain. Moving into H160 format therefore changes wallet
and signing context, not the emission mechanics behind the staked position
([Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

For readers, token bridging vocabulary should stay paired with delegation and emission references.
Bridge prose explains cross-format custody; it does not replace staking reward documentation
([Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation),
[Glossary: TAO](https://docs.learnbittensor.org/resources/glossary#tao-tau)).
