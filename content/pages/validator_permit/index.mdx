# Validator Permit

A validator permit is a Bittensor permission flag for validation rights inside a subnet. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#validator-permit) defines it as a
boolean flag for whether a specific subnet participant has validation rights.

## Subnet Context

Validator permits are subnet-specific. A participant can have validation rights in one subnet
without that statement automatically applying to every other subnet. This is why the glossary also
defines [VPermit](https://docs.learnbittensor.org/resources/glossary#vpermit) as a delegate-level
view of which subnet IDs a delegate is authorized to validate on.

## Relationship to VPermit

Validator permit and VPermit name subnet validation permission at different scopes. The
[Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit)
describes a boolean flag for validation rights inside one subnet, while the
[Glossary: VPermit](https://docs.learnbittensor.org/resources/glossary#vpermit) describes the subnet
IDs where a delegate is authorized to validate.

For readers, validator permit answers a one-subnet permission question, while VPermit summarizes
those permissions across subnets for a delegate. A delegate's VPermit aggregates individual
validator permits rather than replacing them.

References:
[Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit),
[Glossary: VPermit](https://docs.learnbittensor.org/resources/glossary#vpermit)

## Relationship to Stake Weight

The glossary ties validator permits to stake weight by describing them as awarded to the top
participants by stake-weight position. At Taopedia level, the important point is that a permit is
not a general reputation label. It is a subnet validation permission connected to how validator
eligibility is represented for that subnet.

## Hyperparameter Context

Validator permit capacity on a subnet is bounded by the `max_allowed_validators` hyperparameter. The
[Subnet Hyperparameters: MaxAllowedValidators](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#maxallowedvalidators)
documentation lists this as the maximum number of validators on a subnet, with a documented default
of 64. Readers should treat validator permit as a subnet permission flag whose allocation is
constrained by that configured ceiling.

## Relationship to Multiple Mechanisms

A validator permit is usually discussed for one incentive mechanism at a time, but subnets can also
run multiple mechanisms in parallel. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms) notes
that validators must evaluate miners separately for each mechanism.

For readers, a validator permit still marks validation rights inside a subnet, but the mechanism
context matters. Evaluation duties for one mechanism should be read separately when a subnet uses
more than one evaluation system.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)

## Reader Boundary

This article defines the term at a high level. It does not list current permit holders, rank
validators, describe staking or registration steps, or restate operational requirements. Readers
should use official validator references for those details.
