# TAO

TAO is the network token used in Bittensor incentive, staking, subnet reserve, and emissions
context. The [Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction)
describes TAO as the token at the center of Bittensor's incentive system, while
[Emission](https://docs.learnbittensor.org/learn/emissions) describes newly created TAO alongside
subnet-specific alpha tokens.

TAO is network-scoped. In Dynamic TAO, it appears beside subnet-scoped alpha tokens, which makes the
distinction between TAO and subnet alpha important in staking, pool, and emission discussions.

## Network Token Role

TAO is the token of the Bittensor network as a whole. The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction) documentation places
TAO in the network's incentive design.

This makes TAO broader than any one subnet. A subnet can have its own alpha token, but TAO remains
the network-level token used across Bittensor.

## Emission Role

TAO is part of Bittensor's emission system. The
[Emission](https://docs.learnbittensor.org/learn/emissions) documentation describes emissions as
newly created TAO and subnet-specific alpha tokens moving through the system.

That means TAO should be read separately from subnet alpha. Both can appear in the same emission
discussion, but TAO names the network token while alpha names subnet-specific assets.

## Subnet Reserve Role

TAO also appears inside subnet pool language. The
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) documentation
describes subnet pools with a TAO reserve and a subnet-specific alpha reserve.

In that pool setting, TAO is the network-token side of the pair. Alpha reserve is the subnet-token
side. Dynamic TAO connects those two sides inside subnet tokenomics.

## Staking Role

TAO can be used in staking context. The
[Staking and Delegation](https://docs.learnbittensor.org/staking-and-delegation/delegation)
documentation describes staking and delegation as ways to support validators.

In subnet staking, TAO interacts with subnet alpha through the subnet's tokenomics. In root-subnet
context, stake is expressed in TAO. The key reader-facing point is that staking language can involve
TAO, alpha, or the relationship between them.

## RAO Denomination

RAO is a smaller denomination of TAO, not a separate token system. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#rao) defines RAO as one billionth of a
TAO.

This makes RAO useful for precise amounts. TAO names the network token at the familiar unit, while
RAO names the finer denomination of the same token.

## Alpha Token Contrast

TAO and alpha tokens have different scopes. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#alpha-tokens) describes alpha tokens
as subnet-specific assets, while TAO is the network token.

That contrast is central to Dynamic TAO. TAO can sit beside alpha in subnet pools and emissions, but
the two names do not refer to the same asset.

## Term Placement

TAO names the network token. RAO names a smaller denomination of TAO. Alpha tokens name
subnet-specific assets. TAO reserve names the pool-held TAO side of a subnet pool
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

These terms are related because they often appear together in subnet tokenomics, but each one names
a different part of the system.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. The TAO token
concept applies across the Bittensor lifecycle: TAO-denominated incentives and staking can be
observed in localnet for isolated testing, evaluated on testnet for shared non-production
conditions, and relied upon on mainnet for live operation with real emissions.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. TAO amount examples or emission outcomes from one
environment should not be read as representing production subnet performance in another environment.
