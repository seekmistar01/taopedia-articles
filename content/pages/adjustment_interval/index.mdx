# Adjustment Interval

Adjustment interval is a per-subnet hyperparameter, listed as `AdjustmentInterval` in Bittensor
documentation, that sets the number of blocks between automatic recalibrations of the dynamic
registration burn cost. The
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) reference
lists `AdjustmentInterval` alongside the burn band parameters that define the floor and ceiling of
the registration cost being adjusted
([Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register)).

The term names the recalibration cadence. It does not set the burn cost itself; it sets the block
window over which actual registrations are compared to the target registration rate to decide
whether the cost moves up, down, or holds.

## What It Controls

The hyperparameter determines how frequently the subnet's registration burn is recalibrated against
the target registration rate. Each adjustment interval, the network compares how many registrations
occurred during the window to the target rate set by the subnet owner. If actual registrations
exceeded the target, the burn cost rises; if they fell short, the burn cost eases toward the floor
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Burn Cost](https://docs.learnbittensor.org/resources/glossary#burn-cost)).

A shorter adjustment interval means the burn cost responds to registration demand more frequently on
netuid 1 and other active subnets, so spikes in registration activity are reflected in cost sooner.
A longer interval smooths the recalibration over more blocks, making the burn cost less reactive to
short-term fluctuations.

The interval therefore shapes the responsiveness of the registration cost mechanism without setting
the cost band itself. The burn floor and ceiling are separate parameters that define the range
inside which each adjustment operates.

## Distinction from Target Registrations Per Interval

Adjustment interval sets the block window between burn recalibrations. Target registrations per
interval sets how many registrations the mechanism aims for during that window
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Adjustment interval** — blocks between demand-comparison recalibrations.
- **Target registrations per interval** — reference registration count for each window.

## Documented Type and Setter

The [Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)
reference lists `AdjustmentInterval` among the registration-related hyperparameters that subnet
owners can set. Each subnet carries its own hyperparameter state, so the adjustment interval for
netuid 1 can differ from the interval configured on another subnet.

The active value is the per-subnet chain state for that subnet. A reading from one subnet should not
be treated as evidence for the adjustment behavior on any other subnet. Subnet-specific examples
should identify the subnet and its current hyperparameter values.

## Distinction from Burn Half Life

Burn half life and adjustment interval both affect how the registration burn cost changes over time,
but they control different parts of the mechanism
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)). Burn
half life sets the decay horizon over which the elevated burn cost eases back toward the floor
between registration events; adjustment interval sets the block window between demand-comparison
recalibrations.

The two parameters complement each other. Adjustment interval governs when the mechanism
recalibrates the burn cost against demand. Burn half life governs how quickly an elevated burn
decays between registration events inside that cycle. A subnet owner can tune both independently to
shape registration cost behavior on netuid 1 or any other subnet.

- **Adjustment interval** — block window between demand-comparison recalibrations.
- **Burn half life** — decay horizon easing elevated burn between registrations.

## Distinction from Min Burn and Max Burn

Min burn and max burn set the floor and ceiling of the registration burn band. Adjustment interval
sets the block window between demand-comparison recalibrations inside that band
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Burn Cost](https://docs.learnbittensor.org/resources/glossary#burn-cost)).

The band parameters name cost limits; adjustment interval names recalibration cadence.

- **Min burn / max burn** — floor and ceiling of the registration burn band.
- **Adjustment interval** — block window between recalibrations inside that band.

## Distinction from Register

Register names the subnet entry process that places a miner or validator into a UID slot. Adjustment
interval names how often the dynamic registration burn is recalibrated against demand
([Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

Registration events occur inside the mechanism; adjustment interval names the hyperparameter that
sets the comparison window between recalibrations.

- **Register** — the subnet entry action claiming a UID slot.
- **Adjustment interval** — how often the dynamic burn is recalibrated against demand.

## netuid 1 as a Reference Subnet

Hyperparameter examples in this article use netuid 1 as a documented Finney mainnet subnet because
that netuid appears repeatedly in official subnet hyperparameter prose
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

Adjustment interval is per-subnet chain state. The block window configured on netuid 1 governs how
often that subnet recalibrates registration burn against its target registration rate, but the live
value can differ from reference defaults and from the interval set on another netuid
([Glossary: Burn Cost](https://docs.learnbittensor.org/resources/glossary#burn-cost)).

Readers comparing registration-cost responsiveness across subnets should treat netuid 1 as an
illustrative subnet label rather than proof that every subnet shares the same adjustment cadence.
Each netuid carries its own hyperparameter table, and adjustment interval names only the
recalibration window inside that table
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Adjustment interval** — block window between registration burn recalibrations on a subnet.
- **netuid 1** — documented example subnet used for Finney mainnet hyperparameter illustrations.

## Live Values on One Subnet

Adjustment interval is stored separately for each subnet. The block window configured on one netuid
governs how often that subnet recalibrates dynamic registration burn against its target rate, and
the live interval can differ from reference defaults and from the window on another netuid
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Burn Cost](https://docs.learnbittensor.org/resources/glossary#burn-cost)).

Readers should not treat a documented default interval as proof of live chain state everywhere. A
subnet owner may set a shorter or longer recalibration cadence on one netuid while another subnet
keeps a different adjustment window under its own hyperparameter row
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

Adjustment interval names registration-burn recalibration cadence. Yuma Consensus still describes
how validator weights are settled into emission shares on the active subnet each tempo
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

This article's infobox uses netuid 1 as an example when discussing one subnet at a time on Finney
mainnet. That label helps readers anchor examples; it does not mean every subnet shares the same
live adjustment interval
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

## Distinction from Yuma Consensus

Adjustment interval is a per-subnet hyperparameter that sets the block window between dynamic
registration burn recalibrations. Yuma Consensus is the separate tempo process that converts
validator weight submissions into emission shares each epoch
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Adjustment interval** — block window between burn recalibrations.
- **Yuma Consensus** — mechanism that turns validator weights into emission shares. Readers should
  keep retarget-cadence vocabulary separate from settlement vocabulary. Adjustment interval names
  how often certain hyperparameters may change; Yuma converts validator weight submissions into
  emission shares at epoch boundaries on the active subnet
  ([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
  [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Hyperparameter cadence governs rule updates. Yuma governs reward settlement from validator weights
([Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Tempo** — blocks per epoch on the same subnet.

## Reader Boundary

Adjustment interval should not be read as a burn cost value, a registration count, or a miner's
registration status. It names only the subnet hyperparameter that sets the block window between
demand-comparison recalibrations on the selected netuid
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).
