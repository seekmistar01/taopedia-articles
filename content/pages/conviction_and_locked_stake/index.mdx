# Conviction and Locked Stake

Conviction and locked stake describe a Bittensor staking feature where subnet alpha stake can be
locked so that it builds a time-based commitment signal
([Conviction and locked stake](https://docs.learnbittensor.org/staking-and-delegation/conviction-staking),
[Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation)).

The feature is commitment vocabulary. It does not create a separate token or a separate subnet role.

## Locked Stake

Locked stake is subnet alpha stake bound inside a specific subnet lock. The official conviction
documentation explains that the lock prevents the staked alpha balance on that subnet from dropping
below the locked amount
([Conviction and locked stake](https://docs.learnbittensor.org/staking-and-delegation/conviction-staking),
[Glossary: Alpha Tokens](https://docs.learnbittensor.org/resources/glossary#alpha-tokens)).

For readers, the distinction is between stake that supports a position and stake that is also locked
for commitment signaling. Locking adds a constraint to a portion of staked alpha.

## Conviction Score

Conviction is the score that grows from locked stake over time. The conviction documentation
describes the score as a public signal of long-term commitment and explains the score behavior in
the lock model
([Conviction score](https://docs.learnbittensor.org/staking-and-delegation/conviction-staking#the-conviction-score),
[Conviction and locked stake](https://docs.learnbittensor.org/staking-and-delegation/conviction-staking)).

At reference level, conviction should be read as maturing commitment rather than a simple balance
number. The locked amount and the passage of time both matter.

## Lock Modes

The conviction documentation describes decaying and perpetual modes. Decaying mode reduces the
locked amount over time, while perpetual mode keeps the locked amount from decaying until the mode
changes
([Decaying and perpetual modes](https://docs.learnbittensor.org/staking-and-delegation/conviction-staking#decaying-and-perpetual-modes),
[Conviction and locked stake](https://docs.learnbittensor.org/staking-and-delegation/conviction-staking)).

This mode distinction is why conviction should be read as time-sensitive staking context. The mode
changes how the commitment signal evolves.

## Subnet Alpha Context

Conviction and locked stake apply to subnet alpha already staked inside a specific subnet. Alpha
token documentation describes alpha as subnet-specific token context, and staking documentation
places stake inside subnet-specific staking paths
([Glossary: Alpha Tokens](https://docs.learnbittensor.org/resources/glossary#alpha-tokens),
[Staking into Subnets](https://docs.learnbittensor.org/staking-and-delegation/delegation#staking-into-subnets)).

This keeps conviction inside subnet-alpha staking. It is not a root-TAO lock and not a separate
asset layer from the alpha being committed.

## Relationship to Stake

Stake and locked stake are related but different scopes. Stake names the validator-associated amount
inside subnet staking, while locked stake names the portion of subnet alpha constrained by the lock
feature ([Glossary: Stake](https://docs.learnbittensor.org/resources/glossary#stake),
[Conviction and locked stake](https://docs.learnbittensor.org/staking-and-delegation/conviction-staking)).

An alpha position can be staked without being locked. Conviction begins when part of that staked
alpha is locked for commitment signaling.

## Reader Boundary

Conviction should be read as a commitment-signal concept. It is not a validator recommendation,
complete staking status, or protection from subnet deregistration
([Subnet deregistration note](https://docs.learnbittensor.org/staking-and-delegation/conviction-staking#subnet-deregistration),
[Conviction and locked stake](https://docs.learnbittensor.org/staking-and-delegation/conviction-staking)).

The stable article-level point is that locked subnet alpha can build conviction as a visible
long-term commitment signal.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For conviction and
locked stake, that sequence changes how readers should interpret examples of conviction scores, lock
modes, and delegated-stake behavior.

In localnet, conviction and locked stake setups can be exercised in an isolated environment.
Localnet examples reflect local chain state and locally configured lock parameters rather than
production subnet stake behavior.

On testnet, conviction and locked stake behavior can be observed in a shared, non-production
network. Testnet examples show how lock modes, conviction voting, and stake delegation behave under
shared chain conditions while staying separate from mainnet subnet state.

On mainnet, conviction and locked stake describe live delegated-stake behavior on the production
Bittensor network. The
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) reference
describes the staking and delegation context that applies on the production network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A conviction score or locked-stake outcome from one
environment should not be read as evidence of production subnet behavior in another environment.
