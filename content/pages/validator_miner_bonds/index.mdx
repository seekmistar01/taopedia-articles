# Validator-Miner Bonds

Validator-miner bonds are Bittensor consensus signals that connect validator evaluation to miner
outcomes over time. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds) defines bonds
as part of the relationship between a validator and a miner, while
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) explains how validator
evaluation, clipping, bonding, and emissions fit together.

## Consensus Bonding

In Yuma Consensus, validators evaluate miner work and those evaluations are resolved into subnet
outcomes. Bonds sit in that consensus flow after validator weights are compared with the consensus
benchmark. The [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) page places
bonding after clipping and before validator-side emission calculation.

## Smoothing Over Time

Validator-miner bonds are not just a one-moment signal. The
[Exponential Moving Averages](https://docs.learnbittensor.org/learn/ema) page explains that
Bittensor uses EMAs of validator-miner bond strengths when determining emissions inside each subnet.
This smoothing keeps bond behavior tied to a running history rather than only the latest evaluation.

## Relationship to Evaluation

Subnets define useful work, miners produce that work, and validators evaluate it.
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) describes
that miner-and-validator model, while
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) explains how validator
rankings become inputs to consensus. Bonds help connect that evaluation path to validator-side
outcomes.

## Out-of-Consensus Penalty Context

Bonds can also reflect a penalty when a validator's evaluation moves away from consensus. The
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus#penalizing-out-of-consensus-bonds)
documentation explains that when a validator's weight for a miner exceeds that miner's consensus
benchmark, the bond-weight can be blended from the validator's raw weight toward the
consensus-clipped weight. A configurable penalty factor controls how far the bond-weight moves in
that direction, so the amount of above-consensus evaluation that survives into the bond depends on
how the subnet sets that factor.

For readers, the distinction is between the validator's submitted evaluation and the
consensus-aligned bond value that feeds emissions. Because the penalty is configurable, an
out-of-consensus evaluation may still contribute some bond strength, but the configured factor
determines how closely the bond-weight tracks consensus rather than the validator's raw opinion.

## Relationship to Multiple Mechanisms

Validator-miner bonds are usually discussed in a single-mechanism subnet context, but subnets can
also run multiple incentive mechanisms in parallel. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms) defines
multiple incentive mechanisms as independent evaluation systems, each with its own bond pool for
Yuma Consensus calculations.

For readers, bonds still describe how validator evaluation connects to subnet outcomes over time,
but the mechanism context matters. Bond behavior in one mechanism does not substitute for bond
context in another when a subnet runs parallel evaluation paths.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)

## Interpretation Boundaries

Validator-miner bonds should be read as a consensus concept, not as a social endorsement, a
standalone quality score, or a user-facing staking guide. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds) connects the
term to Yuma Consensus, and [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)
places it inside the broader clipping, bonding, and emission process.
