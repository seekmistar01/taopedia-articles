# EdDSA Cryptographic Keypairs

EdDSA cryptographic keypairs are Bittensor wallet-key vocabulary for public and private key pairs
generated with EdDSA. The
[EdDSA Cryptographic Keypairs glossary entry](https://docs.learnbittensor.org/resources/glossary#eddsa-cryptographic-keypairs)
places the term in the keypair layer behind Bittensor key roles.

## Public and Private Sides

A keypair has a public side and a private side. The
[Public Key glossary entry](https://docs.learnbittensor.org/resources/glossary#public-key) places
public keys in shareable verification and account-reference context, while the
[Private Key glossary entry](https://docs.learnbittensor.org/resources/glossary#private-key) places
private keys in authorization context. EdDSA keypairs supply that paired structure.

## Wallet-Key Vocabulary

In Bittensor, EdDSA keypairs belong to wallet-key terminology rather than subnet, consensus, or
tokenomics terminology. The [wallet documentation](https://docs.learnbittensor.org/keys/wallets)
provides the broader setting for Bittensor wallet concepts, while the EdDSA glossary entry narrows
the subject to the cryptographic keypair layer.

## Distinction from Public Key

Public key names the shareable side of a keypair used for account reference and verification. EdDSA
keypairs name the paired structure both sides come from
([Glossary: Public Key](https://docs.learnbittensor.org/resources/glossary#public-key),
[Glossary: EdDSA Cryptographic Keypairs](https://docs.learnbittensor.org/resources/glossary#eddsa-cryptographic-keypairs)).

One side of an EdDSA keypair is public shareable material used for account reference and signature
verification. The other side is private authorization material that must stay protected. The
glossary entries for public key and private key supply that split, while EdDSA keypairs name the
paired structure both sides come from.

That split helps readers avoid treating a public key like signing authority, or a private key like a
destination address. The keypair vocabulary names both sides together; public-key and private-key
vocabulary names what each side does in wallet use.

Wallet documentation remains the place for operational key-handling detail; EdDSA keypairs name the
cryptographic pair structure those roles build on.

## Distinction from Private Key

Private key names the protected signing side of a keypair. EdDSA keypairs name the algorithm context
for the pair rather than key-handling procedures
([Glossary: Private Key](https://docs.learnbittensor.org/resources/glossary#private-key),
[Glossary: EdDSA Cryptographic Keypairs](https://docs.learnbittensor.org/resources/glossary#eddsa-cryptographic-keypairs)).

## EdDSA Keypairs Name Signing Material, Not Balances

EdDSA cryptographic keypairs should not be read as TAO balances, subnet roles, or wallet application
labels. The term names the cryptographic pair structure behind Bittensor signing material
([Glossary: EdDSA Cryptographic Keypairs](https://docs.learnbittensor.org/resources/glossary#eddsa-cryptographic-keypairs),
[Bittensor wallet documentation](https://docs.learnbittensor.org/keys/wallets)).

Coldkey and hotkey vocabulary describes how generated key material is used after the pair exists.
EdDSA names the algorithm context for that paired generation step.

## Algorithm Labels Do Not Replace Key-Management Procedures

EdDSA keypair vocabulary does not provide wallet creation steps, backup policy, or signing
instructions. Key-management guidance covers creation, protection, and use of wallet material
([Working with Keys](https://docs.learnbittensor.org/keys/working-with-keys),
[Handle your Seed Phrase/Mnemonic Securely](https://docs.learnbittensor.org/keys/handle-seed-phrase)).

Readers should keep procedural claims tied to those references rather than to algorithm labels
alone.

## Mnemonic Material Differs From the Keypair Label

Mnemonic vocabulary covers regeneration after loss, while EdDSA keypairs name the cryptographic pair
structure produced during wallet generation. Recovery phrases are not the same label as the
algorithm context
([Glossary: Mnemonic](https://docs.learnbittensor.org/resources/glossary#mnemonic),
[Glossary: EdDSA Cryptographic Keypairs](https://docs.learnbittensor.org/resources/glossary#eddsa-cryptographic-keypairs)).

## Wallet Creation Generates a Fresh Key Pair

[Working with Keys](https://docs.learnbittensor.org/keys/working-with-keys) describes wallet
creation as generating cryptographic key material for coldkeys and hotkeys. EdDSA keypair vocabulary
sits at that generation layer rather than at subnet participation or balance labels
([Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets),
[Glossary: EdDSA Cryptographic Keypairs](https://docs.learnbittensor.org/resources/glossary#eddsa-cryptographic-keypairs)).

For readers, keypair generation is upstream of wallet roles. Coldkey and hotkey labels describe how
generated pairs are used inside wallet structure.

## SS58 Addresses Display Public Key Destinations

The [Glossary: SS58 Encoded](https://docs.learnbittensor.org/resources/glossary#ss58-encoded)
describes the compact format used for public-key-derived addresses, while the
[Glossary: Wallet Address](https://docs.learnbittensor.org/resources/glossary#wallet-address)
describes the destination identifier used for TAO movement. Those terms belong to the public side of
wallet vocabulary rather than to the protected private pair
([Glossary: Public Key](https://docs.learnbittensor.org/resources/glossary#public-key)).

A precise claim should keep address-display vocabulary separate from private signing material.

## Key Pair Scope

EdDSA cryptographic keypairs should be read as the cryptographic key material that backs a Bittensor
wallet account, not as a separate account type or balance unit. Wallet and working-with-keys
references explain how the keypair relates to coldkey and hotkey structure, while the glossary entry
anchors the term itself
([Bittensor Wallets](https://docs.learnbittensor.org/getting-started/wallets),
[Glossary: EdDSA Cryptographic Keypairs](https://docs.learnbittensor.org/resources/glossary#eddsa-cryptographic-keypairs)).

For readers, a precise claim should keep the keypair vocabulary distinct from account, subnet, or
TAO-balance language. The keypair names the signing material, while wallet structure and subnet
participation name the higher-level context that uses that material.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For EdDSA
cryptographic keypairs on a subnet such as netuid 1, that sequence gives readers a boundary for
interpreting keypair examples and signing-context notes.

In localnet, EdDSA keypairs on netuid 1 can be generated and used in an isolated environment. Local
signing examples reflect that test chain rather than production wallet behavior.

On testnet, keypair-backed registration and subnet actions on netuid 1 apply in a shared
non-production network separate from mainnet.

On mainnet, EdDSA keypair examples on netuid 1 should be read as live production signing behavior on
the production Bittensor network
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, so outcomes from one environment should not be treated as
proof of behavior in another.

## Distinction from H160 Address

EdDSA cryptographic keypairs are a native Bittensor wallet key scheme of public and private keys
generated with EdDSA, while an H160 address is the Ethereum-style account format used by EVM wallets
in Bittensor EVM ([Wallets](https://docs.learnbittensor.org/getting-started/wallets),
[Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials)).

Native-keypair vocabulary names the EdDSA signing scheme; H160 vocabulary names the EVM-side address
format.

## Distinction from Mnemonic

Mnemonic names recovery words that can regenerate keys. EdDSA keypairs name the cryptographic pair
structure produced during wallet generation
([Glossary: Mnemonic](https://docs.learnbittensor.org/resources/glossary#mnemonic),
[Glossary: EdDSA Cryptographic Keypairs](https://docs.learnbittensor.org/resources/glossary#eddsa-cryptographic-keypairs)).

## Distinction from Coldkey

EdDSA cryptographic keypairs name the paired signing structure Bittensor wallet keys are built from.
A coldkey is the custody key that holds balances and authorizes high-impact wallet actions
([Glossary: EdDSA Cryptographic Keypairs](https://docs.learnbittensor.org/resources/glossary#eddsa-cryptographic-keypairs),
[Glossary: Coldkey](https://docs.learnbittensor.org/resources/glossary#coldkey)).

- **EdDSA cryptographic keypairs** — algorithm context for paired signing material.
- **Coldkey** — custody key role after key material is generated.

Pair vocabulary names cryptographic structure; coldkey vocabulary names custody authority in wallet
use ([Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets)).

The
[Glossary: EdDSA Cryptographic Keypairs](https://docs.learnbittensor.org/resources/glossary#eddsa-cryptographic-keypairs)
connects EdDSA keypairs to coldkeys and hotkeys in Bittensor wallet vocabulary. Those role labels
describe how generated key material is used on the network, while EdDSA names the cryptographic
method behind the paired public and private keys
([Glossary: Coldkey](https://docs.learnbittensor.org/resources/glossary#coldkey)).

Readers should keep algorithm vocabulary separate from role vocabulary. EdDSA keypairs explain the
pair structure; coldkey and hotkey explain which wallet role uses that material for custody or
routine subnet actions.

## Distinction from Yuma Consensus

EdDSA cryptographic keypairs name the key algorithm behind Bittensor coldkeys and hotkeys. Yuma
Consensus is the separate subnet mechanism that converts weight submissions signed by those keys
into emission shares each tempo
([Glossary: EdDSA Cryptographic Keypairs](https://docs.learnbittensor.org/resources/glossary#eddsa-cryptographic-keypairs),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

- **EdDSA keypairs** — cryptographic keys for Bittensor accounts.
- **Yuma Consensus** — on-chain settlement that turns validator weights into emission shares. EdDSA
  keypairs name signing material; Yuma processes validator weights into rewards
  ([Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions)).

On a subnet such as netuid 1, a [hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey)
derived from an EdDSA keypair can sign registration, weight submission, and other subnet-facing
actions ([Working with Keys](https://docs.learnbittensor.org/keys/working-with-keys)).

Those signed actions feed the subnet incentive path, but EdDSA keypair vocabulary names the signing
material layer rather than
[validator weights](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus), or
[emission](https://docs.learnbittensor.org/learn/emissions) outcomes themselves.

## Distinction from Algorithm

EdDSA names the signature algorithm context, while a keypair names the paired public and private
material produced under that context. In the
[EdDSA glossary entry](https://docs.learnbittensor.org/resources/glossary#eddsa-cryptographic-keypairs),
the term is used for the cryptographic method behind Bittensor public/private keypair terminology.

EdDSA names the signature algorithm context for Bittensor public and private key pairs. It does not
name a transfer, staking action, registration step, or subnet role by itself. The
[Glossary: Public Key](https://docs.learnbittensor.org/resources/glossary#public-key) and
[Glossary: Private Key](https://docs.learnbittensor.org/resources/glossary#private-key) entries
describe what each side of the pair is used for after the keypair exists.

For readers, EdDSA belongs in wallet-security reading when documentation discusses how Bittensor key
material is generated and signed, not when it discusses emissions, validator weights, or subnet
tasks.

## Reader Boundary

EdDSA cryptographic keypairs should not be read as wallet applications, recovery phrases, subnet
roles, or TAO balances. The term names the cryptographic keypair context behind Bittensor signing
material
([Glossary: EdDSA Cryptographic Keypairs](https://docs.learnbittensor.org/resources/glossary#eddsa-cryptographic-keypairs)).
