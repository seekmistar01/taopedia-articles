# Subtensor Events

Subtensor events are named records emitted by the Subtensor runtime. The official
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events) reference describes the
page as a list of runtime events emitted by Subtensor.

## What Events Represent

Events identify activity that the runtime records under a named event. The official reference lists
each event by runtime area and event name, then gives a short summary for many entries. This makes
events a structured reference for interpreting chain activity, not a narrative explanation of the
whole action that produced the record.

Reference: [Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)

## Runtime Areas

The events reference is grouped by runtime areas such as balances, commitments, contracts, EVM,
proxy, registry, safe mode, system, transaction payment, utility, swap, and the Subtensor module.
That grouping helps readers locate the area that emitted a record before reading the specific event
name and summary.

Reference: [Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)

## Bittensor-Specific Context

Some listed events are specific to Bittensor concepts. The Subtensor module section includes event
families for subnet identity, subnet settings, staking activity, weights, root claims, and other
Bittensor runtime behavior. Other sections cover more general runtime areas such as balances,
system, proxy, and utility.

Reference: [Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)

## Careful Interpretation

An event name is a compact runtime record. It should be read with its section, event summary, and
surrounding chain context. A single event entry can show that a record exists, but it should not be
treated as a complete guide to the underlying Bittensor mechanism.

Reference: [Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)

## Relationship to Multiple Mechanisms

Subtensor events include weight-related records from Bittensor subnet activity. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-matrix) notes that each
incentive mechanism has its own weight matrix for independent consensus calculations.

For readers, an event name still identifies a runtime record. Weight-related events should be read
with per-mechanism matrix context when a subnet runs more than one incentive mechanism.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)

## Relationship to Extrinsics

Subtensor events and extrinsics address related but different parts of a submitted chain action. An
extrinsic is a state-changing runtime request, while a Subtensor event is a named record the runtime
emits when activity is recorded. The
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) reference lists
submitted runtime methods, and the
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events) reference lists named
events emitted by the runtime.

For readers, an extrinsic is the submitted action, while an event is a compact runtime record that
may appear when that action is processed. Events help identify activity, but they should not be read
as a complete description of the underlying extrinsic.

References: [Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)

## Relationship to Blocks

Subtensor events and blocks address related but different parts of chain activity. The
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block) describes a block as
containing a collection of transactions, while Subtensor events are runtime records emitted as
accepted actions are processed. The
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events) reference groups events by
runtime area and event name.

For readers, a block names chain ordering and inclusion, while an event names a specific runtime
record associated with activity inside that processing context. An event entry shows that a record
exists, not the full narrative of every action in a block.

References: [Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)

## Relationship to Subtensor Storage

Subtensor events and Subtensor storage address related but different views of runtime state. Storage
entries expose where chain state is recorded, while events expose named runtime activity records.
The [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) reference lists
storage query definitions, and the
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events) reference lists emitted
event names grouped by runtime area.

For readers, storage answers where state lives, while events answer what runtime record was emitted
during activity. Both references should be read with surrounding chain context rather than as
standalone behavioral guides.

References: [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)

## Relationship to Network Context

Subtensor events are read inside a specific Bittensor network environment. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet as deployment contexts, and runtime event records belong to
whichever network environment produced them.

For readers, this keeps event interpretation network-scoped rather than network-agnostic. A mainnet
event record, a testnet event record, and a localnet event record all use the same event vocabulary,
but each one is read against the network context it belongs to.

References: [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)

## Relationship to Subtensor

Subtensor events and Subtensor are related but different parts of Bittensor chain vocabulary.
Subtensor names Bittensor's layer 1 blockchain and system of record, while Subtensor events name
on-chain notifications emitted after runtime actions are processed within that chain layer. The
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor) describes the
blockchain layer, and the [Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)
reference lists emitted event names grouped by runtime area.

For readers, Subtensor names the chain layer, while Subtensor events name the event records emitted
from runtime activity on that layer.

References: [Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)

## Relationship to Mempool

Subtensor events and the mempool are related but different stages of Bittensor chain activity. The
mempool names pending transactions before block inclusion, while Subtensor events name runtime
notifications emitted after accepted actions are processed into blocks. The
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) describes the
pending-transaction context, and the
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events) reference lists emitted
event names grouped by runtime area.

For readers, the mempool belongs to the pre-inclusion waiting stage, while Subtensor events belong
to the post-processing notification stage.

References: [Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)

## Relationship to Netuid

Subtensor events and a netuid are related but different parts of Bittensor vocabulary. A netuid
identifies a subnet on the network, while Subtensor events name runtime notifications that can
include subnet-scoped activity recorded on chain. The
[Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid) places netuid at the
subnet level, and the [Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)
reference lists emitted event names grouped by runtime area.

For readers, a netuid selects which subnet is in scope, while Subtensor events name runtime records
that may include activity tied to that subnet context.

References: [Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)

## Relationship to Transaction Fees

Subtensor events and transaction fees are related but different parts of Bittensor chain vocabulary.
Transaction fees name the costs attached to state-changing actions on Subtensor, while Subtensor
events name runtime notifications emitted after those actions are processed. The
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) documentation describes
fee-bearing actions, and the
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events) reference lists emitted
event names grouped by runtime area.

For readers, transaction fees belong to the cost model for chain actions, while Subtensor events
belong to the notification records emitted when those actions are processed.

References: [Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)
