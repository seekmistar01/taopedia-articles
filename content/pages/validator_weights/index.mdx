# Validator Weights

Validator weights are evaluation signals set by subnet validators. They express how validators
assess miner work inside a subnet before Yuma Consensus filters and aggregates those signals into
subnet outcomes
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The term is subnet-specific. A validator weight only has meaning inside the task, scoring method,
and incentive context that produced it, so it is not the same thing as a general validator
reputation score.

## Evaluation Signal

A validator weight is not a generic reputation score. It is a subnet-specific evaluation of miner
work under the rules of the subnet's incentive mechanism
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

That makes validator weights structured evaluation vocabulary. Similar-looking values can carry
different meaning across subnet tasks because the subnet's work target and scoring method give the
signal its interpretation.

The evaluation role is the first boundary. Validator weights describe miner-work assessments inside
the subnet; they do not describe stake position, permit status, or the whole standing of the
validator that submitted them.

## Consensus Input Role

Validators use weight signals to express relative evaluations of miners. Yuma Consensus combines
those signals into a broader consensus result rather than treating one validator's view as final on
its own ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

This aggregation step is why validator weights matter to rewards. They are evaluation inputs that
help shape how incentives and dividends are distributed inside a subnet, while the consensus and
emission paths describe the later allocation behavior.

The input role is the key boundary. Validator weights enter the consensus process; they are not the
final incentive, dividend, bond value, rank, or trust metric by themselves.

## Vector and Matrix Structure

Validator weights are often read through weight-vector vocabulary. A weight vector is one
validator's structured set of miner evaluations, while validator weights name the evaluation values
inside that consensus input
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

Weight matrix sits one level wider. It is the subnet-level collection of validator weight vectors
used as structured input for Yuma Consensus
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Together, the three terms describe nested parts of the input side: validator weights are the values,
a weight vector groups one validator's values, and the weight matrix groups the subnet's validator
vectors.

## Consensus Score and Clipping

Validator weights and consensus score are related but distinct. Validator weights are evaluation
inputs, while consensus score is the stake-weighted benchmark used to filter outlier weights before
final rank results
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

This separates the submitted signal from the filtering benchmark. The original evaluation signal
enters consensus, while later terms describe how the network filters, aggregates, and interprets the
set of validator signals.

Rank and trust belong after that filtering path. Rank is a miner-side result after validator weights
have been filtered and aggregated, and trust describes how miner support changes through clipping
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

## Stake Influence Boundary

Validator weights are interpreted inside a stake-weighted consensus process. Stake weight explains
how much influence a validator's signal carries, while validator weights express what that validator
evaluated about miner work
([Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

This keeps influence and evaluation separate. Stake weight affects how strongly a signal counts;
validator weights supply the miner-evaluation signal itself.

Validator trust is also distinct. It describes validator-side influence after clipping, while
validator weights are the evaluation signals that enter the process
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

## Quality and Timing Context

Validator weights are most useful when they reflect independent evaluation. Weight copying weakens
that signal by relying on visible weight information instead of fresh miner-work assessment
([The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

Commit Reveal changes the timing around weight visibility. That timing context can reduce the value
of copying newly visible signals without changing what validator weights are.

Validator-miner bonds are another neighboring concept. Bonds carry smoothed validator-miner
relationship history, while validator weights are evaluation signals that feed the consensus path
([Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

The distinction is temporal as well as conceptual. Validator weights describe current evaluation
signals; bonds describe smoothed relationship history shaped by earlier signals and later smoothing
logic.

## Environment Scope

Bittensor separates mainnet, testnet, and localnet environments. Validator-weight examples from one
environment should not be treated as evidence for another because subnet tasks, validators, and
incentive mechanisms can differ
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Introduction to Bittensor: Subnet development](https://docs.learnbittensor.org/learn/introduction#subnet-development)).

Localnet examples are isolated development examples. Testnet examples are shared non-production
examples. Mainnet validator-weight interpretation belongs to the mainnet subnet and consensus
context being discussed.

## Reader Boundary

Validator weights are concept vocabulary for subnet evaluation signals. They are not a miner
leaderboard, a reward guarantee, a validator permit, or a complete explanation of Yuma Consensus
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The stable article-level point is the path: validators evaluate miner work, express that evaluation
as weights, and send those weights into the consensus process that handles filtering and reward
allocation.
