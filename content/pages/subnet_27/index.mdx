Team TBC is Bittensor Subnet 27.

## What Team TBC Provides

Public technical documentation for Team TBC is not available through the on-chain identity alone.
The description, GitHub repository, project URL, and Discord fields are all blank on Finney at the
time of writing.

## Miner and Validator Roles

Subnet 27 operates under the standard Bittensor two-role structure. Miners supply a capability to
the network, and validators evaluate those contributions and set weights that feed into
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus), whose consensus weights then
help determine per-tempo [emissions](https://docs.learnbittensor.org/learn/emissions).

## On-Chain Identity

The on-chain identity for netuid 27 lists the subnet name as Team TBC. All other identity fields —
description, GitHub repository, project URL, and Discord — are blank. Live identity data is
available on [TaoStats](https://taostats.io/subnets/27).

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Team TBC
(SN27), that sequence applies to the standard Bittensor lifecycle: localnet for isolated
development, testnet for shared non-production testing, and mainnet for live operation with real
emissions.

On mainnet, Team TBC (SN27) is registered as the live production subnet at netuid 27. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. Participation examples or emission outcomes from one
environment should not be read as representing production subnet performance in another environment.

## Netuid 27 Identifies the Subnet On-Chain

Bittensor assigns every subnet a unique numeric identifier called a netuid, and Subnet 27 is the
subnet registered at netuid 27
([Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid)). The
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) reference
explains that each subnet runs its own incentive mechanism while sharing the same underlying
Subtensor chain, so the netuid is the stable handle that distinguishes Subnet 27 from every other
subnet.

For a reader, this means "Subnet 27" and "netuid 27" refer to the same on-chain slot. A claim about
Subnet 27 should be tied to that netuid rather than to the registered name alone, because the name
field can be changed on-chain while the netuid stays fixed.

## Distinction from Yuma Consensus

Team TBC on netuid 27 is registered with a name only and blank project identity fields. Yuma
Consensus is the separate mechanism that converts validator weight submissions into emission shares
each tempo ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Subnet 27 (Team TBC)** — sparse on-chain identity at netuid 27.
- **Yuma Consensus** — mechanism that turns validator weights into emission shares.

## Reader Boundary

Subnet 27 should not be read as generic Bittensor subnet documentation, confirmation that the subnet
has a published, working mechanism, or a substitute for the subnet's own primary sources. It names
the on-chain subnet slot registered at netuid 27; see this article's On-Chain Identity section for
its current registered fields
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid)).
