# Batch Transactions

Batch transactions group multiple Bittensor chain actions into one submitted transaction. They are
submission grouping vocabulary, not a new kind of action.

The official [Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions) page
describes Bittensor's batching support as a way to include multiple actions inside one submitted
transaction. The key concept is the boundary between the outer grouped submission and the inner
actions it carries.

## Grouped Action Boundary

The [Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions) source makes
batching a container around multiple actions. The group changes how actions are bundled together,
but each inner action still has its own meaning and result context.

This keeps batching separate from passive observation. A chain read inspects public state, while a
batch transaction belongs to the submitted-action side of the network.

## Failure Boundary

The [Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions) documentation
distinguishes batching forms by failure behavior. At a concept level, the difference is whether a
group can stop after a failed inner action, require every inner action to succeed together, or
continue checking later actions after one fails.

That failure boundary is the main reason batch transactions need separate vocabulary. A reader
should not assume that every grouped submission has the same all-or-nothing behavior.

## Fee Boundary

Batching changes grouping, but it does not erase the fee character of the inner actions. The
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) page explains that batch
fees aggregate the fee behavior of inner actions and include batch-wrapper overhead.

This means a batch should not be treated as a free shortcut. If a grouped submission contains
fee-bearing activity, the group can still carry a fee. If the inner actions are fee-free, the fee
context is different.

## Network Context

Batch examples belong to the network where they are submitted. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, so batch examples should keep their network context
visible when the claim depends on observed behavior.

Local examples can explain grouping mechanics. Testnet examples can show shared testing context.
Mainnet examples belong to primary network context and require the appropriate source context.

## Evidence Boundary

The [Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions) page supports
claims about grouped submissions and their failure forms. It does not provide a blanket success
guarantee, fee quote, or universal result for every grouped action.

When a claim depends on a concrete batch, the article should identify the batching form, network
context, and inner actions. The outer grouped transaction and the carried actions should both be
considered when interpreting evidence.
