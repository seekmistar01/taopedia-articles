# Subnet Hyperparameters

Subnet hyperparameters are protocol state variables that shape how a Bittensor subnet behaves. They
cover timing, validator-weight flow, consensus behavior, emission handling, and lifecycle limits for
a selected subnet
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

They are not the same as the subnet task. An incentive mechanism explains what work miners produce
and how validators evaluate that work; hyperparameters set the protocol envelope around that
incentive system
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

That envelope matters for interpretation. A hyperparameter can change when signals are accepted, how
often epochs are evaluated, or how a subnet's protocol limits are read without changing the
underlying work target.

## Parameter Layer

Hyperparameters belong to the settings layer of a subnet. They can affect cadence, weight submission
timing, consensus processing, and emission handling, but they do not by themselves define the useful
digital work of the subnet
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

That distinction helps readers separate protocol envelope from mechanism design. A hyperparameter
can change when or how signals are processed without changing the subnet's task, scoring model, or
domain-specific evaluation goal.

The safe reading is therefore contextual. Hyperparameters explain conditions around a subnet's
operation; the mechanism explains what the subnet is trying to measure.

## Timing and Weight Flow

Tempo is the clearest timing example because it gives a block-based cadence for subnet epochs. The
hyperparameter reference places tempo alongside other subnet settings that shape validator signal
flow and epoch behavior
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Weight-related settings answer a different timing question: how validator signals are paced or
revealed. Commit Reveal, for example, is a weight-submission mechanism whose timing is interpreted
through subnet settings and tempo-based delays
([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

Timing vocabulary should stay attached to the selected setting. A tempo claim concerns epoch
cadence, while a commit-reveal claim concerns the handling of validator weight submission and reveal
timing.

## Consensus and Emissions

Hyperparameters also frame how consensus and emissions are interpreted. Yuma Consensus explains how
validator weight signals are aggregated into miner incentives and validator dividends, while the
emissions documentation explains how rewards move through subnet contexts
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Hyperparameters provide the subnet-specific timing and boundary layer around those processes. They
help explain when consensus rounds occur, how weight signals are constrained, and how emission
handling is read for a selected subnet.

This makes hyperparameters supporting context rather than the reward mechanism itself. The reward
path still depends on validator signals, Yuma Consensus, and the emission process; the
hyperparameters shape the subnet-specific conditions around that path.

## Lifecycle and Limits

The hyperparameter reference also includes settings that shape lifecycle or limit behavior around a
subnet. Those values belong to the same protocol-envelope category as timing and weight-flow
settings because they describe constraints around a selected subnet
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

That does not make every lifecycle statement a hyperparameter statement. A reader should keep the
specific setting, subnet, and network context attached when a claim depends on a value or limit.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Subnet
Hyperparameters, that sequence changes how an example should be read, because the surrounding
network state differs at each stage.

In localnet, subnet Hyperparameters can be exercised in an isolated development environment, where
the surrounding chain state reflects local configuration rather than production history.

On testnet, subnet Hyperparameters can be observed in a shared, non-production network whose state
is kept separate from mainnet.

On mainnet, subnet Hyperparameters applies on the live, production Bittensor network, where the
surrounding state is real and persistent.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A subnet Hyperparameters example from one environment
should not be read as representing another environment.

## Reader Boundary

A hyperparameter claim should name the behavior being described and stay separate from the subnet
task, incentive mechanism, or live value source. Tempo claims concern epoch cadence; Commit Reveal
claims concern weight-submission timing; neither replaces the work target
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Subnet hyperparameters are best used as context around timing, weights, consensus, emissions, and
limits. They are not an operator manual, deployment status label, or substitute for the subnet's
incentive design
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## Activity Cutoff Sets the Early Weight Window

Subnet [hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) include
activity cutoff measured in blocks within each
[epoch](https://docs.learnbittensor.org/resources/glossary#epoch). Official documentation states
that a validator must submit
[validator weights](https://docs.learnbittensor.org/resources/glossary#validator-weights) within the
first activity-cutoff blocks of an epoch or wait until the next epoch to participate.

That setting shapes validator signal timing. It is one example of how hyperparameters constrain
consensus input without defining what miners produce.

## Min and Max Burn Bound Neuron Registration Cost

The hyperparameter reference lists `MinBurn` and `MaxBurn` as the lower and upper bounds on the
dynamic TAO burn required to register a neuron. The live registration price moves inside that band
rather than staying fixed at either bound
([Register](https://docs.learnbittensor.org/resources/glossary#register)).

Burn hyperparameters therefore belong to the registration envelope. They limit how low or high entry
cost can fall without replacing the registration process itself.

## Tempo Counts Blocks in Each Epoch

The `tempo` hyperparameter sets the block-based cadence for subnet epochs. The
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo) defines tempo as the
interval over which [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)
calculates [emissions](https://docs.learnbittensor.org/resources/glossary#emission) from the latest
ranking [weight matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix).

Tempo is a timing hyperparameter, not a scoring rule. It tells readers how many blocks group epoch
processing; the incentive mechanism still defines the subnet task inside that cadence.

## Environment Scope

Localnet, testnet, and mainnet observations should be read separately. A hyperparameter can have the
same conceptual role across environments while observed values or outcomes belong to the network
where they were measured
([Introduction to Bittensor: Subnet development](https://docs.learnbittensor.org/learn/introduction#subnet-development),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).
