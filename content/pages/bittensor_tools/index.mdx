# Bittensor Tools

Bittensor tools are software interfaces that help people work with the Bittensor network. The tools
overview includes the SDK, terminal interface, and wallet/key material for readers who need a
practical way to reach network features
([Bittensor Tools: SDK](https://docs.learnbittensor.org/concepts/tools#bittensor-sdk),
[Bittensor Tools: CLI](https://docs.learnbittensor.org/concepts/tools#bittensor-cli),
[Bittensor Tools: Wallets and Keys](https://docs.learnbittensor.org/concepts/tools#wallets-and-keys),
[Wallets and Keys](https://docs.learnbittensor.org/keys/wallets)).

The tool name usually answers an access question: which software surface is being used, and which
kind of reader is likely to use it. Subnet, consensus, and governance references still explain the
rules behind the feature being reached
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Governance](https://docs.learnbittensor.org/governance)).

## Interface Layer

Tooling sits around Bittensor roles and features. A tool can help a developer, miner, validator, or
subnet creator reach information or submit network-facing activity
([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools),
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction)).

That makes the tool layer practical rather than purely conceptual. It tells readers where
interaction happens: through a developer library, terminal surface, wallet/key material, or another
user-facing interface ([Wallets and Keys](https://docs.learnbittensor.org/keys/wallets)).

## Tool Families

Bittensor tooling includes developer libraries, terminal interfaces, and user-facing account
material ([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools)).

Those families differ by reader and task. Libraries help software integrate with Bittensor, terminal
interfaces help users interact directly, and wallet/key material helps explain the account and
signing context around network use
([Bittensor Tools: Wallets and Keys](https://docs.learnbittensor.org/concepts/tools#wallets-and-keys),
[Wallets and Keys](https://docs.learnbittensor.org/keys/wallets)).

## SDK Context

The SDK belongs to the developer-library side of the tool family. It gives software a programmatic
way to work with Bittensor features
([Bittensor Tools: SDK](https://docs.learnbittensor.org/concepts/tools#bittensor-sdk),
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction)).

That context is useful when the article is about developer integration or application behavior. It
is less useful by itself when the article is explaining a subnet task, incentive mechanism, or
governance process.

## Terminal and User Interfaces

Terminal and user-facing tools help people interact with Bittensor without writing an integration.
The tools overview treats the terminal interface separately from the SDK because the audience and
use shape are different
([Bittensor Tools: CLI](https://docs.learnbittensor.org/concepts/tools#bittensor-cli)).

Wallet/key material is nearby because accounts and signing are part of using the network. That
material is most useful for explaining account context, while terminal references are most useful
for explaining the user-facing surface
([Bittensor Tools: Wallets and Keys](https://docs.learnbittensor.org/concepts/tools#wallets-and-keys),
[Wallets and Keys](https://docs.learnbittensor.org/keys/wallets)).

## Role and Subnet Context

Miners, validators, subnet creators, and developers can all use tools, but the tool does not define
the role. The Bittensor introduction frames Bittensor around subnets, miners, validators, and
digital commodities; tools are the software surfaces around that structure
([Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

Subnets still provide the work-and-evaluation context. The subnet model explains what is being
coordinated, while the tools overview explains the interface layer around that model.

## Network Boundary

Bittensor documentation separates mainnet, testnet, and localnet contexts
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

A tooling example from one environment should not be treated as proof about another environment.
Localnet examples can show controlled behavior, testnet examples can show shared non-production
conditions, and mainnet references need separate support from mainnet-specific material.

## Reader Boundary

Bittensor tools should be read as practical access vocabulary. The term helps identify whether the
article is discussing a developer library, terminal interface, wallet/key material, or another
surface around Bittensor ([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools),
[Wallets and Keys](https://docs.learnbittensor.org/keys/wallets),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

When the article moves from access into feature behavior, subnet design, role meaning, or network
scope, the nearby mechanism and network references become the stronger sources.
