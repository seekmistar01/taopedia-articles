# Mempool Visibility

Mempool visibility is the exposure of transaction details while a transaction is still pending
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield)).

The term is useful because pending information can matter before a transaction becomes part of
ordered block history. It belongs to transaction-safety context, especially MEV risk, rather than to
final chain-state evidence
([MEV](https://docs.learnbittensor.org/resources/glossary#mev-maximal-extractable-value),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

## Pending Transaction Context

The mempool glossary entry places pending and unconfirmed transactions before block inclusion
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)).

Mempool visibility asks what information is observable during that waiting period. The transaction
has not yet become part of a block, so the relevant context is exposure while it is still waiting
for inclusion.

That timing separates mempool visibility from ordinary block evidence. The mempool term describes a
pending pool of transactions, while a block is the ordered chain data unit that contains accepted
transactions and a block hash
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

## MEV Risk Context

The MEV glossary entry connects maximal extractable value to transaction ordering and information
available before inclusion
([MEV](https://docs.learnbittensor.org/resources/glossary#mev-maximal-extractable-value)).

That makes mempool visibility a safety concept. If sensitive pending details are visible before a
block is produced, another observer may have information that matters for ordering-sensitive
behavior.

Mempool visibility is therefore one part of MEV risk, not the whole risk by itself. MEV context also
depends on ordering, inclusion, and the surrounding transaction environment
([MEV](https://docs.learnbittensor.org/resources/glossary#mev-maximal-extractable-value),
[MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield)).

## Pending Privacy Boundary

Mempool protection should not be read as hiding the final transaction record. MEV Shield addresses
what can be observed before inclusion, while a block still provides the ordered chain context after
the transaction is accepted.

That distinction keeps privacy timing separate from chain history. A shielded pending transaction
can reduce pre-inclusion exposure without changing the fact that included transactions belong to
recorded block context.

## Block Inclusion Boundary

A block is the chain data unit that contains transactions and has a block hash
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

This boundary matters because pending visibility and final chain history answer different questions.
Mempool visibility concerns the pre-inclusion window; block evidence concerns inclusion and ordering
after the chain records the transaction.

## Recorded Evidence Context

Pending visibility is not the same thing as recorded chain state. Chain events and storage are
better sources for what happened after inclusion, while mempool visibility concerns what could be
observed beforehand ([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Events and storage are still important, but they answer later questions. Events name runtime records
emitted during processing, and storage exposes readable state surfaces after the chain has recorded
state ([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

In short, mempool visibility describes pending exposure; blocks, events, and storage describe
recorded chain context.

## Development Stage Context

Bittensor documentation separates mainnet, testnet, and localnet contexts
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Introduction to Bittensor: Subnet development](https://docs.learnbittensor.org/learn/introduction#subnet-development)).

A mempool visibility example from one environment should not be treated as proof about another
environment. Localnet examples can show controlled behavior, testnet examples can show shared
non-production conditions, and mainnet references need separate support from mainnet-specific
material.

## Staking and Unstaking Rely on the Same Pending Window

The
[Glossary: MEV](https://docs.learnbittensor.org/resources/glossary#mev-maximal-extractable-value)
states that MEV attacks can affect
[staking](https://docs.learnbittensor.org/resources/glossary#staking) and
[unstaking](https://docs.learnbittensor.org/resources/glossary#unstaking) when knowledge of pending
transactions lets another party manipulate token prices. Those conversions pass through subnet pool
reserves ([Understanding Slippage](https://docs.learnbittensor.org/learn/slippage)), so a visible
pending stake can reveal an upcoming pool move before inclusion.

Mempool visibility therefore includes delegation conversions, not only discretionary swaps. The
exposure question is what pending details are readable while the transaction waits in the
[mempool](https://docs.learnbittensor.org/resources/glossary#mempool).

## Ordering Attacks Need Readable Pending Details

The
[MEV glossary entry](https://docs.learnbittensor.org/resources/glossary#mev-maximal-extractable-value)
describes front-running, sandwich attacks, and back-running as ordering shapes that depend on
observing pending transactions before block inclusion.
[Understanding Slippage](https://docs.learnbittensor.org/learn/slippage) discusses how visible pool
movement around pending trades can affect execution prices during staking conversions.

Those attacks require foreknowledge of pending trade details. Mempool visibility names that
observation window rather than the final execution outcome once a
[block](https://docs.learnbittensor.org/resources/glossary#block) includes the transaction
([Understanding Slippage](https://docs.learnbittensor.org/learn/slippage)).

## Coldkey-Signed Actions Can Route Through MEV Shield

[MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield) documentation limits the feature
to transactions signed by a [coldkey](https://docs.learnbittensor.org/resources/glossary#coldkey).
[Hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey)-signed submissions are not
supported for shielded pending privacy.

That boundary sits directly on mempool visibility. Shielding encrypts sensitive pending contents for
eligible coldkey-signed actions, reducing what other observers can read before inclusion without
changing runtime acceptance rules after reveal.

## Price Protection Addresses a Different Risk

[MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield) documentation frames encrypted
mempool protection as hiding sensitive pending transaction details before inclusion. Price
protection documentation, by contrast, covers bounds on acceptable execution rates during
staking-related pool movement rather than pending visibility itself
([Price Protection](https://docs.learnbittensor.org/learn/price-protection)).

For readers, mempool visibility and MEV Shield belong to the pre-inclusion observation window. Price
protection belongs to execution-rate limits during pool conversions. The two protections answer
different questions and should not be read as interchangeable
([MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield)).

## Distinction from Mempool

Mempool visibility and the mempool answer different questions about pending transactions. The
mempool names the holding area where unconfirmed transactions wait before block inclusion, while
mempool visibility names whether pending transaction details can be observed by other participants
before that inclusion happens
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield)).

- **Mempool** — pending transactions awaiting inclusion.
- **Mempool visibility** — whether pending details can be observed before ordering.

A transaction can be pending in the mempool without exposing sensitive details to observers when MEV
Shield or similar protections reduce pre-inclusion visibility. Mempool vocabulary describes where
the transaction waits; mempool visibility vocabulary describes what others can read during that wait
([MEV](https://docs.learnbittensor.org/resources/glossary#mev-maximal-extractable-value),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

## Distinction from MEV Shield

Mempool visibility and MEV Shield name a risk and a defense against it. Mempool visibility is the
exposure of pending transaction details before inclusion, while MEV Shield is the encrypted-mempool
protection that reduces what observers can read during that pending window
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield)).

- **Mempool visibility** — the exposure of pending details before inclusion.
- **MEV Shield** — the protection that reduces that pre-inclusion exposure.

Shielding only reduces what can be seen while the transaction is pending; block inclusion and
runtime processing still determine the later record
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

## Distinction from Price Protection

Mempool visibility and price protection address different risks. Mempool visibility concerns what
can be observed before a transaction is included, while price protection sets bounds on the
acceptable execution rate during a staking-related pool conversion
([MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield),
[Price Protection](https://docs.learnbittensor.org/learn/price-protection)).

- **Mempool visibility** — pre-inclusion exposure of pending details.
- **Price protection** — execution-rate bounds during a pool conversion.

## Distinction from Slippage

Mempool visibility concerns what pending transaction details can be observed before inclusion.
Slippage is the gap between expected and received amounts when a pool conversion executes
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Understanding Slippage](https://docs.learnbittensor.org/learn/slippage)).

- **Mempool visibility** — pre-inclusion exposure of pending transaction details.
- **Slippage** — conversion gap during pool movement after execution.

Visible pending stakes can reveal upcoming pool moves; slippage names the execution gap once the
conversion runs ([MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield)).

## Distinction from Yuma Consensus

Mempool visibility describes whether pending extrinsics are observable before inclusion in a block.
It names transaction-queue observability vocabulary, not the on-chain step that turns validator
weight submissions into emission shares
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)).

Yuma Consensus is the on-chain process that converts submitted validator weights into emission
shares after evaluation completes. Pending extrinsics may include weight-setting calls, but Yuma
Consensus names settlement after inclusion, not mempool observation
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers should keep mempool vocabulary separate from the settlement step. Mempool visibility answers
what can be seen before block inclusion; Yuma Consensus answers how validator weights become
emission shares in a subnet
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Mempool visibility** — observability of pending extrinsics before inclusion.
- **Yuma Consensus** — on-chain weight aggregation that settles emissions.

## Reader Boundary

Mempool visibility should be read as pending-exposure vocabulary. It identifies what can be visible
while a transaction is waiting, which is why it sits near MEV and MEV Shield in Bittensor safety
context ([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield)).

Block, event, storage, and network references supply the surrounding record context once the topic
moves from pending exposure to included transaction history.
