# Subtensor Error Codes

Subtensor error codes are diagnostic signals from the blockchain layer beneath Bittensor. They help
identify why a network action failed.

The [Subtensor Error Codes](https://docs.learnbittensor.org/errors) page frames the topic as errors
that can appear through Bittensor-facing interfaces, while the
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor) page describes
standard error categories from the Subtensor layer.

## What Error Codes Describe

An error code usually points to the rule family that rejected an action. The
[Subtensor Error Codes](https://docs.learnbittensor.org/errors) overview groups these failures into
standard Subtensor errors, custom errors, and errors inherited from the underlying Substrate
framework. That grouping matters because a short error message identifies the kind of failure, not
the full context of the attempted action.

## Standard and Custom Families

The [Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor) page describes
named failure categories from the Subtensor layer. These categories cover protocol checks such as
subnet existence, balances, staking conditions, weight-setting constraints, and other runtime
boundaries.

The [Subtensor Custom Errors](https://docs.learnbittensor.org/errors/custom) page lists numeric
custom errors and short descriptions. A numeric code is a lookup key for a specific condition, but
the returned code still needs the attempted action, selected network, and surrounding chain context
to be interpreted safely.

## Error family versus cause

An error family is not always the whole cause. The standard-errors page gives named Subtensor
failure categories, and the custom-errors page gives numeric custom conditions, but both references
are intentionally compact
([Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor),
[Subtensor Custom Errors](https://docs.learnbittensor.org/errors/custom)).

For example, a balance-related failure identifies a balance condition, while a subnet-related
failure identifies a subnet condition. The reason that condition appeared still depends on the
submitted action, selected network, account state, and runtime state at the moment the chain checked
the action. The error code is the diagnostic entry point, not a full reconstruction of everything
that led to the rejection.

This makes error-code articles different from troubleshooting transcripts. A transcript can show the
specific tool output and user action that produced one failure. A reference article names the
failure vocabulary and explains how to read it across many possible actions.

## Rate-limit interpretation

Rate-limit failures are especially easy to misread because they involve timing. The
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits) documentation explains that
Bittensor uses block-based cooldowns for selected actions. In that case, the relevant context is the
action family and the number of blocks since the last accepted action of that kind.

That timing model is narrower than a generic retry rule. Some returned errors can point to rate
limits, but other errors point to missing state, invalid parameters, balance conditions, ownership
conditions, or runtime checks. Treating every error as a wait-and-try-again problem loses the
distinction that the error pages are meant to preserve.

The safer interpretation is to read rate-limit evidence with the action being attempted and the
network where it was submitted. The error code gives the diagnostic label, the rate-limit reference
explains the cooldown model, and the selected network determines the chain history being checked.

## Interpreting a Returned Error

A returned error should be read with the action that produced it. The code or named condition can
identify the failure family, but the attempted
[extrinsic](https://docs.learnbittensor.org/subtensor-api/extrinsics), account role, subnet context,
resource state, and network environment explain why that family appeared in a particular case. The
same error vocabulary can point to different explanations depending on what was being submitted.

This keeps the error reference from becoming an operating manual. The official pages provide error
vocabulary and short descriptions. Exact recovery steps, interface output, and private account
diagnosis depend on the tool, network, and state involved in the failed action.

## Error Signals and Chain Context

Error codes sit beside other chain-observation surfaces, but they answer a different question. A
[storage entry](https://docs.learnbittensor.org/subtensor-api/storage) can show recorded state. An
[event](https://docs.learnbittensor.org/subtensor-api/events) can show a named runtime record
emitted during processed activity. An error code explains a rejection category when the intended
state change does not complete.

These sources are strongest when read together. If a submitted action fails, the attempted action
shows what was requested, the returned error identifies the rejection family, and storage or event
context can help explain the surrounding chain state. No single signal is a complete explanation of
the underlying Bittensor mechanism.

## Network and Retry Context

The network that returns an error is part of the interpretation. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet as deployment contexts. A failed submission returns
feedback from the network endpoint that processed it, so the same error family can appear in
different environments while referring to different chain histories.

Error codes also should not all be reduced to retry advice. The
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits) documentation describes
block-based cooldowns, but many returned errors reflect other protocol checks. Waiting may be
relevant for a rate-limited action; it is not a general interpretation for every Subtensor error.

## Evidence Boundary

The [Subtensor Error Codes](https://docs.learnbittensor.org/errors) overview and related error pages
provide structured diagnostic evidence. They can identify the rejection family and help connect a
failed action to the relevant official mechanism documentation. They do not diagnose private
accounts, reproduce interface output, or prescribe wallet operations.

That boundary matters because an error can involve account state, subnet state, signing authority,
network choice, or timing conditions that a static reference page cannot verify. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) documentation
shows why selected network context matters when interpreting the returned error.
