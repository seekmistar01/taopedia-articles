# Active UID

An active UID is a subnet participant slot that currently counts as usable for Bittensor subnet
participation. The Bittensor glossary defines an active UID as a subnet UID that allows the
associated participant to take part as a subnet validator or subnet miner
([Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid)).

The term matters because it is narrower than registration alone. Registration can place a
participant in a UID slot, but active status describes whether that slot is currently treated as
participating in the subnet context being observed
([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot)).

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For active UID on
a subnet such as netuid 1, that sequence determines which network's metagraph state is being read.

In localnet, active UID observations reflect a locally running chain's metagraph state. Localnet
activity flags come from local validator weights, subnet rules, and local chain state rather than
production Bittensor participant behavior.

On testnet, active UID observations can be read in a shared, non-production network. Testnet
metagraph state shows how activity flags, `activity_cutoff`, and participant counts behave under
shared chain conditions while staying separate from mainnet participant state.

On mainnet, active UID is the live metagraph flag for participants currently participating in a
production Bittensor subnet. The
[Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph) reference exposes the `active`
field per UID, and
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#activitycutoff)
defines `activity_cutoff` as the block-based window that drives the activity boundary
([Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. An active UID observation in one environment should not be
read as evidence of the same participant activity in another environment.

## Distinction from Pruning Score

An active UID records whether a neuron currently meets activity conditions. Pruning score ranks
incumbents when registration pressure displaces a slot on a full subnet
([Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid),
[Glossary: Pruning Score](https://docs.learnbittensor.org/resources/glossary#pruning-score)).

- **Active UID** — current participation-ready status for a subnet slot.
- **Pruning score** — emission standing used when one slot must open.

Activity vocabulary names current status; pruning-score vocabulary names displacement ranking
([Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

## Distinction from Deregistration

An active UID names whether a slot currently counts as participating. Deregistration removes a miner
or validator when capacity and eligibility rules allow replacement
([Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid),
[Glossary: Deregistration](https://docs.learnbittensor.org/resources/glossary#deregistration)).

- **Active UID** — metagraph evidence of current participation standing.
- **Deregistration** — documented removal when a full subnet admits a new registration.

Status vocabulary and removal vocabulary answer different slot-lifecycle questions
([Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)).

## Distinction from Immunity Period

An active UID and the immunity period both describe a UID's standing on a subnet but answer
different questions. An active UID is the participant slot that currently counts as usable for
validating or mining, while the immunity period is the block window that exempts a newly registered
UID from deregistration regardless of its score, giving it time to reach usable standing
([Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid),
[Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Distinction from UID Slot

An active UID is a subnet participant slot that currently satisfies activity conditions. A UID slot
is the positional identifier for a participant inside a subnet regardless of current activity
([Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid),
[Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot)).

- **Active UID** — current participation-ready status for a subnet slot.
- **UID slot** — positional identifier inside a subnet.

Slot vocabulary names position; active vocabulary names whether that position currently counts as
participating ([Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

## Distinction from Yuma Consensus

Active UID is a metagraph flag showing whether a subnet slot currently satisfies activity conditions
for participation. Yuma Consensus is the separate tempo process that converts participating
validators' weight submissions into emission shares each epoch
([Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Active UID** — runtime participation status for a subnet slot.
- **Yuma Consensus** — on-chain settlement that turns validator weights into emission shares. Active
  UID names slot participation status; Yuma processes validator weights into rewards
  ([Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions)).

## Distinction from Subnet Miner

Active UID is a slot-level participation signal for subnet roles. A subnet miner produces the work
validators evaluate inside a subnet market
([Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid),
[Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner)).

- **Active UID** — whether the slot currently counts as participating.
- **Subnet miner** — participant that produces evaluated subnet work.

Slot-status vocabulary and role vocabulary answer different participation questions
([Miners](https://docs.learnbittensor.org/miners)).

## Distinction from Netuid

Active UID claims are scoped to a selected subnet. A netuid is the numeric identifier used to select
that subnet in chain state and tooling
([Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid),
[Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid)).

- **Active UID** — participation status for a slot in the selected subnet.
- **Netuid** — identifier selecting which subnet is being read.

Status vocabulary needs subnet context; netuid vocabulary names that selector
([Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

## Reader Boundary

This article explains how to read the active UID concept in a static reference context. It does not
count current active UIDs, enumerate subnet participants, diagnose account state, or reproduce
interface output. Current readings belong to the relevant subnet, network, block context, and
tooling at the time of observation
([Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid),
[Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

An active UID is useful as evidence about participant-slot activity, not as a complete explanation
of subnet performance. It does not by itself identify why a participant's activity status changed,
how well a miner is performing, or whether a validator's weights are useful to the subnet. Those
claims need role-specific and mechanism-specific sources.

Participant activity can change while the subnet continues to exist, whereas subnet deregistration
describes removal of an entire subnet context under different rules
([Subnet Deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration)).
