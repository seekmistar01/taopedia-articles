# Rate Limits

Rate limits are Subtensor timing rules that pace repeated Bittensor chain actions. They are
described as block-based cooldown periods, so the durable idea is not a wall-clock timer but a
required amount of chain progression before the same kind of action can succeed again
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

The concept is most useful as a boundary term. A rate limit explains when a repeated action is too
soon, not whether the action is useful or appropriate for a particular workflow. It is timing
evidence, not a complete explanation of a user's broader activity.

## Cooldown Meaning

A rate-limited action starts its cooldown after the relevant successful action. The rate-limit
reference distinguishes that from unsuccessful attempts, which can be retried without being treated
as the successful action that starts the cooldown
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

That distinction keeps rejection messages from being overread. The
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits) reference makes the
successful action the pacing point, so a failed action is not automatically proof that every later
attempt must wait the same interval.

## Scope of the Rule

Rate limits are scoped to specific action families. The Bittensor reference groups them across
network-wide limits and subnet-specific limits, so a reader should preserve both the action being
paced and the scope where the rule applies
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

This prevents one cooldown from being generalized to all chain activity. A subnet-specific validator
action, a staking-related action, and a broader network rule can all use rate-limit language while
answering different questions about timing and scope.

## Staking and Validator Examples

The rate-limit reference gives staking operations as a concrete global example: adding stake,
removing stake, or moving stake within a selected subnet is paced at one successful operation per
block, with a rate-limit error when the rule is exceeded
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Staking and Delegation](https://docs.learnbittensor.org/staking-and-delegation/delegation)).

Validator weight-setting is a different example because it is tied to subnet configuration. The
subnet hyperparameter reference includes `weights_rate_limit`, and the rate-limit documentation
connects the effective period to tempo and block timing
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

Those examples show why the action family matters. Staking changes and validator weight updates are
both chain actions, but their pacing rules should not be collapsed into a single generic cooldown.

## Rejection Evidence

A rate-limit error is structured timing feedback from the chain layer. Subtensor error references
group failures that can appear when a submitted action does not pass runtime checks, while the
rate-limit reference explains the cooldown rule behind too-soon repeats
([Subtensor Error Codes](https://docs.learnbittensor.org/errors),
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor),
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

That makes rate limits useful diagnostic evidence, but not complete diagnosis. A rejected action
should still be read with the submitted action, selected network, relevant subnet context, and the
official mechanism documentation for the operation involved.

## Network Context

Network context changes how rate-limit examples should be read. Bittensor documentation separates
mainnet, testnet, and localnet, so a timing example from one environment should not be treated as
proof of identical behavior in another environment
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

The safe article-level claim is that rate limits are block-based Subtensor pacing rules. Current
configuration, rejection behavior, and examples belong to the specific environment and subnet
context that produced them.

## Reader Boundary

This article explains how to interpret rate-limit vocabulary. It does not list every current
cooldown value, prescribe retry timing, diagnose private state, or reproduce interface output. Those
details depend on the action, network, subnet context, and current chain state at the time of
observation ([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).
