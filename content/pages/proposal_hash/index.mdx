# Proposal Hash

A proposal hash is the identifier used to refer to a proposal in Bittensor governance. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#proposal-hash) defines it as a unique
identifier for a proposal used in the voting process.

## Governance Context

The [Senate](https://docs.learnbittensor.org/governance/senate) documentation places proposals in
the Senate review process, while the
[Governance Overview](https://docs.learnbittensor.org/governance) explains that proposals are
created by the Triumvirate and reviewed by the Senate. A proposal hash is the reference for the
proposal, not the proposal body itself.

## Relationship to Proposal

A proposal hash and a proposal refer to related but different parts of Bittensor governance. The
[Glossary: Proposal](https://docs.learnbittensor.org/resources/glossary#proposal) describes a
proposal as a suggestion or plan put forward by the Triumvirate for the Senate to vote on. The
[Glossary: Proposal Hash](https://docs.learnbittensor.org/resources/glossary#proposal-hash)
describes the proposal hash as the unique identifier for a proposal used in the voting process.

For readers, the proposal is the governance item being reviewed, while the proposal hash is the
identifier used to refer to it during Senate voting. A proposal hash should not be read as the
proposal body itself, only as the reference that points to it.

References: [Glossary: Proposal](https://docs.learnbittensor.org/resources/glossary#proposal),
[Glossary: Proposal Hash](https://docs.learnbittensor.org/resources/glossary#proposal-hash)

## How a Proposal Hash Is Used in Senate Voting

When a Triumvirate proposal reaches Senate review, senators need a stable identifier to vote on the
correct item. The proposal hash serves that role. The
[Senate](https://docs.learnbittensor.org/governance/senate) documentation explains that senators
cast approval or disapproval votes with `btcli sudo senate-vote`, referencing the proposal by its
hash rather than by a free-form title or description.

That design keeps Senate voting tied to a specific on-chain proposal record. A senator is not voting
on a summary of the proposal; the hash is the handle the voting command uses to target the exact
proposal under review. If several proposals are active, the hash is what distinguishes one Senate
item from another.

This also means readers should not treat a proposal hash as optional metadata. For Senate
participants, it is the operational identifier that connects a vote action to one proposal record in
the review queue.

Reference: [Senate](https://docs.learnbittensor.org/governance/senate)

## Public Visibility Without Voting Authority

Proposal review in Bittensor is public at the viewing layer even though voting is restricted. The
[Senate](https://docs.learnbittensor.org/governance/senate) documentation states that anyone can
inspect proposals currently before the Senate, including with `btcli sudo proposals`, without
special permissions.

The proposal hash still matters in that public context because it identifies which item in the list
a reader is looking at. A visible proposal hash does not mean the proposal is approved, rejected, or
closed. It only means that identifier points at one governance item currently in the Senate review
path.

For readers, the useful boundary is three-way: the proposal hash identifies the item, public
visibility lets anyone inspect what is under review, and Senate voting authority remains limited to
senators acting on that hash through the documented vote command.

References: [Senate](https://docs.learnbittensor.org/governance/senate),
[Governance Overview](https://docs.learnbittensor.org/governance)

## Where the Hash Fits in the Governance Flow

A proposal hash is most useful when placed inside the full governance sequence rather than read on
its own. The [Governance Overview](https://docs.learnbittensor.org/governance) describes Triumvirate
creation, Senate majority approval, and Triumvirate close before privileged call data executes. The
hash is the reference used while that sequence is underway.

Before Senate approval, the hash identifies the proposal waiting in review. During Senate voting,
the hash is what senators target when recording approval or disapproval. After Senate approval, the
hash still refers to the same proposal record until a Triumvirate member closes it. The hash does
not execute anything by itself; it tracks which proposal record is moving through those stages.

Readers comparing governance terms should keep the roles separate. The proposal is the governance
item with privileged call data. The proposal hash is the identifier for that item during review and
voting. The Senate is the approval body. The Triumvirate both originates proposals and performs the
close step after approval. Subtensor is the chain layer where closed proposal call data may finally
execute.

Reference: [Governance Overview](https://docs.learnbittensor.org/governance)

## Reader Boundary

This article does not list active proposal hashes or describe any proposal outcome. Readers should
use official governance references for proposal records and review context.
