# Mempool

In Bittensor context, a mempool is a temporary holding area for pending blockchain transactions. It
describes a pre-inclusion state, not final chain history.

Pending and unconfirmed transactions sit before block inclusion
([Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)), while a block is the chain
data unit that contains transactions
([Block](https://docs.learnbittensor.org/resources/glossary#block)).

## Pending Transaction Context

The [Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) entry places the term in
the time between transaction submission and block inclusion. A transaction in the mempool has not
yet become part of a block, so the term should be read as pending-state vocabulary rather than
confirmed-history vocabulary.

That boundary keeps mempool claims narrow. A pending transaction may later be included, rejected, or
otherwise absent from the eventual chain record. The mempool term itself does not settle that later
outcome.

## Block Inclusion Boundary

[Block](https://docs.learnbittensor.org/resources/glossary#block) refers to a chain data unit that
contains transactions. The mempool is the waiting area before that inclusion happens. Once a
transaction is included in a block, it is no longer merely pending in the mempool.

This makes blocks and mempools complementary evidence surfaces. The mempool is the pre-inclusion
stage, while a block record gives inclusion and ordering in chain history.

## Extrinsics and Fees

Submitted extrinsics can pass through the mempool before block inclusion. The
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) reference lists
submitted runtime methods, while the
[Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) entry names the
pending-transaction area before those submissions are included.

Fees are related but separate. The
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) page covers fee
accounting for state-changing actions. The mempool does not itself define the fee model; it names
the temporary holding area for unconfirmed submissions.

## Network Scope

A mempool is read inside a specific Bittensor network environment. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, so pending-state claims should keep the selected network
context visible.

A mainnet mempool, a testnet mempool, and a localnet mempool use the same concept but belong to
different environments. A pending transaction observed in one environment should not be treated as
evidence about another environment.

## Temporary Evidence

Mempool evidence is temporary. It can support a claim that a transaction was pending in a selected
context, but it does not prove final inclusion, later state, or every reason for the pending state.
The eventual block record and related events provide different evidence.

The [Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events) reference lists runtime
event names, and [Block](https://docs.learnbittensor.org/resources/glossary#block) supplies the
chain-record context for included transactions. Those sources help separate pre-inclusion
observation from post-inclusion records.

## Visibility and Shielding

Mempool visibility asks what can be seen while a transaction is pending. The mempool term names the
pending area; it does not guarantee that every transaction detail is equally exposed to every
observer.

[MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield/) is an official response to
sensitive-detail exposure before block inclusion. That makes visibility and shielding related to
mempool context, while still remaining separate concepts.

## Reader Boundary

The [Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) entry supports a
high-level definition of the term. It does not provide submission instructions, pending-activity
monitoring procedures, private account diagnosis, or network-state claims.

When a claim depends on a specific pending transaction, it should identify the network and
inspection context. When a claim depends on final chain history, block and event evidence are a
better fit than mempool vocabulary alone.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For the mempool,
that sequence determines which network's pending transaction area is being observed.

In localnet, pending transactions sit in a local chain mempool. Localnet pending-state observations
reflect local chain activity and do not represent production Bittensor transaction state.

On testnet, pending transactions sit in the testnet mempool. Testnet pending-state observations are
separate from mainnet transaction history.

On mainnet, the mempool holds pending Bittensor transactions before they are included in production
blocks. The [Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) glossary entry
describes the pending-state context that applies on the production network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A pending-transaction observation from one environment
should not be read as evidence about transaction state in another environment.
