# Yuma Consensus 3

Yuma Consensus 3, often shortened to YC3, is a refinement of Bittensor's Yuma Consensus bonding
logic. Its main purpose is to make validator-miner bonds more responsive to useful independent
evaluation, especially when a validator recognizes miner quality before that view is widely shared
across the validator set
([How Yuma Consensus 3 Makes Bittensor More Fair](https://docs.learnbittensor.org/learn/yc3-blog)).

The phrase belongs inside consensus vocabulary. Yuma Consensus is the broader process that turns
validator evaluations into miner incentives and validator dividends; YC3 is a bond-scaling
refinement inside that process, not a separate subnet or a general validator operating mode
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Bond Scaling

YC3 centers on per-bond exponential moving average scaling. Instead of applying one adjustment rate
uniformly, YC3 lets individual validator-miner bond relationships move according to their own
alignment with subnet consensus
([How Yuma Consensus 3 Makes Bittensor More Fair](https://docs.learnbittensor.org/learn/yc3-blog),
[Glossary: Exponential Moving Average (EMA)](https://docs.learnbittensor.org/resources/glossary#exponential-moving-average-ema)).

That makes the refinement local to each bond relationship. A validator that finds a useful miner
early can begin building a stronger bond before every other validator has converged on the same
view. The mechanism is meant to distinguish independent evaluation from simple agreement with an
already dominant consensus signal
([The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)).

## Clipping, Bonds, Weight Matrix, and Emission Pipeline

[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) aggregates validator weight
signals within a subnet into miner incentives and validator dividends at
[tempo](https://docs.learnbittensor.org/resources/glossary#tempo) boundaries. YC3 refines how
[validator-miner bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)
respond inside that pipeline rather than replacing the aggregation step itself.

The useful order on a subnet such as netuid 1 is therefore validators submitting weights into the
latest ranking [weight matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
consensus clipping trimming outlying scores toward stake-weighted agreement, bond-weight blending
submitted weights with consensus values using penalty factor β, and emission distribution crediting
miner and validator shares once the tempo interval closes
([Yuma Consensus: Miner emissions](https://docs.learnbittensor.org/learn/yuma-consensus#miner-emissions),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

YC3 adds per-bond EMA scaling and improved fixed-point bond storage on a 0–65535 scale so smaller
validators can accumulate bond mass that earlier 16-bit rounding could erase
([Yuma Consensus 3 (YC3) Migration Guide](https://docs.learnbittensor.org/learn/yuma3-migration-guide)).
The migration guide also corrects bond upscaling when
[consensus](https://docs.learnbittensor.org/resources/glossary#consensus-score) equals zero,
replacing behavior that produced unfair bond distribution with enhanced EMA smoothing in that edge
case
([How Yuma Consensus 3 Makes Bittensor More Fair](https://docs.learnbittensor.org/learn/yc3-blog)).

The β penalty that pulls overstated validator weights back toward the network median remains in
place under YC3. Faster independent bond building and anti-fraud clipping therefore coexist in the
same upgrade rather than trading against each other
([How Yuma Consensus 3 Makes Bittensor More Fair](https://docs.learnbittensor.org/learn/yc3-blog)).

That boundary keeps YC3 distinct from visibility tools such as
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal). Commit Reveal changes when
weight information becomes visible; β and clipping change how far an already submitted weight can
influence bond formation after consensus is known.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. Yuma Consensus 3
examples on netuid 1 need that context because bond behavior, clipping outcomes, and emission
results belong to the network where they were observed
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

In localnet, YC3 bond mechanics can be tested in an isolated environment. Local bond accumulation
and consensus clipping for netuid 1 reflect local configuration rather than production validator
agreement.

On testnet, YC3 flows can be exercised in a shared non-production network. Testnet bond movement and
weight-matrix outcomes for netuid 1 are separate from mainnet subnet state
([Yuma Consensus 3 (YC3) Migration Guide](https://docs.learnbittensor.org/learn/yuma3-migration-guide)).

On mainnet, YC3 concerns live production consensus behavior on the active Bittensor network,
including bond scaling and emission distribution on subnets such as netuid 1.

A YC3 example from one environment should not be treated as proof about another environment. They
use the same consensus vocabulary, but not the same validator set or chain state.

## Distinction from Yuma Consensus

Yuma Consensus 3 and Yuma Consensus relate as a refinement and the broader process. Yuma Consensus
is the aggregation that turns validator weight signals into miner incentives and validator
dividends, while YC3 is a bond-scaling refinement inside that process rather than a separate
consensus system ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[How Yuma Consensus 3 Makes Bittensor More Fair](https://docs.learnbittensor.org/learn/yc3-blog)).

- **Yuma Consensus** — the process that aggregates weights into emissions.
- **Yuma Consensus 3** — a bond-scaling refinement inside that process.

## Distinction from Commit Reveal

Yuma Consensus 3 and Commit Reveal act at different points around validator weights. Commit Reveal
changes when weight information becomes visible, while YC3 changes how far an already submitted
weight influences bond formation after consensus is known
([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal),
[How Yuma Consensus 3 Makes Bittensor More Fair](https://docs.learnbittensor.org/learn/yc3-blog)).

- **Commit Reveal** — controls when weight information becomes visible.
- **Yuma Consensus 3** — controls how submitted weights shape bonds after consensus.

## Distinction from Bonds Penalty

Yuma Consensus 3 refines bond scaling inside Yuma Consensus. Bonds penalty is the hyperparameter
that sets how strongly validator weights above the consensus benchmark are penalized during bonding
([How Yuma Consensus 3 Makes Bittensor More Fair](https://docs.learnbittensor.org/learn/yc3-blog),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Yuma Consensus 3** — bond-scaling refinement inside consensus aggregation.
- **Bonds penalty** — penalty magnitude on weights above the consensus benchmark.

YC3 improves bond accumulation and storage; bonds penalty configures how far outlier weights are
pulled back toward consensus during bonding
([Yuma Consensus: Penalizing out-of-consensus bonds](https://docs.learnbittensor.org/learn/yuma-consensus#penalizing-out-of-consensus-bonds)).

## Distinction from Mechanism

Some subnets can run multiple incentive mechanisms, each with its own evaluation and bond context.
YC3 language should therefore stay attached to the mechanism and subnet context being discussed,
rather than being read as a single global score for all subnet activity
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

Network environment matters as well. Bittensor separates mainnet, testnet, and localnet, and subnet
development can move through isolated testing before production use. A YC3 example from one
environment is evidence about that environment, not automatic proof of the same bond behavior
elsewhere ([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Introduction to Bittensor: Subnet development](https://docs.learnbittensor.org/learn/introduction#subnet-development)).

## Distinction from Evaluation Quality

YC3 is also part of Bittensor's response to copied or delayed evaluation. Weight copying is the case
where a validator relies on public weight information instead of producing an independent
assessment. Commit Reveal and Drand time-lock encryption address visibility and timing around
validator weights, while YC3 addresses how bond movement reflects useful evaluation after weights
enter consensus
([The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal),
[Glossary: Drand/time-lock encryption](https://docs.learnbittensor.org/resources/glossary#drandtime-lock-encryption)).

These concepts reinforce each other without being interchangeable. Commit Reveal is about delayed
visibility, Drand is part of the time-lock context, weight copying names the evaluation-quality
problem, and YC3 names a bond-scaling refinement that can recognize earlier independent evaluation.

## Distinction from Liquid Alpha

YC3 sits near Liquid Alpha because both ideas concern bond responsiveness. Liquid Alpha changes bond
smoothing according to consensus alignment, while YC3 focuses on per-bond EMA scaling so different
validator-miner relationships can adapt at different rates
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights),
[Glossary: Exponential Moving Average (EMA)](https://docs.learnbittensor.org/resources/glossary#exponential-moving-average-ema)).

The useful boundary is conceptual. EMA names the smoothing technique, Liquid Alpha names an
alignment-sensitive adjustment, and YC3 names the newer refinement around validator-miner bond
movement. Treating those terms separately avoids turning one consensus feature into a catch-all
label for every bond or weight behavior.

## Reader Boundary

YC3 is a consensus-reference term for bond scaling. It does not by itself provide a live activation
claim, a subnet-specific setting, a validator strategy, or a guarantee that a particular miner will
receive incentives. Those questions depend on the selected subnet, mechanism, network environment,
and chain state
([How Yuma Consensus 3 Makes Bittensor More Fair](https://docs.learnbittensor.org/learn/yc3-blog),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).
