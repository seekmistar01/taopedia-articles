# Weight Vector

A weight vector is one validator's structured set of miner evaluations inside a Bittensor subnet.
Each element represents a weight assigned to a subnet miner based on the validator's evaluation of
miner performance
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The key scope is one validator before aggregation. Yuma Consensus can combine many such vectors, but
a single vector is still one validator's submitted signal.

## Per-Validator Signal

A weight vector is not the whole subnet result. It is the signal one validator produces after
evaluating miners under the subnet's standards
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

That boundary matters because consensus uses many validator signals. A strong value in one vector
expresses one validator's judgment, not the final rank or emission allocation by itself.

## Task and Mechanism Context

The meaning of a weight vector depends on the subnet task and incentive mechanism. The mechanism
defines what useful work means; the vector records one validator's evaluation relative to that
definition
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

This keeps vector claims from becoming cross-subnet comparisons. A vector from one subnet's task
does not automatically describe performance under another subnet's task or scoring method.

## Matrix Boundary

A weight vector and a weight matrix are related but different scopes. The vector is one validator's
signal, while the matrix is the subnet-level collection of validator vectors used by consensus
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

For readers, this means a vector is an input to the matrix, not the matrix result. The matrix
organizes many vectors before Yuma Consensus filters and aggregates them.

## Consensus Score Boundary

A weight vector is also distinct from consensus score. The vector is one validator's miner
evaluations, while consensus score is the stake-weighted threshold used to filter outlier
submissions before final rank calculations
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

That makes consensus score downstream of submitted weight signals. A vector supplies input; the
consensus score helps decide how far submitted signals can influence the final result.

## Visibility Context

Independent validator evaluation is what gives a weight vector useful information. Weight copying
weakens that purpose when validators rely on visible public weights instead of fresh evaluation
([The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

Commit Reveal can add a timing layer by concealing fresh weight submissions until a reveal period.
That protects weight information before consensus uses it, but it does not change what a vector is:
one validator's structured evaluation signal.

## Multiple-Mechanism Context

When a subnet uses multiple incentive mechanisms, validators can maintain separate weight vectors
for each mechanism. The mechanism context clarifies which evaluation path a vector belongs to
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)).

A vector for one mechanism should not be read as a vector for another mechanism. The term still
means one validator's miner evaluations, but the relevant mechanism is part of the meaning.
