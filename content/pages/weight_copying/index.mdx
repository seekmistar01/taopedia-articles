# Weight Copying

Weight copying is a consensus problem where a validator relies on another validator's weight
information instead of independently evaluating miners. The
[Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)
describes this as free-riding on validator work, and
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) shows why validator weights
matter for miner and validator emissions.

## Independent Evaluation

Independent validator evaluation gives a subnet multiple views of miner performance. When a
validator copies another validator's weights, the signal may look like another evaluation while
adding little independent information. That weakens the evaluation layer that Yuma Consensus relies
on ([Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)).

The issue is not that validators agree. Validators can reach similar conclusions when miners perform
similarly. The problem is dependence on copied weight information rather than independent scoring.

## Consensus Impact

Weight copying matters because validator weights are consensus inputs. The
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) documentation describes how
validator rankings are aggregated into miner incentives and validator dividends. Copied weights can
make the set of inputs less independent than it appears.

This is why weight copying belongs in consensus vocabulary rather than only validator behavior
vocabulary. It affects the quality of the signals that downstream aggregation receives.

## Delayed Visibility

Delayed visibility is one defense against copied weights. The
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal) mechanism separates the
commitment to weights from later visibility, reducing the value of simply copying newly visible
weight information. The
[Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)
places commit reveal in that mitigation context.

The useful idea is timing: if weight information becomes visible only after a delay, copying it is
less useful for producing timely validator evaluation.

## Consensus-Based Weights

Consensus-based weights, also called liquid alpha, are another related defense. The
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights) page
describes bond adjustment based on consensus alignment, while the
[Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor) page
places liquid alpha alongside delayed visibility as a tool against copied or stale weights.

Delayed visibility and consensus-based weights address different parts of the problem. One reduces
the usefulness of copied weight information; the other changes how aligned or stale signals affect
bond outcomes.

## Evidence Boundary

Similar weight patterns do not automatically prove copying. The
[Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)
focuses on dependence on copied or derived weights, while
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) also allows validators to
converge when they evaluate miner performance similarly.

The boundary matters for interpretation. Weight copying is about lack of independent evaluation, not
merely about two validators producing similar-looking scores.

## Development Stage Context

Bittensor's subnet-development path moves from localnet to testnet and then mainnet
([Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)).
For weight copying, that sequence changes how readers should interpret examples of copied or stale
validator-weight behavior.

In localnet, weight-copying mechanics can be tested in an isolated environment. Localnet examples
reflect local validators, local visibility timing, and local chain state rather than production
subnet consensus.

On testnet, weight-copying behavior can be observed in a shared, non-production network. Testnet
examples can show realistic delayed-visibility and validator-evaluation behavior, but they remain
separate from mainnet validator-weight outcomes.

On mainnet, weight copying is a production consensus-quality risk around validator weights and
independent miner evaluation. In
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus), validator weights feed into
miner incentives and validator dividends, so copied or stale signals can affect the quality of
consensus inputs.

Mechanism context also matters. When a subnet uses multiple incentive mechanisms, a weight-copying
example belongs to the relevant mechanism rather than every evaluation path in that subnet
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A weight-copying example from one environment should not
be read as representing validator behavior or consensus outcomes in another environment. It also
needs the mechanism and visibility context that produced the copied-weight pattern.

That keeps local tests, shared testing observations, and production weight-copying context distinct.

## Reading Boundaries

Weight copying is the right term when the focus is validator dependence on copied weight
information. If the focus is the weight signal itself,
[validator weights](https://docs.learnbittensor.org/resources/glossary#validator-weights) is more
precise. If the focus is the aggregation process,
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) is more precise. If the focus
is delayed visibility, [Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal) is
more precise.

These boundaries keep weight copying focused on evaluation-quality risk, not a general description
of every validator-weight pattern.
