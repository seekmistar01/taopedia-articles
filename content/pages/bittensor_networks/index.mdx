# Bittensor Networks

Bittensor networks are the environments where Bittensor activity is interpreted. Mainnet, testnet,
and localnet provide different contexts for primary use, shared testing, and local development
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

That environment label matters because the same Bittensor vocabulary can appear in more than one
context. A value, subnet example, tool display, or chain observation should preserve the network
where it belongs.

## Mainnet, Testnet, and Localnet

Mainnet is the primary Bittensor environment, testnet is the shared testing environment, and
localnet is the local development environment
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Those labels keep examples from being overstated. Localnet is useful for isolated development,
testnet is useful for shared testing, and mainnet is the primary network context.

## Timing and Purpose Details

Each Bittensor network environment has a different purpose
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)). Mainnet is for
transactions with financial value, testnet is for test transactions with no value, and localnet is
for development and testing in a user-controlled environment.

That purpose row is important because it changes how strong a claim can be. A mainnet observation
belongs to the value-bearing production context. A testnet observation belongs to shared testing. A
localnet observation belongs to a private development environment.

Block timing also differs by environment
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)). Mainnet and
testnet use one block every 12 seconds. Localnet can use fast blocks at 0.25 seconds or non-fast
blocks at 12 seconds.

Those timing values help readers interpret local examples. A localnet run in fast-block mode can
produce block-driven effects much faster than mainnet or testnet, so timing evidence should keep the
environment and block mode attached.

Test TAO also differs by environment. The network table lists no test TAO for mainnet, test TAO
available on request for testnet, and local test value available in the local Alice account
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)). That reinforces
the difference between value-bearing and test-only contexts.

Archive and lite access are listed under mainnet rather than testnet or localnet
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)). That keeps
public chain-access examples tied to the network where the table actually lists that access.

These details make the network label more than a name. It carries purpose, timing, and test-value
context that should travel with any chain observation, tool output, or development example. Without
that label, a timing or token example can be easy to read in the wrong environment.

## Why Environment Labels Matter

The environment label changes how readers should interpret an observation. A block height, chain
read, tool display, or subnet example belongs to the network where it was produced
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Bittensor Tools](https://docs.learnbittensor.org/concepts/tools)).

That label prevents examples from drifting between contexts. A localnet observation can help explain
shape or behavior in isolation, while a testnet or mainnet observation belongs to a different shared
chain history. The same rule applies to staking, pool, and emission examples: evidence from one
network should not be read as proof of behavior on another
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. Bittensor networks
vocabulary names those three environments directly, so development-stage context is part of the term
rather than an external add-on
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

In localnet, Bittensor activity can be exercised in an isolated environment for subnet and wallet
development. Local outcomes reflect local configuration and do not represent production network
behavior.

On testnet, Bittensor runs in a shared, non-production network where integration behavior can be
observed without touching mainnet state.

On mainnet, Bittensor networks vocabulary refers to the live production network where primary use,
staking, subnet participation, and emissions apply.

Readers should keep mainnet, testnet, and localnet separate when citing examples. Evidence from one
network environment should not be read as proof of behavior in another.

## Distinction from Mainchain

Bittensor networks and mainchain are related but different terms. Bittensor networks is the broader
environment label covering mainnet, testnet, and localnet, while mainchain names the primary
production network context inside that picture
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Mainchain](https://docs.learnbittensor.org/resources/glossary#mainchain),
[Introduction to Bittensor: Subnet development](https://docs.learnbittensor.org/learn/introduction#subnet-development)).

- **Bittensor networks** — the full set of mainnet, testnet, and localnet environments.
- **Mainchain** — the primary production network context within that set.

## Distinction from Subtensor

Bittensor networks and Subtensor answer different scope questions. Bittensor networks name which
environment an observation belongs to, while Subtensor names the chain layer that records state and
processes extrinsics within each environment
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

- **Bittensor networks** — environment label (mainnet, testnet, localnet).
- **Subtensor** — the chain layer operating inside each environment.

## Distinction from Subnet

A Bittensor network and a subnet name an environment and a market within it. A network is the
mainnet, testnet, or localnet environment where activity is interpreted, while a subnet is an
incentive-based marketplace where miners produce work and validators evaluate it
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

- **Bittensor network** — the environment where activity belongs.
- **Subnet** — a work-and-evaluation market running in that environment.

That distinction keeps subnet vocabulary and environment vocabulary separate: subnets describe
work-and-evaluation markets, while networks describe where observations and examples belong
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

## Distinction from Local Blockchain

A Bittensor network and a local blockchain name an environment set and one private chain. Bittensor
networks covers mainnet, testnet, and localnet, while a local blockchain is the private, isolated
chain used for development and testing
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)).

- **Bittensor networks** — the full set of mainnet, testnet, and localnet.
- **Local blockchain** — the private chain used for development and testing.

Localnet is close to local blockchain vocabulary, but the terms are not identical. A local
blockchain is private, isolated, and used for development and testing, while Bittensor Networks is
the broader environment vocabulary that also includes testnet and mainnet
([Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

This boundary prevents local examples from being read as shared-network evidence. A local blockchain
can be useful for development, but it is still a local environment.

Localnet and testnet are both useful for examples, but they do not mean the same thing. Localnet is
isolated, while testnet is shared testing context
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)).

This helps readers interpret examples without turning the network article into a development guide.
A local example can explain shape or behavior in isolation; a testnet example can show shared
testing context; neither should be treated as the same environment as mainnet.

## Distinction from Tool

Tools can display information from whichever network they are connected to. The
[Bittensor Tools](https://docs.learnbittensor.org/concepts/tools) overview places tools around
Bittensor features, while the
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference gives
the environment vocabulary for interpreting what those tools show.

For readers, the network label is part of the observation. A tool display without environment
context is weaker than one that clearly identifies whether it belongs to mainnet, testnet, or
localnet.

The same BTCLI command can target any of the three networks, so the network selection matters as
much as the command itself when reading or reproducing an example
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

## Distinction from Yuma Consensus

Bittensor networks name the mainnet, testnet, and localnet environments readers use to interpret
chain observations. Yuma Consensus is the on-chain mechanism that converts validator weight
submissions into emission shares each tempo on a live network and is not an environment label
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Network vocabulary tells readers which chain an observation came from. Yuma vocabulary names how
validator weights become emissions on a subnet within that environment.

- **Bittensor networks** — mainnet, testnet, and localnet deployment environments.
- **Yuma Consensus** — on-chain mechanism that turns validator weights into emission shares.

## Reader Boundary

Bittensor Networks should be read as environment-context vocabulary. The reference supports claims
about mainnet, testnet, and localnet separation; it does not redefine miners, validators, staking,
emissions, or subnet mechanisms
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

The network name tells readers where an observation belongs; it does not explain the observation by
itself.
