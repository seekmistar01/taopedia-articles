# Chain Reads

Chain reads describe looking at public Bittensor chain state without asking Subtensor to record a
new action ([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

The term is about observation. It should be read separately from submitted runtime actions that ask
the chain to process a change.

## Read-Only Boundary

The read-versus-submit boundary separates public inspection from state-changing submissions. A chain
read can observe public state, but the read itself does not move value, change stake, or alter a
role ([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

This does not make a chain read a full explanation of why state looks the way it does. It only means
the read is not the same thing as submitting a change.

A read can be useful before a user decides what to do next, but the read is still only inspection.
The state-changing step begins when a submitted action is sent to the runtime
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

## Chain Inspection Context

Chain inspection can show chain state, metadata, blocks, and other public records. Chain reads fit
inside that inspection side of the workflow
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Chain inspection can help readers look at state, metadata, blocks, and records. It does not by
itself explain the full cause of the state being shown
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Different read targets answer different questions. Storage entries show state surfaces, events show
runtime records, and constants show named runtime values
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

## Fee and Submission Context

Reading chain state is free in the Bittensor fee documentation, while submitted state-changing
operations can involve fees
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

That distinction prevents read-only inspection from being mistaken for a submitted transaction. The
read can inform interpretation; the submitted operation remains separate.

Fee language should stay with the submitted action unless the point is specifically that the read is
free ([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

## Reference Definitions and Queried Values

Subtensor references describe what can be inspected, while a chain read returns or displays a value
from the selected chain context. Storage, events, constants, and blocks answer different read
questions ([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

This distinction keeps definitions and observed values from being collapsed into one sentence. A
reference can define what can be read; the read shows an observed value.

The observed value belongs to the network and inspection context where it was read
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

## Network Scope

A chain read belongs to the Bittensor network being inspected. Bittensor network documentation
separates mainnet, testnet, and localnet contexts
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

When precision matters, the network and inspection context should stay attached.

A localnet or testnet read can illustrate how inspection works, but it should not stand in for
another network's chain history
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

## Reader Boundary

Chain reads should be read as read-only observation. They are not proof of intent, a prediction of
later submissions, or a replacement for mechanism documentation
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

The stable article-level point is that chain reads observe public chain state without asking
Subtensor to process a new state-changing submission.

Submitted actions, fees, emitted records, and changed state each need their own surrounding context
when the article goes beyond the read itself
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).
