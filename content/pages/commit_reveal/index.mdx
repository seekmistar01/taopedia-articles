# Commit Reveal

Commit Reveal is a Bittensor mechanism that delays when validator weight signals become visible. The
official Commit Reveal reference frames the mechanism around concealing fresh weights before they
can be reused by observers ([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal),
[Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)).

The term is about visibility timing. It does not change what validator weights mean, and it does not
replace the consensus process that later interprets available validator signals.

## Copying Resistance

The weight-copying problem comes from visible weight information being useful to validators that did
not independently evaluate subnet work
([Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

Commit Reveal reduces that timing advantage by hiding fresh weight signals until a later reveal
point. The protection is narrow: it makes recent signals harder to copy before disclosure, while the
underlying evaluation meaning of validator weights remains separate.

## Timing Model

At a high level, the mechanism separates a concealed phase from a later reveal phase. The Commit
Reveal documentation describes this two-step timing model as the core of the mechanism
([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

That timing model keeps the concept distinct from consensus scoring. Commit Reveal answers when
weight information can be seen; Yuma Consensus explains how available validator signals are
aggregated into subnet outcomes
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Drand Time-Lock Context

Commit Reveal uses Drand time-lock encryption to prevent concealed weight data from being decrypted
before the intended reveal time
([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal),
[Glossary: Drand/time-lock encryption](https://docs.learnbittensor.org/resources/glossary#drandtime-lock-encryption)).

In this context, Drand is timing infrastructure for hidden weights. The important concept is the
visibility gap: weight material can exist before observers can read it.

## Subnet Mechanism Context

The Commit Reveal reference places the mechanism in subnet-specific settings, and subnet
hyperparameter documentation provides the broader vocabulary for subnet mechanism settings
([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

That makes Commit Reveal a subnet mechanism rather than a universal description of every
validator-weight path. A subnet can be discussed in terms of whether this visibility delay is part
of its weight path, while the article definition stays focused on the mechanism itself.

## Consensus Context

Commit Reveal and Yuma Consensus sit at different points in the validator-weight path. Commit Reveal
concerns delayed visibility before weights are openly readable, while Yuma Consensus concerns
aggregation after validator signals are available for consensus processing
([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Keeping that order clear prevents the terms from collapsing into one idea. Commit Reveal protects
the timing of weight disclosure; Yuma Consensus describes how disclosed signals affect subnet
incentives.

## Environment Boundary

Bittensor documentation separates local development, testchain, and mainchain contexts, and it also
separates mainnet, testnet, and localnet network contexts
([Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

A Commit Reveal example from one environment should not be treated as proof about another
environment. Localnet examples can show controlled behavior, testnet examples can show shared
non-production conditions, and mainnet references need separate support from mainnet-specific
material.

## What Commit Reveal Does Not Mean

Commit Reveal does not name a validator strategy, a reward rule, a complete subnet protocol, or a
guarantee about scoring quality
([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Keeping that boundary preserves the term's narrower meaning. Commit Reveal identifies delayed
visibility for validator weight signals, while validator weights, weight copying, Drand, subnet
hyperparameters, and Yuma Consensus describe neighboring parts of the system.
