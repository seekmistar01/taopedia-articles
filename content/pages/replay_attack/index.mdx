# Replay Attack

A replay attack is when an adversary takes a valid, already-signed message and sends it again,
hoping the receiver accepts the repeat as if it were new. The risk is that a legitimate action,
captured once, could be made to happen a second time without the signer intending it.

References:
[bittensor.core.axon](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/)

## What It Would Do

The aim of a replay is duplication. An attacker who observes a valid signed message could try
resubmitting it, hoping the receiver runs its effect again. The attack succeeds only if the
receiving side fails to notice that the very same signed message has already been processed.

References:
[bittensor.core.axon](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/)

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For replay
attacks, that sequence changes whether signed-message examples should be read as local axon traffic
or as production synapse and extrinsic nonce paths.

In localnet, signed synapse and extrinsic flows can be exercised in an isolated environment.
Localnet nonce behavior does not represent production axon freshness rules on live subnets.

On testnet, dendrite-to-axon signed traffic runs on a shared non-production network. Testnet nonce
and extrinsic state are separate from mainnet neuron activity
([bittensor.core.axon](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/)).

On mainnet, replay risk concerns live signed synapse traffic between neurons and live extrinsic
submission on [Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor), where stale
or duplicated nonces must be rejected on the public network
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A replay-attack example from one environment should not be
read as describing production nonce behavior on another network.

## Distinction from Extrinsic Account Nonces

Subtensor extrinsics use account-level nonce expectations so a signed chain transaction cannot be
accepted twice on the same account path
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

Axon verification applies separate nonce tracking to dendrite endpoint keys in signed synapse
traffic
([bittensor.core.axon](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/),
[Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse)).

- **Extrinsic account nonce** — chain replay protection for a submitting account.
- **Axon endpoint nonce** — freshness and monotonic checks for `dendrite.hotkey:dendrite.uuid`.

A replay failure on the axon path therefore reflects stale or reused endpoint nonce state rather
than proof that the hotkey signature was cryptographically invalid
([Glossary: Hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey)).

Axon verification in the Bittensor SDK tracks nonce state per dendrite endpoint key, not by hotkey
alone. Documentation for `bittensor.core.axon` describes checking both freshness and monotonic
increase for the `dendrite.hotkey:dendrite.uuid` pair associated with a synapse
([bittensor.core.axon](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/),
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite)).

For readers, that scope differs from Subtensor account nonces. Extrinsic replay defenses protect
wallet-level chain actions keyed to the submitting account, while axon nonce rules protect off-chain
synapse traffic for the specific dendrite endpoint that produced the signature
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

Both layers resist message reuse but apply to different parts of Bittensor activity: extrinsic
nonces guard on-chain actions such as staking or registration, while axon nonce checking guards
signed off-chain synapse traffic between neurons
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[bittensor.core.axon](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/)).

Bittensor's defense is the nonce, a value attached to signed messages. The documentation describes
the network checking for increasing nonce values as a vital step in preventing replay attacks, and
treating a nonce as valid only when it is fresh, meaning within an acceptable time range. A replayed
message carries a nonce that has already been seen or is no longer fresh, so the receiver recognizes
it as a repeat and turns it away instead of acting on it twice.

References:
[bittensor.core.axon](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/)

## Signed Payload Reuse Is Not Signature Forgery

A replay attack reuses a valid signed message; it does not require breaking the cryptography.
Documentation therefore emphasizes monotonic and fresh nonces so receivers can reject duplicates
even when the signature still verifies
([Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[Glossary: Axon](https://docs.learnbittensor.org/resources/glossary#axon)).

Readers should separate replay from impersonation. Impersonation would require signing as another
hotkey, while replay simply resubmits an already valid payload until nonce state marks it stale or
already consumed ([Understanding Neurons](https://docs.learnbittensor.org/learn/neurons)).

Nonce rejection therefore marks a message as already seen or no longer fresh for that dendrite
endpoint key. It does not, by itself, establish that the signer forged a new payload
([bittensor.core.axon](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/)).

## Distinction from Mempool

A replay attack and the mempool meet at transaction acceptance. A replay attack resends an
already-signed message hoping it is accepted again, while the mempool is the holding area where
submitted transactions wait before inclusion, so a replayed transaction re-enters the mempool and
must be rejected there rather than re-executed
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

## Distinction from Synapse

Replay attack names the reuse of a valid signed message. Synapse names the structured message object
carried through dendrite-to-axon communication
([Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[bittensor.core.axon](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/)).

Misuse-pattern vocabulary and message-object vocabulary answer different questions on the same
signed traffic path. A captured synapse payload could be resubmitted only if nonce and freshness
checks fail.

- **Replay attack** — resubmitting a valid signed message.
- **Synapse** — the structured message object in dendrite-to-axon traffic.

## Distinction from Axon

Replay attack names message reuse at the receiver. Axon names the server-side endpoint that verifies
incoming signed synapse traffic before responding
([Glossary: Axon](https://docs.learnbittensor.org/resources/glossary#axon),
[bittensor.core.axon](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/)).

Attack vocabulary describes duplicate submission; axon vocabulary describes where freshness and
monotonic nonce checks are applied.

- **Replay attack** — duplicate submission of signed traffic.
- **Axon** — the receiver endpoint that applies nonce checks.

## Distinction from Dendrite

Replay attack concerns a signed payload being sent again. Dendrite names the client-side component
that produces and sends synapse requests toward axon endpoints
([Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite),
[Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse)).

Reuse is detected at the receiving axon; dendrite vocabulary names the sending side where the
original signed payload was first emitted.

- **Replay attack** — reuse of a previously signed payload.
- **Dendrite** — the client component that first sends synapse requests.

## Distinction from Commit Reveal

A replay attack and commit reveal both concern how the network handles a message after it is
observed, but they pull in opposite directions. A replay attack tries to make an already-seen signed
message act again, so the defense is detection: reject anything recognized as a repeat. Commit
reveal instead deliberately delays disclosure, solving the weight-copying problem by giving would-be
copiers access only to stale weights so their copied submissions diverge from current consensus
([Glossary: Commit Reveal](https://docs.learnbittensor.org/resources/glossary#commit-reveal)).

- **Replay attack** — defends by recognizing a reused payload and refusing to run it twice.
- **Commit reveal** — defends by withholding fresh information until it is too late to copy.

## Distinction from MEV

A replay attack and MEV both exploit the path a transaction takes toward inclusion, yet they profit
in different ways. A replay attack reuses one already-signed message hoping its effect runs a second
time, gaining nothing from how transactions are ordered. MEV is the profit extracted by reordering,
inserting, or censoring transactions within a block, so its leverage comes from controlling sequence
rather than from duplicating a single signed payload
([Glossary: MEV](https://docs.learnbittensor.org/resources/glossary#mev-maximal-extractable-value)).

- **Replay attack** — value would come from a captured message being accepted again, defeated by
  nonce freshness.
- **MEV** — value comes from manipulating the position of distinct pending transactions.

## Distinction from Yuma Consensus

A replay attack reuses a previously valid signed message in the hope it will be accepted again. It
names transaction-security vocabulary, not the on-chain step that turns validator weight submissions
into emission shares
([Glossary: Replay Attack](https://docs.learnbittensor.org/resources/glossary#replay-attack)).

Yuma Consensus is the on-chain process that converts submitted validator weights into emission
shares after evaluation completes. Nonce and freshness checks defend against replays on signed
messages; Yuma Consensus settles weight outcomes on a subnet
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers should keep replay-defense vocabulary separate from the settlement step. Replay attack
answers what message reuse exploits; Yuma Consensus answers how validator weights become emission
shares in a subnet
([Glossary: Replay Attack](https://docs.learnbittensor.org/resources/glossary#replay-attack),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Replay attack** — reusing a valid signed message without fresh authorization.
- **Yuma Consensus** — on-chain weight aggregation that settles emissions.

## Reader Boundary

This page defines the concept and the role of nonce checking at a high level. It does not specify
the exact nonce rules, time ranges, or signing internals, which are protocol and tooling details
that can change. The durable point is that reuse of a signed message is meant to be detectable, so a
replay does not quietly succeed.

[bittensor.core.axon](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/)

## Dendrite Requests and Axon Responses Use Signed Synapses

Bittensor's [Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite) and
[Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse) describe the
off-chain request path: a dendrite sends a signed synapse to an
[axon](https://docs.learnbittensor.org/resources/glossary#axon), which verifies the signature before
responding. Replay risk sits in that signed message path, not only in chain
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics).

The axon-side documentation referenced in this article focuses on nonce checking at the receiver.
The dendrite and synapse vocabulary names the complementary sender side where a valid signed payload
is first produced and then could be captured for reuse if freshness checks fail.

References: [Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite),
[Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse)

## Freshness Windows Reject Stale Nonces

The axon documentation describes nonce validation as requiring both an increasing value and a fresh
nonce within an acceptable time range. A replayed synapse therefore fails not only when its nonce
was already consumed, but also when the attached nonce is too old to count as current for that
signing [hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey).

That freshness rule limits how long a captured signed message remains dangerous. An adversary who
records a valid synapse cannot rely on it indefinitely; the receiver is expected to treat an expired
nonce as invalid even if the signature itself still verifies mathematically.

References:
[bittensor.core.axon](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/),
[Glossary: Hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey)
