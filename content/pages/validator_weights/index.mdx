# Validator Weights

Validator weights are evaluation signals set by subnet validators. They express how validators
assess miner work inside a subnet before Yuma Consensus filters and aggregates those signals into
subnet outcomes ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The term is subnet-specific. A validator weight only has meaning inside the task, scoring method,
and incentive setting that produced it, so it is different from a general validator reputation
score.

## Evaluation Signal

A validator weight is a subnet-specific evaluation of miner work under the rules of the subnet's
incentive mechanism
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Similar-looking values can carry different meaning across subnet tasks because the subnet's work
target and scoring method give the signal its interpretation.

Validator weights describe miner-work assessments inside the subnet. Stake position, permit status,
and validator standing are separate concepts with their own sources and consensus roles.

## Consensus Input Role

Validators use weight signals to express relative evaluations of miners. Yuma Consensus combines
those signals into a broader consensus result rather than treating one validator's view as final on
its own ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Validator weights enter the consensus process as inputs. Incentives, dividends, bond values, rank,
and trust describe later or neighboring results.

## Vector and Matrix Structure

Validator weights are often read through weight-vector vocabulary. A
[weight vector](https://docs.learnbittensor.org/resources/glossary#weight-vector) is one validator's
structured set of miner evaluations, while validator weights name the evaluation values inside that
consensus input
([Consensus-Based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

[Weight matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix) sits one level
wider: the subnet-level collection of validator weight vectors used as structured input for Yuma
Consensus.

Validator weights are the values; a weight vector groups one validator's values; the weight matrix
groups the subnet's validator vectors.

## Consensus Score Filters Outlier Weights

Validator weights are evaluation inputs, while
[consensus score](https://docs.learnbittensor.org/resources/glossary#consensus-score) is the
stake-weighted benchmark used to filter outlier weights before final rank results.

A single validator weight submission does not by itself define the consensus score reading, because
consensus score is computed from the wider validator-weight set rather than from one submission
alone.

## Rank Follows After Filtering and Aggregation

Validator weights name the evaluation signals validators submit about miner work.
[Rank](https://docs.learnbittensor.org/resources/glossary#rank) names the consensus-adjusted
miner-side result produced after those signals are filtered and aggregated
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

A single validator weight submission does not by itself report where a miner ranks, because rank
belongs to the post-aggregation miner result rather than to one validator's signal alone.

[Trust](https://docs.learnbittensor.org/resources/glossary#trust) describes how miner support
changes through clipping after that filtering path.

## Stake Influence

Validator weights are interpreted inside a stake-weighted consensus process.
[Stake weight](https://docs.learnbittensor.org/resources/glossary#stake-weight) explains how much
influence a validator's signal carries, while validator weights express what that validator
evaluated about miner work.

[Validator trust](https://docs.learnbittensor.org/resources/glossary#validator-trust) describes
validator-side influence after clipping, while validator weights are the evaluation signals that
enter the process.

## Quality and Timing

Validator weights are most useful when they reflect independent evaluation. Weight copying weakens
that signal by relying on visible weight information instead of fresh miner-work assessment
([The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

[Validator-miner bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)
carry smoothed validator-miner relationship history, while validator weights are evaluation signals
that feed the consensus path. Validator weights describe current evaluation signals; bonds describe
smoothed relationship history shaped by earlier signals and later smoothing logic.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For validator
weights on a subnet such as netuid 1, that sequence changes how readers should interpret submitted
weight vectors, permit eligibility, and pre-consensus filtering examples.

In localnet, validator-weight examples can be tested in an isolated environment. Local validator
sets and submitted vectors reflect local chain configuration rather than production subnet consensus
input.

On testnet, validator weights can be observed in a shared, non-production network. Testnet weight
submissions on a selected netuid are separate from mainnet validator-weight state.

On mainnet, validator weights are live consensus-input signals submitted for Yuma Consensus on the
production Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A validator-weight example from one environment should not
be read as representing production consensus input on another network.

## Weights Arrive Through the Tempo Cycle

[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
describes validators as periodically computing weight vectors from miner performance and
transmitting those vectors to the blockchain.

The chain waits for the latest ranking signals from the subnet's validators before assembling weight
matrices for Yuma Consensus input. Epoch documentation states that validator weights are submitted
during the preceding [tempo](https://docs.learnbittensor.org/resources/glossary#tempo), while
emissions from the consensus pass are finalized at the end of each tempo period
([Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

Validator weights therefore belong to a recurring submission window rather than to one isolated
snapshot.

## Multiple Mechanisms Carry Separate Weight Sets

Subnets that run more than one
[incentive mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism)
maintain separate evaluation tracks rather than one blended ranking table
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

Each validator sets weights independently for each mechanism, and each mechanism keeps its own
weight matrix for Yuma Consensus calculations. A high weight in one mechanism's vector does not
automatically carry the same meaning in another mechanism's separate matrix on the same subnet.

## Eligibility Rules Trim Submitted Weights

Epoch documentation lists pre-consensus filtering steps that remove certain submitted weights before
stake-weighted aggregation begins
([Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

Self-assigned weights are removed so validators cannot vote for themselves, with an exception for
the subnet creator when that role exists on the subnet. Weights tied to a neuron before that
neuron's latest [registration](https://docs.learnbittensor.org/resources/glossary#register) are also
dropped so stale assignments do not follow re-registered identities.

Only rows from validators holding a
[validator permit](https://docs.learnbittensor.org/resources/glossary#validator-permit) influence
consensus. Together, those checks limit which submitted validator weights can enter Yuma Consensus
at all.

## Reader Boundary

Validator weights name subnet evaluation signals. They describe submitted miner-work assessments
rather than miner leaderboards, reward guarantees, validator permits, or a complete explanation of
Yuma Consensus ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Specific examples belong to the subnet, mechanism, network, and source that produced the value at
the time of observation.

## Relationship to Weight Setting

Validator weights and weight setting describe the same signal as data and action. Validator weights
are the evaluation values a validator assigns to miners to express how it rates their work, while
weight setting is the action that transmits and records that weight vector on chain, so the weights
are the content and weight setting is the step that commits them before Yuma Consensus reads them
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Validator weights** — the evaluation values a validator assigns to miners.
- **Weight setting** — the on-chain action that submits and records those weights.
