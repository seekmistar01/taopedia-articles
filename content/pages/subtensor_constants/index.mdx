# Subtensor Constants

Subtensor constants are named runtime reference values exposed by Bittensor's Subtensor runtime.
They are grouped by runtime area and provide stable vocabulary for values such as defaults, bounds,
limits, timing parameters, and feature-status flags
([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

The term belongs to runtime-reference vocabulary. A constant identifies a named value in the runtime
metadata; it is not the same thing as a storage entry, event record, extrinsic, or full mechanism
explanation.

## Reference Role

The Subtensor constants reference helps readers identify official names for runtime-defined values.
Each entry should be read with its owning runtime area, entry name, type, access path, and summary
([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

That structure keeps constants from being read as isolated numbers. The name identifies the value,
the type describes the value shape, and the owning runtime area explains where the value belongs.

## Runtime Area Context

Constants are grouped by runtime area. Some areas describe general chain behavior, while the
Subtensor area contains Bittensor-specific values connected to subnet, rate-limit, weight, and
feature-status vocabulary
([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

The runtime-area grouping is part of the meaning of the entry. Similar-looking values can have
different roles when they belong to different areas, so the area should stay attached to any article
claim that cites a constant.

## Constants and Mechanisms

A constant can support a mechanism explanation, but it usually does not replace that explanation.
For example, a rate-limit-related value can help name a bound or default, while the rate-limit
documentation explains the block-based cooldown model
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

This boundary matters for Bittensor-specific topics. A constant may identify a default, limit, or
feature flag, but the related concept page explains how that value fits into user-facing protocol
behavior.

## Storage and Event Boundary

Constants, storage entries, and events answer different reference questions. Constants expose
runtime-defined reference values. Storage entries expose state surfaces. Events name runtime records
emitted during processing
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

A feature can involve all three surfaces without making them interchangeable. A constant can name a
value used by a feature, a storage entry can locate chain state, and an event can record that
runtime activity happened.

## Inspection Context

The chain-inspection documentation includes constants among the chain data readers can inspect
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).
This places constants on the read-only reference side of Bittensor tooling: they can be inspected
without submitting a state-changing runtime action.

Inspection still needs context. The article claim should make clear whether it is citing the
constants reference for a named runtime value or citing another source for observed state, processed
activity, or mechanism behavior.

## Network and Runtime Scope

Subtensor constants belong to a runtime and network context. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, while the constants reference identifies values exposed by
the runtime being discussed.

That means a constant is strongest as evidence for runtime vocabulary and value shape. If an article
discusses how a value behaves across environments or after runtime changes, the claim needs the
surrounding network, runtime, and mechanism context rather than the constant name alone.

## Reader Boundary

Subtensor constants should be read as runtime-reference entries. They are useful for naming
defaults, bounds, limits, timing values, and feature-status values, but they are not a complete
guide to the feature that uses them
([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

The stable article-level point is that Subtensor constants provide named reference values grouped by
runtime area, and those values should be interpreted beside the relevant concept, storage, event, or
mechanism documentation.
