# Batch Transactions

Batch transactions group multiple Bittensor chain actions into one submitted transaction
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

The term belongs to submitted-action grouping vocabulary. It should be read separately from passive
chain reads and from the individual meaning of each inner action.

## Grouped Action Boundary

The official batch-transaction documentation describes batching as a way to include multiple actions
inside one submitted transaction
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

That makes the batch a container. The outer submission groups the actions, but each inner action
still has its own purpose, evidence context, and possible result.

## Failure Boundary

The batch documentation distinguishes batching forms by failure behavior
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

At a concept level, the important distinction is whether the grouped submission stops after a failed
inner action, requires all inner actions to succeed together, or continues evaluating later actions.
Readers should not assume all batches have the same all-or-nothing behavior.

## Fee Boundary

Batching changes grouping, but it does not erase the fee character of the inner actions. The
transaction-fees documentation explains that batch fees aggregate inner-action fee behavior and add
batch-wrapper overhead
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

This means a batch should not be treated as a free shortcut. Fee context still depends on the inner
actions and the wrapper around them.

## Extrinsic Context

Batch transactions sit in the submitted-action side of Subtensor. The Subtensor extrinsics reference
describes runtime methods submitted to the chain
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

That boundary separates batching from chain inspection. Reading state observes the chain; batching
submits grouped calls for processing.

## Network Context

Bittensor documentation separates mainnet, testnet, and localnet contexts
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

A batch example should therefore keep the selected network attached when the claim depends on
observed behavior. The grouping concept can be stable while exact outcomes remain
environment-specific.

## Reader Boundary

The batch-transaction source supports claims about grouped submissions and failure forms
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

It does not provide a blanket success guarantee, exact fee quote, or universal result for every
grouped action. A concrete batch claim should identify the batching form, network context, and inner
actions being interpreted.

## Inner Action Boundary

A batch transaction groups calls, but it does not erase the meaning of each inner action. The batch
source describes grouped submissions, while the extrinsics reference keeps the submitted calls tied
to runtime methods ([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

For readers, the batch is the container and the inner calls are still the actions being evaluated. A
source-backed batch claim should therefore distinguish grouping behavior from the purpose and result
of each call inside the batch.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Batch
Transactions, that sequence changes how readers should interpret batched-call examples, failure
behavior, and fee observations.

In localnet, Batch Transactions examples can be exercised in an isolated environment. Localnet
observations reflect local chain state and local configuration rather than production Bittensor
behavior.

On testnet, Batch Transactions behavior can be observed in a shared, non-production network. Testnet
examples are useful for checking interactions under shared chain conditions while staying separate
from mainnet state.

On mainnet, Batch Transactions describes live submitted chain action behavior on the production
Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. An example or outcome from one environment should not be
read as evidence of production behavior in another environment.
