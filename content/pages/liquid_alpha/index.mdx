# Liquid Alpha

Liquid alpha is the name the documentation gives to the consensus-based weights feature that, when
enabled, lets the protocol vary the smoothing factor used as validator bonds are updated instead of
applying a single fixed factor across the subnet. The smoothing factor is the alpha (α) term in the
exponential moving average that the protocol applies to each validator-miner bond from one epoch to
the next. With liquid alpha, that α is determined for each validator-miner bond pair from the
current consensus relationship rather than being shared uniformly across the subnet.

References:
[Consensus-based Weights / Liquid Alpha](https://docs.learnbittensor.org/concepts/consensus-based-weights),
[Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)

## Bonds and the EMA

A bond represents the investment a validator has made in evaluating a particular miner, and the
protocol smooths each bond across epochs with an exponential moving average so that a single epoch
does not dominate the value. In that moving average, the alpha term controls how much weight the
current epoch carries relative to the accumulated history: a larger alpha reacts faster to the
latest epoch, while a smaller alpha keeps more of the prior value. Liquid alpha is concerned with
how that alpha term is chosen for each bond.

References:
[Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds),
[Glossary: Exponential Moving Average](https://docs.learnbittensor.org/resources/glossary#exponential-moving-average-ema)

## What Liquid Alpha Changes

When liquid alpha is disabled, the bonds moving average uses one fixed alpha across the subnet. When
it is enabled, the protocol instead determines a variable alpha for each validator-miner bond pair,
based on the current consensus for that pair, so the implementation works with a matrix of per-pair
alpha values rather than a single number. Each pair's alpha is held within the configured alpha_low
and alpha_high bounds. The documented motivation is that a validator's dividends become better
correlated with the performance of the specific miner it is setting weights on, which helps
validators identify well-performing miners and bond to them more quickly.

References:
[Consensus-based Weights / Liquid Alpha](https://docs.learnbittensor.org/concepts/consensus-based-weights),
[Subtensor: run_epoch.rs](https://github.com/opentensor/subtensor/blob/main/pallets/subtensor/src/epoch/run_epoch.rs)

## The alpha_low and alpha_high Bounds

The feature is controlled by two subnet hyperparameters, alpha_low and alpha_high, configured
together with an enable flag, so a subnet turns liquid alpha on before the bounds take effect. In
the current Subtensor implementation these are stored as unsigned 16-bit integers, where the full
16-bit range maps onto the interval from 0 to 1. The setter requires both values to be at least one
fortieth of the maximum 16-bit value, which is 1638 and corresponds to roughly 0.025 of full scale,
and it requires alpha_low to be less than or equal to alpha_high, so the two bounds may be set equal
and either may reach the maximum 16-bit value. Because these are subnet hyperparameters, they are
set through the subnet's own configuration rather than by individual validators.

Reference:
[Subtensor: do_set_alpha_values (run_epoch.rs)](https://github.com/opentensor/subtensor/blob/main/pallets/subtensor/src/epoch/run_epoch.rs)

## A Per-Subnet Setting

Liquid alpha is configured per subnet rather than network-wide: one subnet can enable it with its
own alpha_low and alpha_high bounds while another leaves the feature off entirely, and the two
choices are independent. This mirrors how other consensus-related hyperparameters apply to a single
subnet at a time. Whether the feature is enabled, and the alpha_low and alpha_high values in force,
are live per-subnet hyperparameter state that the subnet owner can change; they should be read for
the relevant subnet with the Bittensor CLI when current data is needed rather than assumed from this
reference.

Reference: [Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)

## Consensus Alignment Context

Liquid alpha ties bond smoothing to how closely a validator's weights align with subnet consensus.
The official
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)
documentation frames this as making validator dividends better correlated with the miners a
validator actually weights, rather than applying one uniform smoothing factor across every bond
pair.

For readers, liquid alpha is therefore a consensus-alignment mechanism inside the bond path, not a
separate miner ranking rule or a staking instruction.

References:
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)

## Weight Copying Context

Liquid alpha is discussed alongside delayed weight visibility in the official
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)
documentation because copied or stale weights can diverge from current consensus. The feature does
not, by itself, prove that weights were copied; it changes how quickly bonds can respond when a
validator's weights align with consensus.

For readers, liquid alpha belongs in the bond-smoothing path that helps validators bond to useful
miners, not in a separate accusation model about validator behavior.

References:
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights),
[The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)

## Relationship to Multiple Mechanisms

Liquid alpha is usually discussed for one incentive mechanism at a time, but subnets can also run
multiple mechanisms in parallel. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms) notes
that each mechanism operates with its own bond pool for Yuma Consensus calculations.

For readers, liquid alpha still adjusts bond smoothing inside a subnet, but the mechanism context
matters. Bond smoothing in one mechanism should be read separately when a subnet uses more than one
evaluation system.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)

## Relationship to Yuma Consensus

Liquid alpha and Yuma Consensus address related but different parts of Bittensor subnet incentive
flow. Liquid alpha adjusts validator-miner bond smoothing by consensus alignment, while Yuma
Consensus aggregates validator weight signals into miner incentives and validator dividends. The
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)
documentation describes liquid alpha inside the bond path, and
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) describes how validator
evaluations become consensus outcomes.

For readers, liquid alpha names bond-smoothing behavior inside consensus, while Yuma Consensus names
the aggregation step that turns validator evaluations into subnet-level reward results.

References:
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)

## Relationship to Validator-Miner Bonds

Liquid alpha and validator-miner bonds address related but different parts of subnet bond
processing. Validator-miner bonds describe the smoothed relationship between a validator and miners
over time, while liquid alpha changes how quickly those bonds can respond when a validator's weights
align with consensus. The
[Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)
describes the bond relationship, and the
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)
documentation describes liquid alpha as varying the bond-smoothing factor per validator-miner pair.

For readers, validator-miner bonds name the smoothed relationship, while liquid alpha names a
consensus-alignment adjustment to how that relationship updates.

References:
[Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds),
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)

## Relationship to Weight Copying

Liquid alpha and weight copying address related but different parts of validator evaluation quality.
Weight copying names dependence on public weight signals instead of independent miner evaluation,
while liquid alpha adjusts bond smoothing so copied or stale weights can carry less bond influence
where the feature is active. The
[Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)
documentation describes free-riding on validator weight information, and the
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)
documentation places liquid alpha alongside delayed weight visibility as a defense tool.

For readers, weight copying names the evaluation-quality problem, while liquid alpha names a
bond-path mechanism that can limit how much misaligned weight signals shape bond outcomes.

References:
[The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)

## Relationship to Exponential Moving Averages

Liquid alpha and exponential moving averages address related but different parts of bond smoothing.
Exponential moving averages smooth validator-miner bonds over time using an alpha smoothing factor,
while liquid alpha varies that factor per validator-miner pair based on consensus alignment. The
[Glossary: Exponential Moving Average (EMA)](https://docs.learnbittensor.org/resources/glossary#exponential-moving-average-ema)
describes EMA smoothing in bond updates, and the
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)
documentation describes liquid alpha as changing the smoothing response rather than using one fixed
factor across the subnet.

For readers, exponential moving averages name the general smoothing technique, while liquid alpha
names a per-pair adjustment to the smoothing factor inside that technique.

References:
[Glossary: Exponential Moving Average (EMA)](https://docs.learnbittensor.org/resources/glossary#exponential-moving-average-ema),
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)

## Relationship to Consensus Score

Liquid alpha and a consensus score refer to related but different parts of Yuma Consensus. A
consensus score is the stake-weighted median used to filter outlier validator weight submissions,
while liquid alpha adjusts validator-miner bond smoothing based on how closely a validator's weights
align with consensus. The
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)
describes the agreement threshold applied to validator inputs, and the
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)
documentation describes liquid alpha as varying bond smoothing by consensus alignment.

For readers, a consensus score names input filtering inside consensus, while liquid alpha names a
bond-smoothing adjustment that follows from consensus alignment.

References:
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)

## Relationship to Weight Matrix

Liquid alpha and the weight matrix address related but different parts of subnet weight vocabulary.
The weight matrix collects validator weight vectors as structured input to Yuma Consensus, while
liquid alpha adjusts bond smoothing after weights are compared with consensus alignment. The
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)

[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)
documentation describes liquid alpha inside the bond path.

For readers, the weight matrix names the organized collection of validator evaluations, while liquid
alpha names a per-pair bond-smoothing adjustment inside the consensus path that uses those inputs.

References:
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),

## Relationship to Tempo

Liquid alpha and tempo are related but different parts of subnet timing vocabulary. Tempo names the
block-based cadence that governs epoch transitions inside a subnet, while liquid alpha adjusts
validator-miner bond smoothing across those periodic updates when enabled. The
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo) describes tempo as the
subnet-specific cadence for epochs, and the
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)
documentation describes liquid alpha as changing bond-smoothing response per validator-miner pair.

For readers, tempo names the subnet's periodic cadence, while liquid alpha names a bond-smoothing
adjustment that operates inside consensus work across that cadence.

References: [Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo),
