# Subnet 5: Hone

Hone is Bittensor's fifth subnet (netuid 5), operated by [Manifold](https://www.hone.training/). Its
purpose is to benchmark and advance AI reasoning by pitting miners against
[ARC-AGI-2](https://arcprize.org/) — a suite of novel abstract reasoning tasks designed to resist
pattern-memorisation and measure genuine general intelligence. Miners don't run solvers inside the
validator workflow directly; instead they publish a git repository containing their solution, which
a centralised GPU sandbox clones, builds, and executes in isolation.

References: [manifold-inc/hone (GitHub)](https://github.com/manifold-inc/hone),
[taostats.io/subnets/5](https://taostats.io/subnets/5)

## How the Mechanism Works

Each miner hosts a lightweight HTTP server that exposes a `/info` endpoint. When queried, this
endpoint returns a pointer to the miner's solution repository — an owner, repository name, branch,
and commit hash.

Validators collect these pointers and submit **jobs** to a managed **Sandbox Runner** — a cluster of
H200 GPUs operated by Manifold. The Sandbox Runner:

1. Clones the miner's repository and builds a Docker image from its `Dockerfile`.
2. Runs a **prep phase** (internet-accessible) so the solver can download model weights or other
   assets.
3. Runs an **inference phase** with network access fully disabled, passing a set of held-out
   ARC-AGI-2 problems.
4. Computes `exact_match_rate` — the fraction of problems the solver answers correctly — and returns
   that score to the validator.

Validators aggregate scores and assign on-chain weights. The reward distribution uses **exponential
decay** (factor 0.8 per rank): only the **top 5** miners that exceed a 20% accuracy floor receive
any emission. If no miner clears the floor in a given round, the entire emission for that round is
burned rather than distributed. Miners with identical repo + branch + commit combinations receive a
cached score rather than a redundant evaluation, and a daily submission limit (default: one per
miner per day) prevents gaming through rapid re-submissions. Weights feed into [[Yuma Consensus]] to
determine per-epoch emissions.

References: [manifold-inc/hone (GitHub)](https://github.com/manifold-inc/hone)

## Participating as a Miner

Miners must run an HTTP server (default port 8091) that returns their solution repository pointer at
`GET /info`. The repository must contain a `Dockerfile` that the Sandbox Runner can build; all model
downloads and setup should happen in the prep phase (before network isolation begins).

General setup flow:

1. Register a hotkey on SN5 with `btcli subnet register` for `netuid 5`.
2. Clone the official repository and set up the miner service from its `miner` directory.
3. Configure `miner/.env` with wallet credentials, chain endpoint, and solution repository details.
4. Start the miner services with Docker Compose.
5. Announce the miner's IP address and port on-chain so validators can discover the `/info`
   endpoint.

Solution repositories can use any language or framework that builds inside Docker. The Sandbox
Runner optionally starts a **vLLM sidecar** alongside the inference container for transformer-based
solvers that need GPU-accelerated model inference.

References: [manifold-inc/hone — Miner Setup](https://github.com/manifold-inc/hone#miner-setup),
[miner-solution-example (GitHub)](https://github.com/manifold-inc/hone/tree/main/miner-solution-example)

## Participating as a Validator

Validators do not need a GPU. Their role is to query each registered miner's `/info` endpoint,
forward the solution pointers to the Sandbox Runner API, collect the returned `exact_match_rate`
scores, and set on-chain weights accordingly.

Validator requirements (from the official README):

- 4+ CPU cores
- 8 GB+ RAM
- 20 GB disk
- A PostgreSQL database (provided via Docker Compose)
- Reliable network connection

Validator setup follows the official repository's `validator` workflow: clone the repository, create
`validator/.env` with chain, wallet, and database credentials, and start the validator stack with
Docker Compose.

The validator service polls registered miners, submits jobs to the Sandbox Runner, stores results in
PostgreSQL, and pushes weights to the chain. Because the Sandbox Runner handles all GPU work,
validators have a low hardware bar.

[manifold-inc/hone — Validator Setup](https://github.com/manifold-inc/hone#validator-setup)

## Sandbox Evaluation Context

The [manifold-inc/hone README](https://raw.githubusercontent.com/manifold-inc/hone/main/README.md)
describes Hone as evaluating miner solutions inside a managed GPU sandbox rather than trusting
self-reported results. A miner publishes a solution repository, and the Sandbox Runner clones it,
builds a Docker image, runs a network-accessible prep phase for asset downloads, then runs the
scored inference phase with network access disabled.

That network-isolation step is the part worth understanding for a reader. Because
[ARC-AGI-2](https://arcprize.org/) is designed to resist pattern-memorization, disabling network
access during inference is what keeps the score a measure of the solver's own reasoning rather than
its ability to look answers up at runtime. The sandbox design is therefore central to what SN5's
score actually means, not just an operational detail.

References:
[manifold-inc/hone README](https://raw.githubusercontent.com/manifold-inc/hone/main/README.md),
[ARC-AGI-2 prize and dataset](https://arcprize.org/)

## Scoring and Reward Context

The Hone README describes scoring as an `exact_match_rate` — the fraction of held-out ARC-AGI-2
problems a solver answers correctly — and a reward distribution that uses exponential decay across
only the top-ranked miners above an accuracy floor, with the round's emission burned when no miner
clears that floor.

For a reader, this design says something specific about the subnet's intent. Rewarding only a small
top tier and burning emission when nothing clears the floor means SN5 is built to pay for
demonstrated reasoning capability rather than to spread rewards across marginal attempts. The
cached-score and daily-submission limits described in the README reinforce that focus by reducing
gains from rapid resubmission rather than genuine improvement.

References:
[manifold-inc/hone README](https://raw.githubusercontent.com/manifold-inc/hone/main/README.md),
[taostats.io/subnets/5](https://taostats.io/subnets/5)

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Hone (SN5),
that sequence changes how readers should interpret participation examples and ARC-AGI-2 evaluation
outcomes.

In localnet, miners and validators can develop and test Hone-compatible implementations in an
isolated environment. Localnet sandbox evaluation results and emission outcomes do not represent
production subnet performance.

On testnet, Hone-compatible implementations can be exercised in a shared, non-production network.
Testnet ARC-AGI-2 scores and validator assessments are separate from mainnet subnet state.

On mainnet, Hone (SN5) is the live production subnet where miners solve ARC-AGI-2 abstract reasoning
problems in a secure GPU sandbox and validators score solutions by exact-match accuracy to determine
real Bittensor emissions. The [manifold-inc/hone repository](https://github.com/manifold-inc/hone)
describes the mechanism that applies on the production network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A solution score or emission example from one environment
should not be read as representing production subnet performance in another environment.
