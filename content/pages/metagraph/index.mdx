# Metagraph

The metagraph is Bittensor's structured view of state for a selected subnet. The
[Glossary: Metagraph](https://docs.learnbittensor.org/resources/glossary#metagraph) defines it as a
data structure containing comprehensive information about a subnet's neurons and notes that it helps
calculate emissions. The [Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)
reference describes it as a snapshot of subnet state at a particular block height rather than a
timeless network summary.

## Subnet Context and netuid Selection

A metagraph belongs to a specific subnet context. The subnet metagraph documentation uses netuid 1
as an example and describes the metagraph as a snapshot of subnet state at a particular block
([The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)). A
[netuid](https://docs.learnbittensor.org/resources/glossary#netuid) selects which subnet market the
snapshot represents; a metagraph for one netuid does not describe neuron state across every subnet
at once.

Subnet comparisons should use separate metagraph readings per netuid rather than mixing fields from
different subnet snapshots. Each netuid keeps its own neuron list and incentive fields in metagraph
vocabulary.

## Neuron State Fields

The metagraph carries information about neurons in the selected subnet. That state includes
consensus and incentive-related values: stake-related state, weights, ranks, trust, consensus,
incentives, dividends, emissions, activity state, and bonds
([The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph),
[Understanding Neurons](https://docs.learnbittensor.org/learn/neurons)). These values describe
chain-derived subnet state, not editorial ratings.

The [Glossary: Metagraph](https://docs.learnbittensor.org/resources/glossary#metagraph) notes that
the metagraph helps calculate emissions. It carries neuron-level incentive fields that feed subnet
reward processing, making the metagraph an accounting snapshot for incentive flow rather than a
miner-quality rating by itself.

## Snapshot at a Particular Block

The [Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph) documentation describes
the metagraph as a snapshot of subnet state at a particular block height. That timing detail matters
because metagraph readings are block-specific, not timeless summaries of a subnet.

A metagraph statement should keep both the selected netuid and the block context attached. Without
that pair, the snapshot can be misread as current live state when it is only state as of one
recorded block. A [block](https://docs.learnbittensor.org/resources/glossary#block) names one point
in chain progression; a metagraph names subnet neuron state as represented at that chain-height
context for a selected netuid.

Later blocks can change neuron fields, so comparing metagraphs requires comparing like block heights
or noting that the snapshots come from different points in chain history.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For a metagraph on
a subnet such as netuid 1, that sequence changes what kind of subnet-state evidence a reader is
seeing.

In localnet, a metagraph can show whether a subnet's participant state, weights, and
incentive-related fields are being represented in a controlled environment. That is useful for
understanding the shape of subnet state, but it remains isolated development evidence.

On testnet, a metagraph can reflect subnet state in a shared non-production network. This gives
readers stronger evidence about how participants and validator signals appear together than a
private local run, while still keeping the snapshot separate from production mainnet state.

On mainnet, a metagraph belongs to live Bittensor subnet state. The
[Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph) documentation describes the
metagraph as a snapshot at a particular block, so production readings should keep both the selected
netuid and the block-height context attached.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A metagraph observed in one environment should therefore
not be read as the same kind of evidence as a metagraph observed in another environment.

That distinction also protects block-height language. A local block height, a testnet block height,
and a mainnet block height can each locate a metagraph snapshot, but they locate it inside different
network histories. Local, testnet, and mainnet metagraphs can all describe subnet state, but they do
not carry the same interpretive weight.

## Register Populates the Neuron Rows

[Register](https://docs.learnbittensor.org/resources/glossary#register) describes the subnet-entry
process that grants a [UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot). After
participants enter a subnet, they appear as neuron rows inside the metagraph snapshot for the
selected netuid. A metagraph therefore exposes subnet state after entry rather than describing the
registration transaction itself.

Each [neuron](https://docs.learnbittensor.org/resources/glossary#neuron) names one registered
participant; the metagraph names the subnet-level view that can include many neurons at once. A
[subnet miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner) or
[subnet validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator) role
describes what a participant does inside the subnet, while the metagraph exposes the chain-derived
fields associated with those participants in one snapshot.

## Hotkey and Coldkey Fields Identify Each Neuron Row

The [Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph) documentation lists
`hotkeys` and `coldkeys` among metagraph fields for neurons on a selected subnet. On a subnet such
as netuid 1, each row ties a UID to the operational
[hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey) and owning
[coldkey](https://docs.learnbittensor.org/resources/glossary#coldkey) addresses recorded at the
snapshot block.

Those identity fields are chain-derived state, not wallet guidance. A metagraph reading shows which
keys were associated with each neuron position at that block height rather than ownership after a
later [hotkey swap](https://docs.learnbittensor.org/keys/coldkey-hotkey-security) or
[coldkey swap](https://docs.learnbittensor.org/keys/coldkey-swap).

## Axon Info Records Miner Endpoint Material

The metagraph reference describes `axons` and per-neuron `axon_info` as network connection details
for axon servers. An [axon](https://docs.learnbittensor.org/resources/glossary#axon) represents a
server run by a registered miner that can answer validator requests.

On netuid 1, axon fields sit beside incentive numbers inside the same snapshot. They name how a
miner endpoint was registered at the recorded block, separate from
[rank](https://docs.learnbittensor.org/resources/glossary#rank) or
[emission](https://docs.learnbittensor.org/resources/glossary#emission) values.

## Active UID and Activity Fields

An [active UID](https://docs.learnbittensor.org/resources/glossary#active-uid) describes activity
status for one participant slot. The metagraph exposes activity-related values for neurons across
the selected subnet at once
([The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)). Active status in a
snapshot is block-specific: a slot marked active at one height may differ at a later block.

## Validator Permit Appears as a Metagraph Flag

The subnet metagraph table includes `validator_permit` as a boolean array indicating whether each
neuron has validator permits to set weights and participate in consensus. That flag connects
metagraph vocabulary to
[validator permit](https://docs.learnbittensor.org/resources/glossary#validator-permit) rules
without replacing the permit article itself.

Reading permit status from a metagraph snapshot requires the netuid and block context. The boolean
field describes chain-recorded permit state at one height, not a live guarantee for later blocks.

## Multiple Mechanisms Add Per-Mechanism Fields

Subnets with multiple incentive mechanisms can carry mechanism-specific incentive state. The
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)
documentation describes additional metagraph fields for per-mechanism weights and incentives when a
subnet runs more than one evaluation path
([Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)).

Some weight and incentive values should be read in the context of a particular mechanism, not as one
blended score across every task the subnet supports. The metagraph remains a subnet state view
rather than a scoring rule.

## Yuma Consensus Writes Incentive Fields Into the Snapshot

[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) aggregates validator weight
signals within a subnet into miner incentives and validator dividends, applying consensus clipping,
bonding, and emission calculation at epoch boundaries. The metagraph is where those subnet-level
results appear as structured neuron fields such as weights, ranks, trust, consensus, incentives,
dividends, and emissions at the snapshot block.

For readers, Yuma Consensus names the on-chain process that produces the incentive numbers a
metagraph exposes. The metagraph does not replace Yuma documentation: it records the outcome of that
process for a selected netuid at one block height. Comparing metagraph incentive fields across
blocks tracks how consensus output changed; comparing them without block context can misread stale
state as current.

## Reader Boundary

This page defines the term at a high level. It does not report current subnet membership, current
stake, current emissions, wallet ownership, or live identity fields. Those values change with chain
state and should be checked for the relevant netuid when current data is needed.

## Relationship to UID Slot

Metagraph names the subnet state snapshot for a selected netuid, including participants and their
subnet-level values. UID slot names one participant position inside that snapshot
([Glossary: Metagraph](https://docs.learnbittensor.org/resources/glossary#metagraph),
[Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

A metagraph read therefore spans every indexed participant on the netuid, while a UID slot claim
names one position that a subnet miner or subnet validator occupies
([Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

Name metagraph when covering subnet participant state for a netuid; name UID slot when covering one
indexed position inside that state
([Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner),
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator)).

Metagraph belongs to subnet-state labels, while UID slot belongs to single-position labels inside
that state
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).
