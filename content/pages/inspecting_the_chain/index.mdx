# Inspecting the Chain with Polkadot.js

Inspecting the chain with Polkadot.js means using the Polkadot.js browser app to observe Bittensor
chain data
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

The topic belongs to public chain evidence. Polkadot.js can show what the connected chain exposes,
but the meaning of an observed value still depends on the network, block context, and Subtensor
surface being inspected.

## Inspection Scope

The inspection guide frames Polkadot.js as a way to query chain state, inspect runtime metadata,
browse blocks, inspect events, and submit extrinsics when signing authority is available
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).
Polkadot.js sits inside the broader Bittensor Tools overview alongside other interfaces, such as the
Bittensor SDK, which gives programmatic access to similar network activity rather than a
browser-based inspection view ([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools)).

That makes the tool useful for evidence gathering. It can show a storage value, constant, event, or
block from a connected chain, while the relevant Subtensor reference explains what that surface
means in the protocol ([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

The stable distinction is between observation and interpretation. The interface can reveal a chain
record; the article or reference page still has to explain why that record matters.

Observation is also different from signing or submitting an extrinsic. A public read shows what the
connected chain exposes; an extrinsic asks the runtime to process a submitted action
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

## Chain Surfaces

Storage, constants, events, and blocks are related inspection surfaces, but they are not
interchangeable. Storage names state entries, constants name configured runtime values, events name
records emitted during processing, and blocks provide ordering context
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

This separation helps prevent a single visible value from carrying more weight than it should. A
constant can explain a configured bound, a storage entry can show recorded state, and an event can
show that the runtime emitted a named record.

The surfaces can be read together when they describe the same chain activity. They should still keep
their own roles in prose, because each one answers a different evidence question.

Extrinsics belong beside those read surfaces as a separate submission surface rather than another
read view, so the cited surface should stay attached to each observation
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

## Distinction from Chain Reads

Inspecting the chain with Polkadot.js is one concrete way to perform a chain read. Chain reads are
the general category of looking at public chain state without asking Subtensor to record a change,
while inspecting the chain names a specific tool-based method of doing so
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

Both stay on the read side and neither writes state, which is why fee documentation treats reads
differently from runtime actions that can change chain state
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

- **Chain reads** — the general read-only category of viewing public chain state.
- **Inspecting the chain** — a Polkadot.js method of performing such reads.

## Distinction from Extrinsic

Inspecting the chain and an extrinsic sit on opposite sides of the read/write line. Inspecting the
chain is read-only observation of what the connected chain already exposes, while an extrinsic is a
submitted call that asks the Subtensor runtime to process an action and record a state change, so
inspection looks at chain state without altering it and an extrinsic is how that state is changed
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

- **Inspecting the chain** — read-only observation of existing chain state.
- **Extrinsic** — a submitted call that asks the runtime to record a state change.

## Distinction from Bittensor SDK

Inspecting the chain with Polkadot.js and the Bittensor SDK are two tools that reach similar chain
activity differently. Inspecting the chain uses the Polkadot.js browser app to observe storage,
constants, blocks, and events by hand, while the Bittensor SDK gives programmatic access to similar
network activity from code, so one is a browser-based inspection view and the other a scripting
interface
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Bittensor Tools](https://docs.learnbittensor.org/concepts/tools)).

- **Inspecting the chain** — manual, browser-based observation through Polkadot.js.
- **Bittensor SDK** — programmatic access to similar network activity from code.

## Distinction from Network

The connected network is part of the evidence. Bittensor documentation separates mainnet, testnet,
and localnet contexts
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)).

A Polkadot.js result should therefore be read with its connected endpoint and environment attached.
A value observed in one environment should not be treated as proof of the same state in another.

Block context also matters. The Bittensor glossary describes a block as a unit of blockchain data
with a unique block hash, and the inspection guide includes block browsing as one of the visible
chain surfaces ([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

Storage observed at one block can differ from storage observed at another block on the same network.
For that reason, a useful inspection note keeps both the network and the block context attached to
the value being discussed
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

## Distinction from Subtensor

Polkadot.js inspection connects to Bittensor's Subtensor blockchain layer rather than to a single
subnet application. The
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor) describes
Subtensor as Bittensor's layer 1 blockchain and system of record for transactions, rankings, and
incentive flow.

Subnets define subnet markets and evaluation context, while Subtensor is the shared chain layer that
records protocol activity
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

An inspected value or block therefore belongs to Subtensor chain evidence. It can support
subnet-related questions, but the inspection interface is observing the protocol record layer.

## Distinction from Yuma Consensus

Inspecting the chain with Polkadot.js means using a browser tool to read Subtensor storage,
constants, blocks, and events. It names public chain-observation vocabulary, not the on-chain step
that turns validator weight submissions into emission shares
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

Yuma Consensus is the on-chain process that converts submitted validator weights into emission
shares after evaluation completes. Polkadot.js can display weight-related state at a block, but Yuma
Consensus names the settlement mechanism itself
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers should keep inspection vocabulary separate from the settlement step. Inspecting the chain
answers how to read live Subtensor state; Yuma Consensus answers how validator weights become
emission shares in a subnet
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Inspecting the chain** — read-only observation of Subtensor state through Polkadot.js.
- **Yuma Consensus** — on-chain weight aggregation that settles emissions.

## Reader Boundary

Inspecting the chain is strongest when the observation is narrow: which network, which block or
timeframe, which chain surface, and which value or record was visible. Broader conclusions need the
corresponding protocol reference and surrounding context
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Use Polkadot.js evidence to support claims about what the connected chain exposed.

## Development Stage Context

Bittensor documentation separates localnet, testnet, and mainnet development contexts
([Introduction to Bittensor: Subnet development](https://docs.learnbittensor.org/learn/introduction#subnet-development),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

An inspected value belongs to the network and block context where it was observed.
