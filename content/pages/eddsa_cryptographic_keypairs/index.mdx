# EdDSA Cryptographic Keypairs

EdDSA cryptographic keypairs refer to the public and private key pairs generated with EdDSA, the
signature algorithm named in Bittensor wallet-key terminology. The official
[Glossary](https://docs.learnbittensor.org/resources/glossary#eddsa-cryptographic-keypairs)
describes EdDSA as the algorithm used to generate public and private key pairs for Bittensor
coldkeys and hotkeys.

## Keypair Context

A keypair has two sides: a public side that can be shared for identification or verification, and a
private side that must remain protected. The
[Public Key glossary](https://docs.learnbittensor.org/resources/glossary#public-key) describes the
public key as shareable material used for verification, encryption, and account reference. The
[Private Key glossary](https://docs.learnbittensor.org/resources/glossary#private-key) describes the
private key as the protected component used to authorize activity.

## Wallet-Key Context

In Bittensor, EdDSA keypairs sit under the wallet-key vocabulary rather than naming a separate
network role or asset. The glossary connects EdDSA key generation with coldkeys and hotkeys, while
separate articles can explain those wallet-key roles in more detail.

## Verification and Authorization Context

An EdDSA keypair gives Bittensor wallet terminology its public and private sides. The
[Public Key glossary](https://docs.learnbittensor.org/resources/glossary#public-key) describes a
public key as shareable material used for verification, encryption, and account reference. The
[Private Key glossary](https://docs.learnbittensor.org/resources/glossary#private-key) describes a
private key as protected material used to authorize activity. The keypair is the cryptographic
relationship between those two sides.

That relationship is different from a wallet role. A coldkey and a hotkey can both be based on
cryptographic keypairs, but they carry different meanings in Bittensor's wallet model. The coldkey
is the security-critical ownership key, while the hotkey is the operational key used by miners,
validators, and subnet participants. EdDSA names the signing-key mechanism, not the account's role
or level of authority.

This distinction helps keep address, wallet, and key language separate. A public key can identify or
verify a participant-facing account reference, while a private key is the secret needed to sign.
Wallet labels, addresses, coldkeys, and hotkeys are built around that key material, but they are not
all synonyms for the same thing. The keypair is the cryptographic foundation; the wallet role
explains what that key controls in Bittensor.

For readers, an EdDSA keypair claim is therefore about cryptographic capability rather than a
statement about custody or permissions. It does not by itself tell readers whether key material is
local, hardware-backed, or attached to a particular Bittensor role. Safe storage, wallet placement,
and coldkey or hotkey authority are separate questions handled by wallet documentation. The
algorithm explains how public and private keys pair together; the Bittensor key model explains how
those keys are used.

## Reading the Term

This article is a concept boundary. It does not describe how to create, store, recover, rotate, or
use keys. For Taopedia readers, the useful point is that EdDSA names the cryptographic method behind
the public/private keypair terminology used in Bittensor.
