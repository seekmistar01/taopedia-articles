# Dendrite

A dendrite is the client-side component that sends requests to axon endpoints in Bittensor subnet
communication ([Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite),
[Dendrite reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/dendrite/)).

The term identifies the request side of the axon-dendrite path. It is separate from the subnet role,
subnet task, scoring rule, and chain record that may surround a particular exchange.

## Client-Side Role

Bittensor documentation describes Dendrite as a network client for sending requests to axon
endpoints and processing responses
([Dendrite reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/dendrite/),
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite)).

That makes dendrite a direction and component term. It names the side that contacts an axon, rather
than the full validator program, the task being evaluated, or the economic rule that may later use
the returned information.

## Axon Pairing

Dendrite and axon describe opposite sides of a subnet communication path. Axon is the server-side
endpoint that receives incoming synapse objects, while dendrite is the client-side component that
sends requests to those endpoints
([Glossary: Axon](https://docs.learnbittensor.org/resources/glossary#axon),
[Axon reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/)).

The distinction is useful because both terms can appear near validator and miner behavior. Dendrite
marks the request side; axon marks the receiving endpoint. Neither term, by itself, explains the
subnet's task design or reward calculation.

## Synapse Messages

Subnet communication uses synapse objects for structured information exchange between validators and
miners ([Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[Synapse reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/synapse/)).

Dendrite belongs to the path that sends a synapse toward an axon endpoint and handles the response.
The synapse carries task-specific content, while dendrite describes the client-side route used for
that exchange.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For dendrite on a
subnet such as netuid 1, that sequence changes how readers should interpret request-and-response
examples.

In localnet, a dendrite exchange can test client wiring in isolation on a subnet such as netuid 1.
Localnet examples show communication shape but do not represent live public endpoint behavior.

On testnet, dendrite traffic can be observed in a shared non-production network. Testnet
validator-to-miner communication is separate from mainnet endpoint behavior even when the same
netuid number appears in both environments.

On mainnet, dendrite behavior concerns production validator-to-miner communication on the active
network ([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

An example from one environment should not be treated as proof about endpoint behavior in another.

## Dendrite Processes Axon Responses

The
[Dendrite reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/dendrite/)
describes Dendrite as a network client that sends requests to axon endpoints and processes the
responses it receives. Dendrite vocabulary therefore covers both the outbound request and the
returned response handling on the client side.

That keeps dendrite in the communication path only. Response content still belongs to the synapse
and subnet task being exchanged, not to dendrite as a scoring label.

## Miners Can Also Initiate Dendrite Requests

Dendrite use is not limited to the validator side of an exchange. The glossary describes dendrite as
a client instance used by both subnet validators and subnet miners to transmit information to axons
on miners and validators
([Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite)).

That framing keeps the term role-neutral. Dendrite names the client-side communication role
regardless of whether a validator or a miner happens to be initiating the request, while axon names
the receiving side of that same exchange.

## Synapse Objects Carry the Signed Payload Dendrites Send

[Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse) documentation
describes the structured message object passed through dendrite-to-axon communication. Dendrite
vocabulary names the client that initiates the request; synapse vocabulary names the payload shape
being signed and verified
([Glossary: Axon](https://docs.learnbittensor.org/resources/glossary#axon)).

For readers, replay and freshness checks apply to the signed synapse traffic dendrites emit, while
dendrite itself names only the sending component in that exchange
([Understanding Neurons](https://docs.learnbittensor.org/learn/neurons#neuron-to-neuron-communication)).

## Dendrite Traffic Stays Off-Chain Unless an Extrinsic Is Submitted

Neuron communication through dendrites and axons is separate from Subtensor extrinsic submission. A
dendrite request does not, by itself, mutate on-chain state; extrinsics are the documented path for
chain-recorded actions such as registration or staking
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

For readers, dendrite belongs to subnet task traffic vocabulary, while extrinsics belong to chain
mutation vocabulary on Subtensor
([Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor)).

## Distinction from Axon

Dendrite names the client-side component that sends requests to axon endpoints. Axon names the
server-side endpoint that receives those requests in the axon-dendrite communication path
([Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite),
[Glossary: Axon](https://docs.learnbittensor.org/resources/glossary#axon),
[Understanding Neurons](https://docs.learnbittensor.org/learn/neurons#neuron-to-neuron-communication)).

Dendrite initiates the request path; axon receives the request at the endpoint. They describe
opposite sides of the same communication model rather than interchangeable role labels
([Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse)).

- **Dendrite** — the client-side component that sends requests to axon endpoints.
- **Axon** — the server-side endpoint that receives those requests.

## Distinction from Synapse

Synapse names the structured message object exchanged in subnet communication. Dendrite names the
client-side component that sends and receives those objects during the exchange
([Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite)).

Synapse is the message-object term; dendrite is the communication-component term. One names what is
exchanged and the other names which part of the client stack carries the exchange
([Glossary: Axon](https://docs.learnbittensor.org/resources/glossary#axon)).

- **Synapse** — the structured message object passed between validators and miners.
- **Dendrite** — the client component that sends and receives those messages.

## Distinction from Neuron

A neuron and a dendrite name a participant and a client it uses. A neuron is the subnet participant
identified by a UID, while a dendrite is the client-side component a neuron uses to send requests to
axon endpoints, so the dendrite is something a neuron runs rather than the participant itself
([Glossary: Neuron](https://docs.learnbittensor.org/resources/glossary#neuron),
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite),
[Understanding Neurons](https://docs.learnbittensor.org/learn/neurons#neuron-to-neuron-communication)).

- **Neuron** — the UID-identified subnet participant.
- **Dendrite** — the client-side component a neuron uses to send requests to axon endpoints.

## Distinction from Subnet Validator

Subnet validator names the evaluation role on a subnet. Dendrite names the client-side component a
validator commonly uses to transmit requests to miner axon endpoints
([Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator),
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite),
[Understanding Neurons](https://docs.learnbittensor.org/learn/neurons#neuron-to-neuron-communication)).

Role vocabulary describes who evaluates; dendrite vocabulary describes the client used to initiate
the request side of an exchange.

## Distinction from Subnet Miner

Subnet miner names the work-producing role whose axon endpoints receive validator requests. Dendrite
names the client-side component validators use to reach those endpoints during the exchange
([Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner),
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite),
[Understanding Neurons](https://docs.learnbittensor.org/learn/neurons#neuron-to-neuron-communication)).

Role vocabulary describes who receives evaluation traffic on the miner side; dendrite vocabulary
describes the client used to send that traffic.

## Distinction from Subnet Protocol

Subnet protocol names the rule set for how validators and miners exchange work inside a subnet.
Dendrite names the client-side component used to send and receive synapse objects within that
exchange
([Glossary: Subnet Protocol](https://docs.learnbittensor.org/resources/glossary#subnet-protocol),
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Exchange-rule vocabulary and client-component vocabulary answer different questions on the same
miner-validator path.

## Distinction from Incentive Mechanism

An incentive mechanism defines how miner work is evaluated and how emissions are assigned inside a
subnet. Dendrite is the client-side component that sends and receives synapse objects during that
exchange
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite)).

Dendrite carries request and response traffic between neurons. The incentive mechanism explains what
that traffic means for scoring and emissions on the subnet.

- **Dendrite** — the client component that sends and receives synapse objects.
- **Incentive mechanism** — the subnet rule set that evaluates work and assigns emissions.

## Distinction from Validator

Dendrite language often appears near subnet validators because validators can use client-side
communication to query miner axons. The validator and miner roles remain separate role concepts
([Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator),
[Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner)).

A dendrite can participate in validator-side communication, but it does not define validator
judgment, miner work, or the incentive mechanism. Those meanings come from the subnet task and
evaluation design.

## Distinction from Yuma Consensus

A dendrite is the client-side interface validators use to query miners over the network. It names
the request path for subnet work exchange, not the on-chain tempo pass that turns submitted weight
signals into emission shares
([Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite),
[Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse)).

Yuma Consensus is the on-chain tempo process that aggregates submitted validator weights through
clipping and bonding before miner incentives and validator dividends are credited on a selected
netuid ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission: Distribution](https://docs.learnbittensor.org/learn/emissions#distribution)).

Dendrite queries and synapse payloads belong to off-chain subnet exchange. Validators may use those
responses when forming weight signals, but weight setting records scores on chain separately from
the dendrite call itself
([Glossary: Weight Setting](https://docs.learnbittensor.org/resources/glossary#weight-setting),
[Understanding Neurons](https://docs.learnbittensor.org/learn/neurons#neuron-to-neuron-communication)).

Off-chain validation systems describe subnet-specific judgment before weight signals exist. Dendrite
vocabulary names the networking client that requests miner responses; Yuma names how submitted
weights are settled after that evaluation path completes
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Axon endpoints receive the synapse objects dendrite clients send. That miner-listener path supplies
evaluation inputs upstream of consensus; it does not replace clipping, bonding, or dividend
allocation inside Yuma ([Glossary: Axon](https://docs.learnbittensor.org/resources/glossary#axon),
[Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions)).

Readers should not read successful dendrite queries as proof of final rank or payout amounts.
Networking responses inform off-chain evaluation; role shares still depend on subnet emission
context and the Yuma pass on the relevant netuid
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

- **Dendrite** — validator client interface for miner queries over the network.
- **Yuma Consensus** — tempo pass that turns submitted weights into emission shares.

## Reader Boundary

Dendrite does not name a wallet action, a persistent chain record, a validator score, or a guarantee
that an endpoint will respond in a particular way
([Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

Dendrite identifies the client-side request component, while axon, synapse, subnet task, and
incentive mechanism references describe neighboring parts of the system.
