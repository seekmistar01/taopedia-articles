# Protocol Alpha

Protocol alpha is subnet alpha that the Bittensor protocol itself accumulates, rather than any
individual staker. The official
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)
describes it as alpha tokens the protocol gathers as a result of chain-side TAO buys during reserve
injection, cached per subnet instead of being immediately recycled.

## Where Protocol Alpha Comes From

Protocol alpha is a byproduct of how the network feeds subnet pools. The
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
documentation describes how the coinbase mechanism injects TAO and alpha into each subnet's
liquidity pool every block. The glossary explains that the protocol's own chain-side TAO buys during
that injection leave alpha attributed to the protocol rather than to participants.

For readers, protocol alpha is best understood as alpha that belongs to the system itself, set aside
per subnet rather than handed out as participant rewards.

References:
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha),
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)

## Network and Lifecycle Context

Protocol alpha examples depend on the subnet and network environment being discussed. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, while the
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)
defines protocol alpha as alpha accumulated by the protocol during reserve injection.

In localnet context, a protocol-alpha example mainly explains the accounting idea: some alpha is
held by the protocol rather than by ordinary stakers. It does not imply that the same protocol-held
balance exists on a public subnet.

In testnet context, the example belongs to a shared testing environment. It can help show the
concept outside a private local run, but it remains separate from production mainnet subnet
settlement.

In mainnet context, protocol alpha refers to live subnet tokenomics. The selected subnet, current
protocol-held alpha balance, reserve-injection history, and dissolution context all affect how a
concrete example should be read.

This network context keeps protocol alpha from being mistaken for a user balance or yield source. It
names alpha held by the protocol itself; ordinary staker alpha and validator stake are separate
participant-side concepts.

The lifecycle boundary matters too. Protocol alpha is most important when a subnet dissolves,
because the glossary describes it as participating in pro-rata settlement. During ordinary subnet
operation, it is better read as a cached protocol-side balance than as a claimable participant
position.

When an example includes a concrete protocol-alpha amount, the useful questions are which subnet is
in scope, which network the value comes from, and whether the example is about reserve injection or
dissolution settlement. Those details decide whether the example is teaching the concept or
describing live subnet state.

## How It Differs from Staker Alpha

Protocol alpha sits apart from the alpha that stakers hold. The
[Glossary: Stake](https://docs.learnbittensor.org/resources/glossary#stake) describes participant
stake as tokens delegated to a validator in a subnet, whereas protocol alpha is not tied to any
staker position. The glossary notes that protocol alpha is cached for the subnet instead of being
recycled the way other balances are.

For readers, the distinction is ownership: staker alpha represents a participant's position in a
subnet, while protocol alpha represents value the protocol retains on the subnet's behalf.

References:
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha),
[Glossary: Stake](https://docs.learnbittensor.org/resources/glossary#stake)

## What Happens at Subnet Dissolution

Protocol alpha matters most when a subnet ends. The glossary explains that when a subnet is
dissolved, the protocol's alpha takes part in the pro-rata TAO settlement, reducing staker payouts
proportionally, while the protocol's corresponding TAO share is returned to the chain. The
[Subnet Deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration) documentation
describes the broader lifecycle in which a subnet can be removed.

For readers, protocol alpha is part of the settlement math at dissolution rather than a balance any
participant can withdraw during normal operation.

References:
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha),
[Subnet Deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration)

## Relationship to Coinbase

Protocol alpha and coinbase are related through reserve injection. The coinbase documentation
describes a per-block mechanism that injects TAO and alpha into subnet liquidity pools, while the
protocol alpha glossary entry describes alpha accumulated by the protocol as a result of chain-side
TAO buys during reserve injection.

For readers, coinbase names the recurring process that feeds subnet liquidity. Protocol alpha names
one protocol-held result that can accumulate from that process, separate from participant rewards or
participant stake.

References:
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase),
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)

## Relationship to Alpha Reserve

Protocol alpha and alpha reserve both belong to subnet alpha accounting, but they describe different
locations. The
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) documentation
describes an alpha reserve as the alpha side of a subnet's liquidity pool. Protocol alpha, by
contrast, is alpha attributed to the protocol itself and cached per subnet.

For readers, alpha reserve answers where pool-held subnet alpha sits in the Dynamic TAO market.
Protocol alpha answers which portion of subnet alpha is retained by the protocol rather than held by
ordinary participants.

References: [Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)

## Relationship to Alpha Outstanding

Protocol alpha and alpha outstanding separate protocol-held context from participant-held context.
The subnet overview describes alpha outstanding as subnet alpha held by participants and used around
stake, while the protocol alpha glossary entry describes alpha gathered by the protocol during
reserve injection.

For readers, alpha outstanding belongs to the participant side of subnet tokenomics. Protocol alpha
belongs to the protocol side, which is why it becomes important when a subnet dissolves rather than
when a participant reads a normal stake position.

References: [Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)

## Relationship to Stake

Protocol alpha should not be confused with stake. The
[Glossary: Stake](https://docs.learnbittensor.org/resources/glossary#stake) describes stake as
tokens delegated to a validator in a subnet. Protocol alpha is not delegated by a participant and
does not represent a validator or staker position.

For readers, stake is participant-controlled exposure to a subnet's validator context. Protocol
alpha is protocol-controlled subnet alpha that can affect dissolution settlement without being a
claimable user balance during ordinary subnet operation.

References: [Glossary: Stake](https://docs.learnbittensor.org/resources/glossary#stake),
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)

## Relationship to Subnet Deregistration

Protocol alpha and subnet deregistration meet at the end of a subnet lifecycle. Subnet
deregistration explains how a subnet can be removed from the network, while the protocol alpha
glossary entry explains how protocol-held alpha participates in the pro-rata settlement when a
subnet is dissolved.

For readers, subnet deregistration names the lifecycle event. Protocol alpha names one tokenomics
component that matters during settlement, especially because it is handled separately from
participant-held alpha.

References: [Subnet Deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration),
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)

## Relationship to Recycling

Protocol alpha and recycling are related but distinct parts of Bittensor supply accounting. The
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha) entry
describes protocol alpha as alpha tokens accumulated by the Bittensor protocol from chain-side TAO
buys during reserve injection, and notes that this alpha is cached per subnet in the
`SubnetProtocolAlpha` storage item. The
[Recycling and burning](https://docs.learnbittensor.org/resources/glossary#recycling-and-burning)
glossary entry explains the supply-accounting outcome for tokens that are removed from circulation,
and the same source notes that protocol alpha was previously recycled but is now cached rather than
immediately burned.

For readers, protocol alpha names the alpha the protocol holds on its own behalf, while recycling
names the supply-accounting outcome for tokens removed from circulation. The two are not
interchangeable: protocol alpha is now cached for pro-rata settlement at subnet dissolution rather
than immediately recycled, so it sits in a holding pattern rather than being subtracted from
issuance on receipt.

References:
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha),
[Recycling and burning](https://docs.learnbittensor.org/resources/glossary#recycling-and-burning)

## Reader Boundary

Protocol alpha should not be read as a balance any user can claim, a live figure, or a yield source.
Its size and settlement behavior depend on subnet activity and live chain state. This article
describes the concept of subnet alpha held by the protocol itself and settled when a subnet
dissolves.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For protocol
alpha, that sequence changes how readers should interpret evidence about protocol-held alpha
balances and reserve injection behavior.

In localnet, protocol alpha mechanics can be observed in an isolated environment. Localnet protocol
alpha balances and coinbase reserve injection reflect local subnet configuration and do not
represent production pool state.

On testnet, protocol alpha behavior can be observed in a shared, non-production network. Testnet
protocol alpha balances are separate from mainnet subnet pool reserves.

On mainnet, protocol alpha is an actual protocol-held balance that accumulates from live coinbase
reserve injection and settles when a subnet deregisters. The
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)
describes the per-subnet protocol-owned alpha balance that applies on the production network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A protocol alpha example from one environment should not
be read as representing pool reserve state in another environment.
