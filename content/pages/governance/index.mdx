# Bittensor Governance

Bittensor governance is the proposal and approval path for privileged network changes. The process
separates proposal creation by the Triumvirate from review and approval by the Senate
([Governance Overview](https://docs.learnbittensor.org/governance),
[Senate](https://docs.learnbittensor.org/governance/senate)).

The structure is useful because it keeps proposal origin and proposal approval distinct. A proposal
can be brought forward for review without being the same thing as an approved change.

That makes Bittensor governance a process term. It should be read separately from the individual
body names, the proposal identifier, and the final status of a proposed change.

## Proposal Review Context

A governance proposal is a review item for a privileged change. The
[Governance Overview](https://docs.learnbittensor.org/governance) frames governance around proposal
creation, voting, and approval rather than ordinary subnet operation.

That scope keeps governance vocabulary separate from day-to-day subnet incentives. Governance
concerns privileged network changes; subnet incentive mechanisms concern how subnet work is
evaluated and rewarded
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

Proposal review also keeps status language separate from process language. A submitted proposal, an
approved proposal, and an applied network change are related, but they are not the same governance
state.

## Triumvirate Role

The Triumvirate is the proposal-origin side of the governance process. It creates proposals that can
then be reviewed by the Senate ([Governance Overview](https://docs.learnbittensor.org/governance)).

That means the Triumvirate should not be collapsed into the approval body. Its role is proposal
initiation, not final approval
([Glossary: Triumvirate](https://docs.learnbittensor.org/resources/glossary#triumvirate)).

## Senate Role

The Senate is the approval body in the governance process. The
[Senate](https://docs.learnbittensor.org/governance/senate) reference describes senators as the
voting group that reviews proposals.

This makes Senate approval a governance threshold. Proposal creation starts the review path, while
Senate voting determines whether the proposal clears that approval step.

Senate language should therefore stay attached to review and voting. It should not be used as a
replacement for the proposal itself or for the wider bicameral structure
([Glossary: Bicameral Legislature](https://docs.learnbittensor.org/resources/glossary#bicameral-legislature)).

## Visibility and Voting

Governance records can be visible without making every reader a voter. The
[Senate](https://docs.learnbittensor.org/governance/senate) documentation separates public proposal
visibility from voting, which is restricted to senators.

That distinction prevents two different access ideas from being merged. Public visibility supports
inspection; voting authority belongs to the Senate role.

## Privileged Change Scope

Governance applies to privileged network changes such as parameter changes, upgrades, and related
administrative actions ([Governance Overview](https://docs.learnbittensor.org/governance)).

This is a narrower scope than general Bittensor activity. Governance is not a validator-scoring
mechanism, a staking flow, or an ordinary subnet work loop; it is the review path for elevated
network changes.

The security context is why Sudo belongs nearby. Sudo names the older privileged-authority concept,
while governance names the proposal-and-review process that replaced direct single-key
administration ([Glossary: Sudo](https://docs.learnbittensor.org/resources/glossary#sudo),
[Governance Overview: security](https://docs.learnbittensor.org/governance#security)).

## Reader Boundary

Bittensor governance should be read as proposal-and-approval vocabulary. Triumvirate proposal
creation and Senate approval are distinct parts of the process
([Governance Overview](https://docs.learnbittensor.org/governance),
[Senate](https://docs.learnbittensor.org/governance/senate)).

It does not provide voting instructions, proposal strategy, operational steps, or point-in-time
governance results. Exact proposal outcomes belong to the specific governance record being
consulted.

## Proposal Flow Boundary

Bittensor governance claims should keep the proposal flow visible. The governance overview separates
Triumvirate proposal creation, Senate approval or rejection, and later finalization or application
steps ([Governance Overview](https://docs.learnbittensor.org/governance),
[Senate](https://docs.learnbittensor.org/governance/senate)).

Approval by one body is not the same as a completed governance action. Precise governance language
should identify whether it is discussing proposal creation, Senate review, voting result,
finalization, or application.

## Development Stage Context

Bittensor documentation separates localnet, testnet, and mainnet development contexts
([Introduction to Bittensor: subnet development](https://docs.learnbittensor.org/learn/introduction#subnet-development),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)). For Governance,
that sequence changes how proposal examples, voting observations, and applied-change comparisons
should be interpreted.

In localnet, Governance examples can be exercised in an isolated environment. Localnet observations
reflect local chain state and local configuration rather than production Bittensor behavior.

On testnet, Governance behavior can be observed in a shared, non-production network. Testnet
examples are useful for checking interactions under shared chain conditions while staying separate
from mainnet state.

On mainnet, Governance describes proposal review and approval behavior on the production Bittensor
network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. An example or outcome from one environment should not be
read as evidence of production behavior in another environment.
