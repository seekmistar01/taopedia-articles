# Non-fast Blocks

Non-fast blocks name ordinary block timing in contrast with fast blocks. Official Bittensor
documentation presents the term alongside fast blocks in local-development context.

References:
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)

## Local Context

The term helps readers distinguish two local timing concepts. Fast blocks refer to accelerated local
block production. Non-fast blocks refer to the non-accelerated side of that same local distinction.

References:
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks),
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)

## Twelve-Second Interval Context

The [Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)
describes non-fast blocks as a development-only configuration that adheres to Subtensor's default
12-second block interval, simulating production timing for features like delayed subnet activation.
Readers should treat this as a local timing setting rather than a live mainnet behavior label.

References:
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks),
[Run a Local Bittensor Blockchain Instance](https://docs.learnbittensor.org/local-build/deploy)

## Two Hundred Fifty Millisecond Contrast

Fast blocks and non-fast blocks name opposite sides of the same local timing choice. The
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks) describes
accelerated local production at 250 millisecond intervals, while the
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)
keeps Subtensor's default twelve-second block interval instead.

Choosing non-fast blocks therefore means accepting production-like spacing between locally produced
blocks rather than the immediate execution pace fast blocks provide. Both options remain
development-only settings on a local chain.

References:
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)

## Block-Triggered Process Cadence

Many Subtensor processes advance once per block. Coinbase, for example, runs as a per-block protocol
operation that accumulates pending emissions and checks epoch boundaries. When non-fast blocks
follow a twelve-second interval, those processes advance on the same cadence production networks use
rather than on fast blocks' compressed schedule.

That pacing is one reason developers choose non-fast blocks when they want local observations of
epoch timing, tempo progression, or other block-count-based mechanisms to match production-like
wall-clock behavior.

References:
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase),
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)

## Tempo Wall-Clock Expectations

Subnet tempos are measured in blocks rather than seconds, but block interval still affects how long
a tempo takes in wall-clock time. With non-fast blocks keeping a twelve-second interval, a tempo of
360 blocks spans roughly production-like elapsed time on a local chain instead of the much shorter
elapsed time fast blocks would produce for the same block count.

Readers comparing local tempo or epoch results to public network expectations should keep that
interval attached. Non-fast blocks help local block counts translate into familiar timing, but they
still describe local development rather than live mainnet behavior.

References:
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)

## Relationship to Localnet

Non-fast blocks and localnet are related but name different parts of local development context. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference includes
localnet as a user-controlled development environment, while the
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)
describes ordinary local block timing inside that setting.

For readers, localnet is the environment label, while non-fast blocks are a timing configuration
used when simulating production-like block intervals during local development.

References: [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)

## Relationship to Local Blockchain

Non-fast blocks and local blockchain are related but name different parts of local development
context. The
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)
describes a private chain used for development and testing, while the
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)
describes ordinary local block timing that follows Subtensor's default 12-second interval.

For readers, local blockchain is the isolated development chain, while non-fast blocks are a timing
setting used within that local environment. Neither term describes live mainnet block behavior on

## Reader Boundary

This page defines the term. It does not describe live public network behavior or a subnet reward
mechanism.

## Relationship to Coinbase

Non-fast blocks and coinbase are related but different parts of how block timing affects emission
execution. Non-fast blocks are a development-only configuration that follows Subtensor's default
12-second block interval to simulate production timing. Coinbase is the per-block protocol mechanism
that drives TAO emission, injects TAO into emitting subnets' pool reserves, accumulates pending
emissions, and checks epoch boundaries to trigger Yuma Consensus rounds. The
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)
describes the local timing setting, and the
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
documentation describes the per-block emission operation.

Because coinbase runs once per block, the block interval directly affects how often coinbase
advances pending emissions and how epoch boundaries fall in time. Under non-fast blocks, coinbase
executes on the same 12-second cadence as production, so a local environment can observe emission
accumulation and epoch-triggered distribution at realistic timing. Fast blocks would compress that
cadence instead. Non-fast blocks name the timing configuration; coinbase is the per-block process
whose pacing that configuration is chosen to mirror.

## Development Stage Context

Non-fast blocks is a localnet timing setting that mirrors the 12-second production-network cadence.
It exists so local development can observe emission accumulation and epoch-triggered events at
realistic timing before moving to a public network. Testnet and mainnet do not expose a fast-blocks
or non-fast-blocks configuration; their block intervals are set by the network itself, not by a
local developer option.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. Non-fast blocks belongs to localnet and should not be read
as describing block timing on testnet or mainnet.
