# VPermit

VPermit is a Bittensor validation term for the subnets where a delegate is authorized to validate.
The [Glossary: VPermit](https://docs.learnbittensor.org/resources/glossary#vpermit) defines VPermit
as a list of subnet IDs that indicate which subnets a delegate is authorized to validate on.

## Delegate Context

A delegate can validate in more than one subnet. VPermit is the delegate-level view of that
authorization, while a
[validator permit](https://docs.learnbittensor.org/resources/glossary#validator-permit) is the
subnet-level permission flag for validation rights.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For VPermit on a
subnet such as netuid 1, that sequence changes how readers should interpret evidence about delegate
validation authorization.

In localnet, VPermit assignments can be tested in an isolated environment. Localnet validation
authorization reflects local chain configuration and does not represent production delegate state.

On testnet, VPermit activity can be observed in a shared, non-production network. Testnet
authorization lists are separate from mainnet delegate validation state.

On mainnet, VPermit is the live delegate-level summary of which subnets a delegate is authorized to
validate on ([Glossary: VPermit](https://docs.learnbittensor.org/resources/glossary#vpermit)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A VPermit example from one environment should not be read
as representing delegate authorization or validation rights in another environment.

## Multiple Mechanisms Span Separate Evaluation Contexts

VPermit summarizes the subnets where a delegate is authorized to validate. The
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)
notes that validators must evaluate miners separately for each mechanism.

VPermit still names delegate-level subnet authorization. Validation rights on a subnet that runs
more than one incentive mechanism still span separate evaluation contexts inside that netuid
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

## Distinction from Validator Permit

A validator permit and a VPermit differ in scope. The glossary describes a validator permit as the
per-subnet permission flag awarded to the top neurons by stake weight inside one subnet, while a
VPermit is the delegate-level list of subnet IDs that aggregates those individual permits across
subnets
([Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit),
[Glossary: VPermit](https://docs.learnbittensor.org/resources/glossary#vpermit)).

- **Validator permit** — the validation-rights flag inside one subnet.
- **VPermit** — the delegate-level list of subnets where those permits apply.

## Distinction from Delegate

A delegate and a VPermit name an entity and one of its attributes. The glossary describes a delegate
as a validator that can receive nominated stake, while a VPermit is the list of subnet IDs that
delegate is authorized to validate on
([Glossary: Delegate](https://docs.learnbittensor.org/resources/glossary#delegate),
[Glossary: VPermit](https://docs.learnbittensor.org/resources/glossary#vpermit)).

- **Delegate** — the validator entity that can receive nominated stake.
- **VPermit** — the per-delegate list of subnets it is authorized to validate on.

## Distinction from Stake Weight

Individual validator permits are tied to stake-weight position inside each subnet. VPermit is the
delegate-level list of subnet IDs where those permits apply
([Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit),
[Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Glossary: VPermit](https://docs.learnbittensor.org/resources/glossary#vpermit)).

Stake weight ranks neurons inside one subnet. VPermit summarizes which subnets carry validation
authorization for a delegate.

- **VPermit** — the delegate-level list of authorized subnet IDs.
- **Stake weight** — influence ranking inside one subnet that can affect permit eligibility.

## Distinction from Yuma Consensus

VPermit names where a delegate is authorized to validate across subnets. Yuma Consensus aggregates
validator weights into emission shares on each subnet where validation takes place
([Glossary: VPermit](https://docs.learnbittensor.org/resources/glossary#vpermit),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

VPermit is a permission summary. Yuma Consensus is the aggregation step that turns submitted weights
into incentives and dividends.

- **VPermit** — which netuids a delegate may validate on.
- **Yuma Consensus** — the aggregation mechanism that converts weights into emission shares.

## Distinction from Coinbase

VPermit records where a delegate is authorized to validate across subnets. Coinbase is the per-block
protocol mechanism that drives TAO emission and triggers Yuma Consensus rounds at epoch boundaries
([Glossary: VPermit](https://docs.learnbittensor.org/resources/glossary#vpermit),
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)).

VPermit does not allocate emissions. Coinbase generates and routes emissions on subnets where
validation occurs.

- **VPermit** — delegate-level authorization across subnet IDs.
- **Coinbase** — the per-block emission mechanism on active subnets.

## Distinction from Child Hotkeys

VPermit is the delegate-level list of subnet IDs a delegate is authorized to validate on. A child
hotkey is a separate authorized key that validates on a parent's behalf by re-delegating stake
([Glossary: VPermit](https://docs.learnbittensor.org/resources/glossary#vpermit),
[Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys)).

- **VPermit** — which netuids a delegate may validate on.
- **Child hotkey** — which key carries out validation using routed stake.

A child hotkey is one way validation can be carried out on a parent's behalf after VPermit shows
where the delegate may validate
([Glossary: Delegate](https://docs.learnbittensor.org/resources/glossary#delegate)).

## Reader Boundary

This article defines the term at a high level. It does not list live permissions, rank validators,
describe staking or registration steps, or restate validator requirements. Current permission lists
belong to the relevant delegate, network, and tooling at the time of observation
([Glossary: VPermit](https://docs.learnbittensor.org/resources/glossary#vpermit)).
