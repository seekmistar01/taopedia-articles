# Subnet Liquidity Pools

Subnet liquidity pools are the paired-reserve context for subnet tokenomics in Bittensor. Each pool
connects a TAO reserve with the subnet's alpha-token reserve, giving the subnet a conversion surface
for staking and unstaking
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Slippage](https://docs.learnbittensor.org/learn/slippage)).

The term belongs to reserve vocabulary. It names the pool structure and reserve relationship, not a
static report of exact balances, a one-off balance snapshot, or a complete explanation of every
tokenomics process connected to the pool.

## Reserve Pair

A subnet pool has a TAO reserve and an alpha reserve. The TAO side connects the subnet to
network-level TAO, while the alpha side connects the pool to the subnet-specific token context
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Understanding Subnets: Liquidity Pools](https://docs.learnbittensor.org/subnets/understanding-subnets#liquidity-pools),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Those reserves are separate from alpha outstanding. Pool reserves describe what sits in the pool,
while alpha outstanding describes alpha outside that reserve context.

The paired-reserve view is the reason the pool needs both sides. TAO reserve names the network-token
side of the pool, alpha reserve names the subnet-token side, and subnet liquidity pool names the
structure that holds the pair together
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

A pool statement is therefore pair-aware. A TAO reserve figure or alpha reserve figure can be useful
on its own, but pool interpretation comes from reading the two reserve sides together.

Alpha outstanding sits outside that reserve view. That gives readers three nearby but distinct
tokenomics ideas: TAO reserve, alpha reserve, and alpha outside the reserve
([Emission: Alpha outstanding injection](https://docs.learnbittensor.org/learn/emissions#alpha-outstanding-injection)).

## Emissions and Protocol Alpha

Bittensor emission documentation connects subnet pools to reserve injection. Emission flow can add
TAO into subnet pool context, while the surrounding tokenomics model also separates TAO reserve,
alpha reserve, and alpha-outstanding concepts
([Emission](https://docs.learnbittensor.org/learn/emissions),
[Emission: TAO reserve injection](https://docs.learnbittensor.org/learn/emissions#tao-reserve-injection)).

Protocol alpha is related to subnet pools but not the same term. Protocol alpha names protocol-owned
alpha cached per subnet, while a subnet liquidity pool names the paired reserve structure
([Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)).

That separation is useful because both terms can appear near reserve injection. Pool vocabulary
names the reserve structure, while protocol-alpha vocabulary names a protocol-held alpha category.

Reserve injection concerns movement into subnet pool context. Flow-based emission language explains
how TAO reserve injection is compared across subnets, while broader emissions pages explain how TAO
and alpha rewards move through the network
([Emission](https://docs.learnbittensor.org/learn/emissions),
[Emission: Distribution Across Subnets](https://docs.learnbittensor.org/learn/emissions#distribution-across-subnets)).

This keeps emissions and pools connected without merging them. Emissions explain reward and reserve
movement, while the subnet liquidity pool names the paired reserve surface affected by that
movement.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For subnet
liquidity pools on a subnet such as netuid 1, that sequence changes how an example should be read,
because the surrounding network state differs at each stage.

In localnet, subnet liquidity pools can be exercised in an isolated development environment, where
the surrounding chain state reflects local configuration rather than production history.

On testnet, subnet liquidity pools can be observed in a shared, non-production network whose state
is kept separate from mainnet.

On mainnet, subnet liquidity pools apply on the live, production Bittensor network, where the
surrounding state is real and persistent.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A subnet liquidity pool example from one environment
should not be read as representing another environment.

## Distinction from Alpha Outstanding

A subnet liquidity pool and alpha outstanding name pool-held reserves and alpha outside them. The
pool names the paired TAO and alpha reserves held for conversion, while alpha outstanding names
alpha held by participants outside that reserve context
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Emission: Alpha outstanding injection](https://docs.learnbittensor.org/learn/emissions#alpha-outstanding-injection)).

- **Subnet liquidity pool** — the paired TAO and alpha reserves held for conversion.
- **Alpha outstanding** — alpha held outside the reserve context.

Emission documentation distinguishes alpha held in subnet pool reserves from alpha outstanding held
by participants outside that reserve context. Subnet liquidity pool vocabulary therefore names the
paired TAO-and-alpha reserve structure, not every alpha token associated with the subnet
([Emission: Alpha outstanding injection](https://docs.learnbittensor.org/learn/emissions#alpha-outstanding-injection),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

For readers, reserve depth and outstanding supply answer different tokenomics questions. Pool
reserves shape staking conversions and alpha price quotes, while outstanding alpha can matter for
stake and supply context without entering the reserve ratio directly
([Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)).

The same alpha reserve described above is one side of the Alpha Distribution Ratio comparison. The
[Glossary: ADR](https://docs.learnbittensor.org/resources/glossary#adr-alpha-distribution-ratio)
entry defines ADR as comparing alpha tokens held by participants against alpha tokens remaining in
the AMM pool, and states that a higher ADR means liquidation happens at a deeper discount to spot
price
([Understanding Subnets: Liquidity Pools](https://docs.learnbittensor.org/subnets/understanding-subnets#liquidity-pools)).

ADR is therefore a separate ratio from the reserve pair itself. The reserve pair names what the pool
holds; ADR names how that pool-held alpha compares to alpha held by participants.

Neuron [register](https://docs.learnbittensor.org/resources/glossary#register) fees are paid in TAO,
but glossary wording also states that alpha tokens worth the current swap value of the registration
fee are taken from the subnet's alpha liquidity pool and recycled when a hotkey registers on a
subnet. Subnet liquidity pool vocabulary therefore connects participant entry to reserve movement,
not only to staking conversions
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## Distinction from Reserve Ratio

A subnet liquidity pool and the reserve ratio name a structure and a numeric relationship. The pool
names the paired reserve structure, while the reserve ratio names how the TAO and alpha sides relate
numerically at a given observation time
([Understanding Subnets: Liquidity Pools](https://docs.learnbittensor.org/subnets/understanding-subnets#liquidity-pools),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

- **Subnet liquidity pool** — the paired reserve structure itself.
- **Reserve ratio** — how the two reserve sides compare numerically.

[Emission](https://docs.learnbittensor.org/learn/emissions) documentation describes reserve
injection that adds TAO to the TAO reserve and matched alpha to the alpha reserve. Because both
sides receive injected tokens, emission can deepen both reserves of a subnet's pool together rather
than only one side
([Emission: TAO reserve injection](https://docs.learnbittensor.org/learn/emissions#tao-reserve-injection),
[Emission: Alpha reserve injection](https://docs.learnbittensor.org/learn/emissions#alpha-reserve-injection)).

That deepening changes reserve depth, not what the pool represents: it remains the same paired
TAO-and-alpha reserve structure described above
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

Unstaking documentation reverses the staking conversion through the same paired reserves rather than
withdrawing TAO directly from a hotkey balance. Subnet liquidity pool vocabulary is therefore the
shared conversion surface for both entry and exit
([Glossary: Unstaking](https://docs.learnbittensor.org/resources/glossary#unstaking),
[Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation),
[Slippage](https://docs.learnbittensor.org/learn/slippage)).

For readers, pool prose should stay pair-aware on both directions. A reserve statement about TAO or
alpha depth describes the pool context that staking and unstaking both traverse
([Understanding Subnets: Liquidity Pools](https://docs.learnbittensor.org/subnets/understanding-subnets#liquidity-pools)).

The
[Understanding Subnets: Liquidity Pools](https://docs.learnbittensor.org/subnets/understanding-subnets#liquidity-pools)
describes the ratio between TAO and alpha held in a subnet's pool reserves. Subnet liquidity pool
vocabulary names the paired structure; reserve ratio names how the two sides relate numerically at a
given observation time.

Readers comparing pool depth should keep pair structure and ratio vocabulary separate. The pool
names what exists; reserve ratio names how the two reserve sides compare.

## Distinction from Subnet Markets

A subnet liquidity pool and a subnet market sit on the trading side and the work side of the same
subnet. A subnet liquidity pool is the paired TAO and alpha reserve that prices and trades the
subnet's alpha, while a subnet market is the work-and-evaluation setting where miners produce and
validators evaluate
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Understanding Subnets: Liquidity pools](https://docs.learnbittensor.org/subnets/understanding-subnets#liquidity-pools)).

Pool vocabulary names reserve structure and conversion behavior; subnet-market vocabulary names
where work, evaluation, and reward flow meet for a selected netuid.

- **Subnet liquidity pool** — the TAO/alpha reserve that prices and trades alpha.
- **Subnet market** — the work-and-evaluation economic setting.

## Distinction from Yuma Consensus

Subnet liquidity pools name the paired TAO and alpha reserve structure for staking conversions. Yuma
Consensus is the separate mechanism that converts validator weight submissions into miner incentives
and validator dividends each tempo
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions),
[Understanding Subnets: Liquidity pools](https://docs.learnbittensor.org/subnets/understanding-subnets#liquidity-pools)).

- **Subnet liquidity pool** — paired reserve structure for TAO and alpha conversions.
- **Yuma Consensus** — on-chain settlement that turns validator weights into emission shares. Subnet
  liquidity pools name reserve conversion structure; Yuma processes validator weights into rewards
  ([Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions)).

Subnet liquidity pools provide the reserve surface for staking and unstaking conversions. Yuma
Consensus is the on-chain process that aggregates validator weight signals within a subnet into
miner incentives and validator dividends after those participant roles operate in the subnet context
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Pool vocabulary names reserve structure and conversion behavior; Yuma Consensus names how validator
evaluations become emission outcomes on the subnet where the pool sits.

## Distinction from Constant Product AMM

The slippage documentation describes Bittensor subnet pools as constant product automated market
maker pools. Staking and unstaking therefore pass through paired reserve behavior rather than a
fixed one-sided conversion ([Slippage](https://docs.learnbittensor.org/learn/slippage)).

That is why a conversion cannot be read from one side alone. A change involving one reserve is
interpreted through the paired TAO and alpha reserve relationship.

Constant product context also explains why available pool depth matters. The reserve pair is the
conversion surface for pool actions, so the resulting amount depends on the pool state used for that
action.

This makes the pool different from an ordinary balance list. The reserve pair forms a conversion
surface, so staking, unstaking, and slippage claims need pool context rather than a one-sided token
description.

## Distinction from Staking and Unstaking

When TAO is staked into a subnet, the staking flow passes through the subnet pool reserve context
and produces alpha-denominated stake. When stake is removed, alpha-denominated stake is converted
back through the reserve system
([Staking and Delegation](https://docs.learnbittensor.org/staking-and-delegation/delegation),
[Glossary: Unstaking](https://docs.learnbittensor.org/resources/glossary#unstaking)).

Staking and unstaking stay connected without becoming identical. Staking begins from TAO and creates
subnet alpha stake, while unstaking moves in the opposite direction through the pool context.

The direction matters for interpretation. Staking starts from TAO entering subnet context; unstaking
starts from alpha-denominated stake leaving subnet context through the reserve system
([Staking and Delegation](https://docs.learnbittensor.org/staking-and-delegation/delegation)).

The pool remains the reserve surface through which the action passes; staking and delegation sources
explain the user-facing staking model
([Staking and Delegation](https://docs.learnbittensor.org/staking-and-delegation/delegation)).

## Distinction from Slippage

Slippage is the difference between a simple expected conversion and the result produced by the pool
context used for the action ([Slippage](https://docs.learnbittensor.org/learn/slippage)).

The important point is conceptual: pool reserves affect the resulting TAO or alpha amount. The
amount belongs to the pool state used for that action.

Slippage language belongs with the pool action being discussed. The pool gives the conversion
context; any observed example still needs the source for the amount being described.

Slippage is therefore adjacent to subnet liquidity pools but not identical to them. The pool is the
reserve structure; slippage is the difference created when a conversion moves through that structure
([Slippage](https://docs.learnbittensor.org/learn/slippage),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

For readers, the pool answers where the conversion happens. Slippage answers how the conversion
result can differ from a simple expectation because the reserve state changes during the action.

## Reader Boundary

Subnet liquidity pools are reserve and conversion vocabulary. They connect TAO reserves, alpha
reserves, staking, unstaking, slippage, emissions, and protocol alpha into one pool-centered concept
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Slippage](https://docs.learnbittensor.org/learn/slippage)).

The stable point is that a subnet liquidity pool names the paired TAO and alpha reserve context for
a subnet. Exact values, timing, or examples need the specific source that reports those values at
the time of observation.

## Staking Conversions Move Along the Reserve Pair

Staking documentation describes TAO entering a subnet pool and emerging as alpha-denominated stake
on the other side of the paired reserves. Unstaking reverses that path through the same pool context
rather than withdrawing raw TAO directly from a hotkey balance
([Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation),
[Understanding Subnets: Liquidity Pools](https://docs.learnbittensor.org/subnets/understanding-subnets#liquidity-pools),
[Slippage](https://docs.learnbittensor.org/learn/slippage)).

For readers, subnet liquidity pool vocabulary is therefore the conversion surface behind staking,
registration recycling, and range-based liquidity positions. Each action changes reserve balances
according to pool rules rather than as a simple one-sided transfer
([User Liquidity Positions](https://docs.learnbittensor.org/liquidity-positions),
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)).

That keeps pool prose anchored to paired-reserve behavior instead of isolated token balances
([Emission: TAO reserve injection](https://docs.learnbittensor.org/learn/emissions#tao-reserve-injection)).
