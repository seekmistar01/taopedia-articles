# Synapse

A synapse is the structured message object used in Bittensor subnet communication
([Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[Synapse reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/synapse/)).

The term belongs to communication vocabulary. It should be read separately from the subnet role,
subnet task, and incentive rule that interpret the information carried by the message.

## Message Object Context

The official glossary describes synapse objects as the main vehicle for information exchange between
subnet validators and subnet miners
([Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse)).

That framing makes synapse a message-object term. It names the object exchanged through subnet
traffic, not the whole miner role, validator role, scoring rule, or reward outcome.

## Axon and Dendrite Boundary

Synapse sits between axon and dendrite vocabulary. Axon is the server-side endpoint that receives
incoming synapse objects, while dendrite is the client-side component that sends requests to axon
endpoints and processes responses
([Axon reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/),
[Dendrite reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/dendrite/)).

For readers, dendrite contacts, axon receives, and synapse is the structured message object
exchanged through that path.

## Miner and Validator Exchange

Synapse language often appears near subnet miners and subnet validators because those roles exchange
task-related information. The miner role produces work for a subnet task, and the validator role
evaluates work under the subnet's incentive mechanism
([Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner),
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator)).

The synapse itself does not decide whether a response is useful. It carries the information that a
subnet protocol and scoring model can later interpret.

## Subnet-Specific Payloads

The synapse concept is stable across subnet communication, but the useful content inside a message
depends on the subnet task. Bittensor documentation describes subnet tasks as the work miners
perform for a subnet, and multiple-mechanism documentation notes that different mechanisms can
define different task contexts inside a subnet
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

This means a synapse should not be treated as a universal schema for every subnet. The article-level
point is the communication role; task-specific fields belong to the relevant subnet or API
reference.

## Development and Network Context

Bittensor documentation separates local development, testchain, and mainchain contexts for subnet
work, and it separately describes mainnet, testnet, and localnet network contexts
([Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

A synapse example from one environment should therefore not be read as identical evidence about
response quality, endpoint availability, or subnet scoring in another environment. The message
object can be the same kind of object while the surrounding evidence has a different scope.

## Incentive Mechanism Handoff

Synapse exchange can feed a subnet's evaluation pipeline. The incentive-mechanism documentation
places miner work, validator evaluation, and emission outcomes inside subnet-specific incentive
mechanisms
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

That does not make synapse a reward rule. Synapse names the object carrying task information before
the subnet's own evaluation logic turns responses into weights, scores, or other outcome signals.

## Reader Boundary

Synapse should be read as Bittensor's structured communication object for subnet exchanges. It is
not a miner role, a validator decision, a persistent chain record, or a complete description of any
subnet's protocol ([Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

That boundary keeps the term useful: synapse identifies what is exchanged, while surrounding subnet
documentation explains what the exchanged data means.
