# Validator Trust

Validator trust is a validator-side consensus metric. It describes how much of a validator's weight
signal remains after consensus clipping
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)).

The term belongs to Yuma Consensus because it is read after validator weights are compared with the
broader consensus signal ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Validator-Side Scope

Validator trust is scoped to the validator-side weight signal. It is not a miner-side score and
should not be read as the same concept as trust
([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

For readers, this is the main boundary: trust describes the evaluated miner side, while validator
trust describes the validator-side signal retained after filtering
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)).

## Clipping Context

Consensus filtering can reduce weights that sit outside the consensus boundary. Validator trust
describes how much of the validator's signal remains after that filtering step
([Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

That makes validator trust an alignment metric inside consensus, not a separate reputation label or
validator setting ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Weight Context

Validator weights are the signals validators provide after evaluating miners. Validator trust is
read after those weight signals have been filtered by consensus
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

The distinction is useful: validator weights name the validator evaluation signal, while validator
trust names how much of that signal remains aligned after consensus processing
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)).

Validator trust is easiest to read as retained validator signal. Validator weights express how a
validator evaluated miners, and consensus filtering determines how much of that signal remains in
the consensus-aligned result
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

That reading keeps the metric narrow. Validator trust is not a custody signal, delegation
recommendation, or general reputation label. It says how much of the validator's weight signal
survived the consensus filtering path described by Yuma Consensus
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The metric is therefore about alignment with consensus, not about wallet security, delegation
strategy, or the validator's activity outside the subnet consensus process.

Trust and validator trust are paired terms on different sides of the same consensus process. Trust
looks at the miner-side effect of clipping, while validator trust looks at the validator-side signal
that remains after clipping
([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

This pairing is useful when reading Yuma Consensus metrics together. A miner-side trust value can
help explain how clipping affected support for a miner, while validator trust can help explain how
closely a validator's weight pattern stayed with the broader consensus signal
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)).

Keeping the two terms separate prevents one common mistake: treating validator trust as a miner
performance score, or treating miner-side trust as a validator alignment score.

Validator trust comparisons are clearest inside the same subnet consensus context. The validator
set, incentive mechanism, and weight pattern can differ across subnets, so the same metric name can
carry different surrounding context from one subnet to another
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The stable concept is the retained validator-side signal after consensus filtering. The surrounding
subnet context explains what that retained signal is being compared against.

## Consensus Score Context

Consensus score is the agreement boundary around validator weights. Validator trust is a
validator-side result after that boundary has affected the weight signal
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

The two concepts are related because validator trust depends on how much validator weight survives
consensus filtering ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Emission Context

Validator trust is not itself an emission rate. It is part of interpreting validator-side signal
retention before miner and validator emissions are understood in the broader consensus path
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

This keeps validator trust in consensus vocabulary rather than turning it into a direct reward quote
([Emission: Distribution](https://docs.learnbittensor.org/learn/emissions#distribution)).

## Reader Boundary

Validator trust should be read inside subnet consensus interpretation. It is not a validator guide,
a delegation recommendation, or a general-purpose reputation score
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)).

The stable article-level point is that validator trust explains how much of a validator's weight
signal remains after consensus clipping
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).
