# Relative Stake Weight

Relative stake weight describes a validator's stake-weighted share inside a Bittensor subnet. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#relative-stake-weight) defines it as
one validator's individual stake expressed as a proportion of the stake held by active validators in
the same subnet. This makes it a subnet-local concept rather than a network-wide label.

## Share of Validator Influence

The important idea is proportional influence. Stake weight describes a validator's computed stake
value, while relative stake weight compares that value with the validator set for the subnet being
evaluated. The [Glossary](https://docs.learnbittensor.org/resources/glossary#stake-weight) connects
stake weight to validator consensus power, and the relative-stake entry explains how that influence
is read as a share within a subnet.

## Relationship to Miner Evaluation

Relative stake weight matters after validators evaluate miner work. In
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets), subnet
validators measure miner outputs under subnet-specific standards before those evaluations feed the
consensus process. Relative stake weight helps describe how much influence one validator's signal
has compared with other active validators in that same setting.

## Consensus Context

[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) gives the broader context for
how validator-provided signals are aggregated into subnet outcomes. Relative stake weight should be
read as one way to understand validator influence within that aggregation, not as a separate scoring
model or a standalone measure of miner quality.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from local testing to testchain and then mainchain. For
relative stake weight, that sequence changes how readers should interpret examples of validator
influence.

In local testing, relative stake weight can show how validator influence is represented inside an
isolated subnet environment. That is useful for checking consensus logic, but it does not show a
validator's proportional influence in a live public subnet.

On testchain, relative stake weight can be observed in a shared testing network. This gives stronger
evidence about how stake-weighted influence behaves alongside other validators, miners, epochs, and
subnet rules than a private local run, while still keeping the result separate from production
mainchain activity.

On mainchain, relative stake weight belongs to live subnet consensus state. The
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) documentation describes how
validator signals feed subnet outcomes, so production readings should keep the selected subnet,
active validator set, stake context, and block timing attached.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A relative-stake-weight example in one environment should
not be read as identical evidence about validator influence in another environment.

This distinction matters because relative stake weight is proportional. The network environment
explains what kind of evidence the example provides, while the subnet's active validator set
explains what the proportion is being compared against.

For readers, this keeps relative-stake examples from sounding stronger than their environment
supports. Local, testchain, and mainchain contexts can all contain relative stake weight language,
but they do not carry the same interpretive weight.

## Relationship to Multiple Mechanisms

Relative stake weight describes a validator's stake-weighted share inside a subnet. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms) notes
that each mechanism operates with its own bond pool for Yuma Consensus calculations.

For readers, relative stake weight still names subnet-local validator influence. When a subnet runs
more than one incentive mechanism, bond-pool and consensus context should be read per mechanism.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)

## Interpretation Boundaries

Relative stake weight is not a validator ranking by itself, a staking recommendation, or a current
snapshot of any specific subnet. It is a proportional consensus concept whose meaning depends on the
subnet, the active validator set, and the underlying stake-weight context described in the official
glossary.

## Relationship to Coinbase

Relative stake weight and coinbase are related but different parts of subnet consensus and
emissions. Relative stake weight is a validator's stake-weighted share of influence among the active
validators in a subnet. The
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
documentation describes coinbase as the per-block protocol mechanism that drives TAO emission,
accumulates pending emissions, and triggers Yuma Consensus rounds at epoch boundaries, while the
[Glossary: Relative Stake Weight](https://docs.learnbittensor.org/resources/glossary#relative-stake-weight)
describes the proportional influence measure.

The connection is that coinbase triggers the consensus rounds in which a validator's relative stake
weight matters. Relative stake weight is one of the inputs that shapes how much a validator's signal
counts during Yuma Consensus, and coinbase is the recurring process that fires those epoch
calculations and advances emissions. Relative stake weight describes a validator's proportional
standing among peers; coinbase describes the per-block engine that runs the consensus rounds where
that standing is applied. One is a share-of-influence measure, the other is the emission mechanism
that drives the rounds it feeds into.
