# Rate Limits

Rate limits are Subtensor timing rules that pace repeated Bittensor chain actions. They are measured
in blocks rather than ordinary wall-clock time, so the key idea is chain progression between
accepted actions ([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

The concept explains pacing, not usefulness. A rate limit means the relevant chain action is being
repeated too soon for its rule; it does not say whether that action is economically wise or part of
a broader subnet design.

## Block-Based Cooldowns

A cooldown is anchored to a successful action of the relevant kind. The rate-limit documentation
distinguishes successful actions from failed attempts, so the accepted action is the pacing point
for later timing checks ([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

That distinction keeps the term narrow. A failed attempt can surface an error, but the cooldown rule
is tied to the action that the chain accepted for that action family.

The block unit is important. A cooldown measured in blocks follows chain progression, so its
practical duration depends on the chain environment and the number of blocks that pass before the
next accepted action is eligible
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

## Action Family Scope

Rate limits are scoped to action families. The same rate-limit reference separates global limits
from subnet-specific limits, so the action name and scope need to stay attached to the timing rule
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

This prevents one cooldown from being generalized to all chain activity. A staking-related action, a
weight-setting action, and another subnet action can all use rate-limit vocabulary while answering
different timing questions. Rate-limit examples group pacing by action family such as registration,
staking moves, and subnet-creation steps rather than by every read-only inspection path. When the
real question is a subnet hyperparameter, a transaction fee, or a read-only inspection result rather
than action pacing, the neighboring references answer it instead
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Transaction Fees](https://docs.learnbittensor.org/learn/fees)).

The global-versus-subnet split also changes how examples are read. A global limit describes the same
action family across the chain, while a subnet-configurable limit needs the relevant subnet setting
attached to the block count ([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Example Cooldowns

Staking operations provide one example of globally paced actions. Adding stake, removing stake, and
moving stake appear in the rate-limit context, while staking documentation gives the broader terms
around staking and delegation
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Staking and Delegation](https://docs.learnbittensor.org/staking-and-delegation/delegation)).

Validator weight setting is a different example because its pacing can be tied to subnet settings.
Subnet hyperparameter documentation provides the surrounding vocabulary for subnet-level settings,
while the rate-limit reference keeps the block-based timing model attached
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

The rate-limit reference includes examples with very different cooldown sizes
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)). A general transaction rate
limit is listed as 1 block, while delegate-take changes are listed at 216,000 blocks.

Those examples show why the unit matters. A one-block pause and a multi-day cooldown are both rate
limits, but they describe very different pacing constraints
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

The cooldown starts from a successful rate-limited operation. The rate-limit reference also notes
that unsuccessful operations may be retried
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)). That keeps the cooldown
attached to accepted chain activity rather than every attempted action.

Serving information gives a subnet-scoped example. The rate-limit reference lists serving updates as
configurable per subnet, with a default of 10 blocks
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

Weight setting is scoped differently from those global examples. The reference describes the weight
setting rate limit as configurable per subnet, with a default of 100 blocks and variation by subnet
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

Configurable subnet limits should be read with their subnet setting. A default block count gives
orientation, but the action family and subnet context still determine the effective pacing.

That is why the number alone is incomplete. The action family, scope, and block count travel
together: a 1-block general transaction limit, a 10-block serving update default, a 100-block
weight-setting default, and a 216,000-block delegate-take limit describe different rules, not one
universal cooldown ([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

These examples keep rate-limit prose grounded in block counts rather than ordinary clock time. The
reader still needs the relevant action family and network context to interpret what the cooldown
means in a concrete case.

## Errors and Chain Reads

Rate-limit errors are diagnostic signals from the chain layer. Subtensor error references place
these signals alongside other rejection categories, while the rate-limit documentation explains the
timing rule behind too-soon repeats
([Subtensor Error Codes](https://docs.learnbittensor.org/errors),
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor),
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

The error category is useful, but it is not a complete explanation of the surrounding mechanism. The
relevant action family, selected network, subnet context, and mechanism documentation still matter
for interpretation.

That gives rate-limit errors a narrow role. They can indicate that a repeated action arrived too
soon for the applicable rule, while the rate-limit reference supplies the timing model and the
action reference supplies the meaning of the attempted action
([Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor),
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

Rate limits apply to paced chain actions, not to every way of reading Bittensor information. The
transaction-fee documentation distinguishes state-changing submissions from free chain reads, and
the rate-limit reference focuses on pacing accepted actions
([Transaction Fees](https://docs.learnbittensor.org/learn/fees),
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

That boundary keeps rate-limit language away from passive observation. Read-only chain inspection
can show public state, but the rate-limit rule belongs to accepted chain actions that are subject to
the relevant cooldown.

## Cooldowns Count Blocks, Not Wall-Clock Time

[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits) documentation describes
cooldowns in block counts rather than as fixed calendar intervals. A rate-limited action must wait
until enough blocks pass before the same action family can run again.

That keeps rate-limit reading tied to chain cadence rather than to assumed wall-clock timing.

## Action Families, Hyperparameters, and Errors in One Pacing Pipeline

A rate-limited submission starts with identifying the action family and whether the rule is global
or subnet-configurable ([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

Global examples such as staking moves or general transaction pacing apply across the chain, while
subnet-scoped limits on a subnet such as netuid 1 tie cooldowns to
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) for weight
setting, serving updates, and related validator activity. A successful accepted action starts the
block countdown; a too-soon repeat surfaces a rate-limit error through
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor).

Read-only [chain reads](https://docs.learnbittensor.org/concepts/inspecting-the-chain) and free
storage inspection stay outside that pacing path
([Transaction Fees](https://docs.learnbittensor.org/learn/fees)). Rate limits pace state-changing
submissions, not passive observation of public chain state.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For rate limits on
a subnet such as netuid 1, that sequence changes how subnet-configurable cooldowns should be read.

In localnet, rate-limit examples can test cooldown mechanics in isolation. Localnet block counts for
weight setting or serving updates on netuid 1 reflect that test chain's hyperparameter state rather
than production pacing.

On testnet, subnet-scoped limits on netuid 1 apply in a shared non-production network where
hyperparameter defaults and observed block history are separate from mainnet.

On mainnet, rate-limit interpretation concerns production Subtensor pacing on the active network,
where global cooldowns and subnet-specific limits on netuid 1 both govern accepted chain actions
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A rate-limit example from one environment should not be
read as pacing behavior on another network.

## Distinction from Tempo

Rate limits space repeated chain actions from the same key in blocks. Tempo is the block interval
that defines a subnet epoch and paces consensus and emission
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo),
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Distinction from Transaction Fees

A rate limit and a transaction fee constrain an action on different axes. A rate limit is a timing
rule that paces how often a repeated action can be accepted, while a transaction fee is the cost the
chain charges to process a state-changing action
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Transaction Fees](https://docs.learnbittensor.org/learn/fees)).

- **Rate limit** — a timing rule pacing repeated actions in blocks.
- **Transaction fee** — the cost charged to process a state-changing action.

## Distinction from Subtensor Error Codes

A rate limit and an error code name a rule and the signal it can produce. A rate limit is the timing
rule itself, while a Subtensor error code is the diagnostic returned when an action arrives too soon
for that rule ([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

- **Rate limit** — the timing rule that paces an action.
- **Error code** — the diagnostic returned when a too-soon repeat is rejected.

## Distinction from Subtensor Storage

Rate limits cap how often certain extrinsics can be submitted within a time window. Subtensor
Storage exposes live recorded values such as stake or registration fields at a block height
([Glossary: Rate Limits](https://docs.learnbittensor.org/resources/glossary#rate-limits),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

- **Rate limits** — submission-frequency caps on state-changing actions.
- **Subtensor Storage** — read surface for live chain state at a block.

Rate limits govern action timing; storage reads observe current values without submitting changes
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

## Distinction from Non-fast Blocks

Non-fast blocks and rate limits also separate cadence from the rule being counted. Rate limits are
cooldowns expressed as a number of blocks, while non-fast blocks set how quickly those blocks arrive
locally, so a block-counted rate limit takes longer in non-fast-block mode without its block count
changing ([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)).

- **Non-fast blocks** — the local cadence at which blocks arrive.
- **Rate limits** — block-counted cooldowns, independent of that cadence.

## Distinction from Yuma Consensus

Rate limits name chain-level pacing rules that throttle how often certain extrinsics may run. That
transaction-cadence vocabulary is separate from the tempo pass that converts validator weight
submissions into emission shares
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Subnet hyperparameters can add their own pacing rules for weight commits and serving updates. Yuma
still credits miner incentives and validator dividends from aggregated weights at epoch boundaries
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers should keep chain-rate-limit vocabulary and subnet-settlement vocabulary separate. Rate
limits pace accepted actions; Yuma names how eligible weights become credited outcomes
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

- **Rate limits** — chain-level pacing for accepted extrinsic actions.
- **Yuma Consensus** — on-chain mechanism that turns validator weights into emission shares.

## Reader Boundary

Rate limits do not name a general retry guide, a universal pause for every chain action, or a full
diagnosis of why an operation failed
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor)).

Use the term for block-based pacing rules attached to accepted chain actions.
