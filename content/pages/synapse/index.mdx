# Synapse

A synapse is a structured data object exchanged in Bittensor subnet communication. The official
[Glossary](https://docs.learnbittensor.org/resources/glossary#synapse) describes synapse objects as
the main vehicle for information exchange between subnet validators and subnet miners, and notes
that they are based on Pydantic's BaseModel. The official
[Synapse reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/synapse/)
places Synapse alongside the Axon and Dendrite communication components.

## Message Object Context

Synapse is best read as the message object in the Axon-dendrite path. Axon names the server-side
endpoint, dendrite names the client-side component, and synapse names the object carried through
that request-and-response exchange. This makes synapse communication vocabulary, not a reward rule
or a subnet role.

The term is intentionally narrower than subnet protocol. A subnet protocol can decide what fields
matter and how responses are evaluated, while a synapse is the structured object used to carry the
information for that exchange.

## Relationship to Axons and Dendrites

The official
[Axon reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/)
describes Axon as the server-side interface that services incoming requests. The official
[Dendrite reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/dendrite/)
describes Dendrite as a network client that sends requests to axon endpoints and processes
responses. Synapse sits between those terms as the object being sent, received, and returned.

For Taopedia readers, the practical distinction is simple: axon is the contacted endpoint, dendrite
is the contacting component, and synapse is the structured message object.

## Subnet Communication Context

The glossary ties synapses to subnet miners and subnet validators because those roles exchange
task-specific information. A validator-side process may send a synapse through a dendrite to a
miner-side axon, and the response can contain the information that the subnet's scoring logic needs
to interpret.

This communication context does not define the scoring logic itself. Synapse explains the object
that carries task information; the subnet protocol and scoring model explain what that information
means for the subnet.

## Reader Boundary

Synapse should not be read as a complete subnet interface, a list of required fields, or a universal
request schema for all subnets. Exact fields, validation behavior, and response meaning depend on
the subnet protocol and the current official reference or subnet documentation.
