# Subtensor

Subtensor is Bittensor's layer 1 blockchain.

It is the system of record for transactions, rankings, Yuma Consensus, and incentive flow. It is the
chain layer beneath subnet markets rather than a single subnet, role, token, or interface
([Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction)).

The term is useful because it separates the blockchain record from the subnet markets built on top
of it. Subnets define work and evaluation contexts; Subtensor supplies the shared protocol layer
where recorded activity, consensus results, and tokenomics flow are anchored
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## Distinction from Mainchain

Subtensor and mainchain name related but distinct concepts. Subtensor names Bittensor's layer 1
blockchain and system of record for transactions and rankings, while mainchain names the primary
production network context
([Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Glossary: Mainchain](https://docs.learnbittensor.org/resources/glossary#mainchain),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

- **Subtensor** — the layer 1 blockchain and system of record.
- **Mainchain** — the primary production network context.

## Distinction from Public Subtensor

Subtensor names the chain layer that records network state. Public Subtensor names publicly
reachable access to that Subtensor chain layer
([Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Glossary: Public Subtensor](https://docs.learnbittensor.org/resources/glossary#public-subtensor)).

- **Subtensor** — the layer 1 blockchain and system of record.
- **Public Subtensor** — public access surface for the Subtensor chain layer.

A useful boundary to keep is that Subtensor names where activity is recorded and ordered, not the
logic that decides rewards. Documentation describes Subtensor as the blockchain layer that accepts
signed actions and stores state, while Yuma Consensus is the separate process that interprets
submitted weights into outcomes
([Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

For readers, this separates "where it is written" from "how it is decided." Extrinsics enter
Subtensor history as signed actions, and events and storage expose what was recorded, but the
consensus calculation that turns weights into incentives is its own mechanism
([Subtensor Extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

That layering also explains why the same chain underlies every subnet. Subtensor provides the shared
record and action surface, while each subnet defines its own task and scoring on top, so the chain
layer stays constant even as subnet logic differs
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## Distinction from Proposal

Subtensor is Bittensor's layer-one blockchain and system of record. A proposal is a governance item
submitted for Senate review that, if accepted, can affect privileged Subtensor behavior through the
governance process
([Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Governance Overview](https://docs.learnbittensor.org/governance)).

- **Subtensor** — the chain layer and system of record.
- **Proposal** — a governance item reviewed by the Senate.

## Distinction from BTCLI Permissions

Subtensor names Bittensor's layer 1 blockchain and system of record. BTCLI permissions name which
forms of wallet authority are required before CLI actions can interact with that chain layer
([Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Bittensor CLI: Permissions Guide](https://docs.learnbittensor.org/btcli/btcli-permissions)).

- **Subtensor** — the chain layer itself.
- **BTCLI permissions** — wallet authority required before CLI actions reach that layer.

## Distinction from Bittensor Platform Components

Subtensor and Bittensor platform components name one chain layer and the wider orientation map that
contains it. Platform components keeps subnets, chain state, tokens, roles, tools, and network
environments in one frame, while Subtensor names Bittensor's layer 1 blockchain and system of record
for transactions and rankings
([Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction),
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

- **Bittensor platform components** — the high-level map across subnets, chain, tokens, roles,
  tools, and networks.
- **Subtensor** — the layer 1 chain that records state and runs consensus.

## Distinction from Bittensor EVM Smart Contracts

Bittensor EVM smart contracts are Ethereum-compatible programs executed on Subtensor's EVM runtime.
Subtensor is Bittensor's layer 1 blockchain that records state and runs consensus
([Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials),
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor)).

- **Bittensor EVM smart contracts** — application-layer EVM execution on Bittensor.
- **Subtensor** — the chain layer that hosts that runtime and records outcomes.

Contract deployment changes EVM-side state on Subtensor; it does not replace subnet incentive
settlement through Yuma Consensus
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Distinction from Yuma Consensus

Subtensor is Bittensor's layer 1 blockchain and system of record for extrinsics, balances, and
subnet state. It names the chain layer that stores all on-chain data, not the recurring settlement
step that converts validator weight submissions into emission shares on one netuid
([Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Yuma Consensus is the on-chain mechanism that runs at the epoch boundary on each subnet. It reads
the weight matrix from eligible validators, applies clipping and bonding, and converts the result
into miner incentives and validator dividend shares
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Earlier sections separate Subtensor from EVM contract deployment and from Bittensor networks.
Subtensor records weight submissions and state changes; Yuma settles evaluation signals from
eligible validators on a selected netuid—not the chain layer label itself
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)).

Readers should treat Subtensor as the protocol's blockchain layer, and Yuma as the epoch-boundary
mechanism that allocates subnet rewards from consensus while validators remain active.

- **Subtensor** — blockchain layer and system of record.
- **Yuma Consensus** — epoch-boundary settlement from eligible validator weights to emission shares.

## Distinction from Bittensor Networks

Bittensor networks and Subtensor answer different scope questions. Bittensor networks name which
environment an observation belongs to, while Subtensor names the chain layer that records state and
processes extrinsics within each environment
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

- **Bittensor networks** — environment label (mainnet, testnet, localnet).
- **Subtensor** — the chain layer operating inside each environment.

## Distinction from Fast Blocks

Fast blocks and Subtensor name a timing configuration and the blockchain it runs on. Subtensor is
the Bittensor blockchain that records state and processes transactions, while fast blocks are a
local-development mode that runs a Subtensor instance with accelerated 250 millisecond block
production, so fast blocks are a setting applied to a local Subtensor chain rather than a separate
chain ([Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)).

- **Subtensor** — the Bittensor blockchain that records state and processes transactions.
- **Fast blocks** — a local mode that runs Subtensor with accelerated block production.

## Distinction from Consensus

Yuma Consensus runs within the Subtensor context. Validator rankings of miner performance are
aggregated into miner incentives and validator dividends, and those outcomes belong to Bittensor's
recorded incentive flow ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor)).

This does not make Subtensor a synonym for Yuma Consensus. Subtensor names the chain layer and
recording context; Yuma Consensus names a consensus process that uses validator evaluations to shape
reward allocation. Keeping the terms separate helps distinguish the protocol layer from one of the
processes that runs inside it
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

## Distinction from Emission

Subtensor names the chain layer that records state and processes extrinsics. Emission names the
broader reward and injection flow across the network
([Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Subtensor** — blockchain layer anchoring recorded activity.
- **Emission** — broader reward and allocation flow.

Chain vocabulary and flow vocabulary answer different scope questions
([Emissions](https://docs.learnbittensor.org/learn/emissions)).

## Distinction from Chain Vocabulary

Subtensor is built on Substrate within the Polkadot SDK stack. That foundation explains why related
Bittensor references use vocabulary such as blocks, runtime storage, and extrinsics when discussing
recorded activity on the chain
([Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

Those neighboring terms answer narrower questions. A block is an ordered unit of chain data, and an
extrinsic is a signed action intended for chain inclusion. Subtensor is the broader blockchain layer
where those units and actions fit
([Subtensor Extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

This also explains why Subtensor reference pages discuss storage, events, and extrinsics as related
surfaces. They describe different ways the chain layer exposes state, records activity, and accepts
submitted actions ([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

## Reader Boundary

Subtensor is a protocol reference term for Bittensor's blockchain layer. It is not an infrastructure
operations guide, a subnet ranking report, an endpoint availability report, or a catalog of
interface calls. Claims about a particular network environment, node-access surface, or exact chain
value need the relevant source for that context
([Subtensor Nodes](https://docs.learnbittensor.org/subtensor-nodes),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

The stable article-level point is that Subtensor names the chain layer and system of record. More
specific references should be used when the reader needs storage entries, events, extrinsics,
network access, or node-operation details
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Subtensor Nodes](https://docs.learnbittensor.org/subtensor-nodes)).
