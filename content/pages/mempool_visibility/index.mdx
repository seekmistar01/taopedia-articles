# Mempool Visibility

Mempool visibility is the exposure of pending transaction details before block inclusion
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield/)).

The term belongs to pre-inclusion transaction risk. It should be read separately from final block
history, runtime validity, and the later state produced by a processed transaction.

## Pending Window Context

The mempool glossary entry places pending and unconfirmed transactions before block inclusion
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)).

Mempool visibility asks what information can be observed during that waiting period. The concept is
about exposure before inclusion, not about whether the transaction ultimately appears in a block.

## MEV Risk

The MEV glossary entry connects maximal extractable value to transaction ordering and information
available before inclusion
([MEV](https://docs.learnbittensor.org/resources/glossary#mev-maximal-extractable-value)).

This makes mempool visibility a risk concept. If sensitive pending details are visible before a
block is produced, another observer may have information that can affect ordering-sensitive
behavior.

## MEV Shield Boundary

MEV Shield documentation describes encrypted mempool protection for sensitive transaction contents
while they are pending ([MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield/)).

That makes MEV Shield a visibility protection, not a general guarantee about transaction outcome.
Shielding changes what can be seen during the pending window; runtime rules still determine whether
a submitted action succeeds.

## Block Inclusion Boundary

A block is the chain data unit that contains transactions and has a block hash
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

This boundary matters because pending visibility and final chain history answer different questions.
Mempool visibility concerns the pre-inclusion window; block evidence concerns inclusion and ordering
after the chain records the transaction.

## Network Context

Bittensor documentation separates mainnet, testnet, and localnet contexts
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

A mempool visibility example should therefore keep the selected network attached. A pending-window
observation in one environment is not evidence about exposure in another environment.

## Reader Boundary

Mempool visibility supports claims about information exposure before block inclusion. It does not
prove final inclusion, later state, or every reason another observer could act on pending
information ([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

That boundary keeps the article focused on the information-exposure problem that MEV Shield is
designed to address.
