# Multiple Incentive Mechanisms

Multiple incentive mechanisms are a subnet feature that lets one Bittensor subnet host more than one
independent evaluation system. The official
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)
describes each mechanism as operating with its own bond pool for Yuma Consensus calculations, with
validators evaluating miners separately for each mechanism.

## Contrast with a Single Mechanism

A single
[incentive mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism) names
the rule context that connects subnet work, validator evaluation, and incentive outcomes inside one
subnet. Multiple incentive mechanisms extend that idea: one subnet can run parallel rule contexts
instead of only one.

For readers, the distinction is count and separation. One mechanism means one evaluation context;
multiple mechanisms mean several evaluation contexts can coexist inside the same subnet market.

References: [Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Incentive Mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism)

## Validator and Miner Evaluation Context

When a subnet uses multiple incentive mechanisms, validators must evaluate miners separately for
each mechanism. The glossary also notes that miner performance in one mechanism does not affect a
miner's rating in another. That separation keeps each mechanism's evaluation picture distinct.

This article does not describe how to configure mechanisms, rank miners, or read live subnet
parameters. Those details belong in official subnet documentation and live chain data.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)

## Bond Pool and Yuma Consensus Context

Each mechanism in a multi-mechanism subnet uses its own bond pool for Yuma Consensus calculations.
The [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) documentation describes
how validator-miner bonds feed consensus aggregation, and multiple-mechanism vocabulary adds that
those bond contexts can be separated by mechanism inside one subnet.

For readers, bond pool separation is part of what makes multiple mechanisms distinct from simply
adding more miners or validators to an ordinary single-mechanism subnet.

## Subnet Creator and Emission Context

Subnet creators can use multiple incentive mechanisms to distribute emissions across different kinds
of work or evaluation criteria within one subnet. The official
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)
documentation presents this as a subnet-design choice rather than as a network-wide default.

Multiple incentive mechanisms therefore belong to subnet policy vocabulary. They do not change the
meaning of network-wide TAO emission labels or wallet ownership terms by themselves.

References:
[Glossary: Subnet Creator](https://docs.learnbittensor.org/resources/glossary#subnet-creator),
[Emission](https://docs.learnbittensor.org/learn/emissions)

## Reader Boundary

Multiple incentive mechanisms should not be read as a recommendation to deploy a specific subnet
design, as a live mechanism count for any netuid, or as a substitute for the single-mechanism
incentive vocabulary used in ordinary subnet reading. It is a concept term for parallel evaluation
systems inside one subnet.
