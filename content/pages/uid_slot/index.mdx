# UID Slot

A UID slot is a participant position within a Bittensor subnet. The
[Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot) describes it as a
position occupied by a subnet miner or subnet validator and identified by a unique UID.

## Subnet Role

UID slots give subnet participants a position in the subnet context where miner and validator roles
are interpreted. The [Miners](https://docs.learnbittensor.org/miners) and
[Validators](https://docs.learnbittensor.org/validators) pages describe those two roles at a high
level: miners produce subnet work, and validators evaluate miner responses.

## Slot Capacity and Limits

Each Bittensor subnet has a configurable maximum number of UID slots. The
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) reference
includes `max_allowed_uids` as the ceiling for a given subnet. When slots are full, a new
registration can require displacing an existing participant under the miner-deregistration rules
([Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)).

For readers, the slot ceiling is a real capacity constraint. Once `max_allowed_uids` is reached,
accepting a new registrant depends on displacement rules for that subnet.

## Miner and Validator Roles Share Slot Vocabulary

UID slots organize both subnet miners and subnet validators inside one netuid. The glossary defines
a slot as the position a miner or validator occupies, not two separate slot systems for each role.
Registration vocabulary therefore grants a participant position inside the subnet regardless of
which role the hotkey performs there.

Readers should specify role context when discussing a slot. A UID identifies position inside a
subnet; miner and validator documentation describe what work happens from that position after
registration succeeds. Role-specific performance, permits, and emissions remain downstream of the
slot label itself. Validator permit vocabulary applies only where validating documentation defines
permit rules for that netuid. Miner deregistration and validator deregistration describe different
role-specific displacement paths when a full subnet must free one slot
([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Miners](https://docs.learnbittensor.org/miners),
[Validators](https://docs.learnbittensor.org/validators)).

## Netuid Scoping for Slot Identity

A UID number names a participant position inside one subnet, not a global identity across every
netuid. The same numeric UID can exist on different subnets because netuid selects which subnet
market is in scope. Slot language therefore requires pairing UID with netuid whenever readers
compare participants across markets.

This scoping prevents a common misread. Observing UID 7 on netuid 3 is not the same participant
position as UID 7 on netuid 19. Netuid and UID slot vocabulary should be read together when tracing
registration, displacement, or activity status on chain. Metagraph queries should always include
netuid when resolving a UID slot holder
([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid)).

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For UID slots on a
subnet such as netuid 1, that sequence changes how readers should interpret participant positions,
registration burns, and displacement examples.

In localnet, UID slot examples can be tested in an isolated environment. Local slot occupancy
reflects local chain configuration rather than production subnet participant state.

On testnet, UID slots can be observed in a shared, non-production network. Testnet slot readings on
a selected netuid are separate from mainnet metagraph state
([Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

On mainnet, a UID slot is a live participant position inside production subnet markets on the
connected Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A UID slot example from one environment should not be read
as representing production participant identity on another network.

## Register Purchases Entry Into a Slot

[Register](https://docs.learnbittensor.org/resources/glossary#register) vocabulary describes
purchasing a UID slot through the dynamic registration burn on an existing subnet. A UID slot names
the position that process grants once entry succeeds
([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot)).

Register is the action; the UID slot is the resulting participant position inside the chosen netuid.
Subnet registration at the network level creates netuids; register vocabulary adds participants to
an existing netuid.

## Registration Binds the UID to a Hotkey

The [Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot) states that
the UID is assigned to a hotkey when it registers as a miner or validator. UID slot vocabulary
therefore names the participant position inside a subnet, while registration names how that position
becomes tied to an operational hotkey
([Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register)).

Coldkey authority still governs wallet ownership and stake movement. The slot itself attaches to the
hotkey that registered for the subnet role.

## Active Status Sits on Top of the Slot

A UID slot identifies participant position, while
[active UID](https://docs.learnbittensor.org/resources/glossary#active-uid) status describes whether
that slot currently counts as participating under subnet activity rules. The metagraph exposes an
`active` field in that subnet context
([Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

Readers should not treat slot occupancy and active status as identical claims. A hotkey can hold a
UID slot while activity evidence changes with subnet state and timing.

## Immunity Protects New Slots From Displacement

The [Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period)
describes a post-registration window during which a participant may be protected from displacement.
When a subnet is at capacity, that timing boundary can affect whether a newly registered slot holder
is eligible to be removed when another participant registers into a full subnet
([Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)).

Immunity period and slot vocabulary answer different questions. The slot names the occupied
position; immunity names a timing rule that can shield that position from early displacement.

## Deregistration Vacates One Slot

[Deregistration](https://docs.learnbittensor.org/resources/glossary#deregistration) describes
removing a participant from a subnet, while a UID slot describes the participant position that
process can vacate. When a subnet is at capacity, a new registration may trigger deregistration for
an existing slot holder according to subnet displacement rules.

Subnet deregistration is a different level of state. Participant-level deregistration can vacate one
UID slot while the subnet continues; subnet deregistration ends the subnet context that contained
the collection of UID slots
([Subnet Deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration)).

## Validator Permit Applies After the Slot Exists

A UID slot grants participant position inside a subnet.
[Validator permit](https://docs.learnbittensor.org/resources/glossary#validator-permit) vocabulary
marks which neurons may set weights and participate in consensus, awarded to the top neurons by
stake weight on that netuid. Miners and validators can share slot vocabulary, but permit rules apply
only where validating documentation defines them for that subnet.

## Child Hotkeys Operate From a Parent Slot

A parent hotkey that holds a UID slot can authorize a
[child hotkey](https://docs.learnbittensor.org/validators/child-hotkeys) to validate for it. The
child hotkey is the authorized key, not the slot itself
([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot)).

## A UID Slot Is a Position, Not a Permanent Claim

Documentation treats a UID slot as the indexed position a neuron occupies on a subnet, not a
permanent right to stay. When the subnet is full, the slot can be reclaimed through deregistration
if a new registration arrives and the current holder has the lowest pruning score
([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Mining in Bittensor](https://docs.learnbittensor.org/miners)).

For readers, this reframes a slot as contested capacity. Registration assigns a UID, but ongoing
emissions determine whether that assignment survives the next full-subnet registration event
([Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register)).

That distinction also explains why slot vocabulary is separate from role vocabulary. The same UID
slot can be held by a miner or validator neuron; what matters for retention is the holder's
standing, not the role label alone
([Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner)).

## Distinction from Neuron

A UID slot is the indexed participant position inside a subnet. A neuron is the registered
participant record that occupies that position after registration succeeds
([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Glossary: Neuron](https://docs.learnbittensor.org/resources/glossary#neuron),
[Understanding Neurons](https://docs.learnbittensor.org/learn/neurons)).

- **UID slot** — one indexed position on a netuid.
- **Neuron** — the registered participant record at that position.

## Distinction from Metagraph

A UID slot is one indexed participant position inside a subnet. The metagraph is the per-netuid
snapshot that lists those positions and their values at a block
([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Glossary: Metagraph](https://docs.learnbittensor.org/resources/glossary#metagraph),
[The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

- **UID slot** — one indexed participant position inside that state.
- **Metagraph** — subnet-wide participant state at one block height.

## Distinction from Pruning Score

A UID slot is the participant position a miner or validator occupies in a subnet. The pruning score
ranks which occupied slot is reclaimed first when a full subnet takes a new registration
([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Mining in Bittensor](https://docs.learnbittensor.org/miners),
[Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)).

## Reader Boundary

This article defines the slot concept. It does not list subnet participants, count occupied slots,
or describe current subnet state. Readers should use subnet references for role-specific details.

A UID slot identifies who can occupy a participant position inside a subnet.
[Coinbase](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase) names the
per-block emission operation that advances reward processing; the slot names the position that can
receive subnet outcomes, not the emission mechanism itself.
