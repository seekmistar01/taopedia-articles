# Epoch

An epoch is the consensus period inside a Bittensor subnet
([Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)). It is tied to subnet
tempo and marks the timing window in which subnet consensus work is executed.

The term is useful because it separates timing from scoring. An epoch names the period in which
consensus processing occurs; the subnet's incentive mechanism and validator weights explain what is
being evaluated inside that period.

## Distinction from Tempo

Epoch and tempo are related timing terms on a subnet. Tempo is the hyperparameter that counts how
many blocks make up one consensus cycle, while epoch names the recurring consensus period anchored
to that tempo boundary ([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)).

- **Tempo** — block count for one consensus cycle.
- **Epoch** — the consensus period spanning submission and settlement within that cycle.

Validator weights can be updated across the tempo, but Yuma outcomes tied to that pass finalize at
the epoch boundary when the tempo completes
([Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

## Distinction from Coinbase

Coinbase is the per-block emission operation that accumulates pending emissions and triggers Yuma
Consensus at epoch boundaries. Epoch names the recurring consensus period; coinbase names the
operational step that processes pending emissions when that boundary arrives
([Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)). Coinbase therefore
concerns emission processing at the boundary, while epoch frames the full period during which
validator weights can enter the pass ([Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Epoch** — concept vocabulary for this article's topic.
- **Coinbase** — related term at a different layer of the same flow.

The two labels answer different reader questions on the same path
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## Distinction from Yuma Consensus

An epoch names the consensus window on a subnet from weight submission through settlement at the
tempo boundary. It answers when that pass runs, not how Yuma turns submitted weights into emission
shares ([Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch),
[Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

Yuma Consensus is the on-chain tempo step that reads the weight matrix, applies clipping and
bonding, and then credits miner incentives and validator dividends when that window closes
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Emission: Distribution](https://docs.learnbittensor.org/learn/emissions#distribution)).

The tempo and coinbase sections above already separate epoch timing from emission processing at the
boundary. Epoch vocabulary names the period; Yuma vocabulary names the settlement logic inside it
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo),
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)).

Readers should not read an epoch label as a payout amount or as proof that one validator's latest
submission has settled. Outcomes still depend on the weight matrix recorded for that pass on the
relevant netuid ([Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Epoch** — consensus period from submission through settlement.
- **Yuma Consensus** — tempo step that turns submitted weights into emission shares.

## Distinction from Block

A block is one chain record. An epoch spans the block count set by tempo, so block height locates
one moment inside the wider consensus pass
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)).

Use block for one inclusion point in chain history. Use epoch for the full submission-to-settlement
window on the selected subnet.

## Distinction from Emission

Emission names injection and distribution together. Epoch names the consensus window in which
validator signals are gathered before distribution is credited
([Emission](https://docs.learnbittensor.org/learn/emissions),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)).

- **Epoch** — consensus window for gathering validator signals.
- **Emission** — broader injection-and-distribution flow across the protocol.

Period vocabulary names when signals arrive; emission vocabulary names the wider reward flow
([Emission: Distribution](https://docs.learnbittensor.org/learn/emissions#distribution)).

## Distinction from Kappa

Kappa sets the stake threshold used during an epoch's Yuma pass. Epoch names the period in which
that pass runs
([Subnet Hyperparameters: Kappa](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#kappa),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)). Kappa vocabulary belongs to
clipping support inside the pass; epoch vocabulary names the pass itself.

- **Epoch** — concept vocabulary for this article's topic.
- **Kappa** — related term at a different layer of the same flow.

The two labels answer different reader questions on the same path
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## Distinction from Consensus Score

Consensus score names the stake-weighted benchmark used when clipping validator weights during an
epoch. Epoch names the timing window for that consensus pass
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch),
[Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping)).

## Distinction from Non-fast Blocks

Non-fast blocks name the local twelve-second block cadence. Epoch is measured as a block count set
by tempo, independent of that cadence mode
([Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)).

- **Non-fast blocks** — local cadence at which blocks are produced.
- **Epoch** — block-counted consensus interval set by tempo.

Cadence vocabulary names block production speed; epoch vocabulary names the consensus window
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Reader Boundary

Epoch is the right term when the focus is a subnet consensus period. If the focus is cadence,
[tempo](https://docs.learnbittensor.org/resources/glossary#tempo) is more precise. If the focus is
aggregation logic, [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) is more
precise. If the focus is evaluation signals themselves, validator weights are more precise
([Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)).

These boundaries keep epoch focused on consensus timing, not a subnet task description or a complete
account of reward outcomes.

## Distinction from Subnet Task

An epoch and a subnet task separate timing from the work being timed. A subnet task is the problem
miners are asked to solve and validators score, while an epoch is the recurring block interval over
which weights are tallied and Yuma Consensus runs to distribute emissions for that work
([Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch),
[Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Subnet task** — the problem miners solve and validators score.
- **Epoch** — the recurring interval over which consensus runs and emits.
