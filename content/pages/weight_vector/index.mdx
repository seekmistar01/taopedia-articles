# Weight Vector

A weight vector is one validator's structured set of miner evaluations inside a Bittensor subnet.
Each element represents a weight assigned to a subnet miner according to that validator's evaluation
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The key boundary is scope. A single vector is one validator's signal before aggregation, not the
whole subnet result.

## Per-Validator Signal

A weight vector comes from one validator's evaluation process. It expresses how that validator
scores miners under the relevant subnet task and incentive mechanism
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

That keeps the term narrower than final rank or reward allocation. Consensus can use many vectors,
but each vector remains a per-validator input.

## Task and Mechanism Context

The meaning of a weight vector depends on the subnet task. The incentive mechanism defines useful
work, and the vector records one validator's evaluation relative to that definition
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

This prevents cross-subnet overreach. A vector from one subnet's task does not automatically
describe performance under another subnet's task or scoring method.

## Validator-Weights Boundary

Weight vector and validator weights describe closely related ideas. The vector is the per-validator
structure, while validator weights are the evaluation signals inside that structure
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

For readers, the useful distinction is that a vector organizes one validator's evaluations. The
weights are the values that carry miner-scoring information.

## Consensus Score Boundary

A weight vector is distinct from consensus score. The vector is a submitted evaluation signal, while
consensus score is the stake-weighted threshold used to filter outlier submissions before final rank
results
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

That makes consensus score downstream of validator weight signals. A vector supplies input; the
consensus score helps constrain how submitted signals can influence the final result.

## Stake-Weight Context

Weight vectors are interpreted inside a stake-weighted consensus process. Stake weight helps
determine how much influence a validator's vector carries after it is submitted
([Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

This keeps evaluation and influence separate. The vector records miner evaluations; stake weight
helps determine how strongly that validator signal is counted.

## Rank Context

Rank is downstream of weight vectors. A vector supplies one validator's miner evaluations, while
rank is the consensus-adjusted miner-side result after submitted signals are filtered and aggregated
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

That distinction helps readers avoid treating a single vector as the final subnet outcome. A vector
is input vocabulary; rank is result vocabulary.

## Visibility Context

Independent validator evaluation is what gives a weight vector useful information. Weight copying
weakens that purpose when validators rely on visible public weights instead of fresh evaluation
([The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

Commit Reveal can add a timing layer by concealing fresh weight submissions until a reveal period.
That protects timing around the information, but it does not change the definition of a vector.

## Reader Boundary

Weight vector should be read as per-validator consensus-input vocabulary. The stable point is that a
vector represents one validator's miner evaluations before those signals are organized and
aggregated
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

This article does not provide validator recommendations, miner-performance claims, or cross-subnet
rankings. Exact interpretation belongs to the subnet task and incentive mechanism being discussed.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. The weight vector
concept applies across the Bittensor lifecycle: per-validator miner evaluations can be submitted and
tested in localnet for isolated consensus evaluation, observed on testnet for shared non-production
conditions, and relied upon on mainnet for live operation with real emissions.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. Weight vector examples or consensus input outcomes from
one environment should not be read as representing production subnet performance in another
environment.

## Submission Scope

A weight vector should be read as validator-submitted mechanism input, not as the final consensus
result by itself. Yuma Consensus describes validators assigning weights, while Commit Reveal changes
when submitted weights become visible
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

For readers, the vector is an input into a larger mechanism. Claims about final ranking, incentives,
or dividends need the surrounding consensus and emission context rather than the vector alone.
