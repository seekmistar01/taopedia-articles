# Subtensor Error Codes

Subtensor error codes are diagnostic signals from the blockchain layer beneath Bittensor. They help
readers understand why a network action failed without treating the error text as a complete guide
to the underlying rule. The [Subtensor Error Codes](https://docs.learnbittensor.org/errors) page
frames the topic as errors that can appear through Bittensor-facing interfaces, while the
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor) page describes
standard error categories from the Subtensor layer.

## What error codes describe

An error code usually points to the rule family that rejected an action. The official
[Subtensor Error Codes](https://docs.learnbittensor.org/errors) overview groups these failures into
standard Subtensor errors, custom errors, and errors inherited from the underlying Substrate
framework. That grouping matters because a short error message may identify the category of a
failure, but the surrounding action and source-specific context still determine what it means.

## Standard errors

Standard Subtensor errors are named conditions defined by the Bittensor chain layer. They cover
families such as subnet existence, balance and stake conditions, weight-setting constraints,
network-entry boundaries, and other protocol checks. The
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor) page presents this as
a high-level reference and points readers toward the fuller Subtensor error reference for complete
coverage.

## Custom errors

Custom errors are numeric conditions returned by Subtensor when a more specific rule fails. They can
describe staking limits, rate limits, liquidity limits, shielded-message parsing, and other
chain-layer checks. The [Subtensor Custom Errors](https://docs.learnbittensor.org/errors/custom)
page lists these numeric entries and their short descriptions. For readers, the important idea is
that the number is only a lookup key; it still needs the surrounding Bittensor action to be
interpreted correctly.

## Why source context matters

The same broad error family can appear around different Bittensor activities, including staking,
subnet participation, wallet use, and validator behavior. That is why error-code pages should be
read as diagnostic context, not as standalone operating instructions. The
[Subtensor Error Codes](https://docs.learnbittensor.org/errors) overview connects errors to several
Bittensor-facing interfaces, and the
[Subtensor Custom Errors](https://docs.learnbittensor.org/errors/custom) page shows how short codes
map back to named failure conditions.

## Relation to nearby concepts

Error codes sit close to [[staking_and_delegation|Staking and Delegation]],
[[dynamic_tao|Dynamic TAO]], and [[wallets|Wallets]]. Staking and Delegation explains one common
area where chain-layer rules can reject an action. Dynamic TAO explains the subnet-token and pool
context behind some staking-related outcomes. Wallets provide the user-facing key and balance
context around many network interactions. The
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor) and
[Subtensor Custom Errors](https://docs.learnbittensor.org/errors/custom) pages define the error
families themselves.

## Relationship to Rate Limits

Subtensor error codes and rate limits address related but different sides of chain rejection
feedback. [Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits) pace repeated
actions with block-based cooldowns, while
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor) and
[Subtensor Custom Errors](https://docs.learnbittensor.org/errors/custom) name the failure conditions
the chain returns when an action is rejected.

For readers, a rate-limit rejection is one error-code context among many. Rate limits explain the
cooldown rule behind some too-soon failures, but error codes also cover staking, subnet, and
wallet-related conditions that are not rate limits.

References: [Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)

## Relationship to Multiple Mechanisms

Subtensor error codes can surface around weight-setting constraints and other validator actions. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-matrix) notes that each
incentive mechanism has its own weight matrix on chain.

For readers, a failed weight submission should not be treated as affecting every mechanism in a
subnet. Error diagnosis still depends on which mechanism and action triggered the failure.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)

## Relationship to Extrinsics

Subtensor error codes and extrinsics address related but different parts of a failed chain action.
An extrinsic is a submitted state-changing runtime request, while an error code is diagnostic
feedback returned when that request is rejected. The
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) reference lists
submitted methods, and the [Subtensor Error Codes](https://docs.learnbittensor.org/errors) overview
groups the failure families that can appear around those submissions.

For readers, an extrinsic names what was attempted, while an error code names why the runtime
rejected it. The same error family can appear around different extrinsic types, so the submitted
action still needs to be read alongside the returned code.

References: [Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)

## Relationship to Subtensor Events

Subtensor error codes and Subtensor events address related but different kinds of runtime feedback.
Events are named records emitted when activity is processed, while error codes are rejection signals
returned when an action fails validation. The
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events) reference lists emitted
event names, and the [Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor)
page describes standard failure categories from the Subtensor layer.

For readers, a successful or accepted runtime path may produce events, while a rejected submission
returns an error code instead of completing the intended state change. The two references answer
different questions about chain-layer outcomes.

References: [Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor)

## Relationship to Subtensor

Subtensor error codes and Subtensor address related but different layers of chain feedback. The
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor) defines
Subtensor as Bittensor's layer 1 blockchain and system of record for transactions, while error codes
are diagnostic signals returned when actions against that layer fail protocol checks. The
[Subtensor Error Codes](https://docs.learnbittensor.org/errors) overview connects those failures to
Bittensor-facing interfaces.

For readers, Subtensor names the blockchain layer where rules are enforced, while an error code
names the rejection category surfaced to the caller. Error codes help interpret failure but do not
replace the underlying mechanism documentation for the action being attempted.

References: [Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)

## Relationship to Network Context

Subtensor error codes are interpreted inside a specific Bittensor network environment. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet as deployment contexts, and a failed submission returns
feedback from whichever network endpoint processed the action. The same error family can appear in
each environment, but the surrounding balances, subnet state, and rule context belong to that
network's chain history.

For readers, this matters when debugging across environments. A rejection on testnet may reflect
testnet balances or subnet state even when the error name matches a mainnet failure category.

References: [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)
