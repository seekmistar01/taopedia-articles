# UID Trimming

UID trimming is a subnet-owner operation that reduces the number of registered neuron UIDs on a
subnet down to a chosen maximum and compacts the remaining UIDs into consecutive indices. The
[UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming) guide presents it as a way for
a subnet owner to manage the active participant base while keeping clean, sequential UID numbering.

## What It Represents

UID trimming removes the lowest emitters first until the live count reaches an owner-chosen maximum,
then compresses survivors into consecutive indices. Immune UIDs and owner-owned UIDs are protected;
trimming is a manual sudo action with documented rate limits and minimum target counts
([UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming),
[Glossary: Sudo](https://docs.learnbittensor.org/resources/glossary#sudo)).

Examples tied to netuid 1 illustrate one subnet's owner trim path. They should not be read as
representing production UID layout or trim eligibility on another network
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

## Distinction from Deregistration

Deregistration recycles one incumbent slot when a full subnet admits a new registration. UID
trimming is a subnet-owner sudo action that removes many low emitters in one pass and compresses
survivors into consecutive indices
([UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming),
[Glossary: Deregistration](https://docs.learnbittensor.org/resources/glossary#deregistration),
[Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)).

- **UID trimming** — owner-initiated bulk reduction to a target count with survivor compression.
- **Deregistration** — one incumbent removed when a full subnet accepts a new registration.

## Distinction from Pruning Score

Pruning score ranks miners when a full subnet receives a new registration and one slot must open.
UID trimming is a subnet-owner bulk resize that removes many low emitters down to a chosen maximum
([UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming),
[Glossary: Pruning Score](https://docs.learnbittensor.org/resources/glossary#pruning-score)).

- **Pruning score** — ranking input when one slot opens for a new registrant.
- **UID trimming** — owner-initiated bulk reduction to a target count.

## Distinction from Immunity Period

Immunity period protects a newly registered UID from displacement for a grace window. UID trimming
skips immune UIDs but names the owner resize operation, not the protection rule
([Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period),
[UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming)).

- **UID trimming** — owner-initiated bulk reduction to a target count.
- **Immunity period** — newcomer protection window before displacement.

## Distinction from Max Allowed UIDs

Max allowed UIDs is the subnet capacity ceiling. UID trimming lowers the live count toward a smaller
owner-chosen maximum and compresses survivors into contiguous indices
([Glossary: Max Allowed UIDs](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#maxalloweduids),
[UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming)).

- **Max allowed UIDs** — configured capacity ceiling for registrations.
- **UID trimming** — owner action removing low emitters to a target count.

## Distinction from Register

Register is the subnet-entry path that assigns a UID after paying the registration burn. Trimming
removes low emitters; trimmed participants must register again if they want to return
([Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register),
[UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming)).

- **UID trimming** — owner-initiated removal and compression of low emitters.
- **Register** — subnet entry that assigns a UID, including re-entry after a trim.

## Distinction from Sudo

Sudo is the privileged control mode that lets the subnet owner perform owner-only chain actions. UID
trimming is one specific sudo operation that resizes the participant set
([Glossary: Sudo](https://docs.learnbittensor.org/resources/glossary#sudo),
[UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming)).

- **UID trimming** — the owner sudo action that removes low emitters and compresses survivors.
- **Sudo** — the privileged owner control mode that authorizes trimming and other owner-only
  actions.

## Distinction from Mechanism Count

Mechanism count sets how many incentive mechanisms run on a subnet. UID trimming removes excess UIDs
when live registration exceeds the configured ceiling
([UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Mechanism count** — the configured number of incentive mechanisms on a subnet.
- **UID trimming** — the action that brings live UID count back under the configured ceiling.

## Distinction from Yuma Consensus

UID trimming is the subnet-owner sudo action that reduces active participant slots on a netuid. It
names metagraph capacity management, not the epoch-boundary settlement that converts validator
weight submissions into emission shares
([UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming),
[Glossary: UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot)).

Yuma Consensus is the on-chain mechanism that runs at the epoch boundary on the selected netuid. It
reads the weight matrix from eligible validators who remain on the trimmed metagraph, applies
clipping and bonding, and converts the result into miner incentives and validator dividend shares
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Trimming changes how many UID slots the subnet exposes; Yuma still settles weight signals from
participants who occupy remaining slots. Capacity management and consensus settlement are separate
administrative and reward layers on the same netuid
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

Readers should treat UID trimming as an owner-initiated resize action, and Yuma as the recurring
mechanism that allocates rewards from validator weights on the active metagraph.

- **UID trimming** — owner sudo action that reduces active participant slots on one netuid.
- **Yuma Consensus** — epoch-boundary settlement from eligible validator weights to emission shares.

## Reader Boundary

This page defines the concept at a high level. It does not report which UIDs exist on any particular
subnet, which would be trimmed, or whether an owner has trimmed recently. Those are live chain state
and should be checked for the relevant netuid at the time of observation
([UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming)).
