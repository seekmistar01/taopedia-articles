# Subtensor Storage

Subtensor storage entries are runtime state definitions exposed by the Subtensor runtime
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

The term belongs to chain-state reference vocabulary. It should be read separately from events,
constants, and submitted runtime actions.

## What Storage Entries Represent

Storage entries name places where runtime state is exposed for inspection. A storage reference can
identify the runtime area, entry name, type, access path, and summary for a state surface
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

This makes storage a map of exposed runtime state rather than a full explanation of every mechanism
that changes that state.

## Runtime Areas

Storage entries are grouped by runtime area. The owning area is part of the entry's meaning because
the same kind of state label can carry different context in different runtime areas
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

For readers, a storage item should first be read through its owning area, then through its entry
name and type.

## Definitions and Queried Values

The storage reference defines where state is exposed. A chain read or inspection view can show a
value from a selected network and block context
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

This distinction matters for evidence. The reference gives the map; an observed value belongs to the
chain context where it was read.

## Relationship to Events and Extrinsics

Storage, events, and extrinsics describe different surfaces. Storage exposes state, events name
records emitted during processing, and extrinsics name submitted runtime actions
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

A feature may involve all three surfaces, but they should not be collapsed into one evidence type.

## Network and Block Scope

Storage values are scoped by chain history. The same storage entry name can be meaningful across
environments while returned values belong to separate network and block contexts
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

When precision matters, the network and inspection context should stay attached to the claim.

## Reader Boundary

Subtensor storage should be read as chain-state reference vocabulary. It is not a full mechanism
guide, proof of off-chain intent, or replacement for concept documentation
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

The stable article-level point is that Subtensor storage names runtime state surfaces exposed for
inspection.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Subtensor
Storage, this sequence gives readers a boundary for interpreting storage-query examples, chain-state
notes, and network comparisons.

Localnet examples are isolated and reflect local chain state, so they are useful for controlled
experiments rather than evidence of live Bittensor behavior. Testnet examples add shared
non-production conditions, which can reveal integration behavior without touching mainnet state.

On mainnet, Subtensor Storage examples should be read as live production storage behavior on the
production Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, so outcomes from one environment should not be treated as
proof of behavior in another.
