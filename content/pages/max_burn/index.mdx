# Max Burn

Max burn is a per-subnet hyperparameter that sets the upper bound on the dynamic TAO burn required
to register a neuron on a subnet. Documentation describes it as the upper bound for the dynamic TAO
burn required for neuron registration on the subnet, so it is the ceiling the registration cost
cannot exceed.

## What It Controls

The hyperparameter caps the registration burn rather than fixing it. Registering a neuron requires
burning an amount of TAO, and max burn is the most that amount can be on the subnet. It does not set
the current cost or decide who registers; it only limits how high the burn can climb.

The registration burn is dynamic: it rises as registrations occur and eases back during quieter
periods, moving between min burn and this upper bound. Max burn fixes the top of that range, so even
at peak demand the registration burn is held at or below it
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Default and Setting

The documentation lists a default of 100 TAO and marks the parameter as owner-settable, so a subnet
owner can adjust the ceiling for their subnet. The value in force on netuid 1 can differ from the
documented default and from another subnet's ceiling
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Distinction from BurnHalfLife

Max burn caps the dynamic registration burn on an existing subnet. `BurnHalfLife` names how quickly
that live burn can decay toward the subnet floor when registration demand quiets down
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#burnhalflife),
[Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register)).

- **Max burn** — ceiling on the registration burn band.
- **BurnHalfLife** — decay rate pulling the live burn down during quiet periods.

## Distinction from BurnIncreaseMult

`BurnIncreaseMult` raises the live registration burn after each successful registration inside the
documented min/max band. Max burn names the upper bound that upward stepping cannot exceed
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#burnincreasemult),
[Subnet Hyperparameters: MaxBurn](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#maxburn)).

- **BurnIncreaseMult** — steps the burn up on each successful registration.
- **Max burn** — ceiling the upward stepping cannot exceed.

## Distinction from Min Burn

Max burn and min burn set the upper and lower bounds of the same dynamic registration burn band on a
subnet. Registration cost moves between those limits as demand changes
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Mining: Miner registration](https://docs.learnbittensor.org/miners#miner-registration)).

- **Max burn** — ceiling the registration burn cannot exceed.
- **Min burn** — floor the registration burn cannot fall below.

## Distinction from Burn Cost

Max burn should not be confused with subnet-creation burn cost. Burn cost is the TAO amount recycled
when creating a new subnet, while max burn bounds the separate dynamic burn paid to register a
neuron on a subnet that already exists
([Glossary: Burn Cost](https://docs.learnbittensor.org/resources/glossary#burn-cost),
[Create a Subnet: Burn cost](https://docs.learnbittensor.org/subnets/create-a-subnet#burn-cost),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#maxburn)).

- **Burn cost** — recycled TAO required to create a new subnet market.
- **Max burn** — ceiling on dynamic neuron registration burn on an existing subnet.

## Distinction from Register

When a subnet is full, registration can displace the lowest-ranked existing participant rather than
simply failing. Max burn still bounds the TAO burn for that entry attempt; displacement rules decide
which UID slot changes hands
([Mining: Miner registration](https://docs.learnbittensor.org/miners#miner-registration),
[Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register)).

- **Register** — the action that claims a UID slot by paying the dynamic burn.
- **Max burn** — ceiling on how high that burn can climb.

## Chain Reads for netuid 1

Readers can verify live hyperparameter values for the documented example netuid with
`btcli subnet hyperparameters --netuid 1 --network finney`
([Subnet Hyperparameters: View hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#view-hyperparameters)).

That read path keeps live hyperparameter claims tied to a parseable netuid.

## Per-Subnet Live Value Boundary

Max burn is per-subnet chain state. The registration-burn ceiling configured on one netuid caps how
high the dynamic charge can rise on that subnet, but the live ceiling can differ from reference
defaults and from the ceiling on another netuid
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

This article's infobox uses netuid 1 as an example label when reading one subnet's hyperparameter
table on Finney mainnet. That example helps readers verify one row; it is not proof that every
subnet exposes the same live ceiling
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Max burn governs registration pricing ceilings; Yuma Consensus still settles validator weights into
emission shares each epoch on the active subnet
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Max burn** — ceiling the dynamic registration burn cannot exceed on a subnet.
- **Per-subnet boundary** — live value is chain state on one netuid.

## Distinction from Yuma Consensus

Max burn is a per-subnet hyperparameter that sets the ceiling of the dynamic registration burn band.
Yuma Consensus is the separate tempo process that converts validator weight submissions into
emission shares each epoch
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Max burn** — ceiling the registration burn cannot exceed.
- **Yuma Consensus** — on-chain settlement that turns validator weights into emission shares. Max
  burn names registration burn ceilings; Yuma processes validator weights into rewards
  ([Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions)).

- **Registration vs settlement** — max burn caps registration economics; Yuma aggregates validator
  weights into emission shares at epoch boundaries
  ([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
  [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).
- **Ceiling not price** — max burn bounds the band; live burn still moves inside min/max limits
  ([Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register)).

## Reader Boundary

This page defines the concept at a high level. It does not report the max burn value set on any
particular subnet or the current registration cost. Those are live chain state and should be checked
for the relevant netuid. The default of 100 TAO is the documented value, and the parameter is
owner-settable per subnet.
