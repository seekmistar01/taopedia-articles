# Bittensor Emissions

Bittensor emissions describe how newly created TAO and subnet-specific alpha tokens enter the
network's reward system. The official [Emission](https://docs.learnbittensor.org/learn/emissions)
reference presents emissions as the source of reward distribution across Bittensor's tokenomics.

The term is broader than a single token amount. The
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) reference
separates the network token from subnet-specific alpha token contexts, so an emissions claim should
name which asset and layer it is discussing.

## Asset Scope

Emissions include TAO and subnet alpha tokens. The
[Emission](https://docs.learnbittensor.org/learn/emissions) documentation describes the creation and
distribution of TAO, while the
[alpha tokens](https://docs.learnbittensor.org/resources/glossary#alpha-tokens) glossary entry
places subnet alpha tokens in subnet-specific tokenomics.

That asset boundary is important for readers. TAO belongs to the network-wide side of Bittensor
tokenomics, while alpha tokens belong to individual subnet contexts. Treating both as one undivided
amount can obscure what the emission discussion is actually measuring.

## Injection and Distribution

The [Emission](https://docs.learnbittensor.org/learn/emissions) reference separates the idea of
emission entering subnet contexts from the later distribution of rewards. Injection describes how
new reward material reaches the relevant subnet context; distribution describes how those rewards
are allocated after the subnet context exists.

This distinction keeps emissions from being reduced to one supply-rate sentence. The
[flow-based emissions](https://docs.learnbittensor.org/learn/emissions#flow-based-emissions) section
is one example of emission vocabulary that depends on subnet context rather than only on a
network-wide release schedule.

## Subnet Role Context

Subnets give emissions a work-and-evaluation setting. The
[incentive-mechanism](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
reference explains how subnet incentives connect miner work, validator evaluation, and reward
outcomes.

That means emissions are not only accounting vocabulary. They are also the tokenomic source that
lets subnet incentive mechanisms reward useful work and evaluation. The article-level concept should
stay tied to those roles without becoming a validator guide or a miner guide.

## Consensus Context

Yuma Consensus explains how validator evaluation signals are aggregated into subnet outcomes
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)). Emissions are related
because the reward material being distributed needs a consensus-driven allocation path.

The two concepts answer different questions. Emissions explain what reward material enters the
system, while Yuma Consensus explains how subnet evaluation signals shape reward allocation after
validator signals are available.

## Staking Context

Staking connects token holders to validator-side reward outcomes, so emissions often appear beside
staking and delegation vocabulary. The
[Staking and Delegation](https://docs.learnbittensor.org/staking-and-delegation/delegation)
documentation describes that staking context, while the
[Emission](https://docs.learnbittensor.org/learn/emissions) reference keeps the broader emission
frame around reward creation and distribution.

For readers, staking is one place emissions can be observed, not the whole emissions system.
Emission vocabulary should keep the broader tokenomics context attached.

## Halving and Flow Context

Halving and flow-based emissions refine the broader emissions topic. The
[Halving](https://docs.learnbittensor.org/concepts/halving) reference describes supply-rate
reductions, while the
[flow-based emissions](https://docs.learnbittensor.org/learn/emissions#flow-based-emissions) section
describes a subnet-context distribution model.

Those ideas should not be collapsed into each other. Halving changes emission rate, flow-based
emissions describe subnet allocation context, and Bittensor emissions names the broader reward
creation and distribution topic.

## Reader Boundary

Bittensor emissions should be read as tokenomics and reward-distribution vocabulary. The
[Emission](https://docs.learnbittensor.org/learn/emissions) page and
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) together
support the stable article-level point: emissions connect newly created TAO and subnet alpha tokens
to subnet roles and reward outcomes.

This article does not provide reward forecasts, staking strategy, validator recommendations, market
claims, or point-in-time network summaries. Exact observed values belong to the specific source and
context that produced them.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Bittensor
Emissions, that sequence changes how readers should interpret emission examples, subnet reward
observations, and supply-flow comparisons.

In localnet, Bittensor Emissions examples can be exercised in an isolated environment. Localnet
observations reflect local chain state and local configuration rather than production Bittensor
behavior.

On testnet, Bittensor Emissions behavior can be observed in a shared, non-production network.
Testnet examples are useful for checking interactions under shared chain conditions while staying
separate from mainnet state.

On mainnet, Bittensor Emissions describes live production emission behavior on the production
Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. An example or outcome from one environment should not be
read as evidence of production behavior in another environment.
