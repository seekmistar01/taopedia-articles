# Active UID

An active UID is a UID slot considered active within a specific Bittensor subnet. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#active-uid) describes an active UID as
a subnet slot that allows the associated participant to take part as a subnet validator or subnet
miner.

## Subnet Role

The official [Miners](https://docs.learnbittensor.org/miners) and
[Validators](https://docs.learnbittensor.org/validators) pages describe the two core subnet
participant roles: miners produce subnet work, and validators evaluate miner responses. An active
UID is the active slot through which a subnet participant is associated with one of those roles.

## Activity Tracking

Bittensor tracks each neuron's activity status through the metagraph. The
[Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph) documentation names this the
`active` field, recording whether a participant falls within the `activity_cutoff` window. The
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) reference
defines `activity_cutoff` as the number of blocks for stake to become inactive for Yuma Consensus
epoch purposes. A UID marked active in the metagraph is one whose participant has remained within
that block window.

## Relationship to Multiple Mechanisms

An active UID marks a usable participant slot inside a subnet. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms) notes
that validators must evaluate miners separately for each mechanism.

For readers, an active UID still identifies one subnet participant position. When a subnet runs more
than one incentive mechanism, that participant may be evaluated under separate mechanism rules
inside the same slot.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)

## Relationship to UID Slot

An active UID and a UID slot are related but different parts of Bittensor subnet vocabulary. The
[Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot) describes a
participant position inside a subnet, while the
[Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid) describes a
slot whose participant is considered active for subnet participation purposes.

For readers, a UID slot names the position itself, while an active UID names activity status within
that position. A participant can occupy a slot without meeting the activity conditions that make the
UID active in metagraph terms.

References: [Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid)

## Relationship to Register

An active UID and register are related but different parts of the subnet participation lifecycle.
The [Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register) describes the
subnet-entry process that grants a UID slot, while the
[Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid) describes
ongoing activity status after a participant already holds a slot.

For readers, register names how a participant enters a subnet and receives a UID, while an active
UID names whether that slot currently meets the activity conditions tracked on the metagraph.
Successful registration does not by itself guarantee that a UID will remain active across later
blocks.

References: [Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register),
[Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid)

## Relationship to Netuid

An active UID and a netuid are related but different identifiers inside Bittensor subnet vocabulary.
A netuid selects which subnet context is being discussed, while an active UID identifies an active
participant position within that selected subnet. The
[Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid) places netuid at the
subnet level, and the
[Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid) places active
status at the participant-slot level inside one subnet.

For readers, netuid answers which subnet market is in scope, while an active UID answers which slot
inside that subnet currently counts as active. The same UID number can appear in different subnets,
so active status must always be read together with the netuid that selects the subnet context.

References: [Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid),
[Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid)

## Relationship to Block

An active UID and a block are related through subnet activity timing. Bittensor tracks whether a UID
is active using block-based windows on the metagraph. The
[Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph) documentation describes the
`active` field, and the
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) reference
defines `activity_cutoff` as the number of blocks used for that activity evaluation.

For readers, a block names one unit of chain progression, while an active UID names a slot whose
participant has remained within the subnet's activity window measured across recent blocks. Activity
status can change as new blocks pass, even when the underlying UID slot assignment stays the same.

References: [Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)

## Relationship to Neuron

An active UID and a neuron are related but different parts of Bittensor subnet participant
vocabulary. A neuron names a registered participant record inside a subnet, while an active UID
identifies a participant slot that currently counts as active within that subnet's activity rules.
The [Glossary: Neuron](https://docs.learnbittensor.org/resources/glossary#neuron) describes a
registered subnet participant, and the
[Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid) describes a
UID that is currently active in a subnet.

For readers, a neuron names the participant record term, while an active UID names whether that slot
currently meets the subnet's activity threshold.

References: [Glossary: Neuron](https://docs.learnbittensor.org/resources/glossary#neuron),
[Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid)

## Relationship to Subnet Deregistration

An active UID and subnet deregistration address related but different parts of Bittensor subnet
lifecycle vocabulary. An active UID is a participant slot inside a subnet that currently meets the
subnet's activity rules, while subnet deregistration is the mechanism that removes an entire
non-immune subnet from the network when the subnet limit is reached and a new subnet registration
attempts to enter, selecting the existing subnet with the lowest EMA of alpha price for removal. The
[Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid) defines active
status at the participant-slot level inside one subnet, and the
[Subnet Deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration) documentation
describes the network-level subnet removal process.

Active UID status applies to individual participant slots inside a continuing subnet — a slot can
become inactive while the rest of the subnet keeps operating, and participant-level deregistration
can vacate one slot while other active UIDs in the same subnet remain. Subnet deregistration, by
contrast, removes the entire subnet: every active UID in that subnet simultaneously loses its
assignment because the subnet context no longer exists. Active UID names participant-level activity
status inside a continuing subnet; subnet deregistration ends the subnet that all those UIDs existed
in.

References: [Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid),
[Subnet Deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration)

## Reader Boundary

This article defines the term at a high level. It does not count active UIDs, list subnet
participants, or describe current subnet state. Readers should use official subnet references for
role-specific details.

## Relationship to Coinbase

An active UID and coinbase address different parts of subnet participation and emissions. An active
UID is a participant slot that currently meets a subnet's activity rules. The
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
documentation describes coinbase as the per-block protocol mechanism that drives TAO emission,
accumulates pending emissions, and triggers Yuma Consensus rounds at epoch boundaries, while the
[Glossary: Active UID](https://docs.learnbittensor.org/resources/glossary#active-uid) describes the
activity-status concept.

The connection is that activity status affects how a slot is treated in the consensus rounds
coinbase triggers, while coinbase is the recurring engine that runs those rounds and advances
emissions. Active UID names whether a participant slot currently counts as active under the subnet's
activity window; coinbase names the per-block process that fires the epoch calculations a subnet's
participants feed into. One is a per-slot activity-status measure read from the metagraph, the other
is the emission mechanism that drives the cycle — they describe different layers, not the same
operation.
