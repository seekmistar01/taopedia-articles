# Local Blockchain

A local blockchain is a private Bittensor chain used for development and testing. The term separates
isolated local experiments from public Bittensor network activity
([Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

The concept is environment vocabulary. It does not name a subnet role, a public network, or a
separate tokenomic mechanism.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For local
blockchain work on a subnet such as netuid 1, that sequence is central: a local blockchain belongs
to isolated development rather than shared testnet or production mainnet state
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

In localnet, a local blockchain is the intended environment for netuid 1 subnet and wallet
experiments. Local chain outcomes, including local Yuma and emission runs, reflect local
configuration and do not represent production Bittensor behavior.

On testnet, developers move from private local runs to a shared non-production network. Testnet
netuid 1 state is separate from both local-only chains and mainnet production history
([Emission](https://docs.learnbittensor.org/learn/emissions)).

On mainnet, Bittensor production activity on netuid 1 runs on the live production network rather
than on a developer's local blockchain instance
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Evidence from a local blockchain should not be read as proof of behavior on testnet or mainnet.

## Distinction from Local Wallet

A local wallet holds key material on the user's machine, while a local blockchain is the private,
isolated chain used to develop and test subnet incentive mechanisms. The wallet provides the keys;
the local blockchain is the isolated network those keys act on during development
([Glossary: Local Wallet](https://docs.learnbittensor.org/resources/glossary#local-wallet),
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)).

- **Local wallet** — locally stored keys on the user's machine.
- **Local blockchain** — a private, isolated chain for development and testing.

A local blockchain can exercise subnet incentive machinery in isolation. On a subnet such as netuid
1, developers can run local validator
[weights](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[emission](https://docs.learnbittensor.org/learn/emissions), and
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) paths without treating local
dividend or incentive outcomes as mainnet production state
([Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)).

That makes local blockchain the environment where netuid 1 incentive plumbing can be tested before
the same concepts are observed on testnet or mainnet. Local Yuma runs settle against local chain
state; they do not prove live miner
[incentives](https://docs.learnbittensor.org/resources/glossary#incentives) or validator
[dividends](https://docs.learnbittensor.org/resources/glossary#emission) on the production network
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Fast-block local mode can accelerate epoch feedback for that testing, but the result still belongs
to local evidence unless the example is repeated on testnet or mainnet
([Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)).

## Distinction from Testnet

A local blockchain is a private, isolated chain for development and testing. Testnet is the shared
non-production Bittensor network where developers exercise behavior before mainnet
([Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

- **Local blockchain** — private isolated chain under the developer's control.
- **Testnet** — shared non-production network with separate state from local runs.

Subnet examples on one environment should not be read as proof of behavior on the other
([Introduction to Bittensor: Subnet development](https://docs.learnbittensor.org/learn/introduction#subnet-development)).

## Distinction from Localnet

Localnet is the local development environment label in Bittensor network vocabulary. A local
blockchain is the private chain used inside that environment
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)).

Localnet names the network context. A local blockchain names the isolated chain that supports
development and testing in that context.

- **Localnet** — the local development environment label.
- **Local blockchain** — the private chain used for isolated development and testing.

## Distinction from Public Subtensor

Public Subtensor is public access to the Subtensor chain layer. A local blockchain is a private
chain used for local development and testing
([Glossary: Public Subtensor](https://docs.learnbittensor.org/resources/glossary#public-subtensor),
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)).

Local blockchain results should not be used as evidence about public Subtensor availability or
production chain behavior.

- **Public Subtensor** — public access to the Subtensor chain layer.
- **Local blockchain** — private isolated chain for development and testing.

## Distinction from Mainchain

Mainchain names the primary Bittensor network. A local blockchain is an isolated private chain for
development and testing
([Glossary: Mainchain](https://docs.learnbittensor.org/resources/glossary#mainchain),
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)).

A local blockchain can model Bittensor behavior, but it does not replace mainchain production
network context.

- **Mainchain** — the primary Bittensor network.
- **Local blockchain** — private isolated chain for development and testing.

## Distinction from Subtensor

Subtensor is Bittensor's chain layer and system of record. A local blockchain is a private chain
used to develop and test against that model locally
([Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)).

Subtensor names the production chain layer. A local blockchain names an isolated test chain that can
exercise similar behavior without being public Subtensor state.

- **Subtensor** — Bittensor's chain layer and system of record.
- **Local blockchain** — private isolated chain for development and testing.

## Distinction from Bittensor Networks

A Bittensor network and a local blockchain name an environment set and one private chain. Bittensor
networks covers mainnet, testnet, and localnet, while a local blockchain is the private, isolated
chain used for development and testing
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)).

- **Bittensor networks** — the full set of mainnet, testnet, and localnet.
- **Local blockchain** — the private chain used for development and testing.

## Distinction from Yuma Consensus

Local blockchain names an isolated development chain used for localnet testing. Yuma Consensus can
run on a local blockchain during testing, but localnet results are isolated examples and should not
be treated as mainnet outcomes
([Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Local blockchain** — isolated development chain for localnet.
- **Yuma Consensus** — on-chain settlement that turns validator weights into emission shares. Local
  blockchain names a development chain; Yuma processes validator weights into rewards
  ([Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions)).

## Distinction from Non-fast Blocks

Non-fast blocks and local blockchain sit at different precision. Local blockchain names the private
development chain itself, while non-fast blocks name one timing mode used inside that local
environment
([Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)).

- **Local blockchain** — the isolated development chain.
- **Non-fast blocks** — the default-timing block production mode on that chain.

## Distinction from Private Chain

Local blockchain context means the chain is private to the development environment. Work performed
there can help test Bittensor components and subnet behavior without making claims about public
network state
([Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

That distinction is the point of the term. A local result can be useful evidence for a local test,
but it is not the same kind of evidence as behavior observed in shared network contexts.

The environment label should therefore stay attached whenever a local blockchain example is used.

Localnet is for development and testing in a fully user-controlled environment
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)). That purpose
matches the local blockchain glossary entry: both frame local chain activity as isolated development
context rather than public network activity.

This also changes how local results should be read. A local blockchain can show whether a component
or subnet mechanism behaves as expected inside the private test environment, but it does not prove
the same result on testnet or mainnet
([Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

## Distinction from Block Timing

Fast blocks and non-fast blocks are local-chain timing terms. Fast blocks make local block
production faster for testing, while non-fast blocks keep the ordinary twelve-second local interval
([Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks),
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)).

Both terms belong under local blockchain context. They describe timing behavior inside an isolated
local chain rather than changing the meaning of mainnet or testnet.

Adjusting local block timing does not turn a local chain into mainnet or testnet evidence
([Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks),
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)).

Localnet block cadence can be either one block every 0.25 seconds in fast-block mode or one block
every 12 seconds in non-fast-block mode
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)). That makes
block mode part of the evidence label for local examples.

Mainnet and testnet use one block every 12 seconds
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)). A local
fast-block result can therefore arrive much faster than a public-network example without changing
what the underlying mechanism means.

For readers, the timing distinction is practical. Fast blocks shorten feedback loops during local
testing; non-fast blocks keep the local chain closer to the ordinary 12-second cadence used outside
fast-block local mode.

## Distinction from Development Path

The Bittensor introduction describes subnet development as moving from localnet to testnet and then
mainnet
([Introduction to Bittensor: Subnet development](https://docs.learnbittensor.org/learn/introduction#subnet-development)).

Local blockchain belongs to the isolated local part of that path. Testnet provides shared
non-production context, and mainnet provides the primary network context, so local observations
should stay attached to their environment.

That sequence gives local blockchain a clear scope. It is useful before shared testing because it
lets development work happen without relying on testnet state or mainnet value
([Introduction to Bittensor: Subnet development](https://docs.learnbittensor.org/learn/introduction#subnet-development),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Once an example moves to testnet or mainnet, it no longer has the same evidence status as a local
blockchain example. The network label, block cadence, and value context should move with the claim.

## Reader Boundary

Local blockchain does not name a tutorial, node-operation guide, public network status report, or
source for public chain data
([Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Local blockchain identifies a private, isolated development chain, while localnet, testnet, mainnet,
Public Subtensor, Mainchain, and Subtensor describe neighboring parts of the system.
