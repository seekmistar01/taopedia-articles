# Epoch

An epoch is the consensus period inside a Bittensor subnet
([Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)). It is tied to subnet
tempo and marks the timing window in which subnet consensus work is executed.

The term is useful because it separates timing from scoring. An epoch names the period in which
consensus processing occurs; the subnet's incentive mechanism and validator weights explain what is
being evaluated inside that period.

## Consensus Period

Epoch is timing vocabulary for subnet consensus. It does not name the subnet task, miner output, or
a single validator's evaluation. It names the period in which the subnet's consensus mechanism
processes the relevant validator signals
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Epoch is a boundary term for timing: it tells the reader when consensus work is grouped, not what
useful work the subnet is designed to evaluate.

That distinction keeps epoch prose from becoming a reward explanation. Consensus can produce
outcomes during an epoch, but the epoch itself is the period, not the scoring rule.

## Validator Signal Role

Epochs matter because Bittensor consensus aggregates validator weight signals into subnet outcomes.
In [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus), those weights are
processed over validator stake during the epoch window.

An epoch is not a standalone quality score. It is the timing period for processing consensus inputs,
and its outcomes depend on the validator weight set submitted within it.

Validator weights give the epoch its consensus content. Without the weight signals and stake
context, the epoch is only a timing container
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Multiple Mechanisms

When a subnet uses multiple incentive mechanisms, epoch language still needs mechanism context. The
[Multiple Incentive Mechanisms](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)
documentation explains that each mechanism can carry independent consensus calculations.

Epoch remains a consensus-period term, but the relevant mechanism determines which evaluation path
applies during that period.

This keeps multiple-mechanism prose precise. The epoch is the timing frame; the mechanism identifies
which work-and-evaluation track is being discussed.

## Kappa Sets the Stake Threshold Inside Each Epoch Pass

The hyperparameter [kappa](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#kappa)
sets how much validator stake must support a weight level before that level becomes the clipping
ceiling during an epoch's Yuma pass. With the default kappa of 0.5, at least half of stake-weighted
validator support must agree that a miner deserves weight above a given level before more generous
weights are left unclipped
([Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

Kappa therefore belongs to epoch-level consensus mechanics rather than to wall-clock timing alone.
The same tempo-defined epoch can produce different clipping outcomes if validator stake distribution
changes between passes.

## Distinction from Tempo

Epoch and tempo are related timing terms on a subnet. Tempo is the hyperparameter that counts how
many blocks make up one consensus cycle, while epoch names the recurring consensus period anchored
to that tempo boundary ([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)).

- **Tempo** — block count for one consensus cycle.
- **Epoch** — the consensus period spanning submission and settlement within that cycle.

Validator weights can be updated across the tempo, but Yuma outcomes tied to that pass finalize at
the epoch boundary when the tempo completes
([Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

## Distinction from Coinbase

Coinbase is the per-block emission operation that accumulates pending emissions and triggers Yuma
Consensus at epoch boundaries. Epoch names the recurring consensus period; coinbase names the
operational step that processes pending emissions when that boundary arrives
([Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)). Coinbase therefore
concerns emission processing at the boundary, while epoch frames the full period during which
validator weights can enter the pass ([Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Epoch** — concept vocabulary for this article's topic.
- **Coinbase** — related term at a different layer of the same flow.

The two labels answer different reader questions on the same path
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## Distinction from Yuma Consensus

Epoch names the consensus period from submission through settlement. Yuma Consensus turns submitted
validator weights into miner incentives and validator dividends when that period ends
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch),
[Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

## Distinction from Block

A block is one chain record. An epoch spans the block count set by tempo, so block height locates
one moment inside the wider consensus pass
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)).

Use block for one inclusion point in chain history. Use epoch for the full submission-to-settlement
window on the selected subnet.

## Distinction from Emission

Emission names injection and distribution together. Epoch names the consensus window in which
validator signals are gathered before participant-facing distribution is credited
([Emission](https://docs.learnbittensor.org/learn/emissions),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch),
[Emission: Distribution](https://docs.learnbittensor.org/learn/emissions#distribution)). Activity
cutoff is read with last update inside the same epoch to decide whether a neuron counts inactive
during that pass
([Subnet Hyperparameters: Activity Cutoff](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#activitycutoff),
[Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

- **Epoch** — concept vocabulary for this article's topic.
- **Emission** — related term at a different layer of the same flow.

The two labels answer different reader questions on the same path
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## Distinction from Kappa

Kappa sets the stake threshold used during an epoch's Yuma pass. Epoch names the period in which
that pass runs
([Subnet Hyperparameters: Kappa](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#kappa),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)). Kappa vocabulary belongs to
clipping support inside the pass; epoch vocabulary names the pass itself.

- **Epoch** — concept vocabulary for this article's topic.
- **Kappa** — related term at a different layer of the same flow.

The two labels answer different reader questions on the same path
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## Distinction from Consensus Score

Consensus score names the stake-weighted benchmark used when clipping validator weights during an
epoch. Epoch names the timing window for that consensus pass
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch),
[Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping)).

## Distinction from Non-fast Blocks

Non-fast blocks and an epoch describe different things: a local timing mode and a block-counted
interval. Non-fast blocks keep the ordinary local twelve-second cadence at which blocks are
produced, while an epoch is measured as a number of blocks, so the cadence mode changes how long an
epoch takes in wall-clock time without changing its block count
([Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)).

- **Non-fast blocks** — the local cadence at which blocks are produced.
- **Epoch** — a block-counted interval, independent of that cadence.

Readers should keep period vocabulary separate from settlement vocabulary. An epoch names the
consensus window from weight submission through settlement; Yuma names the mechanism that converts
submitted weights into emission shares when that window closes
([Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Epoch length is set by tempo and subnet hyperparameters. Yuma runs at epoch boundaries to apply
stake weighting, clipping, bonding, and dividend allocation
([Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Tempo** — hyperparameter setting how many blocks compose one epoch.

## Reader Boundary

Epoch is the right term when the focus is a subnet consensus period. If the focus is cadence,
[tempo](https://docs.learnbittensor.org/resources/glossary#tempo) is more precise. If the focus is
aggregation logic, [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) is more
precise. If the focus is evaluation signals themselves, validator weights are more precise
([Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)).

These boundaries keep epoch focused on consensus timing, not a subnet task description or a complete
account of reward outcomes.
