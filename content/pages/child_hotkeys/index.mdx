# Child Hotkeys

A child hotkey is a separate validator key that a delegate validator authorizes to validate on its
behalf in a selected subnet. The documentation calls the originating delegate hotkey the parent
hotkey, and describes child hotkeys as keys the parent re-delegates a portion of its stake to so
they can validate in its place.

## Parent and Child

The parent hotkey is an existing delegate that already holds stake. A child hotkey is a different
key that the parent points a share of that stake toward. The documentation explains that the child
then validates on behalf of the parent, so the parent key does not have to sign every validation
operation itself. This is a delegation relationship between two keys rather than a transfer of
ownership of the underlying stake.

## Subnet Context

Child hotkeys are configured for a specific subnet, identified by netuid; for example, a validator
can assign children for netuid 1 and a different set for another netuid. The documentation states
that a parent may have at most five child hotkeys on a given netuid, and that the same parent can
hold a separate set of children on a different netuid. The configuration is therefore independent
per subnet.

## Child Take

Child take is the share of rewards a child hotkey keeps for validating on the parent's behalf. The
documentation describes it as subnet-specific, ranging from 0 (0%) to 0.18 (18%), with a default of
0, and paid from the dividends the parent's stake would otherwise receive on that subnet. A child
hotkey can also hold a separate delegate take, so the child take rate and the delegate take rate are
described as independent settings.

## Why Validators Use Them

The documented rationale is key security. Validating through children lets the parent delegate
hotkey stay in a secure location while each child signs operations for its assigned subnet,
optionally on a different machine. The documentation notes that if an attacker steals a child
hotkey, only the subnets where that child is used are exposed, rather than the full parent key,
which limits the impact of a compromised validation key.

## Parent Delegate Keeps the Stake

The [Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys) documentation
describes a parent delegate hotkey that already holds stake and assigns a childkey proportion for a
specified `netuid`. The parent remains the delegated stake holder; the child configuration applies
the proportion for that subnet only.

Readers should attach both the parent hotkey and the `netuid` to a child-hotkey claim. The same
parent can use a different child set on another subnet without mixing the two configurations.

## Parent Authorization Boundary

A child hotkey signs validation work because the parent delegate hotkey authorized it for a subnet.
That authorization does not make the child the owner of the parent's stake or the general manager of
the parent's wallet. The child-hotkey documentation keeps the relationship scoped to validating on
the parent's behalf inside the selected `netuid`.

For readers, child-hotkey authority should therefore be read as delegated operational signing, not
as custody transfer. The parent-child relationship explains who may validate for a subnet; it does
not replace the wallet authority model for balances, stake ownership, or coldkey control.

## Child Take From Parent Dividends

Child take is the percentage of the parent's dividends that a child hotkey keeps for validating on
the parent's behalf. The same documentation gives a subnet-specific range from 0 to 0.18, with a
default of 0, and states that the child receives that percentage of the parent's dividends on the
selected subnet.

That wording keeps child take inside dividend-sharing vocabulary for one `netuid`. It is separate
from delegate take settings that apply more broadly to validator-delegator emission sharing.

## Separate Child Hotkeys per Subnet

Validators can run separate child hotkeys per subnet so the parent delegate hotkey does not need to
be used everywhere. The documentation gives the security reason plainly: if one child key is
compromised, only the subnets where that child is configured are exposed rather than every subnet
the parent uses.

That makes child hotkeys a per-subnet operational choice. The parent delegate keeps stake; each
child hotkey can sign validation work for the subnets where the parent authorized it.

The five-child limit and 18% take ceiling remain protocol limits from the documentation rather than
live values reported by this concept article.

## Distinction from Multiple Incentive Mechanisms

Child hotkeys name delegated validation keys a parent hotkey authorizes inside specific subnets,
while multiple incentive mechanisms names parallel evaluation tracks with separate weight matrices
and bond pools on one subnet
([Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

Delegation-key vocabulary and mechanism-scope vocabulary answer different questions. A child hotkey
still names a delegated validation key; weight-related actions it signs should be read with that
mechanism's matrix context
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)).

- **Child hotkeys** — delegated validation keys authorized by a parent hotkey.
- **Multiple incentive mechanisms** — parallel evaluation tracks with separate weight matrices.

## Distinction from Coldkey Swap

Child hotkeys name delegated validation keys a parent hotkey authorizes inside specific subnets,
while coldkey swap names the on-chain identity migration that moves all hotkeys associated with a
coldkey to a new coldkey address without resetting subnet registrations or removing existing stake
([Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys),
[Coldkey Swap](https://docs.learnbittensor.org/keys/coldkey-swap)).

Hotkey-delegation vocabulary and coldkey-identity vocabulary answer different questions. Coldkey
swap governs which coldkey controls associated hotkeys; child hotkeys govern which validation
delegates a parent hotkey authorized within each subnet.

- **Child hotkeys** — delegated validation keys authorized per subnet.
- **Coldkey swap** — coldkey identity migration preserving hotkey state.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from local testing to testchain and then mainchain. For child
hotkeys, that sequence changes how readers should interpret evidence about delegated validation
keys.

In local testing, child hotkey configuration can show whether a parent hotkey is able to authorize a
child to validate inside an isolated subnet environment. That is useful for checking the
parent-child delegation relationship, but it remains isolated feedback from a local environment.

On testchain, child hotkey behavior can be observed in a shared testing network. This gives stronger
evidence about how a child key validates on a parent's behalf and how child take interacts with
parent dividends than a private local run, while still keeping the result separate from production
mainchain activity.

On mainchain, child hotkeys belong to live subnet validation state. The
[Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys) documentation describes
child hotkeys as keys the parent re-delegates a portion of its stake to so they can validate in its
place, so production child hotkey context is tied to active delegation inside a live subnet.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A child hotkey configuration observed in one environment
should not be read as identical evidence about delegated validation activity in another environment.

For readers, this keeps child hotkey examples from sounding stronger than their environment
supports. Local, testchain, and mainchain contexts can all describe parent-child validation
delegation, but they do not carry the same interpretive weight.

## Relationship Scope

Child hotkeys should be read as a key-relationship concept, not as a general staking or delegation
recommendation. Wallet and key documentation separates key custody from staking relationships, so
claims about child hotkeys should preserve whether the source is discussing account structure or
delegated stake behavior
([Bittensor Wallets](https://docs.learnbittensor.org/getting-started/wallets),
[Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation)).

That distinction helps readers avoid treating every hotkey relationship as evidence about validator
selection, emissions, or reward outcomes.

## Five-Child Limit Applies Per netuid

[Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys) documentation states that
a parent may have at most five child hotkeys on a given netuid, while the same parent can hold a
different child set on another netuid. The limit is therefore subnet-local rather than a single
global count across every subnet the parent uses.

Readers should name the netuid when citing the five-child rule. A parent at the limit on netuid 1
does not by itself describe child configuration on another subnet.

## Child Keys Can Operate From Separate Hosts

The child-hotkeys documentation describes validators using children so the parent delegate hotkey
can stay in a secure location while each child signs operations for its assigned subnet, optionally
on a different machine. Child hotkey vocabulary therefore includes an operational deployment choice,
not only a reward-sharing setting.

That security rationale keeps child exposure subnet-scoped. A compromised child key affects only the
subnets where that child is configured, as described in the child-hotkeys reference.

## Changing Children Is Rate-Limited

Setting or revoking children is not something a parent delegate can do at will, block by block.
Official documentation describes a rate limit that permits this action roughly once every 150
blocks, about 30 minutes, and a separate limit allowing a given child hotkey's take rate to be
adjusted only once every 30 days
([Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys),
[Chain Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

Those two limits answer different questions. The 150-block limit paces how often the child set
itself can change, while the 30-day limit paces how often any one child's take percentage can be
adjusted once it is already assigned.

## A Parent Needs a Minimum Stake to Set Children

Setting child hotkeys also requires the parent to clear a minimum total stake. Documentation states
the requirement as 1,000 TAO worth of alpha on mainnet and 100 TAO worth of alpha on testnet,
measured by summing the parent's alpha stake across every subnet it holds, not only the subnet where
children are being set ([Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys)).

That network-wide measurement means a parent's eligibility to set children does not depend only on
its position in one subnet. A parent with modest stake on the target subnet but larger alpha
holdings elsewhere can still clear the threshold once all of its subnet positions are counted
together.

## Distinction from Validator Take

A child hotkey is the key a parent delegate authorizes to validate on its behalf using re-delegated
stake. Validator take is the commission skimmed from validation dividends before the remainder
reaches stakers
([Glossary: Validator Take %](https://docs.learnbittensor.org/resources/glossary#validator-take-),
[Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys)).

Worker identity and commission rate answer different questions. The child hotkey names who signs;
validator take names what share of delegated returns the validator keeps at distribution.

## Distinction from Childkey Take

Childkey take is the fraction a child hotkey retains for validating on behalf of a parent hotkey.
Validator take is the broader commission on delegated-stake dividends before nominators receive the
remainder ([Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys),
[Glossary: Validator Take %](https://docs.learnbittensor.org/resources/glossary#validator-take-)).

A single delegated position can encounter both rates on different relationships: validator take on
the delegate side and childkey take where stake is re-delegated to a child on a named netuid.

## Distinction from Delegate

A delegate is a validator that can receive nominated stake. A child hotkey is an authorized worker
key the parent delegate assigns so validation can be signed on one subnet without using the parent
hotkey directly ([Glossary: Delegate](https://docs.learnbittensor.org/resources/glossary#delegate),
[Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys)).

Role vocabulary names the stake-holding parent; child-hotkey vocabulary names the routed validation
worker configured under that parent on a selected subnet.

## Distinction from Register

Register is the subnet-entry process that grants a UID slot through the dynamic registration burn. A
child hotkey must already be registered on the netuid where it validates
([Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register),
[Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys)).

Entry vocabulary names how a hotkey obtains a slot; child-hotkey vocabulary names how a registered
parent routes stake and signing authority to another registered key on that subnet.

## Child Hotkeys Route a Parent's Stake Proportion

[Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys) documentation describes a
parent delegate hotkey that already holds stake and assigns a childkey proportion for a specified
subnet. The parent remains the delegated stake holder; the child configuration applies a routed
share of that stake so the child can validate on the parent's behalf inside that subnet only
([Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation)).

That routing is subnet-local rather than a transfer of ownership. The same parent can authorize a
different child set on another subnet, and each child proportion applies only where the parent
configured it ([Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid)).

For readers, child hotkey vocabulary therefore includes stake routing as well as signing authority.
The child validates using re-delegated stake from the parent position, while dividends and child
take still flow through the parent-stake relationship described in child-hotkey documentation
([Glossary: Dividends](https://docs.learnbittensor.org/resources/glossary#emission)).

Child take is the percentage of the parent's dividends the child keeps for that routed validation
work. It is paid from dividends the parent's stake would otherwise receive on the selected subnet
and is separate from delegate take settings that apply more broadly to validator-delegator sharing
([Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys),
[Glossary: Validator Take %](https://docs.learnbittensor.org/resources/glossary#validator-take-)).

The parent-child link is operational delegation, not custody transfer. A compromised child key
exposes only the subnets where that child is configured, which is why validators use children to
keep the parent delegate hotkey in a more secure location while routing stake-backed validation per
subnet ([Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys)).

## Distinction from Hotkey

For netuid 1 examples, a child hotkey and a hotkey answer different signing questions in validator
workflows. A hotkey is the operational wallet key used for subnet actions, while a child hotkey is a
separate authorized key a parent delegate assigns so validation can be signed on one subnet without
using the parent hotkey directly
([Glossary: Hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey),
[Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys)).

The parent delegate hotkey remains the stake holder. The child hotkey receives routed stake and
signing authority only where the parent configured it, which is why child-hotkey claims should name
both the parent hotkey and the subnet market involved
([Glossary: Delegate](https://docs.learnbittensor.org/resources/glossary#delegate),
[Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid)).

- **Hotkey** — the operational key tied to a wallet role.
- **Child hotkey** — an authorized worker key under a parent delegate on one subnet.

## Distinction from Netuid

Child hotkeys and netuid answer different scoping questions. Netuid names which subnet market the
configuration applies to, while child hotkey names the authorized validation key the parent assigns
inside that market ([Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid),
[Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys)).

Documentation limits parents to five child hotkeys per netuid and allows a different child set on
another netuid. A child-hotkey statement therefore needs the netuid attached before readers compare
limits, take settings, or routed stake shares across subnets
([Chain Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

Netuid vocabulary sets the subnet boundary. Child-hotkey vocabulary names the operational key
working inside that boundary for a parent delegate on netuid 1
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

## Distinction from Chain Rate Limits

Child hotkeys and chain rate limits govern different parts of the same configuration workflow on
netuid 1. Child hotkeys name the authorized keys and routed stake shares a parent delegate assigns,
while chain rate limits pace how often those assignments or take percentages can change on chain
([Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys),
[Chain Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

The child-hotkeys documentation pairs a five-child limit per netuid with separate pacing rules for
changing the child set and for adjusting an individual child's take percentage. Rate-limit
vocabulary therefore belongs to update timing, not to whether a child key exists at all
([Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys)).

Readers comparing operational freedom should keep the key-routing concept separate from the cooldown
windows that govern when routing or take settings may be edited again on the selected subnet
([Glossary: Validator Take %](https://docs.learnbittensor.org/resources/glossary#validator-take-)).

## Distinction from Subnet Owner Take

On a subnet such as netuid 1, a subnet owner's hotkey can be authorized as a child hotkey — a
delegate re-delegates a portion of its stake to it so the owner validates on netuid 1 and keeps a
childkey take — but that role is separate from the subnet owner take. The child hotkey arrangement
is an optional, per-subnet validation role with its own childkey take on the re-delegated stake,
while subnet owner take is the fixed share of that subnet's total alpha emission allocated to the
owner of netuid 1 for creating and running it, documented at 18% and paid whether or not any
validator re-delegates to the owner
([Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys),
[Emission: Distribution](https://docs.learnbittensor.org/learn/emissions#distribution)).

On netuid 1 a subnet owner can receive both, but they arrive through different paths: a childkey
take only when stake is re-delegated to the owner's hotkey for validation on that subnet, and the
owner take automatically from the subnet's emission split
([Delegation](https://docs.learnbittensor.org/staking-and-delegation/delegation)).

- **Child hotkey** — an optional validating key on a subnet such as netuid 1, earning a per-subnet
  childkey take on re-delegated stake.
- **Subnet owner take** — the fixed 18% protocol share of that subnet's emission paid to its owner.

## Distinction from Stake Weight

Child hotkeys route a parent's stake proportion so a child can validate on its behalf. Stake weight
is the computed influence measure used when validator signals count in subnet consensus
([Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys),
[Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight)).

Routing vocabulary names which key performs validation with a delegated share; stake weight names
how strongly that delegate's evaluations count after consensus processing
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

A child configuration changes who signs validation work; it does not by itself rename stake-weight
influence vocabulary on the selected netuid.

## Reader Boundary

This page defines the concept at a high level. It does not report which child hotkeys a parent
currently has, current take values, current stake proportions, or wallet ownership. Those are live
chain state and change over time; they should be checked for the relevant netuid with the Bittensor
CLI when current data is needed. The numeric rules described here, such as the five-child limit and
the 18% take ceiling, are protocol limits from the documentation.
