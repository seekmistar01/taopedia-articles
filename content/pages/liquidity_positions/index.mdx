# Liquidity Positions

Liquidity positions are range-based liquidity commitments for Bittensor subnet trading pools. The
official [User Liquidity Positions](https://docs.learnbittensor.org/liquidity-positions)
documentation describes the feature as a way to provide trading liquidity for specific subnets
within specified price ranges for a subnet's alpha token.

## Subnet Liquidity Context

Liquidity positions sit in the subnet tokenomics context. Bittensor subnets use TAO and subnet
alpha-token pools, and a liquidity position adds liquidity to that trading context rather than
directly evaluating subnet work. The
[User Liquidity Positions](https://docs.learnbittensor.org/liquidity-positions) source describes the
purpose as supporting trades between TAO and a subnet's alpha token and helping create more liquid
market conditions for the subnet.

This makes a liquidity position different from an ordinary subnet role. It is not mining,
validating, or creating a subnet; it is a liquidity-supply position around a subnet's token market.

## Price Range Context

The feature is based on concentrated liquidity. A position is created for a price range, and it can
accrue fees when trading activity happens within that range. The official source describes liquidity
positions as accumulating fees from subnet trading activity inside the defined price range.

For readers, the useful idea is that a liquidity position is range-sensitive. It is not just a
single static pool deposit; its effect depends on the relationship between the position's range and
the subnet token's trading context.

## Difference from Staking

Liquidity positions and staking are separate concepts. The
[User Liquidity Positions](https://docs.learnbittensor.org/liquidity-positions#liquidity-positions-vs-staking)
source explains that staking supports validators and affects validator stake, consensus power, and
emissions. Liquidity positions instead provide trading liquidity and can accrue fees from activity
in the selected range.

This distinction matters because both concepts involve TAO and subnet alpha context, but they answer
different questions: staking supports validator participation, while liquidity positions support
subnet token-market liquidity.

## Subnet Pool Handoff

Liquidity positions connect to the broader subnet pool design described in the official
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets#liquidity-pools)
documentation. Subnet pools hold TAO and alpha reserves that support staking conversions and trading
context for a subnet's token market.

For readers, a liquidity position is therefore a range-based contribution inside that pool context,
not a substitute for mining, validation, or emission evaluation. Pool reserves shape token-market
conditions; liquidity positions add concentrated liquidity within those conditions.

References:
[Understanding Subnets: Liquidity pools](https://docs.learnbittensor.org/subnets/understanding-subnets#liquidity-pools),
[User Liquidity Positions](https://docs.learnbittensor.org/liquidity-positions)

## Network and Pool Context

Liquidity-position examples depend on both the network and the subnet pool being discussed. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, while the
[User Liquidity Positions](https://docs.learnbittensor.org/liquidity-positions) documentation places
positions inside subnet alpha-token trading ranges.

In localnet context, a liquidity-position example mainly explains the shape of concentrated
liquidity. It can show why a range matters without implying that the same position, fees, or pool
depth exists on a public network.

In testnet context, the example belongs to a shared testing environment. It can help illustrate
public test behavior, but it remains separate from production mainnet liquidity and fee activity.

In mainnet context, a liquidity-position example refers to live subnet market state. The selected
subnet, range, pool depth, trading activity, and fee context all affect how a concrete example
should be read.

This network context keeps liquidity positions from being read as earnings promises. The concept
describes a range-based contribution to subnet trading liquidity; the outcome of any position
depends on current pool state and trading activity.

It also keeps liquidity positions separate from staking. Staking supports validator participation,
while liquidity positions support subnet token-market liquidity. Both can involve TAO and alpha
tokens, but they answer different questions inside Bittensor.

Concrete examples should therefore name what they are describing: the network, the subnet, the
position range, and whether the example is about market depth, fee accrual, or the staking
comparison. Those details decide whether the example is teaching the concept or describing a live
position.

That boundary makes the article useful without turning it into trading or operations advice.

## Relationship to Slippage

Liquidity positions and slippage are related but different parts of Bittensor subnet-market
vocabulary. A liquidity position names concentrated liquidity a participant adds to a subnet pool,
while slippage names the price impact a trade experiences when it moves against available pool
depth. The [User Liquidity Positions](https://docs.learnbittensor.org/liquidity-positions)
documentation describes adding concentrated liquidity to a pool, and the
[Glossary: Slippage](https://docs.learnbittensor.org/resources/glossary#slippage) describes the
difference between expected and executed price as depth is consumed.

For readers, a liquidity position names supplied pool depth, while slippage names a cost that
depends on how much depth a trade consumes. Adding liquidity changes the depth that slippage is
measured against; it is not itself the slippage.

References: [User Liquidity Positions](https://docs.learnbittensor.org/liquidity-positions),
[Glossary: Slippage](https://docs.learnbittensor.org/resources/glossary#slippage)

## Reader Boundary

Liquidity positions should not be read as an earnings promise, trading advice, or operations guide.
Exact ranges, fees, token composition, and subnet-specific availability depend on current official
references and the subnet's market state.

## Relationship to Multiple Mechanisms

Liquidity positions supply trading liquidity for a subnet's TAO and alpha pool. They do not evaluate
miner work under an incentive mechanism. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms) notes
that validators must evaluate miners separately for each mechanism.

For readers, a liquidity position remains a market-liquidity role. Multiple incentive mechanisms
inside the same subnet do not turn a liquidity position into a miner or validator evaluation role.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)

## Relationship to Protocol Alpha

Liquidity positions and protocol alpha address related but different parts of Bittensor subnet pool
vocabulary. A liquidity position is a user-deposited position on a subnet's automated market maker
that provides TAO and alpha liquidity to the pool and earns fees from swaps, while protocol alpha is
the protocol-owned alpha cached per subnet from chain-side TAO buys during reserve injection — a
protocol-held position, not a user-deposited one. The
[User Liquidity Positions](https://docs.learnbittensor.org/liquidity-positions) documentation
describes user-deposited positions and how they earn fees from subnet pool activity, and the
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)
defines protocol alpha as the per-subnet protocol-owned alpha balance produced during reserve
injection.

Both liquidity positions and protocol alpha involve alpha within the same subnet pool system, but
they represent different ownership contexts and origins. A liquidity position is user-managed — the
holder decides when to deposit or withdraw liquidity and earns fees on that deposit. Protocol alpha
is protocol-managed — it accumulates from reserve injection activity and is cleared when the subnet
dissolves rather than being available for user withdrawal. Liquidity position names a user-side pool
role; protocol alpha names a protocol-side pool balance.

References: [User Liquidity Positions](https://docs.learnbittensor.org/liquidity-positions),
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha),
[Emission: TAO reserve injection](https://docs.learnbittensor.org/learn/emissions#tao-reserve-injection)

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For liquidity
positions, that sequence changes how readers should interpret evidence about user pool contributions
and TAO-alpha reserve interactions.

In localnet, liquidity position mechanics can be tested in an isolated environment. Pool reserve
values and position outcomes in localnet reflect local configuration and do not represent production
subnet pool state.

On testnet, liquidity position behavior can be observed in a shared, non-production network. Testnet
pool reserves and position balances are separate from mainnet subnet liquidity state.

On mainnet, liquidity positions represent actual user contributions to live Bittensor subnet pools.
The [User Liquidity Positions](https://docs.learnbittensor.org/liquidity-positions) documentation
describes how pool shares, slippage, and alpha balances interact on the production network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A liquidity position example from one environment should
not be read as representing pool state in another environment.

For readers, this keeps liquidity position examples grounded in their network context. Localnet,
testnet, and mainnet each have separate pool histories, so position values belong to the specific
network being described.
