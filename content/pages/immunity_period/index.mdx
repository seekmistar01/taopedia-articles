# Immunity Period

An immunity period is the grace window after a participant registers into a Bittensor subnet. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#immunity-period) defines it as a
period during which newly registered subnet participants are protected from deregistration for poor
performance.

## Subnet Grace Period

The useful reader-facing idea is simple: a new subnet participant needs time to start contributing
before it can be removed for low performance. The official
[Mining](https://docs.learnbittensor.org/miners#immunity-period) page describes the immunity period
as a number of blocks that begins when a miner or validator registers into a subnet.

This makes immunity period a registration concept rather than a score. It says that a participant is
inside a temporary protection window; it does not say that the participant is performing well or
earning emissions.

## Relationship to Deregistration

Immunity period matters because Bittensor subnets have limited UID slots. The
[Mining](https://docs.learnbittensor.org/miners#miner-deregistration) page explains that, when a
subnet is full and a new registration arrives, the lowest-emission eligible participant can be
replaced. A participant still inside its immunity period is protected from that ordinary
low-performance removal path.

The official
[UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming#protection-from-trimming) page
uses the same boundary in a different context: recently registered UIDs are treated as temporally
immune during trimming.

## Hyperparameter Scope

Immunity period is a subnet hyperparameter measured in blocks. The official
[Subnet Configuration](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#immunityperiod)
page lists `ImmunityPeriod` as the value that defines the post-registration protection window.

Different subnet settings can therefore change how long the window lasts. Taopedia readers should
read the term as a configurable subnet rule, not as a universal promise that every subnet uses the
same number of blocks.

## Default Block Window

Bittensor's default immunity period is 4096 blocks, roughly 13.7 hours at typical block times,
according to the
[Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period).
Subnet creators can configure a different window through the `ImmunityPeriod` hyperparameter listed
in
[Subnet Configuration](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#immunityperiod).
Readers should treat 4096 blocks as the documented default, not a fixed value for every subnet.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from local testing to testchain and then mainchain. For an
immunity period, that sequence changes how readers should interpret block-window examples.

In local testing, an immunity period can show how a newly registered participant remains protected
inside an isolated subnet environment. That is useful for checking registration and displacement
logic, but it does not show how a live public subnet is handling production participants.

On testchain, immunity-period behavior can be observed in a shared testing network. This gives
stronger evidence about how the protection window interacts with other registrations, blocks, and
subnet rules than a private local run, while still keeping the result separate from production
mainchain activity.

On mainchain, immunity-period status belongs to live subnet state. The
[Subnet Configuration](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#immunityperiod)
reference lists `ImmunityPeriod` as a subnet setting, so production readings should keep the
selected subnet, current block context, and configured window attached.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. An immunity-period observation in one environment should
not be read as identical evidence about participant protection in another environment.

This distinction matters because immunity period is a temporary block-count boundary rather than a
participant quality signal. The network environment explains what kind of evidence the block window
represents, while the subnet setting explains how long that window lasts.

For readers, this keeps examples from overstating what they prove. Local, testchain, and mainchain
contexts can all contain immunity-period language, but they do not carry the same interpretive
weight.

## Relationship to Register

An immunity period and register are related but different parts of Bittensor subnet registration
vocabulary. The [Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register)
describes subnet entry as purchasing a UID slot through a dynamic registration burn in TAO, while
the [Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period)
describes a temporary protection window that begins after a participant registers into a subnet.

For readers, register names the entry process, while immunity period names the grace window that
follows that entry before ordinary low-performance removal rules apply.

References: [Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register),
[Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period)

## Relationship to Multiple Mechanisms

An immunity period applies within a subnet after a participant registers. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism) notes that each
subnet has one or more incentive mechanisms.

For readers, immunity period still names a temporary protection window inside one subnet market.
That protection is tied to subnet registration rules, not to a single mechanism label.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Incentive Mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism)

## Relationship to UID Slot

An immunity period and a UID slot are related through subnet displacement rules. The
[Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot) describes a
participant position inside a subnet, while the
[Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period)
describes a temporary protection window that can apply after a participant registers into that slot.

For readers, a UID slot names the occupied position, while immunity period names a post-registration
timing boundary that can protect that slot holder from ordinary low-performance removal while the
window remains active.

References: [Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period)

## Relationship to Netuid

An immunity period and a netuid are related but different parts of Bittensor subnet vocabulary. A
netuid selects which subnet context is in scope, while an immunity period describes a configurable
protection window inside that subnet after registration. The
[Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid) places netuid at the
subnet level, and the
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) reference
lists `ImmunityPeriod` as a subnet-specific setting.

For readers, netuid answers which subnet market is being discussed, while immunity period answers
how long a newly registered participant in that subnet may remain protected from ordinary
displacement rules. Different netuids can use different configured windows.

References: [Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#immunityperiod)

## Relationship to Block

An immunity period and a block are related through subnet timing measurement. Immunity period length
is defined in blocks rather than clock time alone. The
[Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period)
describes the protection window in block terms, and the
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block) describes a block as a
unit of chain data containing transactions on Subtensor.

For readers, each new block advances chain time and can reduce the remaining immunity window for a
recently registered participant. Immunity period should therefore be read as a block-count boundary,
not as a fixed wall-clock duration unless the surrounding network block timing is also in view.

References:
[Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)

## Relationship to Miner Deregistration

An immunity period and miner deregistration address related but different parts of subnet removal
vocabulary. The
[Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period)
describes temporary protection after registration, while
[Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)
describes how a low-performing miner can lose a UID slot when a full subnet accepts a new
registration.

For readers, immunity period names a protection window that can delay ordinary low-performance
removal, while miner deregistration names the miner-specific displacement path once that protection
no longer applies or when other eligibility rules are satisfied.

References:
[Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period),
[Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)

## Relationship to Subnet Deregistration

Immunity period and subnet deregistration share the immunity concept but apply it at different
levels of Bittensor registration vocabulary. The participant-level immunity period is a block-count
window that protects a newly registered miner or validator from within-subnet deregistration for low
performance, while subnet deregistration removes an entire non-immune subnet from the network when
the subnet limit is reached and a new subnet registration attempts to enter, selecting the subnet
with the lowest EMA of alpha price. The
[Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period)
describes the participant-level protection window, and the
[Subnet Deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration) documentation
describes the network-level subnet removal process.

The "non-immune" qualifier in subnet deregistration refers to a network-level protection that newly
registered subnets receive — separately from the participant-level immunity period. A subnet inside
its network-level protection window cannot be selected for subnet deregistration even if its EMA of
alpha price is the lowest among all subnets. The participant-level immunity period inside a subnet
and the subnet-level immunity protection are independent: a participant's registration protection
and the subnet's network-level protection are different things applied at different scopes.

References:
[Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period),
[Subnet Deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration)

## Reader Boundary

Immunity period is not a permanent exemption, a quality score, or a guarantee of rewards. It is a
temporary protection window tied to recent registration. Once the window expires, the participant is
again subject to the subnet's normal deregistration or trimming rules described in the official
documentation.

## Relationship to Coinbase

An immunity period and coinbase address different parts of a participant's subnet lifecycle. An
immunity period is the post-registration window that protects a participant from deregistration for
poor performance. The
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
documentation describes coinbase as the per-block protocol mechanism that drives TAO emission,
accumulates pending emissions, and triggers Yuma Consensus rounds at epoch boundaries, while the
[Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period)
describes the protection window.

The two are largely orthogonal: immunity period governs whether a slot can be removed, while
coinbase governs the emission cycle. A participant inside its immunity window holds a slot that
takes part in the coinbase-triggered epoch processing like any other slot; immunity does not change
how coinbase runs, it only shields the slot from low-performance removal for a time. Immunity period
answers how long a slot is protected from deregistration; coinbase answers how the per-block
emission cycle proceeds regardless of that protection.
