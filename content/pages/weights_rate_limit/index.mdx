# Weights Rate Limit

Weights rate limit is a per-subnet hyperparameter, listed in the documentation as `WeightsRateLimit`
and also referred to as `CommitmentRateLimit`, that sets how long, in blocks, a validator must wait
between [weight setting](https://docs.learnbittensor.org/resources/glossary#weight-vector) commits
on a subnet
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## What It Controls

The hyperparameter paces how often a single validator can resubmit
[validator weights](https://docs.learnbittensor.org/resources/glossary#weight-vector). It does not
define what those weights mean, how
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) aggregates them, or which
miners a validator should score. It only sets the minimum block interval between one weight commit
and the next from the same validator on the selected subnet.

That makes weights rate limit a submission-pacing rule. It limits how frequently a validator can
change its on-chain weight signal, rather than deciding whether a particular submission counts in
the current [epoch](https://docs.learnbittensor.org/resources/glossary#epoch).

## Documented Type, Default, and Setter

The [Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)
reference lists `WeightsRateLimit` / `CommitmentRateLimit` as a `u64` with a default of `100` and
the description "How long, in blocks, a validator must wait between weight commits on a subnet." It
is changed through the `sudo_set_weights_set_rate_limit` extrinsic, exposed in tooling as
`btcli sudo set --param weights_rate_limit`, and the reference marks the permission required to set
it as root rather than subnet owner
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Distinction from Activity Cutoff

Weights rate limit and
[activity cutoff](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#activitycutoff)
answer different timing questions. Activity cutoff is measured in blocks within each epoch and
governs whether a validator's weight submission arrives early enough to count for the current Yuma
Consensus pass. Weights rate limit is measured in blocks between commits and governs how soon the
same validator can submit again after a prior weight commit
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Activity cutoff** — whether a submission arrives inside the current epoch window.
- **Weights rate limit** — how long a validator must wait before the next commit.

## Distinction from Commit Reveal

[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal) changes when committed
weights become visible, while weights rate limit changes how frequently a validator can submit new
commits. Commit reveal is a visibility-timing mechanism; weights rate limit is a submission-pacing
hyperparameter ([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Distinction from Serving Rate Limit

The hyperparameters reference also lists
[ServingRateLimit](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#servingratelimit),
which rate-limits miner calls to `serve_axon` and `serve_prometheus`. That setting paces miner
endpoint registration, not validator weight commits
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## BTCLI Reads the Live Value

Any user can read subnet hyperparameters for a selected netuid with
`btcli subnet hyperparameters --netuid 1`. The command output includes the live `WeightsRateLimit`
value for that subnet at query time
([Subnet Hyperparameters: View hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#view-hyperparameters)).

Live values are network- and subnet-specific and can differ from the documented default, so a pacing
claim should be checked against the hyperparameter table for the relevant netuid and network rather
than assumed from the reference default alone.

## Pacing Commits Discourages Last-Moment Weight Flipping

The reason a minimum wait sits between commits is that without one, a validator could rewrite its
weights repeatedly to track or chase other validators' visible signals. A commit interval makes each
submission more deliberate, since a validator cannot immediately overwrite a commit it just made
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

For readers, this pairs naturally with Commit Reveal. Reveal delays when weights become visible,
while the rate limit slows how often they can change, so together they reduce the value of reacting
to freshly observed signals rather than evaluating independently
([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

That intent keeps the rate limit distinct from aggregation. It shapes the cadence of submissions
upstream, while Yuma Consensus still decides how whatever weights are current get combined into
miner incentives and validator dividends
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).
