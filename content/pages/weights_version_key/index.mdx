# Weights Version Key

Weights version key is a per-subnet hyperparameter that sets a minimum version a validator's weight
submission must carry. Documentation states that if the version key specified in a `set_weights`
call is lower than this system-wide setting, the transaction fails.

## What It Controls

The hyperparameter gates weight submissions by version. When a validator submits weights, the call
carries a version key, and that key must be at least the subnet's weights version key for the
transaction to succeed. It does not change the weights a validator chooses; it only decides whether
a submission is accepted based on the version it declares.

## Why Subnets Use It

Because a submission with a too-low version key fails, raising this setting effectively requires
validators to move to a matching version before their weights are accepted again. That gives a
subnet a way to roll forward expectations for validators in step, since stale submissions are
rejected rather than silently accepted.

## Default and Setting

The documentation lists a default of 0 and marks the parameter as owner-settable, so a subnet owner
can raise it for their subnet. A default of 0 means no minimum is enforced. The value in force is
per-subnet chain state and can differ from the default.

## Subnet Context

Weights version key is one of the subnet hyperparameters, the on-chain state variables that
configure a single subnet, for example netuid 1. Because each subnet carries its own
hyperparameters, the required version is defined per subnet and applies only to weight submissions
on that subnet.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For the weights
version key, that sequence changes which subnet hyperparameter state a reader should assume when
judging whether a `set_weights` submission meets the minimum version.

In localnet, version-key checks can be tested through isolated subnet configuration. Localnet
hyperparameter values do not represent production weight-submission rules on live subnets.

On testnet, the weights version key can be exercised on a shared non-production network. Testnet
hyperparameter state is separate from mainnet subnet data
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

On mainnet, the weights version key concerns live per-subnet settings on the selected
[netuid](https://docs.learnbittensor.org/resources/glossary#netuid) that gate accepted weight
submissions.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A weights-version example from one environment should not
be read as describing production hyperparameter state on another network.

## Distinction from Validator Permit

The weights version key and a validator permit can both block a weights update at submission time,
but for different reasons. The version key check rejects a `set_weights` call whose declared version
is below the subnet's required minimum, while the validator permit authorizes a neuron to set
validator-style weights on other neurons at all
([Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Weights version key** — compatibility gate on declared client version.
- **Validator permit** — authorization gate for counting validator-style weights.

## The Version Rides on the set_weights Extrinsic

Bittensor documents weight submission as a dispatchable
[extrinsic](https://docs.learnbittensor.org/subtensor-api/extrinsics) call that carries both the
[weight vector](https://docs.learnbittensor.org/resources/glossary#weight-vector) and a declared
version key. The weights version key hyperparameter compares that declared value against the subnet
minimum before the runtime accepts the submission
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

That keeps version enforcement at the submission boundary. Validators still choose their own targets
and scores, but the extrinsic must declare a version at or above the subnet floor for the call to
succeed.

## Activity Cutoff Times Submissions Inside the Epoch

Subnet [hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) include an
activity cutoff measured in blocks within each
[epoch](https://docs.learnbittensor.org/resources/glossary#epoch). Documentation states that a
[validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator) who does not submit
[validator weights](https://docs.learnbittensor.org/resources/glossary#weight-vector) within the
first activity-cutoff blocks of the epoch cannot participate in the current epoch.

The weights version key and that timing rule therefore answer different questions. Version key
vocabulary decides whether a submitted weight call is fresh enough in client version, while activity
cutoff vocabulary decides whether the validator acted early enough inside the tempo-sized epoch on a
subnet such as netuid 1.

## Raising the Key Coordinates Validator Upgrades

Because a too-low version key causes the weight submission to fail, raising the subnet's weights
version key is a coordination lever for subnet owners. Hyperparameter documentation marks the
setting as owner-settable, so a subnet such as netuid 1 can require validators to move to a matching
client version before their next accepted weight call.

That is a rollout tool rather than a scoring rule. It does not change how
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) aggregates accepted weights,
but it can reject stale submissions until validators update the version they declare.

## Version Key and Activity Cutoff Gate Different Submission Failures

Documentation treats the weights version key and activity cutoff as separate rejection paths on the
same `set_weights` boundary. A declared version below the subnet floor fails before weights are
recorded, while a validator who misses the early-epoch activity window cannot participate in that
epoch's consensus pass even with a current version
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

For readers, version vocabulary answers whether client software is fresh enough to be accepted, and
activity-cutoff vocabulary answers whether the validator acted inside the epoch window. Both can
block influence, but they describe different clocks on the same submission lifecycle
([Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch),
[Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

## Distinction from Weight Setting

The weights version key is the per-subnet hyperparameter that sets the minimum version a
`set_weights` call must carry to be accepted. Weight setting is the action that submits that vector,
so a call carrying a version below the key is rejected before weights are recorded
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

- **Weights version key** — the minimum version a weight submission must carry to be accepted.
- **Weight setting** — the validator scoring action that must satisfy that gate.

## Distinction from Weights Rate Limit

The weights version key and weights rate limit are different weight-commit hyperparameters. The
weights version key sets the minimum version a `set_weights` call must carry to be accepted, while
weights rate limit sets how many blocks a validator must wait between weight commits
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Weights version key** — the minimum version a weight submission must carry.
- **Weights rate limit** — the blocks a validator must wait before the next commit.

## Distinction from Min Allowed Weights

The weights version key and min allowed weights are different weight-submission hyperparameters. The
weights version key sets the minimum version a `set_weights` call must carry to be accepted, while
min allowed weights sets the minimum number of weights a validator must include in that submission
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Weights version key** — the minimum version a submission must carry.
- **Min allowed weights** — the fewest weights a submission must include.

## Distinction from Commit Reveal Period

The weights version key and commit reveal period both touch weight submissions but solve different
problems. The version key rejects submissions carrying an outdated weights version, gating
compatibility, while commit reveal period sets how many tempos pass before a committed weight set is
revealed under commit-reveal. One screens which submissions are accepted; the other delays when
accepted weights become visible
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

- **Weights version key** — gates submissions by required version.
- **Commit reveal period** — tempos before committed weights are revealed.

## Distinction from Weight Matrix

The weights version key and the weight matrix name different parts of a subnet's consensus state.
The weights version key is the per-subnet hyperparameter that gates which weight submissions are
accepted based on the version carried with the submission, while the weight matrix is the per-subnet
matrix of weights that validators set and that Yuma Consensus reads to produce miner incentives and
validator dividends. The version key controls which submissions reach the matrix; the matrix is the
resulting state those submissions build.

The two operate on different layers of the same flow. The version key may require a submitter to
update its weight version before its weights are accepted into the subnet, while the matrix is the
accumulated record of validator weight assignments that the subnet's epoch cycle then reads
([Glossary: Weights Version Key](https://docs.learnbittensor.org/resources/glossary#weights-version-key),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Distinction from Weight Vector

The weights version key and weight vector submission act on the same extrinsic as gate and payload.
The weights version key is the per-subnet hyperparameter that sets the minimum version a
`set_weights` call must carry, while the weight vector is the scored targets the validator submits;
a call carrying a version below the key is rejected before those weights are recorded
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

Gate vocabulary and payload vocabulary answer different questions on the same submission path. The
version key screens client compatibility; the weight vector carries the miner scores that pass that
gate.

- **Weights version key** — the minimum version a weight submission must carry to be accepted.
- **Weight vector** — the submitted scores the version key must accompany.

## Distinction from Validator-Miner Bonds

The weights version key and validator-miner bonds name different positions in a subnet's consensus
flow. The weights version key is the per-subnet hyperparameter that sets the minimum version a
`set_weights` call must carry for the submission to be accepted, while validator-miner bonds are the
per-validator-per-miner investment scores that EMA-smooth over time and feed into the bond-weighted
consensus aggregation. The version key gates which submissions reach the chain; bonds shape how
accepted submissions aggregate into miner rankings.

The two operate at different stages of the same consensus flow. A validator's weight-setting
submission must carry a version at or above the key to be recorded, and the bonds between that
validator and each miner then influence how consensus aggregates the recorded weights. The version
key is the per-subnet compatibility check; bonds are the per-validator-per-miner smoothing that
helps consensus read those accepted submissions
([Glossary: Weights Version Key](https://docs.learnbittensor.org/resources/glossary#weights-version-key),
[Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Anatomy of Incentive Mechanism](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## Distinction from Subnet Task

The weights version key and subnet task name different positions in a subnet's evaluation flow. The
weights version key is the per-subnet hyperparameter that sets the minimum version a `set_weights`
call must carry for the submission to be accepted, while subnet task is the per-subnet definition of
the work miners perform that validators then score to produce their weights. The version key gates
which submissions reach the chain; the subnet task defines what work those submissions score.

The two operate at different stages of a subnet's evaluation cycle. A subnet creator defines the
subnet task when the subnet is configured, and validators then score miner responses to that task
when submitting their `set_weights` calls. The weights version key checks those submissions for
compatibility before the chain records them. The subnet task is the per-subnet work definition; the
version key is the per-subnet compatibility gate
([Glossary: Weights Version Key](https://docs.learnbittensor.org/resources/glossary#weights-version-key),
[Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Anatomy of Incentive Mechanism](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## Per-Subnet Live Value Boundary

Weights version key is per-subnet chain state. The minimum accepted version configured on one netuid
filters validator weight submissions on that subnet, but the live key can differ from reference
defaults and from the key on another netuid
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

This article's infobox uses netuid 1 as an example label when reading weight-submission rules on one
subnet. That example helps readers verify one table; it is not proof of identical version gates
everywhere ([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Weights version key filters submissions; Yuma Consensus still aggregates eligible weights into
emission shares at epoch boundaries
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Weights version key** — minimum accepted version for weight submissions on a subnet.
- **Per-subnet boundary** — live value is chain state on one netuid.

## Distinction from Yuma Consensus

Weights version key is a per-subnet hyperparameter that sets the minimum version a validator's
`set_weights` call must carry. Yuma Consensus is the separate tempo process that aggregates accepted
weight submissions into emission shares
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Weights version key** — minimum accepted version for weight submissions.
- **Yuma Consensus** — mechanism that turns validator weights into emission shares.

## Reader Boundary

This page defines the concept at a high level. It does not report the weights version key set on any
particular subnet or whether a given validator's submission currently meets it. Those are live chain
state and should be checked for the relevant netuid. The default of 0 is the documented value, and
the parameter is owner-settable per subnet.
