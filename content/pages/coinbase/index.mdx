# Coinbase

Coinbase is the Bittensor mechanism that drives token emission across subnets on every block. The
official
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
documentation describes it as the engine that orchestrates TAO and alpha emission distribution,
connecting subnet validators, subnet miners, and stakers through the flow of emissions. For the
broader picture, the [Emissions](https://docs.learnbittensor.org/learn/emissions) overview frames
how those rewards reach participants.

## Why Coinbase Runs Every Block

Bittensor aims for a continuous flow of liquidity rather than periodic lump payouts. The coinbase
documentation describes the mechanism as the network's economic heartbeat, running each block to
keep value moving into subnets steadily over time.

For readers, coinbase is best understood as a recurring process, not a one-time event. It is the
per-block step that keeps emission and liquidity flowing while leaving the actual reward split to
each subnet's consensus round.

References:
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase),
[Emissions](https://docs.learnbittensor.org/learn/emissions)

## What Coinbase Does Each Block

The coinbase documentation describes three functions performed every block. First, liquidity
injection adds TAO and subnet-specific alpha tokens to each subnet's liquidity pools. Second,
accumulation builds up pending emissions, sometimes called alpha outstanding, that are bound for
distribution to miners and validators in the next round. Third, consensus triggering starts each
subnet's Yuma Consensus epoch.

These steps separate supply from distribution. Coinbase keeps adding and accumulating value each
block, while the actual hand-out to participants happens when a subnet's epoch runs.

References:
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase),
[Emissions](https://docs.learnbittensor.org/learn/emissions)

## How Coinbase Hands Off to Epochs

Coinbase does not pay participants directly. The documentation explains that it triggers each
subnet's Yuma Consensus epoch, the process that turns accumulated emissions into distributed
rewards, and that these epochs are staggered to avoid overloading the chain with simultaneous
computation.

The [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) documentation describes
that distribution step, while an [epoch](https://docs.learnbittensor.org/resources/glossary#epoch)
is the per-subnet cycle in which it happens. Coinbase is the upstream trigger; the epoch is where
accumulated value is shared out.

References: [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)

## Relationship to Emission

Coinbase and emission are related but different scopes in Bittensor tokenomics vocabulary. Emission
names the broader process of generating and allocating TAO and subnet alpha over time, while
coinbase names the recurring block-level mechanism that injects liquidity, accumulates pending
emissions, and triggers subnet consensus epochs. The
[Emissions](https://docs.learnbittensor.org/learn/emissions) overview explains participant reward
allocation, and the
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
documentation describes the coinbase mechanism.

For readers, emission names the reward outcome, while coinbase names the chain process that keeps
that outcome moving block by block.

References: [Emissions](https://docs.learnbittensor.org/learn/emissions),
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)

## Relationship to Block

Coinbase and a block are related because the coinbase process runs at block cadence. A block is one
unit of chain progression on Subtensor, while coinbase is the per-block mechanism that feeds subnets
with liquidity and pending emissions. The
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block) describes a block as a
unit of blockchain data, and the coinbase documentation describes coinbase as running every block.

For readers, block names the timing unit, while coinbase names the tokenomics process that executes
on that timing rhythm.

References: [Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)

## Relationship to Epoch

Coinbase and an epoch are related because coinbase hands accumulated value to subnet consensus
rounds. The coinbase documentation describes the process as triggering each subnet's Yuma Consensus
epoch, while the glossary defines an epoch as the period during which a subnet executes its
consensus mechanism.

For readers, coinbase names the upstream accumulation and trigger process, while epoch names the
subnet consensus period where accumulated emissions can be distributed.

References:
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)

## Relationship to Yuma Consensus

Coinbase and Yuma Consensus are related but separate parts of the reward flow. Coinbase accumulates
pending emissions and triggers the subnet's epoch, while Yuma Consensus turns validator weight
signals into miner incentives and validator dividends. The
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) documentation explains the
distribution process, and the coinbase documentation describes the trigger that starts it.

For readers, coinbase prepares and starts the reward-distribution moment; Yuma Consensus determines
how the accumulated value is allocated.

References: [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)

## Relationship to Protocol Alpha

Coinbase and protocol alpha are related through reserve injection. The coinbase documentation
describes per-block TAO and alpha liquidity injection for subnets, while the
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)
describes alpha accumulated by the protocol as a result of chain-side TAO buys during reserve
injection.

For readers, coinbase names the recurring injection process, while protocol alpha names one kind of
subnet alpha that can accumulate from that process on the protocol side.

References:
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase),
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from local testing to testchain and then mainchain. For
coinbase, that sequence changes how readers should interpret evidence about the per-block emission
mechanism.

In local testing, coinbase can show whether the per-block liquidity injection and epoch-trigger
logic is present inside an isolated environment. That is useful for checking the coinbase
mechanism's behavior, but it remains isolated feedback from a local environment.

On testchain, coinbase behavior can be observed in a shared testing network. This gives stronger
evidence about how per-block injection, emission accumulation, and Yuma Consensus triggering
interact than a private local run, while still keeping the result separate from production mainchain
emission activity.

On mainchain, coinbase drives live TAO and alpha emission across Bittensor subnets. The
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
documentation describes coinbase as the engine that orchestrates TAO and alpha emission
distribution, so production coinbase context is tied to active per-block emission on the live
network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A coinbase result observed in one environment should not
be read as identical evidence about emission behavior in another environment.

For readers, this keeps coinbase examples from sounding stronger than their environment supports.
Local, testchain, and mainchain contexts can all describe per-block emission mechanics, but they do
not carry the same interpretive weight.

## Reader Boundary

Coinbase should not be read as a live emission quote, a per-subnet payout table, or a step-by-step
implementation guide. Block timing and emission amounts depend on live chain state and subnet
context. This article describes the concept of the per-block mechanism that injects liquidity,
accumulates pending emissions, and triggers subnet consensus.
