# Block

In Bittensor, a block is one ordered unit of blockchain data. It groups transactions into chain
history.

[Block](https://docs.learnbittensor.org/resources/glossary#block) is defined in the official
glossary as a data unit containing transactions and a unique block hash. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference places
block timing inside network context.

## Chain Record Context

A [block](https://docs.learnbittensor.org/resources/glossary#block) is strongest as evidence for an
ordered unit of Bittensor chain data. It can support claims about inclusion, ordering, and the
existence of chain-visible records in a selected network context.

The block concept should stay narrower than the activity around it. A block can show that
transactions were included, but it does not by itself explain off-chain intent, future outcomes, or
every reason a state change happened.

## Block Timing Context

The [Block](https://docs.learnbittensor.org/resources/glossary#block) glossary entry places blocks
inside Bittensor chain progression, and the
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet contexts. Timing claims should therefore be read with the
network environment attached.

That context is important because local development settings can differ from shared network
settings. A block timing statement should not be generalized beyond the environment supported by the
source.

## Inclusion Versus Submission

Submission and inclusion are related but separate reading steps. The
[Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) entry names the pre-inclusion
waiting area for pending transactions, while a block is the chain record that contains transactions
after inclusion.

This distinction keeps block vocabulary from replacing extrinsic or mempool vocabulary. A submitted
action is not the same thing as an included record until the chain evidence supports that reading.

## Included Records and Other Evidence

Blocks sit near several other evidence surfaces. The
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) reference names
submitted runtime actions, the
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events) reference lists emitted
runtime records, and the [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)
reference lists queryable state entries.

A block gives those records a place in chain history. It does not make every evidence type
interchangeable. The exact claim determines whether block, extrinsic, event, storage, or fee
evidence is the best source.

## Network Scope

A block is read inside a specific Bittensor network environment. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, so a block from one environment should not be treated as
evidence for another environment.

Network context also affects how examples should be read. A localnet block can demonstrate a local
development record, while a testnet or mainnet block belongs to its own separate chain history.

## Evidence Boundary

The [Block](https://docs.learnbittensor.org/resources/glossary#block) entry supports a narrow claim:
a block is a chain data unit that contains transactions and has a block hash. It does not support
broad claims about motive, custody practice, later state, or every related off-chain event.

Keeping that boundary narrow makes block references more useful. They can anchor chain-visible facts
without becoming a catch-all substitute for transaction analysis, wallet procedures, subnet
evaluation, or network monitoring.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For blocks, that
sequence changes how readers should interpret examples of block-anchored transactions and chain
state.

In localnet, block mechanics can be observed in an isolated environment. Localnet block timing and
transaction inclusion reflect local chain settings and do not represent production chain state.

On testnet, block behavior can be observed in a shared, non-production network. Testnet block
records and extrinsic timing are separate from mainnet chain state.

On mainnet, blocks are live grouped records of Subtensor transactions that form the production
Bittensor blockchain. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
describes the chain environments that apply on the production network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A block example from one environment should not be read as
representing chain state or transaction timing in another environment.
