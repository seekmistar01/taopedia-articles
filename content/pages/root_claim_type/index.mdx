# Root Claim Type

Root claim type is the Keep or Swap setting for alpha dividends earned through Root Subnet staking.
Keep leaves processed dividends as originating-subnet alpha, while Swap converts processed dividends
to TAO and restakes them on Root
([Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

The term belongs to dividend-handling vocabulary. It names the selected outcome for processed root
claims, not a separate staking action, balance report, or reward prediction.

## Claim Setting

Root claim type is the preference applied when accumulated root-claim dividends are processed. It
answers how the dividends are handled, while claim timing answers when accumulated dividends are
processed
([Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

This distinction keeps the setting separate from the claim event. A root claim processes accumulated
dividends; root claim type determines the Keep or Swap outcome during that processing.

The setting should also be kept separate from the existence of dividends. Root claim type can say
which handling path is selected, but it does not by itself prove that a particular amount is ready,
that a claim has already been processed, or that future dividends will follow the same observed
pattern ([Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

That makes root claim type a preference term. It is useful when the article needs to describe the
selected dividend-handling mode, but it should not be stretched into a complete root-staking result
or a general reward statement.

## Keep and Swap Outcomes

Keep is the alpha-retention outcome. Earned alpha remains on the subnet that generated the dividend.
Swap is the TAO-restaking outcome. Accumulated alpha dividends are converted to TAO and restaked on
Root ([Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Both outcomes apply to dividend output, not to the original Root Subnet staking position. Keep does
not evaluate the producing subnet, and Swap should not be confused with unstaking from Root or
moving stake between validators.

The Keep/Swap contrast is therefore an output-handling contrast. Keep preserves the originating
subnet alpha form after processing. Swap changes the processed dividend output into TAO and places
that TAO back into the Root staking context
([Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

Neither label is a quality score for the subnet that produced the dividend. The labels describe how
processed dividend output is handled after it exists in the root-claim flow.

## Root Staking Context

Root claim type belongs to Root Subnet staking. The Root Subnet, also called Subnet Zero, is the
special subnet where validators can register and TAO holders can stake in a subnet-agnostic way, and
a root staker is the TAO holder using that path
([Glossary: Root Subnet](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero),
[Glossary: Root Staker](https://docs.learnbittensor.org/resources/glossary#root-staker)).

This keeps the role and setting connected without merging them. Root staker names who is staking;
root claim type names the dividend-handling preference for processed dividends.

Root proportion is nearby vocabulary but answers a different question. Root proportion describes the
portion of a subnet's dividend context attributed to the Root side, while root claim type describes
how processed root-claim dividends are handled
([Glossary: Root Proportion](https://docs.learnbittensor.org/resources/glossary#root-proportion),
[Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

Keeping those terms separate avoids mixing a subnet-level measure with a holder-level processing
preference. Root proportion helps interpret the dividend split; root claim type names Keep or Swap
handling for processed dividends.

## Emission Scope

Root claim type sits near emissions vocabulary because the underlying subject is dividend handling.
Emissions supply the broader reward context, while root claim type decides whether processed
dividend output remains as alpha or becomes TAO restaked on Root
([Emission](https://docs.learnbittensor.org/learn/emissions),
[Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

The term is narrower than emissions overall. It is the claim-output preference, not the full reward
distribution model and not an exact conversion-result report.

Emission context explains why the output can be discussed as alpha, TAO, dividends, and staking
context in the same page. Root claim type should still stay narrower than the emission process
itself: it describes the selected handling of processed dividend output, not the full path by which
emissions are generated or distributed ([Emission](https://docs.learnbittensor.org/learn/emissions),
[Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

When a statement needs exact amounts, timing, or conversion results, root claim type is not enough
evidence on its own. The statement needs the relevant claim, network, and source context.

## Environment Scope

Bittensor separates localnet, testnet, and mainnet environments. Root-claim-type examples from one
environment should not be treated as evidence for another because balances, dividend state, and
claim outcomes are environment-specific
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Introduction to Bittensor: Subnet development](https://docs.learnbittensor.org/learn/introduction#subnet-development)).

Localnet examples are isolated development examples. Testnet examples are shared non-production
examples. Mainnet root claim type concerns production Root Subnet dividends and production claim
outcomes.

The same Keep or Swap vocabulary can appear across environments while pointing to different chain
contexts. A local example can explain the setting, a testnet example can show non-production use,
and a mainnet example needs support from the production context being discussed
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).
