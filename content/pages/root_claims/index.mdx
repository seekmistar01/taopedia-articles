# Root Claims

Root claims handle alpha dividends from stake placed through the Root Subnet. They determine whether
earned alpha remains as subnet alpha or is converted into TAO and restaked on root
([Root Claims](https://docs.learnbittensor.org/staking-and-delegation/root-claims),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

The mechanism is narrow. It is about dividend treatment after rewards accrue, not about moving the
original root stake or explaining a general staking procedure.

## Root Subnet Context

The Root Subnet is the subnet-agnostic staking environment in Bittensor. Root claims exist because
stake placed there can receive alpha dividends from other subnets even though the underlying root
stake remains tied to the Root Subnet
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Root Claims](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

That separation gives root claims their role. The original stake remains root-routed, while earned
alpha needs a handling rule once it is claimable.

## Keep and Swap Outcomes

Root claims have two outcomes. Keep leaves earned alpha as the alpha of the subnet that produced it.
Swap converts the earned alpha into TAO and restakes that dividend output on the Root Subnet
([Root Claims](https://docs.learnbittensor.org/staking-and-delegation/root-claims),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Both outcomes apply to dividends, not to the original root stake. This distinction keeps root claims
separate from unstaking, moving stake, or changing delegation.

## Claim-Type Context

Root claim type is the preference that decides which outcome a root claim uses. Keep and Swap are
therefore best read as routing choices for dividend output
([Root Claims](https://docs.learnbittensor.org/staking-and-delegation/root-claims),
[Glossary: Root Staker](https://docs.learnbittensor.org/resources/glossary#root-staker)).

The claim event and the claim type are related but different. A claim processes accumulated
dividends; the claim type determines whether the processed dividend remains as alpha or becomes TAO
restaked on root.

## Processing Context

Root claims may be processed automatically or directly, but the same claim type governs the
resulting dividend treatment in either case
([Root Claims](https://docs.learnbittensor.org/staking-and-delegation/root-claims),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

This means the processing trigger is not the core concept. The important article-level idea is the
dividend outcome: Keep preserves alpha exposure from the producing subnet, while Swap routes the
dividend into TAO on root.

## Emission Context

Root-claim swaps sit inside Bittensor emission accounting. They handle dividend output, while
flow-based emissions track subnet inflows and outflows for emission distribution
([Root Claims](https://docs.learnbittensor.org/staking-and-delegation/root-claims),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

This keeps every alpha-to-TAO conversion from being read as the same kind of subnet flow event. Root
claims are a dividend-handling mechanism layered after rewards are earned.

## Reading Root Claims

This article defines root claims as a staking and emissions concept. It does not provide claim
timing advice or a recommendation between Keep and Swap
([Root Claims](https://docs.learnbittensor.org/staking-and-delegation/root-claims),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

The stable article-level point is that root claims explain how dividends from Root Subnet staking
are handled after they accrue.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Root Claims,
this sequence gives readers a boundary for interpreting claim examples and root-network distribution
notes.

Localnet examples are isolated and reflect local chain state, so they are useful for controlled
experiments rather than evidence of live Bittensor behavior. Testnet examples add shared
non-production conditions, which can reveal integration behavior without touching mainnet state.

On mainnet, Root Claims examples should be read as live production root-claim behavior on the
production Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, so outcomes from one environment should not be treated as
proof of behavior in another.

## Ownership Claim Scope

Root claims should keep ownership and network context visible. Governance and subnet sources
distinguish Root/Subnet Zero from ordinary subnet participation, while wallet references keep
ownership authority tied to coldkey control
([Glossary: Root Subnet/Subnet Zero](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero),
[Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets)).

For readers, a root claim is strongest when it names the account authority and the Root context
being claimed. It should not be read as a generic claim about every subnet role or every wallet
action.
