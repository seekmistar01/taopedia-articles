# Validator Weights

Validator weights are the evaluation signals that Bittensor validators submit after judging miner
work on a subnet. A subnet's incentive mechanism defines what useful work means, validators score
miners against that mechanism, and [[yuma_consensus|Yuma Consensus]] aggregates those submitted
signals into miner incentives and validator dividends.

References: [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)

## What a weight represents

A validator weight is not a generic reputation score. It is a subnet-specific judgment about how
well a miner performed under the rules of that subnet's incentive mechanism. Different subnets can
measure different work, so the meaning of a strong weight depends on the task, scoring method, and
evaluation criteria chosen by the subnet.

References:
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[[subnet_creation_mechanisms|Subnet Creation and Incentive Mechanisms]]

## From scoring to consensus

Validators produce weight vectors: one validator's relative evaluation of miners for a subnet. Yuma
Consensus combines the validators' submitted weight vectors into a broader consensus result. That
aggregation step matters because Bittensor is not rewarding a single validator's opinion; it is
using many validator evaluations to decide how miner and validator rewards should be allocated.

References: [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)

## Why weights affect incentives

Weights connect subnet work to economic outcomes. Miners compete to earn stronger evaluations, while
validators are rewarded for producing evaluations that are useful, timely, and consistent with the
consensus process. [[dynamic_tao|Dynamic TAO]] describes the broader emission system; validator
weights explain the subnet-level distribution step that turns evaluations into rewards.

References: [Emission](https://docs.learnbittensor.org/learn/emissions),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)

## Quality risks

The weight system depends on independent validator evaluation. If validators copy visible weights
instead of evaluating miners themselves, the subnet loses useful information and the incentive
mechanism becomes easier to game. Bittensor documentation describes Commit Reveal as one protection:
fresh weights can be concealed before they become visible, making copied signals less useful when
miner performance changes.

References:
[The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)

## Relationship to Consensus Score

Validator weights and consensus score refer to related but different parts of Yuma Consensus. The
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)
frames a consensus score as a stake-weighted median of validator-assigned weights used to filter
outlier submissions before final rank calculations. Validator weights are the evaluation signals
that validators submit and that Yuma Consensus aggregates.

For readers, validator weights name the per-validator evaluation inputs, while consensus score names
the agreement threshold used to filter those inputs. A validator weight should not be read as a
consensus score; it is the submitted signal that the consensus score is computed from.

References:
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)

## Relationship to Multiple Mechanisms

Most validator-weight discussion centers on one incentive mechanism, but subnets can also run
multiple mechanisms in parallel. The official documentation describes validators as setting weights
independently for each mechanism and evaluating miners separately under each mechanism's criteria.

For readers, a validator weight still means a subnet-specific evaluation signal, but the mechanism
context matters. A weight submitted for one mechanism does not describe miner performance in
another, and validator dividends combine performance across mechanisms rather than isolating one
mechanism alone.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)

## Relationship to Weight Matrix

Validator weights and the weight matrix describe the same evaluation data at different scopes. A
validator weight vector is one validator's submitted evaluation signal for miners, while the weight
matrix is the subnet-level collection of those vectors used as input to Yuma Consensus. The
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)
describes the matrix as built from ranking weight vectors submitted by subnet validators.

For readers, validator weights answer what individual validators submit, while the weight matrix
answers how those submissions are organized together before consensus filtering and aggregation.

References:
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from local testing to testchain and then mainchain. For
validator weights, that sequence changes how readers should interpret the evidence represented by a
weight signal.

In local testing, validator weights can show whether an evaluation method can produce a coherent
weight vector for miner work. That is useful for checking the shape of the scoring logic, but it is
still isolated feedback from a local environment.

On testchain, validator weights can be observed in a shared testing network. This gives stronger
evidence about how validator submissions behave together than a private local run, while still
keeping the result separate from production mainchain incentives.

On mainchain, validator weights are part of the live consensus input path. The
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
page describes validators computing weight vectors from subnet evaluation logic, and
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) describes those submitted
weights being aggregated into incentive outcomes.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A weight vector observed in one environment should not be
read as identical evidence about validator evaluation in another environment.

For readers, this keeps weight claims from sounding stronger than their environment supports. Local,
testchain, and mainchain weights can all describe validator evaluation, but they do not carry the
same interpretive weight. The network context is part of the weight claim, not an afterthought.

This distinction is especially important when comparing weights over time. A local or testchain
vector can demonstrate behavior, but mainchain context determines whether the signal was part of
production reward allocation.

## How to read weight claims

A claim about validator weights should be read with its subnet context. The important questions are
what the subnet is measuring, how validators are expected to evaluate miner work, and how Yuma
Consensus uses the submitted signals. A high weight only has meaning inside that evaluation system;
it should not be treated as a universal score across unrelated subnets.

References:
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)

## Relationship to Coinbase

Validator weights and coinbase are related but different parts of how subnet evaluation becomes
emissions. Validator weights are the evaluation signals validators submit after judging miner work.
Coinbase is the per-block protocol mechanism that drives TAO emission, injects TAO into emitting
subnets' pool reserves, accumulates pending emissions, and checks epoch boundaries to trigger Yuma
Consensus rounds. The
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)
describes the evaluation inputs, and the
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
documentation describes the per-block emission operation.

Validator weights are the primary input that coinbase-triggered epoch rounds consume. When coinbase
reaches a subnet's epoch boundary, it runs the Yuma Consensus round that reads the submitted
validator weights, filters and aggregates them, and distributes miner incentives and validator
dividends accordingly. Without fresh validator weights there is little signal for the coinbase round
to act on, and without a coinbase-triggered round the weights are not translated into emissions.
Validator weights supply the judgment; coinbase is the process that converts that judgment into
reward outcomes.
