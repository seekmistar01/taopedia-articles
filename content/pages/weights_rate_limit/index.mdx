# Weights Rate Limit

Weights rate limit is a per-subnet hyperparameter, listed in the documentation as `WeightsRateLimit`
and also referred to as `CommitmentRateLimit`, that sets how long, in blocks, a validator must wait
between [weight setting](https://docs.learnbittensor.org/resources/glossary#weight-vector) commits
on a subnet
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## What It Controls

The hyperparameter paces how often a single validator can resubmit
[validator weights](https://docs.learnbittensor.org/resources/glossary#weight-vector). It does not
define what those weights mean, how
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) aggregates them, or which
miners a validator should score. It only sets the minimum block interval between one weight commit
and the next from the same validator on the selected subnet.

That makes weights rate limit a submission-pacing rule. It limits how frequently a validator can
change its on-chain weight signal, rather than deciding whether a particular submission counts in
the current [epoch](https://docs.learnbittensor.org/resources/glossary#epoch).

## Documented Type, Default, and Setter

The [Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)
reference lists `WeightsRateLimit` / `CommitmentRateLimit` as a `u64` with a default of `100` and
the description "How long, in blocks, a validator must wait between weight commits on a subnet." It
is changed through the `sudo_set_weights_set_rate_limit` extrinsic, exposed in tooling as
`btcli sudo set --param weights_rate_limit`, and the reference marks the permission required to set
it as root rather than subnet owner
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

Because each subnet carries its own hyperparameters, the commit pacing on netuid 1 can differ from
another subnet's limit
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Distinction from Activity Cutoff

Activity cutoff is measured in blocks within each epoch and governs whether a validator's weight
submission arrives early enough to count for the current Yuma Consensus pass. Weights rate limit is
measured in blocks between commits and governs how soon the same validator can submit again after a
prior weight commit
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Activity cutoff** — whether a submission arrives inside the current epoch window.
- **Weights rate limit** — how long a validator must wait before the next commit.

## Distinction from Commit Reveal

[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal) changes when committed
weights become visible, while weights rate limit changes how frequently a validator can submit new
commits. Reveal delays when weights become visible; the rate limit slows how often they can change,
so together they reduce the value of reacting to freshly observed signals rather than evaluating
independently ([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Commit reveal** — hides committed weights until a later reveal.
- **Weights rate limit** — blocks a validator must wait between weight commits.

## Distinction from Serving Rate Limit

[ServingRateLimit](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#servingratelimit)
rate-limits miner calls to `serve_axon` and `serve_prometheus`. That setting paces miner endpoint
registration, not validator weight commits
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Serving rate limit** — how often a miner may update axon/prometheus serving metadata.
- **Weights rate limit** — how long a validator must wait between weight commits.

## Distinction from Tempo

Weights rate limit sets how many blocks a validator must wait between weight commits, while tempo is
the block interval over which rewards accumulate before Yuma Consensus distributes them at the
boundary ([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Weights rate limit** — the blocks a validator must wait between commits.
- **Tempo** — the block interval that bounds reward accumulation and distribution.

## Distinction from Min Allowed Weights

Min allowed weights sets the minimum number of weights a validator must include in a submission,
while weights rate limit sets how many blocks a validator must wait between weight commits
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Min allowed weights** — the fewest weights a submission must include.
- **Weights rate limit** — the blocks a validator must wait before the next commit.

## Distinction from Weights Version Key

The weights version key sets the minimum version a `set_weights` call must carry to be accepted,
while weights rate limit sets how many blocks a validator must wait between weight commits
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Weights version key** — the minimum version a weight submission must carry.
- **Weights rate limit** — the blocks a validator must wait before the next commit.

## Chain Reads for netuid 1

Readers can verify live hyperparameter values for the documented example netuid with
`btcli subnet hyperparameters --netuid 1 --network finney`
([Subnet Hyperparameters: View hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#view-hyperparameters)).

That read path keeps live hyperparameter claims tied to a parseable netuid.

## Per-Subnet Live Value Boundary

Weights rate limit is per-subnet chain state. The block gap configured on one netuid paces how long
a validator must wait between weight commits on that subnet, but the live limit can differ from
reference defaults and from limits on other netuids
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

This article's infobox uses netuid 1 as an example label when reading one subnet's hyperparameter
table on Finney mainnet. That example helps readers verify one row; it is not proof that every
subnet exposes the same live commit pacing
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Weights rate limit paces validator weight submissions; Yuma Consensus still aggregates eligible
weights into emission shares at epoch boundaries on the active subnet
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Weights rate limit** — block gap between validator weight commits on a subnet.
- **Per-subnet boundary** — live limit is chain state on one netuid.

## Distinction from Yuma Consensus

Weights rate limit is the per-subnet hyperparameter that sets the minimum block wait between
validator weight commits. It names submission cadence, not the mechanism that credits emission
shares at the epoch boundary
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Weights Rate Limit](https://docs.learnbittensor.org/resources/glossary#weights-rate-limit)).

Yuma Consensus is the on-chain mechanism that reads the eligible weight matrix, applies clipping and
bonding, and then credits miner incentives and validator dividends on the selected netuid
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Emission: Distribution](https://docs.learnbittensor.org/learn/emissions#distribution)).

The weight-setting section above covers what validators record on chain. Weights-rate-limit
vocabulary names how often those updates may arrive; Yuma vocabulary names settlement once eligible
weights are assembled
([Glossary: Weight Setting](https://docs.learnbittensor.org/resources/glossary#weight-setting)).

Readers should not read the rate limit as a payout quote or scoring rule. It paces submissions
upstream of the matrix Yuma processes on the relevant netuid
([Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Weights rate limit** — minimum block wait between weight commits.
- **Yuma Consensus** — on-chain mechanism that uses validator weights to determine emission shares
  ([Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions)).

## Reader Boundary

Weights rate limit should not be read as activity cutoff, commit reveal, or a weight-quality rule.
It names only the subnet hyperparameter that sets how many blocks a validator must wait between
weight commits on the selected netuid
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).
