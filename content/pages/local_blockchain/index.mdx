# Local Blockchain

A local blockchain is a private Bittensor chain used for development and testing. Official Bittensor
documentation distinguishes local development environments from public network behavior, which makes
the term useful when discussing tools, subnet experiments, and test workflows.

References:
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)

## Development Context

The local blockchain context is isolated from production use. It lets developers exercise Bittensor
components and subnet behavior without treating the result as mainnet state. That separation is the
main point of the term: local behavior is useful for development feedback, but it is not the same as
live network behavior.

References:
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)

## Relationship to Public Subtensor

Local blockchain and Public Subtensor name opposite sides of Bittensor chain access. The
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)
describes a private chain used for development and testing, while the
[Glossary: Public Subtensor](https://docs.learnbittensor.org/resources/glossary#public-subtensor)
describes publicly reachable chain access.

For readers, local blockchain names isolated development context, while Public Subtensor names live
network access context. Outputs from a local chain should not be treated as public Subtensor state.

References:
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Glossary: Public Subtensor](https://docs.learnbittensor.org/resources/glossary#public-subtensor)

## Relationship to Bittensor Networks

Local blockchain and Bittensor Networks name related but different parts of network context. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference includes
localnet as a user-controlled development environment, while the
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)
describes the private chain used in that setting.

For readers, localnet is the environment label, while local blockchain names the isolated
development chain itself. Mainnet and testnet belong to other network contexts rather than to local
blockchain by default.

References: [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)

## Relationship to Non-fast Blocks

Local blockchain and non-fast blocks are related but name different parts of local development
context. The
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)
describes a private chain used for development and testing, while the
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)
describes ordinary local block timing that follows Subtensor's default 12-second interval.

For readers, local blockchain is the isolated development chain, while non-fast blocks are a timing
setting used within it to simulate production-like intervals. Neither term describes live mainnet
behavior on its own.

References:
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)

## Relationship to Fast Blocks

Fast blocks are described in official documentation as a development-only local-chain timing
configuration. Local blockchain context explains why that distinction matters: faster local block
production can help testing while remaining separate from public Bittensor network behavior.

Reference: [Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)

## Relationship to Mainchain

Local blockchain and Mainchain name opposite sides of Bittensor network context. The
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)
describes a private chain used for development and testing, while the
[Glossary: Mainchain](https://docs.learnbittensor.org/resources/glossary#mainchain) describes the
primary Bittensor network used for production purposes.

For readers, local blockchain is isolated development context, while Mainchain is production network
context. Local blockchain should not be read as a stand-in for Mainchain behavior.

References:
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Glossary: Mainchain](https://docs.learnbittensor.org/resources/glossary#mainchain)

## Relationship to Subtensor

Local blockchain and Subtensor refer to related but different parts of Bittensor chain context. The
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)
describes a private, isolated chain used for developing and testing subnet incentive mechanisms. The
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor) describes
Bittensor's layer 1 blockchain and the system of record for transactions and rankings.

For readers, local blockchain is a private development environment, while Subtensor is the
chain-layer concept. Local blockchain should not be read as a replacement for production Subtensor
context.

References:
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor)

## Relationship to Multiple Mechanisms

A local blockchain provides an isolated chain for Bittensor development and testing. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#local-blockchain) describes it as used
for developing and testing subnet incentive mechanisms.

For readers, a local blockchain still names a private development environment. Subnets tested there
may later run more than one incentive mechanism in production, but the local chain itself is not a
mechanism label.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Incentive Mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism)

## Subnet Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as a staged path: first test the incentive mechanism locally, then use
the Bittensor testchain, and finally connect to mainchain. In that sequence, the local blockchain is
the first proving ground, not the final network target.

That stage matters because a subnet is more than an application idea. Its miners and validators need
an incentive mechanism that can be exercised before anyone treats the results as public network
behavior. Local blockchain context gives developers a private place to check whether the basic
interaction between miners, validators, and emissions behaves as intended.

For readers, this also explains why local results should be interpreted carefully. A local
blockchain can show whether a mechanism works inside an isolated environment, but it does not
establish that the mechanism has been accepted by testchain or mainchain. It is useful development
evidence rather than public chain evidence.

The stage boundary keeps the article's terms separate. Local blockchain names the private chain used
for the first development stage. Testchain names a public testing network stage, and mainchain names
production network context. A subnet moving from one stage to another changes the network context
around the same incentive-mechanism idea.

This is why local blockchain belongs in development vocabulary rather than deployment vocabulary. It
helps explain where a subnet can be tried, adjusted, and understood before public network
participation is involved.

It is a controlled comparison point before public-network variables enter the testing path.

## Reader Boundary

This page defines the term. It does not describe a live public network, a production subnet, or a
deployment recommendation. Readers should use current official setup documentation when they need
step-by-step local development instructions.
