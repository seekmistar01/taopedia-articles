Subnet 11, **TrajectoryRL**, is operated by [trajectoryRL](https://github.com/trajectoryRL) and runs
a continuous reinforcement learning competition on the Bittensor network. Its goal is to make AI
agents cheaper, safer, and more reliable by producing optimized agent "skills" — self-contained
instruction files that any compatible agent can install and follow. The output of the competition is
distributed through the [trajrl](https://github.com/trajectoryRL/trajrl) registry, a publicly
installable skill catalog updated automatically as each competition cycle concludes
([source](https://github.com/trajectoryRL/trajectoryRL)).

## How the Mechanism Works

TrajectoryRL structures miner participation around writing **SKILL.md** files — plain-text documents
that teach an AI agent how to handle a set of operational scenarios. Miners package a skill into a
content-addressed bundle identified by its SHA256 hash, upload it to any public URL, and record that
URL as a commitment on-chain. No GPU, persistent server, or continuous uptime is required once the
commitment is registered.

Validators pull each committed pack, verify its hash against the on-chain record, and evaluate the
skill inside isolated sandbox containers drawn from the
[trajrl-bench](https://github.com/trajectoryRL/trajrl-bench) benchmark. Inside each scenario
container, the validator's agent reads the SKILL.md alongside the scenario's own instruction file
and attempts to complete a realistic, shell-verifiable task. A separate verifier container then
scores the output via a test suite, producing a continuous quality score between zero and one per
scenario. The subnet's final score for a miner is the equal-weighted sum across all scenarios, with
cost per scenario tracked as a secondary metric.

Weights follow a winner-take-all design: the miner with the best aggregate score earns the full
emission for that epoch ([source](https://github.com/trajectoryRL/trajectoryRL)). A
similarity-detection layer based on Normalized Compression Distance, combined with a first-mover
threshold, is intended to prevent straightforward copying of high-scoring submissions. Final weight
settlement uses [[Yuma Consensus]], the standard mechanism across all Bittensor subnets.

The project describes a planned Season 2 in which an agent-driven auto-research loop replaces the
hand-designed scenario set. Real user agent trajectories, contributed opt-in with PII stripped,
would flow into a packaging agent that clusters failures, maps them to sandbox services, and
produces new evaluation scenarios without routine human involvement. Winners from each cycle publish
to the skill registry, and the next cycle evaluates against a harder pool that already covers the
previous cycle's solved tasks ([source](https://github.com/trajectoryRL/trajectoryRL)).

## Participating as a Miner

Mining on TrajectoryRL is a writing and reasoning task, not a compute task. A miner authors a
SKILL.md containing strategies, heuristics, and tool-use patterns for an AI agent, packages it into
a verifiable bundle, uploads it to a publicly accessible URL, and submits the URL on-chain. Because
the submission is content-addressed, the chain verifies integrity without any ongoing connection to
the miner.

Effective SKILL.md files tend to be general-purpose rather than narrowly tuned to any single
scenario, since validators run the same file against a broad and evolving set of tasks. Miners can
test their skill locally against the full scenario suite using the same evaluation harness the
validators run, without any chain interaction. The full authoring guide is available in
[MINER_GUIDE.md](https://github.com/trajectoryRL/trajectoryRL/blob/main/docs/MINER_GUIDE.md) in the
repository.

Miners should be aware of the submission terms: copyright in any pack that earns a nonzero reward in
any epoch transfers to trajrl.com. Packs that never earn a reward retain copyright with their
original author ([source](https://github.com/trajectoryRL/trajectoryRL)).

## Participating as a Validator

Validators run the subnet's evaluation harness inside Docker, with automatic image updates managed
by Watchtower so that new scenarios from trajrl-bench are picked up without manual intervention.
Each evaluation cycle, the validator fetches the latest benchmark image, retrieves all current miner
commitments from the chain, downloads and verifies each pack, and runs the agent through every
scenario. The evaluating agent operates inside the same container image as the scenario itself,
giving it access to all scenario-specific dependencies. A separate container then scores the agent's
output and produces a machine-readable test result, from which the validator reads pass rate and
cost per scenario before computing final weights.

Validators must hold a registered seat on the subnet and supply an LLM API key and endpoint for the
agent that solves scenarios. Configuration options are documented in the repository's
[.env.validator.example](https://github.com/trajectoryRL/trajectoryRL/blob/main/.env.validator.example).
