# Activity Cutoff

Activity cutoff is a per-subnet hyperparameter that defines, in blocks, the window a validator has
to submit weights during a Yuma Consensus epoch. Documentation describes it as the number of blocks
for stake to become inactive for the purpose of the epoch: a validator that does not submit weights
within the first activity-cutoff blocks of an epoch cannot participate until the next epoch
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## What It Controls

The hyperparameter ties consensus participation to timely weight submission. Within each epoch, a
validator must submit its weights inside the activity-cutoff window. If it misses that window, the
documentation states it is treated as inactive for the epoch and waits until the next epoch to take
part again. The cutoff is measured in blocks, not in wall-clock time
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Why It Matters

The cutoff keeps an epoch's consensus based on validators that are currently participating. By
requiring weights within a bounded block window, it prevents a validator that has gone quiet within
the epoch from carrying influence into that epoch's result. Activity cutoff therefore gates which
validator weight submissions enter the current Yuma Consensus pass
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For activity
cutoff, that sequence changes how readers should interpret hyperparameter examples and validator
participation windows
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

In localnet, activity cutoff can be exercised on an isolated subnet without production stake
context. Localnet block counts and epoch timing do not represent mainnet validator participation
behavior.

On testnet, activity cutoff applies in a shared non-production network. Testnet epoch windows and
weight-submission outcomes are separate from mainnet subnet state.

On mainnet, activity cutoff is a live per-subnet hyperparameter on the production Bittensor network.
Observed cutoff values and validator participation depend on the selected subnet's on-chain
hyperparameter state
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

An activity-cutoff example from one environment should not be read as representing production
participation rules in another environment.

## Distinction from Last Update

Activity cutoff and last update are read as a pair on metagraph rows. Activity cutoff is the
owner-settable block-count hyperparameter, while last update is the per-neuron block field readers
compare against it
([Subnet Hyperparameters: ActivityCutoff](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#activitycutoff),
[Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

On its own neither value decides activity. Activity cutoff supplies the allowed inactivity span and
last update supplies the block timestamp from which that span is measured, so a neuron counts as
inactive only when the gap since its last update exceeds the cutoff
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Activity cutoff** — the hyperparameter span defining allowed inactivity for epoch purposes.
- **Last update** — the per-neuron block timestamp measured against that span.

## Distinction from Tempo and Epoch

Activity cutoff is the early-epoch block window in which validators must submit weights. Tempo
counts blocks between emission steps; epoch names the wider consensus window grouping those blocks
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)).

- **Activity cutoff** — early-epoch block window for counting weight submissions.
- **Tempo and epoch** — subnet timing cadence and wider consensus window.

Cutoff vocabulary gates timely submission; tempo and epoch vocabulary name the surrounding timing
frame
([Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

## Distinction from Weights Rate Limit

Activity cutoff is the hyperparameter that defines how many blocks stake may remain inactive for
epoch purposes in Yuma Consensus. Weights rate limit is the blocks a validator must wait between
weight commits
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Subnet Hyperparameters: ActivityCutoff](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#activitycutoff)).

- **Activity cutoff** — block-count rule for epoch participation in hyperparameter documentation.
- **Weights rate limit** — blocks between weight commits from the same validator.

## Distinction from Immunity Period

Immunity period shields a newly registered neuron from deregistration for a fixed number of blocks.
Activity cutoff is a separate hyperparameter that governs epoch participation timing
([Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period),
[Subnet Hyperparameters: ActivityCutoff](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#activitycutoff)).

- **Immunity period** — deregistration shield for a new neuron.
- **Activity cutoff** — epoch-participation hyperparameter on the subnet.

## Distinction from Validator Permit

Validator permit records whether a UID may submit validator weights at all. Activity cutoff is a
separate hyperparameter read from subnet state
([Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit),
[Subnet Hyperparameters: ActivityCutoff](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#activitycutoff)).

Readers tracing a failed participation check should compare permit status and cutoff hyperparameter
values on the same netuid rather than treating either field as a full activity verdict by itself
([Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

## Distinction from Commit Reveal

Commit reveal and activity cutoff are different weight-timing rules. Commit reveal delays when
submitted validator weights become visible, concealing fresh weights for a reveal window, while
activity cutoff is the early-epoch block window within which a validator must submit weights to
count in the current Yuma Consensus pass
([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Commit reveal times visibility of a submission; activity cutoff times whether the submission lands
inside the counting window.

## Distinction from Pruning Score

Activity cutoff bounds when validator weights must arrive to count in the current epoch. Pruning
score ranks miners when a full subnet must displace one incumbent slot
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Pruning Score](https://docs.learnbittensor.org/resources/glossary#pruning-score)).

- **Activity cutoff** — validator submission timing gate inside an epoch.
- **Pruning score** — miner ranking input when one slot must open.

Validator-timing vocabulary and displacement-ranking vocabulary answer different subnet questions
([Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)).

## Distinction from Serving Rate Limit

Serving rate limit and activity cutoff are both counted in blocks but govern different actors.
Serving rate limit paces how often a miner may update its axon and prometheus serving metadata,
while activity cutoff is the block window early in an epoch within which a validator must submit
weights to count in the current Yuma Consensus pass
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

One paces a miner's endpoint updates; the other times a validator's weight submission.

## Distinction from Weight Setting

Activity cutoff is the early-epoch block window in which a validator must submit weights to
participate in the current Yuma Consensus pass. Weight setting is the validator action of assigning
scores to miners in those submissions
([Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Activity cutoff** — when weights must arrive to count in the epoch.
- **Weight setting** — how validators score miners in those submissions.

## Distinction from Yuma Consensus

Activity cutoff is a per-subnet hyperparameter that defines how recently a validator must have set
weights for those weights to count in the upcoming Yuma Consensus epoch. It names an eligibility
window on submitted weights, not the consensus algorithm itself
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Activity Cutoff](https://docs.learnbittensor.org/resources/glossary#activity-cutoff)).

Yuma Consensus is the separate tempo process that reads the eligible weight matrix after such
filters, applies stake weighting, clipping, bonding, and dividend logic, and converts the result
into emission shares. Activity cutoff shapes which submissions may enter that process; Yuma
Consensus performs the settlement once eligible weights are known
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers should keep hyperparameter vocabulary separate from mechanism vocabulary. Changing activity
cutoff on a netuid alters freshness requirements for weight inclusion; it does not replace Yuma's
consensus math or emission mapping.

- **Activity cutoff** — freshness window for weights to count in an epoch.
- **Yuma Consensus** — mechanism that turns validator weights into emission shares.
- **Weight setting** — how validators submit the scores activity cutoff filters.

## Distinction from Subnet

Activity cutoff is one of the subnet hyperparameters, the on-chain state variables that configure a
single subnet, for example netuid 1. Like other hyperparameters it is owner-settable, so a subnet
owner can change it for their subnet. The documentation lists a default value of 5000 blocks, but
the value in force is per-subnet chain state and can differ from the default
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Reader Boundary

This page defines the concept at a high level. It does not report the activity-cutoff value set on
any particular subnet, current epoch timing, or which validators are active right now. Those are
live chain state that should be checked for the relevant netuid. The 5000-block figure here is the
documented default, and the value for a given subnet is owner-settable
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).
