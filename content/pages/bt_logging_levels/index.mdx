# Bittensor Logging Levels

Bittensor logging levels are severity labels for diagnostic messages
([Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

They help readers distinguish detailed diagnostic output, normal status messages, successful events,
warnings, errors, and critical failures.

## Severity Ladder

The logging reference presents a ladder from trace and debug through info, success, warning, error,
and critical ([Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

The useful reading is severity and verbosity. Some levels expose more detail, while others indicate
that a message may require attention or describes a failure.

## Bittensor-Specific Levels

Bittensor adds trace and success to the familiar logging ladder. Trace represents fine-grained
diagnostic output, while success marks an event that completed successfully
([Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels),
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction)).

These levels make logs easier to scan without changing the basic idea that the level describes how
the software classified the message.

## Diagnostic Context

A log level is a clue about message importance, not a complete explanation of what happened. The
surrounding message, tool, network, and source context still matter
([Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

For readers, the useful order is level, message, then context. The level says how the software
categorized the message; the context explains where the message belongs.

## Boundary With Error Codes

Logging levels and Subtensor error codes answer different questions. A logging level describes
message severity or verbosity, while an error code identifies a rule family or failure category from
the Subtensor layer ([Subtensor Error Codes](https://docs.learnbittensor.org/errors),
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor)).

Reading them together can be useful, but a log level alone should not be treated as the full source
of truth for a failed network action.

## Source-of-Truth Context

Log messages are strongest when paired with the source that can confirm the underlying claim.
Logging references define message severity, while network and error-code references define
environments and chain-layer failure categories
([Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor)).

This makes logs an interpretation layer rather than the final record. They help readers triage what
deserves attention first.

## Reader Boundary

Bittensor logging levels should be read as diagnostics vocabulary. They are not a complete chain
record, a role guide, or proof that a submitted action succeeded
([Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

The stable article-level point is that logging levels label diagnostic message severity and
verbosity, while stronger claims need supporting source or chain evidence.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For BT Logging
Levels, this sequence gives readers a boundary for interpreting debugging examples, local runs, and
shared-network diagnostics.

Localnet examples are isolated and reflect local chain state, so they are useful for controlled
experiments rather than evidence of live Bittensor behavior. Testnet examples add shared
non-production conditions, which can reveal integration behavior without touching mainnet state.

On mainnet, BT Logging Levels examples should be read as live production diagnostic behavior on the
production Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, so outcomes from one environment should not be treated as
proof of behavior in another.
