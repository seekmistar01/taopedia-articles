# Subnet Task

A subnet task is the work target miners are expected to perform inside a Bittensor subnet. It gives
the subnet a specific digital-commodity goal, while validators evaluate how well miner outputs serve
that goal ([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

The term is narrower than the whole incentive mechanism. A subnet task names what work is being
produced; the incentive mechanism also includes the evaluation design and reward logic around that
work
([Glossary: Incentive Mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## Work Target

The task is the subnet's work definition. Different subnets can ask miners to produce different
outputs, so the task keeps subnet vocabulary tied to the market being discussed rather than to one
universal type of work
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

This makes subnet task a useful boundary term. It answers what work the subnet asks for before the
article moves to who performs the work, how it is evaluated, or how rewards are calculated
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task)).

## Miner and Validator Context

Miners produce work for the task, while validators evaluate that work. A subnet miner is the
task-performing role, and a subnet validator is the role that measures miner performance and sets
weights from the evaluation
([Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner),
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator)).

The task is therefore not the same as either role. It is the work target that makes those roles
meaningful inside the subnet: miners are judged against it, and validators produce evaluations about
it
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## Incentive Mechanism Context

An incentive mechanism combines the task, protocol, and evaluation design. The subnet task is the
work component of that larger mechanism, while evaluation logic determines how validator scores are
formed
([Glossary: Incentive Mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

This distinction helps avoid treating a task statement as a full reward design. A task can say what
miners should produce, but it does not by itself explain all scoring rules, validator behavior, or
emission outcomes ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Weights and Consensus Context

Validator weights are the evaluation signals that validators submit after judging miner work. Yuma
Consensus then aggregates those submitted signals into miner incentives and validator dividends
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The subnet task sits before those consensus outcomes. It defines the work being evaluated; weights
carry evaluations of that work; Yuma Consensus aggregates those weights into incentive results
([Glossary: Incentives](https://docs.learnbittensor.org/resources/glossary#incentives),
[Glossary: Dividends](https://docs.learnbittensor.org/resources/glossary#dividends)).

## Evaluation Quality Context

Weight copying and Commit Reveal sit near subnet-task vocabulary because both relate to validator
evaluation quality and visibility. Weight copying is a failure mode around independent evaluation,
while Commit Reveal controls when fresh weights become visible before consensus uses them
([The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

Those concepts do not redefine the task. The task names the work target; weight copying and Commit
Reveal describe issues or protections around how evaluations of that work reach consensus
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task)).

## Mechanism and Network Context

Some subnets can run multiple incentive mechanisms, and each mechanism can define separate task
context. A task for one mechanism should not be treated as the task for every mechanism in the same
subnet
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)).

Network environment also matters. Localnet, testnet, and mainnet are separate contexts, so a task
example from one environment is evidence about that environment rather than a universal statement
about every Bittensor subnet
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Introduction to Bittensor: Subnet development](https://docs.learnbittensor.org/learn/introduction#subnet-development)).

## Reader Boundary

Subnet task is reference vocabulary for requested work inside a subnet. It is not a miner
performance report, validator strategy, reward guarantee, or proof that every subnet uses the same
evaluation standard. Concrete examples require the selected subnet, mechanism context, network
environment, and relevant official source
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).
