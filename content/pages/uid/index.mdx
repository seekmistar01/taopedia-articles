# UID

A UID is the unique identifier assigned to each participant position in a Bittensor subnet. Each
miner or validator registered to a subnet holds a UID that identifies their position within that
subnet's participant set
([Glossary: UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

UID is identifier vocabulary. It names the number, not the position structure itself. UID slot is
the related term for the participant position that the UID identifies.

## Subnet-Specific Scope

UIDs are subnet-specific. The same UID number can appear in different subnets while referring to
different participants. A UID identifies a position within one subnet's participant set, not a
network-wide identity
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

Netuid identifies the subnet itself, while UID identifies a participant within that subnet
([Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid)). The two terms
answer different questions about the same subnet context.

## Registration and UIDs

A participant receives a UID by registering to a subnet. Registration assigns a UID to a hotkey,
placing that participant into a UID slot within the subnet's capacity
([Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register)).

That means a UID is earned through registration, not self-assigned. If the subnet is at capacity,
registration replaces the lowest-ranked existing participant through the immunity and deregistration
rules.

## Metagraph Context

The metagraph records each subnet's participant set. UIDs appear in metagraph data as identifiers
for the miners and validators indexed in that subnet's state
([Glossary: Metagraph](https://docs.learnbittensor.org/resources/glossary#metagraph)).

Reading a UID from metagraph data gives a position identifier within the subnet's current
participant set. The position's associated state — stake, rank, trust — is indexed by that UID.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For UIDs, that
sequence changes which subnet participant set a reader should assume when matching a UID to a miner
or validator.

In localnet, UID assignment can be tested through isolated subnet registration. Localnet UID slots
do not represent production participant sets on live subnets.

On testnet, registration assigns UIDs on a shared non-production network. Testnet metagraph entries
are separate from mainnet subnet state
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

On mainnet, a UID identifies a participant position on the selected
[netuid](https://docs.learnbittensor.org/resources/glossary#netuid) in live metagraph data
([Glossary: Metagraph](https://docs.learnbittensor.org/resources/glossary#metagraph)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A UID example from one environment should not be read as
identifying a participant on another network.

## Relationship to Yuma Consensus

UID and Yuma Consensus describe related parts of Bittensor's incentive system. Yuma Consensus is the
on-chain process that aggregates validator weight signals within a subnet into miner incentives and
validator dividends, applying consensus clipping, bonding, and emission calculation
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

For readers, uid names a specific part of that incentive picture, while Yuma Consensus names the
consensus process that turns validator weights into the resulting incentives and dividends.

## Reader Boundary

UID is concept vocabulary for subnet participant identification. It should not be read as a network
address, a balance container, or a cross-subnet identity
([Glossary: UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot)).

UIDs can change through deregistration. A UID that identifies one participant today may identify a
different participant in the same subnet after a registration event
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## Immunity Can Shield a New UID From Displacement

[Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)
documentation explains that when a subnet is full, a new registration can replace the
lowest-emission eligible entrant. An entrant still inside the
[immunity period](https://docs.learnbittensor.org/resources/glossary#immunity-period) is protected
from that ordinary removal path.

That timing rule attaches to the UID slot, not to the hotkey in isolation. A newly registered UID
can keep its slot while immunity lasts even if performance is still developing.

## Full Subnets Replace the Lowest Eligible UID

When every [UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot) is occupied, the
subnet reaches its `max_allowed_uids` ceiling and new
[registration](https://docs.learnbittensor.org/resources/glossary#register) attempts depend on
displacement rules rather than empty slots
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

UID vocabulary names which position changes hands. Displacement removes one eligible incumbent so a
new registrant can take that numbered slot under subnet rules.

## Weight Vectors Score Miners at Their UIDs

[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) describes validators
submitting vectors that rank miner work they have evaluated. Those scores attach to miners
identified by UID inside the subnet participant set, not as a network-wide identity label
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

Readers tracing consensus inputs should pair UID with
[netuid](https://docs.learnbittensor.org/resources/glossary#netuid). The same numeric UID on
different subnets refers to different participant positions.

## Relationship to Hotkey

UID and hotkey are related but different registration terms. UID names the subnet-local slot number
a participant occupies inside one subnet, while hotkey names the account key that signs operations
and registers to that subnet
([Glossary: UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Glossary: Hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey)).

For readers, the UID is the position a registration produces and the hotkey is the key that requests
it. Registration assigns a UID to a hotkey, so the hotkey is the account identity while the UID is
the subnet slot that identity earns
([Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register)).

These terms sit on different layers. A hotkey is the same key identity across subnets, while the
same numeric UID on different subnets refers to different participant positions, so one hotkey can
hold different UIDs in different subnets
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

Readers should not treat a UID as proof of a specific hotkey across the network, or a hotkey mention
as a substitute for the per-subnet UID it currently holds inside a chosen subnet.
