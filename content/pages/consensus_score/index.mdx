# Consensus Score

A consensus score is the agreement benchmark Yuma Consensus uses when validator weights are combined
into final rank results. It is a stake-weighted median of validator-assigned weights inside the Yuma
process
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The term matters because validator signals can disagree. Consensus score names the benchmark used to
limit outlier weights before the filtered result contributes to rank.

In plain language, consensus score answers: what is the agreement level among validators for this
miner, measured as a stake-weighted median? Validators whose weights are far from that median are
treated as outliers and have their influence reduced in the final rank calculation
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

For a reader, consensus score differs from a single validator's view. One validator might rate a
miner highly, but if most other validators disagree, that lone signal contributes less to the final
rank after consensus filtering
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

That benchmark belongs inside the Yuma process. It is calculated from validator-submitted weights,
then used before final rank results are produced
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For consensus
score on a subnet such as netuid 1, that sequence changes how readers should interpret the
stake-weighted benchmark, the validator set that produced it, and the rank and trust outcomes that
follow clipping.

In localnet, consensus-score examples can be exercised in an isolated environment. Local validator
weights and stake-weighted medians reflect local chain configuration rather than production subnet
consensus filtering.

On testnet, consensus score can be observed in a shared, non-production network. Testnet benchmark
readings on a selected netuid are separate from mainnet validator-weight and stake-weight state
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

On mainnet, consensus score is the live stake-weighted agreement benchmark Yuma Consensus uses when
filtering production validator weights before rank and trust are calculated on the connected
Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A consensus score example from one environment should not
be read as representing production benchmark behavior on another network.

## Distinction from Kappa

Kappa is the subnet hyperparameter that sets how much stake-weighted validator support must back a
weight level before that level becomes the clipping ceiling. Consensus score is the stake-weighted
median benchmark Yuma compares individual weight values against after that ceiling is derived
([Subnet Hyperparameters: Kappa](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#kappa),
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping)).

- **Kappa** — configured stake fraction input to the clipping ceiling.
- **Consensus score** — resulting agreement benchmark used during clipping.

## Distinction from Validator Trust

Consensus score is the stake-weighted median benchmark used when clipping outlier weights. Validator
trust sums clipped weights a validator successfully contributed across neurons after filtering
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Consensus score** — agreement benchmark inside the clipping step.
- **Validator trust** — validator-side influence after clipping.

## Distinction from Rank

Consensus score is the stake-weighted benchmark used when clipping outlier weights. Rank is the
miner-side standing read after that clipping and aggregation on the selected subnet
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping)).

- **Consensus score** — agreement benchmark inside the clipping step.
- **Rank** — miner-side outcome after filtered weights are combined.

## Distinction from Trust

Consensus score shapes clipping before rank is finalized. Trust compares pre-rank support to final
rank after that step on the selected subnet
([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

Use consensus score for the benchmark; use trust for support preserved through clipping.

## Distinction from Weight Vector

A weight vector is one validator's submitted miner scores. Consensus score is the stake-weighted
median benchmark derived from many such submissions on the selected subnet
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

Weight vectors are inputs; consensus score is the agreement level those inputs produce before rank.

## Distinction from Weight Matrix

A weight matrix stacks validator vectors into subnet-wide consensus input. Consensus score is read
from that grouped input when clipping filters individual weight values on the selected subnet
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Weight matrix** — assembled validator input.
- **Consensus score** — clipping benchmark derived from that input.

## Distinction from Incentives

Consensus score belongs to the clipping step inside Yuma. Incentives are miner-facing emission
outcomes credited after rank is normalized later in the same pass
([Glossary: Incentives](https://docs.learnbittensor.org/resources/glossary#incentives),
[Yuma Consensus: Miner emissions](https://docs.learnbittensor.org/learn/yuma-consensus#miner-emissions)).

- **Consensus score** — clipping benchmark inside Yuma.
- **Incentives** — miner-facing credited results after rank is normalized.

Consensus score answers how clipping is benchmarked; incentives answer what miners are credited.

## Distinction from Dividends

A consensus score and a validator dividend are different objects in the same Yuma Consensus pass on
a subnet such as netuid 1. The consensus score is the stake-weighted agreement benchmark Yuma uses
to clip validator weight signals for a miner — an evaluation intermediate computed during the pass.
Validator dividends are a reward outcome: the validator-side emission share produced after that
pass, shaped by validator-miner bonds and the miner outcomes those bonds point at. One is a
benchmark used inside the calculation; the other is a payout that follows from it
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions)).

The two should not be read as the same thing. A consensus score does not state how much any
validator earns; it states the agreement level a miner's weights are measured against on netuid 1. A
dividend is the downstream emission a validator receives once consensus and bond accounting resolve,
so a consensus-score figure is not a dividend figure, and a dividend does not report a miner's
consensus benchmark
([Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds),
[Emission: Distribution](https://docs.learnbittensor.org/learn/emissions#distribution)).

This parallels the article's distinction from miner incentives: just as the consensus score is not
the miner-side reward, it is also not the validator-side reward. It is the shared agreement
benchmark that sits upstream of both the incentive paid to miners and the dividend paid to
validators on netuid 1 ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Consensus score** — the stake-weighted agreement benchmark Yuma uses when clipping weights on
  netuid 1.
- **Validator dividend** — the validator-side emission outcome produced after that consensus pass.

## Distinction from Weight Copying

Weight copying is when a validator reuses visible weight signals instead of independently evaluating
miners. Consensus score is the stake-weighted median benchmark Yuma Consensus uses to clip outlier
weights, which is the same benchmark copiers may try to forecast from visible submissions
([Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Weight copying** — dependence on another validator's published scores.
- **Consensus score** — clipping benchmark derived from validator submissions.

## Distinction from Subnet Scoring Model

A subnet scoring model is the subnet-specific method a validator uses to judge returned miner work
before submitting weights. Consensus score is the stake-weighted agreement benchmark Yuma Consensus
derives from those submitted weights when clipping outliers on the selected subnet
([Glossary: Subnet Scoring Model](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model),
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Scoring-model vocabulary names upstream per-validator evaluation. Consensus-score vocabulary names
the downstream network benchmark those evaluations are compared against after submission.

## Distinction from Self Weighting

Self-weighting is when a validator assigns inflated weights to entities it controls. Consensus score
is the stake-weighted agreement benchmark Yuma Consensus uses to clip outlier weights that depart
from broader validator agreement on the selected subnet
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Exploit vocabulary names the self-preferencing weight pattern; consensus-score vocabulary names the
clipping benchmark that helps limit how far such weights can depart from stake-weighted agreement.

## Distinction from Yuma Consensus

A consensus score is the stake-weighted agreement figure Yuma uses when it clips validator weights
that depart too far from the group. It names one clipping input inside the tempo pass, not the full
settlement process that follows
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping)).

Yuma Consensus is the broader on-chain tempo step. It reads submitted validator weights, applies
clipping against that agreement benchmark, runs bonding, and then assigns miner incentives and
validator dividends on the selected netuid
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission: Distribution](https://docs.learnbittensor.org/learn/emissions#distribution)).

The subnet scoring model section above covers how validators judge work before submission. Consensus
score vocabulary belongs after submission, when Yuma compares individual weight signals to the
shared benchmark
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

Readers should not treat a consensus score as a payout quote or a miner quality grade on its own. It
is the agreement reference Yuma uses while filtering weight signals before rank and emission
outcomes are assigned ([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

- **Consensus score** — stake-weighted agreement benchmark used when clipping weights.
- **Yuma Consensus** — tempo step that turns submitted weights into emission shares.

## Reader Boundary

Consensus score should not be read as a payout amount, a validator trust score by itself, or proof
that a miner received a specific reward. It names the stake-weighted benchmark used when Yuma
Consensus filters validator weight signals
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).
