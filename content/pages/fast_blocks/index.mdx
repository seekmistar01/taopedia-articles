# Fast Blocks

[Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks) are a development-only
local-chain configuration that accelerates block production. The term belongs to local Bittensor
testing context, not to live mainnet or testnet behavior.

## Local Development Context

Fast blocks are tied to local development. A
[local blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain) is private,
isolated, and used for developing and testing subnet incentive mechanisms. In that setting, faster
block production helps shorten local feedback loops.

## Relationship to Local Blockchain

Fast blocks and local blockchain are related but name different parts of local development context.
The
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)
describes a private chain used for development and testing, while
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks) describes
accelerated block production inside that local setting.

For readers, local blockchain is the isolated development chain, while fast blocks are a timing
configuration used within it. Fast blocks do not by themselves describe mainnet or testnet behavior.

References:
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)

## Relationship to Localnet

Bittensor separates
[mainnet, testnet, and localnet](https://docs.learnbittensor.org/concepts/bittensor-networks)
contexts. Fast blocks sit on the localnet side of that distinction: they describe a local testing
configuration rather than a property of Bittensor network production behavior.

## What the Term Clarifies

[Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks) connect accelerated
block production with rapid local testing. The useful distinction is not that fast blocks are a
separate network or a subnet role; they are a timing behavior inside a local development chain.

## Scope of the Term

[Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks) name local timing
behavior. They do not describe a separate public network, a subnet role, or live production
behavior.

## Relationship to Non-fast Blocks

Fast blocks and non-fast blocks name two local timing configurations. The
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks) describes
accelerated local block production, while the
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)
describes local blocks that follow Subtensor's default 12-second interval.

For readers, fast blocks shorten local feedback loops, while non-fast blocks simulate
production-like timing during development. Both belong to local blockchain context rather than live
mainnet behavior.

References: [Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks),
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)

## Relationship to Bittensor Networks

Fast blocks and Bittensor Networks name related but different parts of local development context.
The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
groups mainnet, testnet, and localnet, while
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks) describes
accelerated block production inside that localnet setting.

For readers, Bittensor Networks is the environment label, while fast blocks are a timing
configuration used within localnet. Fast blocks do not describe live mainnet or testnet behavior.

References: [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)

## Relationship to Subtensor

Fast blocks and Subtensor are related but different parts of Bittensor chain vocabulary. Subtensor
names Bittensor's layer 1 blockchain and system of record, while fast blocks name a development-only
local configuration that accelerates block production on a private Subtensor instance. The
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor) describes the
blockchain layer, and
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks) describes
accelerated local block production.

For readers, Subtensor names the chain layer, while fast blocks name a local timing configuration
used when testing on that layer in development.

References: [Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)

## Relationship to Block

Fast blocks and a block are related but different parts of Bittensor chain timing vocabulary. A
[block](https://docs.learnbittensor.org/resources/glossary#block) names one unit of chain
progression on Subtensor, while fast blocks name a local configuration that produces those units
more quickly during development testing. The
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks) describes
accelerated local block production rather than live mainnet or testnet timing.

For readers, a block is the chain progression unit, while fast blocks name how quickly that unit is
produced inside a local development chain.

References: [Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)

## Relationship to Extrinsics

Fast blocks and extrinsics are related but different parts of Bittensor local development
vocabulary. Extrinsics name signed transactions and calls submitted to Subtensor, while fast blocks
shorten the local wait between block inclusions when developers test those submissions on a private
chain. The [Glossary: Extrinsics](https://docs.learnbittensor.org/resources/glossary#extrinsics)
describes signed transactions submitted to the chain, and
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks) describes
accelerated local block production.

For readers, extrinsics name the submitted chain actions under test, while fast blocks name a local
timing setting that can shorten feedback loops around their inclusion.

References: [Glossary: Extrinsics](https://docs.learnbittensor.org/resources/glossary#extrinsics),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)

## Relationship to Tempo

Fast blocks and tempo are related but different timing concepts in Bittensor vocabulary.
[Tempo](https://docs.learnbittensor.org/resources/glossary#tempo) names the number of blocks between
subnet reward calculation cycles, while fast blocks name accelerated local block production that can
shorten how quickly those block-based cycles advance during development testing. The
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks) describes
local timing behavior rather than live mainnet or testnet cadence.

For readers, tempo counts blocks between reward cycles, while fast blocks change how quickly blocks
are produced inside a local development chain.

References: [Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)

## Relationship to Coinbase

Fast blocks and coinbase are related but different parts of Bittensor block-timing vocabulary. Fast
blocks name a development-only local configuration that accelerates block production on a private
chain, compressing the time between successive blocks. Coinbase is the per-block Subtensor mechanism
that accumulates pending emissions and injects them into subnet pools, triggering Yuma Consensus
rounds when each subnet reaches its epoch boundary. The
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks) describes
accelerated local block production, and the
[Coinbase Implementation](https://docs.learnbittensor.org/subtensor-api/extrinsics#coinbase)
describes the per-block emission mechanism.

Fast blocks change how quickly blocks arrive in a local development chain without changing what
coinbase does inside each block. With fast blocks enabled, coinbase still runs each block —
accumulating pending emissions and checking epoch boundaries — but those invocations happen in
compressed wall-clock time. This lets developers observe multiple coinbase cycles and full epoch
transitions in seconds rather than waiting through full production block intervals.

References: [Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks),
[Coinbase Implementation](https://docs.learnbittensor.org/subtensor-api/extrinsics#coinbase)
