# Subtensor Error Codes

Subtensor error codes are diagnostic signals from the blockchain layer beneath Bittensor. They help
identify why a chain action failed, but they do not provide the full surrounding context by
themselves ([Subtensor Error Codes](https://docs.learnbittensor.org/errors),
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor),
[Subtensor API Errors](https://docs.learnbittensor.org/subtensor-api/errors)).

The term is useful as rejection vocabulary. An error code points toward a rule family, while the
relevant action, subnet context, network context, and mechanism documentation explain how that
family should be interpreted.

## Diagnostic Scope

The Subtensor error overview groups failures into standard Subtensor errors, custom errors, and
errors inherited from the underlying Substrate framework
([Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

That grouping makes error codes classification signals. A short label can tell a reader where to
start, but it should not be treated as a complete reconstruction of every condition that led to the
rejection.

The public error overview also describes returned failures with separate `code`, `message`, and
`data` fields ([Subtensor Error Codes](https://docs.learnbittensor.org/errors)). Those fields keep
two layers of meaning apart. The numeric or named error identifies the broad failure signal, while
the surrounding message and data describe how that signal was exposed by the chain interface.

The generated Subtensor API errors reference adds another layer: runtime error variants are grouped
by pallet and exposed through paths shaped like `api.errors.<Pallet>.<ErrorName>`
([Subtensor API Errors](https://docs.learnbittensor.org/subtensor-api/errors)). That format keeps
the runtime area and error name together, which helps readers avoid treating a short error name as
complete without its owning pallet.

The generated page also pairs each `api.errors` path with a short summary. For example, an error
entry can identify the pallet, exact API label, and short description attached to that runtime error
variant ([Subtensor API Errors](https://docs.learnbittensor.org/subtensor-api/errors)).

That distinction matters for interpretation. A reader can use the returned format to identify the
error family, then use the standard-error or custom-error references to understand what kind of
Subtensor rule was involved
([Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor),
[Subtensor Custom Errors](https://docs.learnbittensor.org/errors/custom)).

## Standard and Custom Families

Standard errors describe named failure categories from the Subtensor layer, while custom errors list
numeric conditions with short descriptions
([Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor),
[Subtensor Custom Errors](https://docs.learnbittensor.org/errors/custom)).

Both families are lookup vocabulary. A standard error can name a broad Subtensor condition, and a
custom error can identify a more specific chain condition. Neither family, by itself, proves the
entire cause without the surrounding chain context.

The generated API reference complements those lists by showing error variants in their runtime area.
A balances error, a system error, and a subtensor-module error are all Subtensor-facing failure
vocabulary, but the pallet grouping narrows which runtime area supplied the error variant
([Subtensor API Errors](https://docs.learnbittensor.org/subtensor-api/errors)).

Substrate errors form the third family in the upstream overview. They are inherited from the
underlying framework on which Subtensor is built, so they are best read as chain-layer failures
rather than as Bittensor-specific mechanism names
([Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

The family label also says something about where to look next. Standard and custom Subtensor errors
point back to Bittensor chain rules, while inherited Substrate errors may reflect lower-level
transaction, dispatch, or framework checks before a reader reaches a Bittensor mechanism-specific
explanation.

That makes the family label a routing aid. It points the reader toward the relevant error table or
runtime area, while the action that failed still supplies the rest of the interpretation context.

## Family and Cause Boundary

An error family is not always the whole cause. A balance-related failure identifies a balance
condition, while a subnet-related failure identifies a subnet condition, but the reason that
condition appeared depends on the action and context being checked
([Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor),
[Subtensor Custom Errors](https://docs.learnbittensor.org/errors/custom)).

This keeps error-code prose different from a troubleshooting transcript. A reference article can
name the failure vocabulary and explain how to read it across many possible cases without
reproducing tool output or prescribing a recovery path.

The same label can also appear near different kinds of actions. For example, a balance, stake,
subnet, authorization, or rate-limit label identifies the rejected condition, not the whole user
intent that produced it
([Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor),
[Subtensor Custom Errors](https://docs.learnbittensor.org/errors/custom)).

That is why error codes are most useful as anchors for reading other chain references. They narrow
the failure domain, but storage state, emitted events, rate-limit rules, and the submitted action
still provide the context that turns a label into a full explanation.

## Standard Errors Name Recognizable Subtensor Conditions

The Subtensor Standard Errors reference lists specific named conditions such as
`SubNetworkDoesNotExist`, described as the specified subnet not existing, `NotEnoughStake`,
described as the caller lacking enough stake to perform the action, and
`HotKeyNotRegisteredInSubNet`, described as the hotkey not being registered in the specified subnet
([Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor)).

These names give the family label a concrete face. A `NotEnoughStake` condition and a
`SubNetworkDoesNotExist` condition both come from the standard-error family, but they point to
different missing preconditions: an account-level balance check versus a subnet-existence check.

## Authority and Coldkey-Association Errors Gate Who Can Act

The same reference lists `NonAssociatedColdKey` for a request made by a coldkey not associated with
the targeted hotkey account, and `BadOrigin` for a wallet that lacks the required root or
subnet-owner permission for the attempted call
([Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor)).

These two named errors connect standard-error vocabulary to the signing-authority boundary already
covered above: a coldkey-hotkey mismatch and a missing-permission condition are both authority
failures rather than balance or subnet-existence failures.

## Custom Errors Use Numbered Codes With Short Descriptions

The Subtensor Custom Errors reference lists numbered conditions such as error 2, `BalanceTooLow`,
described as the stake amount held being less than the amount requested, and error 3,
`SubnetDoesntExist`, described as the named subnet not existing
([Subtensor Custom Errors](https://docs.learnbittensor.org/errors/custom)).

Reading a custom-error number alone is not enough; the same reference's accompanying name and
description give the number its meaning, consistent with the surrounding-context point already made
above.

## A Documented Minimum Stake Amount Has Its Own Custom Error

The same reference lists error 1, `StakeAmountTooLow`, for a staked or unstaked amount that falls
below a documented minimum of 2,000,000 RAO, equal to 0.002 TAO
([Subtensor Custom Errors](https://docs.learnbittensor.org/errors/custom)).

That figure is the documented floor checked by this specific custom error, separate from the general
balance and stake-availability conditions named elsewhere in the same reference.

## Coldkey Swap Scheduling and Liquidity Each Have a Named Custom Error

The same reference lists error 0, `ColdKeyInSwapSchedule`, for a coldkey scheduled to be swapped
that therefore cannot process transfer operations, and error 7, `InsufficientLiquidity`, for a
subnet pool that lacks enough liquidity for the attempted transaction
([Subtensor Custom Errors](https://docs.learnbittensor.org/errors/custom),
[Rotate or Swap your Coldkey](https://docs.learnbittensor.org/keys/coldkey-swap)).

These two named errors connect custom-error vocabulary to topics already covered elsewhere: a
coldkey-swap-in-progress condition and a pool-liquidity condition each surface through their own
specific numbered error rather than a generic rejection.

## Distinction from Extrinsics

An extrinsic names the runtime call being attempted. A Subtensor error code classifies why
validation or execution failed ([Subtensor Error Codes](https://docs.learnbittensor.org/errors),
[Subtensor Extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor)). The extrinsic
identifies intent and method; the error code explains the failure reason returned by the runtime.

- **Subtensor error codes** — failure classification returned by the runtime.
- **Extrinsics** — the signed call whose validation or execution failed.

## Distinction from Subtensor Events

A Subtensor error code and an event answer opposite outcomes. An error code names the rejection
category when an intended state change does not complete, while an event names a record the runtime
emitted while processing activity ([Subtensor Error Codes](https://docs.learnbittensor.org/errors),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

- **Error code** — the rejection category when an action fails.
- **Subtensor event** — a record emitted while the runtime processes activity.

## Distinction from Rate Limits

A Subtensor error code and a rate limit name a signal and a rule. A rate limit is the block-based
timing rule that paces a repeated action, while an error code such as `SettingWeightsTooFast` is the
diagnostic returned when that timing rule is violated
([Subtensor Error Codes](https://docs.learnbittensor.org/errors),
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

- **Rate limit** — the block-based timing rule that paces an action.
- **Error code** — the diagnostic returned when a rule like that is violated.

A weight-submission attempt made too soon after a previous one surfaces the specific
`SettingWeightsTooFast` standard error rather than a generic rejection
([Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor),
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

## Distinction from Bittensor Logging Levels

Bittensor logging levels and Subtensor error codes answer different diagnostic questions. A logging
level describes how the software classified a message's severity or verbosity, while a Subtensor
error code names the rule family or failure category returned from the chain layer when validation
or execution fails ([Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors),
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor)).

- **Bittensor logging levels** — how the software labels message severity and verbosity in local
  output.
- **Subtensor error codes** — chain-layer classifications that identify which rule failed or which
  failure family applies.

## Distinction from Yuma Consensus

Subtensor error codes classify why a chain action was rejected. They name diagnostic vocabulary for
failed submissions, not the settlement step that turns validator weights into emission shares
([Subtensor Error Codes](https://docs.learnbittensor.org/errors),
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor)).

Named standard errors such as `SettingWeightsTooFast` and `NotEnoughStake` illustrate the boundary.
Those labels identify rejected preconditions on a call, while Yuma Consensus names how submitted
weights become emission shares after evaluation completes
([Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers comparing a failed weight submission to subnet settlement should start from the error
family, then read Yuma Consensus documentation only when the question is how weights become shares
rather than why a call failed
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

- **Subtensor error codes** — failure labels for rejected chain actions.
- **Yuma Consensus** — on-chain weight aggregation that settles emissions.

## Distinction from Rate-Limit

Rate-limit failures are easy to misread because they involve timing. Bittensor rate-limit
documentation explains that selected chain actions use block-based cooldowns
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

That timing model is narrower than a generic wait-and-try-again rule. Some errors point to rate
limits, while other errors point to missing state, invalid parameters, balance conditions, authority
conditions, or Subtensor checks.

## Distinction from Chain Signal

Error codes sit beside other chain-observation surfaces. Storage entries describe recorded state,
events describe emitted runtime records, and errors describe rejection categories when an intended
state change does not complete
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

Those signals are strongest when read in their own lanes. Storage is not an error, an event is not a
full diagnosis, and an error code is not a complete description of the underlying Bittensor
mechanism.

The official overview notes that errors can surface through client, SDK, or extrinsic interfaces
([Subtensor Error Codes](https://docs.learnbittensor.org/errors)). The conceptual meaning is
therefore the chain signal, not the particular interface that displayed it.

The generated API path is also an interface label, not the whole failure story. It gives a stable
way to name the runtime error variant, while storage reads, emitted events, and the submitted action
describe the surrounding chain context
([Subtensor API Errors](https://docs.learnbittensor.org/subtensor-api/errors),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

The submitted action is often an extrinsic, which is the transaction or call surface used to ask
Subtensor to process a runtime action. Reading the error label beside the extrinsic keeps the
failure vocabulary attached to what the chain was asked to do
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

This keeps the concept portable. A reader comparing an error seen through a wallet, an application,
or a chain transaction interface can still start from the same Subtensor error family, then add the
interface-specific context only after the chain-layer signal is understood.

## Reader Boundary

Subtensor error codes do not name a complete recovery guide, a full incident report, or a guarantee
that the same label has the same practical meaning in every context
([Subtensor Error Codes](https://docs.learnbittensor.org/errors),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Subtensor error codes identify rejection vocabulary, while rate limits, storage, events, subnet
settings, and mechanism references describe neighboring parts of the system.

## Custom and Standard Errors Need Surrounding Context

Custom errors may expose a number first, but the custom-error table supplies the Bittensor-specific
description that gives that number meaning
([Subtensor Custom Errors](https://docs.learnbittensor.org/errors/custom)).

Standard errors name recognizable Subtensor conditions, but the short name still needs the action
and network around it.

## Network Boundary

A localnet or testnet failure can illustrate an error family, but it should not be treated as proof
about another network environment
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).
