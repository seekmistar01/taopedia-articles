# Batch Transactions

Batch transactions group multiple Bittensor chain actions into one outer transaction
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

The term describes packaging. A batch can carry several inner calls, but the calls still keep their
own purpose, fee context, and possible outcome.

## What It Represents

The batch documentation names three utility-pallet variants: `batch`, `batch_all`, and
`force_batch`. The variants share the same grouping idea but differ in how they handle inner-call
failure ([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

`batch` stops at the first failed inner call while earlier successful calls remain applied.
`batch_all` rolls back all inner calls if any one fails. `force_batch` continues attempting later
calls after an earlier failure
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

Batching changes grouping, not the rules inside each call. Inner actions still incur their own fees
plus wrapper overhead, and each call still requires the signer authority that action demands
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Bittensor CLI: Permissions Guide](https://docs.learnbittensor.org/btcli/btcli-permissions)).

SDK helpers that submit multiple individual extrinsics in sequence are not the same as a utility
batch extrinsic. A helper name containing "multiple" does not prove that `batch`, `batch_all`, or
`force_batch` was used on chain
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

Observed batch records depend on which network you connect to. Examples from one environment should
not be read as production chain activity on another
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

## Distinction from Transfer

Batch transactions group multiple chain actions into one outer transaction. A transfer is a single
action that sends TAO from one wallet address to another
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions),
[Glossary: Transfer](https://docs.learnbittensor.org/resources/glossary#transfer)).

- **Batch transactions** — one outer transaction bundling multiple chain actions.
- **Transfer** — a single action sending TAO between wallet addresses.

## Distinction from Mempool

Batch transactions package several inner calls into one outer transaction. The mempool is the queue
where that outer transaction waits before inclusion
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

- **Batch transactions** — grouped outer submission carrying several inner calls.
- **Mempool** — pre-inclusion queue for submitted transactions.

## Distinction from Extrinsics

A batch transaction is a utility-pallet extrinsic that carries several runtime calls. An ordinary
extrinsic is a single submitted call; inner calls keep their own method meaning and per-call result
handling but dispatch from the batch's origin
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

- **Extrinsic** — a single submitted runtime call.
- **Batch transaction** — a utility extrinsic that carries several calls as one transaction.

## Distinction from Transaction Fees

A batch groups calls into one outer submission. Fee documentation still charges inner-call fees plus
batch wrapper overhead, so grouping does not make the inner actions free
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

- **Batch transaction** — how calls are grouped into one submission.
- **Transaction fee** — the cost, charged as inner-call fees plus wrapper overhead.

## Distinction from Runtime Calls

Batch transactions group multiple inner actions under one outer submission. Runtime calls are the
individual methods the runtime evaluates inside that wrapper
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

- **Batch transactions** — grouped outer packaging with a failure model.
- **Runtime calls** — the inner methods each call executes.

## Distinction from Yuma Consensus

Batch transactions group multiple runtime calls into one outer extrinsic submission on Subtensor.
Yuma Consensus is the epoch-boundary mechanism that converts eligible validator weight submissions
into emission shares on a subnet
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Batch transaction** — groups multiple runtime calls under one outer submission.
- **Yuma Consensus** — epoch-boundary settlement from validator weights to emission shares.

## Reader Boundary

Batch transactions identify grouping behavior. They are not a fee waiver, a guarantee that every
inner call succeeds, or a replacement for the individual action references
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

Use the batch term when the grouping behavior matters. Use the individual extrinsic reference when
the question is what a particular inner call does.
