Extrinsics are submitted actions that ask the Bittensor chain to perform a runtime operation. The
official [Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) reference
lists extrinsic methods exposed by the Subtensor runtime.

For Taopedia readers, the useful idea is that an extrinsic is a state-changing chain action, not a
passive read. Reading chain data is different from submitting an action that the runtime must
process.

## How Extrinsics Fit

Extrinsics are the transaction-like side of interacting with Subtensor. They are used when software
or a user asks the chain to update state through a supported runtime call.

The official [transaction fees](https://docs.learnbittensor.org/learn/fees) page is useful context
because it separates chain reads from state-changing actions. Chain reads are free, while
state-changing actions can carry fees depending on the operation.

## Why the Distinction Matters

The read-versus-submit distinction helps explain why some Bittensor actions only inspect current
state, while others ask the chain to record a change. An extrinsic belongs to the second group: it
is evaluated by the runtime and can affect chain state if accepted.

This article does not try to catalog every extrinsic. The official reference is the right source for
the current method list and exact behavior.

## Relationship to Mempool

Extrinsics and the mempool are related but name different parts of a submitted chain action. An
extrinsic is a state-changing runtime request, while the
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) is where pending
transactions wait before block inclusion.

For readers, an extrinsic is the submitted action itself, and the mempool is the pending holding
period before that action may be bundled into a block. A chain read is not an extrinsic, and a
pending mempool entry has not yet become accepted chain history.

References: [Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)

## Relationship to Blocks

Submitted extrinsics are bundled into blocks for the runtime to process. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#block) describes a block as a unit of
chain data containing transactions, so an accepted extrinsic becomes part of a block rather than
remaining a standalone request. For readers, this keeps the extrinsic as the submitted action and
the block as the unit that records and orders those actions on chain.

References: [Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)

## Relationship to Multiple Mechanisms

Extrinsics are state-changing chain actions processed by the Subtensor runtime. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-matrix) notes that each
incentive mechanism has its own weight matrix for independent consensus calculations.

For readers, an extrinsic still names a submitted runtime action. Weight-related extrinsics in a
subnet that runs more than one incentive mechanism should be read with that mechanism's matrix
context.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)
