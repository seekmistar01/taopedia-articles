# Subtensor Constants

Subtensor constants are named runtime values exposed by the official
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants) reference. That
source describes the page as compile-time runtime constants for the Subtensor runtime and their
respective values.

## What Constants Represent

The constants reference lists named entries under runtime areas. For each entry, it gives the
constant name, data type, access path, value, and a short summary where one is available. This makes
the page a compact reference for runtime-defined values, not a full explanation of the feature that
uses each value.

Reference: [Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)

## Runtime Areas

The official reference is grouped by runtime areas such as aura, balances, commitments, contracts,
crowdloan, drand, proxy, registry, safe mode, scheduler, system, timestamp, utility, swap, and the
Subtensor module. That grouping helps readers identify the part of the runtime that owns a constant
before reading the entry name and summary.

Reference: [Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)

## Bittensor-Specific Context

The Subtensor module section contains Bittensor-specific constants for subnet defaults, registration
bounds, rate limits, weight-related defaults, Liquid Alpha status, and Yuma3 status. Other sections
cover broader runtime areas such as account balances, contracts, proxy behavior, scheduled calls,
system limits, and timestamp timing.

Reference: [Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)

## Relationship to Multiple Mechanisms

Subtensor constants are protocol parameters on the Bittensor chain layer. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism) notes that each
subnet has one or more incentive mechanisms.

For readers, constants still name chain-layer settings.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Incentive Mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism)

## Relationship to Transaction Fees

Subtensor constants and transaction fees are related but describe different parts of the Bittensor
runtime. Subtensor constants are named compile-time values exposed by the runtime reference across
areas including balances and system, while transaction fees are the runtime-computed costs that
depend on weight and length parameters drawn from those constant areas. The
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants) reference lists
entries for the balances and system runtime areas that contain weight-related parameters, and the
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) documentation explains
how fee amounts are computed from weight and length components.

For readers, a Subtensor constant names a fixed runtime value; a transaction fee is the computed
cost that depends on those values and the specific action being submitted. Constants provide the
parameters; fees are the amounts those parameters produce.

References: [Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)

## Relationship to Subtensor

Subtensor constants and Subtensor are related but different scopes in Bittensor vocabulary.
Subtensor names the layer 1 blockchain and runtime environment, while Subtensor constants name fixed
runtime values exposed by that environment. The
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor) describes the
blockchain layer, and the
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants) reference describes
compile-time constants for the Subtensor runtime.

For readers, Subtensor names the chain layer; Subtensor constants name one reference surface for
fixed values used by that layer.

References: [Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)

## Relationship to Subtensor Storage

Subtensor constants and Subtensor storage are related but different runtime reference surfaces.
Constants expose fixed values grouped by runtime area, while storage entries expose chain state that
can change as the network processes activity. The
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants) reference describes
compile-time runtime constants, and the
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) reference lists runtime
storage entries and query paths.

For readers, constants answer "what fixed value does this runtime area expose?", while storage
answers "what state value does this runtime area currently hold?"

References: [Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Relationship to Rate Limits

Subtensor constants and rate limits are related because some runtime constants help define timing or
limit behavior, while rate limits describe the cooldown rules users encounter at the chain level.
The constants reference groups Bittensor-specific entries such as rate-limit defaults under the
Subtensor module, and the [Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)
documentation explains block-based cooldown behavior.

For readers, a Subtensor constant can be one parameter exposed by the runtime, while a rate limit is
the user-visible rule that determines when repeated actions can succeed again.

References: [Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)

## Relationship to Existential Deposit

Subtensor constants and existential deposit are related because existential deposit is exposed as a
named balance constant. The constants reference lists `balances.existentialDeposit` as the minimum
amount required to keep an account open, while the
[Glossary: Existential Deposit](https://docs.learnbittensor.org/resources/glossary#existential-deposit)
defines the term as the minimum TAO amount required for an account to exist on the Bittensor
blockchain.

For readers, existential deposit names the account-existence threshold; Subtensor constants name the
runtime-reference surface where that threshold is exposed.

References: [Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Glossary: Existential Deposit](https://docs.learnbittensor.org/resources/glossary#existential-deposit)

## Careful Interpretation

A constant entry shows the name and value exposed by the runtime reference. It should be read with
its runtime area and entry summary. Current network behavior can also depend on mutable storage,
events, and feature-specific documentation, so a constant alone should not be treated as a complete
description of the underlying mechanism.

Reference: [Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)
