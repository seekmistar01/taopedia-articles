# Subtensor Storage

Subtensor storage entries are named places where Bittensor's Subtensor runtime exposes chain state
for inspection ([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

They help readers find which state surfaces exist, how those surfaces are named, and how an
inspected value should stay tied to its chain context.

## What Storage Shows

Storage entries name places where runtime state is exposed for inspection. A storage entry can
identify the runtime area, entry name, type, lookup shape, and summary for a state surface
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

This makes storage a map of readable runtime state rather than a full explanation of every mechanism
that changes that state. A value read from a storage surface still belongs to the network and block
context where the read happened.

## Generated Reference Shape

The generated storage reference identifies entries through the `api.query.<Pallet>.<storage_item>`
shape ([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)). That path keeps
the runtime area and storage item name attached in one lookup label.

The listing records generated-reference metadata for the connected Subtensor runtime, including the
endpoint used to build the page
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)). That gives readers a
source for the storage surface rather than leaving the entry name as an isolated label.

Each entry can include a type, modifier, interface path, and summary. The modifier helps show
whether the storage item is required or optional in the generated reference, while the summary gives
the short description attached to that state surface
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

## Storage Entry Examples

The storage listing includes general runtime areas and Bittensor-specific areas. Examples include
`balances`, `safeMode`, `subtensorModule`, `system`, `timestamp`, and `transactionPayment`
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Some entries expose general chain state. The generated reference lists `adminUtils.palletVersion` as
a required `u16` storage item summarized as returning the current pallet version from storage
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Other entries expose account or balance-related state. The same reference includes
`balances.account` as a storage surface for account-balance data
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Subtensor-module storage entries expose Bittensor-specific state surfaces. The generated listing
includes entries for subnet activity, emission accounting, stake relationships, and feature
settings, with each entry keeping its storage item name, type, and summary together
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

## Where Entries Appear

Storage entries are grouped by runtime area. The owning area is part of the entry's meaning because
similarly named state surfaces can carry different context in different runtime areas
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

For readers, the runtime area is the first layer of interpretation. The area says which part of the
runtime owns the entry before the entry name and type narrow the meaning.

## Storage Maps Need Keys Before a Read

The
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
guide draws a practical line between two storage shapes in the chain-state view. Some storage items
are single values that can be queried without extra input, while others are maps keyed by an
account, a [netuid](https://docs.learnbittensor.org/resources/glossary#netuid), or another
identifier.

Map-shaped entries need that key filled in before a read can return a value. A subnet-scoped surface
such as immunity period therefore belongs to one netuid at a time, and an account-balance surface
belongs to one account at a time. A global spacing value for subnet registration can be queried
without that extra key field.

That key step sits between the storage listing and the returned value. The catalog shows the entry
name and type shape, while the supplied key narrows which row of mutable chain state is being read
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

## Network-Wide Accounting Items Need No Key

The Subtensor Storage reference lists `totalIssuance` and `totalStake` as single network-wide items,
and lists `totalNetworks` as a count of how many subnets currently exist in the catalog
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

These three entries are the no-key-required shape described above: each is one global accounting
value rather than a per-subnet lookup, so reading them does not require supplying a netuid.

## Several Catalog Entries Are Keyed by NetUID

The same reference lists `lastUpdate`, `validatorPermit`, `emission`, and `subnetworkN` as
map-shaped entries keyed by netuid rather than single global values
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

These are the keyed shape described above: reading any of them requires supplying the netuid for the
subnet in question, the same key requirement that applies to any other map-shaped storage entry.

## The Storage Catalog Follows Runtime Spec

The [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) reference is
generated from the connected Subtensor runtime spec rather than maintained as a standalone prose
list. Its grouped entries describe the storage surfaces the runtime exposes at generation time,
including metadata about the endpoint used when the page was built.

Runtime upgrades can change that exposed surface by adding, renaming, or retiring storage items.
Inspection guidance notes that runtime metadata must be refreshed after a Bittensor runtime upgrade
so connected tools still describe the live chain accurately
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

That generation link separates catalog vocabulary from observed chain history. A listed storage
entry names a surface the runtime can expose, while a value returned from a particular network and
block belongs to that moment in chain history on the connected
[Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor) network.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Subtensor
storage on a subnet such as netuid 1, that sequence changes which network and block context a reader
should assume when interpreting a storage read.

In localnet, storage surfaces on netuid 1 can be inspected against an isolated local blockchain.
Localnet storage values do not represent production chain state.

On testnet, storage reads on netuid 1 can be exercised in a shared non-production network. Testnet
entries are separate from mainnet state
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

On mainnet, Subtensor storage on netuid 1 concerns live production chain state at the selected block
height on the connected network
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A storage example from one environment should not be read
as representing production chain state in another environment.

## Practical Limits

Subtensor storage can show what state was exposed at an inspected chain context, but it does not
prove off-chain intent or fully explain why that state changed
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

When a concrete question depends on cause or result, the storage value should be read alongside the
submitted action, emitted events, and the relevant concept page. That gives readers more context
than the storage entry name alone.

## Distinction from Metagraph

Subtensor storage and the metagraph expose chain state at different levels. Subtensor storage is the
set of named runtime entries that hold raw chain state, while the metagraph is a derived snapshot
that assembles per-subnet neuron information from that state
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

- **Subtensor storage** — the named runtime entries holding raw chain state.
- **Metagraph** — a derived per-subnet snapshot assembled from that state.

Among the netuid-keyed entries, the catalog ties `subnetworkN` to the number of registered UIDs on a
subnet, which is the storage-layer counterpart to the metagraph's `n` field: both name the same
registered-neuron count exposed through different reference surfaces
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

## Distinction from Chain Reads

Subtensor storage is one chain-read path that returns values at keys. Chain reads more broadly cover
storage, events, constants, and other inspection without submitting a new action
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Inspecting the Chain](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)). Storage answers
what value lives at a key; chain reads name the broader non-submission inspection family.

- **Subtensor storage** — key-value reads of current chain state.
- **Chain reads** — broader inspection family including storage, events, and constants.

Direct storage queries read named state surfaces from chain storage. The same inspection guide
places [runtime calls](https://docs.learnbittensor.org/subtensor-api/runtime) on a neighboring
read-only path that executes functions which may involve computation rather than returning one
stored value alone
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

Examples include bundled subnet information that covers price, emission, and reserve-related fields,
along with fee estimation for a proposed extrinsic length. Those answers can draw on storage
internally, but they are returned as computed read-only results rather than as one labeled storage
item from the catalog.

When a question needs a catalogued state surface, storage vocabulary is the natural starting point.
When the question needs a derived or bundled answer, runtime-call vocabulary may fit better even
though both paths stay on the inspection side of the chain.

On a subnet such as netuid 1, map-shaped Subtensor-module storage entries expose stake, weight, and
subnet activity surfaces keyed by
[netuid](https://docs.learnbittensor.org/resources/glossary#netuid)
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Those reads show chain state at a chosen block, while
[validator weights](https://docs.learnbittensor.org/resources/glossary#weight-vector) submission and
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) processing explain how subnet
signals become [emission](https://docs.learnbittensor.org/learn/emissions) outcomes
([Emission](https://docs.learnbittensor.org/learn/emissions)).

Storage therefore names inspectable state surfaces; it does not replace the incentive mechanism that
updates many of those surfaces over time.

## Distinction from Subtensor Constants

Subtensor storage and Subtensor constants expose different kinds of values. A storage entry names a
state surface that can change as the chain advances, while a constant names a fixed runtime-defined
value built into the runtime
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

- **Subtensor storage** — a readable state surface that can change.
- **Subtensor constant** — a fixed runtime-defined value.

## Distinction from Extrinsics

Subtensor storage and an extrinsic sit on the read side and the write side. A storage entry exposes
state for inspection, while an extrinsic is a submitted runtime action that can change that state
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

- **Subtensor storage** — readable chain state.
- **Extrinsic** — a submitted action that can change state.

Storage, events, and extrinsics describe different Subtensor surfaces. Storage exposes state, events
name emitted runtime records, and extrinsics name submitted runtime actions
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

A feature can involve all three surfaces without making them interchangeable. Storage answers what
state surface can be read; events answer what runtime record was emitted; extrinsics answer what
action was submitted.

For example, after a submitted action is processed, an event can show what the runtime recorded and
a storage read can show a relevant state surface afterward. Those are related observations, but they
are not the same artifact ([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

## Distinction from Subtensor Events

A Subtensor storage entry and a Subtensor event answer different questions. A storage entry names a
state surface that can be read after processing, while an event names a record emitted while the
runtime processes activity
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

- **Subtensor storage** — a readable state surface after processing.
- **Subtensor event** — a record emitted during runtime activity.

## Distinction from Yuma Consensus

Subtensor storage names the on-chain state items the Subtensor runtime persists between blocks. Yuma
Consensus is the separate subnet process that reads and updates relevant storage when converting
validator weight submissions into emission shares each tempo
([Navigating Subtensor](https://docs.learnbittensor.org/navigating-subtensor),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Subtensor storage** — on-chain state persisted by the runtime.
- **Yuma Consensus** — on-chain settlement that turns validator weights into emission shares.
  Subtensor storage names persisted chain state; Yuma processes validator weights into rewards
  ([Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions)).

## Distinction from Network

Storage values are scoped by chain history. The same storage entry name can be meaningful across
environments, while returned values belong to separate network and block contexts
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

The same reference lists `blockEmission` as the storage item that exposes the network's current
per-block TAO mint amount
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

That entry is the storage-layer counterpart to block reward vocabulary: block reward names the
concept, while `blockEmission` names the specific catalog entry that surfaces the current figure
without a netuid key.

## Reader Boundary

The storage listing tells readers where state is exposed. A chain read or inspection view can then
show a value from a selected network and block context
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

The entry name tells readers which surface was inspected; the returned value tells readers what that
surface showed in the chain context where the read happened.
