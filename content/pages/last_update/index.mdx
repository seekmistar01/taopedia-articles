# Last Update

Last update is a per-neuron value in a subnet's metagraph that records the block at which a neuron's
data was most recently updated. Documentation presents it as the marker of a neuron's latest update
and ties it directly to whether that neuron counts as currently active.

## What It Records

The value is a block height, not a wall-clock time. It captures the most recent point at which a
neuron's state changed on the subnet, such as a validator setting weights. Because it is measured in
blocks, its meaning is read against the current block rather than against a calendar.

## Determining Activity

Last update is the input to a subnet's activity test. The documentation describes a neuron as
inactive once its last update is far enough in the past: when the last update plus the activity
cutoff is below the current block, the neuron is treated as inactive. Until that point it is still
within the active window ([The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph),
[Subnet Hyperparameters: Activity Cutoff](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#activitycutoff)).

## Why It Matters

The point of tracking last update is to keep consensus based on participants that are currently
contributing. A neuron whose last update has fallen outside the activity window is treated as
inactive, so a stale or absent participant does not keep influencing an epoch's outcome. This is how
the network distinguishes recently active neurons from ones that have gone quiet.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For last update,
that sequence changes how readers should interpret neuron activity timestamps and metagraph snapshot
examples.

In localnet, last-update fields can be tested in an isolated environment. Localnet block heights do
not represent production neuron activity.

On testnet, metagraph activity fields can be exercised in a shared non-production network. Testnet
last-update values are separate from mainnet subnet state.

On mainnet, last update describes live production neuron activity on subnets such as netuid 1.
Observed timestamps depend on the selected subnet, block height, and activity-cutoff settings
([The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A last-update example from one environment should not be
read as representing production neuron activity in another environment.

## Metagraph Views Expose Last Update Per Neuron

[The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph) presents subnet neurons as
structured rows within a metagraph snapshot. Last update is one of the per-neuron fields readers
encounter when inspecting that row data alongside stake, rank, and incentive-related values.

That placement keeps last update in observation vocabulary. It is a field returned with other neuron
state in a subnet snapshot, not a separate wallet label or account setting.

## Snapshots Are Tied to a Block Height

[The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph) describes the metagraph as
a snapshot of subnet state at a particular block. Last update uses the same block-height vocabulary:
it records the block of a neuron's most recent change, which readers then compare against the
snapshot's current block context.

Because both values are block-based, the field is meant to be read inside chain timing rather than
as a calendar timestamp.

## Activity Fields Sit Beside Incentive Metrics

Metagraph material groups last update with other neuron activity and incentive-related values such
as ranks, trust, and consensus scores. The activity test that combines last update with activity
cutoff therefore sits among the same row of chain-derived performance fields rather than in a
separate identity record.

Readers comparing whether a neuron is active should keep that row context in view. Last update is
one activity input within a broader metagraph snapshot, not a standalone status badge.

## Distinction from Registration

Last update and the registration block are two different per-neuron block markers in a subnet's
metagraph on a subnet such as netuid 1. Last update (`last_update`) records the block at which a
neuron's data was most recently updated — its latest-activity marker — while the registration block
(exposed as `block_at_registration`) records the block at which the neuron first registered to the
subnet. One tracks the most recent update; the other tracks entry, so the gap between them measures
how long the neuron has been on netuid 1
([The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

The two answer different questions. The registration block anchors lifecycle calculations such as
the immunity period that shields a newly registered UID on netuid 1, while last update reflects
whether a neuron has acted recently. A long registration age does not imply a recent update, and a
recent update does not change when the neuron registered
([Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period),
[The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

- **Last update** — the block of a neuron's most recent data update on netuid 1.
- **Registration block** — the block at which the neuron first registered to the subnet.

## Distinction from Activity Cutoff

Last update and activity cutoff are always read as a pair, but they record different kinds of subnet
state. Last update is the per-neuron block height of the neuron's most recent on-chain change, while
activity cutoff is the owner-settable hyperparameter that sets how many blocks of inactivity are
allowed before stake is treated as inactive for epoch participation in Yuma Consensus
([The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph),
[Subnet Hyperparameters: Activity Cutoff](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#activitycutoff)).

For readers, last update answers when the neuron last acted; activity cutoff answers how long
silence is tolerated before that neuron stops counting as active. Neither field alone decides status
— the comparison is whether the gap from `last_update` to the current block exceeds the subnet's
cutoff window
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

- **Last update** — the block of a neuron's most recent data update.
- **Activity cutoff** — the hyperparameter span that defines allowed inactivity before epoch
  exclusion.

## Distinction from Active Status

Last update and the metagraph `active` field sit on the same neuron row but name different layers of
the activity test. `last_update` records the block timestamp of the neuron's latest change, while
`active` is the derived boolean that reports whether that neuron still falls inside the subnet's
activity window at the snapshot block
([The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

For readers inspecting a metagraph export, `last_update` is the raw staleness input and `active` is
the computed outcome after the cutoff rule is applied. A neuron can have a recent `last_update` yet
still read inactive if the subnet's cutoff is short, and a stale `last_update` should not be read
without the hyperparameter that defines the allowed gap
([The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

- **Last update** — the block height of the neuron's most recent on-chain update.
- **Active** — the metagraph flag showing whether the neuron is inside the activity window.

## Distinction from Immunity Period

Last update and the immunity period both use block heights, but they gate different lifecycle
questions. Last update tracks whether a neuron has contributed recently enough to count as active,
while the immunity period is the post-registration grace window that shields a newly assigned UID
from ordinary pruning or trimming for a documented block span
([Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period),
[The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

For readers, immunity protects entry timing — a young registration may be safe from displacement
even before it has built performance history — whereas last update measures ongoing participation
after the neuron exists. A neuron inside immunity can still have a stale last update, and a recent
last update does not extend or replace the immunity window counted from registration
([UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming),
[Understanding Neurons](https://docs.learnbittensor.org/learn/neurons)).

- **Last update** — whether the neuron has updated recently enough to count as active.
- **Immunity period** — the post-registration block window that protects a new UID from pruning or
  trimming.

## Distinction from Yuma Consensus

Last update names the metagraph field that records the block of a neuron's most recent update. Yuma
Consensus is the separate tempo process that converts validator weight submissions into emission
shares and does not by itself store that activity marker
([The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Last update** — block height of a neuron's most recent update.
- **Yuma Consensus** — on-chain settlement that turns validator weights into emission shares. Last
  update names neuron activity timing; Yuma processes validator weights into rewards
  ([Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions)).

## Reader Boundary

This page defines the concept at a high level. It does not report the last update of any particular
neuron, whether a given neuron is active right now, or the activity-cutoff value on a subnet. Those
are live chain state and should be checked for the relevant netuid, for example netuid 1. The
activity rule described here is the documented relationship, not a live reading
([The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).
