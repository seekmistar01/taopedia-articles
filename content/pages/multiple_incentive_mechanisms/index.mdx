# Multiple Incentive Mechanisms

Multiple incentive mechanisms are a subnet feature that lets one Bittensor subnet host more than one
independent evaluation system. The official
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)
describes each mechanism as operating with its own bond pool for Yuma Consensus calculations, with
validators evaluating miners separately for each mechanism.

## Contrast with a Single Mechanism

A single
[incentive mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism) names
the rule context that connects subnet work, validator evaluation, and incentive outcomes inside one
subnet. Multiple incentive mechanisms extend that idea: one subnet can run parallel rule contexts
instead of only one.

For readers, the distinction is count and separation. One mechanism means one evaluation context;
multiple mechanisms mean several evaluation contexts can coexist inside the same subnet market.

References: [Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Incentive Mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism)

## Validator and Miner Evaluation Context

When a subnet uses multiple incentive mechanisms, validators must evaluate miners separately for
each mechanism. The glossary also notes that miner performance in one mechanism does not affect a
miner's rating in another. That separation keeps each mechanism's evaluation picture distinct.

This article does not describe how to configure mechanisms, rank miners, or read live subnet
parameters. Those details belong in official subnet documentation and live chain data.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)

## Bond Pool and Yuma Consensus Context

Each mechanism in a multi-mechanism subnet uses its own bond pool for Yuma Consensus calculations.
The [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) documentation describes
how validator-miner bonds feed consensus aggregation, and multiple-mechanism vocabulary adds that
those bond contexts can be separated by mechanism inside one subnet.

For readers, bond pool separation is part of what makes multiple mechanisms distinct from simply
adding more miners or validators to an ordinary single-mechanism subnet.

## Subnet Creator and Emission Context

Subnet creators can use multiple incentive mechanisms to distribute emissions across different kinds
of work or evaluation criteria within one subnet. The official
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)
documentation presents this as a subnet-design choice rather than as a network-wide default.

Multiple incentive mechanisms therefore belong to subnet policy vocabulary. They do not change the
meaning of network-wide TAO emission labels or wallet ownership terms by themselves.

References:
[Glossary: Subnet Creator](https://docs.learnbittensor.org/resources/glossary#subnet-creator),
[Emission](https://docs.learnbittensor.org/learn/emissions)

## Relationship to Yuma Consensus

Multiple incentive mechanisms and Yuma Consensus address related but different parts of subnet
consensus vocabulary. Multiple incentive mechanisms describe parallel evaluation systems inside one
subnet, each with its own bond pool for Yuma Consensus calculations, while Yuma Consensus aggregates
validator weight signals into miner incentives and validator dividends inside each mechanism
context. The
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)
defines separate bond pools per mechanism, and
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) describes clipping, bonding,
and emission calculation.

For readers, multiple incentive mechanisms name parallel evaluation paths inside one subnet, while
Yuma Consensus names the aggregation process that runs separately within each mechanism's bond pool.

References:
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)

## Relationship to Validator Weights

Multiple incentive mechanisms and validator weights address related but different parts of subnet
evaluation structure. Validator weights are the evaluation signals validators submit for miners,
while multiple incentive mechanisms require those submissions to be handled separately for each
parallel evaluation system inside one subnet. The
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)
describes the per-validator evaluation inputs, and the
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)
notes that miner performance in one mechanism does not affect rating in another.

For readers, validator weights name the evaluation submission concept, while multiple incentive
mechanisms name the parallel evaluation contexts those submissions must be read within.

References:
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)

## Relationship to Weight Matrix

Multiple incentive mechanisms and the weight matrix address related but different parts of subnet
weight vocabulary. The weight matrix collects validator weight vectors as structured input to Yuma
Consensus, while multiple incentive mechanisms mean a subnet can maintain separate weight-matrix and
bond-pool contexts for each parallel evaluation system. The
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)
describes the matrix built from validator submissions, and the
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)
defines independent bond pools per mechanism.

For readers, the weight matrix names organized validator input for one mechanism context, while
multiple incentive mechanisms name when a subnet runs more than one such context in parallel.

References:
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)

## Relationship to Weight Copying

Multiple incentive mechanisms and weight copying address related but different parts of subnet
evaluation quality. Weight copying describes when a validator relies on public weight information
instead of producing an independent evaluation, while multiple incentive mechanisms separate
evaluation and bond-pool context so copied behavior in one mechanism does not automatically define
performance in another. The
[The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)
documentation describes copied or derived weights as the problem behavior, and the
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)
notes separate miner ratings per mechanism.

For readers, weight copying names an evaluation-quality problem, while multiple incentive mechanisms
name parallel evaluation contexts that keep mechanism-specific ratings separate.

References:
[The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)

## Reader Boundary

Multiple incentive mechanisms should not be read as a recommendation to deploy a specific subnet
design, as a live mechanism count for any netuid, or as a substitute for the single-mechanism
incentive vocabulary used in ordinary subnet reading. It is a concept term for parallel evaluation
systems inside one subnet.
