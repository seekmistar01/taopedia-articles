# Drand Time-Lock Encryption

Drand time-lock encryption helps Bittensor Commit Reveal keep newly submitted validator weights
concealed until a later reveal round. In
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal), validators commit weight
information before it becomes readable, reducing the usefulness of copied public weights.

## Delayed Visibility

Delayed visibility matters because validator weights are valuable consensus inputs. A validator that
can simply reuse public weight signals adds less independent evaluation to a subnet. The
[Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)
centers on that free-riding risk, and Commit Reveal reduces the window in which fresh weight signals
can be copied.

## Drand Round Boundary

Time-lock encryption binds access to a later round instead of immediate decryption. In the
[Drand/time-lock encryption glossary entry](https://docs.learnbittensor.org/resources/glossary#drandtime-lock-encryption),
encrypted data becomes decryptable only after a future Drand round, which makes the round boundary
part of the visibility rule.

## Commit Reveal Context

Commit Reveal uses concealed weight submission before later reveal. Drand time-lock encryption fits
that process as the timing layer: the committed weight data remains unreadable until the target
round, then the revealed information can enter the consensus flow. The
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal) reference links this delay
to copied-weight protection.

## Consensus Boundary

Drand time-lock encryption controls when concealed weight information becomes readable; it does not
score miners or allocate emissions. After reveal,
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) is the aggregation context
for validator evaluations, miner incentives, and validator dividends. The delayed-visibility layer
protects inputs before that aggregation step.

## Multiple-Mechanism Context

When a subnet uses more than one incentive mechanism, each mechanism has its own weight context. The
[multiple incentive mechanisms](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)
material keeps those evaluation contexts separate, so delayed access to concealed weights should be
read inside the mechanism whose weights are being revealed.

## Reading the Term

The term names a timing and visibility tool, not a separate consensus algorithm. It is closest to
Commit Reveal, weight-copying protection, and validator-weight handling. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) distinction also
keeps examples in production, shared testing, and local environments from being treated as the same
evidence.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Drand
Time-Lock Encryption, this sequence gives readers a boundary for interpreting encryption examples
and timing-assumption notes.

Localnet examples are isolated and reflect local chain state, so they are useful for controlled
experiments rather than evidence of live Bittensor behavior. Testnet examples add shared
non-production conditions, which can reveal integration behavior without touching mainnet state.

On mainnet, Drand Time-Lock Encryption examples should be read as live production encryption
behavior on the production Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, so outcomes from one environment should not be treated as
proof of behavior in another.
