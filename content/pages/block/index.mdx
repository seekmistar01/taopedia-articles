# Block

A block is a unit of data in the Bittensor blockchain. It contains transactions and has a unique
identifier called a block hash
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor)).

The term matters because it separates accepted chain history from pending transaction context. A
transaction waiting in the mempool is not yet the same as a transaction included in a block.

After inclusion, the block gives readers the chain-history location for the transaction. Events and
storage can then supply runtime-record and state context around that included activity
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

## Chain Unit Context

Blocks give chain activity an ordered record. In Bittensor, the block concept belongs to Subtensor,
the layer 1 blockchain and system of record for transactions and rankings
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor)).

A block is not a role, reward rule, or subnet mechanism. It is a blockchain data unit that records
included transactions.

The block hash is part of that reference value. It gives the block a specific identifier, while the
transactions inside the block remain separate runtime actions with their own meanings
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

## Block Hash Boundary

A block hash identifies the block as an ordered chain-history unit; it is not the same thing as the
runtime action inside that block. The block glossary places the hash on the block itself, while
extrinsic documentation describes the submitted calls that can be included within block history
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

For readers, that means a block hash is useful for locating the chain context around activity. The
meaning of the included transaction or call still comes from the relevant extrinsic, event, storage,
or mechanism reference.

## Timing Context

Bittensor documentation presents ordinary public-network block processing around a twelve-second
cadence, while local development context can use different local timing modes
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)).

That timing context should be read with the network environment attached. Mainnet, testnet, and
localnet examples are not interchangeable just because they all use block vocabulary.

## Mempool Boundary

The mempool is the pending-transaction holding area before block inclusion. A block is the ordered
unit that includes accepted transactions
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

This distinction is the most useful way to read the pair: mempool means waiting before inclusion;
block means recorded after inclusion.

That is why mempool visibility and block evidence support different safety questions. Mempool
visibility concerns exposed pending details; a block confirms that accepted transactions were
included in chain history ([MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield/),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

## Extrinsic Boundary

Extrinsics are runtime actions. Blocks are the chain data units that can include and order accepted
extrinsics ([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

Extrinsic names the action surface, while block names the ordered record that can contain accepted
actions.

## Fee Boundary

Transaction fees belong to state-changing actions that the chain processes. Blocks are where
accepted transaction records are ordered, so fee-bearing actions and block inclusion are adjacent
concepts rather than synonyms
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

The block itself names the chain unit. Fee rules describe cost accounting for state-changing
actions.

## Storage and Event Boundary

Storage entries and events are related chain-observation surfaces, but they do not mean the same
thing as a block. Storage describes recorded state, while events describe emitted runtime records
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

A block can provide the ordered context in which chain activity is recorded. Storage and events help
interpret what the chain contains or emitted around that context.

For a concrete included transaction, the surrounding references each add a different kind of detail.
The extrinsic reference names the runtime action, fee documentation explains cost components for
state-changing actions, events name runtime records, and storage exposes readable state surfaces
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

## Metagraph Snapshots Anchor to One Block Height

The [Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph) documentation describes
the metagraph as subnet state at a particular block height and lists a `block` field for the height
when the snapshot was retrieved. On a subnet such as netuid 1, that number ties incentive fields to
one point in chain history rather than to a timeless summary.

Block vocabulary therefore locates subnet snapshots in time. A metagraph reading should keep both
the selected netuid and the block height attached when neuron fields are interpreted.

## Registration Records Include Block Heights

The metagraph reference includes `block_at_registration` as registration blocks for each neuron,
used for [immunity period](https://docs.learnbittensor.org/resources/glossary#immunity-period)
calculations. A [block](https://docs.learnbittensor.org/resources/glossary#block) number therefore
marks when a participant entered a subnet, not only when a snapshot was taken.

That timing field connects block units to participant lifecycle vocabulary. Registration block
heights belong to chain history, while pending mempool activity still waits before inclusion.

## Block Time, Tempo, Epoch, and Yuma Pipeline

Bittensor block time names how often Subtensor produces a new ordered chain record. Public-network
documentation centers on a twelve-second cadence, while local development can use faster local modes
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)).

[Tempo](https://docs.learnbittensor.org/resources/glossary#tempo) counts blocks between subnet
emission steps on a selected netuid. Metagraph metadata exposes tempo and blocks-since-last-step
fields tied to that subnet update rhythm, so block numbers mark both inclusion history and subnet
timing boundaries ([The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

[Epoch](https://docs.learnbittensor.org/resources/glossary#epoch) vocabulary names longer subnet
timing windows that group many blocks together. Tempo is the subnet-specific block interval over
which [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) calculates emissions
from the latest available ranking weight matrix; epoch language helps readers place those tempo
boundaries inside broader subnet cadence without treating every block as a reward event.

The useful pipeline order on a subnet such as netuid 1 is therefore block production at chain
cadence, included weight-setting extrinsics in chain history, tempo reaching its block interval,
Yuma Consensus aggregating the latest validator weight matrix, and emission distribution crediting
miner and validator shares at the tempo boundary
([Emission](https://docs.learnbittensor.org/learn/emissions),
[Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

Block vocabulary locates each included extrinsic in ordered chain history. Tempo counts how many
blocks elapse before the next Yuma emission step; Yuma Consensus names the aggregation that turns
validator weights into incentives and dividends once that interval closes.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. Block examples on
a subnet such as netuid 1 need that context because chain history, inclusion records, and metagraph
snapshots belong to the network where they were observed
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

In localnet, block examples are isolated development records. Local block cadence and metagraph
snapshots for netuid 1 reflect local chain configuration rather than production Subtensor history.

On testnet, block inclusion and metagraph snapshots can be observed in a shared non-production
network. Testnet block heights and tempo readings for netuid 1 are separate from mainnet chain state
([The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

On mainnet, block interpretation concerns production Subtensor records on the active network,
including subnet timing and weight-setting history on live subnets such as netuid 1.

A block example from one environment should not be treated as proof about another environment. They
use the same block concept, but not the same chain history.

## Distinction from Tempo

A block is one chain record. Tempo counts how many blocks pass before the next subnet emission step
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo),
[The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

- **Block** — one ordered unit of chain history.
- **Tempo** — the block interval between subnet emission steps.

## Distinction from Epoch

Epoch names longer subnet timing windows that group many blocks. A block height still marks where an
included extrinsic sits inside that wider cadence
([Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

Use block when locating one inclusion point; use epoch when naming the wider consensus window.

## Distinction from Extrinsics

An extrinsic is a submitted runtime action. A block is the container that records which extrinsics
were included at one height
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

Extrinsic vocabulary names the action; block vocabulary names the inclusion record.

Readers use block height when they need to locate when an extrinsic became chain history.

## Distinction from Mempool

The mempool holds pending transactions before inclusion. A block names the point where accepted
transactions become chain history
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

Pending and included are different stages: mempool first, block after inclusion.

A pending extrinsic is not yet chain history; a block record is.

## Distinction from Coinbase

Coinbase is the per-block emission process that runs inside block production. Block vocabulary names
the chain record itself, not the emission engine inside it
([Coinbase documentation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

A block can contain many kinds of activity; coinbase names one recurring emission step within that
cadence.

Block height also helps readers line up metagraph snapshots with the chain moment they describe
([The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

## Reader Boundary

Block should be read as chain-history vocabulary. It names a Bittensor blockchain data unit that
contains transactions and has a block hash
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor)).

Mempool, extrinsic, fee, storage, event, and network references remain useful because they explain
nearby questions that the block term alone does not answer.
