# Chain Reads

Chain reads describe looking at public Bittensor chain state without asking Subtensor to record a
new action. They are observation, not authorization.

The official [Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) page says
that reading chain state is free. That statement sets the main boundary: reading public chain
information is different from submitting a state-changing action for the runtime to process.

## Why the Boundary Matters

The read-versus-submit boundary separates public inspection from actions that need signing
authority. A chain read can observe public state, but it does not move funds, change stake, register
anything, or alter an on-chain role.

This does not make chain data private or risk-free to interpret. It only means the read itself is
not the same as giving permission for Subtensor to do something new. The
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) documentation supports
that distinction by separating free reads from fee-bearing state changes.

## Chain Inspection Context

The official
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
page describes chain inspection as a way to look at Bittensor chain state, metadata, blocks, and
other public records. Chain reads fit inside that inspection side of the workflow.

This makes chain reads useful evidence for claims about what a selected network exposes through
public state. They are weaker evidence for claims about intent, cause, future outcomes, or private
operational decisions.

## Fee and Action Context

Because the [official fee page](https://docs.learnbittensor.org/learn/fees) treats reading chain
state as free, chain reads should not be described as fee-bearing transactions. Fees belong to the
action side of the network when a submitted operation asks Subtensor to process a change.

The boundary is useful in articles about wallets, staking, transfers, and validator activity because
those topics can involve both observation and action. A chain read can inform a later decision, but
the later action remains separate and may require signing authority, fees, role eligibility, or
other conditions.

## Network and Block Scope

A chain read belongs to the Bittensor network being inspected. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) page separates
mainnet, testnet, and localnet contexts, so a fact observed on one network should not automatically
be treated as the same fact on another network.

Chain reads are also scoped by chain history. Public state can change as blocks are produced and as
submitted actions are included. When precision matters, a claim should identify the network and
inspection context instead of speaking as if every Bittensor environment has the same state.

## Reference Definitions and Queried Values

Official Subtensor references describe what can be inspected; a chain read returns or displays a
particular value from the selected chain context. The
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) reference can define a
storage entry, while a query result shows a value for a chosen context.

The same distinction applies across other inspection surfaces. The
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events) reference describes emitted
runtime records, while the
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants) reference describes
runtime constants. A chain read may use these sources, but definitions and observed values should
not be collapsed into one claim.

## Evidence Boundary

The
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
guide frames chain inspection as a way to look at public chain state. Chain reads can therefore
support claims about observed storage values, block records, metadata, or emitted records when the
claim matches the inspected context.

They do not prove why a value changed, whether a later submission will succeed, or whether a wallet
should take a specific action. Those claims require separate source support. A chain read answers
what is visible in a selected context; it does not replace mechanism documentation or operating
guidance.
