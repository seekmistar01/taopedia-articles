# Rank

Rank is a Bittensor consensus metric for summarizing miner evaluation inside a subnet. It is the
aggregate miner-side result produced by Yuma Consensus from validator weight signals
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The term belongs to the miner-evaluation side of consensus. It is not one validator's raw view and
not a general reputation label for every Bittensor activity.

In plain language, rank answers: after all validators have submitted weights and consensus has run,
where does each miner stand in the subnet's evaluation result? A higher rank means the miner
received stronger consensus-adjusted support from validators
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

For a reader, rank differs from a single validator's raw score. Raw validator weights are inputs;
rank is the aggregated output after Yuma Consensus has compared, filtered, and normalized those
inputs across the validator set
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Consensus Result

Rank appears after validator weight signals have been compared, filtered, and aggregated. It answers
where a miner lands in the subnet's consensus-adjusted evaluation result
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

This makes rank a result term. Validator weights are inputs; rank is the miner-side result built
after those inputs pass through consensus.

The result is still subnet-scoped. A rank value belongs to the subnet task, validators, and miner
set that produced it rather than to a universal miner leaderboard across Bittensor
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

That scope keeps rank tied to the work being evaluated. The same consensus term can appear across
subnets, but the underlying miner outputs and validator standards come from the selected subnet
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## From Weights to Rank

Validator weights express evaluations of miner work, while rank summarizes the consensus-adjusted
result of those evaluations. A single validator weight signal can influence the process, but rank is
not a single validator's opinion
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

That distinction keeps input terminology separate from result terminology. Weights describe
evaluation signals; rank describes the aggregate miner-side result.

Weight matrix vocabulary sits one layer earlier. The matrix groups validator weight vectors as
structured input, while rank is interpreted after the Yuma process filters and aggregates those
signals
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The sequence is therefore input, filtering, and miner-side result. Validator weights and matrices
sit before rank, while rank appears after consensus has processed those inputs
([Consensus-Based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

## Consensus Score

Consensus score and rank describe different parts of Yuma Consensus. Consensus score is the
stake-weighted benchmark used for clipping outlier validator weights, while rank is the miner-side
result after filtering and aggregation
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

Consensus score helps shape rank, but rank is the processed miner-side result rather than the
benchmark itself.

The distinction also keeps input, filtering, and result vocabulary separate: validator weights enter
the process, consensus score helps filter them, and rank names the adjusted miner-side outcome.

## Rank and Trust Terms

Rank, trust, and validator trust are related consensus terms. Rank names the miner-side aggregate
result. Trust describes how much miner-side support remains after clipping, while validator trust
describes validator-side influence after filtering
([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

They are complementary views of one consensus process. Rank explains the miner-side result, while
trust and validator trust describe how much signal remains after clipping or filtering
([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)).

## Emission Role

Rank matters because Yuma Consensus uses miner evaluation when distributing subnet incentives. It
belongs near emissions terminology, but it is not a reward promise by itself
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

The useful reading is that rank contributes to how miner-side outcomes are shaped in consensus.
Emissions provide the broader reward-flow setting.

Rank helps explain miner-side ordering inside the consensus process; emissions explain how reward
flow is handled at the tokenomics layer
([Emission](https://docs.learnbittensor.org/learn/emissions)).

## Subnet Scope

Rank is meaningful inside the subnet that produced it. Subnets define work markets where miners
produce outputs and validators evaluate them according to subnet-specific standards
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

Rank belongs with the relevant subnet task and evaluation setting. It is a consensus result for that
setting, not a universal quality label across unrelated work.

Subnet scope also guards against comparing unlike work. A rank result from one subnet reflects that
subnet's task and incentive mechanism, not an all-purpose score for miners in unrelated subnet
settings
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For rank on a
subnet such as netuid 1, that sequence changes how readers should interpret miner-side consensus
outcomes, the validator set that produced them, and the tempo boundary where those results are
credited.

In localnet, rank examples can be exercised in an isolated environment. Local validator weights and
aggregated miner results reflect local chain configuration rather than production subnet evaluation
history.

On testnet, rank can be observed in a shared, non-production network. Testnet rank readings on a
selected netuid are separate from mainnet validator-weight and miner-evaluation state
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

On mainnet, rank is the live miner-side consensus result Yuma Consensus produces from production
validator weights before subnet incentives are allocated on the connected Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A rank example from one environment should not be read as
representing production miner evaluation on another network.

## Rank Sums Post-Clip Weights by Stake

[Yuma Consensus: Miner emissions](https://docs.learnbittensor.org/learn/yuma-consensus#miner-emissions)
documentation calculates each miner's aggregate rank as the stake-weighted sum of post-clip
[validator weights](https://docs.learnbittensor.org/resources/glossary#weight-vector). Rank is
therefore a merged miner-side result, not one validator's raw score on its own
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight)).

That formula keeps input and result vocabulary separate. Validator weights express what a validator
scored; [stake weight](https://docs.learnbittensor.org/resources/glossary#stake-weight) scales how
strongly each surviving post-clip signal counts when those evaluations are combined into rank.

## Consensus Score Clips Before Rank Finalizes

Before rank is calculated, Yuma Consensus compares validator weights against the stake-weighted
[consensus score](https://docs.learnbittensor.org/resources/glossary#consensus-score) benchmark and
clips values that exceed it. When a submitted weight exceeds the clipped benchmark for a miner,
neither the miner nor the submitting validator receives emissions for the excess portion
([Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping)).

Rank is read from the post-clip signal set.
[Trust](https://docs.learnbittensor.org/resources/glossary#trust) then compares final rank to
pre-rank support to show how much evaluation signal survived that filtering step rather than naming
the benchmark itself.

## Miner Incentives Normalize Rank Outcomes

Miner-side [incentives](https://docs.learnbittensor.org/resources/glossary#incentives) are derived
from rank after clipping and aggregation. Yuma Consensus normalizes those rank outcomes into the
miner emission vector for the epoch, which is then allocated when tempo-bound distribution runs
([Yuma Consensus: Miner emissions](https://docs.learnbittensor.org/learn/yuma-consensus#miner-emissions),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Rank therefore belongs to the miner evaluation path inside consensus, while emission vocabulary
names the broader tokenomics flow that later distributes value from those outcomes.

## Rank Names the Miner Result After Weights Merge

The [Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank) describes rank as the
final aggregate judgment Yuma Consensus computes from validator miner-ratings. It is the miner-side
result after many validator signals are compared, not one validator's raw score on its own
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

That keeps rank in result vocabulary. Validator weights name the inputs; rank names where each miner
lands after those inputs pass through consensus processing
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

## Late Weight Submissions Can Miss the Current Epoch

Subnet [hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) include an
activity cutoff measured in blocks within each epoch. A validator that does not submit weights
within the first activity-cutoff blocks of an epoch cannot participate until the next epoch
([Subnet Hyperparameters: Activity Cutoff](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#activitycutoff)).

That timing rule affects which validator signals can enter the consensus pass that later produces
rank. Validators who miss the early window sit out the current epoch rather than joining that round
of aggregation
([Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

## Distinction from Weight Matrix

The weight matrix collects validator weight vectors as structured input for Yuma Consensus. Rank is
the miner-side standing read after that grouped input is processed, not from one isolated vector
inside the matrix
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Weight matrix** — the grouped set of validator weight vectors entering consensus.
- **Rank** — miner-side standing after consensus processes that input.

## Distinction from Trust

Rank is the miner-side aggregate standing after Yuma Consensus filters and combines validator
weights. Trust is the ratio of final rank to pre-rank that shows how much support survived clipping
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Rank vocabulary names standing after aggregation. Trust vocabulary names how much of the pre-rank
signal survived filtering.

## Distinction from Consensus Score

Consensus score is the stake-weighted benchmark used to clip outlier validator weights. Rank is read
after that benchmark has shaped miner-side support on the selected subnet
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping)).

Consensus-score vocabulary names the agreement benchmark. Rank vocabulary names miner-side standing
after aggregation.

## Distinction from Incentives

Incentives name miner-side emission outcomes credited after rank is normalized. Rank is the standing
input that precedes that credited result on the selected subnet
([Glossary: Incentives](https://docs.learnbittensor.org/resources/glossary#incentives),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus: Miner emissions](https://docs.learnbittensor.org/learn/yuma-consensus#miner-emissions)).

Incentive vocabulary belongs to credited results. Rank vocabulary belongs to consensus standing
before that credit.

## Distinction from Weight Vector

A weight vector is one validator's submitted evaluation signals. Rank is interpreted after the full
validator set is filtered and aggregated into miner-side outcomes
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

Weight-vector vocabulary belongs to input signals. Rank vocabulary belongs to miner-side standing
after those signals are processed.

## Distinction from Validator Trust

Validator trust measures validator-side clipped influence across neurons. Rank measures miner-side
standing after the same Yuma pass on the selected subnet
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

The two terms share one Yuma pass but answer different reader questions on the selected subnet.

## Reader Boundary

Rank is a consensus-result term for miners after Yuma Consensus. It is not a universal quality
label, reward guarantee, or one validator's raw score
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Specific rank examples belong to the subnet, network, validator set, and source that produced them.

## Relationship to Multiple Incentive Mechanisms

Rank and multiple incentive mechanisms connect a consensus standing to a subnet's split evaluation.
Rank is the consensus standing that orders miners, while multiple incentive mechanisms is the case
where one subnet runs more than one evaluation mechanism, each with its own bond pool
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)).

On a multi-mechanism subnet, rank is computed separately within each mechanism rather than as one
combined ordering, so a single rank value belongs to one mechanism's evaluation track rather than
the whole subnet ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Relationship to Incentive Compatibility

Rank and incentive compatibility connect a mechanism to a design goal. Rank is the consensus
standing that orders miners and drives their reward, while incentive compatibility is the goal that
honest, high-quality work is the most profitable choice
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

For readers, rank is the mechanism and incentive compatibility is the goal it serves. Rank being
earned by genuine quality is what makes the subnet incentive-compatible, so rank delivers the
alignment that incentive compatibility names rather than being that goal itself
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Relationship to Trust

Rank names a miner's relative standing after validator weights are combined in Yuma Consensus. Trust
measures how much of that pre-consensus rank survived clipping in the same pass
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Readers comparing outcome level to clipping impact need both terms. Rank names the aggregated
standing used for miner-side emissions, while trust names the ratio between final rank and pre-rank
after outlier weights were removed
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Incentives](https://docs.learnbittensor.org/resources/glossary#incentives)).

Use rank when covering miner standing in the emission split; use trust when covering how much of
that standing survived consensus clipping rather than the standing value itself
([Emission](https://docs.learnbittensor.org/learn/emissions),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Standing outcome and clipping-survival ratio answer different reader questions in the same epoch
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).
