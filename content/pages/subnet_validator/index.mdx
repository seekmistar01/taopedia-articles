# Subnet Validator

A subnet validator is the evaluating role inside a Bittensor subnet. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#subnet-validator) describes a
validator, or subnet validator, as the subnet role that evaluates miner performance and sets weights
based on miner output, while
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) describes
validators as measuring the work miners produce.

## Role in a Subnet

The validator role is tied to the subnet's standards. A validator evaluates miner work according to
what that subnet is trying to produce, so the meaning of a useful evaluation depends on the subnet's
task and incentive design.
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) explains that
subnets can have different goals and standards, and
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
places evaluation inside the subnet's incentive mechanism.

## Relationship to Miners

Subnet validators and subnet miners form the basic evaluation relationship inside a subnet. Miners
produce work for the subnet, and validators measure that work against the subnet's standards. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#subnet-validator) defines the
validator side around miner output, while
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) describes
miners as producing a specific digital commodity.

## Relationship to Weights

Weights are how validator evaluation is represented for the consensus process, but the validator is
the evaluating role rather than the weight signal itself. This distinction keeps the role separate
from the data that later moves through consensus.
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) describes validator weights
as inputs to consensus, and the
[Glossary](https://docs.learnbittensor.org/resources/glossary#subnet-validator) ties validators to
weights based on miner output.

## Incentive Mechanism Context

A subnet validator's evaluation only makes sense inside the subnet's incentive mechanism. The task
defines the requested work, the protocol shapes miner-validator interaction, and the evaluation
method gives validators a way to judge the result.
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
describes these pieces together, while
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) explains why
subnet evaluation differs across subnet goals.

## Relationship to Multiple Mechanisms

A subnet validator is usually discussed for one incentive mechanism at a time, but subnets can also
run multiple mechanisms in parallel. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms) notes
that validators must evaluate miners separately for each mechanism.

For readers, a subnet validator still evaluates miner work inside a subnet market, but the mechanism
context matters. Evaluation in one mechanism does not substitute for evaluation duties in another
when a subnet uses more than one evaluation system.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)

## Reader Boundary

Subnet validator should be read as a role concept, not as a procedure or claim about any particular
validator. The [Glossary](https://docs.learnbittensor.org/resources/glossary#subnet-validator) keeps
the term scoped to evaluating miner output, and
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) provides the
broader subnet context without requiring procedural detail in this article.
