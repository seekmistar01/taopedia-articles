# Transaction Fees

Transaction fees are costs the Bittensor chain can attach to submitted state-changing actions
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

The term describes costs attached to submitted actions. It should be read separately from chain
reads, swap fees, and every other economic effect that can surround a transaction.

## Base Fee Context

The official fee documentation describes base transaction fees as having weight and length
components ([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

Weight is tied to computational load, while length is tied to submitted action size. This base-fee
context belongs to the chain action being submitted, not to passive observation.

In plain terms, the base fee is the cost attached to processing a submitted action. The extrinsics
reference describes those submitted runtime actions, while the fee page explains the cost categories
that can apply to them
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

## Chain Reads

The fee documentation identifies chain reads as free
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

That distinction matters because reading chain state does not ask Subtensor to record a state
change. Fee language should therefore stay attached to submitted actions rather than being applied
to every form of chain inspection.

A read can help someone inspect public state before deciding what to do next, but the read is not
the paid transaction. The fee appears when a state-changing call is submitted
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

## Swap Fee Boundary

The fee documentation treats swap fees as separate from base transaction fees
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

A stake-related action can therefore involve more than one cost concept. The base fee belongs to the
submitted chain action, while a swap fee belongs to the liquidity-linked part of the action.

So a reader should not treat a swap fee as just another name for the base transaction fee. They can
appear near the same action, but the fee page keeps the categories separate
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

## Batch Context

Batch transaction fees are described as aggregating the fee behavior of inner calls and adding
batch-wrapper overhead ([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

This means grouped submissions should still be read through their components. Grouping calls does
not erase the fee behavior of fee-bearing activity inside the batch.

For a batch, the wrapper adds its own overhead and the inner calls still matter. A batch is a
grouped submission, not a way to make every inner action free
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

## Recycling Context

The recycling and burning glossary entry describes collected transaction fees as recycled tokens
deducted from the issuance record for future emission
([Recycling and burning](https://docs.learnbittensor.org/resources/glossary#recycling-and-burning)).

That places transaction fees near issuance accounting. It does not make every transaction fee the
same as generic burning.

Recycling explains where collected fees go in issuance accounting. It does not change what the fee
was charged for at submission time
([Recycling and burning](https://docs.learnbittensor.org/resources/glossary#recycling-and-burning),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

## Network Context

Bittensor documentation separates mainnet, testnet, and localnet contexts
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Fee examples should keep the selected network attached when observed behavior matters. The
article-level concept can be stable while exact observed outcomes remain environment-specific.

That means an example fee should be tied to the network where the action was evaluated. A localnet
or testnet example can explain the idea, but it should not be read as a universal fee quote
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

## Reader Boundary

The official fee page supports claims about fee categories, free reads, batch behavior, and fee
recycling ([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

It is not a permanent quote for every action and not a calculator. Exact cost numbers need the
submitted action, network context, and evidence used for that observation.

The stable article-level point is that transaction fees are costs for submitted state-changing
actions. Free reads, swap fees, batch grouping, recycling, and network context are separate ideas
that help readers avoid overgeneralizing a fee example
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).
