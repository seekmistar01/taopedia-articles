# Non-fast Blocks

Non-fast blocks name the local-development timing mode that keeps Subtensor's default 12-second
block interval
([Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks),
[Run a Local Bittensor Blockchain Instance](https://docs.learnbittensor.org/local-build/deploy)).

The term is easiest to understand as the contrast to fast blocks. Both terms belong to local
blockchain context.

## Local Development Context

The local blockchain glossary entry describes a local blockchain as a private chain used for
development and testing, while the non-fast-blocks glossary entry describes ordinary local block
timing
([Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)).

Non-fast blocks therefore describe a local timing choice. They are not a separate public network,
subnet role, or incentive rule.

## Twelve-Second Timing

The official glossary describes non-fast blocks as following Subtensor's default 12-second block
interval
([Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)).

That interval gives local testing production-like spacing between blocks. It should be read as a
local timing setting rather than a broad claim about every Bittensor environment.

## Fast Blocks Contrast

Fast blocks and non-fast blocks name opposite local timing choices. Fast blocks accelerate local
block production to 250 millisecond intervals, while non-fast blocks keep the default 12-second
interval ([Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks),
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)).

The contrast is about development feedback. Fast blocks shorten local waiting time; non-fast blocks
preserve timing that is closer to default Subtensor cadence.

## Block-Based Process Context

Many Bittensor mechanisms are described in block terms. Non-fast blocks can make local observations
of block-counted behavior easier to compare with production-like timing because each local block
keeps the default interval
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

This does not make non-fast blocks a mechanism. They are the local timing context in which other
block-based mechanisms can be tested.

## Network Context

Bittensor documentation separates mainnet, testnet, and localnet contexts
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Non-fast blocks belong to the localnet side of that distinction. A local timing mode should not be
treated as an automatic statement about shared public network behavior.

## Reader Boundary

Non-fast blocks should be read as an ordinary local block-timing configuration. They are not a
separate chain category, a subnet task, a reward mechanism, or a guarantee that every environment
uses the same timing
([Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

That boundary keeps the term useful: non-fast blocks explain local timing, while the relevant subnet
or protocol documentation explains what the block-based behavior means.
