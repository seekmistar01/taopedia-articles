# Bittensor Networks

Bittensor Networks are the environments where Bittensor activity is interpreted. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet context, while
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction) frames Bittensor as
a platform made from subnets, blockchain state, and supporting tools.

## Mainnet, testnet, and localnet

Mainnet is the production environment for real-value Bittensor activity. Testnet is a shared testing
environment, and localnet is a user-controlled local environment for development and experiments.
The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
groups these environments together so readers can separate production context from testing context
before interpreting examples elsewhere in the docs.

## Networks are not subnets

A network environment is broader than a subnet. In
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets), a subnet is
an incentive-based marketplace where miners produce work and validators evaluate it. A Bittensor
network can provide the environment where subnet activity is tested or used, while a subnet remains
one market inside that environment. This is why
[[subnet_creation_mechanisms|Subnet Creation and Incentive Mechanisms]] is related but not the same
topic.

## Why context matters

Network context changes how a reader should understand examples, articles, and tool references. A
localnet example belongs to local testing, a testnet reference belongs to shared testing, and a
mainnet reference belongs to production context. The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction) presents subnets,
the blockchain, and supporting tools as separate platform components, so network context should not
be collapsed into any single component.

## Local testing context

Localnet is useful because it gives developers a controlled Bittensor environment before relying on
shared network conditions. It is still a network context, not a subnet and not a separate protocol
role. The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)
reference places localnet alongside mainnet and testnet, while the
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction) describes subnet
development as moving from local testing toward broader Bittensor environments.

## Network names in documentation

When a Bittensor page names a network, read the name as context for where the example or concept
belongs. Network labels do not explain miners, validators, staking, or subnet mechanics by
themselves. They help keep [[mining_and_validating|Mining and Validating]],
[[staking_and_delegation|Staking and Delegation]], and subnet articles clear about the environment
being discussed. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference is the
right place to confirm network-specific details.

## Relationship to Mainchain

Bittensor Networks and Mainchain name related but different network-context ideas. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference groups
mainnet, testnet, and localnet, while the
[Glossary: Mainchain](https://docs.learnbittensor.org/resources/glossary#mainchain) describes the
primary Bittensor network used for production purposes.

For readers, Bittensor Networks is the broader environment label, while Mainchain is the production
subset inside that picture. A mainnet example and a Mainchain reference both belong to production
context, but not every network environment is Mainchain.

References: [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Mainchain](https://docs.learnbittensor.org/resources/glossary#mainchain)

## Relationship to Subtensor

Bittensor Networks names the environments where Bittensor activity is interpreted, while Subtensor
names the blockchain layer that runs in each environment. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference lists
mainnet, testnet, and localnet as deployment contexts, and the
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor) describes
Subtensor as Bittensor's layer 1 blockchain and system of record for transactions and rankings.

For readers, Bittensor Networks is the network-context layer that says where activity is
interpreted, and Subtensor is the chain layer that records that activity. The same Subtensor concept
applies inside each network environment, but production, test, and local contexts are still read
separately.

References: [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor)

## Relationship to Local Blockchain

Localnet runs as a local blockchain. The
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)
describes a local blockchain as private, isolated, and used for developing and testing subnet
incentive mechanisms. This places local blockchain as the chain-level implementation of the localnet
environment that [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)
describes.

For readers, the relationship keeps two terms distinct: localnet is the network-environment label,
and local blockchain is the chain-software label for that same local context. Mainnet and testnet do
not collapse into local blockchain, because each network environment has its own deployment context.

References:
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)

## Relationship to Fast Blocks

Fast blocks are a local-development timing configuration that belongs to localnet context. The
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks) describes
fast blocks as a development-only local-chain configuration that accelerates block production. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference places
localnet alongside mainnet and testnet, so fast blocks sit on the localnet side of that distinction.

For readers, fast blocks are a network-specific behavior: they apply inside the localnet
environment, not in production mainnet or shared testnet. Reading the term alongside Bittensor
Networks keeps fast blocks from being mistaken for a production chain behavior.

References: [Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)

## Relationship to Public Subtensor

Bittensor Networks and Public Subtensor name related but different parts of chain access context.
The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
groups mainnet, testnet, and localnet, while the
[Glossary: Public Subtensor](https://docs.learnbittensor.org/resources/glossary#public-subtensor)
describes publicly reachable access to the Subtensor chain layer.

For readers, Bittensor Networks is the environment label, while Public Subtensor names public chain
access within mainnet and testnet rather than isolated local development. Public Subtensor does not
rename Subtensor or create a separate blockchain.

References: [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Public Subtensor](https://docs.learnbittensor.org/resources/glossary#public-subtensor)

## Relationship to Non-fast Blocks

Non-fast blocks are a local-development timing configuration that belongs to localnet context. The
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)
describes non-fast blocks as a development-only configuration that follows Subtensor's default
12-second block interval. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference places
localnet alongside mainnet and testnet, so non-fast blocks sit on the localnet side of that
distinction.

For readers, non-fast blocks are a network-specific behavior: they apply inside the localnet
environment, not in production mainnet or shared testnet. Reading the term alongside Bittensor
Networks keeps non-fast blocks from being mistaken for a production chain behavior.

References:
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)
