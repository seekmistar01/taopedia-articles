# Bittensor Wallet

A Bittensor wallet is the identity and ownership container for a participant on the network. The
official
[Glossary: Bittensor Wallet](https://docs.learnbittensor.org/resources/glossary#bittensor-wallet)
describes it as a digital wallet that holds core ownership in the Bittensor network and underlies
user operations across staking, delegation, and subnet participation.

## Local and External Wallet Boundaries

Bittensor documentation also distinguishes where wallet material is stored, such as local software
wallets versus external wallet integrations. Those labels describe storage and tooling context
around the same ownership idea rather than separate network identities
([Glossary: Local Wallet](https://docs.learnbittensor.org/resources/glossary#local-wallet),
[Glossary: External Wallet](https://docs.learnbittensor.org/resources/glossary#external-wallet)).

For readers, Bittensor wallet still names the participant ownership container. Local wallet,
external wallet, and wallet location are adjacent tooling terms, not replacements for the core
wallet concept.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from local testing to testchain and then mainchain. For a
Bittensor wallet, that sequence changes how readers should interpret evidence about key and identity
management ([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

In local testing, a Bittensor wallet can be used to manage coldkeys and hotkeys inside an isolated
environment. That is useful for checking wallet creation and key registration mechanics, but it
remains isolated feedback from a local environment.

On testchain, Bittensor wallet behavior can be observed in a shared testing network. This gives
stronger evidence about how coldkeys, hotkeys, and staking operations interact with on-chain
identity than a private local run, while still keeping the result separate from production mainchain
key management.

On mainchain, a Bittensor wallet holds a participant's actual on-chain identity, stake, and key
registrations. The
[Glossary: Bittensor Wallet](https://docs.learnbittensor.org/resources/glossary#bittensor-wallet)
defines the wallet as an identity and ownership container, so production wallet context is tied to
real coldkey and hotkey ownership on the active network.

For readers, this keeps Bittensor wallet examples from sounding stronger than their environment
supports. Local, testchain, and mainchain contexts can all describe key and identity management, but
they do not carry the same interpretive weight.

## Cryptographic, Hardware, and Software Senses of Wallet

The [Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets) documentation
distinguishes three senses of the term that are easy to confuse. The cryptographic wallet is the key
material — essentially the coldkey that defines on-chain identity. A hardware wallet is a device
that holds that key material and signs across a protected device boundary. A wallet application is
software that uses the key material to perform blockchain operations.

For readers, the Bittensor wallet concept on this page is the cryptographic wallet: the on-chain
identity that persists even if every application or device is removed, because it can be recovered
from the seed phrase. Hardware devices and wallet applications are ways to hold or use that
identity, not separate identities
([Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets)).

## One Cryptographic Wallet, Many Applications

The same documentation notes that any cryptographic wallet can be loaded into any number of wallet
applications, and that the on-chain wallet survives even if every application is closed or its
device is destroyed, since it can be restored from the seed phrase. That is why this page treats the
wallet as an ownership container rather than a single installed program
([Working with Keys](https://docs.learnbittensor.org/keys/working-with-keys)).

Keeping that separation in view helps readers avoid equating a wallet application's local state with
the underlying cryptographic wallet that the seed phrase actually recovers.

## Distinction from Seed Phrase

A Bittensor wallet is the identity container for keys and stake. A seed phrase is offline recovery
material that can restore that wallet's key access
([Glossary: Bittensor Wallet](https://docs.learnbittensor.org/resources/glossary#bittensor-wallet),
[Glossary: Seed Phrase](https://docs.learnbittensor.org/resources/glossary#seed-phrase)).

Ownership-container vocabulary and secret backup vocabulary require opposite handling.

## Distinction from Coldkeys

A Bittensor wallet organizes coldkey and hotkey roles. A coldkey is the custody key inside that
wallet package ([Glossary: Coldkey](https://docs.learnbittensor.org/resources/glossary#coldkey),
[Glossary: Bittensor Wallet](https://docs.learnbittensor.org/resources/glossary#bittensor-wallet)).

Container vocabulary names the whole ownership package; coldkey vocabulary names one authority role
inside it.

Wallet vocabulary in Bittensor separates long-term ownership from operational keys. Wallet
documentation describes coldkeys as the security-critical key that controls TAO and stake, and
hotkeys as operational keys used for day-to-day network roles. A Bittensor wallet is the container
that organizes those keys rather than a single key file label
([Glossary: Coldkey](https://docs.learnbittensor.org/resources/glossary#coldkey),
[Glossary: Hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey),
[Wallets, Coldkeys and Hotkeys in Bittensor](https://docs.learnbittensor.org/keys/wallets)).

For readers, wallet names the ownership package, while coldkey and hotkey name specific key roles
inside that package.

## Distinction from Coldkey Swap

Coldkey swap migrates on-chain identity to a new coldkey while preserving hotkey registrations. A
Bittensor wallet still names the ownership container whose controlling coldkey can change through
that migration ([Coldkey Swap](https://docs.learnbittensor.org/keys/coldkey-swap),
[Glossary: Bittensor Wallet](https://docs.learnbittensor.org/resources/glossary#bittensor-wallet)).

Wallet vocabulary persists across swap; coldkey swap vocabulary names the migration event.

## Distinction from Proxy Wallets

A Bittensor wallet is the owned identity package. A proxy wallet grants scoped authority for another
account to act on that package's behalf
([Proxies overview](https://docs.learnbittensor.org/keys/proxies),
[Glossary: Bittensor Wallet](https://docs.learnbittensor.org/resources/glossary#bittensor-wallet)).

Ownership vocabulary names the wallet container; proxy vocabulary names delegated operating
permission around it.

## Distinction from Hotkeys

Hotkeys are operational keys inside a Bittensor wallet. The wallet names the container that groups
those keys under coldkey ownership
([Glossary: Hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey),
[Glossary: Bittensor Wallet](https://docs.learnbittensor.org/resources/glossary#bittensor-wallet)).

Container vocabulary and operational-key vocabulary answer different wallet-structure questions. The
wallet groups the coldkey with the hotkeys it controls for staking, delegation, and subnet
participation, rather than naming any single operational key on its own
([Glossary: Hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey)).

## Distinction from Local Wallet

Local wallet names where wallet files are stored on a machine. Bittensor wallet names the on-chain
identity and ownership container those files represent
([Glossary: Local Wallet](https://docs.learnbittensor.org/resources/glossary#local-wallet),
[Glossary: Bittensor Wallet](https://docs.learnbittensor.org/resources/glossary#bittensor-wallet)).

Storage-context vocabulary and identity-container vocabulary sit on adjacent but different layers.

## Distinction from Hotkey-Coldkey Pair

A hotkey-coldkey pair names how operational and ownership keys bind for authentication. A Bittensor
wallet is the broader container that organizes those keys and stake
([Glossary: Hotkey-Coldkey Pair](https://docs.learnbittensor.org/resources/glossary#hotkey-coldkey-pair),
[Glossary: Bittensor Wallet](https://docs.learnbittensor.org/resources/glossary#bittensor-wallet)).

Pair vocabulary names one binding relationship; wallet vocabulary names the full ownership package.

## Distinction from Delegation

Bittensor wallet ownership also frames staking participation. The
[Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation)
describes how TAO holders stake to validators, and wallet material is what authorizes those
financial operations on chain. Wallet therefore belongs to the identity side of staking vocabulary,
not to validator performance or subnet emission formulas
([Delegation](https://docs.learnbittensor.org/staking-and-delegation/delegation),
[Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation)).

This article does not describe how to create, import, or back up a wallet. Those steps belong in
wallet documentation.

## Distinction from Yuma Consensus

A Bittensor wallet holds coldkey and hotkey pairs used to sign extrinsics and manage stake. It names
custody tooling vocabulary, not the on-chain step that turns validator weight submissions into
emission shares ([Wallets](https://docs.learnbittensor.org/keys/wallets),
[Glossary: Coldkey](https://docs.learnbittensor.org/resources/glossary#coldkey)).

Yuma Consensus is the on-chain process that converts submitted validator weights into emission
shares after evaluation completes. Wallets sign operations that move stake or set weights, but Yuma
Consensus settles scoring outcomes on a subnet
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers should keep wallet vocabulary separate from the settlement step. Bittensor wallet answers
where keys are stored and signed; Yuma Consensus answers how validator weights become emission
shares in a subnet ([Wallets](https://docs.learnbittensor.org/keys/wallets),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Bittensor wallet** — tooling that holds coldkey and hotkey signing material.
- **Yuma Consensus** — on-chain weight aggregation that settles emissions.

## Reader Boundary

Bittensor wallet should not be read as a balance report, a validator recommendation, or a command
tutorial. It is a concept term for the identity container behind Bittensor key and stake operations.
Exact addresses, balances, and backup steps belong in wallet references and live chain data.
