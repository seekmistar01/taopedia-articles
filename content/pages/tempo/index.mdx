# Tempo

A tempo is a block-based timing interval used by Bittensor subnets. Rewards accumulate during a
tempo and are distributed at the interval boundary through
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)
([Emission](https://docs.learnbittensor.org/learn/emissions)).

As a subnet hyperparameter, `tempo` gives subnet processes a cadence measured in blocks. It is a
timing concept rather than a definition of what useful subnet work means
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Block-Based Cadence

Tempo names a cadence, not a task. A subnet's incentive mechanism defines the work being evaluated;
tempo defines the block interval over which periodic subnet processes are grouped
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

That distinction keeps timing vocabulary separate from evaluation vocabulary. Tempo helps determine
when subnet work is processed, while the incentive mechanism explains what validators are evaluating
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Block Count Versus Wall Clock

Tempo is counted in blocks, so its wall-clock length depends on the network context producing those
blocks. Bittensor network documentation separates localnet timing modes from testnet and mainnet,
including fast-block and non-fast-block local settings
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)).

For readers, that means a tempo example should keep the block count and the environment label
together. The tempo value explains how many blocks group the subnet process; the surrounding network
context explains how quickly those blocks pass in that example.

## Emission Timing Context

Bittensor emission has an accumulation-and-distribution rhythm. Rewards accumulate during a tempo
and are distributed through consensus at the interval boundary
([Emission](https://docs.learnbittensor.org/learn/emissions)).

Yuma Consensus is the aggregation step that uses validator-submitted signals to determine how those
rewards are allocated inside the subnet. Tempo supplies the cadence for that periodic process
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Hyperparameter Boundary

Tempo is a subnet hyperparameter, so it should not be read as one universal value for every subnet.
The stable concept is block-based cadence; exact values belong with the relevant subnet parameter
source ([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

This also means a tempo claim needs context. A statement about cadence should say whether it is
explaining the concept or referring to a specific subnet setting
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

## Epoch Boundary

Tempo and epoch are related but distinct timing terms. Tempo is the subnet-specific cadence for
epoch transitions, while an epoch is the consensus period produced by that cadence
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

For readers, tempo names the interval; epoch names the period in which consensus work is carried
out. A tempo should not be treated as the full consensus process
([Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)).

## Commit Reveal Timing

Commit Reveal uses tempo-based timing for delayed weight visibility. Its reveal period is expressed
in tempos, making tempo relevant to when concealed validator weights become usable
([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

This is a timing relationship, not a new definition of tempo. Tempo remains the block-based cadence;
Commit Reveal uses that cadence as part of delayed weight submission and reveal timing
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Multiple Mechanisms Scope

When a subnet has multiple incentive mechanisms, tempo still names the shared subnet cadence. The
mechanisms can have separate weight and consensus paths, but that does not require treating tempo as
a separate schedule for each mechanism
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

Multiple mechanisms clarify how evaluation is organized; tempo clarifies when periodic processing
occurs
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)).

## Development Stage Context

Bittensor's subnet-development path moves from localnet to testnet and then mainnet
([Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)).
For tempo, that sequence changes how readers should interpret examples of block cadence, epoch
boundaries, and reward timing.

In localnet, tempo behavior can be tested in an isolated environment. Localnet examples reflect
local block production, local subnet parameters, and local chain state rather than production subnet
timing.

On testnet, tempo can be observed in a shared, non-production network. Testnet examples can show
realistic block progression and interval boundaries, but they remain separate from mainnet emission
and consensus outcomes.

On mainnet, tempo is the production block cadence that groups subnet processes into periodic reward
and consensus timing. The [Emission](https://docs.learnbittensor.org/learn/emissions) documentation
describes rewards accumulating during a tempo and being distributed through consensus at the
interval boundary.

Because tempo is a subnet hyperparameter, a tempo example belongs to the relevant subnet setting
rather than every subnet on Bittensor
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A tempo example from one environment should not be read as
representing block cadence, epoch timing, or emission timing in another environment.

## Reader Boundary

Tempo should be read as a block-count cadence for subnet processes. It is not a wall-clock promise,
a subnet quality signal, or a substitute for Yuma Consensus. It tells readers how timing is grouped
for rewards, epochs, and related subnet processes
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

The article-level point is stable even when specific values differ by subnet: tempo is the cadence
that helps turn block progression into periodic subnet consensus and emission timing
([Emission](https://docs.learnbittensor.org/learn/emissions)).

## Injection Runs Every Block, Distribution Waits for Tempo

Official [Emission](https://docs.learnbittensor.org/learn/emissions) documentation separates the
emissions process into two stages: injection and distribution. Injection adds new liquidity to each
subnet's pools every [block](https://docs.learnbittensor.org/resources/glossary#block), while
distribution waits until the end of each
[tempo](https://docs.learnbittensor.org/resources/glossary#tempo).

The [Glossary: Emission](https://docs.learnbittensor.org/resources/glossary#emission) entry states
the same rhythm in plain terms. Every block,
[TAO](https://docs.learnbittensor.org/resources/glossary#tao) is injected into each subnet, and
every tempo the accumulated value is extracted by participants including miners, validators,
stakers, and subnet creators.

That split explains why tempo reading must keep two speeds in view. Block progression continuously
feeds subnet pools, but participant-facing reward allocation from that buildup is grouped and
settled on the tempo boundary through
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus).

References: [Emission](https://docs.learnbittensor.org/learn/emissions),
[Glossary: Emission](https://docs.learnbittensor.org/resources/glossary#emission)

## Miner Ranks Refresh From the Weight Matrix

The [Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo) defines tempo as the
block interval over which [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)
calculates [emissions](https://docs.learnbittensor.org/resources/glossary#emissions) from the latest
available ranking [weight matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix).

The [Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank) adds that each miner's
[rank](https://docs.learnbittensor.org/resources/glossary#rank) score is the final aggregate
judgment computed by Yuma Consensus from validator miner-ratings submitted during that tempo.
Outlier weights are removed through consensus clipping before those ranks are finalized.

Those ranks then feed incentive allocation inside the subnet. Tempo therefore marks not only when
rewards are distributed, but also when the consensus layer refreshes performance scores from the
gathered validator signals that formed the weight matrix.

References: [Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)

## Activity Cutoff Sets the Early Epoch Window

Subnet [hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) include an
activity cutoff measured in blocks within each
[epoch](https://docs.learnbittensor.org/resources/glossary#epoch). Because an epoch spans the block
count set by the subnet's tempo hyperparameter, that cutoff applies inside a tempo-sized consensus
period rather than as a separate schedule.

Official documentation states that if a
[validator](https://docs.learnbittensor.org/resources/glossary#validator) does not submit
[validator weights](https://docs.learnbittensor.org/resources/glossary#validator-weights) within the
first activity cutoff blocks of the epoch, that validator cannot participate in the current epoch
and must wait until the next epoch begins.

That rule ties tempo reading to participation timing inside the interval. Tempo sets how long each
epoch lasts, while activity cutoff sets how early in that epoch a validator must act to be included
in the current consensus pass.

References:
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)
