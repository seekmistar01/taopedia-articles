# Batch Transactions

Batch transactions group multiple Bittensor chain actions into one submitted transaction
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)). The term describes
how actions are packaged for submission, not a passive chain read and not a new meaning for every
inner action.

## Grouped Submission

A batch is a container for submitted calls. The outer transaction groups the calls, while each inner
call keeps its own purpose, inputs, and possible result
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

That makes batching useful to describe packaging. It does not turn several runtime actions into a
single conceptual action, and it does not replace the need to understand what each inner call does.

## Failure Behavior

Batch forms differ in what happens after an inner action fails. At a reference level, the important
question is whether the grouped submission stops after a failed inner action, requires all inner
actions to succeed together, or continues evaluating later actions
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

Readers should identify the batch form before interpreting a failed inner action. The same inner
call can have different practical meaning depending on the wrapper around it.

## Fees and Submitted Actions

Batching changes grouping, but it does not make the inner actions free. Bittensor's fee
documentation explains that batch fees aggregate inner-action fee behavior and add wrapper overhead
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

The submitted-action context also separates batches from chain reads. Reading state observes the
chain; a batch submits grouped calls for runtime processing
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

## Inner Call Context

The batch wrapper answers how calls are submitted together. The inner calls still need their own
runtime-method context, because each call can touch a different part of Subtensor and can produce
its own events or state changes
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

This keeps the grouping behavior separate from the purpose of the calls inside the batch. A batch
can be one submitted transaction while still containing several actions that readers should
interpret separately.

## Network and Interpretation

A batch example belongs to the Bittensor network where it is submitted. Mainnet, testnet, and
localnet are separate contexts, so observed behavior from one environment should not stand in for
another network's chain history
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

A concrete example should name the batch form, the network context, and the inner calls being
interpreted. The grouping concept can stay stable while fees, failures, and resulting state depend
on the selected wrapper and calls.
