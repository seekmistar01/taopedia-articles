# Subnet Protocol

A subnet protocol is the rule set that defines how validators and miners interact inside a Bittensor
subnet
([Glossary: Subnet Protocol](https://docs.learnbittensor.org/resources/glossary#subnet-protocol),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

The term belongs to incentive-mechanism vocabulary. It names the interaction pattern around work,
not the work target itself and not the scoring method.

## Interaction Rules

The protocol describes how a validator can request work and how a miner can return a response for
evaluation. Incentive-mechanism documentation places protocol beside task and scoring model as part
of subnet design
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Subnet Protocol](https://docs.learnbittensor.org/resources/glossary#subnet-protocol)).

For readers, protocol is the exchange pattern. It is the connective layer between requested work and
validator evaluation.

## Relationship to Miners and Validators

Subnet protocols sit between the two subnet roles. Miners produce responses within the subnet's
rules, and validators evaluate the returned work against subnet standards
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Glossary: Subnet Protocol](https://docs.learnbittensor.org/resources/glossary#subnet-protocol)).

The protocol helps organize how those roles interact. It does not replace either role.

## Relationship to the Task

The subnet task and subnet protocol are related but separate ideas. The task describes the work
target, while the protocol describes the interaction pattern used around that work
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Subnet Protocol](https://docs.learnbittensor.org/resources/glossary#subnet-protocol)).

This distinction keeps "what miners produce" separate from "how validators and miners exchange
requests and responses."

## Incentive Context

A subnet protocol matters because interaction rules are part of how a subnet turns work into
evaluated output. The protocol works alongside the task and scoring model inside the incentive
mechanism
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

Each subnet can define its own goals and standards, so protocol details should be read in the
context of the subnet's incentive design.

## Multiple-Mechanism Context

When a subnet runs multiple incentive mechanisms, protocol expectations should be read in the
mechanism context being discussed. Different mechanisms can evaluate different kinds of miner work
inside the same subnet
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)).

This keeps one mechanism's interaction pattern from being treated as the whole subnet's only
protocol context.

## Reader Boundary

Subnet protocol should be read as a concept article term. It is not a claim that every subnet uses
the same exchange pattern, not a code specification, and not a deployment-status report
([Glossary: Subnet Protocol](https://docs.learnbittensor.org/resources/glossary#subnet-protocol),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

The stable article-level point is that a subnet protocol defines the request-and-response rules by
which miners and validators interact inside a subnet.
