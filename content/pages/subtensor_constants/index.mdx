# Subtensor Constants

Subtensor constants are named values in Bittensor's Subtensor runtime.

## What Constants Represent

In the [Subtensor Constants reference](https://docs.learnbittensor.org/subtensor-api/constants), a
constant is a runtime-defined value rather than an account, subnet, or balance record. Constants are
useful for identifying fixed values exposed by the runtime, such as defaults, bounds, type limits,
timing values, or feature-status values.

That makes constants different from ordinary article prose about a Bittensor feature. A constant can
name one value used by a feature, but it does not explain every rule or state transition connected
to that feature. The constants reference is therefore a map of named runtime values, not a full
mechanism guide.

## Runtime Areas

The [Subtensor Constants reference](https://docs.learnbittensor.org/subtensor-api/constants) groups
constants by runtime area. Examples include aura, balances, commitments, contracts, crowdloan,
drand, safe mode, scheduler, system, timestamp, utility, swap, and the Subtensor module.

The runtime-area grouping is part of the meaning of a constant. A value under balances, timestamp,
system, or the Subtensor module belongs to that owning area first. The same kind of number or flag
can carry different meaning when it appears under a different runtime area.

## Bittensor-Specific Context

The Subtensor module section of the
[constants reference](https://docs.learnbittensor.org/subtensor-api/constants) contains
Bittensor-specific entries for subnet defaults, rate-limit defaults, weight-related defaults, Liquid
Alpha status, and Yuma3 status. Other runtime areas cover broader chain behavior such as balances,
contracts, scheduled work, system limits, and timestamp timing.

This split helps separate Bittensor-specific values from general runtime values. A constant under
the Subtensor module is closer to subnet and incentive-system vocabulary, while a constant under
balances or system belongs to broader chain accounting and execution context.

## Interpreting Constant Entries

A constant entry in the [reference](https://docs.learnbittensor.org/subtensor-api/constants) should
be read as a bundle of context. The runtime area identifies ownership, the name identifies the
entry, the type describes the value shape, the access path shows where the reference exposes it, and
the summary gives the short human-readable description.

That structure keeps constants from being read as isolated numbers. The value itself matters, but
the owning runtime area and summary explain what kind of value it is. This is especially important
for Bittensor-specific constants because defaults, bounds, feature flags, and timing values can sit
near the same subnet or consensus vocabulary while serving different purposes.

The constants page is most useful when each row is read as a reference entry, not as a complete
feature description. The
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants) reference exposes the
runtime area, constant name, type, access path, and summary. Those fields work together: the name
tells which value is being referenced, the type tells the value shape, and the runtime area tells
where the value belongs.

This matters because similar-looking values can play different roles. A timing value, a limit, a
default, and a feature-status flag may all appear as constants, but they answer different questions.
The constant confirms that the runtime exposes a named value; the surrounding feature documentation
explains how that value fits into user-facing behavior.

In practice, constants are a good starting point for precise vocabulary. If a page discusses a
runtime bound, default, or status flag, the constants reference can identify the official name and
owning runtime area. If the question is how that value affects chain behavior, the reader needs the
related concept, storage, event, or mechanism reference as well. That keeps constants useful for
naming runtime values without stretching them into complete explanations of the feature that uses
the value.

The
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
guide includes constants among the chain data that can be inspected. That places constants in the
read-only reference side of Bittensor tooling: they help readers observe exposed runtime metadata
without submitting a state-changing action.

This keeps constants separate from storage and events. A constant can show a runtime-defined value,
a storage item can show mutable state, and an event can show a runtime record emitted during
processed activity ([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

For Taopedia, that distinction is useful because it prevents a constant from being treated as proof
of current mutable state. Constants give the named reference value; storage, events, and other
chain-observation surfaces give different evidence about what the selected network is doing.

## Fixed Values and Mutable State

Constants expose fixed runtime values, while
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) exposes state entries
that can change as blocks are processed. Events provide another view by naming runtime records
emitted during activity through the
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events) reference.

Those three reference surfaces answer different questions. A constant can expose a default, bound,
timing value, or feature flag. A storage entry can locate a mutable state value. An event can name a
runtime record produced during processing. They can appear near the same feature without meaning the
same thing.

## Network and Runtime Scope

Subtensor constants belong to a runtime context. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, while
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
includes constants among the chain data readers can inspect.

A constant value is therefore clearest when it is tied to the runtime and network context being
discussed. A concept article may explain the role of a parameter, while the constants reference
names the value exposed by a runtime. If a runtime or network context changes, the article should
not treat an older constant value as a timeless explanation of Bittensor behavior.

## Defaults, Bounds, and Feature Flags

Many Bittensor-specific entries in the
[Subtensor Constants reference](https://docs.learnbittensor.org/subtensor-api/constants) are most
useful as defaults, bounds, or status flags. Subnet defaults describe starting assumptions for
subnet behavior. Rate-limit defaults and other bounds describe limits around repeated or
resource-sensitive actions. Liquid Alpha and Yuma3 status constants show how named protocol features
can appear in the runtime reference.

These entries still need surrounding context. A default is not always the final value used by every
runtime path, a bound is not a full policy explanation, and a feature-status constant is not a
complete feature guide. The [Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)
documentation, for example, explains cooldown behavior that cannot be reduced to the existence of a
single constant.
