# Bittensor Governance

Bittensor governance is the network process for reviewing and executing privileged changes. The
current model moves administrative authority away from a single `sudo` key and toward a bicameral
structure: the Triumvirate creates proposals, and the Senate approves or rejects them.

References: [Governance Overview](https://docs.learnbittensor.org/governance),
[Senate](https://docs.learnbittensor.org/governance/senate)

## Why governance matters

Privileged network changes can affect subnet creation, runtime parameters, chain upgrades, and other
configuration that shapes how Bittensor operates. Governance raises the review threshold for those
changes by separating proposal creation from Senate approval.

Reference: [Governance Overview](https://docs.learnbittensor.org/governance)

## Proposal flow

A proposal wraps the call data for a privileged action. It does not execute immediately when it is
created. The official governance docs describe two required steps before execution:

- A majority of the Senate must approve the proposal.
- A Triumvirate member must close the proposal after approval.

When a proposal is closed successfully, the proposal call data is executed in the same block as the
close action. This gives the Senate a review gate while leaving final execution with the
Triumvirate-controlled proposal process.

Reference: [Governance Overview](https://docs.learnbittensor.org/governance)

## The Senate

The Senate is the governance body that reviews Triumvirate proposals. The official Senate docs
describe membership as stake-based and explain how senators view proposals, vote, and participate in
the approval process.

For readers, the practical point is that governance participation is not just a social process. It
is tied to Bittensor account roles, wallet permissions, and proposal records. The BTCLI permissions
guide documents the CLI permission model for governance-related actions.

References: [Senate](https://docs.learnbittensor.org/governance/senate),
[BTCLI Permissions Guide](https://docs.learnbittensor.org/btcli/btcli-permissions)

## Viewing and voting

Proposal visibility is public. Anyone can inspect proposals currently before the Senate without
special permissions. Voting is restricted to senators, and senators use the proposal hash to cast an
approval or disapproval vote.

Reference: [Senate](https://docs.learnbittensor.org/governance/senate)

## Bicameral responsibilities

Bittensor governance separates proposal creation from proposal approval. The governance overview
describes the Triumvirate as the body that creates proposals and the Senate as the body that
approves them. That split replaces the older pattern where privileged network actions depended on a
single `sudo` key.

The two bodies therefore play different roles in the same process. A Triumvirate proposal brings a
privileged call before governance, while Senate approval supplies the review threshold before that
call can be executed. This keeps governance vocabulary from treating proposal creation and approval
as the same action.

References: [Governance Overview](https://docs.learnbittensor.org/governance),
[Senate](https://docs.learnbittensor.org/governance/senate)

## Proposal execution lifecycle

A governance proposal is not only a discussion item. It wraps call data for a privileged action,
waits for Senate approval, and then requires a Triumvirate member to close it after approval. When
the close action succeeds, the call data executes in the same block as that close.

This lifecycle gives governance proposals three separate stages: creation, Senate review, and
closure-driven execution. A proposal that has been created but not approved remains pending. A
proposal that has been approved but not closed has passed the Senate gate but has not yet executed.
The execution point is the successful close action.

Reference: [Governance Overview](https://docs.learnbittensor.org/governance)

## Senate membership and voting

The Senate docs describe the Senate as a group of delegates who participate in proposals and control
a significant portion of total network stake. Once the required conditions are met by a coldkey and
hotkey pair, that senator can vote on Triumvirate proposals.

Viewing and voting have different permission boundaries. Proposal viewing is public, while proposal
voting is limited to senators. This distinction explains why a user can inspect Senate proposals
without holding Senate permissions, but cannot cast an approval or disapproval vote unless the
account satisfies the Senate requirements.

References: [Senate](https://docs.learnbittensor.org/governance/senate),
[BTCLI Permissions Guide](https://docs.learnbittensor.org/btcli/btcli-permissions)

## Privileged change scope

Governance is scoped to privileged network changes. The governance overview names examples such as
changing hyperparameters, creating new subnetworks, and chain upgrades as actions that previously
required privileged permission. Under governance, those changes move through proposal review instead
of relying on one privileged key.

This scope separates governance from ordinary subnet work. Miners and validators participate in
subnet activity, while governance handles administrative changes that can alter network rules,
runtime settings, or subnet configuration. Governance is therefore the review path for privileged
changes, not the everyday process by which miners produce work or validators score it.

References: [Governance Overview](https://docs.learnbittensor.org/governance),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)

## Practical implications

- **For stakers:** Delegation can affect governance because Senate participation is stake-based.
- **For senators:** Proposal review requires understanding the call data and operational impact of
  the proposed change.
- **For subnet creators and operators:** Governance can affect network parameters, subnet
  configuration, and the rules that determine who can participate.
- **For readers:** Governance records should be read through proposal details, Senate voting status,
  and the execution step that closes an approved proposal.

References: [Governance Overview](https://docs.learnbittensor.org/governance),
[Senate](https://docs.learnbittensor.org/governance/senate)

## Relationship to Multiple Mechanisms

Governance reviews privileged network and subnet configuration changes. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#subnet-creator) notes that subnet
creators can configure multiple incentive mechanisms to distribute emissions across different types
of work or evaluation criteria.

For readers, governance still names proposal review for privileged changes. Subnet markets may later
run more than one incentive mechanism under rules set through subnet design.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Subnet Creator](https://docs.learnbittensor.org/resources/glossary#subnet-creator)

## Relationship to Coinbase

Bittensor governance and coinbase are related but different parts of Bittensor network vocabulary.
Governance is the bicameral review process where the Triumvirate creates proposals and the Senate
approves or rejects them, gating privileged changes to network configuration. Coinbase is the
per-block Subtensor mechanism that accumulates pending emissions and injects them into subnet pools,
triggering Yuma Consensus rounds when subnets reach their epoch boundary. The
[Governance Overview](https://docs.learnbittensor.org/governance) describes the proposal and Senate
model, and the
[Coinbase Implementation](https://docs.learnbittensor.org/subtensor-api/extrinsics#coinbase)
describes the per-block emission mechanism.

Governance proposals can target the runtime parameters that coinbase operates under — such as
emission schedules, subnet configuration, or other privileged network-level settings that shape the
per-block emission pipeline. When a proposal is approved and closed, its call data executes in the
same block, after which coinbase incorporates the updated configuration in subsequent block cycles.
Governance names the review gate for those kinds of privileged changes; coinbase names the mechanism
that runs each block under whatever configuration governance has established.

References: [Governance Overview](https://docs.learnbittensor.org/governance),
[Coinbase Implementation](https://docs.learnbittensor.org/subtensor-api/extrinsics#coinbase)

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from local testing to testchain and then mainchain. For
governance, that sequence changes how readers should interpret evidence about proposal review and
Senate approval.

In local testing, governance mechanics can be exercised inside an isolated environment to check
whether proposal submission and approval flows work correctly. That is useful for verifying
governance behavior, but it remains isolated feedback from a local environment.

On testchain, governance behavior can be observed in a shared testing network. This gives stronger
evidence about how Triumvirate proposals and Senate votes interact with privileged network changes
than a private local run, while still keeping the result separate from production mainchain
governance state.

On mainchain, governance controls live privileged network changes through real Senate votes and
Triumvirate proposals. The [Governance Overview](https://docs.learnbittensor.org/governance)
documentation describes the review process, so production governance context is tied to actual
on-chain proposals and votes on the active network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A governance outcome observed in one environment should
not be read as identical evidence about network-configuration changes in another environment.

For readers, this keeps governance examples from sounding stronger than their environment supports.
Local, testchain, and mainchain contexts can all describe the proposal and approval process, but
they do not carry the same interpretive weight.
