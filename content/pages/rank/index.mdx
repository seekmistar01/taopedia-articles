# Rank

Rank is a Bittensor consensus metric for summarizing miner evaluation inside a subnet. It is the
aggregate miner-side judgment produced by Yuma Consensus from validator ratings
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The term belongs to the miner-evaluation side of consensus. It is not one validator's raw view and
not a general reputation score for every Bittensor activity.

## Consensus Role

Rank is the aggregate result after consensus compares validator weight signals and limits outlying
values. It answers where a miner landed in the subnet's evaluation result after filtering and
aggregation ([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

This makes rank an outcome term. Validator weights are inputs; rank is the consensus-adjusted
miner-side result built after those inputs are processed.

## From Weights to Rank

Validator weights are input signals, while rank is a post-aggregation output. A weight vector
represents one evaluator's signal about miners. Rank represents the consensus-adjusted result built
after those signals are compared, filtered, and aggregated
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

This separation keeps the vocabulary clear. Rank should not be read as a single validator's opinion.

## Rank, Trust, and Validator Trust

Rank, trust, and validator trust are related consensus terms, but they do not describe the same
thing. Rank names the miner-side aggregate result. Trust describes how much a miner-side outcome
changes after clipping, and validator trust describes how much of a validator's signal remains after
clipping ([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)).

They are complementary views of the same consensus process. Rank explains the miner-side result;
trust terms help explain clipping effects.

## Emission Context

Rank matters because Yuma Consensus uses aggregate miner evaluation when calculating miner
emissions. It is part of the path from validator rankings, stake weighting, and clipping into
emission calculation ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emissions](https://docs.learnbittensor.org/learn/emissions)).

Rank is not a free-standing reward promise. It is a miner-side output inside a process that also
considers validator stake, consensus clipping, and incentive flow.

## Subnet Context

Rank is meaningful inside the subnet and incentive context that produced it. Subnets define their
own work and evaluation roles, and multiple mechanisms can separate evaluation paths inside one
subnet ([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Multiple Incentive Mechanisms](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

For that reason, a rank value should not be treated as a universal quality score across unrelated
subnet tasks.

## Reader Boundary

Use rank when the focus is the final miner-side aggregate judgment. Use consensus score for the
clipping benchmark, validator weights for the inputs, and trust terms for how much clipping changed
the signals ([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

These boundaries keep the article focused on the concept that matters: rank explains the outcome
after validator evaluation and consensus filtering.
