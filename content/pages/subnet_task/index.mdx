# Subnet Task

A subnet task is the work target a Bittensor subnet asks miners to perform. It gives the subnet's
incentive design a concrete output to request and evaluate
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

The term belongs to subnet design vocabulary. It names the requested work, while protocol, scoring,
and consensus vocabulary describe other parts of how that work is exchanged, evaluated, and
connected to incentives
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

That makes the task the first concept to identify before reading a subnet's protocol, scoring model,
or incentive mechanism. The task says what miners are trying to produce; the surrounding mechanism
explains how that work is requested, measured, and rewarded.

## Requested Work

The task is the work a subnet is built to request. Subnet documentation describes subnets as markets
with their own goals and standards, while the glossary places the subnet task in the work miners
perform ([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task)).

This makes the task the starting point for interpreting the subnet's purpose. It identifies what
kind of useful output is being requested before the reader turns to protocol or scoring details.

The task can be broad or narrow depending on the subnet, but it stays tied to the work target. A
claim about a subnet task is clearest when it names the subnet and the work being requested
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## Miner and Validator Roles

Miners perform the task, and validators evaluate the returned work against the subnet's standards
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Those roles keep the task from becoming a generic project description. The task identifies the work
target; miner and validator terms identify who produces and evaluates work around that target.

That role split keeps the article centered on work rather than account details. Miners and
validators matter here because the task moves through production and evaluation before incentives
are assigned.

## Protocol Distinction

Subnet task and subnet protocol are related but separate ideas. The task names the requested work,
while the protocol names the interaction pattern used to request and return that work
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Subnet Protocol](https://docs.learnbittensor.org/resources/glossary#subnet-protocol)).

That distinction keeps content and exchange separate. A task can describe the desired output even
when the protocol details explain the request and response format around that output.

This is why task language is useful even when protocol details vary. The task remains the requested
work target, while the protocol explains how subnet components exchange that work
([Glossary: Subnet Protocol](https://docs.learnbittensor.org/resources/glossary#subnet-protocol)).

## Scoring Distinction

The subnet scoring model is adjacent but different. The task says what work is being requested,
while the scoring model describes how returned work is judged
([Glossary: Subnet Scoring Model](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

This distinction matters because a task alone does not say whether a particular output was useful.
The scoring model supplies the evaluation method that judges returned work against the task.

Task and scoring therefore sit in sequence. The task defines the work target first; the scoring
model evaluates whether returned work satisfies that target
([Glossary: Subnet Scoring Model](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model)).

## Incentive Mechanism

A subnet task is one part of a larger incentive mechanism. The incentive mechanism also includes the
protocol around the task and the evaluator method that connects performance to incentives
([Glossary: Incentive Mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

This keeps the task from carrying the full weight of the incentive system. A task names what is
requested; the incentive mechanism explains how requests, scoring, and incentives fit together.

That makes task language narrower than incentive-mechanism language. The task supplies the work
target; the mechanism supplies the surrounding evaluation and reward design.

## Multiple Mechanism Scope

When multiple incentive mechanisms operate inside a subnet, the mechanism matters for task
interpretation. Each mechanism can have its own evaluation system and bond-pool frame
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)).

That makes mechanism scope part of a task claim. A task described for one mechanism does not
automatically describe every mechanism inside the same subnet.

## Validators Encode Task Performance in Weight Vectors

[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) describes validators
submitting vectors that rank miner work they have evaluated against the subnet task.
[Validator weights](https://docs.learnbittensor.org/resources/glossary#weight-vector) therefore
carry task-performance scores rather than describing the task definition itself
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Subnet task names what miners should produce; validator weights name how returned work was judged
relative to that target.

## Consensus Turns Task Evaluations Into Incentives

Incentive-mechanism documentation places task design, protocol, and scoring inside subnet design,
then connects validator evaluations to emission outcomes through
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus). Miner
[incentives](https://docs.learnbittensor.org/resources/glossary#incentives) and validator
[dividends](https://docs.learnbittensor.org/resources/glossary#emission) sit downstream of the
task-evaluation path rather than inside the task label itself
([Emission](https://docs.learnbittensor.org/learn/emissions)).

That ordering keeps vocabulary precise. The task defines requested work; consensus and emissions
describe how evaluated work is rewarded after many validator signals merge.

## Multiple Mechanisms Can Carry Separate Tasks

When a subnet runs
[multiple incentive mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms),
each mechanism can maintain independent evaluation tracks with separate weight matrices and bond
pools
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

Task reading therefore needs mechanism context on those subnets. The same netuid can host more than
one work target depending on which incentive mechanism is being discussed.

## Distinction from Subnet Miner

Subnet task names the work a subnet expects participants to perform. Subnet miner names the role
that performs that work
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

Subnet-task vocabulary belongs to the requested work definition. Subnet-miner vocabulary belongs to
the performing role that produces that work.

## Distinction from Subnet Validator

Subnet task names the work a subnet expects participants to perform. Subnet validator names the role
that evaluates miner work inside the subnet
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Subnet-task vocabulary belongs to the requested work definition. Subnet-validator vocabulary belongs
to the evaluating role that scores that work.

## Distinction from Subnet Protocol

Subnet task names the work a mechanism asks miners to produce. Subnet protocol names the interaction
pattern for how validators request that work and miners return responses
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Subnet Protocol](https://docs.learnbittensor.org/resources/glossary#subnet-protocol),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Task vocabulary describes what work means; protocol vocabulary describes how that work is exchanged.

## Distinction from Subnet Scoring Model

Subnet task names the work a mechanism asks miners to produce. Subnet scoring model names how
validators evaluate responses against that work target
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Subnet Scoring Model](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Task vocabulary describes what work means; scoring vocabulary describes how returned work is judged.

## Relationship to Synapse

Subnet task and synapse are related but different subnet terms on the same work-exchange path.
Subnet task names the work a subnet expects miners to perform, while synapse names the structured
message object exchanged when validators request that work and miners return results
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[Understanding Neurons](https://docs.learnbittensor.org/learn/neurons#neuron-to-neuron-communication)).

For readers, subnet task belongs to requested-work vocabulary and synapse belongs to
communication-object vocabulary. Dendrite and axon vocabulary name the client and server components
that carry those objects during the exchange
([Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite),
[Glossary: Axon](https://docs.learnbittensor.org/resources/glossary#axon)).

Neuron communication documentation keeps work-target vocabulary separate from message-object
vocabulary so readers can tell which term applies to requested work and which applies to the carried
exchange schema on the subnet
([Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner)).

## Relationship to Subnet Scoring Model

Subnet task and subnet scoring model are related but different subnet design terms. Subnet task
names the work miners are asked to produce, while subnet scoring model names how returned work is
judged against that target inside the incentive mechanism
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Subnet Scoring Model](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model)).

For readers, subnet task belongs to requested-work vocabulary and subnet scoring model belongs to
evaluation-method vocabulary in the same subnet context
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

A subnet task statement does not by itself prove that a particular output was useful, because subnet
scoring model vocabulary names the evaluation method rather than the work target alone
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

Use subnet task when the article covers expected work and subnet scoring model when it covers how
returned work is judged on the subnet
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Distinction from Multiple Incentive Mechanisms

Subnet task names the work target inside a subnet. Multiple incentive mechanisms names the case
where one subnet runs parallel evaluation tracks with separate weight matrices and bond pools
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Subnet-task vocabulary belongs to the requested work definition. Multiple-incentive-mechanisms
vocabulary belongs to mechanism-scope labels that identify which parallel evaluation track is in
view.

## Relationship to Register

Subnet task and register are related but different subnet participation terms. Subnet task names the
work a subnet expects miners to perform, while register names the subnet-entry process that gives a
participant a UID slot after paying the dynamic registration burn
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register),
[Mining: Miner registration](https://docs.learnbittensor.org/miners#miner-registration)).

For readers, subnet task belongs to requested-work vocabulary and register belongs to subnet-entry
vocabulary. A participant must register before miner or validator roles can perform or evaluate the
task on a selected netuid
([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

Subnet documentation keeps work-target language separate from registration-process language so
readers can tell whether the article is describing expected work or the step that obtains a subnet
position
([Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner)).

## Relationship to UID Slot

Subnet task and UID slot are related but different subnet terms. Subnet task names the work target
inside a subnet, while UID slot names the participant position identified by a unique UID where
subnet miner or subnet validator roles operate
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

Readers should use subnet task when the article covers expected work and UID slot when it names
which participant position on a selected netuid is in scope
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner),
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator)).

On a selected netuid, task reading identifies what work the subnet requests, while UID slot reading
identifies which indexed participant is being discussed in miner or validator claims
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Mining: Miner registration](https://docs.learnbittensor.org/miners#miner-registration)).

## Subnet Task, Weights, and Emissions on netuid 1

On a subnet such as netuid 1, the
[subnet task](https://docs.learnbittensor.org/resources/glossary#subnet-task) defines the work
miners produce and validators evaluate
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Validators submit
[validator weights](https://docs.learnbittensor.org/resources/glossary#weight-vector) encoding those
evaluations, and [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) aggregates
the signals into miner [incentives](https://docs.learnbittensor.org/resources/glossary#incentives)
and validator [dividends](https://docs.learnbittensor.org/resources/glossary#emission)
([Emission](https://docs.learnbittensor.org/learn/emissions)). Subnets with
[multiple incentive mechanisms](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)
keep separate task and evaluation paths before those weight signals enter the shared consensus
layer.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For subnet task on
a subnet such as netuid 1, that sequence changes how readers should interpret requested-work
examples.

In localnet, subnet-task examples on netuid 1 can be tested in isolation. Local miner outputs and
validator evaluations reflect local chain configuration rather than production subnet task design.

On testnet, a subnet task on netuid 1 can be observed in a shared non-production network. Testnet
task descriptions are separate from mainnet subnet task state
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task)).

On mainnet, a subnet task on netuid 1 describes live requested work on the production Bittensor
network for the subnet or mechanism being discussed.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A subnet-task example from one environment should not be
read as representing production task design on another network.

## The Task Defines What "Good Work" Means for a Subnet

A subnet task is more than a label for activity; documentation frames it as the thing that gives
validator evaluation something concrete to measure. Without a defined task, there is no stable basis
for comparing miner outputs, so the task is what makes scoring meaningful in the first place
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

For readers, this is why the task sits upstream of weights and emissions. Validators encode their
judgment of task performance into weight vectors, and consensus turns those into incentives, so a
poorly defined task weakens every step that follows
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

That dependency also explains why tasks differ so much across subnets. Each subnet defines its own
task to match the useful work it targets, which is why a strong result on one subnet's task does not
translate into another's
([Glossary: Subnet Scoring Model](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model)).

## Distinction from Rank

A subnet task and rank describe a miner's path as work and result. The subnet task is the specific
work a subnet defines for miners to perform and be evaluated on, while rank is the consensus result
that orders miners after validators' clipped weights are merged, so task performance feeds the
weights that determine rank
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Distinction from Metagraph

Subnet task and metagraph are related but different subnet terms. Subnet task names the work target
inside a subnet, while metagraph names the subnet state snapshot for a selected netuid, including
participants and their subnet-level values
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Metagraph](https://docs.learnbittensor.org/resources/glossary#metagraph),
[The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

Readers should use subnet task when the article covers expected work and metagraph when it reads
subnet participant state for a named netuid rather than the work definition alone
([Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

- **Subnet task** — the work miners are asked to produce inside a subnet.
- **Metagraph** — the subnet state snapshot for a selected netuid at one block height.

## Distinction from Epoch

Subnet task and epoch are related but different subnet terms. Subnet task names the work miners are
asked to produce inside a subnet, while epoch names the consensus period in which validator signals
for that subnet are processed
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Readers should use subnet task when the article covers expected work and epoch when it covers the
timing window in which subnet consensus groups validator evaluations on a selected netuid
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

- **Subnet task** — the work miners are asked to produce inside a subnet.
- **Epoch** — the consensus period in which validator signals for that subnet are processed.

## Distinction from Tempo

Subnet task names the work a subnet expects miners to perform. Tempo names the block-based cadence
that groups periodic subnet processes such as weight processing and reward distribution
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

The [Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo) entry defines tempo
as a subnet hyperparameter measured in blocks, while the subnet task label stays on the requested
work evaluated inside that subnet
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Work-target vocabulary and timing vocabulary answer different reader questions. Use subnet task when
naming expected work on a netuid; use tempo when naming how many blocks pass between periodic subnet
processing steps on that netuid
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

- **Subnet task** — the work miners are asked to produce inside a subnet.
- **Tempo** — the block cadence for periodic subnet processing on that netuid.

## Distinction from Deregistration

Subnet task and deregistration are related but different subnet terms. Subnet task names the work
miners are asked to produce, while deregistration names the removal of a low-performing subnet miner
or subnet validator when capacity and replacement rules apply
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Deregistration](https://docs.learnbittensor.org/resources/glossary#deregistration),
[Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)).

Readers should use subnet task when the article covers expected work and deregistration when it
covers loss of a subnet position on a selected netuid rather than the work definition itself
([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

- **Subnet task** — the work miners are asked to produce inside a subnet.
- **Deregistration** — removal of a subnet miner or validator when replacement rules apply.

## Distinction from Weight Setting

Subnet task names the work miners are asked to produce inside a subnet. Weight setting names the
validator process of assigning scores to that work before consensus runs
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Subnet-task vocabulary belongs to the work target miners return. Weight-setting vocabulary belongs
to the validator scoring step applied to that output
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator)).

## Reader Boundary

Subnet task is requested-work terminology. It is not the scoring model, the protocol, the consensus
result, or a claim that every returned output is useful
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Subnet Scoring Model](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

A subnet task identifies the work miners are asked to produce inside a subnet or mechanism.
