# Bittensor Logging Levels

Bittensor logging levels are severity labels for diagnostic messages. They help readers distinguish
fine-grained trace output, normal informational messages, successful events, warnings, errors, and
critical failures. The [Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels) page
lists the Bittensor level names and their severity order.

## Severity ladder

The core idea is that lower-numbered levels are more verbose and higher-numbered levels are more
severe. `TRACE` and `DEBUG` are useful for detailed diagnostic context, `INFO` and `SUCCESS` mark
normal or successful events, `WARNING` calls attention to possible problems, and `ERROR` or
`CRITICAL` indicate serious failure conditions. The
[Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels) page gives the level names,
numeric values, and short meanings.

## Bittensor-specific levels

Bittensor adds two custom levels to the familiar logging ladder: `TRACE` and `SUCCESS`. `TRACE`
represents very fine-grained diagnostic output, while `SUCCESS` marks an event that completed
successfully. These levels make logs easier to scan without changing the basic idea that the level
describes message severity. The
[Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels) page identifies both custom
levels.

## Diagnostic context

A log level is a clue about message importance, not a complete explanation of what happened. For
example, an error-level message can point a reader toward a failure area, but the surrounding
message, interface, and source context still matter. Bittensor's
[Subtensor Error Codes](https://docs.learnbittensor.org/errors) overview makes the same distinction
for chain-layer errors by grouping failures into standard, custom, and inherited error categories.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from local testing to testchain and then mainchain. Logging
levels can appear at each stage, but the environment determines how much weight a log message should
carry.

In local testing, `TRACE`, `DEBUG`, `INFO`, or `SUCCESS` output can help a developer understand what
a local process attempted inside an isolated environment. That is useful diagnostic context, but it
should not be read as proof that the same event happened on a public Bittensor network.

On testchain, logs can accompany shared testing-network activity. This gives stronger context than a
private local run, while still keeping the message separate from production mainchain behavior and
from canonical chain records.

On mainchain, logs can help interpret production software behavior, but they should be tied to the
network, role, process, account or subnet context, and relevant block or transaction evidence. A log
line alone is not the same as a verified on-chain result.

The [Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels) page defines severity
labels, while the [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)
reference separates mainnet, testnet, and localnet contexts. Reading those sources together keeps
log severity separate from network evidence.

This distinction matters because a logging level describes message importance, not final truth. A
`SUCCESS` message may be useful in context, and an `ERROR` message may identify a failure path, but
the claim being made still needs the right chain, tool, or protocol evidence.

For readers, this keeps log examples from sounding stronger than their environment supports. Local,
testchain, and mainchain contexts can all contain Bittensor logs, but they do not carry the same
interpretive weight.

## Boundary with error codes

Logging levels and Subtensor error codes answer different questions. A logging level describes how
severe or verbose a message is. An error code identifies a rule family or failure category returned
by the Subtensor layer. Reading them together can be useful, but a log level alone should not be
treated as the full source of truth for a failed network action. The
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor) reference shows how
specific chain-layer failure names sit behind short error messages.
