# Subtensor Constants

Subtensor constants are named runtime values exposed by Bittensor's Subtensor runtime. They are
grouped by runtime area and help readers identify values such as defaults, bounds, limits, timing
parameters, and feature-status flags
([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

The term is about named values built into the runtime. It should be read separately from storage
entries, events, and submitted runtime actions.

## What Constants Show

Each constant entry gives readers a runtime area, entry name, type, lookup shape, and summary for a
named value ([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

That structure keeps constants from being read as isolated numbers. The name identifies the value,
the type describes the value shape, and the runtime area explains where the value belongs.

Constants are useful when a reader needs to know what the runtime calls a value or where that value
is grouped. They do not by themselves explain every feature that uses the value.

## Lookup Shape

The Subtensor constants page identifies each entry through the `api.consts.<Pallet>.<constant_name>`
shape ([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)). That shape
keeps the runtime area and constant name attached in one lookup label.

This matters for readers because the same short word can be ambiguous without the pallet context.
For example, a balance-related constant belongs under the balances area, while a Bittensor-specific
constant can belong under the Subtensor module area.

The constants page also states that the values are read from the connected node. Chain-inspection
documentation places constants on the read-only side of Bittensor tooling, so constants stay tied to
the runtime and network context that exposed them
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

## Runtime Areas

Constants are grouped by runtime area. Some areas describe general chain behavior, while the
Subtensor area contains Bittensor-specific values connected to subnet, rate-limit, weight, and
feature-status vocabulary
([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

The runtime area is part of the meaning of the entry. Similar-looking values can have different
roles when they belong to different areas, so the area should stay attached when the value is
discussed.

For readers, the area answers where the constant belongs before the entry name and type narrow what
the value means.

## Constant Examples

The constants listing includes general runtime areas and Bittensor-specific areas. Examples include
`aura`, `balances`, `safeMode`, `subtensorModule`, `system`, and `transactionPayment`
([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

Some constants are general chain reference values. The constants page lists
`api.consts.balances.existentialDeposit` under the balances area, which keeps the value tied to
balance-accounting context rather than to Subtensor-specific subnet behavior.

Other constants sit in the Subtensor module area. The constants page includes Subtensor-module
constants such as `alphaHigh`, `alphaLow`, and `hotkeySwapOnSubnetInterval`, with each entry keeping
the lookup label, type, and summary together
([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

These examples make the reading pattern concrete: first identify the runtime area, then the constant
name, then the type and summary. The value is easier to interpret when those parts stay together.

## Constants and Features

A constant can name a value used by a feature, but it usually does not replace the feature
explanation. For example, a rate-limit-related value can help name a bound or default, while the
rate-limit documentation explains the block-based cooldown model
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

This distinction is useful for Bittensor-specific topics. A constant may identify a default, limit,
or feature flag; the related concept page explains how that value fits into protocol behavior.

When an article discusses a rate limit, the constant can help identify the runtime value, while the
rate-limit page explains the reader-facing behavior
([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

## Crowdloan Constants Bound Campaigns

Beyond subnet examples, the constants page lists a `crowdloan` runtime area with deposit, duration,
and refund-batch limits for collective funding campaigns
([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)). Those values name
rules such as the creator deposit required to open a campaign, the minimum and maximum block
duration a campaign may span, and how many contributors can be refunded in one refund pass.

Crowdloan guidance ties those bounds to real campaign design. Campaign duration must stay within the
minimum and maximum window on production-style networks, where the documented range maps to about
seven days at the low end and about sixty days at the high end
([Crowdloans](https://docs.learnbittensor.org/subnets/crowdloans),
[Create a Subnet with a Crowdloan](https://docs.learnbittensor.org/subnets/crowdloans/crowdloans-tutorial)).

Readers therefore meet crowdloan limits in two related places. Constants expose the runtime
reference values, while crowdloan documentation explains how contributors fund an action such as
subnet registration and how failed campaigns return deposits
([Crowdloans](https://docs.learnbittensor.org/subnets/crowdloans)).

## Slot Duration Anchors Block Time

Many Bittensor limits are expressed in blocks rather than in seconds. The constants page lists
`api.consts.aura.slotDuration` with a twelve-thousand-millisecond summary for how long an Aura slot
should run ([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)). That
entry sits in the general `aura` area rather than in the Subtensor-module bucket, because block
timing is shared chain infrastructure.

Rate-limit documentation describes cooldowns as block-based waits after a successful operation
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)). When a reader wants wall-
clock intuition, the slot-duration constant provides the millisecond-per-slot anchor that sits under
those block counts.

The connection is interpretive, not interchangeable. A rate-limit page explains the behavior and
error pattern, while the Aura constant names the slot timing reference that helps turn a block count
into an approximate time estimate.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Subtensor
constants on a subnet such as netuid 1, that sequence changes how an example should be read, because
the surrounding network state differs at each stage.

In localnet, Subtensor constants on netuid 1 can be inspected in an isolated development
environment. Localnet constant reads and hyperparameter examples reflect that test chain rather than
production subnet history.

On testnet, constants and hyperparameters on netuid 1 apply in a shared non-production network whose
runtime values and subnet state are separate from mainnet.

On mainnet, Subtensor constants on netuid 1 concern the live production runtime and the subnet
hyperparameters that govern tempo, participation limits, and related bounds on the selected subnet
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A constant or hyperparameter example from one environment
should not be read as representing another environment.

## Distinction from Subnet Hyperparameters

A Subtensor constant and a subnet hyperparameter name a runtime default and a live per-subnet value.
`initial` constants such as `initialTempo` name compile-time startup defaults, while subnet
hyperparameters show the tempo, UID cap, and immunity values a subnet actually uses after creation
([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Subtensor constant** — a compile-time runtime default or bound.
- **Subnet hyperparameter** — the live per-subnet value used after creation.

On a subnet such as netuid 1, `initial` Subtensor-module constants such as `initialTempo`,
`initialMaxAllowedUids`, and `initialImmunityPeriod` name compile-time startup defaults the runtime
uses when a parameter is first set, while live
[subnet hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) show the
tempo, UID cap, and immunity settings that subnet uses after creation
([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)). The
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo) entry describes tempo as
a subnet setting measured in blocks, which readers usually read from hyperparameter context rather
than from a one-time `initial` constant alone, so an `initial` constant should not be mistaken for
live subnet state
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

Those timing and participation values sit upstream of the subnet incentive loop. Validators submit
[validator weights](https://docs.learnbittensor.org/resources/glossary#weight-vector) on the
subnet's tempo schedule, and [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)
turns those signals into miner incentives and validator dividends
([Emission](https://docs.learnbittensor.org/learn/emissions)). Constants therefore help name runtime
defaults and bounds, while hyperparameters and consensus documentation explain how those values
participate in live subnet behavior.

## Distinction from Subtensor Storage

Subtensor constants expose fixed protocol parameter values in the API. Subtensor storage queries
return the current on-chain value stored at a key
([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Inspecting the Chain](https://docs.learnbittensor.org/concepts/inspecting-the-chain)). Constants
answer what the protocol defines; storage answers what the chain currently holds.

- **Subtensor constants** — fixed protocol parameter values in the API.
- **Subtensor storage** — current on-chain values at queried keys.

Constants, storage entries, and events answer different questions. Constants expose runtime-defined
values, storage entries expose state surfaces, and events name runtime records emitted during
processing ([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

A feature can involve all three surfaces without making them interchangeable. A constant can name a
value used by a feature, a storage entry can locate chain state, and an event can show that runtime
activity happened.

The distinction keeps readers from treating a constant as mutable state or as a record of an action.
A constant can name a value, storage can show a state surface, and an event can show runtime
activity ([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

## Distinction from Subtensor Events

A Subtensor constant and a Subtensor event sit on the value-definition side and the activity-record
side. A constant exposes a named runtime value, while an event names a runtime record emitted during
processing ([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

- **Subtensor constant** — a named runtime value.
- **Subtensor event** — a record emitted when runtime activity happens.

## Distinction from Yuma Consensus

Subtensor constants are fixed protocol values defined in the Subtensor runtime configuration. They
name hard-coded limits and parameters the chain enforces, not the recurring settlement step that
converts validator weight submissions into emission shares on one netuid
([Navigating Subtensor](https://docs.learnbittensor.org/navigating-subtensor),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

Yuma Consensus is the on-chain mechanism that runs at the epoch boundary on each subnet. It reads
the weight matrix from eligible validators, applies clipping and bonding under those runtime rules,
and converts the result into miner incentives and validator dividend shares
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Earlier sections separate subtensor constants from hyperparameters and from runtime storage keys.
Constants constrain how Yuma executes; Yuma settles evaluation signals from eligible validators on
that netuid—not the constant definition table itself
([Glossary: Hyperparameters](https://docs.learnbittensor.org/resources/glossary#hyperparameters)).

Readers should treat subtensor constants as runtime configuration vocabulary, and Yuma as the
epoch-boundary mechanism that allocates subnet rewards from consensus while validators remain
active.

- **Subtensor constants** — fixed protocol values in the runtime.
- **Yuma Consensus** — epoch-boundary settlement from eligible validator weights to emission shares.
  Subtensor constants name fixed runtime values; Yuma processes validator weights into rewards
  ([Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions)).

## Distinction from Inspection

Chain-inspection documentation includes constants among the chain data readers can inspect
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).
This places constants on the read-only side of Bittensor tooling: they can be inspected without
submitting a state-changing runtime action
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

Subtensor constants also belong to a runtime and network context. Mainnet, testnet, and localnet are
separate environments, so examples should keep the environment label attached when behavior across
networks matters ([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

## Distinction from Network

## Reader Boundary

Subtensor constants are useful for naming defaults, bounds, limits, timing values, and
feature-status values, but they are not a complete guide to the feature that uses them
([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

When a concrete question depends on behavior, the constant should be read beside the relevant
concept page and any related storage or event context.
