# UID

A UID is the unique identifier assigned to each participant position in a Bittensor subnet. Each
miner or validator registered to a subnet holds a UID that identifies their position within that
subnet's participant set
([Glossary: UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

UID is identifier vocabulary. It names the number, not the position structure itself. UID slot is
the related term for the participant position that the UID identifies
([Glossary: UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot)).

## What It Represents

UIDs are subnet-specific. The same UID number can appear in different subnets while referring to
different participants. Netuid identifies the subnet itself, while UID identifies a participant
within that subnet ([Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

On netuid 1, a UID identifies a participant position in that subnet's metagraph. It should not be
read as identifying a participant on another network
([Glossary: Metagraph](https://docs.learnbittensor.org/resources/glossary#metagraph),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

## Distinction from Hotkey

A UID names a subnet-local slot number. A hotkey names the account key that registers to and signs
for that subnet ([Glossary: UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Glossary: Hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey)).

- **Hotkey** — account identity that registers and signs on the subnet.
- **UID** — numbered slot that identity occupies on the selected subnet.

A hotkey persists across subnets while the UID is the positional index a subnet assigns at
registration, so one hotkey can hold different UIDs in different subnets
([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets)).

## Distinction from Register

Register names the action that places a participant into a subnet. A UID is the slot number assigned
by that registration
([Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register),
[Glossary: UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot)).

- **Register** — registration action that places a participant on a subnet.
- **UID** — numbered position assigned by that registration.

## Distinction from Metagraph

The metagraph records each subnet's participant set. A UID is the index label for one miner or
validator row in that subnet state
([Glossary: Metagraph](https://docs.learnbittensor.org/resources/glossary#metagraph),
[Glossary: UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot)).

- **Metagraph** — full participant table for one subnet.
- **UID** — index label for one row inside that table.

## Distinction from Weight Vector

Validators score miners by UID inside a subnet. A weight vector attaches those scores to the UID
positions listed in the subnet metagraph
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **UID** — participant slot that weight entries reference.
- **Weight vector** — scored targets attached to those slots in one submission.

## Distinction from Immunity Period

A UID names a participant position in a subnet. Immunity period is the block window that shields a
newly assigned UID from deregistration
([Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period),
[Glossary: UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **UID** — numbered participant position in a subnet.
- **Immunity period** — shield window for a newly assigned UID.

## Distinction from Netuid

A netuid identifies a whole subnet, while a UID identifies one participant's position inside a
subnet. The same UID number can exist on many subnets, so a UID only pins down a participant when
read together with its netuid
([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid)).

- **Netuid** — identifier of a whole subnet.
- **UID** — identifier of one participant position within a subnet.

## Distinction from Neuron

UID names the numbered slot identifier inside a subnet. Neuron names the registered participant
record tied to that slot in subnet state
([Glossary: UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Glossary: Neuron](https://docs.learnbittensor.org/resources/glossary#neuron)).

- **UID** — subnet-local slot number for a participant position.
- **Neuron** — registered participant record tied to that slot.

## Distinction from Yuma Consensus

UID is the subnet-local slot number that identifies one participant position on a netuid. It names
addressing vocabulary in the metagraph and in weight vectors, not the settlement step that converts
those indexed signals into emission shares
([Glossary: UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

Yuma Consensus is the on-chain mechanism that runs at the epoch boundary on the selected netuid. It
reads the weight matrix from eligible validators—each row keyed by validator UID and each column
targeting miner UIDs—applies clipping and bonding, and converts the filtered result into incentive
and dividend shares ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Earlier sections separate UID from netuid and from the neuron record tied to a slot. A UID tells
readers which row or column is meant in subnet state; it does not by itself prove rank, trust, or
payout amount after consensus settles.

Readers should treat UID as participant slot addressing on one subnet, and Yuma as the mechanism
that settles the UID-indexed weight matrix into emissions
([Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions)).

- **UID** — subnet-local slot number for one participant position.
- **Yuma Consensus** — epoch-boundary settlement from the UID-indexed weight matrix to emission
  shares.

## Reader Boundary

UID is concept vocabulary for subnet participant identification. It should not be read as a network
address, a balance container, or a cross-subnet identity
([Glossary: UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot)).
