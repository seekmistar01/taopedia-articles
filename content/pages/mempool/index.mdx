# Mempool

A mempool is a temporary holding area for pending blockchain transactions. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#mempool) defines it as the place where
pending and unconfirmed transactions sit before being included in a block.

## Pending Transaction Context

The mempool belongs to the time between transaction submission and block inclusion. A transaction in
the mempool has not yet become part of a block, so readers should treat the term as describing a
pending state rather than final chain history.

## Relationship to Blocks

The glossary defines a [block](https://docs.learnbittensor.org/resources/glossary#block) as a unit
of blockchain data containing transactions. The mempool is the waiting area before that inclusion
happens. Once a transaction is included in a block, it is no longer merely pending in the mempool.

## Relationship to Mempool Visibility

The mempool holds pending transactions before block inclusion. Mempool visibility names whether
details of those pending transactions are exposed to observers during that wait. The
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) defines the mempool
as the place where pending and unconfirmed transactions sit.

For readers, mempool is the pending-state container, while mempool visibility is the exposure
question about what can be seen before inclusion. A transaction can sit in the mempool without every
detail being equally visible.

References: [Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)

## Relationship to Extrinsics

Extrinsics are the state-changing chain actions that pass through the mempool. The official
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) reference describes
extrinsics as submitted actions that ask the Subtensor runtime to perform an operation, and the
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) describes the
mempool as the place where pending and unconfirmed transactions sit before being included in a
block.

For readers, this places extrinsics on the submission side of the mempool boundary. An extrinsic
that has been submitted but not yet included in a block belongs to the pending-transaction state the
mempool names; once the runtime accepts it into a block, it moves past the mempool stage.

References: [Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)

## Relationship to Transaction Fees

Pending mempool transactions can carry fees. The official
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) page describes fees as
costs the chain attaches to state-changing actions, with a base transaction fee made up of weight
and length components. The
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) places pending
transactions in the mempool before block inclusion, where those fee-bearing actions wait alongside
other unconfirmed submissions.

For readers, the connection is between the fee model and the pending-state concept. The mempool does
not itself set fees; it names the holding area for submitted actions whose fee accounting is
evaluated when the runtime processes them into a block.

References: [Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)

## Relationship to Network Context

A mempool is read inside a specific Bittensor network environment. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet as deployment contexts, and a pending transaction belongs
to whichever network environment it was submitted into.

For readers, this keeps mempool state network-scoped rather than network-agnostic. A mainnet
mempool, a testnet mempool, and a localnet mempool all describe the same pending-transaction
concept, but each one is read against the network context it belongs to.

References: [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from local testing to testchain and then mainchain. Mempool
examples can appear at each stage, but the network determines what the pending transaction can
prove.

In local testing, a mempool example can show a submitted transaction waiting inside an isolated
local blockchain. That is useful for understanding pending state and block inclusion timing, but it
should not be read as evidence about public Bittensor network activity.

On testchain, mempool behavior can be observed in a shared testing network. This gives stronger
evidence about public submission and inclusion behavior than a private local run, while still
keeping the result separate from production mainchain activity.

On mainchain, mempool context belongs to live production transaction flow. A pending transaction
claim should keep the selected network, submitted extrinsic or transaction, visibility context, and
eventual block inclusion status attached.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, and the
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) defines the mempool
as the place where pending and unconfirmed transactions sit before block inclusion. Reading those
together keeps the pending state tied to its environment.

This distinction matters because mempool evidence is temporary. A transaction can be visible as
pending before it is included, rejected, replaced, or otherwise absent from final chain history, so
the stage and block context should remain attached.

For readers, this keeps mempool examples from sounding stronger than their environment supports.
Local, testchain, and mainchain contexts can all include pending transactions, but they do not carry
the same interpretive weight.

## Relationship to Subtensor

The mempool and Subtensor are related but different parts of Bittensor chain vocabulary. Subtensor
names Bittensor's layer 1 blockchain and system of record, while the mempool names the temporary
area where pending transactions wait before block inclusion on that chain. The
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor) describes the
blockchain layer, and the
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) describes the
pending-transaction holding area before inclusion.

For readers, Subtensor names the chain layer, while the mempool names the pre-inclusion waiting area
on that layer.

References: [Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)

## Relationship to Subtensor Events

The mempool and Subtensor events are related but different parts of Bittensor chain vocabulary. The
mempool names pending transactions before block inclusion, while Subtensor events name on-chain
notifications emitted after runtime actions are processed into blocks. The
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) describes the
pending-transaction context, and the
[Glossary: Subtensor Events](https://docs.learnbittensor.org/resources/glossary#subtensor-events)
describes event records emitted by the chain.

For readers, the mempool belongs to the pre-inclusion waiting stage, while Subtensor events belong
to the post-processing notification stage after inclusion.

References: [Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Subtensor Events](https://docs.learnbittensor.org/resources/glossary#subtensor-events)

## Relationship to Subtensor Storage

The mempool and Subtensor storage are related but different parts of Bittensor chain vocabulary. The
mempool names pending transactions before block inclusion, while Subtensor storage names the
on-chain data the runtime keeps after those transactions are processed into blocks. The
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) describes the
pending-transaction context, and the
[Glossary: Subtensor Storage](https://docs.learnbittensor.org/resources/glossary#subtensor-storage)
describes on-chain storage maintained by the runtime.

For readers, the mempool belongs to the pre-inclusion waiting stage, while Subtensor storage belongs
to the persisted on-chain state after inclusion.

References: [Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Subtensor Storage](https://docs.learnbittensor.org/resources/glossary#subtensor-storage)

## Relationship to MEV Shield

The mempool and MEV Shield are related but name opposite sides of pre-inclusion transaction
handling. The [Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)
describes the pending-transaction holding area, while
[MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield/) encrypts sensitive transaction
contents during that wait before block inclusion.

For readers, the mempool names the waiting area, while MEV Shield names one official response to
sensitive-detail exposure before inclusion.

References: [Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield/)

## Reader Boundary

This article defines the term at a high level. It does not describe how to submit transactions,
monitor pending activity, or inspect current network state. Readers should use official transaction
and chain references for those details.
