# Block

In Bittensor, a block is an ordered unit of blockchain data that contains transactions and has a
unique block hash ([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

The term belongs to chain-record vocabulary. It should be read separately from pending submission,
event interpretation, and storage-state queries.

## Chain Record Context

The official glossary defines a block as a data unit containing transactions and a unique block hash
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

That makes a block useful evidence for inclusion and ordering. It can anchor a chain-visible record
without explaining every cause or later consequence connected to that record.

## Inclusion Versus Submission

Submission and inclusion are different reading steps. The mempool glossary entry describes the
pre-inclusion waiting area for pending transactions, while a block is the chain record after
inclusion ([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

This boundary keeps block vocabulary precise. A submitted action is not the same thing as an
included record until chain evidence supports that reading.

## Events and Storage Context

Blocks sit near other Subtensor evidence surfaces. Events can show named runtime records emitted
during processing, while storage entries can show queryable chain state
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

A block gives included records a place in chain history. It does not make events, storage values,
and submitted calls interchangeable evidence.

## Extrinsic Context

The Subtensor extrinsics reference describes submitted runtime methods
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

Block evidence can show that an extrinsic appeared in chain history, but the meaning of the result
may depend on associated events, resulting state, and the runtime rule being invoked.

## Network Context

Bittensor documentation separates mainnet, testnet, and localnet contexts
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

A block should therefore be read inside the network where it was produced. A block from one
environment is not evidence of the same chain state in another environment.

## Development Context

Bittensor documentation describes subnet development as moving through localnet, testnet, and
mainnet contexts
([Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)).

Localnet blocks can demonstrate local chain behavior, testnet blocks belong to a shared testing
network, and mainnet blocks belong to production chain history. The block concept stays the same,
but the evidence scope changes with the environment.

## Reader Boundary

The block glossary entry supports a narrow claim: a block is a chain data unit that contains
transactions and has a block hash
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

That boundary keeps block references useful. They can anchor chain-visible facts without becoming a
substitute for transaction analysis, runtime documentation, or network-specific context.
