# Bonds Penalty

Bonds penalty is a per-subnet hyperparameter, listed in the documentation as `BondsPenalty`, that
sets the magnitude of the penalty applied when validator weights exceed the subnet consensus
benchmark in [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus). The
hyperparameter reference describes it as the magnitude of the penalty subtracted from weights for
exceeding consensus and points readers to the out-of-consensus bond penalty section of the Yuma
Consensus documentation
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## What It Controls

The hyperparameter configures how strongly out-of-consensus
[validator weights](https://docs.learnbittensor.org/resources/glossary#weight-vector) are pulled
toward the consensus benchmark before bond updates are computed. It does not define the subnet task,
which miners a validator should score, or how
[clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping) removes weights above the
stake-backed benchmark.

Yuma Consensus documentation states that when a submitted weight exceeds a miner's consensus
evaluation, its bond value is penalized by a configurable factor. Bonds penalty is the subnet
hyperparameter that names that penalty magnitude for the selected subnet
([Yuma Consensus: Penalizing out-of-consensus bonds](https://docs.learnbittensor.org/learn/yuma-consensus#penalizing-out-of-consensus-bonds)).

The documentation expresses the adjusted bond-weight as a blend between the submitted weight and the
consensus benchmark, with the penalty factor controlling how far the result is pulled toward
consensus. Bonds penalty therefore belongs to the validator-side bond path, not to miner incentive
ranking alone ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Documented Type, Default, and Setter

The [Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)
reference lists `BondsPenalty` as a `u16` with a default of `0`. It is changed through the
`sudo_set_bonds_penalty` extrinsic, and the reference marks the permission required to set it as
subnet owner. The hyperparameter table lists no `btcli sudo set` helper for this field
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

Because each subnet carries its own hyperparameters, the penalty magnitude is defined per subnet and
applies only within that subnet's Yuma Consensus path.

## Distinction from Clipping

[Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping) removes validator weights
above the stake-backed consensus benchmark so neither the miner nor the validator receives emissions
for the excess portion. Bonds penalty adjusts bond-weight calculation when a validator's submitted
weight exceeds consensus, before the smoothed
[validator-miner bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)
are updated ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Clipping** — caps excess weight so emissions are not paid on the clipped portion.
- **Bonds penalty** — pulls bond-weight toward consensus when a weight exceeds the benchmark.

## Distinction from Kappa

[Kappa](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#kappa) sets how much
stake-weighted validator support a weight level must reach before it becomes the clipping ceiling
during Yuma Consensus. Bonds penalty names the out-of-consensus bond penalty magnitude, not the
stake fraction used to compute the clipping benchmark
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## BTCLI Reads the Live Value

Any user can read subnet hyperparameters for a selected netuid with
`btcli subnet hyperparameters --netuid 1`. The command output includes the live `BondsPenalty` value
for that subnet at query time
([Subnet Hyperparameters: View hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#view-hyperparameters)).

Live values are network- and subnet-specific and can differ from the documented default, so a
penalty claim should be checked against the hyperparameter table for the relevant netuid and network
rather than assumed from the reference default alone.

## Why the Default Leaves Bonds Unpenalized

The documented default of zero is worth reading as a deliberate starting point, not just an empty
value. With the penalty magnitude at zero, the blend between a submitted weight and the consensus
benchmark leaves the submitted weight effectively unchanged, so out-of-consensus weights are not
pulled toward the benchmark on the bond path
([Yuma Consensus: Penalizing out-of-consensus bonds](https://docs.learnbittensor.org/learn/yuma-consensus#penalizing-out-of-consensus-bonds)).

For readers, this means bonds penalty is opt-in per subnet. A subnet that wants to discourage
validators from over-rating miners beyond consensus raises the value, while a subnet leaving the
default in place is choosing not to apply this particular bond-path pressure
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

That opt-in nature is why bonds penalty and clipping should not be read as the same safeguard.
Clipping always caps emissions on excess weight, whereas bonds penalty only shapes bond updates when
a subnet has chosen a non-zero magnitude
([Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping)).
