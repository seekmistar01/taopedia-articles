# Liquidity Positions

Liquidity positions are range-based liquidity commitments for Bittensor subnet pools. The official
[User Liquidity Positions](https://docs.learnbittensor.org/liquidity-positions) reference describes
supplying liquidity for a specific subnet across a selected range.

The concept belongs to subnet tokenomics. It is about supplying depth to a TAO-and-alpha pool, not
about mining, validation, or subnet work evaluation.

## Subnet Pool Context

Bittensor subnets use pools with TAO and subnet alpha sides. The
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets#liquidity-pools)
reference places liquidity pools inside the Dynamic TAO subnet model.

A liquidity position is a contribution inside that pool context. It adds range-based liquidity to a
subnet pool rather than replacing the pool's reserve structure.

## Range Context

Liquidity positions use concentrated liquidity. The
[User Liquidity Positions](https://docs.learnbittensor.org/liquidity-positions) reference describes
positions as being supplied over a selected range.

That makes the term range-sensitive. A liquidity position is not just a generic balance in a pool;
it is liquidity supplied under a range condition.

## Staking Contrast

Liquidity positions and staking answer different questions. The
[Liquidity Positions vs. Staking](https://docs.learnbittensor.org/liquidity-positions#liquidity-positions-vs-staking)
section distinguishes liquidity provision from staking.

Staking supports validator-related subnet participation. A liquidity position supports pool depth
for TAO-and-alpha subnet liquidity. Both can involve TAO and alpha, but they are different concepts.

## Slippage Context

Liquidity positions are related to slippage because both concern available pool depth. The
[Slippage](https://docs.learnbittensor.org/learn/slippage) reference explains how pool depth affects
swap outcomes.

A liquidity position is supplied depth. Slippage is an effect measured against available depth. The
two concepts are connected, but one should not be used as a synonym for the other.

## Fee Context

The [User Liquidity Positions](https://docs.learnbittensor.org/liquidity-positions) reference
describes fee context around subnet pool activity inside the selected range.

That boundary matters for reference prose. The article can describe fee context without promising an
outcome for any specific subnet, range, or amount of supplied liquidity.

## Reserve Boundary

Liquidity positions sit around the subnet reserve pair. TAO reserve names the pool-held TAO side,
and alpha reserve names the paired subnet-alpha side
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets#liquidity-pools),
[Emission: Alpha reserve injection](https://docs.learnbittensor.org/learn/emissions#alpha-reserve-injection)).

That reserve pair is the pool context. A liquidity position adds range-based depth around the pool;
it does not rename either reserve side.

## Environment Boundary

Liquidity-position examples need their network and subnet context. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, while liquidity-position documentation places the feature
inside subnet pools.

An example from one environment should not be read as the same kind of evidence for another
environment. The concept is stable, but concrete pool depth, fee context, and range behavior depend
on the source context.

## Reader Boundary

Liquidity positions are range-based liquidity commitments inside subnet pools
([User Liquidity Positions](https://docs.learnbittensor.org/liquidity-positions)). They are not an
earnings promise, staking guide, subnet quality signal, or operator procedure.

When the focus is validator support, staking is the more precise concept. When the focus is pool
depth and range-based liquidity, liquidity position is the more precise term.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Liquidity
Positions, that sequence changes how readers should interpret position examples, pool observations,
and subnet-liquidity comparisons.

In localnet, Liquidity Positions examples can be exercised in an isolated environment. Localnet
observations reflect local chain state and local configuration rather than production Bittensor
behavior.

On testnet, Liquidity Positions behavior can be observed in a shared, non-production network.
Testnet examples are useful for checking interactions under shared chain conditions while staying
separate from mainnet state.

On mainnet, Liquidity Positions describes live production liquidity behavior on the production
Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. An example or outcome from one environment should not be
read as evidence of production behavior in another environment.
