# Transaction Fees

Transaction fees are costs the Bittensor chain can attach to state-changing actions. They are
separate from free chain reads.

The official [Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) page
separates base chain transaction fees from swap fees that can apply when stake-related activity
interacts with subnet liquidity. It also states that reading chain state is free.

## Base Chain Fee Context

The [Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) documentation
describes base transaction fees as having weight and length components. Weight is tied to the
computational load of a submitted action, while length is tied to the size of that submitted action.

This base-fee concept belongs to state-changing activity. It should not be collapsed into every
economic effect that can occur around a Bittensor action, and it should not be applied to passive
chain reads.

## Swap Fee Boundary

The [Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) source treats swap
fees as separate from the base transaction fee. Stake and unstake activity can involve an
amount-based swap fee when liquidity is transacted through subnet markets.

That boundary matters because a stake-related action can involve more than one cost concept. The
base chain fee belongs to the submitted action, while the swap fee belongs to the liquidity-linked
part of the action.

## Fee-Free Activity

The [official fee page](https://docs.learnbittensor.org/learn/fees) identifies chain reads as free
and lists some weight-setting and commit-reveal activity as fee-free. The useful distinction is not
whether an action appears small, but whether the chain classifies that action as fee-bearing.

This keeps observation separate from submitted action. A chain read inspects state without asking
Subtensor to record a change, while fee-bearing activity belongs to the submitted-action path.

## Grouped Actions

Grouped actions do not erase fee behavior. The
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) page explains that batch
transaction fees aggregate the fee behavior of the inner calls and include batch-wrapper overhead.

This means a grouped submission should be read through its components. A batch containing
fee-bearing activity can carry a fee, while a group of fee-free actions has a different fee context.

## Recycling Context

Transaction fees also sit near recycling vocabulary. The
[Recycling and burning](https://docs.learnbittensor.org/resources/glossary#recycling-and-burning)
glossary entry describes collected transaction fees as recycled tokens deducted from the issuance
record for future emission.

This context prevents collected fees from being treated as a generic burn in every article. Fee
recycling is an issuance-accounting concept, while transaction fees are the cost layer on submitted
actions.

## Network Context

Transaction-fee examples belong to a selected Bittensor network environment. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, so fee examples should keep the network context visible
when the claim depends on observed state.

The same fee rule family can be discussed across environments, but balances, subnet markets, and
observed fee outcomes belong to the network where the action is evaluated.

## Evidence Boundary

The [Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) page supports
conceptual claims about fee categories, fee-free activity, batch fee behavior, and fee recycling. It
does not provide a permanent quote for every action or replace network-specific inspection.

When exact cost matters, the claim should identify the action, network, and source context. This
article is a concept reference, not a calculator, operating guide, or guarantee of a future fee
outcome.
