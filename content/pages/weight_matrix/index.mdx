# Weight Matrix

A weight matrix is the subnet-level collection formed from validator weight vectors. The Bittensor
glossary describes it as the matrix made from ranking weight vectors submitted by subnet validators
and used as an input to Yuma Consensus.

References: [Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)

## What the Matrix Represents

Each validator weight vector is one validator's relative evaluation of miner work inside a subnet.
The weight matrix puts those vectors into a shared structure so the consensus process can compare
validator signals at the subnet level instead of treating one validator's view as the whole result.

References: [Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Validator Weights](https://docs.learnbittensor.org/resources/glossary#weight-vector)

## Why It Matters

Yuma Consensus depends on many validator signals. The weight matrix is the organized input that lets
those signals be combined, filtered, and interpreted as a collective subnet result. This keeps the
article-level meaning focused on aggregation: the matrix is not a separate scoring rule, and it is
not a public quality label for a subnet.

References: [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)

## Relationship to Weight Vectors

A weight vector is the per-validator signal. A weight matrix is the set of those signals arranged
for the subnet's consensus step. This distinction matters because claims about one validator's
vector do not automatically describe the matrix result after other validators' vectors are included.

Reference: [Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)

## Relationship to Multiple Mechanisms

A weight matrix is usually discussed for one incentive mechanism at a time, but subnets can also run
multiple mechanisms in parallel. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-matrix) notes that when
multiple incentive mechanisms are used, each mechanism has its own weight matrix for independent
consensus calculations.

For readers, a weight matrix still collects validator weight vectors for Yuma Consensus, but the
mechanism context matters. A matrix for one mechanism does not describe miner evaluation in another
when a subnet uses more than one evaluation system.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)

## Rank Output Context

The weight matrix organizes validator weight vectors as input to Yuma Consensus. The related
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank) term names the final
aggregate judgment produced after that consensus step processes combined validator signals. Readers
should treat the matrix as the structured input collection and rank as a downstream consensus result
used when subnet emissions are allocated.

References: [Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)

## Careful Interpretation

A weight matrix is a consensus input, not an operating guide. It should not be used to infer exact
participant outcomes without the surrounding Yuma Consensus context, the subnet's incentive
mechanism, and the validator weights that formed the matrix.

References: [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Validator Weights](https://docs.learnbittensor.org/resources/glossary#weight-vector)

## Relationship to Yuma Consensus

A weight matrix and Yuma Consensus address related but different parts of Bittensor subnet incentive
flow. A weight matrix collects validator weight vectors as structured input for a subnet, while Yuma
Consensus aggregates those inputs into miner incentives and validator dividends. The
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)
describes the matrix formed from validator weight vectors, and
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) describes how those inputs
become consensus outcomes.

For readers, a weight matrix names the organized collection of validator signals, while Yuma
Consensus names the aggregation step that turns that collection into shared incentive results.

References:
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)

## Relationship to Validator Weights

A weight matrix and validator weights address related but different parts of subnet evaluation
structure. Validator weights are the per-validator evaluation signals validators submit for miners,
while a weight matrix arranges those signals together as input to Yuma Consensus. The
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)
describes the per-validator evaluation inputs, and the
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)
describes the subnet-level collection built from those vectors.

For readers, validator weights name individual evaluation submissions, while a weight matrix names
the combined structure those submissions form before aggregation.

References:
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)

## Relationship to Consensus Score

A weight matrix and a consensus score refer to related but different parts of Yuma Consensus. A
weight matrix organizes validator weight vectors as input, while a consensus score is the
stake-weighted median used to filter outlier submissions before final rank calculations. The
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)
describes the structured validator input, and the
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)
describes the agreement threshold applied to those inputs.

For readers, a weight matrix names the collected validator signals, while a consensus score names
the filtering step that helps consensus ignore out-of-line submissions.

References:
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)

## Relationship to Tempo

A weight matrix and tempo are related but different parts of subnet timing vocabulary. Tempo names
the block-based cadence that governs epoch transitions inside a subnet, while a weight matrix
collects the validator weight vectors submitted during that subnet's periodic work. The
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo) describes tempo as the
subnet-specific cadence for epochs, and the
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)
describes the matrix built from validator weight vectors across that cadence.

For readers, tempo names the subnet's periodic cadence, while a weight matrix names the structured
validator inputs that Yuma Consensus can aggregate within that cadence.

References: [Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)

## Relationship to Commit Reveal

A weight matrix and commit reveal both concern validator weights, but they describe different
things. A weight matrix is the subnet-level collection formed from the weight vectors validators
submit, used as an input to Yuma Consensus, as the
[Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix) glossary entry
describes. Commit reveal, by the
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal) documentation, is a scheme
in which validators first submit a hidden, committed version of their weights and only reveal the
actual values after a delay, so current weights are not visible to others in real time.

For readers, the weight matrix is the assembled set of validator weights, while commit reveal
governs when and how those weights become visible. The two are not interchangeable: the weight
matrix is the data Yuma Consensus reads, whereas commit reveal is the timing scheme that conceals
each validator's contribution to that matrix until reveal. Commit reveal exists so a validator
cannot simply copy another's current weights, because the live values entering the matrix stay
hidden until the reveal step.

References: [Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)

## Relationship to Coinbase

A weight matrix and coinbase are related but different parts of the consensus-to-emission path. A
weight matrix is the subnet-level collection of validator weight vectors used as input to Yuma
Consensus. Coinbase is the per-block protocol mechanism that drives TAO emission, injects TAO into
emitting subnets' pool reserves, accumulates pending emissions, and checks epoch boundaries to
trigger Yuma Consensus rounds. The
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)
describes the structured validator inputs, and the
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
documentation describes the per-block emission operation.

The weight matrix is the data structure that coinbase-triggered epoch rounds process. When coinbase
reaches a subnet's epoch boundary, the Yuma Consensus round it runs reads the weight matrix, applies
clipping and bonding, and produces the miner incentives and validator dividends that coinbase then
distributes. The matrix is the organized input; coinbase is the protocol trigger that activates the
consensus computation over it each epoch. One names how validator signals are collected for
aggregation, while the other names the mechanism that runs that aggregation and pays out its
results.
