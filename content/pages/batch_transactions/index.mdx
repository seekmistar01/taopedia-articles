# Batch Transactions

Batch transactions group multiple Bittensor chain actions into one outer transaction
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

The term describes packaging. A batch can carry several inner calls, but the calls still keep their
own purpose, fee context, and possible outcome.

The batch documentation names three utility-pallet variants: `batch`, `batch_all`, and
`force_batch`. The variants share the same grouping idea but differ in how they handle inner-call
failure ([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

That distinction matters when reading chain activity. The outer transaction explains why several
calls appear together, while the variant name explains the failure model that applies to the group.
Neither piece replaces the source for the inner call itself
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

## Grouped Calls

A batch is a container for runtime calls. The outer transaction groups the calls, while each call
keeps its own method and result surface
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

That makes batching grouping vocabulary rather than a new action type. The batch wrapper tells
readers how the calls are packaged; the inner extrinsic references still explain what the individual
calls do.

For example, a batch can contain actions that belong to different parts of Subtensor. Reading the
batch alone identifies the grouped submission, not the full meaning of every call inside it.

This is why batch references are most useful beside, rather than instead of, extrinsic references.
The batch wrapper describes how a set of calls was submitted; the individual extrinsic entry
describes what a particular call changes when it is evaluated
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

The batch documentation also separates utility batch wrappers from SDK helpers that submit multiple
individual extrinsics, so "multiple" in a helper name should not be treated as proof that a single
batch extrinsic was used
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

## Failure Variants

Batch variants differ by failure behavior. The useful question is whether the grouped transaction
allows partial success, requires all inner calls to succeed together, or continues evaluating later
calls after a failure
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

The basic `batch` form is for grouped calls where partial success can be acceptable. The `batch_all`
form is the all-or-nothing variant: all inner calls must succeed together, or none of them are
applied. The `force_batch` form continues past failures so later calls can still be evaluated
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

That makes the variant name part of the reference meaning. The same list of inner calls can describe
different behavior when wrapped by a different batch form.

The practical reading is therefore two-step: identify the wrapper, then read the inner calls through
that wrapper's failure rule. Without the wrapper name, a grouped list of calls does not by itself
tell a reader whether later calls could continue after an earlier failure.

## Fees and Submission

Batching changes grouping, but it does not make the inner actions free. Bittensor's fee
documentation explains that batch fees aggregate inner-action fee behavior and add wrapper overhead
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

That keeps batch vocabulary separate from fee vocabulary. The batch describes how calls are grouped;
the fee reference explains how costs are counted.

The fee context also differs from the failure context. Failure behavior explains which inner calls
can take effect; fee documentation explains how the grouped submission and its inner calls are
charged.

Those claims stay separate. A batch may make a set of calls easier to discuss as one submission, but
fee analysis still belongs to the fee documentation and the specific calls inside the batch
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

## Signer Authority

Batching does not remove the signer requirements for the actions inside the group. The permissions
guide separates permissionless actions from hotkey and coldkey authority, while the batch reference
describes how multiple runtime calls can be submitted together
([Bittensor CLI: Permissions Guide](https://docs.learnbittensor.org/btcli/btcli-permissions),
[Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

That keeps signer authority separate from packaging. A batch wrapper can group calls, but each inner
action should still be read with the key, role, and balance requirements that apply to that action.

## Reads and Chain Records

Batch transactions belong to state-changing action vocabulary, not read-only inspection. Reading
state observes the chain; a batch asks the runtime to process grouped calls
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

After processing, related records can come from events, storage, or errors. Those surfaces describe
what the chain emitted, stored, or rejected after the submitted batch was evaluated
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

The selected batch form affects how those later signals are read. A failed `batch_all` has different
meaning from a `force_batch` that records one failed inner call and still evaluates later calls
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

The same separation helps when tracing a result. The batch wrapper is the submission context;
events, storage, and errors are the records that help explain what the runtime did with that
submission.

## Wrapper and Inner-Call Reading

A batch transaction has two useful reading layers: the wrapper and the inner calls. The wrapper
explains the group behavior, while the inner calls explain the runtime actions being requested
inside that group ([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

| Reading layer    | What it explains                                          | What it does not replace           |
| ---------------- | --------------------------------------------------------- | ---------------------------------- |
| Batch wrapper    | Why several calls share one outer submission              | The meaning of each inner call     |
| Failure variant  | Whether failure stops, rolls back, or continues the group | The reason a specific call failed  |
| Inner extrinsic  | The actual runtime method being requested                 | The wrapper's group-level rule     |
| Signer authority | Which key or role can authorize the action                | The batch packaging itself         |
| Fee context      | How grouped submission and inner calls are charged        | Runtime success or state change    |
| Event record     | What the runtime emitted while processing                 | The full explanation of every call |
| Storage state    | What public state remains inspectable later               | The submitted call sequence        |

This split keeps the variant name useful without overloading it. A `batch_all` wrapper tells readers
to expect all-or-nothing group behavior, while a `force_batch` wrapper tells readers that later
calls can still be evaluated after an earlier failure
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

The inner extrinsics still carry the action meaning. If one inner call changes staking state and
another updates a setting, the batch wrapper explains their shared submission, not a new combined
mechanism ([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

Signer authority stays attached to the calls being made. The permissions guide separates hotkey,
coldkey, and permissionless contexts, so a batch should not be read as removing the authority needed
by an inner action
([Bittensor CLI: Permissions Guide](https://docs.learnbittensor.org/btcli/btcli-permissions),
[Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

Fee and result evidence come afterward. Fees explain the cost side of grouped submission, while
events, storage, and errors help readers inspect what happened after the runtime evaluated the batch
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

For readers, the practical order is to identify the wrapper, read each inner extrinsic, check
authority and fee context, and then use events, errors, or storage to understand the observed
result.

That order also helps with partial results. A grouped submission can produce mixed evidence across
inner calls, so event and error records should be read with the wrapper variant that produced them
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

Without that wrapper context, the same event list can be harder to interpret.

## Network Context

A batch example belongs to the Bittensor network where it is observed. Mainnet, testnet, and
localnet are separate contexts, so observed behavior from one environment should not stand in for
another network's chain history
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

The grouping concept can stay stable while fees, failures, and resulting state depend on the
selected wrapper, network, and inner calls.

## Reading the Term

Batch transactions identify grouping behavior. They are not a fee waiver, a guarantee that every
inner call succeeds, or a replacement for the individual action references
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

Use the batch term when the grouping behavior matters. Use the individual extrinsic reference when
the question is what a particular inner call does.
