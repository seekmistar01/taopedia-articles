# H160 Address

An H160 address is the Ethereum-style wallet address used by EVM wallets in Bittensor EVM. The
[Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials) page says Bittensor
wallets use Polkadot-style SS58 addresses, while Ethereum wallets use H160 addresses
([Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials)).

The practical point is simple: an H160 address belongs to the Ethereum-compatible side of Bittensor
EVM. It is not the same thing as a native Bittensor coldkey address, hotkey address, or SS58 wallet
address.

## Why It Exists

Bittensor EVM lets Ethereum-style smart contracts run on the Bittensor blockchain. That makes it
possible to use familiar EVM wallet concepts in a Bittensor setting. H160 addresses are part of that
EVM wallet side, even though the chain context differs from Ethereum mainnet
([Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials)).

For a reader, the safest interpretation is: H160 is the EVM wallet address format used inside
Bittensor EVM.

## What It Identifies

An H160 address identifies an EVM wallet account. In Bittensor EVM, that is the account an
Ethereum-style wallet can use for EVM smart-contract activity on Bittensor
([Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials)).

That identity does not extend to native Bittensor key vocabulary. An H160 address does not identify
a native Bittensor hotkey or coldkey, and it does not prove that the holder can sign native
Subtensor actions from a related SS58-looking address.

## Common Confusions

An H160 address can be misunderstood in several ways:

- It is not an Ethereum mainnet activity marker just because it is Ethereum-style.
- It is not a native Bittensor SS58 wallet address.
- It is not a coldkey or hotkey label.
- It is not proof that someone controls a converted SS58 representation.
- It is not a replacement for checking the intended destination before moving funds.

These distinctions matter because wallet mistakes can be irreversible. Readers should keep the EVM
wallet address, native Bittensor wallet address, and signing key separate in their mental model.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For H160
addresses, that sequence changes how readers should interpret evidence about EVM address usage and
conversion on Bittensor
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

In localnet, H160 address interactions can be tested in an isolated environment; the results reflect
local EVM configuration and do not represent address state or conversion behavior on a production
network. On testnet, H160 address behavior can be observed in a shared, non-production EVM network,
which separates the result from a private local environment while keeping EVM activity outside
mainnet wallet state. On mainnet, an H160 address identifies an EVM wallet on the live Bittensor EVM
layer ([Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials)).

For readers, this keeps H160 address examples from being treated as carrying more weight than their
environment supports. Localnet, testnet, and mainnet contexts can all describe EVM address
operations, but they carry different interpretive weight.

## EVM Wallets Sign Contract Calls, Not Native Actions

The [Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials) overview
distinguishes EVM wallets from native Bittensor wallets by signing scope. An H160-backed EVM wallet
signs smart-contract interactions on the EVM layer, while native Bittensor wallets sign Subtensor
transactions through Bittensor wallet tooling.

Readers should not assume an H160 address can authorize native staking, registration, or governance
actions just because a converted SS58 representation exists
([Convert Ethereum H160 Address to Substrate SS58](https://docs.learnbittensor.org/evm-tutorials/convert-h160-to-ss58)).

## Distinction from Coldkey

An H160 address and a coldkey identify wallets on two different sides of Bittensor. An H160 address
is the Ethereum-style account format used by EVM wallets in Bittensor EVM, while a coldkey is the
native SS58 owning key that authorizes Subtensor actions such as staking and transfers, so holding
or converting an H160 address does not grant the signing control a coldkey has over native wallet
authority ([Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials),
[Wallets, Coldkeys and Hotkeys in Bittensor](https://docs.learnbittensor.org/keys/wallets)).

- **H160 address** — the Ethereum-style EVM wallet address used in Bittensor EVM.
- **Coldkey** — the native SS58 key that authorizes Subtensor wallet actions.

A staking precompile is one specific case where an H160 address does feed into native coldkey
accounting. Tutorial documentation describes EVM code calling this precompile to reach the same
underlying staking actions Subtensor exposes natively, `add_stake` and `remove_stake`, to delegate
or undelegate stake to a hotkey
([Staking Precompile](https://docs.learnbittensor.org/evm-tutorials/staking-precompile)).

When another smart contract calls that precompile, the documentation states the precompile uses the
calling contract's own address as the coldkey for the resulting stake, since the precompile has no
way to see the original caller behind it. That is a documented exception tied to this one call path,
not a general reversal of the rule that an H160 address does not otherwise identify a native
coldkey.

## Distinction from Hotkey

An H160 address and a hotkey identify keys on two different sides of Bittensor. An H160 address is
the Ethereum-style account format used by EVM wallets in Bittensor EVM, while a hotkey is the native
SS58 key used for frequent network operations such as validation and mining
([Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials),
[Wallets, Coldkeys and Hotkeys in Bittensor](https://docs.learnbittensor.org/keys/wallets)).

- **H160 address** — the Ethereum-style EVM wallet address used in Bittensor EVM.
- **Hotkey** — the native SS58 key used for frequent network operations.

## Distinction from SS58 Encoded

An H160 address and an SS58 encoded address are two wallet-address formats. H160 is the
Ethereum-style format used by EVM wallets in Bittensor EVM, while SS58 is the encoding format for
public-key-derived native Bittensor wallet addresses; converting between them does not move the
private key ([Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials),
[Glossary: SS58 Encoded](https://docs.learnbittensor.org/resources/glossary#ss58-encoded)).

- **H160 address** — the Ethereum-style EVM wallet address format.
- **SS58 encoded** — the native Bittensor address encoding format.

H160 and SS58 are different address styles. H160 is associated with Ethereum-style wallets; the
[SS58 Encoded glossary](https://docs.learnbittensor.org/resources/glossary#ss58-encoded) describes
SS58 as the encoding format for public-key-derived Bittensor wallet addresses
([Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials)).

The Bittensor EVM documentation explains that a user can convert between H160 and SS58
representations. That conversion is useful when a transfer or account view needs to cross the
boundary between an EVM wallet and a Subtensor wallet view. The same documentation gives the
important warning: conversion does not produce the corresponding private key. In other words,
turning an address into another representation does not give signing control over the other wallet
([Convert Ethereum H160 Address to Substrate SS58](https://docs.learnbittensor.org/evm-tutorials/convert-h160-to-ss58)).

## Distinction from Token Bridging

An H160 address and token bridging name a destination and a movement. The H160 address is the
EVM-side account that can hold TAO in the EVM layer, while token bridging is the capability of
moving TAO into that EVM context
([Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials),
[Token Bridging](https://docs.learnbittensor.org/evm-tutorials/bridge-vtao)).

- **H160 address** — the EVM-side destination for TAO.
- **Token bridging** — the movement that puts TAO into that destination.

## Distinction from Bittensor EVM

H160 addresses appear wherever the EVM side of Bittensor needs Ethereum-style account references for
smart-contract interactions and EVM wallet activity
([Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials)).

Native Bittensor coldkeys, hotkeys, validators, and subnet roles keep their own meanings on the
Subtensor side. An H160 address names an EVM wallet account, not a native wallet role.

- **H160 address** — the Ethereum-style account format for EVM wallet activity on Bittensor.
- **Bittensor EVM** — the on-chain EVM layer where those accounts and smart contracts operate.

## Distinction from Bittensor EVM Smart Contracts

Bittensor EVM smart contracts use Ethereum-style H160 addresses, the 20-byte address format the EVM
layer works with. The smart contract is the executable code on Subtensor's EVM, while the H160
address is the Ethereum-compatible identifier used to deploy and call it; tooling converts between
H160 and Subtensor's native SS58 format when the two layers interact
([Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials),
[Convert H160 to SS58](https://docs.learnbittensor.org/evm-tutorials/convert-h160-to-ss58)).

## Distinction from Mnemonic

An H160 address and a mnemonic name an identifier and a recovery secret. An H160 address is the
Ethereum-style account format that an EVM wallet uses in Bittensor EVM, while a mnemonic is a
sequence of words used to regenerate keys and restore coldkeys and hotkeys in the Bittensor wallet,
so an H160 address is something a wallet exposes as a destination, whereas a mnemonic is the
recovery material that can rebuild the underlying native keys
([Glossary: Mnemonic](https://docs.learnbittensor.org/resources/glossary#mnemonic)).

- **H160 address** — the Ethereum-style EVM account identifier in Bittensor EVM.
- **Mnemonic** — the word sequence that regenerates and restores native Bittensor keys.

## Distinction from Wallet Address

An H160 address and a wallet address name a format and a destination role. An H160 address is the
Ethereum-style account format used by EVM wallets in Bittensor EVM, while a wallet address is the
unique identifier derived from the public key, used as a destination for sending and receiving TAO
tokens in the Bittensor network, so an H160 address describes the encoding an EVM account carries,
whereas a wallet address describes the destination role that a native public-key-derived identifier
fills
([Glossary: Wallet Address](https://docs.learnbittensor.org/resources/glossary#wallet-address)).

- **H160 address** — the Ethereum-style EVM account format used in Bittensor EVM.
- **Wallet address** — the public-key-derived identifier used as a send/receive destination for TAO.

Conversion documentation explains that users can translate between H160 and SS58 representations
when a transfer or account view crosses wallet styles, but conversion does not produce the
corresponding private key
([Convert Ethereum H160 Address to Substrate SS58](https://docs.learnbittensor.org/evm-tutorials/convert-h160-to-ss58)).

H160-to-SS58 conversion therefore belongs to address-format vocabulary. Wallet control still follows
whichever private key can sign in the intended environment
([Glossary: SS58 Encoded](https://docs.learnbittensor.org/resources/glossary#ss58-encoded)).

## Distinction from Yuma Consensus

An H160 address names the Ethereum-style account format used in EVM contexts on Bittensor. Yuma
Consensus is the on-chain mechanism that converts validator weight submissions into emission shares
each tempo and is not an EVM address format
([H160 Address](https://docs.learnbittensor.org/resources/glossary#h160-address),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

H160 vocabulary belongs to EVM account identification. Yuma Consensus vocabulary belongs to
validator weight settlement on a subnet.

- **H160 address** — Ethereum-style account identifier in EVM contexts.
- **Yuma Consensus** — on-chain mechanism that turns validator weights into emission shares.

## Reader Boundary

This article explains the term, not a setup procedure. It does not describe how to use an EVM
wallet, deploy a contract, or perform address-conversion steps.

An H160 address is best understood as the Ethereum-style address for the EVM wallet side of
Bittensor EVM. When an address crosses into SS58 representation, the user should treat that as an
address-format bridge, not as a transfer of private keys or native wallet authority.
