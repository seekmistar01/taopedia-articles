# Commit Reveal

Commit Reveal is a Bittensor mechanism for delaying public visibility of fresh validator weights. It
conceals weights before they become visible inputs to [[yuma_consensus|Yuma Consensus]], reducing
the value of copying another validator's recent weight signals
([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

## Why It Exists

Validator weights need to become visible for transparent consensus records, but immediate visibility
can invite weight copying. The
[Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)
describes the risk: validators can free-ride on visible weight information instead of independently
evaluating miners.

Commit Reveal addresses that timing problem by making recent weight information harder to reuse
while it is still valuable. It does not replace independent evaluation; it protects the visibility
path around that evaluation.

## Concealment Flow

With Commit Reveal enabled, validator weights pass through a concealment phase before they become
visible for consensus processing. The
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal) documentation describes this
as a commit phase followed by a later reveal phase. Once revealed, the weights can be used by the
consensus process.

That flow separates evaluation from immediate public visibility. The mechanism is about when weight
information becomes visible, not about changing what miners are being evaluated for.

## Drand Timing

Commit Reveal uses Drand time-lock encryption so concealed weights cannot be decrypted before the
target reveal time ([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)). This
timing property is central to the mechanism because it prevents early access to the concealed weight
information.

The result is a protocol-level timing barrier around validator weights. Delayed visibility makes
copied information less useful because the copied view is older by the time it can be seen.

## Subnet Configuration

Commit Reveal is controlled through subnet hyperparameters. The
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal) page describes
`commit_reveal_weights_enabled` and `commit_reveal_period`, and the
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) reference
places those settings in the wider subnet-configuration vocabulary.

This keeps the term scoped. Commit Reveal is a configurable weight-visibility mechanism, not a
universal state of every subnet or a replacement for a strong evaluation design.

## Consensus Context

Commit Reveal and Yuma Consensus describe different stages of the same validator-weight path.
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) explains how validator
evaluation signals are aggregated into subnet outcomes. Commit Reveal describes how those signals
can be concealed before they become visible to that process.

The distinction is useful: Commit Reveal is about delayed visibility, while Yuma Consensus is about
aggregation after validator signals are available for processing.

## Reading Boundaries

Commit Reveal is the right term when the focus is delayed visibility for validator weights. If the
focus is copied-weight behavior,
[weight copying](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor) is more
precise. If the focus is the evaluation signal itself,
[validator weights](https://docs.learnbittensor.org/resources/glossary#validator-weights) is more
precise. If the focus is aggregation,
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) is more precise.

These boundaries keep Commit Reveal focused on concealment timing, not a complete validator guide or
a full explanation of subnet incentives.
