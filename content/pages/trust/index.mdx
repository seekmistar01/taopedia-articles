# Trust

Trust is a Bittensor consensus metric for the miner side of evaluation. It describes how much of a
miner's original validator support remains after consensus clipping
([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The term belongs to Yuma Consensus terminology because it is read after validator weight signals are
filtered and aggregated.

## Consensus Clipping Context

Validators submit weight signals after evaluating miner work. Consensus filtering can reduce outlier
support before the final miner-side result is read
([Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

Trust describes how much miner support remains after that filtering. It is therefore about
consensus-adjusted support, not a separate miner label outside the consensus process.

The clipping context is the main boundary. Trust does not name the original validator weights and
does not name final emission allocation; it describes what remains of miner-side support after
consensus filtering
([Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

## Rank Context

Rank and trust are connected but not identical. Rank names the consensus-adjusted miner result,
while trust describes how much support was preserved between pre-rank and final rank
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

Rank says where the miner landed after aggregation. Trust says how much clipping changed the
pre-rank-to-rank relationship.

This keeps the result and the comparison separate. Rank is the adjusted miner-side outcome, while
trust describes the preservation of support between the pre-rank signal and the final rank
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

## Pre-Rank Relationship

Trust is easiest to read as a comparison between a miner's pre-rank and final rank. Pre-rank is the
support pattern before consensus filtering, while rank is read after outlier weights have been
clipped ([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The useful reading is relational. Trust does not replace rank; it explains how strongly the final
rank stayed aligned with the pre-clipping support represented by validator weights.

That relationship makes trust useful for reading clipping effects without turning it into a separate
scorecard. The term is about the change between pre-rank and rank, not about miner labeling, subnet
operation, or a subnet-independent reputation value.

## Validator Trust Boundary

Trust is read from the evaluated miner's side of the process. Validator trust is read from the
validator side, describing how much of a validator's submitted signal remains after filtering
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

Keeping the two terms separate avoids treating trust as a universal quality score for every role in
a subnet.

The boundary is role direction. Trust is miner-side, because it concerns support for an evaluated
miner. Validator trust is validator-side, because it concerns how much of a validator's submitted
signal remains after filtering
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)).

## Same-Context Comparison

Trust comparisons are most meaningful when the compared values come from the same subnet consensus
context. Yuma Consensus operates over validator weights inside a subnet, so miner-side trust values
are clearest when read against the process that produced them
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

Across different subnets, the task, validator set, incentive mechanism, and weight pattern can
differ. Trust still has the same definition, but the surrounding consensus setting can change what a
particular value is telling the reader.

This is why same-context comparisons are safer than cross-subnet comparisons. A trust value is most
informative when the subnet task, validator set, and consensus process are part of the comparison
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

## Reader Boundary

Trust should be read inside subnet consensus interpretation. It is not an operating instruction, a
staking guide, or a promise about future miner results
([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The stable article-level point is that trust explains how consensus clipping changes miner-side
support between pre-rank and final rank.

## Environment Boundary

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. Trust keeps the same miner-side clipping meaning across
those environments, but a concrete trust value belongs to the network environment and subnet
consensus context where it was produced.

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
places subnet development across localnet, testnet, and mainnet. Trust examples are clearest when
the environment, subnet task, validator set, and clipping context stay attached.
