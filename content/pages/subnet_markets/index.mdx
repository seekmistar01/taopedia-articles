# Subnet Markets

Subnet markets are the economic and evaluation contexts created by Bittensor subnets. Each subnet
defines a market for a specialized digital commodity
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction)).

The term is useful because it keeps the subnet market separate from the whole Bittensor network. A
subnet narrows the work category and evaluation context, while the broader protocol supplies the
chain and tokenomics environment around it.

## Market Role

The Bittensor introduction describes subnets as specialized competition markets. Miners produce work
for the market, validators evaluate that work, and the protocol connects evaluation to reward
allocation ([Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

That market role is narrower than the whole network. A subnet market names one work-and-evaluation
context; Bittensor names the broader protocol where many subnet markets can operate.

## Work and Evaluation

Subnet markets need both production and evaluation. The incentive-mechanism documentation explains
how subnet creators define tasks and validation logic, while Yuma Consensus connects validator
evaluation to miner incentives and validator dividends
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

For readers, the market is not just the work category. It is the context where work, evaluation, and
reward signals are connected.

## Token Context

Subnet markets also have a token side. The subnet documentation describes TAO reserves, alpha
reserves, and alpha outstanding as part of subnet tokenomics, while emissions documentation connects
TAO and alpha emissions to subnet reward flow
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

This means a subnet-market claim should separate work context from token context. The work market
names what is being produced and evaluated; the token context explains how TAO and alpha are used
around that market.

## Emission Context

Emissions make subnet markets economically meaningful inside Bittensor. Emission documentation
describes TAO and alpha-token flows, while Yuma Consensus describes how validator evaluation
contributes to reward allocation ([Emission](https://docs.learnbittensor.org/learn/emissions),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The important boundary is that emissions do not define the task by themselves. They are the reward
flow around a market whose task and evaluation logic come from the subnet's incentive design.

## Multiple-Mechanism Boundary

Some subnets can use more than one incentive mechanism. The multiple-mechanism documentation
describes separate mechanism contexts inside a subnet
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

For subnet markets, that means one subnet market can contain more than one evaluation path. A
market-level statement should stay clear about whether it describes the whole subnet or one
mechanism inside it.

## Network Evidence Boundary

Subnet-market examples should keep their network environment attached. The Bittensor networks
reference separates mainnet, testnet, and localnet, and the introduction describes subnet
development across local and shared environments
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)).

This boundary prevents an example from one environment from being treated as evidence for another.
The market concept is stable, but observed activity belongs to the source environment.

## Coinbase Boundary

Coinbase is adjacent to subnet markets because it is part of the emission process. The coinbase
documentation describes a per-block protocol process that drives TAO emission, pending-emission
accounting, and epoch-boundary activity
([Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)).

The market names the work, evaluation, and tokenomics context. Coinbase names a protocol process
that helps feed emission flow into that context. The two terms are related but not interchangeable.

## Reader Boundary

Subnet markets should be read as per-subnet economic and evaluation contexts. They connect work,
validation, staking context, TAO and alpha tokenomics, and emission outcomes without replacing the
specific documentation for any one mechanism
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

The stable article-level point is that a subnet market names the context where a subnet's work,
evaluation, and reward flow meet. Claims about observed activity need the relevant network,
mechanism, and source context.
