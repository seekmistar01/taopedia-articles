# Self-Weighting

Self-weighting is when a single validator assigns inflated weights to entities it controls, such as
its own miner registrations, to try to steer more reward toward itself. Bittensor's consensus is
arranged so that doing this on its own does not pay
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## What It Represents

Self-weighting names a validator inflating grades for entities it controls to steer reward toward
itself. Consensus is stake-weighted agreement across validators, not any one validator's opinion;
weights that outlie that agreement are clipped
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Validators are also rewarded for agreeing with consensus, so self-preferencing tends to be
self-defeating: the inflated portion is clipped, and departing from consensus can weaken the
validator's own standing.

Examples tied to netuid 1 illustrate one subnet's clipping path. They should not be read as
representing production consensus behavior on another network
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

## Distinction from Consensus Score

Self-weighting is when a validator assigns inflated weights to entities it controls. Consensus score
is the stake-weighted median benchmark that clipping uses to trim outlier weights in Yuma Consensus
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping)).

- **Self-weighting** — validator inflating weights for entities it controls.
- **Consensus score** — stake-weighted benchmark used to clip outlier weights.

## Distinction from Rank

Self-weighting is a lone validator assigning inflated weights to entities it controls. Rank is the
consensus standing produced after all validators' weights are clipped toward the stake-weighted
majority ([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Self-weighting** — lone validator inflating weights for entities it controls.
- **Rank** — miner-side standing produced after clipped aggregation.

## Distinction from Trust

Self-weighting is a lone validator assigning inflated weights to entities it controls. Trust is a
consensus metric describing how much of a miner's support survives after consensus clips outlier
weight signals ([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Self-weighting** — a lone validator inflating weights for entities it controls.
- **Trust** — how much miner support survives consensus clipping.

## Distinction from Validator-Miner Bonds

Self-weighting is inflated validator preference in weight submissions. Validator-miner bonds are the
smoothed relationship values Yuma uses when mapping miner outcomes back to validators
([Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Self-weighting** — exploit pattern of self-preferencing weight submissions.
- **Validator-miner bonds** — smoothed validator-miner relationship used during consensus.

## Distinction from Stake Weight

Self-weighting is a validator assigning inflated weights to entities it controls. Stake weight is
the stake-derived value that scales how much that validator's weights count in consensus
([Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Self-weighting** — exploit pattern of inflating controlled entities' weights.
- **Stake weight** — stake-derived scale for how strongly weights count in consensus.

## Distinction from Collusion

Self-weighting is one validator inflating grades for entities it controls without a coordinating
group. Collusion is multiple validators acting together toward shared targets; a coalition can pool
stake-weighted signal that a lone validator cannot
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Yuma Consensus: Clipping excess weights](https://docs.learnbittensor.org/learn/yuma-consensus#clipping-excess-weights)).

- **Self-weighting** — one validator inflating weights for entities it controls.
- **Collusion** — multiple validators coordinating inflated weights toward chosen miners.

## Distinction from Weight Copying

Self-weighting is a validator inflating grades for entities it controls. Weight copying is
validators reusing visible weight signals instead of performing independent evaluation
([Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Glossary: Weight Copying](https://docs.learnbittensor.org/resources/glossary#weight-copying)).

- **Self-weighting** — inflating weights for entities the validator controls.
- **Weight copying** — reusing another validator's visible weights without independent evaluation.

## Distinction from Sybil Attack

A Sybil actor multiplies identities at registration, while self-weighting is a permitted validator
biasing weight submissions after it already holds a subnet slot
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Self-weighting** — one validator favoring its own side through inflated weights.
- **Sybil attack** — one actor registering many identities to seek leverage from numbers.

## Distinction from Validator Permit

Self-weighting is biased scoring by a validator that may already hold a permit. The validator permit
is the on-chain flag recording whether a UID may submit non-self weights that count in consensus
([Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Self-weighting** — a validator inflating weights for entities it controls.
- **Validator permit** — the on-chain right to submit non-self weights that count in consensus.

## Distinction from Yuma Consensus

Self-weighting names the pattern where a validator assigns disproportionate weight to its own hotkey
or related entities. That submission-risk vocabulary is separate from the tempo pass that filters
and settles weights into emission shares
([Glossary: Self Weighting](https://docs.learnbittensor.org/resources/glossary#self-weighting),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Yuma Consensus clips weights that exceed the stake-weighted consensus benchmark, then converts the
filtered result into miner incentives and validator dividends at the epoch boundary
([Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Self-weighting describes what a validator attempted in its submission. Yuma describes how those
signals are filtered and credited on the relevant netuid
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

- **Self-weighting** — validator overweighting its own hotkey or controlled entities.
- **Yuma Consensus** — epoch-boundary settlement with clipping, bonding, and dividend allocation.

## Reader Boundary

This page defines the concept and Bittensor's consensus-level resistance at a high level. It does
not claim manipulation is impossible in every case, since a participant holding enough stake is a
different situation, nor does it quantify any threshold. The stake distribution and consensus values
that decide how strong the resistance is on any subnet are live chain state and change over time
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).
