# Root Subnet

The Root Subnet, also called Subnet Zero, is a special Bittensor subnet. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero) describes it
as a subnet where miners cannot register and no validation work is performed, while validators can
register and TAO holders can stake to those validators.

## Subnet-Agnostic Staking

Root Subnet staking is subnet-agnostic. Instead of tying stake directly to one ordinary subnet's
local context, the glossary describes Root as a way for TAO holders to stake to validators across
the wider Bittensor setting.

## Validator Weight Context

The [Glossary](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero) explains
why subnet-agnostic Root staking matters for validator weight: a validator's weight in a subnet
includes both their share of that subnet's alpha and their share of staked TAO in Subnet Zero.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from local testing to testchain and then mainchain. For the
Root Subnet, that sequence changes how readers should interpret examples of subnet-agnostic staking
context.

In local testing, a Root Subnet example can show how root-routed stake or validator standing is
represented inside an isolated environment. That is useful for checking the concept, but it does not
show live public Root staking or production validator standing.

On testchain, Root-related behavior can be observed in a shared testing network. This gives stronger
evidence about how validator registration, stake context, and subnet-agnostic weighting interact
than a private local run, while still keeping the result separate from production mainchain
activity.

On mainchain, the Root Subnet belongs to live Bittensor staking and governance context. The
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) documentation
describes Root as a special subnet for validator registration and TAO-holder staking, so production
readings should keep the network, validator set, stake context, and block timing attached.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A Root Subnet example in one environment should not be
read as identical evidence about validator standing or staking context in another environment.

This distinction matters because the Root Subnet is subnet-agnostic, but not environment-agnostic.
The network environment explains what kind of evidence the example provides, while the Root context
explains how TAO staking is being read.

For readers, this keeps Root Subnet examples from sounding stronger than their environment supports.
Local, testchain, and mainchain contexts can all contain Root Subnet language, but they do not carry
the same interpretive weight.

## Relationship to Root Claims

The Root Subnet and root claims address related but different parts of Bittensor staking context.
The [Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) overview
describes the Root Subnet as a special subnet where validators register and TAO holders can stake in
a subnet-agnostic way, while the
[Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims) describes
how alpha dividends from that stake are handled.

For readers, the Root Subnet is the staking environment, while root claims name the
dividend-handling mechanism for stake placed there.

References: [Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)

## Relationship to Related Terms

Root Subnet is the base term. Root claims describe how dividends are handled for stake placed
through Root, while root proportion describes how much of a subnet's dividend context is attributed
to Root-routed TAO stake. Those related terms should not be treated as the same concept as Root
Subnet itself.

## Relationship to Multiple Mechanisms

The Root Subnet is a special subnet where miners cannot register and no validation work is
performed. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms) notes
that validators must evaluate miners separately for each mechanism when a subnet runs more than one
incentive mechanism.

For readers, Root Subnet still names subnet-agnostic validator staking. That special role is
separate from the per-mechanism miner evaluation context used in ordinary subnets.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)

## Relationship to Senate

The root subnet and the Senate are related but different parts of Bittensor governance vocabulary.
The root subnet names the network-level context where validators hold subnet-agnostic stake
standing, while the Senate names the governance body formed from the highest-ranked of those
validators to review proposals. The
[Glossary: Root Network](https://docs.learnbittensor.org/resources/glossary#root-network) describes
the root validator context, and the
[Glossary: Senate](https://docs.learnbittensor.org/resources/glossary#senate) describes the elected
delegates who approve or reject Triumvirate proposals.

For readers, the root subnet names the staking and ranking context, while the Senate names a
governance role drawn from the top of that ranking. Standing in the root subnet is what can place a
validator into the Senate; the Senate is not the ranking itself.

References:
[Glossary: Root Network](https://docs.learnbittensor.org/resources/glossary#root-network),
[Glossary: Senate](https://docs.learnbittensor.org/resources/glossary#senate)

## Relationship to Root Claim Type

The Root Subnet and root claim type are related but distinct parts of Bittensor staking vocabulary.
The Root Subnet is the special subnet where validators register and TAO holders can stake in a
subnet-agnostic way, while root claim type is the per-staker Keep-or-Swap dividend-handling
preference the
[Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims) names for
alpha dividends earned through that root-routed stake.

For readers, the Root Subnet is the staking environment that produces root-routed dividends, while
root claim type names the dividend-handling setting a root staker chooses inside that environment.
The Root Subnet governs where the staking position lives; root claim type governs what happens to
the alpha dividends that position earns. A reader who understands Root as the subnet-agnostic
staking context can use root claim type to identify how the resulting dividends are routed.

References:
[Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims),
[Glossary: Root Subnet/Subnet Zero](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero)

## Reader Boundary

This article defines the term at a high level. It does not list current Root validators, describe
staking steps, compare staking choices, or explain root-claim settings. Readers should use official
staking references for those details.
