# Multiple Incentive Mechanisms

Multiple incentive mechanisms describe the case where one Bittensor subnet contains more than one
separate evaluation mechanism. Each mechanism has its own evaluation system and bond pool for Yuma
Consensus calculations
([Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

In plain language, multiple incentive mechanisms answers: can one subnet run more than one
independent evaluation track? The answer is yes — each track has its own evaluation system and bond
pool, so a miner's performance in one mechanism does not automatically carry into another
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

For a reader, this distinction matters most when reading weight or rating data from a
multi-mechanism subnet. A score from mechanism A is not a score from mechanism B, even though both
mechanisms share the same subnet market and UID space
([Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)).

The term is narrower than incentive mechanism by itself. An incentive mechanism connects work,
evaluation, and rewards; multiple incentive mechanisms name the structure where several such systems
coexist inside one subnet
([Glossary: Incentive Mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

## Distinction from Incentive Mechanism

Multiple incentive mechanisms names parallel evaluation tracks inside one subnet, each with its own
weights and bond pool. An incentive mechanism names one evaluation design inside that subnet
([Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms),
[Glossary: Incentive Mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism)).

- **Multiple incentive mechanisms** — parallel evaluation paths with separate bond pools.
- **Incentive mechanism** — one evaluation design inside a subnet.

Plural structure vocabulary and single-mechanism vocabulary answer different scope questions
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

## Distinction from Subnet Task

Subnet task names the work a mechanism is meant to evaluate. Multiple incentive mechanisms names
parallel evaluation tracks, each of which can carry its own task target
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)).

On a multi-mechanism subnet, readers should attach both a mechanism label and a task label before
comparing outcomes.

## Distinction from Rank

Rank is computed separately within each mechanism when a subnet runs more than one. Multiple
incentive mechanisms therefore changes how rank should be read: there is no single combined ordering
across mechanisms on the same netuid
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Rank comparisons across mechanisms on one subnet need the mechanism index named in the claim.

## Distinction from Weight Matrix

Each mechanism keeps its own weight matrix on a multi-mechanism subnet. Multiple incentive
mechanisms is the structural label; the weight matrix is the per-mechanism scoring table
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

Readers inspecting metagraph columns should expect separate mechanism weight fields when more than
one mechanism is active.

## Distinction from Yuma Consensus

Multiple incentive mechanisms names parallel evaluation tracks with separate bond pools inside one
subnet. Yuma Consensus settles validator weight submissions into emission shares separately for each
mechanism each tempo
([Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Multiple incentive mechanisms** — parallel evaluation paths inside one subnet.
- **Yuma Consensus** — per-mechanism settlement from validator weights to emission shares.

Structure vocabulary names how many tracks run; Yuma vocabulary names settlement within each track
([Emission](https://docs.learnbittensor.org/learn/emissions)).

## Distinction from Mechanism Count

Multiple incentive mechanisms name the subnet model in which more than one evaluation path runs in
parallel inside one subnet. Mechanism count names the `MechanismCount` hyperparameter that sets how
many such paths a subnet is configured to run
([Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

For readers on a subnet such as netuid 1, the model explains why weights, bond pools, and rank stay
separated by evaluation track. The hyperparameter answers how many tracks that subnet runs according
to its hyperparameter table, which can be read with `btcli subnet hyperparameters --netuid 1`
([Subnet Hyperparameters: View hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#view-hyperparameters),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

- **Multiple incentive mechanisms** — parallel evaluation paths inside one subnet.
- **Mechanism count** — the configured number of those paths.

## Distinction from Max Allowed UIDs

Multiple incentive mechanisms and max allowed UIDs answer different capacity questions on the same
subnet. Multiple incentive mechanisms names how many evaluation paths share the subnet's UID
population, while max allowed UIDs sets the ceiling on how many registered neurons that population
can hold on a subnet such as netuid 1
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

Official documentation ties the two through a product rule: mechanism count multiplied by max
allowed UIDs must not exceed 256. Adding mechanisms therefore shrinks the UID headroom available
under the same ceiling, so capacity planning on netuid 1 should read both hyperparameters together
rather than in isolation
([Subnet Hyperparameters: MaxAllowedUIDs](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#maxalloweduids)).

- **Multiple incentive mechanisms** — parallel evaluation structure inside one subnet.
- **Max allowed UIDs** — the ceiling on registered UID slots those mechanisms share.

## Distinction from UID Trimming

Multiple incentive mechanisms and UID trimming both touch subnet capacity, but they name different
layers. Multiple incentive mechanisms describes how many evaluation paths run inside a subnet, while
UID trimming is the owner action documented for bringing an oversized UID set back under the
configured ceiling on a subnet such as netuid 1
([UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

For readers, trimming addresses live over-capacity relative to max allowed UIDs. Raising the number
of mechanisms can require lowering that ceiling first because the product of mechanism count and max
allowed UIDs must not exceed 256. Trimming is the documented reduction path; multiple-mechanism
structure names how many evaluation paths share whatever capacity remains
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Multiple incentive mechanisms** — parallel evaluation paths inside one subnet.
- **UID trimming** — the documented action that brings live UID count back under the configured
  ceiling.

## Distinction from Bond Pool

The glossary ties multiple mechanisms to separate bond pools. Yuma Consensus uses validator weight
signals when calculating miner incentives and validator dividends, and the separate bond-pool frame
keeps those calculations divided by mechanism
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)).

The bond-pool detail is structural. It explains why the mechanism split can affect how evaluation
signals are interpreted before rewards are assigned.

## Reader Boundary

Multiple incentive mechanisms are subnet-internal evaluation structure. The term does not by itself
say how many mechanisms any particular subnet has, which mechanism is preferable, or how rewards
were distributed in a specific case
([Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

The stable reference point is separation: several evaluation mechanisms can exist inside one subnet,
with their own evaluation systems and bond pools.
