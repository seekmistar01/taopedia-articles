# Consensus-Based Weights

Consensus-based weights, also known as liquid alpha, make validator-miner bond smoothing depend on
how validator weights align with consensus. The mechanism sits inside the Yuma Consensus bond path
between validator-weight signals and validator-side outcomes
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The core idea is alignment-sensitive smoothing. Stronger consensus alignment can make the related
bond response more direct, while weaker alignment keeps the response more conservative.

That places the mechanism on the bond-response side of Yuma Consensus. Validator weights provide
evaluation signals; consensus-based weights adjust how validator-miner bonds respond after those
signals are compared with consensus
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights),
[Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)).

## Liquid Alpha

Liquid alpha is another name for consensus-based weights. The name highlights that the alpha term in
bond smoothing can vary rather than staying fixed across every validator-miner pair
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights),
[Exponential Moving Averages](https://docs.learnbittensor.org/learn/ema)).

Liquid alpha and consensus-based weights name the same mechanism. They do not name separate reward
types or separate miner-scoring systems.

Here, alpha is the EMA smoothing parameter, not a subnet alpha token. That keeps the mechanism tied
to bond smoothing rather than token ownership
([Exponential Moving Averages](https://docs.learnbittensor.org/learn/ema),
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

This terminology matters because alpha-token vocabulary is common elsewhere in Bittensor.
Consensus-based weights use alpha for smoothing responsiveness, not pool-held alpha, alpha
outstanding, or stake.

## Bond Smoothing

Bittensor uses exponential moving averages to smooth validator-miner bonds, so bond state does not
change entirely from one evaluation step to the next. Consensus-based weights make the smoothing
factor responsive to consensus alignment
([Exponential Moving Averages](https://docs.learnbittensor.org/learn/ema),
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

That places the mechanism after validator weights are compared with consensus. It does not replace
weight input, clipping, rank calculation, or Yuma Consensus itself.

The smoothing role is narrow. Consensus-based weights affect how validator-miner bond history
responds, while the surrounding Yuma Consensus path still defines how weights, ranks, and emissions
are interpreted.

That makes the mechanism a response-shaping step. It changes how quickly bond history reacts after
consensus comparison, while validator weights and the consensus process still provide the input
side.

The important movement is in the bond history, not in the original validator-weight submission. A
validator still submits weights; consensus-based weights change how the bond path reacts around
alignment
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Validator Dividends

Consensus-based weights affect the bond path that feeds validator dividends. Yuma Consensus routes
validator emissions through bonds to miners weighted by miner emission results, and liquid alpha
adjusts how quickly those bonds respond to consensus-aligned evaluation
([Yuma Consensus: validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions),
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

The result is still validator-side consensus behavior, not a new miner-ranking rule.

That distinction keeps dividend language from becoming a broader performance label. The mechanism
changes bond responsiveness; it does not directly rewrite the task being evaluated by a subnet.

The dividend connection is therefore downstream. Consensus-based weights adjust the bond state that
feeds validator-side emissions, while dividends name the validator-side result
([Yuma Consensus: validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions),
[Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)).

## Weight Copying

Consensus-based weights are often discussed near the weight-copying problem because copied or stale
weights can diverge from consensus. The mechanism can make bond response more sensitive to whether
validator weights remain aligned
([The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

This does not make liquid alpha an accusation model. Weight copying names an evaluation-quality
problem; consensus-based weights name a bond-path response to consensus alignment.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For
consensus-based weights on a subnet such as netuid 1, that sequence changes how readers should
interpret bond-smoothing and liquid-alpha examples tied to validator weights and clipping.

In localnet, consensus-based weight mechanics can be tested in an isolated environment. Localnet
bond history does not represent production validator-miner relationships.

On testnet, liquid-alpha bond smoothing can be exercised in a shared non-production network. Testnet
consensus alignment outcomes are separate from mainnet subnet state.

On mainnet, consensus-based weights apply as a live Yuma Consensus bond-smoothing step on production
subnets. Observed bond response depends on the selected subnet's validator weights and consensus
context
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A liquid-alpha example from one environment should not be
read as representing production bond behavior in another environment.

## Bond Blending Pulls Out-of-Consensus Weights Toward Benchmark

When a validator assigns a miner weight above the clipped
[consensus score](https://docs.learnbittensor.org/resources/glossary#consensus-score) benchmark, the
bond update blends that submission toward the benchmark rather than accepting the full value. A
configurable penalty factor controls how strongly out-of-consensus weights are pulled back, and an
exponential moving average then smooths
[validator-miner bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)
across epochs ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

Liquid alpha adjusts how quickly that smoothed bond history responds when submitted weights align or
fall out of alignment with consensus rather than replacing clipping itself.

## Clipping Filters Weights Before Bond Smoothing Responds

Consensus-based weights are adjacent to clipping but not identical to it. Clipping limits outlier
validator weights above the consensus benchmark before final
[rank](https://docs.learnbittensor.org/resources/glossary#rank) calculations, while liquid alpha
adjusts how bond history responds after weights are compared with consensus
([Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping)).

Clipping is a filtering step and liquid alpha is a bond-smoothing step. Both sit inside the
consensus path, but they answer different questions about validator-weight signals.

## Dividends Pair Smoothed Bonds With Miner Incentives

Each validator's share of validator-side emissions equals the sum of that validator's bonds
multiplied by each miner's
[incentive](https://docs.learnbittensor.org/resources/glossary#incentives) outcome
([Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions)).

Consensus-based weights shape how quickly those smoothed bonds respond to alignment, while
[dividends](https://docs.learnbittensor.org/resources/glossary#emission) name the validator-side
emission result after bonds and miner incentives are paired in the same epoch.

## Commit Reveal and Liquid Alpha Address Different Copy Risks

Delayed visibility through Commit Reveal addresses timing around visible weights, while
consensus-based weights address bond responsiveness around alignment
([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

The relationship is complementary. Commit Reveal reduces the value of copying fresh weights before
they are revealed; consensus-based weights change how bonds respond when submitted weights align or
fall out of alignment
([The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)).

## Validator Trust Reads Alignment, Not Bond Smoothing

[Validator trust](https://docs.learnbittensor.org/resources/glossary#validator-trust) sums clipped
weights a validator successfully contributed across neurons, which measures validator-side influence
after filtering. Consensus-based weights instead adjust how bond history responds when alignment
changes over repeated epochs.

The two terms share one Yuma pass but answer different reader questions. Validator trust summarizes
clipped influence; liquid alpha shapes bond responsiveness around that alignment path.

## Why Make the Smoothing Factor Variable at All

Documentation frames a fixed smoothing factor as a single trade-off: a high value reacts quickly but
can overreact to noisy or copied signals, while a low value is stable but slow to reward genuine
improvement ([Exponential Moving Averages](https://docs.learnbittensor.org/learn/ema)). Making the
factor depend on consensus alignment is described as a way to avoid committing to one fixed point on
that trade-off.

For readers, that is the purpose behind the "liquid" name. Documentation ties the variable factor to
how well a validator's weights agree with consensus, so bond history can respond more directly when
evaluation is aligned and more cautiously when it is not
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

That intent also explains why the mechanism is discussed near the weight-copying problem.
Documentation presents alignment-sensitive smoothing as reducing how much a stale or copied signal
that drifts from consensus can shape bond history, without replacing clipping or Commit Reveal
([The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)).

## Distinction from Exponential Moving Average

Consensus-based weights make the bond-smoothing alpha parameter vary with consensus alignment. EMA
is the broader smoothing method that updates validator-miner bond history gradually across epochs
([Exponential Moving Averages](https://docs.learnbittensor.org/learn/ema),
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

For netuid 1 readers, alpha here names the smoothing parameter in the bond update, not subnet alpha
tokens held in pool reserves
([Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)).

## Distinction from Clipping

Clipping limits outlier validator weights above the consensus benchmark before rank is calculated.
Consensus-based weights adjust how bond history responds after weights are compared with consensus
([Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping),
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

- **Clipping** — filters submitted weights against the benchmark.
- **Consensus-based weights** — shapes bond responsiveness around alignment.

## Distinction from Validator Trust

Validator trust sums clipped weight influence a validator contributed across neurons on netuid 1.
Consensus-based weights adjust how quickly validator-miner bonds respond when alignment changes over
repeated epochs
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Validator trust is a clipped-influence reading for one pass. Liquid alpha is the bond-response rule
across epochs
([Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)).

## Distinction from Dividends

Dividends are the validator-side emission outcomes paired from bonds and miner incentives.
Consensus-based weights shape how quickly bonds respond; they do not by themselves name a payout
amount
([Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions),
[Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

Dividend vocabulary belongs at the emission result. Consensus-based-weight vocabulary belongs on the
bond-smoothing step that feeds that result on the selected subnet.

## Distinction from Commit Reveal

Commit Reveal delays visible weight publication to reduce fresh-signal copying. Consensus-based
weights change bond responsiveness when submitted weights align or drift from consensus after they
are processed ([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal),
[The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)).

The two mechanisms address different copying risks in the same Yuma pass on netuid 1
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

## Reader Boundary

Consensus-based weights are a Yuma Consensus bond-smoothing concept. They are not a separate subnet
token, a new miner task, or a complete explanation of validator rewards
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights),
[Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)).

The mechanism only becomes meaningful with the subnet, incentive mechanism, and consensus process
that interpret validator weights
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).
