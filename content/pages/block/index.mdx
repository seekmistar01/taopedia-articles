# Block

A block is a unit of data in the Bittensor blockchain. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#block) describes a block as containing
a collection of transactions and a unique identifier called a block hash.

## Chain Timing Context

The glossary states that Bittensor processes one block every 12 seconds. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference also
lists block processing for mainnet and testnet as one block every 12 seconds, while separating those
network contexts from localnet behavior.

## Relationship to Mempool

Blocks and mempools name opposite sides of transaction inclusion on Subtensor. The
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) defines the mempool
as the place where pending transactions wait, while the
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block) describes a block as
containing a collection of transactions.

For readers, the mempool is the pre-inclusion waiting period, while a block records which pending
transactions were accepted and ordered on chain. A transaction leaves the pending mempool context
once it is included.

References: [Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)

## Relationship to Extrinsics

Blocks and extrinsics name related but different parts of Subtensor activity. An extrinsic is a
submitted state-changing runtime action, while the
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block) describes a block as
containing a collection of transactions.

For readers, extrinsics are the submitted actions, while blocks are the units that include and order
them once accepted on chain.

References: [Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)

## Relationship to Transaction Fees

Blocks and transaction fees address related but different parts of chain activity. The
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block) describes a block as
containing a collection of transactions, while
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) describes the costs the
chain can attach to state-changing actions that may be included there.

For readers, a block names chain ordering and inclusion, while transaction fees describe fee
accounting for fee-bearing actions recorded inside that block.

References: [Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)

## Relationship to Network Context

A block is read inside a specific Bittensor network environment. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet as deployment contexts, and block history belongs to
whichever network environment produced it.

For readers, this keeps block records network-scoped rather than network-agnostic. A mainnet block,
a testnet block, and a localnet block all describe the same block concept, but each one is read
against the network context it belongs to.

References: [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)

## Relationship to Subtensor

Blocks are recorded on the Subtensor chain layer. The official
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor) defines
Subtensor as Bittensor's layer 1 blockchain and system of record for transactions, while the
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block) describes a block as
containing a collection of transactions with a unique block hash.

For readers, Subtensor names the blockchain layer, while a block names one ordered unit of chain
data recorded inside that layer.

References: [Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)

## Relationship to Multiple Mechanisms

A block records accepted chain actions on Subtensor. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-matrix) notes that each
incentive mechanism has its own weight matrix for independent consensus calculations.

For readers, a block still names chain ordering and inclusion. Weight-related actions inside a block
should be read with per-mechanism matrix context when relevant.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)

## Relationship to Coinbase

A block and coinbase are related through the timing mechanism that drives Bittensor's per-block
reward flow. A block is the fundamental unit of chain data—each block groups a collection of
transactions and a unique block hash—while coinbase is the per-block execution that injects
liquidity into subnet pools, accumulates pending emissions, and triggers each subnet's Yuma
Consensus epoch. The
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
documentation describes coinbase as a process that runs every block, making the block both the
data-record unit and the timing trigger that activates the coinbase mechanism once per chain step.

For readers, a block names the data record side of each chain step—what transactions were accepted
and ordered on Subtensor—while coinbase names the emission side of that same step: what liquidity
was injected and which subnet epochs were triggered. Both happen once per block, but they describe
different aspects of what each block produces. A claim about chain history or transaction ordering
belongs to the block concept, while a claim about per-step emission accumulation or epoch triggering
belongs to coinbase.

References:
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)

## Reading a Block Reference

A block reference is strongest when it is used to point at an ordered unit of Bittensor chain data.
The official glossary defines a block as containing transactions and a unique block hash, while the
chain-inspection documentation describes looking at chain state, blocks, and runtime metadata
through an explorer. Together, those sources make a block useful as evidence for chain-visible
records, not as a general explanation of why those records exist.

The useful question is usually narrow: what did this network record at this point in chain history?
A block citation can support a claim about the existence or ordering of included chain activity, but
it should not be stretched into a claim about motive, wallet intent, future state, or every related
off-chain event.

References: [Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)

## Included Records and Other Evidence

Blocks sit near several other evidence surfaces. Extrinsics name submitted chain actions, events
record emitted outcomes around chain activity, and storage entries expose state that can be queried.
A block can help locate where chain activity was included, but the exact claim determines which
evidence type is most relevant.

For example, a claim that an action was submitted belongs close to extrinsic vocabulary. A claim
that an event was emitted belongs close to event evidence. A claim about current state belongs close
to storage or chain-read evidence. A block gives those records a place in chain history, but it does
not make every evidence type interchangeable.

References: [Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Network and Timing Scope

Block claims should keep their network context visible. The Bittensor Networks documentation
separates mainnet, testnet, and localnet, and the existing block-timing context belongs inside that
environment framing. A block observed on one network is not automatically evidence for another
network.

Timing language should be handled the same way. The article can explain the general block concept
and cite official network timing references, but it should not invent live chain-height claims or
pretend a specific historical block is current. When precision matters, the safer prose names the
network and the block context being discussed.

References: [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)

## Inclusion Versus Submission

Submission and inclusion are related but separate reading steps. The mempool article describes the
pre-inclusion waiting area for pending transactions, while the block glossary describes blocks as
containing collections of transactions. That means a submitted action should not be treated as
included until the chain record supports that reading.

This distinction is helpful without turning the article into an operational guide. A reader can use
block vocabulary to understand where a chain action appears in recorded history, while extrinsic and
mempool vocabulary explain other stages around submission and waiting. The terms should reinforce
one another rather than collapse into a single generic "transaction happened" label.

References: [Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)

## What a Block Claim Does Not Prove

A block record does not, by itself, explain whether a participant made a good decision, whether a
future action will succeed, or whether a subnet's later state will remain the same. It is a chain
record boundary. Other claims need sources that match their subject: fees for cost claims, events
for emitted records, storage for current state, and official network references for environment
context.

Keeping the boundary narrow makes block references more useful. They can anchor chain-visible facts
without becoming a catch-all substitute for transaction analysis, wallet procedures, subnet
evaluation, or live network monitoring.

References: [Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Reader Boundary

This article defines the term at a high level and explains how to interpret block references. It
does not describe transaction submission, endpoint details, current chain height, or any specific
block record. Readers should use official network references for network-specific timing context.
