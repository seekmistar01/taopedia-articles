# Subtensor Events

Subtensor events are named records emitted by Bittensor's Subtensor runtime
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

The term belongs to runtime-record vocabulary. It should be read separately from storage entries and
submitted runtime actions.

## What Events Represent

Events identify named records emitted during runtime processing. An event entry gives vocabulary for
a record, but it does not by itself explain every rule that led to the record
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

For readers, the event name is evidence that a type of runtime record exists. Mechanism
documentation is still needed when the article explains why the record matters.

That boundary keeps the event reference from becoming a full mechanism explanation. The events page
names record vocabulary; concept sources explain the rules and consequences around those records
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

## Runtime Areas

Event names are grouped by runtime area. The runtime area is part of the event's meaning because a
record under one area can carry different context from a similar record under another area
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Reading the area and event name together is more precise than reading the event name alone.

Runtime-area context also improves citation quality. A source-backed statement about one event area
should stay attached to that area rather than becoming a claim about every Subtensor event
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

## Definitions and Emitted Records

The events reference describes event vocabulary. A chain-inspection surface can show a specific
emitted record in a selected network and block context
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

This keeps vocabulary separate from observation. The reference names the event type; an emitted
record belongs to the history that produced it.

That distinction matters because an event definition and an observed emitted record support
different claims. The definition supports vocabulary; the observed record supports a claim about a
particular chain context ([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

## Relationship to Storage and Extrinsics

Storage, events, and extrinsics describe different runtime surfaces. Storage exposes state, events
name records emitted during processing, and extrinsics name submitted runtime actions
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

A single feature can involve all three surfaces. Keeping them separate prevents an event name from
being treated as the full lifecycle of a feature.

For readers, this keeps evidence types from drifting together. Storage can support a claim about
exposed state, an event can support a claim about emitted runtime activity, and an extrinsic can
support a claim about a submitted action
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

The separation is especially useful when an article explains a failed or successful action. The
submitted action, emitted record, and resulting state surface are connected, but each one is a
different source of evidence.

## Network and Block Scope

Events are interpreted inside chain history. The same event name can appear across network
environments, but emitted records belong to the chain context where they were produced
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

For article claims, event vocabulary is reusable, while observed event records should stay tied to
their runtime and network context.

Examples should also keep their environment label attached. A localnet or testnet event example can
illustrate event vocabulary, but it should not be treated as evidence for another network
environment ([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

That framing is enough for a concept article. Subtensor events define emitted-record vocabulary,
while specific emitted records remain tied to the environment and block context where they were
observed.

## Reader Boundary

Subtensor events should be read as runtime-record vocabulary. They are not proof of off-chain
intent, not a complete mechanism explanation, and not a substitute for storage or extrinsic context
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

The stable article-level point is that Subtensor events name records emitted during runtime
processing.

The article should therefore avoid presenting event names as instructions. The source-backed
definition is about the runtime record vocabulary, not about which event a reader should search for
in a particular decision.

It should also avoid treating an event name as proof of intent outside the runtime. Events are
records emitted by the chain layer; broader explanations still need the relevant concept or
mechanism source ([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).
