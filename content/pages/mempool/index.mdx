# Mempool

The Bittensor mempool is the pending-transaction holding area before block inclusion. It names the
state where unconfirmed transactions wait before they become part of a block
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

The useful distinction is pending versus included. A mempool entry is not yet final chain history; a
block is the unit that records included transactions.

## Pending, Included, and Observed States

Mempool language is useful because it separates several nearby transaction states. The glossary
defines mempool as the pending-transaction area before block inclusion, while block vocabulary names
the unit that records included transactions
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

| State                | What the reader can say                              | What the reader should not infer                          |
| -------------------- | ---------------------------------------------------- | --------------------------------------------------------- |
| Submitted extrinsic  | A runtime action was sent toward chain processing    | It has not necessarily become block history.              |
| Mempool entry        | The transaction is pending before inclusion          | It is not final evidence of success.                      |
| Included transaction | A block contains the transaction record              | The mempool stage no longer describes it.                 |
| Event record         | The runtime emitted follow-up evidence               | The event is not the same object as the pending entry.    |
| Storage state        | Public chain state can be inspected after processing | A storage value is not proof of what was pending earlier. |
| Fee context          | A state-changing action may carry cost               | Fee vocabulary does not define pending status.            |

The table keeps timing and evidence separate. An extrinsic names the submitted action surface,
mempool names the waiting area, and block vocabulary names the included chain-history surface
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

Events and storage answer later questions. Event references describe records emitted by the runtime,
and storage references describe public state that can be inspected; neither source turns the
pre-inclusion mempool into final chain history
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Fee context is another neighboring layer. Transaction-fee documentation belongs to the cost of
state-changing actions, while mempool vocabulary belongs to the pending interval before inclusion
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)).

That distinction helps safety wording as well. MEV Shield documentation is concerned with sensitive
pending transaction details, but MEV Shield is a protection context rather than a synonym for the
mempool itself ([MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield/),
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)).

For readers, the safest phrasing is stage-specific. Use mempool when the claim is about pending
transactions before block inclusion, use block when the claim is about included chain history, and
use events or storage when the claim is about evidence produced after runtime processing.

The same rule applies when a transaction later fails. A failure signal belongs to runtime outcome
evidence, while mempool still names the earlier pending stage before the chain reached that outcome
([Subtensor Error Codes](https://docs.learnbittensor.org/errors),
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)).

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For mempool
examples tied to netuid 1 stake, weight, or fee extrinsics, that sequence changes how readers should
weigh pending-transaction evidence
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

In localnet, mempool behavior on netuid 1 can be tested in an isolated environment. Local pending
pools reflect local configuration and do not represent mainnet ordering or MEV conditions.

On testnet, mempool state on netuid 1 can be observed in a shared non-production network. Testnet
pending extrinsics and inclusion timing are separate from mainnet Subtensor history
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

On mainnet, mempool entries for netuid 1 actions carry production pending-state meaning before they
become included chain evidence on the live Bittensor network
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)).

A mempool example from one environment should not be treated as proof about another environment.

## Distinction from Block

The mempool holds pending transactions; a block records which transactions were included at one
height ([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

Readers often treat submission as final history. Mempool vocabulary marks the waiting stage; block
vocabulary marks the inclusion record.

## Distinction from Transfer

The mempool holds pending transactions awaiting block inclusion. A transfer is one such transaction,
a send of TAO from one wallet address to another, that waits in the mempool before it is included
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Transfer](https://docs.learnbittensor.org/resources/glossary#transfer)).

Queue vocabulary names where transactions wait; transfer vocabulary names one action that waits
there.

## Distinction from Extrinsics

An extrinsic is a submitted runtime action. The mempool can hold that action while it waits;
inclusion in a block is what makes it part of chain history
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)).

Extrinsic names the action type; mempool names the pre-inclusion waiting state for that action.

## Distinction from Fees

The mempool holds pending transactions awaiting inclusion. Transaction fees are the cost attached to
submitted extrinsics
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Transaction Fees](https://docs.learnbittensor.org/resources/glossary#transaction-fees)).

- **Mempool** — pending transaction queue before inclusion.
- **Transaction fees** — cost attached to submitted extrinsics.

Queue vocabulary and fee vocabulary answer different transaction questions
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

## Distinction from Yuma Consensus

The mempool is the pending-transaction layer before block inclusion. That pre-inclusion queue is
separate from the subnet mechanism that aggregates validator weights into emission shares each tempo
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Extrinsics such as weight submissions pass through mempool and block production before Yuma can read
them on chain. Mempool vocabulary names visibility before inclusion; Yuma names settlement after
weights are recorded ([Emission](https://docs.learnbittensor.org/learn/emissions)).

Transaction fees and inclusion timing belong to the chain layer. Miner incentives and validator
dividends still follow the documented weight-aggregation path on the selected netuid
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

Readers should not treat mempool visibility as consensus outcome. Pending weights are not settled
shares until the tempo pass runs on included state
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Mempool** — pre-inclusion pending transaction layer.
- **Yuma Consensus** — validator-weight aggregation for emissions.

## Distinction from Mempool Visibility

The mempool is the queue of pending transactions. Mempool visibility is whether those pending
details can be observed before ordering, which matters for MEV risk
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield)).

- **Mempool** — pending transactions awaiting inclusion.
- **Mempool visibility** — whether pending details can be observed before ordering.

Mempool visibility does not change whether a transaction is pending; it changes how much of that
pending state other participants can see before inclusion.

## Distinction from Batch Transactions

Batch transactions package several inner calls into one outer transaction. The mempool is the queue
where that outer transaction waits before inclusion
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

- **Mempool** — pending transactions awaiting inclusion.
- **Batch transaction** — one outer submission wrapping multiple inner calls.

Readers should keep pending-transaction vocabulary separate from settlement vocabulary. The mempool
holds extrinsics waiting for inclusion; Yuma names how included validator weight submissions are
aggregated into emission shares each tempo
([Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Mempool ordering concerns which extrinsics reach a block first. Yuma concerns how validator weights
already on chain are processed into miner and validator dividends
([Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Extrinsic** — signed action submitted to Subtensor for inclusion.

## Reader Boundary

Mempool does not name a transaction tutorial, monitoring guide, source for final chain status, or
guarantee that a pending transaction will be included
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

Mempool identifies pending transaction state before block inclusion, while block records,
extrinsics, fees, MEV Shield, and network context describe neighboring parts of the system.

## Distinction from Transaction Fees

A mempool and transaction fees address where a transaction waits versus what it costs. The mempool
is the staging area holding submitted extrinsics before they are included in a block, while
transaction fees are the amount paid to process an extrinsic, which can influence how quickly it
leaves the mempool ([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

- **Mempool** — the staging area of pending extrinsics awaiting inclusion.
- **Transaction fees** — the amount paid to process an extrinsic.
