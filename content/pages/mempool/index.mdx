# Mempool

The Bittensor mempool is the pending-transaction holding area before block inclusion. It names the
state where submitted, unconfirmed transactions wait before they become part of a block
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

For Taopedia readers, the useful distinction is pending versus included. A mempool entry is not yet
final chain history; a block is the unit that records included transactions.

## Pending Transaction Context

Mempool vocabulary belongs to the interval after submission and before block inclusion. During that
interval, a transaction is waiting to be processed into chain history
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

This makes the mempool a timing and state concept rather than a reward mechanism. It explains where
pending transactions sit, not whether the submitted action will ultimately succeed.

## Relationship to Blocks

Blocks are the chain units that contain transactions. The mempool is the waiting area before that
inclusion happens ([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)).

Once a transaction is included in a block, the reader should treat it as block history rather than
as a merely pending mempool entry.

## Relationship to Extrinsics

Extrinsics are submitted runtime actions. In Bittensor context, a submitted extrinsic can pass
through the pending-transaction stage before block inclusion
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)).

This keeps the concepts separate. Extrinsic names the submitted action; mempool names the pending
state before inclusion.

## Relationship to Fees

Some submitted actions carry transaction fees. The fee model belongs to the action being processed,
while the mempool names the pending area where unconfirmed submissions wait
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)).

For readers, the mempool does not set fee rules. It sits before the block-processing stage where the
runtime evaluates submitted actions.

## Network Context

Bittensor network vocabulary separates mainnet, testnet, and localnet. A mempool should be read
inside the network context where the transaction was submitted
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)).

The same mempool concept can apply across environments, but each environment has its own pending
transaction context.

## Reading the Term

Mempool should be read as high-level transaction vocabulary. It is not a tutorial for submitting
transactions, a monitoring guide, or a source for public chain status
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

The stable article-level point is that the mempool holds pending transactions before block
inclusion.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Mempool, this
sequence gives readers a boundary for interpreting pending-transaction examples and block-inclusion
observations.

Localnet examples are isolated and reflect local chain state, so they are useful for controlled
experiments rather than evidence of live Bittensor behavior. Testnet examples add shared
non-production conditions, which can reveal integration behavior without touching mainnet state.

On mainnet, Mempool examples should be read as live production transaction behavior on the
production Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, so outcomes from one environment should not be treated as
proof of behavior in another.

## Pending State Scope

Mempool language should be kept separate from finalized chain state. Transaction and block
references place submitted activity into chain processing, while MEV Shield documentation focuses on
protecting sensitive pending transaction details before inclusion
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield/)).

For readers, a pending transaction is not the same as a completed on-chain result. Claims about
mempool visibility should avoid implying final execution, final fee outcome, or finalized state
until block inclusion is established.
