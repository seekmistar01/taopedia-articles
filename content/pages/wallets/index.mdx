# Bittensor Wallets

A Bittensor wallet represents a user or entity's identity and signing authority on the network. It
stores cryptographic keys used to access TAO, delegate stake, and authorize operations
([Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets)).

- A wallet contains or references two key roles: a security-critical coldkey, and one or more
  operational hotkeys.
- Wallet operations include creating/importing keys, viewing balances, staking/unstaking, and
  registering participants.

References: [Wallets (learnbittensor.org)](https://docs.learnbittensor.org/keys/wallets),
[BTCLI Reference](https://docs.learnbittensor.org/btcli)

## What a wallet does

- Proves identity for transactions and on-chain actions.
- Controls TAO and stake (via the coldkey) and runs roles like miners/validators (via hotkeys).
- Interfaces with CLI (`btcli`) or SDKs to manage keys and send transactions.

## Wallet Identity and Applications

A Bittensor wallet is the identity-and-key layer, while a wallet application is the software layer
used to work with that identity. The
[Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets) documentation
describes a wallet as the structure built around coldkeys and hotkeys, and the
[BTCLI reference](https://docs.learnbittensor.org/btcli) provides one interface for wallet-related
actions. The wallet and the tool are related, but they are not the same concept.

This distinction matters because Bittensor articles often mention wallets next to CLI commands,
SDKs, browser tools, hardware wallets, and local wallet files. Those tools can expose balances,
prepare transactions, or request signatures, but the wallet concept remains the key-backed identity
and authority model underneath them.

Coldkeys and hotkeys also keep wallet authority separated inside that identity model. The coldkey
controls TAO, stake, and high-authority ownership actions. Hotkeys support operational activity for
miners, validators, and subnet participation. A wallet article therefore names the container and key
structure, while coldkey and hotkey articles explain the different authority roles inside it.

Storage labels belong to a different layer again. A local wallet, an external wallet, or a hardware
wallet describes where key material or signing capability is managed. Those labels can affect
security practice, but they do not create a new network identity separate from the coldkey and
hotkey relationship described in the wallet documentation.

Importing or moving wallet files changes which interface can access signing material; it should not
be read as creating a separate on-chain wallet identity
([Working with Keys](https://docs.learnbittensor.org/keys/working-with-keys)).

For readers, the safest interpretation is layered: wallet names identity and signing authority,
wallet applications provide interfaces, and storage terms describe where sensitive key material is
kept or used. Keeping those layers separate prevents a tool name, file location, address, and
private key from being treated as interchangeable wallet terms.

References: [Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets),
[Working with Keys](https://docs.learnbittensor.org/keys/working-with-keys),
[BTCLI Reference](https://docs.learnbittensor.org/btcli)

## Managing a wallet (conceptual)

- Create or import a wallet using the CLI or SDK.
- Securely store the seed phrase (mnemonic) offline.
- Generate a hotkey for operational roles.

## Security guidelines

- Keep seed phrases private and offline.
- Verify you are using official documentation and released binaries.
- Use hardware-backed or dedicated secure storage for the coldkey if possible.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Wallets, this
sequence gives readers a boundary for interpreting wallet examples and network-selection notes.

Localnet examples are isolated and reflect local chain state, so they are useful for controlled
experiments rather than evidence of live Bittensor behavior. Testnet examples add shared
non-production conditions, which can reveal integration behavior without touching mainnet state.

On mainnet, Wallets examples should be read as live production wallet behavior on the production
Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, so outcomes from one environment should not be treated as
proof of behavior in another.

## Coldkey Public Key Names the Wallet

The official [Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets) guide
describes the cryptographic wallet as closely tied to the coldkey that controls assets and serves as
public identity. The coldkey public key uniquely identifies that wallet on the network and acts as
its address for transactions.

That public-side material is what senders need when moving TAO toward a holder. The
[Glossary: Wallet Address](https://docs.learnbittensor.org/resources/glossary#wallet-address)
defines a wallet address as a public-key-derived identifier used for sending and receiving TAO,
which matches the coldkey-facing role in wallet documentation.

Hotkey public keys still matter for subnet-facing identity, but fund receipt vocabulary usually
centers on coldkey address material rather than on operational hotkey labels.

References: [Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets),
[Glossary: Wallet Address](https://docs.learnbittensor.org/resources/glossary#wallet-address)

## On-Chain Wallet State Follows Key Pairs

Official wallet documentation describes a wallet as the on-chain records tied to coldkey and hotkey
key pairs, including TAO and alpha balances and transaction history
([Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets)).

Those records persist independently of any one wallet application install. The same cryptographic
wallet can be loaded into different software or hardware interfaces while referring to the same
on-chain identity centered on the controlling coldkey
([Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets)).

A missing local wallet file therefore changes access to signing material, not the existence of the
underlying on-chain wallet records themselves.

References: [Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets),
[Glossary: Bittensor Wallet](https://docs.learnbittensor.org/resources/glossary#bittensor-wallet)

## TAO Custody Belongs to the Coldkey

Official wallet guidance assigns TAO balances, stake management, and high-authority ownership
actions to the coldkey. Hotkeys support mining, validation, and other subnet operations that need
operational signing rather than fund custody
([Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets)).

In the intended wallet model, hotkeys should not hold TAO even when a transfer to a hotkey address
might be technically possible. Operational keys are meant to stay on connected nodes, while the
coldkey keeps custody of TAO and stake
([Glossary: Hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey)).

Wallet-level questions about unstaked TAO therefore belong with coldkey vocabulary until a separate
staking or delegation action attaches value to a validator hotkey on a subnet.

References: [Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets),
[Glossary: Hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey)
