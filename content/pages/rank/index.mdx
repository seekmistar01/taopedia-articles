# Rank

Rank is a Bittensor consensus metric for summarizing miner evaluation inside a subnet. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#rank) describes rank as the final
aggregate judgment produced by Yuma Consensus from validator ratings, while
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) explains how validator
evaluations are filtered and combined into miner emission vectors.

## Consensus Role

Rank belongs to the miner-evaluation side of Yuma Consensus. It is not a single validator's raw
view, and it is not a general reputation score for every Bittensor activity. It is the aggregate
result after the consensus process compares validator weight signals and limits outlying values.

That makes rank useful as a compact answer to one question: after validator evaluation and consensus
filtering, where did the miner land in the subnet's evaluation result? The
[Glossary](https://docs.learnbittensor.org/resources/glossary#rank) ties the term directly to Yuma
Consensus, and the [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) page
places that result inside the broader flow from validator evaluation to miner emissions.

## From Weights to Rank

Validator weights are input signals; rank is a post-aggregation output. The
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)
describes the weight values validators use to evaluate miners, while
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank) describes the final
aggregate judgment produced from those ratings.

This separation keeps the vocabulary clear. A weight vector represents one evaluator's signal about
miners. Rank represents the consensus-adjusted result built after those signals are compared,
filtered, and aggregated by Yuma Consensus.

## Rank, Trust, and Validator Trust

Rank, trust, and validator trust are related consensus terms, but they do not describe the same
thing. Rank names the miner-side aggregate result. The
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust) describes how much a
miner's pre-rank outcome changes after clipping, and
[Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)
describes how much of a validator's weight signal remains after clipping.

A plain reading is enough: rank says how the miner was evaluated after aggregation, while trust
terms help explain how much the clipping step changed the miner-side result or validator-side
signal. They are best read as complementary views of the same consensus process.

## Emission Context

Rank matters because Yuma Consensus uses aggregate miner evaluation when calculating miner
emissions. The [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) documentation
connects validator rankings, stake weighting, clipping, and emission calculation in one process.

The metric should still be read with care. Rank is not a free-standing promise of rewards, and it
does not replace the surrounding consensus machinery. It is a miner-side output inside a process
that also considers validator stake, consensus clipping, and incentive flow.

## Subnet Context

Rank is meaningful inside the subnet and incentive context that produced it. The
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) page explains
that subnets define their own work and evaluation roles. The
[Multiple Incentive Mechanisms](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)
documentation adds that performance in one mechanism does not affect rating in another mechanism.

For that reason, a rank value should not be treated as a universal quality score across unrelated
subnet tasks. It is best read as a consensus-adjusted outcome for a miner in the relevant subnet
evaluation path.

## Reading Boundaries

Rank is the right term when the focus is the final miner-side aggregate judgment. If the question is
about the benchmark used for clipping,
[Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score) is the closer
term. If the question is about how much clipping changed an outcome, trust or validator trust is
more precise.

These boundaries make the article easier to use: rank explains the outcome, validator weights
explain the inputs, consensus score explains the clipping benchmark, and trust terms explain how
much clipping changed the signals.
