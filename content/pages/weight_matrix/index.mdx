# Weight Matrix

A weight matrix is the subnet-level collection formed from validator weight vectors. It organizes
the ranking weight vectors submitted by subnet validators so Yuma Consensus can process them as
structured consensus input
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The term separates individual validator submissions from the combined input that consensus operates
on. A single vector is one validator's evaluation; the matrix is the collection of those vectors.

## Structured Input Context

Each validator weight vector represents one validator's relative evaluation of miner work. The
weight matrix arranges those vectors together so consensus can compare and aggregate validator
signals
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)).

That makes the matrix an input structure, not a separate scoring rule. It exists so Yuma Consensus
can work with many validator submissions at once.

## Yuma Consensus Boundary

Yuma Consensus uses validator signals to produce subnet incentive outcomes. The weight matrix is the
organized input to that process; Yuma Consensus is the aggregation process that filters, bonds, and
turns those inputs into outcomes
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

For readers, this keeps matrix language separate from result language. The matrix is not the final
rank, dividend, or incentive result.

## Consensus Score Boundary

The weight matrix and consensus score describe different parts of the same Yuma path. The matrix
collects validator weight vectors, while consensus score is the stake-weighted threshold used to
filter outlier submissions before final rank calculations
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)).

That distinction prevents the input collection from being confused with the filtering benchmark. The
matrix supplies the set of signals; the consensus score is derived during filtering.

## Timing and Visibility Context

Tempo and Commit Reveal add timing context around validator weights. Tempo provides the block-based
cadence for subnet epochs, while Commit Reveal can delay visibility of fresh validator weight
submissions ([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

Those timing concepts do not replace the matrix. They explain when related validator-weight signals
are processed or revealed, while the weight matrix names the structured collection itself.

## Multiple-Mechanism Context

When a subnet uses multiple incentive mechanisms, each mechanism can have its own weight matrix for
independent consensus calculations. The mechanism context tells readers which evaluation path the
matrix belongs to
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)).

A matrix for one mechanism should not be treated as the matrix for another mechanism. The term still
means a validator-weight collection, but the relevant mechanism is part of the meaning.

## Reader Boundary

Weight matrix should be read as consensus input vocabulary. It is not an operating guide, a public
quality label, or a complete explanation of reward outcomes
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The stable article-level point is that the matrix organizes validator weight vectors for consensus.
Everything downstream depends on how Yuma Consensus filters and aggregates that organized input.
