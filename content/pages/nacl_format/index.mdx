# NaCl Format

NaCl Format is Bittensor terminology for an encryption format used around legacy wallet files
([Glossary: NaCl Format](https://docs.learnbittensor.org/resources/glossary#nacl-format)).

The term belongs to wallet-file security vocabulary. It describes a protection format around stored
wallet material, not a subnet role, tokenomics concept, consensus mechanism, or network-status
label.

## Format Scope

NaCl Format names a file-format protection layer. Bittensor wallet documentation provides the
broader wallet setting, while the NaCl glossary term narrows the subject to legacy wallet-file
encryption ([Bittensor wallet documentation](https://docs.learnbittensor.org/keys/wallets),
[Glossary: NaCl Format](https://docs.learnbittensor.org/resources/glossary#nacl-format)).

That scope is narrower than wallet behavior. A wallet file can have a protection format without the
format saying what balance, subnet role, or staking state is associated with the wallet.

## Stored Material Boundary

NaCl Format should be read as stored-material vocabulary. It concerns the protection format around a
wallet file, not every security property of the environment where that file exists
([Glossary: NaCl Format](https://docs.learnbittensor.org/resources/glossary#nacl-format),
[Bittensor wallet documentation](https://docs.learnbittensor.org/keys/wallets)).

That boundary prevents overstatement. A protected legacy file format can limit what the file itself
reveals, but it does not replace private-key custody, workstation security, or source-specific
wallet guidance.

## Seed Phrase Boundary

NaCl Format and seed-phrase handling describe different wallet-safety layers. NaCl Format concerns
the protection format around stored legacy wallet files, while seed-phrase guidance concerns the
recovery words that can regenerate wallet key material
([Glossary: NaCl Format](https://docs.learnbittensor.org/resources/glossary#nacl-format),
[Handle your Seed Phrase/Mnemonic Securely](https://docs.learnbittensor.org/keys/handle-seed-phrase)).

That distinction keeps file-format language from sounding like a backup plan. A protected wallet
file and a protected seed phrase both belong to wallet security, but they answer different recovery
and storage questions.

## Key-Regeneration Boundary

NaCl Format and regenerating a key both sit near wallet recovery, but they answer different
questions. NaCl Format names the legacy encryption format around stored wallet material, while
regenerating a key names recreating a lost or deleted coldkey or hotkey from the associated mnemonic
([Glossary: NaCl Format](https://docs.learnbittensor.org/resources/glossary#nacl-format),
[Glossary: Regenerating a Key](https://docs.learnbittensor.org/resources/glossary#regenerating-a-key)).

For readers, the format protects local wallet material at rest; the recovery operation depends on
separate recovery words and current key-handling guidance. A NaCl-format file should therefore not
be treated as the same thing as a mnemonic, a regenerated key, or proof that recovery material has
been preserved.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For NaCl Format,
that sequence mainly matters when legacy wallet-file examples are drawn from different network
environments during development or testing.

In localnet, NaCl Format examples can be exercised with local wallet files in an isolated
environment. Local file-protection behavior does not represent production wallet custody on live
networks.

On testnet, legacy wallet files used in shared testing can demonstrate format compatibility under
non-production conditions while staying separate from mainnet wallet state
([Bittensor wallet documentation](https://docs.learnbittensor.org/keys/wallets)).

On mainnet, NaCl Format vocabulary still describes legacy wallet-file protection on the production
network context where those files are used
([Glossary: NaCl Format](https://docs.learnbittensor.org/resources/glossary#nacl-format)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A wallet-file example from one environment should not be
read as representing production file custody in another environment.

## Legacy Files Use NaCl-Style Encryption at Rest

The [Glossary: NaCl Format](https://docs.learnbittensor.org/resources/glossary#nacl-format)
describes NaCl Format as encryption used around legacy Bittensor wallet files. The term therefore
belongs to local file-protection vocabulary rather than to on-chain roles or subnet participation
labels ([Bittensor wallet documentation](https://docs.learnbittensor.org/keys/wallets)).

Readers should keep NaCl Format on the stored-file side of wallet security. It names how some legacy
wallet material is protected at rest, not what that material can do after unlock.

## Format Protection Does Not Authorize Transactions

[Working with Keys](https://docs.learnbittensor.org/keys/working-with-keys) documentation covers how
keys are used once available, while NaCl Format names only the legacy wrapper around stored wallet
files. A protected file format can limit casual disclosure of file contents; it does not replace
private-key handling or transaction review
([Glossary: Private Key](https://docs.learnbittensor.org/resources/glossary#private-key)).

Authorization still follows whichever key material signs an action after the wallet is unlocked.

## Distinction from Encrypting the Hotkey

NaCl format names an encryption format used around legacy wallet files. Encrypting the hotkey is the
operation that protects a hotkey's key material
([Glossary: NaCl Format](https://docs.learnbittensor.org/resources/glossary#nacl-format),
[Glossary: Encrypting the Hotkey](https://docs.learnbittensor.org/resources/glossary#encrypting-the-hotkey)).

Format vocabulary describes how stored bytes are written; encryption-operation vocabulary names when
protection is applied.

## Distinction from Local Wallet

Local wallet names wallet files kept on the user's machine. NaCl format names how some legacy wallet
files may be encrypted on disk
([Glossary: Local Wallet](https://docs.learnbittensor.org/resources/glossary#local-wallet),
[Glossary: NaCl Format](https://docs.learnbittensor.org/resources/glossary#nacl-format)).

Wallet-type vocabulary and file-format vocabulary answer different security questions. For article
use, the local-file context should stay attached: a NaCl-format statement is about the stored file
format, not about a remote account database, subnet-role record, wallet balance, or runtime
authorization event. That also keeps NaCl Format separate from wallet-location vocabulary, since
wallet location identifies where local wallet files are stored while NaCl Format identifies the
legacy protection format around stored material
([Glossary: Wallet Location](https://docs.learnbittensor.org/resources/glossary#wallet-location),
[Glossary: NaCl Format](https://docs.learnbittensor.org/resources/glossary#nacl-format)).

Legacy context matters because wallet vocabulary can mix several layers: the key material, the local
file, the account context, and the protection format around stored material. NaCl Format belongs to
the file-protection layer
([Bittensor wallet documentation](https://docs.learnbittensor.org/keys/wallets),
[NaCl Format glossary entry](https://docs.learnbittensor.org/resources/glossary#nacl-format)).

This makes the term useful for compatibility reading. It explains how a legacy wallet file is
protected at rest, while other wallet and key terms explain how the material is used or referenced.

## Distinction from Mnemonic

NaCl Format names legacy encryption around stored wallet files. A mnemonic is the recovery phrase
used to regenerate wallet keys after loss
([Glossary: NaCl Format](https://docs.learnbittensor.org/resources/glossary#nacl-format),
[Glossary: Mnemonic](https://docs.learnbittensor.org/resources/glossary#mnemonic),
[Handle Seed Phrase](https://docs.learnbittensor.org/keys/handle-seed-phrase)).

- **NaCl Format** — file-at-rest encryption format for legacy wallet files.
- **Mnemonic** — recovery words that can regenerate keys after loss.

File-format vocabulary and recovery vocabulary answer different wallet-security questions
([Bittensor wallet documentation](https://docs.learnbittensor.org/keys/wallets)).

NaCl Format concerns legacy wallet-file encryption, while
[Handle Seed Phrase](https://docs.learnbittensor.org/keys/handle-seed-phrase) guidance covers
mnemonic recovery material. The two belong to wallet security but answer different questions: file
format at rest versus regeneration after loss
([Glossary: Mnemonic](https://docs.learnbittensor.org/resources/glossary#mnemonic)).

Readers should not treat a NaCl-format label as interchangeable with seed-phrase backup vocabulary.

## Distinction from SS58 Encoded

The NaCl format and SS58 encoding name two different things about a wallet. The NaCl format is the
legacy encryption format that protects stored wallet-file material at rest, while SS58 encoding is a
compact representation of the public keys corresponding to a wallet's coldkey and hotkey, used as
the wallet addresses for TAO transfers
([Glossary: SS58 Encoded](https://docs.learnbittensor.org/resources/glossary#ss58-encoded)).

- **NaCl format** — the encryption wrapper around stored legacy wallet files.
- **SS58 encoded** — the address encoding of a wallet's public keys.

## Distinction from EdDSA Cryptographic Keypairs

The NaCl format and EdDSA cryptographic keypairs name different layers of wallet cryptography. The
NaCl format is the encryption format used to protect stored legacy wallet files at rest, while EdDSA
cryptographic keypairs is the algorithm that generates the public and private key pairs for the
coldkeys and hotkeys in a Bittensor wallet
([Glossary: EdDSA Cryptographic Keypairs](https://docs.learnbittensor.org/resources/glossary#eddsa-cryptographic-keypairs)).

- **NaCl format** — how stored wallet-file material is encrypted at rest.
- **EdDSA cryptographic keypairs** — the algorithm that generates the wallet's coldkey and hotkey
  pairs.

## Distinction from Yuma Consensus

NaCl format describes the encoded key representation Bittensor tooling uses for certain wallet key
exports. It names key-encoding vocabulary, not the on-chain step that turns validator weight
submissions into emission shares
([Glossary: NaCl Format](https://docs.learnbittensor.org/resources/glossary#nacl-format),
[Wallets](https://docs.learnbittensor.org/keys/wallets)).

Yuma Consensus is the on-chain process that converts submitted validator weights into emission
shares after evaluation completes. Key encoding affects local file formats; Yuma Consensus settles
scoring outcomes on a subnet
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers should keep key-format vocabulary separate from the settlement step. NaCl format answers how
keys are serialized for tooling; Yuma Consensus answers how validator weights become emission shares
in a subnet
([Glossary: NaCl Format](https://docs.learnbittensor.org/resources/glossary#nacl-format),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **NaCl format** — encoded key representation for wallet exports.
- **Yuma Consensus** — on-chain weight aggregation that settles emissions.

## Distinction from Key Vocabulary

NaCl Format sits near public-key and private-key terminology because wallet files protect key
material. Public keys are shareable reference material, while private keys are authorization
material that must remain protected
([Glossary: Public Key](https://docs.learnbittensor.org/resources/glossary#public-key),
[Glossary: Private Key](https://docs.learnbittensor.org/resources/glossary#private-key)).

The NaCl term does not rename either key type. It identifies the legacy protection format around
stored wallet material, while public-key and private-key vocabulary identify what the material is
used for.

## Reader Boundary

NaCl Format should be used narrowly in Taopedia prose. It can support statements about legacy
wallet-file protection and wallet-security terminology, but it should not be used to infer staking
status, subnet participation, balance state, validator status, or wallet-tool behavior
([NaCl Format glossary entry](https://docs.learnbittensor.org/resources/glossary#nacl-format),
[Bittensor wallet documentation](https://docs.learnbittensor.org/keys/wallets)).

The stable article-level point is that NaCl Format is format vocabulary. It does not name a recovery
process, wallet procedure, subnet action, address, public key, private key, or balance state.
