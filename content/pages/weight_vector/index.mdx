# Weight Vector

A weight vector is one validator's structured set of miner evaluations inside a Bittensor subnet.
Each element represents a weight assigned to a subnet miner according to that validator's evaluation
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The scope is narrow. A single vector is one validator's signal before aggregation, not the whole
subnet result. The individual values inside the set are validator weights
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

## Per-Validator Signal

A weight vector comes from one validator's evaluation process. It expresses how that validator
scores miners under the relevant subnet task and incentive mechanism
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

That keeps the term narrower than final rank or reward allocation. Consensus can use many vectors,
but each vector remains a per-validator input.

Yuma Consensus compares many validator signals. One vector expresses one validator's view; it does
not stand in for every validator's evaluation.

That makes vector language useful when the focus is the validator that supplied the structured
evaluation signal.

## Task and Mechanism Scope

The meaning of a weight vector depends on the subnet task. The incentive mechanism defines useful
work, and the vector records one validator's evaluation relative to that definition
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

This prevents cross-subnet overreach. A vector from one subnet's task does not automatically
describe performance under another subnet's task or scoring method.

The mechanism supplies the meaning of the entries. Without the subnet task and scoring rules, the
vector is only a structure; the incentive mechanism explains what the entries are evaluating.

The same vector shape can appear in different subnet tasks, but interpretation belongs to the task
and scoring model that produced the values.

## Value, Vector, and Matrix

Weight vector and validator weights describe closely related ideas. The vector is the per-validator
structure, while validator weights are the evaluation signals inside that structure
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

The useful distinction is scale. Validator-weight language is more precise for an individual score,
while weight-vector language is more precise for the whole structured set from one validator.

Weight vector and weight matrix are different levels of the same input vocabulary. A vector belongs
to one validator, while the matrix is the subnet-level collection of validator weight vectors
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

This distinction keeps one validator's view from being mistaken for the combined input set. The
matrix vocabulary names the grouped input; vector vocabulary names one validator's structured
evaluation set.

Consensus-based weights documentation uses the broader weight language for validator-submitted
signals. The vector term keeps that broader process grounded in one validator's submitted set
([Consensus-Based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

## Consensus and Filtering Terms

A weight vector is distinct from consensus score. The vector is an evaluation signal, while
consensus score is the stake-weighted benchmark used to filter outlier weights before final rank
results
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

That makes consensus score downstream of validator weight signals. A vector supplies input; the
consensus score helps constrain how those signals can influence the final result.

Rank is downstream as well. A vector supplies one validator's miner evaluations, while rank is the
consensus-adjusted miner-side result after signals are filtered and aggregated
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Submitted vectors are therefore pre-filtering inputs. Consensus score and rank appear after many
validator inputs are compared, clipped, and aggregated
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Consensus-Based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

## Stake Influence

Weight vectors are interpreted inside a stake-weighted consensus process. Stake weight helps
determine how much influence a validator's vector carries during consensus
([Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

This keeps evaluation and influence separate. The vector records miner evaluations; stake weight
helps determine how strongly that validator signal is counted.

Stake weight does not replace vector vocabulary. It describes influence around the signal, while the
vector describes the structured miner-evaluation input.

That split keeps the value and its influence separate. The vector records evaluations; stake-related
terms explain how much effect a validator's submitted signal can have.

## Visibility and Timing

Independent validator evaluation is what gives a weight vector useful information. Weight copying
weakens that purpose when validators rely on visible weights instead of fresh evaluation
([The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

Commit Reveal can add a timing layer by delaying visibility of fresh weight signals. That protects
timing around the information, but it does not change the definition of a vector.

## Weight Setting Builds One Validator's Vector

[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) describes each subnet
validator as periodically submitting a vector of weights that ranks miner work. Weight setting is
the validator-side process; the
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector) names
the structured score set that process produces.

That keeps process and structure separate. Weight setting explains what a validator does; a weight
vector names the per-validator miner scores that action creates.

## set_weights Carries the Vector On Chain

Bittensor documents weight submission as a dispatchable
[extrinsic](https://docs.learnbittensor.org/subtensor-api/extrinsics) call that carries a
validator's weight vector for a subnet. The vector term describes the submitted scores; the
extrinsic is how those scores reach chain state for later consensus processing
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Readers should not treat the vector as only an off-chain concept. On-chain submission is what makes
one validator's structured evaluation available to Yuma Consensus.

## Vectors Can Rank Only Evaluated Miners

[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) states that each validator
submits a vector ranking the value of work for each miner that validator has evaluated. A weight
vector therefore reflects observed miner work under the subnet task, not an automatic score for
every registered [UID](https://docs.learnbittensor.org/resources/glossary#uid-slot) on the subnet.

That scope keeps vector reading tied to evaluation. The structure holds one validator's miner
judgments, and those judgments apply to miners the validator actually scored.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For weight vector
examples on a subnet such as netuid 1, that sequence changes how readers should interpret examples
tied to subnet state, validator activity, or chain observations.

In localnet, weight vector examples can be tested in an isolated environment. Local chain state
reflects local configuration rather than production network behavior.

On testnet, weight vector can be observed in a shared, non-production network. Testnet readings on a
selected netuid are separate from mainnet state
([Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

On mainnet, weight vector applies on the live production Bittensor network for subnets such as
netuid 1 and the environment context being discussed.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A weight vector example from one environment should not be
read as representing production behavior on another network.

## Distinction from Weight Matrix

A weight vector is one validator's miner weight assignments in a single submission. A weight matrix
is the combined validator-to-miner assignment table across the validator set on the selected subnet
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)).

Weight-vector vocabulary names one submission. Weight-matrix vocabulary names the grouped input Yuma
reads before consensus aggregation.

## Distinction from Consensus Score

Consensus score is the stake-weighted benchmark used to clip outlier weights. A weight vector is the
per-validator input evaluated before that benchmark is formed on the selected subnet
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping)).

Weight-vector vocabulary belongs to submitted signals. Consensus-score vocabulary belongs to the
agreement benchmark derived from those signals.

## Distinction from Rank

Rank is the miner-side standing after validator weights are filtered and aggregated. A weight vector
is one validator's raw evaluation input before that aggregation on the selected subnet
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

Weight-vector vocabulary belongs to input signals. Rank vocabulary belongs to miner-side outcomes
after processing.

## Distinction from Stake Weight

Stake weight names how much validator influence each submitted weight carries in consensus math. A
weight vector names which miners a validator scored, not the stake backing that validator
([Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

Stake-weight vocabulary belongs to influence scaling. Weight-vector vocabulary belongs to evaluation
targets in one submission.

## Distinction from Weight Setting

Weight setting is the per-validator action that submits one weight vector. The weight vector is the
structured payload that action records on the selected subnet
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

Weight-setting vocabulary names the submission act. Weight-vector vocabulary names the evaluation
content inside that submission.

## Distinction from Trust

A weight vector is one validator's submitted evaluation signals, while trust is interpreted after
the full validator set is filtered and aggregated into miner-side outcomes
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

The weight vector is an input one validator submits; trust is a downstream miner-side reading of how
that evaluation survived aggregation.

## Distinction from Validator Trust

A weight vector is one validator's submitted evaluation signals, while validator trust summarizes
that validator's clipped influence after the full weight matrix is filtered
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)).

Weight-vector vocabulary belongs to the input signals; validator-trust vocabulary belongs to the
validator-side measure of how much of that input survived clipping.

## Reader Boundary

Weight vector names per-validator consensus-input vocabulary. It describes one validator's
structured miner-evaluation set before subnet-level matrix and consensus processing, not stake
weight, rank, or a complete reward model
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Specific examples belong to the subnet, mechanism, network, and source that produced the vector.
Validator weights, weight matrices, consensus score, rank, and stake weight describe neighboring
parts of the same consensus process.

## Relationship to Weight Matrix

Weight vector is one validator's list of miner weight assignments in a single submission. Weight
matrix is the combined table of validator-to-miner assignments across validators on a subnet
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)).

Yuma Consensus reads many validator vectors together when building subnet-wide scoring inputs.
Weight vector therefore names one submission's assignments, while weight matrix names the combined
picture on a netuid such as netuid 1
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Use weight vector when covering one validator's submission; use weight matrix when covering the
combined assignment table rather than a single vector
([Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)).

Single submission and combined table answer different reader questions in the same weight cycle
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## Relationship to Validator-Miner Bonds

A weight vector and validator-miner bonds describe a per-round signal and the persistent
relationship that forms from it. A weight vector is the set of weights one validator assigns to
miners in a single round, while validator-miner bonds are the smoothed connections that build up as
a validator repeatedly weights the same miners across rounds
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)).

The two terms answer different questions. The weight vector names the immediate evaluation a
validator submits, while bonds name the accumulated relationship that consensus uses to map miner
outcomes back to validators, so a single weight vector is the input and bonds are the slower-moving
state derived from many such vectors
([Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds),
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

## Relationship to Weights Version Key

The weight vector and the weights version key act on the same extrinsic as payload and gate. The
weight vector is the scored targets a validator submits, while the weights version key is the
per-subnet hyperparameter that sets the minimum version a `set_weights` call must carry
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

The gate is checked before the payload is recorded. A `set_weights` call carrying a version below
the key is rejected before those weights are recorded, so a weight vector counts only when its
submission meets the subnet's weights version key
([Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).
