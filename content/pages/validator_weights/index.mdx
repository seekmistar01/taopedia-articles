# Validator Weights

Validator weights are evaluation signals submitted by subnet validators. They express how validators
assess miner work inside a subnet, and Yuma Consensus aggregates those signals into subnet outcomes
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The term is subnet-specific. A validator weight only has meaning inside the task, scoring method,
and incentive context that produced it.

That makes validator weights different from general validator reputation. They are structured
evaluation signals for a specific subnet and incentive mechanism, not a universal score that travels
across unrelated subnet tasks.

## Evaluation Signal

A validator weight is not a generic reputation score. It is a subnet-specific evaluation of miner
work under the rules of the subnet's incentive mechanism
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

That means similar-looking values can carry different meaning across different subnet tasks. The
subnet's work target and scoring method give the signal its interpretation.

## From Evaluation to Consensus

Validators use weight signals to express relative evaluations of miners. Yuma Consensus combines
those signals into a broader consensus result rather than treating one validator's view as final on
its own ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

This aggregation step is why validator weights matter to rewards. They are evaluation inputs that
help shape how incentives and dividends are distributed inside a subnet.

The input role is the key boundary. Validator weights enter the consensus process; they are not the
final incentive, dividend, or bond value by themselves.

That also separates validator weights from rank. Rank is a miner-side consensus outcome, while
validator weights are submitted evaluations that help produce the later outcome
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

## Consensus Score Boundary

Validator weights and consensus score are related but distinct. Validator weights are evaluation
inputs, while consensus score is the stake-weighted threshold used to filter outlier submissions
before final rank results
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

For readers, this means a validator weight should not be read as the consensus score itself. One is
an input signal; the other is a filtering threshold around submitted signals.

## Weight Vector Boundary

Validator weights are often read through weight-vector vocabulary. A weight vector is one
validator's structured set of miner evaluations, while validator weights name the evaluation signals
inside that consensus input
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

The distinction is mainly scope. Weight vector names the per-validator structure; validator weights
name the evaluation values that carry miner-scoring information.

This means a validator can submit a weight vector, and that vector contains validator weights. The
vector is the container; the weights are the evaluation entries used by consensus.

## Stake-Weight Context

Validator weights are interpreted inside a stake-weighted consensus process. Stake weight explains
how much influence a validator's signal carries, while validator weights express what that validator
evaluated about miner work
([Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

This keeps influence and evaluation separate. Stake weight affects how strongly a signal counts;
validator weights supply the miner-evaluation signal itself.

That separation prevents stake-weight language from replacing weight language. A validator can have
influence through stake weight, but the submitted validator weights still express what that
validator evaluated.

## Quality and Timing Context

Validator weights are most useful when they reflect independent evaluation. Weight copying weakens
that signal by relying on visible weight information instead of fresh evaluation
([The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

Commit Reveal can delay visibility of fresh weight signals until a reveal period. That timing
context reduces the usefulness of copying newly visible signals without changing what validator
weights are.

Validator-miner bonds are another neighboring concept. Bonds carry smoothed validator-miner
relationship history, while validator weights are the submitted evaluation signals that can feed the
consensus path
([Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

## Practical Limits

Validator weights should be read as consensus inputs, not as universal miner ratings or cross-subnet
comparisons. A weight claim should stay tied to the subnet, incentive mechanism, and Yuma Consensus
context that give it meaning
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Validator weights do not provide validator recommendations, miner-performance claims, or
point-in-time network results. They explain the submitted signal type used inside subnet consensus.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. Validator weight examples or consensus outcomes from one
environment should not be read as representing production subnet performance in another environment.

Use validator-weight language when the focus is a submitted miner-evaluation signal. Use weight
vector, consensus score, stake weight, validator-miner bond, incentive, or dividend language when
those specific parts of the consensus path are the focus.

The narrow use keeps the term useful: validator weights name what validators submit, while the rest
of the consensus vocabulary describes how those submissions are filtered, weighted, smoothed, and
converted into outcomes.
