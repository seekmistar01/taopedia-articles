# Liquid Alpha

Liquid alpha is Bittensor's consensus-based approach to varying validator-miner bond smoothing by
pair. Instead of applying one fixed smoothing factor across a subnet, the smoothing response can
change according to consensus alignment
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

The term belongs to the Yuma Consensus bond path. It affects how bond history responds after
validator weights are compared with consensus, rather than replacing validator evaluation itself
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Bond-Smoothing Context

Validator-miner bonds use exponential moving average smoothing. Alpha is the smoothing factor in
that moving average, so it controls how quickly a bond responds to new information
([Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds),
[Exponential Moving Averages](https://docs.learnbittensor.org/learn/ema)).

Liquid alpha changes how that factor is selected. The smoothing value can vary for each
validator-miner pair, which makes the mechanism pair-specific instead of subnet-wide only.

## Pair-Specific Response

Consensus alignment is the reader-facing reason liquid alpha varies by pair. When a pair is closer
to consensus, the bond response can be more direct; weaker alignment can keep the response more
conservative
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

That makes liquid alpha a bond-path mechanism. It should not be read as a separate miner-ranking
rule, a validator strategy, or proof that a particular validator behaved well or badly.

## Bounds and Configuration

Liquid alpha operates inside low and high alpha bounds. Those bounds limit the range of possible
smoothing responses, so pair-specific alpha values remain inside a configured interval
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

For readers, the bounds are a scope guard. Liquid alpha can vary the smoothing factor, but the
variation is constrained by the configured range.

## Consensus Alignment

Liquid alpha depends on the relationship between validator weights and consensus. Validator weights
are the evaluation signals that validators submit, while Yuma Consensus aggregates those signals
into subnet outcomes
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The mechanism therefore sits after evaluation in the bond response path. It adapts bond smoothing
around consensus alignment; it does not define the task being evaluated.

## Weight-Copying Boundary

Liquid alpha is often discussed near weight-copying defenses because copied or stale weights can
relate poorly to consensus alignment. Commit Reveal and weight-copying documentation describe the
visibility and copying problem around submitted weights
([The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

That relationship should stay bounded. Weight copying names an evaluation-quality concern; liquid
alpha names a bond-smoothing response that uses consensus alignment.

## Reader Boundary

Liquid alpha should be read as a consensus bond-smoothing concept. The stable point is that the
smoothing factor for validator-miner bonds can vary by pair within bounds, based on consensus
alignment
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

This article does not provide staking instructions, validator recommendations, copying accusations,
or performance claims. Exact outcomes belong to the subnet context and consensus data being
analyzed.

## Staking Boundary

Liquid alpha should be kept separate from ordinary TAO staking language. Alpha tokens are
subnet-specific assets, while staking and delegation documentation describes how TAO holders support
validators and receive reward exposure
([Glossary: Alpha Tokens](https://docs.learnbittensor.org/resources/glossary#alpha-tokens),
[Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation)).

That distinction helps readers track what asset and context a claim is using. Liquid alpha belongs
to subnet alpha-token liquidity, not every staking position or every wallet balance.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Liquid Alpha,
that sequence changes how readers should interpret liquidity examples, subnet-alpha observations,
and staking comparisons.

In localnet, Liquid Alpha examples can be exercised in an isolated environment. Localnet
observations reflect local chain state and local configuration rather than production Bittensor
behavior.

On testnet, Liquid Alpha behavior can be observed in a shared, non-production network. Testnet
examples are useful for checking interactions under shared chain conditions while staying separate
from mainnet state.

On mainnet, Liquid Alpha describes live production alpha-liquidity behavior on the production
Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. An example or outcome from one environment should not be
read as evidence of production behavior in another environment.
