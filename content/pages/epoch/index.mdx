# Epoch

An epoch is a consensus period inside a Bittensor subnet. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#epoch) defines an epoch as the period
during which a subnet executes its consensus mechanism, with the period determined by the subnet's
tempo. [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) provides the broader
consensus context for how validator evaluation signals become subnet outcomes.

## Consensus Period

The useful way to read epoch is as a period of subnet consensus work. It is not the subnet task, the
miner output, or the validator's individual evaluation; it is the period in which the subnet's
consensus mechanism runs over the relevant signals. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#epoch) ties the term directly to
subnet consensus, and [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)
explains the consensus process that interprets validator signals.

## Relationship to Tempo

Epoch and tempo are related but not identical. Tempo describes the cadence that determines when
epochs run, while epoch names the consensus period itself. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#tempo) describes tempo as the
subnet-specific cadence for epochs, and the
[Glossary](https://docs.learnbittensor.org/resources/glossary#epoch) defines epoch as the period
that cadence produces.

## Relationship to Validator Signals

Epochs matter because Bittensor consensus uses validator evaluation signals rather than a single
validator's view. [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) describes
validator weights as inputs to subnet consensus, and the
[Glossary](https://docs.learnbittensor.org/resources/glossary#epoch) places epoch at the period
where the subnet executes that consensus mechanism.

## Subnet Context

An epoch is scoped to a subnet's consensus process. Different subnets can define different work and
standards, so the epoch is best understood as part of the subnet's own mechanism rather than a
standalone quality score.
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
places task, protocol, and evaluation method inside subnet design, while
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) explains how evaluation
signals are resolved by consensus.

## Relationship to Multiple Mechanisms

An epoch is usually discussed for one incentive mechanism at a time, but subnets can also run
multiple mechanisms in parallel. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-matrix) notes that each
mechanism has its own weight matrix for independent consensus calculations.

For readers, an epoch still names a subnet consensus period, but the mechanism context matters.
Consensus work in one mechanism should be read separately when a subnet uses more than one
evaluation system.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)

## Relationship to Netuid

An epoch and a netuid are related but different parts of Bittensor subnet vocabulary. A netuid
selects which subnet context is in scope, while an epoch names a consensus period inside that
subnet's mechanism. The
[Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid) places netuid at the
subnet level, and the [Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)
defines an epoch as the period during which a subnet executes its consensus mechanism.

For readers, netuid answers which subnet market is being discussed, while an epoch answers which
consensus period inside that subnet is meant. Epoch language should stay paired with the netuid that
selects the subnet context.

References: [Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)

## Relationship to Block

An epoch and a block are related through subnet timing measurement. Epoch transitions are driven by
subnet tempo, which is expressed in blocks rather than clock time alone. The
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block) describes a block as a
unit of chain data on Subtensor, and the
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo) describes tempo as the
subnet-specific cadence for epochs measured in blocks.

For readers, blocks mark chain progression that subnet tempo counts toward epoch boundaries. An
epoch should be read as a consensus period tied to block-based cadence rather than as a standalone
wall-clock interval.

References: [Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch),
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)

## Relationship to Subnet Hyperparameters

An epoch and subnet hyperparameters are related but different parts of subnet timing vocabulary. An
epoch names the consensus period itself, while subnet hyperparameters such as `tempo` configure the
block-based cadence that determines when those periods run. The
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) reference
lists tempo and related subnet settings for a selected subnet.

For readers, subnet hyperparameters are the configurable inputs, while an epoch is the consensus
period those settings help produce. Different netuids can use different tempo values and therefore
different epoch cadences.

References: [Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)

## Relationship to Yuma Consensus

An epoch and Yuma Consensus address related but different parts of subnet outcome processing. An
epoch names the consensus period inside a subnet, while
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) describes how validator
evaluation signals are aggregated into subnet outcomes during that work. The
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch) ties epoch directly to
subnet consensus execution.

For readers, an epoch names the period boundary, while Yuma Consensus names the consensus process
that interprets validator signals within subnet mechanism design. Epoch language should not be read
as a substitute for the full Yuma documentation.

References: [Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)

## Relationship to Coinbase

Epoch and coinbase address related but different parts of Bittensor's per-block timing. An epoch is
the consensus period during which a subnet executes its consensus mechanism, while coinbase is the
per-block process that accumulates pending emissions and triggers each subnet's Yuma Consensus
epoch. The
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
documentation describes coinbase as a process that runs every block, injects liquidity into subnet
pools, and triggers each subnet's Yuma Consensus epoch at the appropriate interval. The
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch) defines an epoch as the
period during which a subnet executes its consensus mechanism.

For readers, an epoch names the consensus period—what happens inside the subnet when its consensus
mechanism runs—while coinbase names the per-block driver that determines when each subnet's epoch is
triggered. Coinbase runs every block; an epoch runs at the cadence that coinbase triggers for each
subnet's tempo. The two are additive: coinbase provides the per-block triggering mechanism, and the
epoch is the consensus period that results. A claim about what consensus computes during one period
belongs to epoch vocabulary, while a claim about what drives the per-block timing that starts each
period belongs to coinbase.

References:
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)

## Reader Boundary

Epoch should be read as a concept about consensus periods, not as a claim about exact duration or a
procedure for any particular subnet. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#epoch) provides the stable concept,
and the [Glossary](https://docs.learnbittensor.org/resources/glossary#tempo) provides the related
cadence term without requiring exact timing detail in this article.
