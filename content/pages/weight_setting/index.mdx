# Weight Setting

Weight setting is the validator process of assigning numerical scores to miners in a Bittensor
subnet. Validators observe miner work and submit weights that reflect how well each miner performed
the subnet task ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

Weight setting is process vocabulary for the evaluation side of subnet consensus. It does not name
the emission outcome or the miner incentive that follows from consensus.

## Role in Yuma Consensus

Weight setting feeds into Yuma Consensus, which aggregates validator evaluations into miner
incentives and validator dividends. Validators set weights; Yuma Consensus processes those weights
into consensus scores and emission shares
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

This keeps weight setting separate from the consensus calculation itself. Weight setting is the
input side; Yuma Consensus is the aggregation and output side.

## Eligibility Rules Trim Which Submissions Count

Not every weight a validator sets ends up counted. Epoch documentation lists pre-consensus filtering
steps that remove certain submitted weights before stake-weighted aggregation begins
([Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

Self-assigned weights are removed so validators cannot vote for themselves, with an exception for
the subnet creator when that role exists on the subnet. Weights tied to a neuron before that
neuron's latest [registration](https://docs.learnbittensor.org/resources/glossary#register) are also
dropped, and only rows from validators holding a
[validator permit](https://docs.learnbittensor.org/resources/glossary#validator-permit) influence
consensus. Weight setting therefore names the act of submitting, not a guarantee that every
submission reaches the consensus calculation.

## Subnet Context

Weight setting is subnet-specific. Each subnet defines what miners produce, and validators in that
subnet evaluate the output by setting weights
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

This keeps weight setting tied to the subnet's evaluation context. It is not a network-wide scoring
mechanism or a ranking system independent of a particular subnet's task definition.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For weight setting
on a subnet such as netuid 1, that sequence changes which subnet evaluation context a reader should
assume when interpreting validator score submissions.

In localnet, weight-setting flows can be tested in an isolated environment. Localnet validator
weights do not represent production consensus outcomes.

On testnet, weight setting can be exercised on a shared non-production network. Testnet consensus
results on a selected netuid are separate from mainnet metagraph and subnet state
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

On mainnet, weight setting concerns live validator evaluations on the selected
[netuid](https://docs.learnbittensor.org/resources/glossary#netuid) that feed Yuma Consensus.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A weight-setting example from one environment should not
be read as representing production consensus behavior in another environment.

## Each Setting Produces a Weight Vector

[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) documentation describes each
subnet validator as periodically submitting a vector of weights that ranks the miners that validator
has evaluated. Weight setting is the validator-side action that turns observed miner work into that
per-validator score set.

The [Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)
names one validator's structured miner scores. Those individual vectors are combined into the
subnet-level
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix), which
Yuma Consensus reads as its ranking input. Weight setting therefore describes building one
validator's row in that wider matrix, not the subnet-wide result after aggregation
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)).

[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
documentation adds that validators compute weight vectors from miner performance and transmit them
to the chain. The chain waits for the latest ranking signals from the subnet's validators before
assembling the matrix for consensus. One validator's setting step supplies evaluation material; the
full matrix appears only after those submissions are gathered across the subnet.

## Stake Weight Scales How Much a Setting Counts

[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) documentation explains that
the algorithm combines validator submissions using stake-weighted aggregation rather than treating
every setting as one equal vote. When consensus compares weights for a miner, validator stake helps
determine how much each submitted score influences the merged result.

[Understanding Subnets: Validator stake weight](https://docs.learnbittensor.org/subnets/understanding-subnets#validator-stake-weight)
documentation describes validator stake weight as the influence measure used during consensus. The
[Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight) defines
stake weight as the amount of stake backing a neuron, which affects how strongly that neuron's
validator weights count when those scores are merged with others in the same subnet.

That split keeps evaluation and influence separate in weight-setting vocabulary. Validator weights
express what a validator scored about miner work; stake weight expresses how strongly that
validator's setting counts inside the aggregation step that follows submission
([Understanding Subnets: Validator stake weight](https://docs.learnbittensor.org/subnets/understanding-subnets#validator-stake-weight),
[Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight)).

## Weight Setting Follows the Tempo Cycle

[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
documentation places weight setting inside a recurring evaluation rhythm rather than a one-off
snapshot. Validators compute weight vectors from miner performance and transmit them as part of the
subnet's ongoing incentive process.

The [Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo) defines tempo as the
subnet-specific interval over which Yuma Consensus calculates emissions from the latest available
ranking weight matrix.
[Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)
documentation states that validator weights are submitted during the preceding tempo, while emission
results from the consensus pass are finalized at the end of each tempo period.

Weight setting therefore belongs to a periodic submission window inside subnet timing. Validators
may update scores across the tempo, but the miner-facing and validator-facing outcomes tied to those
settings are credited when epoch processing completes at the tempo boundary rather than at every
intermediate submission moment
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

## Validator Weights, Weight Matrix, and Yuma Consensus Input

Weight setting is the validator-side process that produces validator weights. Each submission
becomes one validator's
[weight vector](https://docs.learnbittensor.org/resources/glossary#weight-vector), and the chain
assembles those vectors into the subnet-level
[weight matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix) that
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) reads as ranking input
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

The Subnet Metagraph documentation lists a Weights field defined as the weight matrix formed from
validator weight vectors and used as input for Yuma Consensus, matching the assembled-matrix step
this section describes ([Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

The useful pipeline order is therefore miner evaluation, weight setting, vector submission, matrix
assembly, and Yuma Consensus filtering and aggregation. Weight setting names the evaluation step;
validator weights name the submitted scores; the weight matrix names the assembled input set; Yuma
Consensus names the process that turns that matrix into miner incentives and validator dividends.

A weight-setting schedule or tempo window does not by itself report every validator weight value
stored for consensus processing, because the matrix names the gathered input rather than the
submission process alone
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Relationship to Weights Version Key

Weight setting and the weights version key act on the same submission as action and gate. Weight
setting is the act of submitting a weight vector, while the weights version key is the per-subnet
hyperparameter that sets the minimum version a `set_weights` call must carry
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

The key gates the action before any weights are recorded. A `set_weights` call carrying a version
below the key is rejected, so a weight-setting submission counts only when its version meets the
subnet's weights version key
([Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

## Relationship to Validator-Miner Bonds

Weight setting and validator-miner bonds connect submitted weights to accrued relationships. Weight
setting is the per-round action by which a validator assigns weights to miners, while
validator-miner bonds are the smoothed, time-accumulated relationships built from those weights
([Glossary: Weight Setting](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)).

Each weight-setting submission updates the bonds that shape a validator's later emission outcomes,
so weight setting is the recurring input and validator-miner bonds are the smoothed state that
carries those inputs into Yuma Consensus reward calculations
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Relationship to Subjective Evaluation

Weight setting and subjective evaluation meet where judgment becomes a signal. Weight setting is the
action each validator uses to submit its own scores, while subjective evaluation is the challenge
that subnet work often has no single correct answer, so validators may score the same submission
differently
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

Because there is no answer key, consensus over the submitted weights is what resolves the
subjectivity. Weight setting supplies each validator's judgment as a signal, and Yuma Consensus
aggregates those signals into the agreed result
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Relationship to Subnet Task

Weight setting names the validator process of assigning scores to miners. Subnet task names the work
those scores judge inside the subnet
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

The [Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task) entry
labels requested miner output, while weight setting labels the validator step that ranks that output
before consensus runs ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator)).

Use subnet task when naming expected work on a netuid; use weight setting when naming the validator
scoring step applied to that work
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner)).

Weight setting belongs to evaluation-process labels, not to the work target miners are asked to
produce on the subnet
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

## Setting Weights Is Not the Same as Them Counting

A validator can complete a `set_weights` submission and still have it contribute nothing to
consensus. Weight setting is only the act of submitting; whether those weights count depends on
separate gates such as holding a validator permit, meeting the minimum-weight count, and arriving
within the activity-cutoff window of the epoch
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

For readers, this separates the action from its effect. A submitted weight vector is an input
attempt, not a guaranteed consensus contribution, because eligibility and timing rules decide which
submissions enter aggregation
([Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

That gap is also why timing matters as much as content. Even a well-formed submission is ignored for
the current epoch if it misses the cutoff, so weight setting has to happen both correctly and early
enough to be aggregated.

## Reader Boundary

Weight setting is concept vocabulary for the validator evaluation process in Bittensor subnets. The
term should not be read as a miner ranking table, a staking guide, or an emission forecast
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Actual weight values and their emission effects depend on subnet context, validator behavior, and
consensus configuration. The concept travels across subnets; observed outcomes belong to specific
environments ([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).
