# Validator Deregistration

Validator deregistration is the validator-specific case of deregistration in a Bittensor subnet. The
[Validating: Validator deregistration](https://docs.learnbittensor.org/validators#validator-deregistration)
documentation describes validators being removed when emissions are low and subnet registration
rules require displacement from a full subnet.

## Validator-Specific Scope

The broader [Deregistration](https://docs.learnbittensor.org/resources/glossary#deregistration)
glossary entry applies to subnet miners or subnet validators. Validator deregistration narrows that
concept to validators: the participant being removed is a validator occupying a subnet UID slot.

This keeps the term role-specific rather than a separate scoring system. It names which type of
subnet participant is removed, while the broader deregistration concept names the general removal
process for low-performing subnet participants.

## Validation Permit Context

Validator deregistration differs from miner deregistration because validator participation also
involves validation-permit context inside a subnet. The
[Validating: Validator deregistration](https://docs.learnbittensor.org/validators#validator-deregistration)
documentation explains that losing validation permit and emission standing can make a validator
eventually eligible for displacement when a subnet is full and a new registration arrives
([Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit)).

For readers, validator deregistration should be read as subnet-participation vocabulary tied to
validator role context, not as a live ranking table or a manual recommendation.

## UID Slot and Registration Context

Like miner deregistration, validator deregistration belongs to limited subnet UID capacity. The same
validating documentation places displacement in the context of a full subnet and a new registration
event. [Register](https://docs.learnbittensor.org/resources/glossary#register) names the entry
process; validator deregistration names the validator-side exit path when subnet rules select a
low-performing validator for removal
([Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)).

This article does not list current validators, count open slots, or describe registration commands.
Those details belong in subnet references and live chain data.

## Contrast with Miner Deregistration

Miner deregistration and validator deregistration share the broader deregistration label but apply
to different subnet roles. Miner deregistration names removal of a low-performing miner, while
validator deregistration names removal of a low-performing validator under validator-specific permit
and emission context described in validating documentation.

For readers, choosing the correct term depends on which subnet role is being removed, not on whether
the subnet uses one or many incentive mechanisms.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For validator
deregistration on a subnet such as netuid 1, that sequence changes how readers should interpret
examples of validator removal.

In localnet, validator deregistration can show that validator-slot replacement logic is represented
inside an isolated subnet environment. That is useful for checking how a validator UID can be
vacated on a selected netuid, but it does not show that a live public subnet validator has been
removed.

On testnet, validator deregistration behavior can be observed in a shared testing network. Testnet
displacement on a subnet such as netuid 1 gives stronger evidence about how validator removal
interacts with permits, emissions, new registrations, and immunity-period boundaries than a private
local run, while still keeping the result separate from production mainnet activity.

On mainnet, validator deregistration belongs to live subnet state. The
[Validating: Validator deregistration](https://docs.learnbittensor.org/validators#validator-deregistration)
reference describes validator displacement in the full-subnet context, so production readings should
keep the selected netuid, validator UID slot, block timing, and eligibility context attached.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A validator deregistration example in one environment
should not be read as identical evidence about validator removal in another environment.

This distinction matters because validator deregistration is a role-specific subnet outcome, not a
standalone validator quality label. Capacity, validator emissions, permit context, immunity-period
status, and network environment all affect what the example proves.

For readers, this keeps validator deregistration examples from sounding stronger than their
environment supports. Local, testnet, and mainnet contexts can all contain validator deregistration
language, but they do not carry the same interpretive weight.

## Low Emissions Can Precede Validator Displacement

The
[Validating: Validator deregistration](https://docs.learnbittensor.org/validators#validator-deregistration)
documentation describes validators being removed when emissions are low and subnet registration
rules require displacement from a full subnet. Validator deregistration vocabulary therefore ties
removal to subnet participation and emission standing, not to a generic quality label detached from
subnet context.

Readers should attach the selected netuid and registration context when citing displacement. The
validating documentation names the validator-side exit path; live eligibility belongs to current
subnet state.

## Validator Permit Loss Ends Weight-Setting Rights

Validator deregistration documentation connects displacement to validation-permit and emission
context inside a subnet
([Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit)).
Losing validator permit and emission standing can make a validator eligible for removal when a full
subnet accepts a new registration.

Permit vocabulary names weight-setting authorization among qualifying stake-weight neurons;
validator deregistration names removal from the subnet participant set when documented rules require
it.

## Full Subnet Capacity Triggers Displacement on New Registration

Validator deregistration belongs to limited
[UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot) capacity inside a selected
[netuid](https://docs.learnbittensor.org/resources/glossary#netuid). When a subnet is full and a new
[register](https://docs.learnbittensor.org/resources/glossary#register) event arrives, subnet rules
can select a low-performing validator for removal to make room.

A UID slot names the occupied validator position; validator deregistration names the validator-side
process that can vacate that position under documented displacement rules
([Validating: Validator deregistration](https://docs.learnbittensor.org/validators#validator-deregistration)).

## Immunity Period Can Delay Validator Displacement

The [Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period)
describes temporary protection after registration. Validator deregistration describes how a
low-performing validator can lose a UID slot once that protection no longer applies or when other
eligibility rules are satisfied.

Immunity period names a protection window that can delay ordinary low-performance removal; validator
deregistration names the validator-specific displacement path after that boundary.

## Multiple Mechanisms Still Use One Validator UID

Subnets can run more than one incentive mechanism in parallel. The
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)
notes that validators must evaluate miners separately for each mechanism
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

Validator deregistration still describes removal of a validator from a subnet UID slot. Mechanism
count belongs with subnet incentive design rather than with the validator deregistration label
itself.

## Participant Removal Differs From Subnet Deregistration

Validator deregistration removes one validator from a UID slot inside a subnet that continues to
exist. [Subnet deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration)
removes an entire non-immune subnet from the network when the number of subnets exceeds 128 and that
subnet has the lowest EMA of alpha price.

Validator deregistration is scoped to one participant inside one subnet; the subnet continues to
exist, and other validators and miners retain their slots. Subnet deregistration removes the subnet
itself, ending all activity for every participant registered in that netuid.

Readers should keep participant-level validator removal separate from whole-subnet sunset events.

## Child Hotkeys Delegate Validation Without Replacing UID Ownership

A [child hotkey](https://docs.learnbittensor.org/validators/child-hotkeys) is a separate key that a
parent hotkey authorizes to validate on its behalf by re-delegating a portion of the parent's stake,
without transferring ownership of that stake. Validator deregistration is the removal of a validator
from a subnet when emissions are low and registration rules require displacement from a full subnet
([Validating: Validator deregistration](https://docs.learnbittensor.org/validators#validator-deregistration)).

Child hotkeys answer which key a parent authorizes to carry out validation; validator deregistration
answers how a registered validator's UID slot is removed. Deregistration ends a registered
validator's place in the subnet, whereas a child hotkey is a delegated validation key acting on a
parent's re-delegated stake-weight.

## Deregistration Ends Epoch Dividends From Coinbase

While registered, a validator can earn validator dividends from the emissions
[coinbase](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase) distributes each
[epoch](https://docs.learnbittensor.org/resources/glossary#epoch). After validator deregistration,
the validator is no longer an active subnet participant and no longer shares in those distributions.

Coinbase is the recurring process that produces and distributes emissions each epoch; validator
deregistration is the lifecycle event that removes a validator from the subnet where it would
otherwise take part in that flow.

## Validator Deregistration Removes Evaluation Standing

Validator deregistration ends a neuron's participation as a subnet validator, not merely its wallet
association. Documentation describes removal from a UID slot, which stops that neuron from
submitting weights that count in consensus until it registers again
([Validating: Validator deregistration](https://docs.learnbittensor.org/validators#validator-deregistration),
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator)).

For readers, this is why deregistration affects emissions paths. A deregistered validator no longer
participates in the evaluation and dividend flow that coinbase and Yuma Consensus advance each tempo
([Coinbase documentation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)).

The same pruning-score ranking that displaces miners can displace validators when a full subnet
needs a slot, so validator deregistration is not a separate penalty category but the same
competition rule applied to the evaluating role
([Mining in Bittensor](https://docs.learnbittensor.org/miners)).

## Validator Permit and UID Removal Are Separate Limits

Subnet hyperparameter documentation lists max allowed validators separately from max allowed UIDs. A
validator can lose a UID through deregistration even while validator-permit vocabulary describes how
many neurons may hold validator authorization on the subnet
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit)).

For readers, validator deregistration names UID-slot removal for the evaluating role. Permit limits
shape how many validators can submit weights, but displacement still follows pruning-ranked removal
when the subnet is full ([Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph),
[Validating: Validator deregistration](https://docs.learnbittensor.org/validators#validator-deregistration)).

## Weight Submission Ends When the Validator Slot Vacates

Validator deregistration stops a neuron from submitting weights that count in Yuma Consensus until
it registers again on the subnet. That is why the term belongs to participation vocabulary rather
than wallet deletion ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

For readers, deregistered validators no longer influence miner incentives or validator dividends on
that netuid, even if the underlying keys remain usable elsewhere in Bittensor
([Emission](https://docs.learnbittensor.org/learn/emissions),
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator)).

## Distinction from Miner Deregistration

Deregistration uses the same emission-based pruning rule for every neuron on a full subnet. Miner
deregistration names the case where the displaced participant is a miner, while validator
deregistration names the case where the displaced participant is a validator
([Glossary: Deregistration](https://docs.learnbittensor.org/resources/glossary#deregistration),
[Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration),
[Validating: Validator deregistration](https://docs.learnbittensor.org/validators#validator-deregistration)).

The chain-level selector is pruning-score ranking, not role type. The two terms describe which
subnet role lost the UID once displacement occurs.

## Distinction from UID Trimming

Validator deregistration removes one participant when a full subnet accepts a new registration. UID
trimming is a subnet-owner bulk resize that removes many low emitters down to a chosen maximum
([Validating: Validator deregistration](https://docs.learnbittensor.org/validators#validator-deregistration),
[UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming)).

- **Validator deregistration** — single-slot displacement at registration time.
- **UID trimming** — owner-initiated bulk reduction and compression.

## Distinction from Subnet Deregistration

Validator deregistration vacates one UID row inside a subnet that continues to exist. Subnet
deregistration removes an entire subnet to free a network slot when the global subnet cap binds
([Subnet deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration),
[Validating: Validator deregistration](https://docs.learnbittensor.org/validators#validator-deregistration)).

A low-performing validator can be displaced without dissolving the subnet itself.

## Distinction from Validator Permit

Validator permit authorizes qualifying neurons to submit weights on a subnet. Losing permit can
reduce validator emissions, but permit status is not the displacement selector when a full subnet
accepts a new registration
([Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit),
[Validating: Validator deregistration](https://docs.learnbittensor.org/validators#validator-deregistration)).

Displacement still follows pruning-ranked removal on the selected subnet
([Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)).

## Distinction from Register

Register is the entry action that pays a dynamic burn to claim a UID. Validator deregistration is
the removal outcome when a full subnet must displace an eligible incumbent for that entry
([Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register),
[Validating: Validator deregistration](https://docs.learnbittensor.org/validators#validator-deregistration)).

Registration and displacement are linked on a full subnet, not independent steps
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Distinction from Pruning Score

Pruning score ranks which incumbent is weakest when displacement is required on a full subnet.
Validator deregistration names the validator-specific removal event that follows that ranking
([Glossary: Pruning Score](https://docs.learnbittensor.org/resources/glossary#pruning-score),
[Validating: Validator deregistration](https://docs.learnbittensor.org/validators#validator-deregistration)).

Pruning-score vocabulary belongs to the ranking input. Validator-deregistration vocabulary belongs
to the validator slot vacated after eligibility rules are satisfied
([Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period)).

## Reader Boundary

This article defines validator deregistration at a high level. It does not list current validators,
rank subnet participants, or describe registration commands. Live displacement eligibility belongs
in validator references and chain state
([Validating: Validator deregistration](https://docs.learnbittensor.org/validators#validator-deregistration)).

## Relationship to Pruning Score

Validator deregistration and the pruning score act on the same removal as action and selector. The
pruning score is the per-neuron value that ranks which UID is most expendable when a full subnet
must displace one, while validator deregistration is the validator-specific removal that frees the
slot, so the score chooses which validator is removed and deregistration carries it out
([Mining in Bittensor](https://docs.learnbittensor.org/miners),
[Glossary: Deregistration](https://docs.learnbittensor.org/resources/glossary#deregistration),
[Validating: Validator deregistration](https://docs.learnbittensor.org/validators#validator-deregistration)).

- **Pruning score** — the ranking that selects which UID is removed.
- **Validator deregistration** — the removal that frees the selected validator's slot.
