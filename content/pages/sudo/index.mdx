# Sudo

Sudo is the Bittensor glossary term for privileged administrative authority. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#sudo) describes it as a privileged key
for administrative actions that has been replaced by the governance protocol for enhanced security.

## Governance Context

The [Governance Overview](https://docs.learnbittensor.org/governance) explains that Bittensor
governance moves network management away from centralized control over time. In that model, the
Triumvirate creates proposals and the Senate reviews them, replacing single-authority administration
with proposal review.

## Security Transition

Before Bittensor governance, administrative actions required permission through a single privileged
sudo key. The [Governance Overview](https://docs.learnbittensor.org/governance#security) explains
that governance replaced that single-key model so approving privileged changes now requires both
Senate majority support and Triumvirate closure of a proposal.

## Proposal Review Context

The governance replacement for Sudo is not just a new key name. The
[Governance Overview](https://docs.learnbittensor.org/governance) describes a proposal flow in which
the Triumvirate creates a proposal containing privileged call data, the Senate approves or rejects
it, and a Triumvirate member closes the proposal after approval. When closure succeeds, the call
data executes in the same block as the close action.

That lifecycle separates authority into stages. Proposal creation identifies the privileged action
being requested. Senate voting supplies the approval gate. Closure supplies the execution step after
approval. Sudo vocabulary points to the older single-authority model, while governance vocabulary
points to this staged proposal-review path.

The distinction matters when reading old or general administrative references. A statement about
Sudo describes privileged administrative authority; a statement about governance describes a
proposal process with named bodies and state transitions. The terms are historically related, but
they do not describe the same operating model.

The Senate documentation also separates visibility from authority. Proposals before the Senate can
be viewed publicly, but approval and disapproval votes belong to senators. That makes the governance
replacement more transparent than a hidden administrative key while still limiting the actual
approval role to the Senate process described in the official documentation.

For readers, the practical boundary is scope. Sudo is useful as a legacy authority concept, not as a
current instruction to look for one active administrative key. Current privileged-change context
belongs with proposals, Senate approval, Triumvirate closure, and official governance records. The
term is therefore best read as historical authority vocabulary inside a governance article.
Governance records, not a supposed current sudo holder, are the current source for privileged-change
review context.

## Reader Boundary

This article describes Sudo as a governance term. It does not identify any live holder, expose key
material, or describe privileged action execution. Readers should use official governance references
for current proposal-review context.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Sudo, that
sequence changes how readers should interpret evidence about privileged administrative actions and
governance flows.

In localnet, sudo authority and governance mechanics can be tested in an isolated environment.
Localnet privileged actions do not represent production governance decisions or real chain state.

On testnet, governance flows can be observed in a shared, non-production network. Testnet sudo and
proposal interactions are separate from mainnet governance state.

On mainnet, privileged administrative authority is handled through the live governance protocol —
Triumvirate proposals reviewed and approved by the Senate before executing on-chain. The
[Governance Overview](https://docs.learnbittensor.org/governance) describes the proposal lifecycle
that replaces the older sudo model on the production network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A sudo or governance example from one environment should
not be read as representing privileged-change authority or proposal outcomes in another environment.
