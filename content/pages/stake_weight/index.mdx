# Stake Weight

Stake weight is the subnet-level stake measure used to describe validator influence in Bittensor
consensus. It is the computed stake value tied to consensus power and emissions inside a subnet
([Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The term links staking vocabulary to consensus evaluation. It answers how much influence a submitted
evaluation carries after stake context is applied, while validator weights answer what was signaled
about miner work.

## Consensus Influence

In Yuma Consensus, validators submit evaluation signals and the consensus process weights those
signals before producing subnet outcomes
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

This measure is the influence side of that path. It does not replace the evaluation signal; it
scales how strongly the submitted signal participates in the subnet consensus process
([Glossary](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

This is why the term matters in plain English: two validators can submit signals into the same
subnet process, but their stake context affects how those signals count when consensus is computed.

## Components and Neighboring Terms

The measure combines alpha stake, TAO stake, and TAO weight in subnet context
([Glossary](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Glossary: TAO Weight](https://docs.learnbittensor.org/resources/glossary#tao-weight)).

The staking-total term and the validation-eligibility term have different jobs. One describes
backing, while the other concerns eligibility in a subnet
([staking total](https://docs.learnbittensor.org/resources/glossary#effective-stake),
[validation eligibility](https://docs.learnbittensor.org/resources/glossary#validator-permit)).

Together, these terms give readers a compact map: backing, eligibility, TAO contribution, and the
consensus measure are related but separate concepts.

## Signal Path

Validator weights are the evaluation values. Weight vectors organize one validator's submissions,
and weight matrices collect those vectors for the subnet-level consensus process
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)).

The measure sits beside that input structure as the influence value applied around validator
submissions. Later result terms such as rank, trust, and validator trust describe what happens after
the signals are filtered and interpreted
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)).

## Subnet and Environment Scope

The measure belongs inside the subnet context that uses it. Validators evaluate miner work according
to subnet-specific standards, and Yuma Consensus interprets those signals inside that setting
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Bittensor separates mainnet, testnet, and localnet environments. Examples from one environment
should not be treated as evidence for another because subnet tasks, validators, and stake conditions
can differ ([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Introduction to Bittensor: Subnet development](https://docs.learnbittensor.org/learn/introduction#subnet-development)).

Localnet examples are isolated development examples. Testnet examples are shared non-production
examples. Mainnet interpretation belongs to the mainnet subnet and consensus context being
discussed.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For stake weight
on a subnet such as netuid 1, that sequence changes how readers should interpret validator influence
readings and metagraph stake fields.

In localnet, stake-weight examples can be exercised in an isolated environment. Local validator
influence values reflect local chain state rather than production subnet consensus.

On testnet, stake weight can be observed in a shared, non-production network. Testnet stake-weight
readings on a selected netuid are separate from mainnet metagraph state
([Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

On mainnet, stake weight is the live subnet-level validator influence measure used in production
consensus context on the connected Bittensor network
([Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A stake-weight example from one environment should not be
read as representing production validator influence on another network.

## Stake Weight Differs From Effective Stake

Effective stake and validator permit vocabulary describe backing and eligibility. Stake weight
describes consensus influence after those staking contexts are already in view
([Glossary: Effective Stake](https://docs.learnbittensor.org/resources/glossary#effective-stake),
[Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit)).

Keeping those terms separate avoids reading a permit flag or staking total as the same thing as a
consensus influence measure. A higher effective stake total does not automatically mean the same
stake weight reading, because stake weight is the influence term used in consensus rather than a
duplicate balance label
([Delegation](https://docs.learnbittensor.org/staking-and-delegation/delegation)).

## TAO Weight Scales the TAO Stake Component

[TAO weight](https://docs.learnbittensor.org/resources/glossary#tao-weight) names the relative role
of TAO stake when stake weight is derived alongside
[alpha stake](https://docs.learnbittensor.org/resources/glossary#protocol-alpha) on an ordinary
subnet. Stake weight is the combined consensus influence result; TAO weight is one input parameter
inside that calculation
([Understanding Subnets: Validator stake weight](https://docs.learnbittensor.org/subnets/understanding-subnets#validator-stake-weight)).

A TAO weight parameter does not by itself report a validator's full stake weight reading, because
stake weight combines stake components into the influence value consensus uses.

## Stake Weight Sits Between Weight Inputs and Rank Outputs

Validator weights, weight vectors, and weight matrices describe the evaluation inputs Yuma Consensus
receives. Rank, trust, and validator trust describe filtered outcomes after those inputs are
interpreted. Stake weight names the influence value applied along that path rather than replacing
either side
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Rank Sums Post-Clip Weights by Stake

[Yuma Consensus: Miner emissions](https://docs.learnbittensor.org/learn/yuma-consensus#miner-emissions)
documentation calculates each miner's aggregate rank as the stake-weighted sum of post-clip
validator weights. Stake weight therefore scales how much each submitted evaluation can move the
merged miner result inside a subnet
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

That formula keeps evaluation and influence separate. Validator weights express what a validator
scored; stake weight expresses how strongly that score counts when those signals are combined.

## Validator Permits Follow Stake-Weight Ranking

The
[Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit)
describes the flag as awarded to the top neurons by stake weight on a subnet. Stake weight is
therefore tied both to consensus influence and to which neurons hold validation rights in that
subnet
([Understanding Subnets: Validator stake weight](https://docs.learnbittensor.org/subnets/understanding-subnets#validator-stake-weight)).

Permit vocabulary answers whether weight submission rights exist. Stake weight explains the ranking
context that determines who can receive those rights.

## Clipping Benchmarks Use a Stake Fraction

[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) clipping uses a
stake-weighted benchmark for each miner. Weights that exceed that benchmark are clipped, so neither
the miner nor the validator receives emissions for the excess portion. The kappa hyperparameter sets
what fraction of total stake must support a weight level before it becomes the benchmark
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

Stake weight therefore appears twice in the filtering path: it helps set the benchmark and it scales
how surviving weights combine into rank after clipping.

## Root Subnet Stake Weight Skips the Alpha Component

The formula changes on the Root Subnet. On an ordinary subnet, validator stake weight is alpha stake
plus TAO stake scaled by TAO weight, but on Subnet Zero a validator's stake weight is simply their
staked TAO
([Understanding Subnets: Validator stake weight](https://docs.learnbittensor.org/subnets/understanding-subnets#validator-stake-weight)).

That difference follows from scope: Subnet Zero has no alpha token of its own, so the alpha-stake
term in the ordinary-subnet formula has nothing to add. Root validator stake weight is read directly
from staked TAO rather than from a blended alpha-and-TAO calculation
([Glossary: Root Subnet/Subnet Zero](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero)).

## Metagraph Fields Expose Stake Weight Readings

The [subnet metagraph](https://docs.learnbittensor.org/subnets/metagraph) exposes stake-weight
fields for each neuron on a netuid. Those readings help readers connect staking context to consensus
influence without treating metagraph totals as payout amounts or rank outcomes on their own
([Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight)).

Metagraph stake weight belongs to the selected subnet and network environment. A testnet metagraph
field on one netuid is not evidence for mainnet influence on another network.

## Distinction from Validator Weights

Validator weights are evaluation signals a validator assigns to miners. Stake weight is the computed
stake value that scales how strongly that validator's signal counts in Yuma Consensus
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Weights vocabulary names what was scored; stake weight vocabulary names influence around that score.

## Distinction from Effective Stake

Effective stake names total TAO staked behind a validator, including delegated stake
([Glossary: Effective Stake](https://docs.learnbittensor.org/resources/glossary#effective-stake)).

Stake weight is the subnet consensus influence measure derived from stake components on that subnet
([Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Understanding Subnets: Validator stake weight](https://docs.learnbittensor.org/subnets/understanding-subnets#validator-stake-weight)).

Backing totals and consensus influence answer different reader questions on the staking path.

## Distinction from Relative Stake Weight

Stake weight is one validator's computed influence value on a subnet. Relative stake weight divides
that value by the total stake weight across validators on the same subnet
([Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Glossary: Relative Stake Weight](https://docs.learnbittensor.org/resources/glossary#relative-stake-weight)).

Absolute influence and proportional share within the validator set are separate readings.

## Distinction from Validator Trust

Validator trust is a validator-side outcome after Yuma Consensus filters and aggregates submitted
weights. Stake weight scales influence before and during that aggregation path
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Influence input and post-filter validator-side result belong to different stages of the same path.

## Distinction from Blockchain Validator vs Subnet Validator

Stake weight and the blockchain-versus-subnet-validator comparison clarify which validator role
stake weight applies to. A blockchain validator secures the chain at the network layer, while a
subnet validator evaluates miners inside a subnet, and stake weight is the value that scales a
subnet validator's consensus influence, not the block-production duty of a blockchain validator
([Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Glossary: Blockchain Validator vs Subnet Validator](https://docs.learnbittensor.org/resources/glossary#blockchain-validator-vs-subnet-validator),
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

- **Blockchain validator vs subnet validator** — separates network-layer block production from
  subnet evaluation roles.
- **Stake weight** — subnet-validator consensus influence, not a blockchain-validator duty.

## Distinction from Conviction and Locked Stake

Stake weight names the influence measure derived from staked value. Conviction and locked stake name
the separate time-based commitment signal produced by locking already-staked alpha
([Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Conviction and locked stake](https://docs.learnbittensor.org/staking-and-delegation/conviction-staking)).

- **Stake weight** — computed consensus influence from staked value.
- **Conviction and locked stake** — time-based commitment from locking staked alpha.

## Distinction from Max Validators

Max validators is the hyperparameter ceiling on how many validator permits a subnet can issue. Stake
weight is the computed influence measure used to rank validators for those permit slots
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight)).

- **Max validators** — configured cap on validator permits.
- **Stake weight** — computed influence that ranks who receives permits.

The hyperparameter sets the ceiling; stake-weight ranking fills permit slots up to that maximum
([Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit)).

## Distinction from Yuma Consensus

Stake weight names how much influence a validator's submitted weights carry based on the
stake-weight calculation that combines subnet alpha stake and global TAO stake for that hotkey. Yuma
Consensus is the separate mechanism that uses those stake-weighted signals when converting validator
weights into emission shares each tempo
([Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Stake weight** — stake-backed influence behind validator weight signals.
- **Yuma Consensus** — mechanism that turns validator weights into emission shares.

## Reader Boundary

Stake weight should not be read as a payout amount, a validator trust score by itself, or proof that
a miner received a specific reward. It names validator influence inside subnet consensus—the
computed stake value that scales how strongly a submitted evaluation counts
([Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Validator weights say what a validator signaled about miner work. Stake weight explains how strongly
that signal participates when Yuma Consensus filters and combines validator submissions
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).
