# Root Claim Type

Root claim type is the Keep or Swap setting for handling alpha dividends earned through Root Subnet
staking. Keep leaves processed dividends as originating-subnet alpha, while Swap converts processed
dividends to TAO and restakes them on Root
([Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

The term belongs to dividend-handling vocabulary. It names the selected outcome for processed root
claims, not a separate staking action or balance report.

## Claim Setting

Root claim type is the preference applied when accumulated root-claim dividends are processed. It
chooses between the Keep and Swap dividend outcomes
([Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

That makes root claim type different from claim timing. Claim timing answers when accumulated
dividends are processed; root claim type answers how those dividends are handled.

## Keep Outcome

Keep leaves earned alpha on the subnet that generated the dividend
([Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

For readers, Keep is the alpha-retention outcome. It does not change the definition of Root Subnet
staking, and it does not by itself describe the quality of the subnet that produced the dividend.

## Swap Outcome

Swap converts accumulated alpha dividends to TAO and restakes the resulting TAO on the Root Subnet
([Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

For readers, Swap is the TAO-restaking outcome. It should not be confused with unstaking from Root,
moving stake between validators, or changing the underlying Root Subnet staking position.

## Relationship to Root Claims

Root claims name the broader process for handling accumulated dividends. Root claim type is the
setting used inside that process
([Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

This keeps the vocabulary precise. A root claim is the processing event; root claim type is the Keep
or Swap setting that determines the dividend outcome during processing.

## Root Staking Context

Root Subnet is the staking context where TAO can be delegated through validators, and root claim
type describes what happens to dividends produced through that position
([Glossary: Root Subnet](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero),
[Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

This makes root claim type adjacent to root staker vocabulary. Root staker names the TAO holder;
root claim type names the dividend-handling preference tied to claim processing.

## Root Staker Context

A root staker is the TAO holder using Root Subnet staking. Root claim type matters to that role
because it determines how claimable dividends from the root-staked position are handled
([Glossary: Root Staker](https://docs.learnbittensor.org/resources/glossary#root-staker),
[Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

This keeps the role and setting connected without merging them. Root staker names who is staking;
root claim type names the Keep or Swap preference for processed dividends.

## Emission Context

Root claim type also belongs near emissions vocabulary because the underlying subject is dividend
handling. Emissions supply the broader reward context, while root claim type decides whether the
processed dividend stays as alpha or becomes TAO restaked on Root
([Emission](https://docs.learnbittensor.org/learn/emissions),
[Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

This makes the term narrower than emissions overall. It is the claim-output preference, not the full
reward distribution model.

## Reading the Term

Root claim type should be read as a concept for dividend handling. It does not describe how to set
the preference, initiate a claim, quote claim thresholds, or report exact conversion outcomes
([Root Claim overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

The stable article-level point is that root claim type names the Keep or Swap setting that
determines whether processed Root Subnet dividends remain as subnet alpha or are converted to TAO
and restaked on Root.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Root Claim
Type, this sequence gives readers a boundary for interpreting claim examples and root-network
accounting notes.

Localnet examples are isolated and reflect local chain state, so they are useful for controlled
experiments rather than evidence of live Bittensor behavior. Testnet examples add shared
non-production conditions, which can reveal integration behavior without touching mainnet state.

On mainnet, Root Claim Type examples should be read as live production root-claim behavior on the
production Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, so outcomes from one environment should not be treated as
proof of behavior in another.
