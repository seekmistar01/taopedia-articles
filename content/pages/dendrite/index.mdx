# Dendrite

A dendrite is the client-side component used to send requests to axon endpoints in Bittensor subnet
communication ([Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite),
[Dendrite reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/dendrite/)).

The term belongs to the axon-dendrite communication path. It should be read separately from the
subnet role, subnet task, and incentive rule that interpret the response.

## Client-Side Context

The official glossary defines a dendrite as a client instance that communicates with axons, and the
API reference describes Dendrite as a network client that sends requests to axon endpoints and
processes responses
([Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite),
[Dendrite reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/dendrite/)).

That framing makes dendrite a client-side communication term. It names the contacting component, not
the full subnet protocol or the reward rule that may use the returned data.

## Relationship to Axons

Dendrite and axon describe opposite sides of the communication path. Axon is the server-side
endpoint that receives incoming synapse objects, while dendrite is the client-side component that
sends requests to those endpoints
([Glossary: Axon](https://docs.learnbittensor.org/resources/glossary#axon),
[Axon reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/)).

For readers, dendrite initiates the request path and axon receives it. The two terms describe
direction in subnet communication rather than a complete account of subnet behavior.

## Synapse Exchange

Subnet communication carries structured synapse objects. The glossary describes synapse objects as
the main vehicle for information exchange between subnet validators and subnet miners
([Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[Synapse reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/synapse/)).

Dendrite belongs to the path that sends a synapse toward an axon endpoint and processes the
response. It does not define the task-specific content inside the synapse object.

## Miner and Validator Boundary

Dendrite language often appears near subnet validators because validators can use client-side
communication to query miner axons. The miner and validator roles are still separate role concepts
([Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner),
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator)).

A dendrite can be part of validator-side communication, but dendrite does not by itself define
validator judgment, miner work, or the subnet's incentive rule.

## Subnet Protocol Context

Different subnets can define different tasks and evaluation standards. Bittensor documentation
connects subnet tasks to miner work, and incentive-mechanism documentation places miner work,
validator evaluation, and emission outcomes inside subnet-specific mechanisms
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Dendrite is the client-side communication component in that setting. The subnet protocol explains
what request and response content means for evaluation.

## Development and Network Context

Bittensor documentation separates local development, testchain, and mainchain contexts, and also
separates mainnet, testnet, and localnet network contexts
([Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

A dendrite example from one environment should not be treated as identical evidence about response
quality, endpoint availability, or validator behavior in another environment. The client-side
concept can stay the same while the evidence has a different scope.

## Reader Boundary

Dendrite should be read as Bittensor's client-side communication component for subnet exchange. It
is not a complete validator implementation, a scoring decision, a persistent chain record, or a
guarantee about endpoint behavior
([Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

That boundary keeps the term useful: dendrite identifies the client-side request component, while
surrounding subnet documentation explains what the response means.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Dendrite, this
sequence gives readers a boundary for interpreting client examples, miner-query behavior, and
network communication notes.

Localnet examples are isolated and reflect local chain state, so they are useful for controlled
experiments rather than evidence of live Bittensor behavior. Testnet examples add shared
non-production conditions, which can reveal integration behavior without touching mainnet state.

On mainnet, Dendrite examples should be read as live production validator communication on the
production Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, so outcomes from one environment should not be treated as
proof of behavior in another.
