# Inspecting the Chain with Polkadot.js

Inspecting the chain with Polkadot.js means using the Polkadot.js browser app to look at Bittensor's
blockchain layer, Subtensor. The official
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
page frames the app as a way to query chain state, inspect runtime metadata, browse blocks, and
submit extrinsics when signing authority is available.

For Taopedia readers, the page is most useful as a concept boundary rather than a procedural guide.
The important distinction is between observing public chain information and authorizing state
changes. Polkadot.js can do both, but those two uses have very different security implications.

## Chain Inspection Context

Bittensor's chain state is public, so many questions can be answered without loading private wallet
material. Polkadot.js gives readers a direct view into Subtensor data such as storage items, runtime
constants, blocks, events, and metadata. That makes it useful as a chain-inspection tool even when
the user does not intend to submit a transaction.

The tool should not be confused with Taopedia itself, BTCLI, or the Bittensor SDK. Taopedia explains
concepts, BTCLI and the SDK expose Bittensor-specific workflows, and Polkadot.js is a general
Substrate-family interface for reading chain state and interacting with runtime calls.

## Network Connection Boundary

The first boundary is which network is being inspected. The official source distinguishes mainnet
Finney from testnet and custom endpoints. A query result only makes sense in the context of the
connected endpoint, so a claim about balances, constants, storage, or blocks should name whether it
comes from mainnet, testnet, or another chain endpoint.

This matters for Taopedia because an article should not treat a value seen on one endpoint as a
universal Bittensor fact. Some chain data is live, some differs by network, and some changes after a
runtime upgrade or governance action.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from local testing to testchain and then mainchain. Chain
inspection with Polkadot.js can appear at each stage, but the connected endpoint determines what the
inspection actually shows.

In local testing, Polkadot.js inspection can show storage values, metadata, blocks, or events from
an isolated local blockchain. That can be useful for understanding a development environment, but it
should not be read as evidence about public Bittensor network state.

On testchain, inspection can show records from a shared testing network. This gives stronger
evidence about public-network assumptions than a private local run, while still keeping the result
separate from production mainchain activity.

On mainchain, inspection belongs to production Bittensor chain context. A mainchain inspection claim
should keep the endpoint, network, pallet or query surface, account or subnet key, and block context
attached because live state can change as new blocks are produced.

The
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
source frames Polkadot.js as a way to query chain state, inspect metadata, browse blocks, and submit
extrinsics when signing authority is available. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference keeps
mainnet, testnet, and localnet separate.

This distinction matters because inspection is strongest as evidence about the connected chain
context. It can show what is visible through an endpoint, but it does not by itself explain
off-chain intent, future transaction outcomes, or the same value on another network.

For readers, this keeps Polkadot.js examples from sounding broader than their endpoint supports.
Local, testchain, and mainchain contexts can all involve chain inspection, but they do not carry the
same interpretive weight.

## Storage Queries

Storage queries read data stored by runtime pallets. In Bittensor context, that can include
Subtensor module values, proxy data, balance data, subnet information, or other chain state. Some
storage items are simple values, while others are maps keyed by accounts, netuids, or other
identifiers.

For readers, a storage query is a read operation. It can answer questions about current chain state,
but it does not by itself prove why the state changed or authorize a new change. That distinction
helps keep storage articles separate from extrinsic articles: storage describes what is recorded;
extrinsics describe calls that can change state.

## Constants and Runtime Metadata

Runtime constants are values included in the chain runtime. The official Polkadot.js page uses them
as a separate inspection area from ordinary storage. Constants are useful when a reader needs to
understand configured limits or deposits that the runtime exposes, while metadata helps describe the
available pallets, calls, storage entries, and types.

The key caution is that "constant" does not mean timeless. A runtime constant may remain stable
until the runtime changes, but runtime upgrades can alter the available metadata or values. This is
why precise documentation should connect constant claims to the inspected network and source.

## Runtime Calls

Runtime calls are read-only functions that can compute or return derived information. They are
different from plain storage reads because the runtime may do work before returning the result. The
official inspection page presents runtime calls as useful for derived values such as subnet
information or transaction-fee estimates.

This distinction is helpful for interpreting source material. A storage value is data at a storage
location. A runtime call can package or compute information from the runtime interface. Both are
read paths, but they answer different kinds of questions.

## Extrinsic Boundary

Polkadot.js can also submit extrinsics, but that crosses from inspection into action. An extrinsic
is a signed or otherwise authorized call submitted to the chain. The official page describes this
area as useful for operations not covered by other tooling or for signing flows such as air-gapped
devices.

For Taopedia readers, the boundary is security-sensitive: viewing chain state is not the same as
signing a call. Any article that discusses Polkadot.js should avoid implying that inspection
requires private keys. Private key exposure belongs to wallet and workstation-security topics, not
ordinary public chain browsing.

## Blocks and Events

The block explorer view is useful for seeing recent blocks, included extrinsics, and emitted events.
Blocks give chain data a time and ordering context. Events help explain what the runtime emitted
after calls were included, which can be useful when interpreting transaction outcomes or chain
activity.

This makes block inspection different from a simple current-state query. A state query shows what is
true at the queried point. Block and event inspection helps explain when an included action appeared
and what records accompanied it.

## Historical and Metadata Context

The official source notes that chain state can be inspected at a specific block and that metadata
matters for tools such as Polkadot Vault. That gives readers two useful boundaries. Historical
queries can explain past state, while current queries explain the current endpoint state. Metadata
updates matter because tooling must understand the runtime it is interacting with.

This article should therefore be read as a map of inspection concepts, not as a live explorer,
endpoint list, signing guide, or transaction tutorial.

## Inspection as Evidence

Polkadot.js inspection can provide evidence for claims about public chain state. Storage values,
runtime constants, blocks, events, metadata, and runtime-call results are all visible through the
connected chain interface, so they can help a reader verify that a record exists or that a value was
exposed by a particular runtime context.

That evidence still has limits. A visible record does not by itself explain off-chain intent, wallet
custody practices, project governance choices, or why a participant submitted an action. Inspection
is strongest when the article claim is about chain-visible state; it is weaker when the claim
depends on documents, source repositories, social context, or private operational decisions.

## Citation and Network Context

The network being inspected is part of the evidence. The Bittensor network docs distinguish mainnet,
testnet, and local network environments, and the inspection docs distinguish endpoint selection from
the state being queried. A claim about a value, record, constant, block, or event is therefore
clearer when it names the network context or avoids pretending that one environment settles every
Bittensor claim.

For Taopedia-style reference writing, chain inspection is useful as corroboration rather than as a
replacement for official conceptual sources. Official docs can explain what a storage area, event
family, runtime call, or constant means; inspection can show how a connected chain exposes that
surface. Those are complementary source types, not interchangeable ones.

## Authority and Observation Boundaries

Inspection should stay separate from authority. Looking at storage, constants, metadata, blocks, or
events does not require a reader to control the account involved in those records. Submitting an
extrinsic is different because it can require signing authority and can change chain state if the
runtime accepts the action.

This boundary is why a chain-inspection article should not blur public observation with wallet
operation. Public records can be inspected without exposing custody material, while signed actions
belong to wallet, permissions, and transaction topics. Keeping those categories separate reduces
confusion between what the chain makes observable and what an account holder can authorize.

## Relationship to Chain Reads

Inspecting the chain with Polkadot.js and chain reads are related but different scopes in Bittensor
chain observation vocabulary. Chain reads name observation of public chain state without submitting
a new action, while inspecting the chain with Polkadot.js names a tool-based workflow for querying
storage, constants, blocks, and runtime calls on Subtensor. The
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) documentation states
that reading chain state is free, and the
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
documentation frames the app as a way to query chain state and inspect runtime metadata.

For readers, chain reads name the observation concept, while inspecting the chain with Polkadot.js
names one interface used to carry out that kind of public inspection.

References: [Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)

## Relationship to Subtensor

Inspecting the chain with Polkadot.js and Subtensor are related but different parts of Bittensor
chain vocabulary. Subtensor names Bittensor's layer 1 blockchain and system of record, while
inspecting the chain with Polkadot.js names a tool workflow for looking at that chain's public
state, constants, blocks, and runtime interface. The
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor) describes the
blockchain layer, and the
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
documentation frames the app as a way to query Subtensor data.

For readers, Subtensor names the chain layer, while inspecting the chain with Polkadot.js names a
tool path for observing public information on that layer.

References: [Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)

## Relationship to Block

Inspecting the chain with Polkadot.js and a block are related but different parts of Bittensor chain
vocabulary. A block names one unit of chain progression on Subtensor, while inspecting the chain
with Polkadot.js includes a block explorer view for seeing recent blocks, included calls, and
emitted events. The [Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)
describes a block as a unit of chain data, and the
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
documentation describes browsing blocks as part of chain inspection.

For readers, a block names one progression unit on chain, while inspecting the chain with
Polkadot.js names a tool workflow that can browse those units and their included records.

References: [Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)

## Relationship to Transaction Fees

Inspecting the chain with Polkadot.js and transaction fees are related but different parts of
Bittensor fee vocabulary. Transaction fees name the TAO cost of state-changing chain actions, while
inspecting the chain with Polkadot.js can use runtime calls to estimate fees before submission. The
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) documentation explains
fee-bearing and fee-free actions, and the
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
documentation presents runtime calls as useful for derived values such as transaction-fee estimates.

For readers, transaction fees name the cost of submitted actions, while inspecting the chain with
Polkadot.js can help estimate those costs through read-only runtime calls.

References: [Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
