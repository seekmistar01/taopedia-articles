# Weights Version Key

Weights version key is a per-subnet hyperparameter that sets a minimum version a validator's weight
submission must carry. Documentation states that if the version key specified in a `set_weights`
call is lower than this system-wide setting, the transaction fails.

## What It Controls

The hyperparameter gates weight submissions by version. When a validator submits weights, the call
carries a version key, and that key must be at least the subnet's weights version key for the
transaction to succeed. It does not change the weights a validator chooses; it only decides whether
a submission is accepted based on the version it declares.

Raising the setting requires validators to move to a matching version before their weights are
accepted again, so stale submissions are rejected rather than silently accepted
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Default and Setting

The documentation lists a default of 0 and marks the parameter as owner-settable, so a subnet owner
can raise it for their subnet. A default of 0 means no minimum is enforced. The value in force on
netuid 1 can differ from the documented default and from another subnet's required version
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Distinction from Validator Permit

The version key check rejects a `set_weights` call whose declared version is below the subnet's
required minimum, while the validator permit authorizes a neuron to set validator-style weights on
other neurons at all
([Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Weights version key** — compatibility gate on declared client version.
- **Validator permit** — authorization gate for counting validator-style weights.

## Distinction from Weight Setting

The weights version key sets the minimum version a `set_weights` call must carry to be accepted.
Weight setting is the action that submits that vector, so a call carrying a version below the key is
rejected before weights are recorded
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

- **Weights version key** — the minimum version a weight submission must carry to be accepted.
- **Weight setting** — the validator scoring action that must satisfy that gate.

## Distinction from Weights Rate Limit

The weights version key sets the minimum version a `set_weights` call must carry to be accepted,
while weights rate limit sets how many blocks a validator must wait between weight commits
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Weights version key** — the minimum version a weight submission must carry.
- **Weights rate limit** — the blocks a validator must wait before the next commit.

## Distinction from Min Allowed Weights

The weights version key sets the minimum version a `set_weights` call must carry to be accepted,
while min allowed weights sets the minimum number of weights a validator must include in that
submission
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Weights version key** — the minimum version a submission must carry.
- **Min allowed weights** — the fewest weights a submission must include.

## Distinction from Commit Reveal Period

The version key rejects submissions carrying an outdated weights version, while commit reveal period
sets how many tempos pass before a committed weight set is revealed under commit-reveal. One screens
which submissions are accepted; the other delays when accepted weights become visible
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

- **Weights version key** — gates submissions by required version.
- **Commit reveal period** — tempos before committed weights are revealed.

## Distinction from Weight Matrix

The weights version key gates which weight submissions are accepted by client version. The weight
matrix is the accumulated validator weight state Yuma Consensus reads each epoch
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)).

- **Weights version key** — compatibility gate on declared submission version.
- **Weight matrix** — accumulated validator weights read by consensus.

## Distinction from Weight Vector

The weights version key sets the minimum version a `set_weights` call must carry, while the weight
vector is the scored targets the validator submits
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Weights version key** — the minimum version a weight submission must carry to be accepted.
- **Weight vector** — the submitted scores the version key must accompany.

## Distinction from Validator-Miner Bonds

The weights version key gates whether a `set_weights` submission is accepted. Validator-miner bonds
EMA-smooth validator investment in miners during consensus aggregation
([Glossary: Weights Version Key](https://docs.learnbittensor.org/resources/glossary#weights-version-key),
[Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)).

- **Weights version key** — compatibility gate before weights are recorded.
- **Validator-miner bonds** — smoothed validator-miner investment during aggregation.

## Distinction from Subnet Task

The weights version key gates submission compatibility. Subnet task defines the work miners perform
that validators score when submitting weights
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task)).

- **Weights version key** — minimum version a weight submission must carry.
- **Subnet task** — defined work miners perform inside the subnet.

## Chain Reads for netuid 1

Readers can verify live hyperparameter values for the documented example netuid with
`btcli subnet hyperparameters --netuid 1 --network finney`

That read path keeps live hyperparameter claims tied to a parseable netuid.

## Per-Subnet Live Value Boundary

Weights version key is per-subnet chain state. The minimum accepted version configured on one netuid
filters validator weight submissions on that subnet, but the live key can differ from reference
defaults and from the key on another netuid
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

This article's infobox uses netuid 1 as an example label when reading one subnet's hyperparameter
table on Finney mainnet. That example helps readers verify one row; it is not proof of identical
version gates everywhere
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Weights version key filters submissions; Yuma Consensus still aggregates eligible weights into
emission shares at epoch boundaries on the active subnet
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Weights version key** — minimum accepted version for weight submissions on a subnet.
- **Per-subnet boundary** — live value is chain state on one netuid.

## Distinction from Yuma Consensus

Weights version key is the per-subnet hyperparameter that sets the minimum version a validator's
`set_weights` call must declare. It names a compatibility gate on the submission path, not the
settlement step that converts aggregated weights into emission shares
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Weight Setting](https://docs.learnbittensor.org/resources/glossary#weight-setting)).

Yuma Consensus is the on-chain mechanism that runs at the epoch boundary on the selected netuid. It
reads the weight matrix from eligible validators whose submissions pass version and structural
checks, applies clipping and bonding, and converts the result into miner incentives and validator
dividend shares ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

When subnet owners raise the version key, stale calls are rejected before weights enter the matrix.
That gate screens compatibility; Yuma still settles only the accepted submissions that survive
clipping and bonding—not the version number by itself
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

Readers should treat weights version key as a per-netuid compatibility floor, and Yuma as the
mechanism that allocates rewards from accepted weight vectors at the epoch boundary.

- **Weights version key** — minimum declared version required for a valid weight submission.
- **Yuma Consensus** — epoch-boundary settlement from accepted validator weights to emission shares.

## Reader Boundary

This page defines the concept at a high level. It does not report the weights version key set on any
particular subnet or whether a given validator's submission currently meets it. Those are live chain
state and should be checked for the relevant netuid. The default of 0 is the documented value, and
the parameter is owner-settable per subnet.
