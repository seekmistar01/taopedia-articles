# Inspecting the Chain with Polkadot.js

Inspecting the chain with Polkadot.js means using the Polkadot.js browser app to observe Bittensor
chain data
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

The topic belongs to chain evidence and tool-interface vocabulary. It should be read separately from
concept explanation, signing authority, and state-changing submissions.

## Inspection Context

The inspection guide describes Polkadot.js as a way to query chain state, inspect runtime metadata,
browse blocks, inspect events, and submit extrinsics when authority is available
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

That makes the tool useful for evidence gathering. It can show what the connected chain exposes, but
it does not by itself explain every mechanism behind the value being inspected.

Polkadot.js can provide a direct view into chain surfaces, while the relevant Subtensor reference
explains what the observed surface means.

## Network Scope

The connected network is part of the evidence. Bittensor documentation separates mainnet, testnet,
and localnet contexts
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)).

A Polkadot.js result should therefore be read with its connected endpoint and environment attached.
A value observed in one environment should not be treated as proof of the same state in another.

Localnet inspection can demonstrate local mechanics. Testnet inspection can show shared
non-production behavior. Mainnet inspection is the production chain context. The interface can look
similar across those settings while the evidence scope is different.

## Read-Only Surfaces

Polkadot.js can inspect read-oriented Subtensor surfaces such as storage entries, runtime constants,
runtime metadata, blocks, and events
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

Reference pages and inspection results answer different questions. A reference page names the
runtime surface, while an inspection result shows what the connected chain exposes in the selected
context.

That keeps read-only evidence bounded. A storage lookup, constant value, or metadata view shows a
chain surface in context; it does not replace the documentation that explains the broader concept.

Runtime metadata and constants are especially useful for understanding what the connected chain
exposes before any transaction is signed. Metadata shows available runtime structure, constants show
configured values, and storage entries show selected state
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

## Blocks and Events

Block browsing gives ordering context. A block can show included calls, while events can show named
runtime records emitted during processing
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

That makes block and event inspection different from a storage lookup. Storage can show a value,
while blocks and events help identify when runtime activity appeared and which records accompanied
it.

The distinction helps avoid overreading one inspection result. A block, event, and storage value can
be related evidence, but they answer different questions about chain activity.

## Extrinsic Boundary

Polkadot.js can expose runtime calls and can also submit extrinsics when signing authority is
available. The Subtensor extrinsics reference describes submitted runtime methods
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

This is the main boundary for readers. Observing chain data is not the same as submitting a
state-changing action, and selecting a call interface is not evidence that a call succeeded.

Inspecting the chain is a read-side activity. The Polkadot.js inspection source explains how readers
can view chain state, while BTCLI permissions distinguish permissionless reads from actions that
require private key authority
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Bittensor CLI: Permissions Guide](https://docs.learnbittensor.org/btcli/btcli-permissions)).

That distinction matters for safety language. Looking up public state should not be described as
equivalent to signing an extrinsic, transferring value, setting weights, or changing staking
positions.

## Reader Boundary

Inspecting the chain with Polkadot.js should be read as public chain inspection through a Substrate
interface. The evidence shows that a connected chain exposed a record, value, metadata item, block,
event, or runtime call in the inspected context
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

The stable reference point is inspection scope: Polkadot.js can show chain data, while the network,
endpoint, block context, and action type determine what that observation can support.

## Subtensor Names the Connected Chain

Polkadot.js inspection connects to Bittensor's Subtensor blockchain layer rather than to a single
subnet application. The
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor) describes
Subtensor as Bittensor's layer 1 blockchain and system of record for transactions, rankings, and
incentive flow.

Subnets define subnet markets and evaluation context, while Subtensor is the shared chain layer
where balances, staking records, and subnet registrations are recorded
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

An inspected value or block therefore belongs to Subtensor chain evidence. It can support
subnet-related questions, but the inspection interface is observing the protocol record layer.

References: [Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)

## Constants Differ from Stored State

Runtime constants and storage entries can both appear in Polkadot.js, but they answer different
inspection questions. The Subtensor constants reference groups named runtime values such as
defaults, bounds, and limits, while the storage reference exposes places where chain state is
recorded ([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Constants name reference values configured in the runtime. Storage entries name state surfaces that
can change through accepted chain activity. The constants documentation distinguishes constants from
mutable storage, which helps keep inspection prose precise when both values appear in one workflow.

A constant read shows what the connected runtime calls a limit or default. A storage read shows what
the connected chain currently records at the selected lookup. Those results can be related, but they
are not the same kind of evidence.

References: [Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Block Context Anchors State Reads

Chain inspection often happens at a selected block context even when the interface looks like a
simple lookup. The [Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)
describes a block as a unit of Bittensor blockchain data that contains transactions and has a unique
block hash.

Stored state is read in chain-history context. A storage value observed at one block can differ from
the value at another block on the same network, so the block reference should stay attached to the
inspection result ([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Block browsing and direct storage lookups serve related but different roles. Browsing shows ordering
and included activity, while the selected block context tells readers which point in chain history a
state read represents.

References: [Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)
