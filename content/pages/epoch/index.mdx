# Epoch

An epoch is the consensus period inside a Bittensor subnet
([Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)). The period is tied to
subnet tempo, and it is the timing context in which subnet consensus work is executed.

## Consensus Period

Epoch is timing vocabulary for subnet consensus. It does not name the subnet task, the miner output,
or one validator's individual evaluation. It names the period in which the subnet's consensus
mechanism runs over the relevant validator signals
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

That makes epoch useful as a boundary term. It tells the reader when consensus work is grouped, not
what useful work the subnet is designed to evaluate.

## Tempo Boundary

Epoch and tempo are related but different timing terms. Tempo is the cadence that determines when
epochs occur, while epoch is the consensus period produced by that cadence
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

This distinction keeps interval language clear. Tempo describes the cadence; epoch describes the
period in which consensus runs at that cadence.

## Validator-Signal Context

Epochs matter because Bittensor consensus uses validator evaluation signals. In
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus), validator weights are
aggregated into subnet outcomes. The epoch is the timing period in which that consensus work is
organized.

The term should therefore be read with the validator-signal context attached. An epoch is not a
standalone quality score; it is a period for processing consensus inputs.

## Subnet Mechanism Context

Each subnet defines useful work and evaluation standards through its incentive mechanism. The
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
documentation places task design, protocol, and scoring method inside subnet design. Epoch belongs
to that setting as timing context for the subnet's consensus period.

This keeps epoch claims scoped to the subnet mechanism being discussed. The period has meaning
because it belongs to a specific subnet's consensus path.

## Multiple Mechanisms

When a subnet uses multiple incentive mechanisms, epoch language still needs mechanism context. The
[Multiple Incentive Mechanisms](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)
documentation explains that each mechanism can have independent consensus calculations.

That means epoch remains a subnet consensus-period term, but the relevant mechanism determines which
evaluation path is being discussed during that period.

## Reading Boundaries

Epoch is the right term when the focus is a subnet consensus period. If the focus is cadence,
[tempo](https://docs.learnbittensor.org/resources/glossary#tempo) is more precise. If the focus is
aggregation logic, [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) is more
precise. If the focus is the evaluation signals, validator weights are more precise.

These boundaries keep epoch focused on consensus timing, not a subnet task description, a validator
ranking, or a full explanation of reward outcomes.
