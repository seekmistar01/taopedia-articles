# Local Blockchain

A local blockchain is a private Bittensor chain used for development and testing. It belongs to
local development context, not public network context.

[Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain) is described
in the official glossary as a private, isolated chain for developing and testing subnet incentive
mechanisms. The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)
reference places localnet beside mainnet and testnet as a distinct environment.

## Development Context

The [Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain) context
is isolated from mainnet and testnet. It lets developers evaluate Bittensor components and subnet
behavior without treating the result as shared network state.

That separation is the main point of the term. Local behavior can be useful development evidence,
but it should not be read as proof of the same state, timing, or behavior in another Bittensor
environment.

## Localnet Relationship

Local blockchain and localnet are closely related but not identical terms. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference uses
localnet as the local environment label, while
[Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain) names the
private chain used inside that context.

This distinction keeps the environment label separate from the chain instance. Localnet is the
context; local blockchain is the isolated chain where development testing happens.

## Public Network Boundary

Local blockchain sits on the opposite side of public chain access. The
[Public Subtensor](https://docs.learnbittensor.org/resources/glossary#public-subtensor) glossary
entry refers to publicly reachable access to Subtensor, while local blockchain refers to private
development context.

Outputs from a local blockchain should not be treated as Public Subtensor state. They can explain
local behavior, but public-access claims need evidence from the public network context being
discussed.

## Timing Context

Local blockchain articles often sit near fast-block and non-fast-block vocabulary. The
[Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks) and
[Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks) glossary
entries describe local-chain timing configurations.

Those timing terms remain local-development context. They can help explain local testing behavior,
but they should not be generalized to mainnet or testnet without a source that supports the network
being discussed.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from local testing to testchain and then mainchain. A local
blockchain belongs to the first stage of that path.

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
keeps local evidence in its proper place: a local result can inform development, a testnet result
can support shared testing context, and a mainnet result belongs to production network context.

## Evidence Boundary

The [Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain) entry
supports claims about private local development chains. It does not support endpoint
recommendations, deployment instructions, public network-state claims, or private account diagnosis.

When an article uses local blockchain evidence, it should keep the local context visible. Claims
about public network behavior, production use, or subnet economics need source support from the
environment where those claims apply.
