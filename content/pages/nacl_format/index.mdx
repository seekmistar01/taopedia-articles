# NaCl Format

NaCl Format is Bittensor terminology for a secure encryption format used around legacy wallet files.
The [NaCl Format glossary entry](https://docs.learnbittensor.org/resources/glossary#nacl-format)
places the term in wallet-security vocabulary rather than subnet, tokenomics, or consensus
vocabulary.

## Format Role

NaCl Format names an encryption-format role. It is not a Bittensor account role, address format, or
network feature. In the
[NaCl Format glossary entry](https://docs.learnbittensor.org/resources/glossary#nacl-format), the
important boundary is that the term points to a format using the NaCl library for wallet-file
security context.

## Legacy Wallet Context

The term is tied to legacy wallet security. Bittensor's
[wallet documentation](https://docs.learnbittensor.org/keys/wallets) gives the broader wallet
context, while the NaCl glossary entry narrows the subject to an encryption format used for older
wallet-file protection. That makes NaCl Format a compatibility and security term, not a new wallet
type.

Legacy context is important because wallet vocabulary can otherwise mix several different layers:
the account role, the key material, the local file, and the protection format around that file. NaCl
Format belongs to the last layer. It describes how stored wallet material is protected in a legacy
file-format context, while the wallet article explains the surrounding key and account concepts
([Bittensor wallet documentation](https://docs.learnbittensor.org/keys/wallets),
[NaCl Format glossary entry](https://docs.learnbittensor.org/resources/glossary#nacl-format)).

That keeps the phrase from being read as a network-role status. A wallet file can use a protection
format without saying whether the associated account is staking, assigned to a subnet role, holding
TAO, or participating in validation. Those are chain-state and role questions, not NaCl-format
questions.

## Encryption Boundary

Encryption format and wallet behavior are related but different ideas. NaCl Format concerns how
wallet-file material is protected in a legacy format; it does not define staking, emissions, subnet
roles, or address ownership. The
[NaCl Format glossary entry](https://docs.learnbittensor.org/resources/glossary#nacl-format) keeps
the term at that file-format level.

The boundary is also an at-rest boundary. NaCl Format describes protection around stored wallet
material, while wallet use still depends on how the relevant key is handled when an action must be
authorized. The wallet docs provide the broader custody context, and the
[Private Key glossary entry](https://docs.learnbittensor.org/resources/glossary#private-key) anchors
the protected authorization side of a key pair.

This distinction prevents overstating. A protected file format can reduce the meaning of a copied
legacy file, but it does not turn an unsafe workstation into a safe one, replace private-key
custody, or explain every wallet threat. It is one storage-format concept inside the larger wallet
security model.

## Relationship to Wallets

Wallets are the surrounding concept, because wallet files and wallet security provide the setting
where NaCl Format appears. The [wallets guide](https://docs.learnbittensor.org/keys/wallets)
provides the broader Bittensor wallet context, while NaCl Format remains narrower: it is one
encryption-format label inside that context.

## Relationship to Local Wallet Files

NaCl Format is most naturally read next to local wallet-file vocabulary. A local wallet describes
wallet files kept on a user's machine, while NaCl Format describes a legacy encryption-format label
around protected wallet-file material
([Bittensor wallet documentation](https://docs.learnbittensor.org/keys/wallets),
[NaCl Format glossary entry](https://docs.learnbittensor.org/resources/glossary#nacl-format)).

For article use, the local-file context should stay attached. A NaCl-format statement is about the
stored file format, not about a remote account database, subnet-role record, or balance context. If
the question is where wallet files live, wallet-location vocabulary is more precise. If the question
is how a legacy file is protected, NaCl Format is the narrower term.

## Relationship to Public and Private Keys

NaCl Format sits near public-key and private-key terminology because wallet files protect key
material. A [public key](https://docs.learnbittensor.org/resources/glossary#public-key) is shareable
material for account reference, while the
[NaCl Format glossary entry](https://docs.learnbittensor.org/resources/glossary#nacl-format)
concerns the protection format around legacy wallet files. The terms belong to the same security
area but answer different questions.

Private-key vocabulary remains separate as well. A private key names protected authorization
material, while NaCl Format names a file-format protection context around stored wallet material
([Private Key glossary entry](https://docs.learnbittensor.org/resources/glossary#private-key),
[NaCl Format glossary entry](https://docs.learnbittensor.org/resources/glossary#nacl-format)). That
difference matters because the format label does not itself identify which wallet role is being
authorized.

Public-key vocabulary points the other way. Public keys and wallet addresses can be shared for
identification or destination reference, but NaCl Format belongs to the protection of stored wallet
material. It should not be used as a synonym for an address, public key, or account identifier.

## Reader Boundary

NaCl Format should be used narrowly in Taopedia prose. It can support statements about legacy
wallet-file protection and wallet-security terminology. It should not be used to infer staking
status, subnet participation, balance state, validator status, or wallet-tool behavior without a
separate source for those statements
([NaCl Format glossary entry](https://docs.learnbittensor.org/resources/glossary#nacl-format),
[Bittensor wallet documentation](https://docs.learnbittensor.org/keys/wallets)).

That boundary keeps the article readable. NaCl Format is a storage-format term inside wallet
security context. Wallets, wallet locations, public keys, private keys, hotkeys, coldkeys, and
chain-state records provide neighboring vocabulary for other questions.

## Reading the Term

NaCl Format is best read as format vocabulary. It does not name a recovery process, wallet
procedure, subnet action, or balance state. For concept reading, the
[NaCl glossary entry](https://docs.learnbittensor.org/resources/glossary#nacl-format) is the anchor
for the term, with Bittensor wallet documentation supplying the surrounding context.
