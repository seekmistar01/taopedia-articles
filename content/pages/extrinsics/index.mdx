# Extrinsics

Extrinsics are submitted actions that ask the Bittensor chain to perform a runtime operation. They
belong to the action side of Subtensor interaction.

The official [Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)
reference lists methods exposed by the Subtensor runtime. It is a method reference for submitted
actions, not a catalog of passive chain reads.

## What Extrinsics Represent

An extrinsic is a submitted request for the runtime to process. The
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) reference lists
runtime methods that can be called through that submitted-action path. The method name identifies
the action family, while the selected network and surrounding chain state determine the context of
the submission.

This makes extrinsics different from ordinary observation. A method entry can show that a runtime
action exists, but it does not by itself prove that a particular submission will succeed or explain
every requirement around the action.

## Read Versus Submit Boundary

The [Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) documentation
separates free chain reads from state-changing actions. That distinction is the main boundary for
extrinsics: reading public chain state observes the network, while submitting an extrinsic asks the
runtime to process a change.

The boundary matters for terminology and safety. Inspecting chain data does not sign a submitted
action, pay a fee, or ask Subtensor to record a new state change. An extrinsic belongs to the side
of the workflow where the runtime evaluates a submitted call.

## Method Reference Scope

The [Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) page is a
runtime method reference. It can identify supported submitted actions, but it is not a complete
procedure guide for every wallet, interface, account condition, or network state that may affect a
particular submission.

That scope helps avoid over-reading method names. A listed method establishes the runtime action
surface. Successful use still depends on the rule checks, state, signing authority, fees, and
network context involved in the actual submission.

## Submission and Outcome Context

An extrinsic describes what was submitted to the runtime. The outcome may involve a completed state
change, emitted events, changed storage values, fees, or an error code when the runtime rejects the
request. These surrounding signals answer different questions from the method reference itself.

The [Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events) reference lists emitted
runtime records, the [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)
reference lists queryable state entries, and the
[Subtensor Error Codes](https://docs.learnbittensor.org/errors) pages describe rejection families.
Together, they provide context around what happened after a submitted action was processed or
rejected.

## Network Scope

Extrinsics are interpreted inside a selected Bittensor network environment. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet as deployment contexts. A submission belongs to the chain
history of the environment where it was processed.

The same method name can appear across environments while referring to different chain histories. A
testnet submission does not establish mainnet state, and a localnet submission remains confined to
that local environment.

## Evidence Boundary

The [Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) reference is
evidence for submitted runtime actions. It can identify the action family sent to Subtensor and help
connect the submission to fees, events, storage, or errors. It does not by itself explain off-chain
intent, private custody practice, interface behavior, or every cause behind the resulting state.

This boundary keeps extrinsics from replacing source-specific mechanism documentation. The method
reference names the submitted action surface; related official sources such as
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) and
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) explain fee
handling and network context.
