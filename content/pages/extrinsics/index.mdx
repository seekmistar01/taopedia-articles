Extrinsics are submitted actions that ask the Bittensor chain to perform a runtime operation. The
official [Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) reference
lists extrinsic methods exposed by the Subtensor runtime.

For Taopedia readers, the useful idea is that an extrinsic is a state-changing chain action, not a
passive read. Reading chain data is different from submitting an action that the runtime must
process.

## How Extrinsics Fit

Extrinsics are the transaction-like side of interacting with Subtensor. They are used when software
or a user asks the chain to update state through a supported runtime call.

The official [transaction fees](https://docs.learnbittensor.org/learn/fees) page is useful context
because it separates chain reads from state-changing actions. Chain reads are free, while
state-changing actions can carry fees depending on the operation.

## Why the Distinction Matters

The read-versus-submit distinction helps explain why some Bittensor actions only inspect current
state, while others ask the chain to record a change. An extrinsic belongs to the second group: it
is evaluated by the runtime and can affect chain state if accepted.

This article does not try to catalog every extrinsic. The official reference is the right source for
the current method list and exact behavior.

## Relationship to Mempool

Extrinsics and the mempool are related but name different parts of a submitted chain action. An
extrinsic is a state-changing runtime request, while the
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) is where pending
transactions wait before block inclusion.

For readers, an extrinsic is the submitted action itself, and the mempool is the pending holding
period before that action may be bundled into a block. A chain read is not an extrinsic, and a
pending mempool entry has not yet become accepted chain history.

References: [Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)

## Relationship to Blocks

Submitted extrinsics are bundled into blocks for the runtime to process. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#block) describes a block as a unit of
chain data containing transactions, so an accepted extrinsic becomes part of a block rather than
remaining a standalone request. For readers, this keeps the extrinsic as the submitted action and
the block as the unit that records and orders those actions on chain.

References: [Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)

## Relationship to Transaction Fees

Extrinsics and transaction fees address related but different parts of a submitted chain action. An
extrinsic is a state-changing runtime request, while
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees) describes the costs the
chain can attach to those actions, including weight and length components.

For readers, an extrinsic names the submitted action, while transaction fees describe the fee
accounting that may apply when that action is processed.

References: [Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)

## Relationship to Multiple Mechanisms

Extrinsics are state-changing chain actions processed by the Subtensor runtime. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-matrix) notes that each
incentive mechanism has its own weight matrix for independent consensus calculations.

For readers, an extrinsic still names a submitted runtime action. Weight-related extrinsics in a
subnet that runs more than one incentive mechanism should be read with that mechanism's matrix
context.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)

## Relationship to Subtensor Events

Extrinsics and Subtensor events address related but different parts of a submitted chain action. An
extrinsic is a state-changing runtime request, while a Subtensor event is a named record the runtime
may emit when that activity is processed. The
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) reference lists
submitted methods, and the [Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)
reference lists runtime event names grouped by runtime area.

For readers, an extrinsic is the submitted action, while an event is a compact record that can help
identify what the runtime recorded. Events should not be treated as a complete substitute for
reading the extrinsic reference or the surrounding chain context.

References: [Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)

## Relationship to Subtensor Storage

Extrinsics and Subtensor storage address related but different steps in chain interaction. An
extrinsic asks the runtime to change state, while storage entries name where current state can be
queried afterward. The
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) reference lists
state-changing methods, and the
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) reference lists query
definitions for existing runtime state.

For readers, extrinsics belong to the submission side of the workflow, while storage belongs to the
inspection side. Chain reads query storage without submitting an extrinsic, which is why the two
references answer different questions about the same network.

References: [Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Relationship to Subtensor

Extrinsics and Subtensor address related but different parts of Bittensor chain interaction. The
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor) defines
Subtensor as Bittensor's layer 1 blockchain and system of record for transactions, while an
extrinsic is a submitted runtime action processed inside that layer. The
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) reference lists the
methods exposed for those submissions.

For readers, Subtensor names the blockchain layer, while an extrinsic names one state-changing
request sent to that layer. Passive chain reads inspect Subtensor state without using an extrinsic.

References: [Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)

## Relationship to Network Context

Extrinsics are submitted inside a specific Bittensor network environment. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet as deployment contexts, and a submitted extrinsic affects
only the chain history of the network endpoint being used. The same extrinsic method name can appear
in each environment, but the resulting state belongs to separate networks.

For readers, this matters before submission. Software must target the intended network endpoint,
because an extrinsic accepted on testnet does not change mainnet state, and localnet submissions
remain confined to that development chain.

References: [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)

## Relationship to Coinbase

Extrinsics and coinbase are related but different parts of Bittensor chain operation vocabulary.
Extrinsics are signed transactions and calls that users and applications submit to Subtensor — the
on-chain actions that change state by registering, staking, transferring, or configuring the
network. Coinbase is the protocol-internal per-block operation that accumulates pending emissions,
injects TAO and alpha into subnet pool reserves, and checks each subnet's epoch boundary to trigger
Yuma Consensus rounds — a block-level process the protocol runs automatically, not one submitted by
users. The
[Subtensor extrinsics reference](https://docs.learnbittensor.org/subtensor-api/extrinsics) describes
the user-submitted actions, and the
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
documentation describes the automatic per-block protocol operation.

Both produce state changes on Subtensor, but from different origins. Extrinsics are submitted
externally by users and applications; coinbase runs internally each block as a protocol-driven
operation without an external submission. Extrinsics answer what users and applications can do to
change network state; coinbase answers what the protocol itself executes each block to advance
emission and consensus state.

References:
[Subtensor extrinsics reference](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
