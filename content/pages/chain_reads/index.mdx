# Chain Reads

Chain reads describe looking at public Bittensor chain state without asking Subtensor to record a
new action. Bittensor's fee documentation treats reads as free inspection, while runtime actions can
change state and involve fees
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

The term is about observation. It is separate from an extrinsic, transaction, or other action that
asks the chain to process a change.

## Read Scope

A chain read can show public state, metadata, blocks, storage values, or related records, but the
read itself does not move TAO, change stake, register a role, or alter subnet settings
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

This distinction matters because inspection and action answer different questions. A read can show
what is visible on chain, while an extrinsic asks the runtime to process a state-changing action
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

The useful reference point is the source of evidence. A chain read is evidence that a connected
chain exposed a value or record; it is not evidence that a new chain action occurred.

That makes chain reads useful for checking public facts before describing them. The read can anchor
the observation, but the claim should still name the surface that was read and avoid turning the
observation into a broader mechanism claim
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

## Inspection Surfaces

Different read targets answer different questions. Storage entries expose stored chain state, events
record runtime activity, constants name runtime values, and blocks place observations in chain order
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

The inspecting-the-chain guide places chain reads inside public chain inspection. A read can show
what a selected surface contains, but it does not by itself explain every cause behind the displayed
state
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

The surface distinction keeps read evidence precise. Storage entries describe stored state, events
describe emitted records, constants describe runtime reference values, and blocks describe ordering
context. A chain read should preserve which surface supplied the evidence
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

Constants and storage entries are both readable, but they do not answer the same question. Reading a
constant is reference inspection; reading storage is state inspection. Both can be chain reads, but
the source surface changes how the result should be described
([Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Runtime metadata belongs beside those read surfaces as structure evidence. It shows available
runtime structure, while storage and events show recorded values and emitted records in context
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

## Runtime Calls Stay on the Read Path

The inspecting-the-chain guide places runtime calls on a read-only path that can return computed
answers such as bundled subnet information or fee estimates without submitting a state-changing
extrinsic
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Runtime Calls](https://docs.learnbittensor.org/subtensor-api/runtime)).

That keeps runtime-call results inside chain-read vocabulary when the question is inspection rather
than submission.

## Block Context Narrows Historical Reads

A value read from one point in chain history should not automatically be treated as the value at
every later point on the same network
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Chain-read examples should keep block and network context attached when the observed value matters.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For chain reads on
a subnet such as netuid 1, that sequence changes how storage, event, and block-height examples
should be interpreted.

In localnet, chain reads can be tested against an isolated local blockchain. Storage and event
surfaces on netuid 1 reflect that test chain's history rather than production Subtensor state.

On testnet, read-only inspection can be exercised in a shared non-production network. Testnet block
heights and storage entries for netuid 1 are separate from mainnet state.

On mainnet, chain reads concern live production Subtensor state at selected block heights. Observed
values on netuid 1 depend on the network, surface, and block context where the read was taken
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A chain-read example from one environment should not be
read as representing production chain state in another environment.

## Distinction from Extrinsics

A chain read and an extrinsic sit on opposite sides of the chain's request boundary. A chain read
queries public state — storage, events, constants, or runtime calls — without asking Subtensor to
record anything, and the documentation treats it as free inspection, while an extrinsic is a
submitted, fee-bearing call that asks the runtime to change state such as staking, transferring, or
registering, so a read observes what already exists and an extrinsic is what produces a new recorded
change ([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

- **Chain read** — free, read-only observation of public chain state.
- **Extrinsic** — a submitted, fee-bearing call that changes chain state.

[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants) document
protocol-wide fixed values, while
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) exposes per-key state
that changes as blocks advance. Chain reads can target either surface depending on whether the
reader needs a static parameter or a live field
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

For readers, constants vocabulary names durable configuration, while storage vocabulary names
mutable chain state at a block height. Both remain read-only observation paths
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

A typical inspection path starts with choosing the network and block context, then selecting the
read surface that answers the question
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) exposes recorded values
such as stake, registration, or subnet pool state on a subnet such as netuid 1.
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events) show emitted runtime
records after actions occur, while
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants) and
[Runtime Calls](https://docs.learnbittensor.org/subtensor-api/runtime) supply reference values and
computed answers without submitting a change.

When the goal shifts from observation to mutation, the path crosses into
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) and fee-bearing
transactions ([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)). Chain
reads inform those later actions but do not replace them.

## Distinction from Transaction Fees

A chain read and a transaction fee sit on opposite sides of the free/paid boundary. Documentation
treats a chain read as free inspection of public state, while a transaction fee is the cost charged
to process a state-changing action
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

- **Chain read** — free, read-only observation of public state.
- **Transaction fee** — the cost charged to process a state-changing action.

Read-only inspection is separate from state-changing actions, so fee language belongs with the
action unless the point is specifically that a read is free
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

That boundary keeps chain-read vocabulary from being treated as transaction vocabulary. A read can
inform a later decision, but the read itself is not the fee-bearing action.

The fee split also keeps article prose from overclaiming. A read may be part of understanding chain
state before an action, but the action and its fee behavior belong to the transaction or extrinsic
surface, not to the read itself
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

## Distinction from Subtensor Events

A chain read and a Subtensor event differ in what they show. A chain read inspects current public
state at a surface such as storage, while an event is a record the runtime emitted after an action
was processed ([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

- **Chain read** — inspection of current public state.
- **Subtensor event** — a record emitted after an action is processed.

[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events) document emitted records
that follow successful extrinsic processing. Chain reads of events therefore show historical
outcomes rather than pending intent in the mempool
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

For readers, event vocabulary complements storage reads. Storage can show current values at a block,
while events show which actions produced observable state changes on the connected network
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

## Distinction from Inspecting the Chain with Polkadot.js

Inspecting the chain with Polkadot.js is one concrete way to perform a chain read. Chain reads are
the general category of looking at public chain state without asking Subtensor to record a change,
while inspecting the chain names a specific tool-based method of doing so
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

Both stay on the read side and neither writes state, which is why fee documentation treats reads
differently from runtime actions that can change chain state
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

- **Chain reads** — the general read-only category of viewing public chain state.
- **Inspecting the chain** — a Polkadot.js method of performing such reads.

## Distinction from Public Subtensor

Public Subtensor names where the chain is reached publicly; a chain read names the read-only
inspection performed through an available access path
([Subtensor Nodes](https://docs.learnbittensor.org/subtensor-nodes),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Subtensor storage documentation describes the readable chain surfaces exposed through that layer,
separate from the access route itself.

## Distinction from Bittensor CLI: Permissions Guide

Chain reads name observation of public chain state without submitting a new action, while BTCLI
permissions place permissionless reads in the lowest-risk category of the CLI permission model
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Bittensor CLI: Permissions Guide](https://docs.learnbittensor.org/btcli/btcli-permissions)).

Observation vocabulary and CLI-permission vocabulary answer different questions. Chain reads name
the observation concept; BTCLI permissions name which CLI tasks stay permissionless.

- **Chain reads** — observation of public chain state without submitting an action.
- **BTCLI permissions** — which CLI tasks require no private key.

## Distinction from Yuma Consensus

Chain reads are queries that return live hyperparameter, metagraph, or storage values from Bittensor
without submitting a state-changing extrinsic. They name observation vocabulary, not the on-chain
step that turns validator weight submissions into emission shares
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

Yuma Consensus is the on-chain process that converts submitted validator weights into emission
shares after evaluation completes. Read queries can show weights or emissions outcomes, but Yuma
Consensus names the settlement mechanism rather than the query format
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers should keep read-query vocabulary separate from the settlement step. Chain reads answer what
live values can be queried; Yuma Consensus answers how validator weights become emission shares in a
subnet ([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Chain reads** — live queries of chain state without changing it.
- **Yuma Consensus** — on-chain weight aggregation that settles emissions.

## Distinction from Evidence

Chain reads are evidence of visible public chain state from the selected read surface. They are
strongest when paired with the surface type being used: storage state, an emitted event, a runtime
constant, or a block-context observation
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

That evidence still has limits. A storage read can show a recorded value, but surrounding mechanism
documentation may be needed to explain why the value exists. An event can show that runtime activity
was emitted, but the event name is not the same as the full action reference
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

## Distinction from Network

A chain read belongs to the network being inspected. Mainnet, testnet, and localnet are separate
Bittensor network contexts, so an observation from one environment should not be treated as the
chain history of another
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

## Reader Boundary

Chain reads identify read-only observation. They do not prove intent, predict a later action,
replace mechanism documentation, or show private state that is not on chain
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Use the term when the article is about observing public chain state.
