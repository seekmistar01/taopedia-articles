Subnet 116 is a Bittensor subnet with no registered on-chain identity.

## What Subnet 116 Provides

The Finney identity query for netuid 116 returns no registered project information. The subnet name,
description, GitHub repository, project URL, and Discord fields are all absent from the on-chain
record.

## Miner and Validator Roles

Subnet 116 operates under the standard Bittensor two-role structure. Miners supply a capability to
the network, and validators evaluate those contributions and set weights. Those weights feed into
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus), which converts validator
assessments into emissions that reward the strongest performers.

## On-Chain Identity

The Finney identity query for netuid 116 returns no registered identity record. All fields — subnet
name, description, GitHub repository, project URL, and Discord — are absent. Live subnet data is
available on [TaoStats](https://taostats.io/subnets/116).

## Relationship to Yuma Consensus

Subnet 116 is registered on Bittensor and uses
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) to convert validator weight
vectors into the emission shares distributed to miners and validators within the subnet each tempo.
The [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) documentation describes
how validator weight submissions are aggregated into consensus weights for each miner registered on
the subnet.

Because Subnet 116 has no registered on-chain identity and no public source describing how
validators score miners, no source-backed description of its incentive mechanism is available. The
[Emission](https://docs.learnbittensor.org/learn/emissions) documentation describes how consensus
weights determine each participant's share of the subnet's accumulated emission each tempo.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Subnet 116,
that sequence applies to the standard Bittensor lifecycle: localnet for isolated development,
testnet for shared non-production testing, and mainnet for live operation with real emissions.

On mainnet, Subnet 116 is registered as the live production subnet at netuid 116. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. Participation examples or emission outcomes from one
environment should not be read as representing production subnet performance in another environment.

## Netuid 116 Identifies the Subnet On-Chain

Bittensor assigns every subnet a unique numeric identifier called a netuid, and Subnet 116 is the
subnet registered at netuid 116
([Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid)). The
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) reference
explains that each subnet runs its own incentive mechanism while sharing the same underlying
Subtensor chain, so the netuid is the stable handle that distinguishes Subnet 116 from every other
subnet.

For a reader, this means "Subnet 116" and "netuid 116" refer to the same on-chain slot. A claim
about Subnet 116 should be tied to that netuid rather than to the registered name alone, because the
name field can be changed on-chain while the netuid stays fixed.

## Reader Boundary

Subnet 116 should not be read as generic Bittensor subnet documentation, confirmation that the
subnet has a published, working mechanism, or a substitute for the subnet's own primary sources. It
names the on-chain subnet slot registered at netuid 116; see this article's On-Chain Identity
section for its current registered fields
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid)).
