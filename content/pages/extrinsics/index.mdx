# Extrinsics

Extrinsics are chain actions that ask the Bittensor runtime to process an operation
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

The term belongs to the action side of Subtensor interaction. It is separate from chain reads, which
observe public state without asking the runtime to process a state change.

## Read and Write Boundary

Transaction-fee documentation separates free chain reads from state-changing actions
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

That distinction is the main boundary for extrinsics. Reading public chain state observes the
network, while an extrinsic asks the runtime to process an action. Storage and inspection references
belong to read-focused claims, while the extrinsics reference belongs to action-focused claims
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

## Signing Boundary

An extrinsic still needs the right authority before it can be submitted as a signed action. The
BTCLI permissions guide separates actions that require no private key from actions requiring hotkey
or coldkey authority, so the extrinsic name alone does not prove which wallet key should sign it
([Bittensor CLI: Permissions Guide](https://docs.learnbittensor.org/btcli/btcli-permissions),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

For readers, this keeps the action surface separate from custody. Extrinsics describe what the
runtime is asked to process, while wallet permissions describe who can authorize that request.

## Method Reference Scope

The Subtensor extrinsics reference can identify supported action families, but it is not a complete
explanation of every interface, condition, or network context that may affect an action
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

That keeps method names tied to what was requested. Claims about success, failure, emitted records,
or resulting state need the relevant event, storage, error, fee, or mechanism source.

## Outcome Evidence

An extrinsic names the requested runtime action. The outcome may involve changed storage, emitted
records, transaction fees, or a rejection when the runtime does not accept the action
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

Those surfaces answer different questions. The extrinsic identifies the action surface; events,
storage, and errors provide surrounding evidence about what happened afterward.

## Some Extrinsics Are Fee-Free

Not every extrinsic carries the base weight-and-length fee that state-changing actions usually pay.
Fee documentation lists `set_weights`, `commit_weights`, `reveal_weights`, and `sudo` as extrinsics
that pay neither the weight component nor the length component
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

That exception list is about specific named methods rather than a general rule. An extrinsic's fee
status still has to be checked against the documented fee-free list rather than assumed from the
general state-changing-action pattern described above.

## Submission Lifecycle

An extrinsic is easiest to read as one stage in a chain-action lifecycle. It begins as a submitted
runtime action, can pass through pending transaction context, and then needs block, event, error, or
storage evidence before stronger outcome claims are made
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

| Stage             | What the stage names                             | Useful neighboring source      |
| ----------------- | ------------------------------------------------ | ------------------------------ |
| Action surface    | The method submitted to the runtime              | Subtensor extrinsics reference |
| Pending context   | The transaction waiting before block inclusion   | Mempool glossary               |
| Included context  | The transaction recorded in a block              | Block glossary                 |
| Runtime record    | A named event emitted while processing activity  | Subtensor Events               |
| Rejection context | An error explaining why a call was not accepted  | Subtensor Error Codes          |
| State surface     | Public state inspected after processing          | Subtensor Storage              |
| Cost context      | Transaction fees attached to state-changing work | Transaction Fees               |

This lifecycle keeps the action name from doing too much work. The extrinsic name tells readers what
was submitted, while the block or mempool context tells them where the transaction sits in chain
timing ([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

Events and errors add runtime evidence. Events are named records emitted by the runtime, while error
codes explain rejected or failed runtime paths; both are later evidence around the submitted action,
not replacements for the extrinsic surface itself
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

Storage answers a different question again. A storage value can show public chain state after
processing, but it is not the submitted action and it is not the pending transaction stage
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

Fee context also belongs beside, rather than inside, the action name. Transaction-fee documentation
describes costs for state-changing work, while the extrinsics reference identifies callable runtime
methods ([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

For readers, the practical sequence is: name the extrinsic when describing the requested action,
name mempool or block context when describing timing, and use events, errors, storage, or fees when
describing evidence produced around the action.

The same sequence helps avoid overclaiming success. A submitted extrinsic can identify intent and
method, but success language should wait for the runtime evidence that belongs to the selected
network and block context ([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. Extrinsics are
interpreted inside the selected Bittensor network environment where they were submitted
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Localnet extrinsic examples can test action wiring in isolation. Testnet examples add shared
non-production chain history. Mainnet extrinsic interpretation concerns production Subtensor records
on the active network.

## Permission and Nonce Checks Precede State Change

Submitted extrinsics pass permission, nonce, and fee gates before the runtime applies a state
change. A listed method name therefore does not guarantee execution even when parameters are well
formed
([Subtensor Pallets and Extrinsics](https://docs.learnbittensor.org/navigating-subtensor/subtensor-extrinsics),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

For readers, extrinsic vocabulary names the requested action, while error-code and event vocabulary
name why a submission failed or what changed after inclusion
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

## Distinction from Chain Reads

Extrinsics and chain reads sit on opposite sides of Subtensor interaction. An extrinsic asks the
runtime to process a state-changing action, while a chain read observes public state without
requesting that change
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

- **Extrinsics** — submitted runtime actions that can change chain state.
- **Chain reads** — observation of public state without a state-changing request.

## Distinction from Batch Transactions

Extrinsics and batch transactions differ in execution shape. A single extrinsic names one submitted
runtime action, while a batch wrapper groups several inner extrinsics inside one outer transaction
without renaming the underlying action families
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions)).

- **Extrinsic** — one submitted runtime action at the action surface.
- **Batch transaction** — one outer submission that wraps multiple inner extrinsics.

## Distinction from Public Key

Extrinsics and the public key are related but different parts of Bittensor chain action vocabulary.
Extrinsics name signed transactions and calls submitted to Subtensor, while the public key is the
shareable key material used to identify accounts and verify the signatures on those submissions
([Subtensor Extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Glossary: Public Key](https://docs.learnbittensor.org/resources/glossary#public-key)).

- **Extrinsic** — the submitted action sent to Subtensor.
- **Public key** — the shareable account reference that signs and identifies that submission.

## Distinction from Private Key

Extrinsics and the private key are related but different parts of Bittensor chain action vocabulary.
Extrinsics name signed transactions and calls submitted to Subtensor, while the private key is the
secret key material used to produce the signature that authorizes those submissions
([Subtensor Extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Glossary: Private Key](https://docs.learnbittensor.org/resources/glossary#private-key)).

- **Extrinsic** — the submitted action sent to Subtensor.
- **Private key** — the secret that signs an extrinsic to authorize it.

## Distinction from Crowdloans

A crowdloan and an extrinsic relate as a funding campaign and the action it can execute. Crowdloan
documentation describes a campaign that, on finalization, runs exactly one specified action — a call
(an extrinsic) or a target-address transfer — chosen when the campaign is created
([Crowdloans](https://docs.learnbittensor.org/subnets/crowdloans),
[Crowdloan Lifecycle](https://docs.learnbittensor.org/subnets/crowdloans#crowdloan-lifecycle),
[Subtensor Extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

- **Crowdloan** — the pooled-funding campaign that triggers one finalization action.
- **Extrinsic** — the on-chain call a campaign can execute when it finalizes.

## Distinction from Runtime Action

The Subtensor extrinsics reference lists runtime methods exposed as chain actions
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

A clear extrinsic example should name the requested runtime method before describing whether the
request changed chain state, produced events, or failed validation
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)).

A method name identifies the action family. Fees, selected network, and surrounding chain context
help explain how that action should be read in a particular example
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

These calls begin outside the blockchain runtime, which is why Bittensor documentation names them
extrinsics rather than internal runtime steps: the runtime checks permissions and then decides
whether to accept the submitted action
([Subtensor Pallets and Extrinsics](https://docs.learnbittensor.org/navigating-subtensor/subtensor-extrinsics),
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor)).

## Distinction from Block

Extrinsics are adjacent to block and mempool vocabulary. The mempool is the pending-transaction area
before block inclusion, while a block is the ordered chain unit that can contain accepted
transactions ([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

This boundary keeps timing clear. An extrinsic can name the action being processed, while mempool
and block vocabulary describe pending and included chain contexts.

## Distinction from Yuma Consensus

An extrinsic is a signed transaction payload a participant submits to change Bittensor chain state.
It names the submission unit for on-chain actions, not the on-chain step that turns validator weight
submissions into emission shares
([Glossary: Extrinsic](https://docs.learnbittensor.org/resources/glossary#extrinsic)).

Yuma Consensus is the on-chain process that converts submitted validator weights into emission
shares after evaluation completes. Weight-setting extrinsics feed consensus inputs, but Yuma
Consensus names the settlement mechanism rather than the extrinsic format itself
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers should keep submission vocabulary separate from the settlement step. Extrinsics answer what
action is being submitted; Yuma Consensus answers how validator weights become emission shares in a
subnet ([Glossary: Extrinsic](https://docs.learnbittensor.org/resources/glossary#extrinsic),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Extrinsic** — signed transaction payload that changes chain state.
- **Yuma Consensus** — on-chain weight aggregation that settles emissions.

## Reader Boundary

Extrinsics do not name an operator guide, interface tutorial, chain-read surface, or proof that an
action succeeded ([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

Extrinsics identify runtime action vocabulary, while chain reads, storage, events, errors, fees,
mempool, blocks, and network context describe neighboring parts of the system.

## The Extrinsic Catalog Follows Runtime Spec

The [Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) reference is
generated from the Subtensor runtime spec rather than maintained as a standalone prose list. Its
grouped entries describe the dispatchable action types the connected runtime exposes at generation
time.

Runtime upgrades can change that exposed surface by adding, renaming, or retiring action types.
Inspection guidance notes that runtime metadata must be refreshed after a Bittensor runtime upgrade
so connected tools still describe the live chain accurately
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

That generation link separates catalog vocabulary from block history. A listed action type names
what the runtime can accept, while an extrinsic observed in a particular
[block](https://docs.learnbittensor.org/resources/glossary#block) is evidence from that moment in
chain history on the connected network.

References: [Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)

## Parameters Complete the Submission

The
[Subtensor Pallets and Extrinsics](https://docs.learnbittensor.org/navigating-subtensor/subtensor-extrinsics)
overview lists both a call method and its arguments for each dispatchable action. Official
documentation also directs readers to the generated
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) reference for the
complete parameter specification on the connected runtime.

A method name therefore identifies an action family, but the filled-in arguments select the intended
target and amount within that family. Staking, registration, and subnet-scoped actions often need a
[netuid](https://docs.learnbittensor.org/resources/glossary#netuid) or account identifier attached
before the submission fully describes what was requested.

Without those arguments attached, two submissions can share a method label while requesting
different subnet markets or account targets. Parameter fields therefore belong in the same reading
step as the action name when precision matters.

References:
[Subtensor Pallets and Extrinsics](https://docs.learnbittensor.org/navigating-subtensor/subtensor-extrinsics),
[Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid)

## Distinction from Mempool

Extrinsics and the mempool relate as the items and the place that holds them. An extrinsic is a
signed instruction submitted to the chain to change state, while the mempool is the staging area
that holds submitted extrinsics until a validator includes them in a block
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

- **Extrinsic** — a signed instruction submitted to change chain state.
- **Mempool** — the staging area holding pending extrinsics before inclusion.
