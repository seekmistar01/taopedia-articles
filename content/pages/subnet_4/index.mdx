# Subnet 4: Targon

Targon is Bittensor Subnet 4. [TaoStats](https://taostats.io/subnets/4) lists the live on-chain
identity name as Targon and describes it as an "Incentivized Compute Marketplace powered by the
Targon Virtual Machine (TVM)." The subnet's official
[manifold-inc/targon](https://github.com/manifold-inc/targon) repository presents Targon as a
confidential decentralized AI cloud and a secure AI infrastructure platform built around
Confidential Compute and Protected PCIe technology.

## What Targon Provides

According to the [official repository](https://github.com/manifold-inc/targon), Targon focuses on
secure AI infrastructure for decentralized compute. The project README says the platform uses
Confidential Compute and Protected PCIe technology to provide a secure execution environment from
hardware to application layers.

The same source says Targon is designed for verifiable and trustworthy operation across the stack,
including hardware-enforced protection, secure boot, remote attestation, and isolated execution
environments for AI workloads.

## Confidential Compute Context

The [Targon repository](https://github.com/manifold-inc/targon) frames the subnet around
confidential compute and Protected PCIe rather than around ordinary GPU capacity alone. That framing
matches the [Targon website](https://targon.com), which presents Targon as a confidential compute
cloud for secure GPU and CPU rentals used in AI training and deployment.

For a reader, this changes how the compute-marketplace description should be interpreted. Targon's
public materials emphasize the security properties of the execution environment, not just the amount
of hardware available to run an AI workload.

Confidential compute is relevant because outsourced inference normally requires trust in a remote
machine operator. Targon's public description instead points to hardware-enforced memory protection,
secure boot, and isolated execution as parts of the stack that are meant to reduce that trust
assumption. The useful service is therefore secure AI execution capacity: compute that can be
associated with a protected environment.

The same source describes the platform as securing the path from hardware to application layers.
That is why the article treats Targon as infrastructure for protected AI workloads rather than as a
general-purpose model leaderboard or prompt marketplace.

## Attestation Context

Remote attestation is one of the recurring terms in Targon's public materials. At the article level,
the important point is that attestation gives participants a way to reason about where and how a
workload ran before trusting the result. In a decentralized compute setting, that is separate from
simply receiving an inference response: the worker's execution environment is part of the value
being evaluated.

The repository groups hardware-level protection, key management, and isolated execution alongside AI
infrastructure capabilities. Those details describe the trust boundary around the workload: the
model or application is valuable, but so is evidence that it ran inside the expected protected
environment.

Targon's README also connects remote attestation to protected model execution and verifiable
computation. In this article, that connection distinguishes Targon from compute subnets that focus
mainly on raw capacity, uptime, or benchmark throughput.

## Miner and Validator Roles

Targon follows the standard Bittensor subnet structure. Miners supply compute capability to the
network, and validators evaluate miner performance and set weights. Those validator assessments feed
into [[Yuma Consensus]], which determines how subnet emissions are distributed.
