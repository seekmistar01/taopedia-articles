# MEV Shield

MEV Shield is Bittensor's encrypted mempool-protection feature. It hides sensitive transaction
details during the pending window before block inclusion.

The official [MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield/) page describes this
as encrypted mempool protection. The
[MEV glossary entry](https://docs.learnbittensor.org/resources/glossary#mev-maximal-extractable-value)
defines the risk area behind the feature.

## Mempool Visibility Context

The [Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) entry places pending
transactions before block inclusion. MEV Shield belongs to that pre-inclusion window because it
limits what sensitive details are visible while a protected transaction is waiting.

This makes MEV Shield different from post-inclusion privacy. It is about hiding sensitive contents
before the chain can process them, not about erasing records after activity is included.

## MEV Risk Context

The
[MEV glossary entry](https://docs.learnbittensor.org/resources/glossary#mev-maximal-extractable-value)
connects MEV risk to pending transaction visibility. If sensitive details are exposed before
inclusion, another observer may be able to use that timing and ordering information.

MEV Shield is one official Bittensor response to that exposure. It should be read as a protection
for the pending visibility problem, not as a general guarantee that every transaction outcome will
match a user's preference.

## Shielded Protection Boundary

The [MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield/) documentation explains that
sensitive transaction contents are encrypted while pending and revealed only when the chain can
process them. The protection is therefore about information exposure during the waiting period.

That boundary matters because shielding and execution are separate questions. MEV Shield changes
what can be seen before inclusion; it does not replace the runtime rules, fee rules, or price
movement controls that apply to the submitted action.

## Price Protection Boundary

MEV Shield and price protection address related but different risks. MEV Shield hides sensitive
pre-inclusion details, while
[Price Protection](https://docs.learnbittensor.org/learn/price-protection) defines guardrails for
adverse price movement around staking-related activity.

The two protections should not be blended into one concept. A source-backed claim should say whether
it concerns pending visibility, adverse price movement, fees, or inclusion evidence.

## Network Context

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. MEV Shield examples should therefore keep network context
visible because pending-visibility evidence belongs to the environment being observed.

Local examples can explain the protection model. Testnet examples can show shared testing context.
Mainnet claims need source support appropriate to production network behavior.

## Evidence Boundary

The [MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield/) page supports claims about
encrypted mempool protection. It does not support fee quotes, price guarantees, wallet advice,
private account diagnosis, or claims that every other transaction risk has been removed.

When a claim depends on a concrete protected transaction, the article should identify the network,
pending-visibility context, and any separate protection source being used. MEV Shield is a
visibility-protection concept, not a replacement for every other chain-action reference.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For MEV Shield,
that sequence determines which network's pending transaction window the protection applies to.

In localnet, MEV Shield behavior can be tested in an isolated environment. Localnet encrypted
mempool protection reflects local chain settings and does not represent production transaction
exposure.

On testnet, MEV Shield can be observed in the shared, non-production network's pending window.
Testnet protection behavior is separate from mainnet transaction exposure.

On mainnet, MEV Shield is the live encrypted mempool protection that hides sensitive Bittensor
transaction details before production block inclusion. The
[MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield/) documentation describes the
protection model that applies on the production network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. An MEV Shield example from one environment should not be
read as representing protection behavior or pending visibility in another environment.
