# Alpha Sigmoid Steepness

Alpha sigmoid steepness is a per-subnet hyperparameter, listed in the documentation as
`AlphaSigmoidSteepness`, that shapes how alpha values are assigned based on validator voting
alignment in [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus). The
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) reference
lists it as the control that determines how alpha values are assigned according to how a validator's
weights align with consensus.

## What It Controls

The hyperparameter sets the steepness of the sigmoid curve that maps a validator's consensus
alignment to the alpha used when smoothing
[validator-miner bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds).
It works together with the liquid alpha feature, which varies the bond-smoothing factor between an
`alpha_low` and `alpha_high` bound rather than holding it constant
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

A steeper curve makes the assigned alpha shift more sharply as a validator moves toward or away from
consensus, while a gentler curve spreads that response over a wider alignment range. It does not set
the alpha bounds themselves, which belong to the liquid alpha range.

## Documented Type, Default, and Setter

The [Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)
reference lists `AlphaSigmoidSteepness` as an `i16` with a default of `1000`. Because each subnet
carries its own hyperparameters, the steepness is defined per subnet and applies only within that
subnet's bond-smoothing path.

The value in force is per-subnet chain state and can differ from the documented default, so a
subnet's real steepness should be read for that subnet rather than assumed from the reference value.

## Distinction from Liquid Alpha

Liquid alpha is the feature that varies the bond-smoothing factor by consensus alignment, while
alpha sigmoid steepness sets how sharp that variation is across the alignment range
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

- **Liquid alpha** — enables a variable, consensus-aligned bond-smoothing factor.
- **Alpha sigmoid steepness** — sets how steeply that factor responds to alignment.

## Distinction from Bonds Moving Average

Alpha sigmoid steepness and bonds moving average both touch the bond path but answer different
questions. Steepness shapes the response curve that assigns alpha from consensus alignment, while
bonds moving average sets the moving-average behavior applied to validator-miner bonds over time
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Alpha sigmoid steepness** — the sharpness of the alpha-assignment curve.
- **Bonds moving average** — the smoothing applied to bonds across epochs.

## Per-Subnet Hyperparameter State

The [Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)
reference stores alpha sigmoid steepness separately for each netuid. The steepness in force on one
subnet can differ from the documented default and from the value configured on another subnet
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

Readers comparing bond-path behavior should read the hyperparameter row for the netuid they mean
rather than inferring steepness from reference defaults alone
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

## Why Steepness Belongs to the Bond Path

The steepness control matters because liquid alpha decides how fast bonds adapt to a validator's
alignment, and a sigmoid is what turns a continuous alignment signal into the assigned alpha.
Steepness sets how decisively that mapping reacts near the consensus boundary
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights),
[Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)).

For readers, this places alpha sigmoid steepness inside the validator bond-smoothing mechanics
rather than in miner-incentive ranking. It tunes how alignment is translated into bond adaptation,
not what task a subnet rewards
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Distinction from Alpha Low and Alpha High

Alpha sigmoid steepness works within the range that alpha low and alpha high define. Alpha low and
alpha high fix the lower and upper endpoints of the bond-smoothing coefficient, while alpha sigmoid
steepness sets how sharply the coefficient moves between those endpoints
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

The bounds set where the coefficient can land; steepness sets the sharpness of its movement within
that range.

- **Alpha low / alpha high** — the lower and upper endpoints of the coefficient range.
- **Alpha sigmoid steepness** — how sharply the coefficient moves within that range.

## Distinction from EMA

The alpha that the steepness curve assigns is the EMA smoothing coefficient used when updating
validator-miner bonds. EMA is the technique that blends each epoch's new bond value with the prior
one, while alpha sigmoid steepness shapes how the coefficient feeding that EMA is set from a
validator's consensus alignment under the liquid alpha feature
([EMA](https://docs.learnbittensor.org/learn/ema),
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights),
[Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)).

- **EMA** — the moving-average technique that blends new and prior bond values.
- **Alpha sigmoid steepness** — shapes the coefficient that EMA uses under liquid alpha.

## netuid 1 as a Reference Subnet

Hyperparameter examples in this article use netuid 1 as a documented Finney mainnet subnet because
that netuid appears repeatedly in official liquid-alpha and bond-smoothing prose
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

Alpha sigmoid steepness is per-subnet chain state. The steepness value configured on netuid 1 shapes
how sharply liquid alpha responds inside that subnet's bond-smoothing path, but the live value can
differ from reference defaults and from steepness configured elsewhere
([Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

Readers should keep the steepness hyperparameter separate from subnet performance claims. Alpha
sigmoid steepness names a bond-path response shape inside Yuma Consensus; netuid 1 provides a
concrete subnet context for reading that setting in official examples
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Alpha sigmoid steepness** — liquid-alpha response steepness inside bond smoothing on a subnet.
- **netuid 1** — documented example subnet used for Finney mainnet hyperparameter illustrations.

## Distinction from Yuma Consensus

Alpha sigmoid steepness is a per-subnet hyperparameter that shapes how sharply the liquid-alpha
coefficient responds to a validator's consensus alignment inside Yuma Consensus bonding. It names
the steepness of that response curve on a selected netuid, not the full epoch settlement that
converts validator weights into emission shares
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

Yuma Consensus is the tempo process that aggregates eligible validator weights, applies clipping,
updates validator-miner bonds—including liquid-alpha adjustments bounded by alpha high and alpha
low—and maps the result into miner incentives and validator dividends
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Steepness therefore names a bonding-input control, not an emission outcome. Rank, dividends, and
miner incentives describe settlement results after Yuma processes clipped weights and updated bonds;
alpha sigmoid steepness configures how alignment is translated into the coefficient that feeds those
bond updates when liquid alpha is enabled
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Dividends](https://docs.learnbittensor.org/resources/glossary#emission)).

Liquid alpha must be enabled for steepness to matter on a subnet. When it is off, the hyperparameter
does not change how bonds smooth even though Yuma still runs its full weight-to-emission settlement
path ([Glossary: Liquid Alpha](https://docs.learnbittensor.org/resources/glossary#liquid-alpha)).

When liquid alpha is active, steepness works together with alpha low and alpha high inside the same
bond path. Those bounds set the coefficient range; steepness sets how sharply the mapping moves
across that range before Yuma converts the resulting bond state into emission shares each tempo
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

Readers should keep this hyperparameter separate from alpha-token supply language. Alpha sigmoid
steepness configures how quickly bond smoothing reacts to consensus alignment; it does not set
emission rates or replace Yuma's consensus math.

- **Alpha sigmoid steepness** — sharpness of the liquid-alpha response curve inside bonding.
- **Yuma Consensus** — aggregation that turns filtered weights and bond state into emission shares.
- **Liquid alpha** — feature that varies the bond-smoothing coefficient by consensus alignment.

## Reader Boundary

This page defines alpha sigmoid steepness at a high level. It does not report the live steepness
value on any particular subnet or whether liquid alpha is enabled there. Those are per-subnet
hyperparameter settings that should be checked for the relevant netuid
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).
