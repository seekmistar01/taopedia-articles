# Weight Vector

A weight vector is one validator's set of miner evaluations inside a Bittensor subnet. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-vector) defines it as a vector
maintained by each subnet validator, with each element representing the weight assigned to a subnet
miner based on performance. [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)
provides the broader context for how validator signals become subnet consensus inputs.

## Per-Validator Signal

The important scope is one validator's view. A weight vector is not the whole subnet result; it is
the signal one validator produces after evaluating miners under the subnet's standards. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-vector) frames the vector at
the validator level, while [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)
explains that consensus depends on validator-submitted weights.

## Relationship to Miner Evaluation

Each weight in the vector corresponds to a validator's evaluation of miner work in the subnet. That
evaluation only makes sense in relation to the subnet's task and standards.
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
explains that subnet design defines how miners are evaluated, and the
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-vector) connects weight vectors
to miner performance.

## Relationship to the Weight Matrix

A weight vector and a weight matrix are related but separate concepts. The vector is one validator's
signal, while the matrix is the subnet-level collection of validator vectors used by consensus. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-vector) describes weight
vectors as inputs that combine into a matrix, and the
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-matrix) defines the matrix as
the collection used by Yuma Consensus.

## Consensus Context

Weight vectors matter because Bittensor does not treat one validator's evaluation as the final
subnet result. [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) describes a
process that aggregates validator weights, and the
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-vector) identifies weight
vectors as the validator-level inputs to that process.

## Relationship to Multiple Mechanisms

A weight vector is usually discussed for one incentive mechanism at a time, but subnets can also run
multiple mechanisms in parallel. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-vector) notes that when
multiple incentive mechanisms are used, validators maintain separate weight vectors for each
mechanism.

For readers, a weight vector still names one validator's miner evaluations, but the mechanism
context matters. A vector for one mechanism does not describe miner performance in another when a
subnet uses more than one evaluation system.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)

## Relationship to Yuma Consensus

A weight vector and Yuma Consensus address related but different parts of Bittensor subnet incentive
flow. A weight vector is one validator's miner evaluations inside a subnet, while Yuma Consensus
aggregates many validator weight signals into miner incentives and validator dividends. The
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)
describes the per-validator signal, and
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) describes how those signals
become consensus outcomes.

For readers, a weight vector names one validator's evaluation submissions, while Yuma Consensus
names the aggregation step that turns many such vectors into shared subnet results.

References:
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)

## Relationship to Validator Weights

A weight vector and validator weights address related but different parts of subnet evaluation
structure. Validator weights are the evaluation signals validators submit for miners, while a weight
vector is the structured form of one validator's relative miner evaluations inside a subnet. The
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)
describes the per-validator evaluation inputs, and the
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)
describes one validator's vector of miner weights.

For readers, validator weights name the evaluation submission concept, while a weight vector names
one validator's arranged set of those evaluations before matrix aggregation.

References:
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)

## Relationship to Consensus Score

A weight vector and a consensus score refer to related but different parts of Yuma Consensus. A
weight vector is one validator's miner evaluations, while a consensus score is the stake-weighted
median used to filter outlier submissions before final rank calculations. The
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)
describes the per-validator input, and the
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)
describes the agreement threshold applied to those inputs.

For readers, a weight vector names one validator's signal, while a consensus score names the
filtering step that helps consensus ignore out-of-line submissions.

References:
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)

## Relationship to Commit Reveal

A weight vector and Commit Reveal address related but different parts of validator-weight
processing. A weight vector names one validator's miner evaluations, while Commit Reveal can conceal
those evaluations until a later reveal time before consensus uses them publicly. The
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)
describes the per-validator signal, and the
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal) documentation describes
concealed weight submission and delayed reveal.

For readers, a weight vector names the evaluation content, while Commit Reveal names a visibility
and timing layer that can protect fresh weight information before aggregation.

References:
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)

## Relationship to Weight Copying

A weight vector and weight copying address related but different parts of subnet evaluation quality.
A weight vector is one validator's structured miner evaluations inside a subnet, while weight
copying describes when a validator relies on public weight information instead of producing an
independent evaluation. The
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)
describes one validator's vector of miner weights, and the
[The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)
documentation describes copied or derived weights as the problem behavior.

For readers, a weight vector names one validator's evaluation signal, while weight copying names
when that signal may add less independent information than it appears.

References:
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor)

## Relationship to Drand Time-Lock Encryption

A weight vector and Drand time-lock encryption address related but different parts of delayed weight
visibility. A weight vector names one validator's miner evaluations, while Drand time-lock
encryption provides the timing source that prevents concealed weights from being decrypted before
the target reveal round inside Commit Reveal. The
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)
describes the per-validator signal, and the
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal) documentation describes
Drand as the reveal timing mechanism.

For readers, a weight vector names the evaluation content, while Drand time-lock encryption names
the timing layer behind when concealed weight information becomes available.

References:
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)

## Relationship to Liquid Alpha

A weight vector and liquid alpha address related but different parts of consensus bond processing. A
weight vector is one validator's miner evaluations, while liquid alpha adjusts validator-miner bond
smoothing based on how closely those evaluations align with consensus. The
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)
describes one validator's relative miner evaluations, and the
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)
documentation describes liquid alpha as varying bond-smoothing response per validator-miner pair.

For readers, a weight vector names one validator's evaluation input, while liquid alpha names a
bond-path adjustment that responds to how that input aligns with consensus.

References:
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Consensus-based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)

## Relationship to Rank

A weight vector and rank address related but different parts of Yuma Consensus vocabulary. A weight
vector is one validator's miner evaluations, while rank names the final aggregate judgment produced
for a miner after many such vectors are filtered and aggregated. The
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)
describes one validator's vector of miner weights, and the
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank) describes the final
aggregate judgment from validator ratings.

For readers, a weight vector names one validator's evaluation input, while rank names the miner-side
consensus outcome built from many combined inputs.

References:
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)

## Reader Boundary

Weight vector should be read as a consensus input concept, not as a universal score, a standalone
quality label, or a procedure for evaluating any particular miner. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-vector) keeps the term scoped
to validator-maintained weights, and
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
keeps the meaning tied to subnet-specific evaluation.
