# Root Staker

A root staker is a TAO holder who stakes through validators registered on the Root Subnet. The
official
[Glossary: Root Subnet](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero)
describes Root Subnet, also called Subnet Zero, as a special subnet where validators can register
and TAO holders can stake to those validators in a subnet-agnostic way.

## Root Subnet Context

Root staking is different from staking directly inside one ordinary subnet. The Root Subnet glossary
entry explains that miners cannot register there and no validation work is performed in that subnet,
while validators can register and TAO holders can attach stake through that path. For readers, root
staker names the TAO-holder role on that subnet-agnostic staking route.

This keeps two ideas separate: Root Subnet names the special subnet context, while root staker names
the participant role that supplies TAO through validators on that path.

## Root Staker vs Alpha Staker

Bittensor staking prose also distinguishes root-side TAO staking from subnet-local alpha staking.
The official [Emission: Distribution](https://docs.learnbittensor.org/learn/emissions#distribution)
documentation describes emissions to stakers on the root subnet separately from emissions to stakers
holding subnet alpha in a mining subnet. Root staker therefore belongs to the TAO-on-root
vocabulary, not to subnet-alpha staking vocabulary.

For readers, the useful boundary is asset and path: root staker describes TAO staked through Root
Subnet validators, while alpha staker language describes stake held in a subnet's alpha context.

## Choosing a Root Subnet Validator

A root staker commits TAO by delegating to a validator registered on the Root Subnet. The
[Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation)
describes TAO holders staking through validators, and the
[Glossary: Root Subnet](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero)
places that path on Subnet Zero where validators register and TAO holders attach stake. The
validator choice determines where the delegated TAO sits on that subnet-agnostic route, while root
staker names the participant who supplies the delegation.

Readers should keep the role separate from any one validator identity or live ranking table. Root
staker vocabulary applies to the TAO-holder role on the Root Subnet path regardless of which
registered validator receives the stake. Commission settings, validator lists, and current
delegation commands belong in official staking documentation rather than in this concept article.

References:
[Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation),
[Glossary: Root Subnet](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero)

## Dividends From Root-Routed Stake

Root-routed stake can produce alpha dividends that the root-claim mechanism later processes. The
[Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims) describes
how accumulated dividends from Root Subnet stake can be handled through Keep or Swap modes when
claims run. Root staker names who holds the root-routed staking position; root claims and root claim
type name the dividend-handling layer documented separately in official staking material.

Readers should not read every TAO balance or subnet alpha position as a root staker outcome. This
role applies to TAO delegated through Root Subnet validators. Pending dividend totals, claim timing,
and Keep or Swap preferences belong in root-claim references and live chain state rather than in
this definition.

References:
[Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims),
[Emission: Distribution](https://docs.learnbittensor.org/learn/emissions#distribution)

## Subnet-Agnostic Path Versus Ordinary Subnet Stake

Root staking routes TAO through Subnet Zero instead of describing subnet-local alpha stake inside an
ordinary mining subnet. The
[Glossary: Root Subnet](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero)
states that TAO holders can stake to validators there in a subnet-agnostic way, while the official
[Emission: Distribution](https://docs.learnbittensor.org/learn/emissions#distribution) documentation
treats emissions to root-subnet stakers separately from emissions to stakers holding subnet alpha in
a mining subnet.

For readers comparing routes, the useful distinction is asset and staking path rather than validator
ranking. Root staker belongs to TAO-on-root vocabulary; alpha staker language belongs to subnet
alpha context. Neither term replaces coldkey, hotkey, or wallet vocabulary used elsewhere in
Bittensor documentation.

References:
[Glossary: Root Subnet](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero),
[Emission: Distribution](https://docs.learnbittensor.org/learn/emissions#distribution)

## Relationship to Root Claims

Root claims are related but not the same term. The official
[Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims) describes
how alpha dividends earned through Root Subnet stake can be handled through swap or keep modes. Root
staker names the staking role; root claims name the dividend-handling mechanism for stake placed on
Root Subnet validators.

This article does not describe claim settings, dividend outputs, or staking steps. Those details
belong in official root-claim and staking references.

## Relationship to Root Proportion

Root staker and root proportion are related but describe different levels of the same staking
context. Root staker names the participant role — a TAO holder who stakes through validators on Root
Subnet. Root proportion is a subnet-level measure that quantifies how much of that subnet's dividend
context belongs to the root-staking path. The
[Glossary: Root Proportion](https://docs.learnbittensor.org/resources/glossary#root-proportion)
defines it as the relative weight of TAO staked to validators through the root subnet compared with
the subnet's alpha context.

For readers, root staker names who is staking through Root Subnet; root proportion names how much of
a given subnet's dividends that root-staking path is allocated. One names the participant role, the
other names the subnet-level dividend measure that reflects the aggregate of that role.

References:
[Glossary: Root Proportion](https://docs.learnbittensor.org/resources/glossary#root-proportion),
[Glossary: Root Subnet](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero)

## Relationship to Root Claim Type

Root staker and root claim type are related but describe different layers of the root-staking flow.
Root staker names the participant role — a TAO holder who stakes through validators on the Root
Subnet. Root claim type is the per-staker dividend-handling preference the
[Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims) names:
Keep, where accumulated alpha dividends remain as the alpha of the originating subnet, or Swap,
where dividends are converted to TAO and restaked on Root. The same source notes this preference
applies to every root claim, automatic or manual.

For readers, root staker names who holds the root-routed stake, while root claim type names how that
staker's alpha dividends are handled once a claim is processed. A root staker chooses a claim type;
the claim type then determines whether the next dividend flows as alpha or as TAO restaked on Root.

References:
[Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims),
[Glossary: Root Subnet](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero)

## Relationship to Autostaking

Root staker and autostaking address related but different parts of Bittensor staking vocabulary. A
root staker is a TAO holder who stakes TAO through the Root Subnet path — a deliberate decision to
commit TAO to a validator on Subnet Zero and receive alpha dividends that can later be claimed as
TAO or restaked. Autostaking is the feature that lets a subnet miner automatically route emission
proceeds to a chosen validator staking position as they arrive, without the miner manually
submitting a staking action each time. The
[Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation)
describes how TAO holders stake through validators, and the
[Autostaking](https://docs.learnbittensor.org/miners/autostaking) documentation describes how miners
can route their emission proceeds to a staking position automatically.

Both describe ways that TAO or emission proceeds end up staked to a validator, but the source and
actor differ. A root staker holds TAO and chooses to commit it on Root Subnet. Autostaking routes a
miner's own emission proceeds to a staking position automatically as each emission round settles.
Root staker is a TAO holder's deliberate staking role; autostaking is a miner's automated emission
routing feature.

References:
[Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation),
[Autostaking](https://docs.learnbittensor.org/miners/autostaking)

## Relationship to Child Hotkeys

A root staker and a child hotkey relate to validation from different sides. A root staker is a TAO
holder who stakes through validators registered on the Root Subnet, as the
[Glossary: Root Subnet](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero)
describes. A child hotkey, by the
[Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys) documentation, is a
separate key that a parent hotkey authorizes to validate on its behalf by re-delegating a portion of
the parent's stake to it, without transferring ownership of that stake.

For readers, a root staker names a participant who supplies TAO stake through Root Subnet
validators, while a child hotkey names a key authorized to carry out validation for a parent. The
two are not interchangeable: a root staker delegates economic TAO stake through the Root Subnet,
whereas a child hotkey is a delegated validation key that acts on a parent's re-delegated
stake-weight without owning the stake. A root staker supplies stake; a child hotkey performs
validation on a parent's behalf.

References:
[Glossary: Root Subnet](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero),
[Child Hotkeys](https://docs.learnbittensor.org/validators/child-hotkeys)

## Reader Boundary

Root staker should not be read as a live balance report, a validator recommendation, or a synonym
for nominator in every context. It is a role term for TAO holders who use the Root Subnet staking
path. Exact balances, validator choices, and current subnet state belong in official staking
documentation and live chain data.

## Relationship to Coinbase

A root staker and coinbase are related but different parts of Bittensor's staking and emission flow.
A root staker is a TAO holder who stakes through validators on the Root Subnet and earns alpha
dividends from that position. The
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
documentation describes coinbase as the per-block protocol mechanism that drives TAO emission,
accumulates pending emissions, and triggers Yuma Consensus rounds at epoch boundaries, while the
[Glossary: Root Subnet](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero)
describes the root-staking path.

The connection is that coinbase is part of the upstream emission process that the dividends a root
staker earns ultimately originate from, while the root staker is a participant role on the staking
side. Coinbase advances emissions and fires the epoch calculations each block; the alpha dividends
that reach a root staker's position then arise from that cycle and are handled afterward through
root claims. A root staker names who supplies TAO on the root path; coinbase names the per-block
emission engine whose output those dividends come from — not the same kind of thing.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For a root staker,
that sequence changes how readers should interpret evidence about Root Subnet staking positions and
dividend outcomes.

In localnet, root staker mechanics can be tested in an isolated environment. Localnet Root Subnet
stake positions and dividend outcomes reflect local chain configuration and do not represent
production Root Subnet state.

On testnet, root staker activity can be observed in a shared, non-production network. Testnet Root
Subnet staking outcomes are separate from mainnet production delegation state.

On mainnet, a root staker holds a live TAO staking position on a Root Subnet validator and
participates in production alpha dividend distribution. The
[Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation)
describes the delegation mechanics that apply on the production network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A root staker example from one environment should not be
read as representing stake amounts or dividend outcomes in another environment.
