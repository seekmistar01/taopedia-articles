# Subnet Scoring Model

A subnet scoring model is the evaluation method validators use when judging miner responses inside a
Bittensor subnet
([Glossary: Subnet Scoring Model](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

The term belongs to incentive-mechanism vocabulary. It names how responses become evaluation scores,
not the work target, interaction protocol, or final consensus result.

## Evaluation Role

The scoring model explains how returned work is judged. The subnet task describes what miners are
asked to produce, while the scoring model describes how validators evaluate the responses they
receive
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Subnet Scoring Model](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model)).

This keeps task vocabulary separate from scoring vocabulary. The task asks for work; the scoring
model evaluates returned work.

## Protocol Boundary

Subnet protocol and subnet scoring model are adjacent but different. The protocol describes the
request-and-response pattern between validators and miners, while the scoring model describes how
returned responses are judged
([Glossary: Subnet Protocol](https://docs.learnbittensor.org/resources/glossary#subnet-protocol),
[Glossary: Subnet Scoring Model](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model)).

For readers, the protocol creates the interaction surface. The scoring model explains the evaluation
method applied to that surface.

## Scores Before Weights

A scoring model sits earlier in the evaluation path than final weight signals. Weight vectors are
structured signals formed after validator evaluation and sent onward for consensus context
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Subnet Scoring Model](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model)).

This distinction keeps scoring formulas and weight vocabulary separate. The scoring model produces
evaluation scores; weight vocabulary describes the signals that later carry evaluation into
consensus.

## Role Context

Miners produce responses for the subnet, and validators evaluate those responses under subnet
standards. Subnet documentation describes miners as producing a digital commodity and validators as
measuring miner work
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Glossary: Subnet Scoring Model](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model)).

The scoring model is the method validators apply to that work. It links miner responses to validator
evaluation without replacing either role.

## Multiple-Mechanism Boundary

When a subnet uses multiple incentive mechanisms, each mechanism can have its own scoring context.
The multiple-mechanism documentation describes separate mechanism contexts inside one subnet
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Subnet Scoring Model](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model)).

For readers, a scoring standard for one mechanism should not be treated as the evaluation method for
every mechanism in the subnet.

## Consensus Boundary

Yuma Consensus is downstream of validator evaluation. A scoring model can shape the evidence a
validator uses, while consensus aggregates validator signals into incentive and dividend outcomes
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

This boundary keeps evaluation methods separate from final consensus outputs. Scoring is part of how
work is judged; consensus is the protocol process that aggregates validator signals.

## Reader Boundary

Subnet scoring model should be read as concept vocabulary for evaluation inside a subnet incentive
mechanism. It is not a universal benchmark, a deployment claim, or a full implementation description
for any specific subnet
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

The stable article-level point is that a subnet scoring model converts miner responses into
evaluation scores that validators can use inside a subnet incentive mechanism.
