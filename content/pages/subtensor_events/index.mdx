# Subtensor Events

Subtensor events are named records emitted by Bittensor's Subtensor runtime, listed in the
[Subtensor Events reference](https://docs.learnbittensor.org/subtensor-api/events).

## What Events Represent

In the [Subtensor Events reference](https://docs.learnbittensor.org/subtensor-api/events), each
event is presented under a runtime area with an event name and, where available, a short summary.
That makes events a vocabulary for runtime records rather than a full explanation of every rule that
led to those records.

An event entry identifies that a named record can be emitted by the runtime. It does not, by itself,
explain the whole mechanism behind the activity. A feature article may need the event name, the
surrounding runtime area, and the mechanism documentation that explains why the record matters.

## Runtime Areas

The [Subtensor Events reference](https://docs.learnbittensor.org/subtensor-api/events) groups event
names by runtime area. Examples include balances, commitments, contracts, EVM, registry, safe mode,
system, utility, swap, and the Subtensor module.

The runtime area is part of the event's meaning. The same general kind of record can carry different
context when it belongs to balances, system, swap, or the Subtensor module. Reading the area and the
event name together is more precise than reading the event name alone.

## Bittensor-Specific Context

The Subtensor module section of the
[events reference](https://docs.learnbittensor.org/subtensor-api/events) contains Bittensor-specific
event families around subnet settings, staking activity, weights, root claims, and related runtime
behavior. Other runtime areas cover broader chain behavior such as balances, contracts, system, and
utility.

This split helps separate Bittensor-specific activity from general runtime activity. Events under
the Subtensor module are closer to subnet and incentive-system vocabulary, while events under
balances or system belong to broader accounting and execution context.

## Reading Event Definitions

A definition in the [events reference](https://docs.learnbittensor.org/subtensor-api/events) should
be read as a compact bundle of context. The runtime area identifies ownership, the event name
identifies the record inside that area, and the summary gives the short human-readable description.

That structure keeps event names from being treated as standalone explanations. A named event is a
record emitted by the runtime, not a full narrative of intent, cause, or downstream effect. The
owning area and summary give the event its reference meaning.

The events reference is therefore strongest as a naming source. If a page needs the official name
for a runtime record, the [Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)
reference can identify the owning runtime area and event name. If the page needs to explain why the
record is emitted, it also needs the related mechanism, storage, or extrinsic reference.

That distinction is important for Bittensor-specific events. A staking, weight, root-claim, or
subnet-setting event can sit near several concepts at once. The event name identifies the runtime
record; the concept article explains what the activity means for readers.

## Event Definitions and Emitted Records

The events reference describes event vocabulary. It is not the same thing as an event history for
every account, subnet, or block. A definition can show that an event type exists and how the runtime
names it, while an emitted record belongs to the processing context that produced it.

This distinction keeps vocabulary separate from observation. The event definition gives the map. A
block explorer or chain-inspection view can show a specific emitted record. The
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
guide provides that inspection context, while the events reference names the event types.

An emitted record also needs block and network context. The same event type can appear in different
environments, but a specific event belongs to the chain history that produced it. That means an
event name can be reusable vocabulary while the emitted instance remains tied to the selected
network, block, and surrounding runtime activity
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

For Taopedia, this keeps event evidence precise. Events can support claims about named runtime
records, but they do not automatically identify every account decision, off-chain process, or
mechanism detail behind that record. The emitted event is evidence of runtime activity; the
interpretation still depends on the surrounding source context.

## Block and Network Scope

Events are interpreted inside chain history. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, while
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
includes events among the records readers can inspect.

The same event name can appear across network environments, but emitted records belong to the
history that produced them. A concept article should therefore treat event vocabulary as reusable
while keeping observed event records tied to their runtime and network context.

## Events as Evidence

Subtensor events can support claims about chain-visible activity, but they do not explain every
cause behind that activity. An event record can show that the runtime emitted a named notification
during processing. It should be paired with surrounding storage, block, and mechanism context when a
claim depends on more than the event name.

This keeps event evidence from being overextended. Events are strongest as evidence that the runtime
recorded a named activity. They are weaker as a standalone explanation of off-chain intent, wallet
custody practice, project design, or every rule involved in the underlying Bittensor mechanism. The
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) reference provides a
separate view of where state is exposed.

Storage, events, and extrinsics therefore work best as complementary references. Storage describes
where mutable state is exposed, events describe records emitted during processing, and extrinsics
describe callable runtime actions
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

A single feature can involve all three surfaces. A call can request a change, storage can expose the
resulting state, and an event can record that processing occurred. Keeping those surfaces separate
helps avoid treating an event name as the entire lifecycle of the feature. It also keeps event
references useful for what they do best: naming the runtime record that appeared during processing.

That is why event references are clearest when they identify the emitted record and then point to
storage, extrinsic, or mechanism context for the rest of the explanation.
