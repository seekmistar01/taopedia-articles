# Neuron

A neuron is a registered participant in a Bittensor subnet
([Glossary: Neuron](https://docs.learnbittensor.org/resources/glossary#neuron),
[Understanding Neurons](https://docs.learnbittensor.org/learn/neurons)). The term appears when
describing miners, validators, and metagraph state for subnet participants.

## Subnet Participant

The term neuron is used inside subnet context. A subnet can contain miner and validator
participants, and those participants are represented in subnet state rather than as generic network
accounts ([Understanding Neurons](https://docs.learnbittensor.org/learn/neurons)).

A [subnet miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner) names the
task-performing role inside a subnet, while a
[subnet validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator) names the
evaluation role. A neuron is the participant record term; miner and validator are role terms for
what that participant may do inside the subnet.

## Metagraph Context

The subnet metagraph documentation describes the metagraph as a snapshot containing information
about neurons in a selected subnet
([The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph),
[Glossary: Metagraph](https://docs.learnbittensor.org/resources/glossary#metagraph)). Each
registered neuron appears as state inside that metagraph view rather than as a standalone wallet
label.

Reading neuron count or participant fields therefore requires both the selected
[netuid](https://docs.learnbittensor.org/resources/glossary#netuid) and the block height of the
snapshot. A neuron row describes chain-recorded subnet participation at one point in history.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For a neuron on a
subnet such as netuid 1, that sequence changes how readers should interpret participant evidence.

In localnet, a neuron record can show that registration, UID assignment, and metagraph fields are
represented inside an isolated subnet environment. That is useful for checking participant lifecycle
shape, but it does not show live public subnet membership.

On testnet, neuron state can be observed in a shared non-production network. Testnet participant
records are separate from mainnet subnet state even when the same netuid number appears in both
environments.

On mainnet, a neuron belongs to live Bittensor subnet state. Production readings should keep the
selected netuid, block height, and network environment attached
([The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A neuron observed in one environment should not be read as
identical evidence about participant state in another.

## UID and Hotkey-Coldkey Pair

Each neuron in a subnet is identified by a unique UID and linked to a hotkey-coldkey pair for
authentication and operations
([Understanding Neurons](https://docs.learnbittensor.org/learn/neurons),
[Glossary: Hotkey-Coldkey Pair](https://docs.learnbittensor.org/resources/glossary#hotkey-coldkey-pair)).

A [UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot) describes a participant
position inside a subnet; a neuron names the registered participant associated with that position in
subnet state. Neuron is therefore a subnet participant record, not a wallet file label on its own.

## Axon and Dendrite Roles

Neuron architecture documentation places miners and validators on opposite sides of the same
communication model. Miners deploy axon servers to receive validator requests, while validators use
dendrite clients to transmit data to those axon endpoints. Synapse objects carry the structured
information exchanged through that path
([Understanding Neurons](https://docs.learnbittensor.org/learn/neurons),
[Glossary: Axon](https://docs.learnbittensor.org/resources/glossary#axon),
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite),
[Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse)).

A neuron can be a miner or a validator in subnet terms, while axon, dendrite, and synapse name how
subnet messages move between them.

## Register Assigns the UID Before Metrics Mature

[Understanding Neurons](https://docs.learnbittensor.org/learn/neurons) lists registration and UID
assignment as the first lifecycle steps on a subnet such as netuid 1, followed by an
[immunity period](https://docs.learnbittensor.org/resources/glossary#immunity-period) before
ordinary displacement rules apply in the usual way. Performance fields such as
[rank](https://docs.learnbittensor.org/resources/glossary#rank),
[trust](https://docs.learnbittensor.org/resources/glossary#trust), and
[incentive](https://docs.learnbittensor.org/resources/glossary#incentives) build after that entry
window rather than at the instant of registration.

Neuron vocabulary therefore names a participant record that exists once
[register](https://docs.learnbittensor.org/resources/glossary#register) succeeds, even before later
evaluation metrics reflect sustained work. Readers should not treat a fresh UID as immediately
exposed to every deregistration rule.

## Immunity After Registration

The neuron lifecycle documentation lists registration and UID assignment as the first steps,
followed by an immunity period during which a newly registered neuron is protected from ordinary
pruning. Performance metrics such as rank, trust, consensus, and incentive build after that grace
window rather than at the instant of registration
([Understanding Neurons](https://docs.learnbittensor.org/learn/neurons),
[Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period)).

Immunity gives a newly registered neuron time to begin producing and being evaluated before
low-performance replacement rules apply in the usual way.

## Multiple Mechanisms Keep Separate Evaluation Tracks

Subnets can run more than one incentive mechanism in parallel
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)).
Validators must evaluate miners separately for each mechanism.

A neuron still names one registered subnet participant, but the mechanism context matters.
Evaluation and performance readings for one mechanism should not be assumed to describe the same
neuron in another when a subnet uses more than one evaluation system.

## Incentive Mechanism and Subnet Task Context

An [incentive mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism)
covers the task, protocol, and evaluation method that define how a subnet connects work to rewards.
A [subnet task](https://docs.learnbittensor.org/resources/glossary#subnet-task) names the work
miners are expected to perform inside that subnet's incentive design.

A neuron names who is registered in a subnet; an incentive mechanism names the evaluation rules that
govern how that participant's role is judged.

## Validator Weights and Yuma Consensus

[Validator weights](https://docs.learnbittensor.org/resources/glossary#validator-weights) are the
evaluation signals that validator-role participants submit for miners after judging their work.
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) aggregates those weight
signals into miner incentives and validator dividends, applying clipping, bonding, and emission
calculation at epoch boundaries.

A neuron names a participant record inside a subnet; validator weights name the evaluation
submissions produced by validator-role participants; Yuma Consensus names the aggregation process
that uses those submissions to update incentive fields visible in metagraph state.

## Participant Removal Versus Subnet Removal

Participant-level deregistration can vacate one neuron slot while the subnet continues.
[Subnet deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration) removes an
entire subnet context and ends every neuron record within it.

A neuron is one registered participant inside a subnet; subnet deregistration removes the subnet
context that held all of its neuron records.

## Child Hotkeys Delegate Validation From a Parent Slot

A [child hotkey](https://docs.learnbittensor.org/validators/child-hotkeys) is a separate key that a
parent hotkey authorizes for validation work. The neuron is the participant record; the child hotkey
is delegated validation authority, not the participant record itself.

## Coinbase Drives Emissions for Registered Neurons

[Coinbase](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase) is the per-block
emission operation that can trigger epoch processing and reward distribution. A neuron is the
registered subnet participant who can receive miner incentives or validator dividends after
evaluation; coinbase advances chain-level emissions and triggers the epoch calculations on a
recurring basis.

Registration places a neuron in subnet state; coinbase and Yuma Consensus determine what that
participant earns based on stake, weights, and related eligibility rather than on registration
alone.

## Reader Boundary

This page defines the term at a high level. It does not report current subnet membership, live
neuron counts, wallet ownership, or current rewards. Those values change with chain state and should
be checked from current Bittensor sources when needed.

## Relationship to UID Slot

Neuron is the registered participant record inside a subnet. UID slot is the position label that
record occupies on a selected netuid
([Glossary: Neuron](https://docs.learnbittensor.org/resources/glossary#neuron),
[Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Understanding Neurons](https://docs.learnbittensor.org/learn/neurons)).

Metagraph and registration documentation use both terms together. Neuron names the participant row
in subnet state, while UID slot names the indexed position that row holds after registration
succeeds ([The Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph),
[Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register)).

Use neuron when discussing registered participant records; use UID slot when naming which position
index on the netuid is in scope
([Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner),
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator)).

Participant record and position index answer different reader questions in the same subnet context
([Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid)).
