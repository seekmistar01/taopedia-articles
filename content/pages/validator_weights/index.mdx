# Validator Weights

Validator weights are evaluation signals set by subnet validators. They express how validators
assess miner work inside a subnet before Yuma Consensus filters and aggregates those signals into
subnet outcomes
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The term is subnet-specific. A validator weight only has meaning inside the task, scoring method,
and incentive setting that produced it, so it is different from a general validator reputation
score.

## Evaluation Signal

A validator weight is a subnet-specific evaluation of miner work under the rules of the subnet's
incentive mechanism
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

Similar-looking values can carry different meaning across subnet tasks because the subnet's work
target and scoring method give the signal its interpretation.

Validator weights describe miner-work assessments inside the subnet. Stake position, permit status,
and validator standing are separate concepts with their own sources and consensus roles.

## Consensus Input Role

Validators use weight signals to express relative evaluations of miners. Yuma Consensus combines
those signals into a broader consensus result rather than treating one validator's view as final on
its own ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

This aggregation step is why validator weights matter to rewards. They are evaluation inputs that
help shape how incentives and dividends are distributed inside a subnet, while the consensus and
emission paths describe the later allocation behavior.

Validator weights enter the consensus process as inputs. Incentives, dividends, bond values, rank,
and trust describe later or neighboring results.

## Vector and Matrix Structure

Validator weights are often read through weight-vector vocabulary. A weight vector is one
validator's structured set of miner evaluations, while validator weights name the evaluation values
inside that consensus input
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Consensus-Based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

Weight matrix sits one level wider. It is the subnet-level collection of validator weight vectors
used as structured input for Yuma Consensus
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Together, the three terms describe nested parts of the input side: validator weights are the values,
a weight vector groups one validator's values, and the weight matrix groups the subnet's validator
vectors.

This nesting keeps submitted values separate from subnet-wide aggregation. One validator can submit
a weight vector, while Yuma Consensus evaluates the wider matrix of validator signals.

## Consensus Score and Clipping

Validator weights and consensus score are related but distinct. Validator weights are evaluation
inputs, while consensus score is the stake-weighted benchmark used to filter outlier weights before
final rank results
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

This separates the submitted signal from the filtering benchmark. The original evaluation signal
enters consensus, while later terms describe how the network filters, aggregates, and interprets the
set of validator signals.

Rank and trust belong after that filtering path. Rank is a miner-side result after validator weights
have been filtered and aggregated, and trust describes how miner support changes through clipping
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

That sequence matters because a submitted weight is pre-filtering evidence. After clipping and
aggregation, rank, trust, incentive, and dividend language describes processed outputs rather than
the original validator submission.

## Stake Influence

Validator weights are interpreted inside a stake-weighted consensus process. Stake weight explains
how much influence a validator's signal carries, while validator weights express what that validator
evaluated about miner work
([Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

This keeps influence and evaluation separate. Stake weight affects how strongly a signal counts;
validator weights supply the miner-evaluation signal itself.

Validator trust is also distinct. It describes validator-side influence after clipping, while
validator weights are the evaluation signals that enter the process
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

## Quality and Timing

Validator weights are most useful when they reflect independent evaluation. Weight copying weakens
that signal by relying on visible weight information instead of fresh miner-work assessment
([The Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

Commit Reveal changes the timing around weight visibility. That timing can reduce the value of
copying newly visible signals without changing what validator weights are.

Validator-miner bonds are another neighboring concept. Bonds carry smoothed validator-miner
relationship history, while validator weights are evaluation signals that feed the consensus path
([Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights)).

The distinction is temporal as well as conceptual. Validator weights describe current evaluation
signals; bonds describe smoothed relationship history shaped by earlier signals and later smoothing
logic.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Validator
Weights, that sequence changes how an example should be read, because the surrounding network state
differs at each stage.

In localnet, validator Weights can be exercised in an isolated development environment, where the
surrounding chain state reflects local configuration rather than production history.

On testnet, validator Weights can be observed in a shared, non-production network whose state is
kept separate from mainnet.

On mainnet, validator Weights applies on the live, production Bittensor network, where the
surrounding state is real and persistent.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A validator Weights example from one environment should
not be read as representing another environment.

## Reader Boundary

Validator weights name subnet evaluation signals. They describe submitted miner-work assessments
rather than miner leaderboards, reward guarantees, validator permits, or a complete explanation of
Yuma Consensus
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Specific examples belong to the subnet, mechanism, network, and source that produced the value.

## Weights Arrive Through the Tempo Cycle

Official incentive-mechanism documentation describes validators as periodically computing
[weight vectors](https://docs.learnbittensor.org/resources/glossary#weight-vector) from miner
performance and transmitting those vectors to the blockchain
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

The chain then waits for the latest ranking signals from the subnet's validators before assembling
[weight matrices](https://docs.learnbittensor.org/resources/glossary#weight-matrix) for Yuma
Consensus input. That collection step treats validator weights as submitted evaluation material that
must be gathered subnet-wide rather than as one validator's standalone verdict.

Epoch documentation states that validator weights are submitted during the preceding
[tempo](https://docs.learnbittensor.org/resources/glossary#tempo), while emissions from the
consensus pass are finalized at the end of each tempo period
([Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch),
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

A tempo is the subnet-specific interval, described in the glossary as a 360-block period over which
Yuma Consensus calculates emissions from the latest available ranking weight matrix. Validator
weights therefore belong to a recurring submission window rather than to one isolated snapshot.

References:
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)

## Multiple Mechanisms Carry Separate Weight Sets

Subnet documentation describes subnets that run more than one
[incentive mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism) as
maintaining separate evaluation tracks rather than one blended ranking table
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

In that layout, each validator sets weights independently for each mechanism, and each mechanism
keeps its own weight matrix for Yuma Consensus calculations. The
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector) notes
the same pattern at validator scope: when multiple mechanisms are used, validators maintain separate
weight vectors for each one.

Each mechanism also keeps independent bond pools and independent emissions paths tied to the weights
validators submit for that mechanism's work. A high weight in one mechanism's vector therefore does
not automatically carry the same meaning in another mechanism's separate matrix on the same subnet.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)

## Eligibility Rules Trim Submitted Weights

Not every value a validator submits survives into consensus unchanged. Epoch documentation lists
pre-consensus filtering steps that remove certain submitted weights before stake-weighted
aggregation begins
([Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

Self-assigned weights are removed so validators cannot vote for themselves, with an exception for
the subnet creator when that role exists on the subnet. Weights tied to a neuron before that
neuron's latest [registration](https://docs.learnbittensor.org/resources/glossary#register) are also
dropped so stale assignments do not follow re-registered identities.

Only rows from validators holding a
[validator permit](https://docs.learnbittensor.org/resources/glossary#validator-permit) influence
consensus. The glossary describes that permit as a validation-rights flag required for setting
weights and participating in consensus, awarded to the top neurons by stake weight on the subnet.

Together, those checks limit which submitted validator weights can enter Yuma Consensus at all.

References:
[Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch),
[Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit)

## Network Examples

Validator-weight examples from one environment are not evidence for another because subnet tasks,
validators, and incentive mechanisms can differ
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).
