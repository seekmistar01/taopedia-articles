# Subnet Markets

Subnet markets are the economic and evaluation contexts created by Bittensor subnets. Each subnet
defines a market for a specialized digital commodity
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction)).

The term is useful because it keeps one subnet market separate from the whole Bittensor network. A
subnet narrows the work category and evaluation context, while the broader protocol supplies the
chain and tokenomics environment around it.

## Market Role

Bittensor subnets are specialized competition markets. Miners produce work for the market,
validators evaluate that work, and the protocol connects evaluation to reward allocation
([Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

That role is narrower than the whole network. A subnet market names one work-and-evaluation context;
Bittensor names the broader protocol where many subnet markets can operate.

This distinction keeps market claims scoped. A statement about one subnet market should not be read
as a statement about every subnet or about Bittensor as a whole
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## Work and Evaluation

Subnet markets need both production and evaluation. Incentive-mechanism documentation explains how
subnet creators define tasks and validation logic, while Yuma Consensus connects validator
evaluation to miner incentives and validator dividends
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

For readers, the market is not just the work category. It is the context where work, evaluation, and
reward signals are connected.

The evaluation side matters because Bittensor markets are not generic listings of services. They are
incentive environments where validator assessment connects work quality to rewards
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Token Context

Subnet markets also have a token side. Subnet documentation describes TAO reserves, alpha reserves,
and alpha outstanding as part of subnet tokenomics, while emissions documentation connects TAO and
alpha emissions to subnet reward flow
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

This means a subnet-market claim should separate work context from token context. The work market
names what is being produced and evaluated; the token context explains how TAO and alpha are used
around that market.

That separation avoids making tokenomics carry the whole definition. A subnet market is not just an
asset context, and it is not just a work category; it is where the work and token contexts meet.

## Emission Context

Emissions make subnet markets economically meaningful inside Bittensor. Emission documentation
describes TAO and alpha-token flows, while Yuma Consensus describes how validator evaluation
contributes to reward allocation ([Emission](https://docs.learnbittensor.org/learn/emissions),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The important boundary is that emissions do not define the task by themselves. They are the reward
flow around a market whose task and evaluation logic come from the subnet's incentive design.

Emission language should therefore stay attached to the market being discussed. The reward flow
helps explain why evaluation matters, but it does not replace the subnet-specific task definition.

## Network Context

Bittensor documentation separates localnet, testnet, and mainnet environments
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Subnet-market examples should keep that environment boundary attached. A localnet or testnet example
can illustrate market mechanics, but it is not evidence of a mainnet subnet-market outcome.

The concept can remain stable while examples belong to different environments. The environment label
should stay with the example so readers do not move a test context into a production claim.

## Reader Boundary

Subnet markets should be read as per-subnet economic and evaluation contexts. They connect work,
validation, staking context, TAO and alpha tokenomics, and emission outcomes without replacing the
specific documentation for any one incentive design
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

The stable article-level point is that a subnet market names the context where a subnet's work,
evaluation, and reward flow meet. Claims about observed activity need the relevant network and
source context.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Subnet
Markets, this sequence gives readers a boundary for interpreting market examples, liquidity notes,
and network-comparison context.

Localnet examples are isolated and reflect local chain state, so they are useful for controlled
experiments rather than evidence of live Bittensor behavior. Testnet examples add shared
non-production conditions, which can reveal integration behavior without touching mainnet state.

On mainnet, Subnet Markets examples should be read as live production subnet-market behavior on the
production Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, so outcomes from one environment should not be treated as
proof of behavior in another.
