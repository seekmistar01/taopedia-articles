# Subnet Scoring Model

A subnet scoring model is the evaluation method validators use when judging miner responses inside a
Bittensor subnet. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model) defines the term
as the part of an incentive mechanism that converts miner responses into numerical scores, and
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
places it beside the subnet task and miner-validator interaction rules.

## Evaluation Role

The scoring model explains how returned work is judged. The subnet task describes what miners are
asked to produce, while the scoring model describes how validators evaluate the responses they
receive.
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
separates the task, protocol, and scoring model as related parts of a subnet incentive mechanism.

## Relationship to Miners and Validators

Miners produce responses for the subnet, and validators evaluate those responses under the subnet's
standards. [Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)
describes subnets as markets where miners produce a digital commodity and validators measure miner
work, while the [Glossary](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model)
ties the scoring model to validator evaluation of miner responses.

## Relationship to Weights

A scoring model is not the final subnet consensus result. It belongs earlier in the evaluation path,
where validators translate miner responses into evaluation signals. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-vector) describes weight
vectors as validator-maintained weights for subnet miners, and
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) explains that
validator scores become inputs for later consensus.

## Consensus Context

Yuma Consensus uses validator-provided signals after miners have already been evaluated under the
subnet's standards. This keeps the scoring model separate from the consensus process that combines
validator signals. [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) provides
the consensus context, and
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
describes scoring as part of subnet incentive design.

## Relationship to Multiple Mechanisms

Most scoring-model discussion centers on one incentive mechanism, but subnets can also run multiple
mechanisms in parallel. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model) notes that when
multiple incentive mechanisms are used, each mechanism can have its own scoring model for
independent evaluation, and the
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)
docs explain how multiple mechanisms can operate inside one subnet.

For readers, a subnet scoring model still names the evaluation method for miner responses, but the
mechanism context matters. A scoring standard for one mechanism does not automatically describe how
miners are judged in another, and validators are expected to evaluate miners separately for each
mechanism they validate.

## Reader Boundary

Subnet scoring model names the evaluation method, not a universal benchmark for all Bittensor
subnets. A useful scoring model depends on the subnet task, the miner-validator interaction pattern,
and the kind of work the subnet is meant to evaluate.
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)
frames scoring models around the desired subnet outcome, while
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) explains why
subnet goals and standards differ.

## Relationship to Coinbase

A subnet scoring model and coinbase sit at different ends of the same incentive pipeline. A subnet
scoring model is the evaluation method that converts miner responses into numerical scores, which
validators then express as weight signals. Coinbase is the per-block protocol mechanism that drives
TAO emission, injects TAO into subnet pool reserves each block, accumulates pending emissions, and
checks epoch boundaries to trigger Yuma Consensus rounds. The
[Glossary: Subnet Scoring Model](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model)
defines the evaluation method, and the
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
documentation describes the per-block emission operation.

The scoring model is near the start of the pipeline, and coinbase is near the end. A scoring model
turns miner responses into scores that shape validator weights; those weights become inputs to Yuma
Consensus, and coinbase is the mechanism that triggers the Yuma Consensus round at each epoch
boundary and distributes the resulting miner incentives and validator dividends. The scoring model
determines how well a miner is judged to have performed, while coinbase converts the consensus built
from those judgments into actual emission outcomes. One defines the quality signal; the other
performs the distribution that the signal ultimately influences.
