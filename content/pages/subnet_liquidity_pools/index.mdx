# Subnet Liquidity Pools

Subnet liquidity pools are the reserve context for subnet tokenomics in Bittensor. Each pool
connects a TAO reserve with the subnet's alpha-token reserve
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Slippage](https://docs.learnbittensor.org/learn/slippage)).

The term should be read as reserve vocabulary. It names the pool structure and the reserve
relationship, not a static report of exact balances or a complete explanation of every tokenomics
process connected to the pool.

## Reserve Pair

A subnet pool has a TAO reserve and an alpha reserve. The TAO side connects the subnet to
network-level TAO, while the alpha side connects the pool to the subnet-specific token context
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Those reserves should be kept separate from alpha outstanding. Pool reserves describe what sits in
the pool, while alpha outstanding describes alpha outside that reserve context.

This reserve pair also keeps TAO-side and alpha-side statements from being merged into one vague
pool claim. A source about the TAO reserve is not automatically a source about alpha outside the
pool.

## Constant Product Context

The slippage documentation describes Bittensor subnet pools as constant product automated market
maker pools. Staking and unstaking therefore pass through paired reserve behavior rather than a
fixed one-sided conversion ([Slippage](https://docs.learnbittensor.org/learn/slippage)).

That is why pool claims should name both reserve sides together. A change involving one side of the
pool is interpreted through the paired TAO and alpha reserve relationship.

Constant product context belongs to the pool structure. It should not be used as shorthand for
validator behavior, subnet quality, or every emission-related result connected to a subnet.

## Staking and Unstaking Context

When TAO is staked into a subnet, the staking flow passes through the subnet pool reserve context
and produces alpha-denominated stake. When stake is removed, alpha-denominated stake is converted
back through the reserve system
([Staking and Delegation](https://docs.learnbittensor.org/staking-and-delegation/delegation),
[Glossary: Unstaking](https://docs.learnbittensor.org/resources/glossary#unstaking)).

This keeps staking and unstaking connected without making them identical. Staking begins from TAO
and creates subnet alpha stake, while unstaking moves in the opposite direction through the pool
context.

## Slippage Boundary

Slippage is the difference between a simple expected conversion and the result produced by the pool
context used for the action ([Slippage](https://docs.learnbittensor.org/learn/slippage)).

For a reference page, the important point is conceptual: pool reserves affect the resulting TAO or
alpha amount. A static article should not treat one example as a general pool rule.

Slippage language should therefore remain tied to the pool action being discussed. The pool gives
the conversion context; the surrounding article still needs the source for the example it describes.

## Emissions and Protocol Alpha

Bittensor emission documentation connects subnet pools to reserve injection. Emission flow can add
TAO into subnet pool context, while the surrounding tokenomics model also separates TAO reserve,
alpha reserve, and alpha-outstanding concepts
([Emission](https://docs.learnbittensor.org/learn/emissions),
[Emission: TAO reserve injection](https://docs.learnbittensor.org/learn/emissions#tao-reserve-injection)).

Protocol alpha is related to subnet pools but not the same term. Protocol alpha names protocol-owned
alpha cached per subnet, while a subnet liquidity pool names the paired reserve structure
([Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)).

That separation is useful because both terms can appear near reserve injection. Pool vocabulary
names the reserve structure, while protocol-alpha vocabulary names a protocol-held alpha category.

## Network Context

Bittensor documentation separates localnet, testnet, and mainnet environments
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Subnet liquidity pool examples should keep that environment boundary attached. A localnet or testnet
example can illustrate reserve behavior, but it is not evidence of a mainnet pool outcome.

## Reader Boundary

Subnet liquidity pools should be read as reserve and conversion vocabulary. They help explain why
TAO reserves, alpha reserves, staking, unstaking, slippage, emissions, and protocol alpha need
separate wording
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Slippage](https://docs.learnbittensor.org/learn/slippage)).

The stable article-level point is that a subnet liquidity pool names the paired TAO and alpha
reserve context for a subnet. Claims about exact values, timing, or examples need the specific
source that reports those values.
