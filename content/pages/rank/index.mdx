# Rank

Rank is a Bittensor consensus metric for summarizing miner evaluation inside a subnet. It is the
aggregate miner-side result produced by Yuma Consensus from validator weight signals
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The term belongs to the miner-evaluation side of consensus. It is not one validator's raw view and
not a general reputation label for every Bittensor activity.

## Consensus Result

Rank appears after validator weight signals have been compared, filtered, and aggregated. It answers
where a miner lands in the subnet's consensus-adjusted evaluation result
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

This makes rank a result term. Validator weights are inputs; rank is the miner-side result built
after those inputs pass through consensus.

The result is still subnet-scoped. A rank value belongs to the subnet task, validators, and miner
set that produced it rather than to a universal miner leaderboard across Bittensor
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## From Weights to Rank

Validator weights express evaluations of miner work, while rank summarizes the consensus-adjusted
result of those evaluations. A single validator weight signal can influence the process, but rank is
not a single validator's opinion
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

That distinction keeps input terminology separate from result terminology. Weights describe
submitted evaluation signals; rank describes the aggregate miner-side result.

This also keeps a single validator's evaluation from being mistaken for consensus. The weight signal
can contribute to rank, but rank is only meaningful after those signals have been processed through
the subnet consensus mechanism
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Rank and Trust Terms

Rank, trust, and validator trust are related consensus terms. Rank names the miner-side aggregate
result. Trust indicates how much miner-side signal remains after clipping, while validator trust
indicates how much of a validator's signal remains after filtering
([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

They are complementary views of one consensus process. Rank explains the miner-side result; trust
terms help explain clipping effects.

The relationship is useful because the terms answer different questions. Rank asks where a miner
lands after aggregation; trust and validator trust describe how much signal remains after clipping
or filtering ([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)).

## Emission Context

Rank matters because Yuma Consensus uses miner evaluation when distributing subnet incentives. It
belongs near emissions terminology, but it is not a reward promise by itself
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

The useful reading is that rank contributes to how miner-side outcomes are shaped in consensus.
Emissions provide the broader reward-flow context.

That boundary keeps rank from being read as a payout table. Rank helps explain miner-side ordering
inside the consensus process; emission pages explain how reward flow is handled at the tokenomics
layer ([Emission](https://docs.learnbittensor.org/learn/emissions)).

## Subnet Context

Rank is meaningful inside the subnet context that produced it. Subnets define work markets where
miners produce outputs and validators evaluate them according to subnet-specific standards
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

For that reason, rank should be read with the relevant subnet task and evaluation context attached.
It is a consensus result for that setting, not a universal quality label across unrelated work.

Subnet context also guards against comparing unlike work. A rank result from one subnet reflects
that subnet's task and incentive mechanism, so it should not be treated as an all-purpose score for
miners in unrelated subnet settings
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## Reader Boundary

Use rank when the focus is the miner-side aggregate result after consensus filtering. Use consensus
score for the agreement threshold, validator weights for the inputs, and trust terms for how much
clipping changed the signals
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

The stable article-level point is that rank explains the consensus-adjusted miner result after
validator evaluation signals are filtered and aggregated.

## Environment Boundary

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. Rank keeps the same consensus-result meaning across those
environments, but a concrete rank value belongs to the network environment, subnet task, validator
set, and miner set that produced it.

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
places subnet development across localnet, testnet, and mainnet. Rank examples are clearest when the
environment and subnet context stay attached to the consensus result being discussed.
