# Self-Weighting

Self-weighting is when a single validator assigns inflated weights to entities it controls, such as
its own miner registrations, to try to steer more reward toward itself. On a subnet, for example
netuid 1, the worry is a validator grading its own side generously. Bittensor's consensus is
arranged so that doing this on its own does not pay
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## Why a Lone Validator Cannot Decide

Consensus is a stake-weighted agreement taken across validators, not any one validator's opinion.
The documentation describes weights that outlie that agreement being clipped, so an inflated
self-weight that departs from what the broader stake supports is trimmed away rather than paid out.
A single validator cannot move the agreement point by itself unless it holds enough stake to shift
the weighted middle
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## The Alignment Incentive

There is also a pull in the other direction. Validators are rewarded for agreeing with consensus, so
a validator that deviates to favor its own side works against its own standing rather than for it.
Self-weighting therefore tends to be self-defeating: the inflated portion is clipped, and the act of
departing from consensus can weaken the validator's own position
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## Validator Permits Gate Non-Self Weight Submissions

Documentation awards
[validator permits](https://docs.learnbittensor.org/resources/glossary#validator-permit) to the top
[stake](https://docs.learnbittensor.org/resources/glossary#stake)-weight neurons on a subnet. Epoch
consensus then treats permit status as part of which validator weight rows participate in the normal
aggregation path for that subnet
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

That gate is narrower than a blanket ban on every self-directed weight. Permit rules focus on
non-self weight submissions and standard consensus participation, while separate protocol rules can
still allow limited self-weight or owner-related exceptions. Self-weighting therefore remains a
consensus-clipping problem even when a neuron can submit a narrow self-directed weight row
([Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit)).

## Bonds Scale How Validator Signals Count Toward Miners

The
[Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds)
describes bond formation as starting with an instant bond for each validator-miner pair, then
smoothing those values through an exponential moving average. A validator's
[stake](https://docs.learnbittensor.org/resources/glossary#stake) scales how much instant bond mass
it can add when its evaluation input for a miner is applied.

Self-weighting therefore competes inside bond-weighted consensus math, not inside a validator's
private scoring notebook. Inflated self-preference still has to survive stake-weighted agreement and
bond-weighted dividend logic after clipping removes outlying weights
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Clipping, Bonds, and Yuma Settlement on netuid 1

On a subnet such as netuid 1, [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)
aggregates validator [weights](https://docs.learnbittensor.org/resources/glossary#weight-vector)
into miner [incentives](https://docs.learnbittensor.org/resources/glossary#incentives) and validator
[dividends](https://docs.learnbittensor.org/resources/glossary#emission) only after clipping trims
weights that depart from the stake-weighted
[consensus score](https://docs.learnbittensor.org/resources/glossary#consensus-score)
([Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping)).

Self-weighting on netuid 1 therefore faces the full subnet pipeline: permit gating, bond-weighted
aggregation, clipping against consensus score, and only then emission settlement through the subnet
[emission](https://docs.learnbittensor.org/learn/emissions) path
([Emission](https://docs.learnbittensor.org/learn/emissions)).

Self-weighting names the exploit pattern; Yuma names the consensus machinery that limits how far a
lone validator's inflated grades can move netuid 1 payouts.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For self-weighting
on a subnet such as netuid 1, that sequence changes how readers should interpret stake-weighted
consensus and validator clipping examples
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

In localnet, consensus resistance mechanics on netuid 1 can be tested in an isolated environment.
Localnet stake distributions do not represent production validator influence.

On testnet, validator weight and stake patterns on netuid 1 can be exercised in a shared
non-production network. Testnet consensus outcomes are separate from mainnet subnet state
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

On mainnet, self-weighting resistance on netuid 1 concerns live production validator stake and Yuma
Consensus on the selected subnet. Observed stake distributions and consensus values depend on that
subnet's current chain state.

A self-weighting example from one environment should not be read as representing production
consensus behavior in another environment.

## Distinction from Consensus Score

Self-weighting is when a validator assigns inflated weights to entities it controls. Consensus score
is the stake-weighted median benchmark that clipping uses to trim outlier weights in Yuma Consensus
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping)).

Exploit vocabulary and clipping-benchmark vocabulary answer different questions on the same weight
path. Self-weighting names the self-preferencing weight pattern; consensus score names the benchmark
that helps limit how far such weights can depart from broader validator agreement
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

Use self-weighting when covering a validator favoring its own side; use consensus score when
covering the clipping benchmark rather than the exploit behavior itself
([Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator)).

## Distinction from Rank

Self-weighting and rank describe the same manipulation attempt as the action and the result it
targets. Self-weighting is a lone validator assigning inflated weights to entities it controls,
while rank is the consensus standing produced after all validators' weights are clipped toward the
stake-weighted majority, so a single inflated weight is clipped away and the self-dealt miner's rank
does not rise ([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Distinction from Stake Weight

Self-weighting and stake weight meet at how much a manipulation can matter. Self-weighting is a
validator assigning inflated weights to entities it controls, while stake weight is the
stake-derived value that scales how much that validator's weights count in consensus, so a
self-weighting attempt moves the outcome only in proportion to the attacker's stake weight and stays
negligible from a low-stake validator
([Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Distinction from Collusion

Self-weighting and collusion both try to steer rewards through biased validator weights, but the
coordination model differs. Self-weighting is one validator inflating grades for entities it
controls without a coordinating group, while collusion is multiple validators acting together toward
shared targets. Both run into consensus clipping, but a lone validator lacks coalition stake to
shift the agreement point, whereas a coordinated group can pool stake-weighted signal
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Yuma Consensus: Clipping excess weights](https://docs.learnbittensor.org/learn/yuma-consensus#clipping-excess-weights)).

For readers, similar-looking weight matrices therefore need actor-count context. Self-weighting
turns on whether one permitted validator can move enough stake-backed agreement to survive clipping
on its own; collusion turns on whether a coalition can do the same at tempo-bound emission
settlement ([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

- **Self-weighting** — one validator inflating weights for entities it controls.
- **Collusion** — multiple validators coordinating inflated weights toward chosen miners.

## Distinction from Weight Copying

Self-weighting and weight copying can both produce aligned-looking validator rows, but they describe
different failures. Self-weighting is a validator inflating grades for entities it controls, while
weight copying is validators reusing visible weight signals instead of performing independent
evaluation — alignment without private coordination
([Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Glossary: Weight Copying](https://docs.learnbittensor.org/resources/glossary#weight-copying)).

For readers, copying concerns whether a submission reflects independent measurement, while
self-weighting concerns whether inflated self-preference can move stake-weighted consensus payouts
after clipping. Both interact with Yuma Consensus, but only self-weighting names deliberate
self-preference rather than imitated signals
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

- **Self-weighting** — inflating weights for entities the validator controls.
- **Weight copying** — reusing another validator's visible weights without independent evaluation.

## Distinction from Sybil Attack

Self-weighting and a Sybil attack both seek more reward than honest participation would allow, but
they operate on different layers. A Sybil actor multiplies identities at registration, while
self-weighting is a permitted validator biasing weight submissions after it already holds a subnet
slot. Identity flooding pays burn cost and competes for UID ceilings; self-weighting must survive
stake-weighted clipping inside the consensus pass
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

For readers, Sybil resistance is mostly economic at entry, while self-weighting resistance is
algorithmic inside weight aggregation. A crowd of low-stake registrations does not substitute for
one validator's stake-backed ability to move the agreement point
([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight)).

- **Self-weighting** — one validator favoring its own side through inflated weights.
- **Sybil attack** — one actor registering many identities to seek leverage from numbers.

## Distinction from Validator Permit

Self-weighting and the validator permit sit on the same submission path but name different ideas.
Self-weighting is biased scoring by a validator that may already hold a permit, while the validator
permit is the on-chain flag recording whether a UID may submit non-self weights that count in
consensus at all
([Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

For readers, holding a permit is necessary for a validator's weights to matter in the normal
aggregation path, but it does not make self-preference profitable. Permit rules gate participation;
clipping still trims inflated weights that depart from the stake-weighted consensus benchmark
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Yuma Consensus: Clipping excess weights](https://docs.learnbittensor.org/learn/yuma-consensus#clipping-excess-weights)).

- **Self-weighting** — a validator inflating weights for entities it controls.
- **Validator permit** — the on-chain right to submit non-self weights that count in consensus.

## Distinction from Yuma Consensus

Self weighting names the exploit pattern where a validator assigns disproportionate weight to its
own hotkey. Yuma Consensus is the separate mechanism that clips outlier weights against the
consensus score before converting submissions into emission shares each tempo
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Self weighting** — validators overweighting their own hotkey.
- **Yuma Consensus** — mechanism that turns validator weights into emission shares.

## Reader Boundary

This page defines the concept and Bittensor's consensus-level resistance at a high level. It does
not claim manipulation is impossible in every case, since a participant holding enough stake is a
different situation, nor does it quantify any threshold. The stake distribution and consensus values
that decide how strong the resistance is on any subnet are live chain state and change over time
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).
