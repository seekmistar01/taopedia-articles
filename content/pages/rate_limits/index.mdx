# Rate Limits

Rate limits are Subtensor timing rules that pace repeated Bittensor chain actions. They are measured
in blocks rather than ordinary wall-clock time, so the key idea is chain progression between
accepted actions ([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

The concept is useful for understanding pacing. A rate limit explains that an action is being
repeated too soon for the relevant rule; it does not by itself explain whether the action is useful
or part of a broader workflow.

## Cooldown Meaning

A cooldown begins from a successful action of the relevant kind. The
[rate-limit](https://docs.learnbittensor.org/learn/chain-rate-limits) material distinguishes
successful actions from failed attempts, so the accepted action is the pacing point.

That distinction keeps failed attempts from being overread. A failed attempt can return an error,
but the cooldown rule is tied to the successful action that the chain accepted for that action
family.

In practical reading, the accepted action is the anchor. Later attempts are compared with that
accepted action and with the number of blocks that have passed
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

## Scope of the Rule

Rate limits are scoped to action families. The same
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits) reference separates global
limits from subnet-specific limits, so a reader should keep the action and scope attached.

This prevents one cooldown from being generalized to all chain activity. A staking-related action, a
weight-setting action, and another subnet action can all use rate-limit language while answering
different timing questions.

The action family matters as much as the number of blocks. A staking cooldown and a subnet-specific
weight-setting cooldown should not be treated as the same rule
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Staking and Weight Examples

Staking operations provide one example of globally paced actions. Adding stake, removing stake, and
moving stake are discussed in the rate-limit context, while
[Staking and Delegation](https://docs.learnbittensor.org/staking-and-delegation/delegation) gives
the broader staking terms around those actions.

Validator weight setting is a different example because it is tied to subnet settings. The
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) reference
places weight-setting pace in subnet settings, while
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits) keeps the block-based timing
model attached.

These examples show why the action name should stay near the cooldown. The timing rule is not a
general pause for all Bittensor activity
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Staking and Delegation](https://docs.learnbittensor.org/staking-and-delegation/delegation)).

## Error Context

Rate-limit errors are diagnostic signals from the chain layer. The
[Subtensor Error Codes](https://docs.learnbittensor.org/errors) overview and
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor) reference place these
signals alongside other rejection categories, while the
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits) page explains the timing rule
behind too-soon repeats.

That makes a returned rate-limit error useful, but not a complete diagnosis. The attempted action,
selected network, subnet context, and relevant mechanism documentation still determine how the
signal should be interpreted.

A rate-limit error says the timing rule matters. It does not replace the staking, weight-setting, or
subnet-specific documentation for the action being attempted
([Subtensor Error Codes](https://docs.learnbittensor.org/errors),
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

## Transaction Boundary

Rate limits apply to submitted chain actions, not to every way of reading Bittensor information. The
[Transaction Fees](https://docs.learnbittensor.org/learn/fees) documentation distinguishes
state-changing submissions from free chain reads, and the
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits) reference focuses on pacing
accepted actions.

That boundary keeps rate-limit language away from passive observation. A reader should ask which
submitted action is being paced before treating timing as a rate-limit issue.

Read-only chain inspection can show public state, but the rate-limit rule belongs to accepted
submitted actions ([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).

## Reader Boundary

Rate limits should be read as block-based pacing rules for repeated chain actions
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)). They are not a general
retry guide, a complete troubleshooting transcript, or a list of every runtime value.

This article does not prescribe action timing, reproduce interface output, or diagnose private
state. Exact interpretation belongs to the submitted action, selected network, subnet context, and
official mechanism involved.

Network context still matters for examples. A localnet or testnet timing example can explain the
idea, but it should not stand in for another network's chain history
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits)).
