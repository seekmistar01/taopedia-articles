# Subnet Creator

A subnet creator is the role that defines what a Bittensor subnet is meant to produce and how that
work should be evaluated
([Glossary: Subnet Creator](https://docs.learnbittensor.org/resources/glossary#subnet-creator),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

The term belongs to subnet design vocabulary. It should be read separately from miner and validator
roles that operate inside the subnet after the work and evaluation context exist.

## Design Role

The creator role sits upstream of miner work and validator evaluation. It gives the subnet a work
target, an interaction context, and an evaluation frame so subnet roles are not working against a
generic network-wide task
([Glossary: Subnet Creator](https://docs.learnbittensor.org/resources/glossary#subnet-creator),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

For readers, the creator shapes the context. Miners produce work inside that context, and validators
evaluate work inside that context.

## Task Boundary

A subnet creator defines the digital task miners are asked to perform. The subnet task names the
work target, while the creator role names who establishes that target for the subnet
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Subnet Creator](https://docs.learnbittensor.org/resources/glossary#subnet-creator)).

This boundary keeps the role and the task separate. The creator is not the task itself; the creator
is the design role that makes the task meaningful for the subnet.

## Protocol and Scoring Context

Incentive-mechanism documentation separates task, protocol, and scoring model as related parts of
subnet design. A creator's design context can include each of those pieces without making them the
same term
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Subnet Creator](https://docs.learnbittensor.org/resources/glossary#subnet-creator)).

The subnet protocol describes how miners and validators interact. The scoring model describes how
returned work is evaluated. The creator role sits around those design choices rather than replacing
them.

## Miner and Validator Boundary

Subnet creators are not the same role as miners or validators. Miners produce work for the subnet,
and validators evaluate that work against subnet standards
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Glossary: Subnet Creator](https://docs.learnbittensor.org/resources/glossary#subnet-creator)).

This distinction matters because creator vocabulary is about design authority and context. Miner and
validator vocabulary is about operating inside the resulting subnet.

## Multiple-Mechanism Boundary

A subnet creator can shape more than one incentive mechanism inside a subnet. Multiple-mechanism
documentation describes separate mechanism contexts that can operate in parallel inside one subnet
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Subnet Creator](https://docs.learnbittensor.org/resources/glossary#subnet-creator)).

For readers, one mechanism's task, protocol, or scoring model should not automatically be treated as
the whole subnet's design. The mechanism context should stay attached to the claim.

## Network Evidence Boundary

Subnet-creator examples should keep their network environment attached. The Bittensor Networks
reference separates mainnet, testnet, and localnet as different environments
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

The creator concept is stable, but observed subnet behavior belongs to the environment where it was
observed. A static article should not turn one environment's subnet example into a general claim
about every subnet.

## Reader Boundary

Subnet creator should be read as a role concept. It is not a subnet launch guide, operations
checklist, funding explainer, or claim about one specific subnet's quality
([Glossary: Subnet Creator](https://docs.learnbittensor.org/resources/glossary#subnet-creator),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

The stable article-level point is that a subnet creator defines a subnet's work and evaluation
context, while miners and validators operate inside that created context.
