# Validator Trust

Validator trust is a per-validator metric in Bittensor's consensus process. The official glossary
describes it as measuring how much weight a validator successfully contributes after weights are
clipped by consensus.

References: [Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)

## What It Represents

Validators submit weight signals after evaluating miner work in a subnet. Consensus filtering can
reduce the influence of outlier assignments. Validator trust describes the remaining consensus
aligned portion of a validator's submitted weights, so it is about validator alignment rather than a
standalone miner score.

References: [Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)

## Relationship to Consensus Score

Consensus score and validator trust describe different sides of the same filtering process. The
consensus score is the agreement threshold around a particular target, while validator trust is a
per-validator view of how much submitted weight remains after that filtering step.

References: [Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)

## Relationship to Trust

The glossary distinguishes trust from validator trust. Trust is described for the evaluated target,
while validator trust is described for the validator that submitted weights. This distinction helps
avoid reading validator trust as a universal quality label for every participant in a subnet.

References: [Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)

## Relationship to the Weight Matrix

The weight matrix collects validator weight vectors for a subnet's consensus step. Validator trust
is read after those submitted weights have been compared with the broader consensus signal. The
matrix is the shared input; validator trust is one way of interpreting each validator's retained
influence after filtering.

References: [Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)

## Emissions Influence Context

Validator trust describes how much of a validator's submitted weight remains after consensus
filtering. That retained influence sits upstream of subnet emission allocation in
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus), where validator signals are
combined before miner and validator emissions are determined. The term is not an emission rate; it
names validator-side signal retention in that consensus flow.

References: [Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from local testing to testchain and then mainchain. For
validator trust, that sequence changes how readers should interpret evidence about validator-side
signal retention.

In local testing, validator trust can show whether the consensus path is retaining or clipping a
validator's submitted weights. That is useful for checking the shape of the metric, but it remains
isolated feedback from a local environment.

On testchain, validator-trust behavior can be observed in a shared testing network. This gives
stronger evidence about how validator submissions align with consensus than a private local run,
while still keeping the result separate from production mainchain incentives.

On mainchain, validator trust belongs to live Yuma Consensus output. The
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) documentation describes
validator weights being clipped and aggregated into incentive outcomes, so production validator
trust context is tied to live reward allocation.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A validator-trust value observed in one environment should
not be read as identical evidence about validator alignment in another environment.

Because validator trust is a validator-side outcome metric, its environment also affects which
validator submissions and subnet conditions it summarizes. A local result, a testchain result, and a
production outcome are not interchangeable evidence sources.

For readers, this keeps validator-trust examples from sounding stronger than their environment
supports. Local, testchain, and mainchain contexts can all describe validator alignment, but they do
not carry the same interpretive weight.

## Relationship to Multiple Mechanisms

Validator trust is usually discussed for one incentive mechanism at a time, but subnets can also run
multiple mechanisms in parallel. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#weight-matrix) notes that each
mechanism has its own weight matrix for independent consensus calculations.

For readers, validator trust still describes how much of a validator's submitted weight remains
after consensus filtering, but the mechanism context matters. Validator trust for one mechanism
should be read separately when a subnet uses more than one evaluation system.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)

## Careful Interpretation

Validator trust should be read inside its subnet and consensus context. It is not an instruction for
running a validator, not a general reputation score, and not a promise about future outcomes. Exact
calculation details belong in the official references; the Taopedia-level meaning is that validator
trust reflects how much of a validator's submitted signal remains aligned with consensus.

References: [Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)

## Relationship to Rank

Validator trust and rank are related but different parts of Bittensor consensus vocabulary. The
[Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)
describes validator trust as the measure of how much of a validator's submitted signal remains
aligned with consensus, and the
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank) describes rank as the
final aggregate judgment of a miner after consensus filtering.

For readers, validator trust names a validator-side alignment metric, while rank names a miner-side
performance score. One measures the validator; the other measures the miner. The two should not be
read as the same consensus output.

References: [Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)

## Relationship to Stake Weight

Validator trust and stake weight are related but distinct parts of Bittensor consensus vocabulary.
Validator trust is a validator-side alignment metric that measures how much of a validator's
submitted signal stays aligned with consensus after clipping. Stake weight is the computed total
stake value for a validator that determines their consensus power and emissions in a subnet. The
[Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)
entry describes validator trust as the sum of clipped weights a validator sets, and the
[Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight) entry
describes stake weight as the alpha-plus-TAO stake blend that feeds validator influence.

For readers, validator trust names how aligned a validator's submitted signal is, while stake weight
names how much influence that validator has on the consensus inputs the alignment is measured
against. The two are independent measurements on the same validator: stake weight sets the
validator's standing in the consensus arithmetic, and validator trust records how much of that
validator's signal the consensus kept after clipping. A validator with high stake weight can still
have low validator trust if their signal is repeatedly clipped, and vice versa.

References:
[Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight)

## Relationship to Subnet Deregistration

Validator trust and subnet deregistration address different parts of Bittensor consensus and network
removal vocabulary. Validator trust is a per-validator metric inside Yuma Consensus that describes
how much of a validator's submitted weight signal remains aligned with consensus after clipping,
while subnet deregistration removes an entire non-immune subnet from the network when the subnet
limit is reached and a new subnet registration attempts to enter, selecting the subnet with the
lowest EMA of alpha price. The
[Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust)
describes validator trust as a measure of how much weight a validator successfully contributes after
clipping, and the
[Subnet Deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration) documentation
describes the network-level subnet removal process.

Validator trust is a within-subnet signal about validator alignment — it is not the selection signal
for subnet deregistration. Subnet deregistration uses the EMA of alpha price as a network-level
measure of subnet demand rather than individual validator trust values. When a subnet is
deregistered, every participant — including validators with high trust — loses their slot
simultaneously because the entire subnet context is removed from the network.

References:
[Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Subnet Deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration)

## Relationship to Coinbase

Validator trust and coinbase are related but different parts of Bittensor's consensus pipeline.
Validator trust is the per-validator metric for how much of a validator's submitted weight signal
survives Yuma Consensus clipping. Coinbase is the per-block protocol mechanism that drives TAO
emission, injects TAO into emitting subnets' pool reserves, accumulates pending emissions, and
checks epoch boundaries to trigger Yuma Consensus rounds. The
[Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust) glossary entry
describes the metric, and the
[Coinbase Implementation](https://docs.learnbittensor.org/navigating-subtensor/emissions-coinbase)
documentation describes the per-block emission operation.

Validator trust is one of the values that coinbase-triggered epoch rounds produce. When coinbase
runs a subnet's Yuma Consensus round at its epoch boundary, the round compares submitted weights
against the consensus signal, and the portion of each validator's weight that survives clipping is
recorded as that validator's trust for the epoch. Validator trust is therefore an output of the
consensus computation coinbase drives, alongside other epoch outputs such as incentives and
dividends. Coinbase runs the round; validator trust is one measurement that round produces about how
aligned each validator's signal was.
