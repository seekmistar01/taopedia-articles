# Serving Rate Limit

Serving rate limit is a per-subnet hyperparameter, listed in the documentation as
`ServingRateLimit`, that rate-limits how often a miner can call the `serve_axon` and
`serve_prometheus` extrinsics to register or update its network endpoint information on a subnet
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## What It Controls

The hyperparameter paces miner-side endpoint registration, not validator weight submission. It does
not define what work a
[subnet miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner) produces or how
validators score that work. It only limits how frequently a miner can publish or refresh its
on-chain serving metadata through the documented serving extrinsics.

Miners commonly expose an [axon](https://docs.learnbittensor.org/resources/glossary#axon) server so
validators can reach them over the subnet communication path. Serving rate limit governs how often
the on-chain endpoint record for that axon may be rewritten, not whether the miner is registered or
whether the live server is reachable at query time
([Understanding Neurons](https://docs.learnbittensor.org/learn/neurons#neuron-to-neuron-communication)).

## Documented Type, Default, and Setter

The [Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)
reference lists `ServingRateLimit` as a `u64` with a default of `50` and the description "Rate limit
for calling `serve_axon` and `serve_prometheus` extrinsics used by miners." It is changed through
the `sudo_set_serving_rate_limit` extrinsic, exposed in tooling as
`btcli sudo set --param serving_rate_limit`, and the reference marks the permission required to set
it as subnet owner
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

Because each subnet carries its own hyperparameters, the pacing rule is defined per subnet and
applies only within that subnet's miner-serving path.

## Distinction from Weights Rate Limit

The hyperparameters reference also lists
[WeightsRateLimit](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#weightsratelimit--commitmentratelimit),
which sets how long, in blocks, a validator must wait between weight commits. That setting paces
validator [weight setting](https://docs.learnbittensor.org/resources/glossary#weight-vector), not
miner endpoint registration
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Serving rate limit** — how often a miner can update axon/prometheus serving metadata.
- **Weights rate limit** — how long a validator must wait between weight commits.

## Distinction from Tempo

Both serving rate limit and [tempo](https://docs.learnbittensor.org/resources/glossary#tempo) are
counted in blocks, which makes them easy to confuse, but they govern different things. Tempo is the
length of a subnet's consensus cycle — the cadence at which Yuma Consensus processes validator
weights — while serving rate limit only paces how often a miner may refresh its serving metadata
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Serving rate limit** — the minimum block gap between a miner's endpoint updates.
- **Tempo** — the block length of the subnet's consensus cycle.

## BTCLI Reads the Live Value

Any user can read subnet hyperparameters for a selected netuid with
`btcli subnet hyperparameters --netuid 1`. The command output includes the live `ServingRateLimit`
value for that subnet at query time
([Subnet Hyperparameters: View hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#view-hyperparameters)).

Live values are network- and subnet-specific and can differ from the documented default, so a pacing
claim should be checked against the hyperparameter table for the relevant netuid and network rather
than assumed from the reference default alone.
