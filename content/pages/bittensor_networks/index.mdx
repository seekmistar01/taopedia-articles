# Bittensor Networks

Bittensor Networks are the environments where Bittensor activity is interpreted. They separate
mainnet, testnet, and localnet context.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
defines those environments, while
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction) frames Bittensor as
a platform made from subnets, blockchain state, and supporting tools.

## Mainnet, Testnet, and Localnet

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
groups mainnet, testnet, and localnet as separate environments. Mainnet is the primary Bittensor
environment, testnet is used for shared testing, and localnet is used for local development and
experiments.

That vocabulary keeps examples from being overstated. A localnet example can demonstrate behavior
inside an isolated environment, while a testnet example belongs to shared testing context and a
mainnet example belongs to the primary network context.

## Networks Are Not Subnets

A network environment is broader than a subnet. The
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) documentation
describes a subnet as an incentive-based marketplace where miners produce work and validators
evaluate it. A Bittensor network is the environment where that activity is tested or used.

This is why [[subnet_creation_mechanisms|Subnet Creation and Incentive Mechanisms]] is related but
not the same topic. Subnets are markets inside the broader Bittensor setting; networks identify the
environment where examples, observations, and actions belong.

## Why Context Matters

Network context changes how a claim should be read. A localnet observation, a testnet observation,
and a mainnet observation can use similar vocabulary while describing different chain histories. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference is the
source for keeping those environments separate.

The context layer also prevents a tool example from sounding stronger than its environment supports.
If an article describes a value, block, subnet behavior, or chain inspection result, the network
context determines how broadly that evidence can be used.

## Development Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from local testing to testchain and then mainchain. The
Bittensor Networks reference provides the environment vocabulary for that path: localnet, testnet,
and mainnet.

Local testing can explain how a mechanism behaves in a controlled setting. Testnet can show shared
testing-network behavior. Mainnet context belongs to the primary Bittensor environment. These
contexts can all involve similar concepts, but they do not carry the same evidentiary weight.

## Local Blockchain Context

Localnet is closely related to local blockchain terminology. The
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain)
describes a local blockchain as private, isolated, and used for development and testing. That makes
local blockchain the chain-level local context, while Bittensor Networks is the broader environment
vocabulary.

The distinction helps keep local examples separate from shared network examples. A local blockchain
can be useful for development, but it should not be treated as proof of the same state on testnet or
mainnet.

## Evidence Boundary

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
supports claims about environment context. It does not redefine miners, validators, staking,
subnets, emissions, or chain mechanisms. Those topics need their own mechanism-specific sources.

When an article uses a network label, the label should explain where the claim is being observed. It
should not substitute for evidence about what a subnet does, why a value changed, or how a protocol
mechanism works.
