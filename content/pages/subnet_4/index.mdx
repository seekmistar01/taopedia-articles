# Subnet 4: Targon

Targon is Bittensor Subnet 4, a subnet focused on confidential AI compute. Its public materials
frame the project as a confidential decentralized AI cloud built around Confidential Compute and
Protected PCIe technology ([Subnet 4 on TaoStats](https://taostats.io/subnets/4),
[Targon repository](https://github.com/manifold-inc/targon)).

## What the Subnet Provides

Targon's project materials describe secure AI infrastructure rather than a model leaderboard or
prompt marketplace. The core theme is protected execution for AI workloads: hardware-supported
isolation, memory protection, secure boot, and remote attestation around compute used for model
inference or related AI infrastructure
([Targon repository](https://github.com/manifold-inc/targon)).

That framing matters because the subnet is not just about finding raw GPU or CPU capacity. The
reader-facing concept is protected compute capacity: AI work should be associated with a more
verifiable execution environment than an ordinary remote machine.

## Confidential Compute Context

Confidential compute is the security context behind Targon's positioning. The repository emphasizes
Confidential Compute, Protected PCIe, GPU trusted execution environments, memory isolation, and
remote attestation as parts of the infrastructure stack
([Targon repository](https://github.com/manifold-inc/targon)).

This separates Targon from a generic compute-marketplace description. A generic marketplace asks
whether compute is available. Targon's public framing asks whether the compute environment can be
protected and reasoned about when sensitive AI workloads run outside the requester's own hardware.

## Attestation and Protected Execution

Remote attestation gives the subnet's compute story a verification angle. In Targon's materials,
attestation appears alongside hardware-level security guarantees, protected model execution, and
verifiable computation. The point is not only that a machine returned a result, but that the
environment around that result is part of the value being offered
([Targon repository](https://github.com/manifold-inc/targon)).

Protected execution also keeps the article from overstating the service as general AI hosting. The
distinctive claim is narrower: Targon centers secure infrastructure for AI workloads that benefit
from isolation and attestation.

## On-Chain Identity

Live data for Subnet 4 is available on [TaoStats](https://taostats.io/subnets/4). Readers can also
reproduce the registered chain identity fields with the Bittensor CLI command documented in the
[btcli reference](https://docs.learnbittensor.org/btcli/#btcli-subnets-get-identity):
`btcli subnets get-identity --netuid 4 --network finney`.

That Finney identity reports the subnet name as Targon, and the GitHub repository is
[manifold-inc/targon](https://github.com/manifold-inc/targon), which documents the
confidential-compute behavior described above.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Subnet 4
Targon on netuid 4, that sequence changes how readers should interpret confidential-compute
examples, validator weights, and emission outcomes.

In localnet, Targon examples can help explain development behavior in an isolated environment.
Localnet confidential-compute demonstrations do not represent production subnet performance on live
networks.

On testnet, Subnet 4 activity can be observed in a shared, non-production network. Testnet miner
scores and validator weights on netuid 4 are separate from mainnet subnet state
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

On mainnet, Subnet 4 Targon is a live Bittensor subnet whose fee-based validator weights and Yuma
Consensus emissions apply on the production network
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A netuid 4 example from one environment should not be read
as representing production Targon performance or emissions in another environment.

## Concept Boundaries

Subnet 4 should be read as a Targon overview, not as a hardware compatibility matrix, or service
availability dashboard. The stable concept is the combination of Bittensor subnet incentives with
confidential-compute infrastructure for protected AI workloads
([Targon repository](https://github.com/manifold-inc/targon),
[Subnet 4 on TaoStats](https://taostats.io/subnets/4)).

Exact hardware availability, deployment details, pricing, and project-specific operating details
belong in the project's own materials. The stable distinction is that Targon is a
confidential-compute subnet, not a general AI application article.

## Miner and Validator Roles

Subnet 4 operates under the standard Bittensor two-role structure. Miners supply the subnet's
capability and validators evaluate those contributions and set weights. Reward distribution follows
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus).

## Protected Compute, Not Raw Capacity Listings

The same public materials emphasize Confidential Compute, Protected PCIe, memory isolation, and
remote attestation as infrastructure guarantees. Availability of ordinary compute hours alone does
not map onto Targon's stated value proposition without those protection and verification features.

## Attestation Is Part of the Offered Environment

The repository describes remote attestation alongside hardware-supported isolation and verifiable
computation. A returned inference result should not be treated as equivalent evidence without the
attestation and protected-execution context the project associates with its compute stack.

Validator weights still flow through Yuma Consensus to determine emissions each tempo
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

## Reader Boundary

Subnet 4 Targon should not be read as a generic GPU marketplace or prompt-leaderboard subnet. The
[Targon repository](https://github.com/manifold-inc/targon) frames SN4 around confidential
decentralized AI compute with protected execution for sensitive workloads rather than
undifferentiated remote capacity.
