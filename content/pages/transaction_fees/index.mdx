# Transaction Fees

Transaction fees are costs the Bittensor chain can attach to state-changing actions
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

The term describes chain-processing costs. It is separate from chain reads, swap fees, batch
grouping, and every other economic effect that can appear near a transaction.

## Base Fee Components

Bittensor fee documentation describes base transaction fees as having weight and length components
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

Weight is tied to computational load, while length is tied to transaction size. This base-fee
context belongs to the state-changing action, not to passive observation.

Fee-bearing extrinsics pay both components from the sender's TAO free balance. The weight fee is
based on the `ref_time` part of transaction weight, and the length fee is charged per encoded byte
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

The fee documentation states the total base transaction fee as the sum of the weight fee and the
length fee. For length, the documented charge is 1 rao per byte of the encoded extrinsic. For
weight, the documented formula scales the `ref_time` component by `50,000 / 10^9` rao
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

This makes weight a fee-input measure, not a promise that an extrinsic will be included or succeed.
The extrinsic reference still describes the runtime call being submitted, while the fee reference
describes how a fee-bearing submission is charged
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

A base-fee example can therefore describe the charge formula without describing every economic
effect around the transaction. Swap fees, batch wrapper overhead, and recycling accounting remain
nearby fee concepts, but they answer different parts of the cost picture.

## Fee-Free and Alpha Cases

The fee documentation lists fee-free extrinsics that pay neither the weight component nor the length
component. The examples include `set_weights`, `commit_weights`, `reveal_weights`, and `sudo`
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

Alpha fallback is a separate fee-payment case. When the sender's TAO balance is insufficient, the
fee documentation describes transaction fees being paid in alpha instead
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

The fee-free list is about whether the chain charges the base components at all. Alpha fallback is
about the asset used to pay fees when a fee-bearing transaction cannot be covered by TAO.

These cases should not be collapsed into one rule. A fee-free extrinsic is listed as not paying the
base components; alpha fallback is still a fee-payment path, only with a different asset source
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

## Reads, Swaps, and Batches

The fee documentation identifies chain reads as free
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

That distinction matters because reading chain state does not ask Subtensor to record a state
change. Fee language should stay attached to state-changing actions rather than every form of chain
inspection ([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

The same distinction helps when a read appears immediately before a transaction. The read can show
state used by the user or interface, while the later submitted action is the fee-bearing event
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

The fee documentation treats swap fees as separate from base transaction fees
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

A stake-related action can therefore involve more than one cost concept. The base fee belongs to the
chain action, while a swap fee belongs to the liquidity-linked part of the action. The two can
appear near each other without meaning the same thing.

The fee page gives the swap fee as 0.05% of transacted liquidity for staking and unstaking
operations ([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)). That
percentage belongs to the amount-based swap-fee side of the action, while the base transaction fee
still follows weight and length.

Batch transaction documentation describes grouped calls, and the fee documentation describes batch
fees as combining inner-call behavior with batch-wrapper overhead
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

This means grouped activity should still be read through its components. A batch is a grouped
submission pattern, not a way to erase fee-bearing activity inside the batch.

Together, these neighboring concepts keep the transaction-fee term narrow. Reads are inspection,
swap fees are liquidity-linked, and batches describe grouped submission; none of those labels
replace the base fee components.

## Extrinsic Submission and Fee Pipeline

Transaction fees sit on the extrinsic submission path. A state-changing
[extrinsic](https://docs.learnbittensor.org/subtensor-api/extrinsics) names the runtime call being
submitted; fee documentation explains how that submission is charged before the chain records the
resulting state change
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

The useful pipeline order is therefore extrinsic selection, fee calculation from weight and length,
payment from TAO free balance or alpha fallback, block inclusion of the accepted transaction, and
downstream accounting of the collected fee. Fee language belongs to the submission and payment step;
block inclusion confirms the charge landed in chain history without redefining what triggered it
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

Weight-setting extrinsics such as `set_weights`, `commit_weights`, and `reveal_weights` are listed
as fee-free, so consensus participation through weight submission does not carry the same base fee
formula as many other state-changing actions
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

Fee-free listing is separate from swap fees or alpha fallback. It names extrinsics the chain does
not charge through the standard weight-and-length base formula.

## Fee Recycling, Burning, and Emission Pipeline

Paid base fees are deducted from
[TotalIssuance](https://docs.learnbittensor.org/resources/glossary#issuance), and the
[Recycling and burning](https://docs.learnbittensor.org/resources/glossary#recycling-and-burning)
glossary entry describes collected transaction fees as recycled tokens removed from the issuance
record for future [emission](https://docs.learnbittensor.org/resources/glossary#emission)
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

For readers, the pipeline order is: a fee-bearing extrinsic pays weight and length charges, the
collected base fee reduces total issuance, recycling returns that value to the protocol supply path,
and the broader emission process continues distributing newly minted value at the adjusted supply
pace ([Emission](https://docs.learnbittensor.org/learn/emissions)).

[Burning](https://docs.learnbittensor.org/resources/glossary#burning) is a separate supply movement
that removes existing tokens from spendable circulation. Recycling transaction fees changes issuance
accounting for fees already collected; burning addresses a different token exit path. Neither
replaces the base fee formula that charged the extrinsic in the first place.

That accounting path explains where base fees go after payment. It does not change what triggered
the fee, but it connects chain charges to broader tokenomics vocabulary.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For transaction
fees on a subnet such as netuid 1, that sequence changes how readers should interpret fee quotes,
RAO-denominated charges, and stake-related swap fees across environments.

In localnet, transaction-fee examples can be exercised in an isolated environment. Local fee
parameters and staking swap examples on netuid 1 reflect local chain configuration rather than
production fee levels.

On testnet, transaction fees can be observed in a shared, non-production network. Testnet fee quotes
and stake-related swap charges for netuid 1 are separate from mainnet extrinsic costs
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

On mainnet, transaction fees are live production charges for extrinsic length and weight on the
connected Bittensor network, including stake-related actions on production subnet pools such as
netuid 1 ([Glossary: RAO](https://docs.learnbittensor.org/resources/glossary#rao)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A fee example from one environment should not be read as
representing production transaction costs in another environment.

## Distinction from Mempool

A transaction fee is the cost the chain attaches to a state-changing action. The mempool is the
holding area where submitted transactions wait before a block includes them
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

## Distinction from Swap Fees

A base transaction fee and a swap fee are two different costs on a stake-related action. The base
transaction fee follows the extrinsic's weight and length, while the swap fee is a percentage of
transacted liquidity charged on staking and unstaking conversions through the pool
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Understanding Slippage](https://docs.learnbittensor.org/learn/slippage)).

- **Transaction fee** — the weight-and-length cost of processing the extrinsic.
- **Swap fee** — a liquidity-percentage cost on the pool conversion.

A stake-related action can therefore involve both a base chain fee on the extrinsic and a
liquidity-linked swap fee on the converted amount, so transaction-fee vocabulary stays attached to
the chain-processing charge while swap-fee vocabulary belongs to the pool conversion side
([Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation),
[Understanding Slippage](https://docs.learnbittensor.org/learn/slippage)).

It is easy to assume a transaction fee flows to a validator the way a tip might. Bittensor
documentation instead frames fees as the cost the chain attaches to state-changing actions, with the
deducted amount handled through recycling-and-burning accounting rather than paid out as a
validator's earnings ([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Recycling and burning](https://docs.learnbittensor.org/resources/glossary#recycling-and-burning)).

For readers, this separates fee vocabulary from reward vocabulary. A validator's earnings come from
the emission and consensus path, while a fee is what a submitter pays to have an action processed,
so the two should not be read as the same flow
([Emission](https://docs.learnbittensor.org/learn/emissions)).

That framing also clarifies why fee size relates to chain work rather than reward size.
Documentation ties the base fee to the action being submitted, so a fee reflects the cost of
processing that call, not the value of any downstream reward
([Glossary: Extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

## Distinction from Burning

A transaction fee and burning describe a charge and a supply outcome. A transaction fee is the cost
charged to process an action; collected base fees are recycled for future emission, whereas burning
removes tokens from spendable circulation without that re-emission path
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Glossary: Burning](https://docs.learnbittensor.org/resources/glossary#burning)).

- **Transaction fee** — the cost charged to process an action (then recycled).
- **Burning** — removal of tokens from circulation without re-emission.

## Distinction from Batch Transactions

A batch transaction and a transaction fee name a grouping and a cost. A batch groups calls into one
outer submission, while the fee documentation still charges inner-call fees plus batch wrapper
overhead, so grouping does not make the inner actions free
([Batch Transactions](https://docs.learnbittensor.org/learn/batch-transactions),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

- **Batch transaction** — how calls are grouped into one submission.
- **Transaction fee** — the cost, charged as inner-call fees plus wrapper overhead.

## Distinction from Chain Reads

A chain read and a transaction fee sit on opposite sides of the free/paid boundary. Documentation
treats a chain read as free inspection of public state, while a transaction fee is the cost charged
to process a state-changing action
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

- **Chain read** — free, read-only observation of public state.
- **Transaction fee** — the cost charged to process a state-changing action.

## Distinction from Recycling and Burning

Recycling and burning name a supply-accounting outcome, while a transaction fee names a charge. A
transaction fee is the cost charged to process an action; its collected amount is then handled
through recycling rather than the fee itself being a removal category
([Glossary: Recycling and burning](https://docs.learnbittensor.org/resources/glossary#recycling-and-burning),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

- **Transaction fee** — the cost charged to process an action.
- **Recycling/burning** — how the removed amount is treated for future emission.

## Distinction from Yuma Consensus

Transaction fees are costs the Bittensor chain attaches to state-changing actions. They name
chain-processing cost vocabulary, not the on-chain step that turns validator weight submissions into
emission shares ([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

Yuma Consensus is the on-chain process that converts submitted validator weights into emission
shares after evaluation completes. Base transaction fees cover extrinsic inclusion and execution;
Yuma Consensus settles scoring outcomes on a subnet
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers should keep transaction-fee vocabulary separate from the settlement step. Transaction fees
answer what a state-changing action costs to submit; Yuma Consensus answers how validator weights
become emission shares in a subnet
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Transaction fees** — chain-processing cost for state-changing extrinsics.
- **Yuma Consensus** — on-chain weight aggregation that settles emissions.

## Reader Boundary

Transaction fees should not be read as a permanent quote for every action, a calculator result, or a
complete description of all costs that can appear near a transaction. The term names
chain-processing costs tied to state-changing extrinsics on Subtensor
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor)).

Use transaction fee language for base weight-and-length costs on state-changing actions. Use
neighboring vocabulary when the question is a read, swap fee, batch wrapper, recycling accounting
record, or a fee observation tied to one network environment
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).
