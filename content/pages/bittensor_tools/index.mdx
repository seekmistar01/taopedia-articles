# Bittensor Tools

Bittensor tools are software interfaces around the Bittensor network. They include the SDK, terminal
surfaces, and wallet/key material used to interact with network features
([Bittensor Tools: SDK](https://docs.learnbittensor.org/concepts/tools#bittensor-sdk),
[Bittensor Tools: CLI](https://docs.learnbittensor.org/concepts/tools#bittensor-cli),
[Bittensor Tools: Wallets and Keys](https://docs.learnbittensor.org/concepts/tools#wallets-and-keys),
[Wallets and Keys](https://docs.learnbittensor.org/keys/wallets)).

The tool name answers an access question: which software surface is being used, and which audience
is it meant to serve. Subnet, consensus, and governance references still explain the network rules
behind the feature being reached
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Governance](https://docs.learnbittensor.org/governance)).

## Interface Layer

Tooling sits around Bittensor roles and features. A tool can help a developer, miner, validator, or
subnet creator reach information or submit network-facing activity
([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools),
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction)).

That makes the tool layer practical rather than purely conceptual. It identifies the interface used
to reach a feature: a developer library, terminal surface, wallet/key system, or user-facing tool
([Wallets and Keys](https://docs.learnbittensor.org/keys/wallets)).

## Tool Families

Bittensor tooling includes developer libraries, terminal interfaces, and account material
([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools)).

Those families differ by reader and task. Libraries support software integration, terminal surfaces
support direct interaction, and wallet/key material explains the account and signing surface around
network use
([Bittensor Tools: Wallets and Keys](https://docs.learnbittensor.org/concepts/tools#wallets-and-keys),
[Wallets and Keys](https://docs.learnbittensor.org/keys/wallets)).

## SDK Surface

The SDK belongs to the developer-library side of the tool family. It gives software a programmatic
surface for Bittensor features
([Bittensor Tools: SDK](https://docs.learnbittensor.org/concepts/tools#bittensor-sdk),
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction)).

That makes SDK language strongest when the subject is integration, application behavior, or a
software surface around the network. It is weaker by itself when the subject is a subnet task,
incentive mechanism, or governance process.

## Terminal and User Interfaces

Terminal and user-facing tools help people interact with Bittensor without writing an integration.
The tools overview treats the terminal surface separately from the SDK because the audience and use
shape are different
([Bittensor Tools: CLI](https://docs.learnbittensor.org/concepts/tools#bittensor-cli),
[Bittensor CLI](https://docs.learnbittensor.org/btcli),
[BTCLI Overview](https://docs.learnbittensor.org/btcli/overview)).

Wallet/key material is nearby because accounts and signing are part of using the network. That
material is useful for explaining account context, while terminal references identify the
user-facing surface
([Bittensor Tools: Wallets and Keys](https://docs.learnbittensor.org/concepts/tools#wallets-and-keys),
[Wallets and Keys](https://docs.learnbittensor.org/keys/wallets)).

Signing context matters for both the SDK and terminal surfaces, but account vocabulary still differs
from the tool interface being named: the SDK belongs to developer-library integration, while
terminal and user-facing tools support direct interaction without writing application code
([Bittensor Tools: SDK](https://docs.learnbittensor.org/concepts/tools#bittensor-sdk),
[Bittensor Tools: CLI](https://docs.learnbittensor.org/concepts/tools#bittensor-cli),
[BTCLI Overview](https://docs.learnbittensor.org/btcli/overview)).

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. Bittensor
documentation also separates mainnet, testnet, and localnet contexts
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

A tooling example belongs to the environment where it was observed. Localnet examples can show
controlled behavior, testnet examples can show shared non-production conditions, and mainnet
references concern production network use.

## Tools Do Not Define Miner or Validator Roles

Miners, validators, subnet creators, and developers can all use tools, but the interface does not
create the role. Role vocabulary describes participation in subnet work and evaluation, while tool
vocabulary describes how a participant reaches network features
([Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction),
[Bittensor Tools](https://docs.learnbittensor.org/concepts/tools)).

Keeping that boundary clear prevents CLI or SDK examples from being read as protocol definitions.

## Distinction from BTCLI

Bittensor tools and BTCLI relate as a category and a member. Bittensor tools name the whole
interface layer — developer libraries, terminal surfaces, and wallet/key material — while BTCLI is
the specific command-line tool within that layer
([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools),
[Bittensor CLI](https://docs.learnbittensor.org/btcli)).

- **Bittensor tools** — the whole interface layer around the network.
- **BTCLI** — the specific command-line tool inside that layer.

## Distinction from Wallets

Bittensor tools and a wallet name an interface layer and the account material it acts on. Tools are
the software surfaces used to reach network features, while a wallet is the coldkey/hotkey account
and signing material those surfaces operate with
([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools),
[Wallets and Keys](https://docs.learnbittensor.org/keys/wallets)).

- **Bittensor tools** — the software surfaces for interacting with the network.
- **Wallet** — the account and signing material those surfaces act on.

## Distinction from Bittensor SDK

Bittensor tools names the whole interface layer around the network. The Bittensor SDK is the
developer library surface used to integrate applications with Bittensor programmatically
([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools),
[Bittensor Tools: SDK](https://docs.learnbittensor.org/concepts/tools#bittensor-sdk)).

- **Bittensor tools** — SDK, terminal, wallet, and user-facing surfaces together.
- **Bittensor SDK** — the developer library integration surface inside that layer.

Category vocabulary and library vocabulary answer different interface questions
([Environment Variables](https://docs.learnbittensor.org/sdk/env-vars)).

## Distinction from Bittensor Platform Components

Bittensor platform components is the high-level map across subnets, chain, tokens, roles, tools, and
networks. Bittensor tools names the interface layer — SDK, terminal, wallet, and user-facing
surfaces ([Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction),
[Bittensor Tools](https://docs.learnbittensor.org/concepts/tools)).

- **Bittensor platform components** — orientation map across major system pieces.
- **Bittensor tools** — SDK, terminal, wallet, and user-facing interface layer.

Map vocabulary orients readers across layers; tools vocabulary names the interface surfaces they use
([Bittensor Tools: SDK](https://docs.learnbittensor.org/concepts/tools#bittensor-sdk)).

## Distinction from Subnet

Miners, validators, subnet creators, and developers can all use tools, but the tool does not define
the role. The Bittensor introduction frames Bittensor around subnets, miners, validators, and
digital commodities; tools are the software surfaces around that structure
([Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

Subnets still provide the work-and-evaluation context. The subnet model explains what is being
coordinated, while the tools overview explains the interface layer around that model.

## Distinction from Yuma Consensus

Bittensor tools are the CLI and SDK utilities participants use to inspect chain state, manage
wallets, and interact with subnets. They name operator tooling vocabulary, not the on-chain step
that turns validator weight submissions into emission shares
([Bittensor CLI](https://docs.learnbittensor.org/btcli),
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction)).

Yuma Consensus is the on-chain process that converts submitted validator weights into emission
shares after evaluation completes. Tools can read weights and hyperparameters, but Yuma Consensus
performs the settlement mechanism on chain
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers should keep tooling vocabulary separate from the settlement step. Bittensor tools answer how
participants query or submit actions; Yuma Consensus answers how validator weights become emission
shares in a subnet ([Bittensor CLI](https://docs.learnbittensor.org/btcli),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Bittensor tools** — CLI and SDK utilities for chain and wallet interaction.
- **Yuma Consensus** — on-chain weight aggregation that settles emissions.

## Reader Boundary

Bittensor tools should not be read as subnet roles, consensus mechanisms, or governance processes.
They name practical access vocabulary for SDK, terminal, wallet, and user-facing surfaces around the
network ([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools),
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction)).

Tools answer which software surface is being used. Subnet, consensus, and governance references
explain the network concepts those surfaces reach
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Governance](https://docs.learnbittensor.org/governance)).
