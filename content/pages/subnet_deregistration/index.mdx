# Subnet Deregistration

Subnet deregistration is the Bittensor mechanism that removes a subnet from the network to make room
for a new one. The official
[Subnet Deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration) documentation
describes it as a lifecycle process that maintains network quality by removing weaker subnets so new
subnets can register within the network's fixed subnet limit.

## Why Subnets Get Deregistered

Bittensor caps how many subnets can occupy the network at once. The deregistration documentation
states the network supports up to 128 subnet slots, so once that limit is full a new subnet cannot
register unless an existing one leaves.

Deregistration addresses that pressure by efficiency. The documentation frames it as removing
subnets that consume TAO emissions without providing value and unlocking the TAO resources locked in
those subnet pools, so capacity flows toward subnets that hold more market value.

References: [Subnet Deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)

## How a Subnet Is Selected

The process is triggered by demand for a slot. The documentation explains that when the subnet limit
is reached and a new subnet attempts to register, the network selects a subnet to remove rather than
rejecting the newcomer outright.

The selection metric is market price, not a generic performance grade. The documentation states that
the subnet chosen for deregistration is the non-immune subnet with the lowest EMA price, meaning its
exponential moving average of alpha price. Here, "underperforming" is a conceptual shorthand: the
actual selection compares EMA price across eligible subnets. The on-chain behavior behind this lives
in the Subtensor
[network source code](https://github.com/opentensor/subtensor/blob/main/pallets/subtensor/src/subnets/subnet.rs).

References: [Subnet Deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration),
[Subtensor network source code](https://github.com/opentensor/subtensor/blob/main/pallets/subtensor/src/subnets/subnet.rs)

## Immunity and Rate Limiting

New subnets are not removed immediately. The deregistration documentation describes an immunity
period, currently about four months, during which a freshly registered subnet is protected from
deregistration, giving it time to establish its EMA price.

The mechanism is also paced. The documentation notes a rate-limiting window of about two days
between registrations and deregistrations, which connects subnet lifecycle changes to the broader
[Rate Limits in Bittensor](https://docs.learnbittensor.org/learn/chain-rate-limits) model that
spaces sensitive chain actions over time.

References: [Subnet Deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration),
[Rate Limits in Bittensor](https://docs.learnbittensor.org/learn/chain-rate-limits)

## Related Concepts

Subnet deregistration should be separated from participant deregistration. Subnet deregistration
removes an entire subnet to free a network slot, while
[deregistration](https://docs.learnbittensor.org/resources/glossary#deregistration) can also refer
to removing a miner or validator from a UID slot inside a subnet.

The selection rule depends on alpha-price context, not a manual review of subnet content. The subnet
deregistration documentation describes the target as the non-immune subnet with the lowest EMA
price, while the [Alpha Price](https://docs.learnbittensor.org/resources/glossary#alpha-price)
glossary entry defines alpha price as the value of a subnet's alpha token relative to TAO. The
non-immune qualifier matters because a new subnet can be protected by an
[immunity period](https://docs.learnbittensor.org/resources/glossary#immunity-period) while it has
time to establish market history.

Rate limits and settlement context describe what happens around the lifecycle event. Rate limits
pace subnet registrations and deregistrations; they do not decide which subnet is selected. If a
subnet dissolves,
[protocol alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha) is part of the
tokenomics settlement context rather than an ordinary participant-held stake.

References: [Subnet Deregistration](https://docs.learnbittensor.org/subnets/subnet-deregistration),
[Rate Limits in Bittensor](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Glossary: Protocol Alpha](https://docs.learnbittensor.org/resources/glossary#protocol-alpha)

## Reader Boundary

Subnet deregistration should not be read as a prediction about any specific subnet, a live ranking,
or a step-by-step guide. The subnet limit, immunity period, and rate-limit values can change on
chain, so current figures belong in official references and live chain state. This article describes
the concept of removing the lowest-EMA-price subnet to free a slot.
