# Chain Reads

Chain reads describe looking at public Bittensor chain state without asking Subtensor to record a
new action. Bittensor's fee documentation treats reads as free inspection, while submitted runtime
actions can change state and involve fees
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

The term is about observation. It should be kept separate from an extrinsic or other submitted
action that asks the chain to process a change.

## Read Versus Submit

A chain read can show public state, metadata, blocks, storage values, or related records, but the
read itself does not move TAO, change stake, register a role, or alter subnet settings. The
state-changing step begins when an action is submitted to the runtime
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

This distinction is useful because many Bittensor workflows begin with inspection. A reader can use
a read to understand what is visible on-chain, then treat any later submission as a separate action.

## Inspection Surfaces

Different read targets answer different questions. Storage entries expose stored chain state, events
record runtime activity, constants name runtime values, and blocks place observations in chain order
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants)).

The
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
guide places chain reads inside public chain inspection. A read can show what a selected surface
contains, but it does not by itself explain every cause behind the displayed state.

## Fees and Network Scope

Read-only inspection is separate from submitted state changes, so fee language belongs with the
submitted action unless the point is specifically that a read is free
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics)).

A chain read also belongs to the network being inspected. Mainnet, testnet, and localnet are
separate Bittensor network contexts, so an observation from one environment should not be treated as
the chain history of another
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)).

## Interpretation Limits

Chain reads are read-only observations. They are not proof of intent, a prediction of later
submissions, or a replacement for mechanism documentation
([Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

When an article goes beyond the read itself, submitted actions, fees, emitted events, and changed
state should each keep their own context. That separation keeps a displayed value from being
mistaken for the action that produced it.
