# Weight Matrix

A weight matrix is the subnet-level collection formed from validator weight vectors. It organizes
validator-assigned weight vectors so Yuma Consensus can process them as structured consensus input
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The term separates one validator's vector from the combined input set. One vector represents one
validator's relative evaluation of miner work; the matrix is the organized collection of those
vectors for the subnet consensus process.

That collection makes the term useful when the subject is subnet-level consensus input. A single
validator vector can explain one validator's view, while the matrix shows the set of validator
vectors available for Yuma processing
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Input Collection

Each validator weight vector represents one validator's relative evaluation of miner work. The
weight matrix arranges those vectors together so Yuma Consensus can compare and aggregate validator
signals as a group
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)).

The structure preserves the difference between one validator's signal and the subnet-level
collection. A weight vector carries one evaluator's miner weights; the matrix is the organized set
of those vectors available to the subnet consensus process
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)).

This makes the matrix an input structure, not a separate scoring rule. It gives the consensus
process a common shape for reading many validator weight vectors together.

The practical reading is collection-first: a matrix belongs with the subnet and mechanism whose
validators produced the vectors.

Consensus-based weights documentation uses the broader weight language for validator-submitted
signals. The matrix term names the subnet-level collection of those validator inputs
([Consensus-Based Weights](https://docs.learnbittensor.org/concepts/consensus-based-weights)).

That collection still belongs to a selected subnet and mechanism. The same matrix shape can appear
across subnets, but the miner work and validator scoring rules give the entries their meaning
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## Consensus Processing

Yuma Consensus uses validator signals to produce subnet incentive outcomes. The weight matrix is the
organized input to that process; Yuma Consensus is the aggregation process that filters, bonds, and
turns those inputs into outcomes
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

This keeps matrix language separate from result language. The matrix is the structured input that
Yuma processing reads before rank, dividend, and incentive terms become meaningful
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Filtering and Output Terms

The weight matrix and consensus score describe different parts of the same Yuma process. The matrix
collects validator weight vectors, while consensus score is the stake-weighted benchmark used to
filter outlier weights before final rank calculations
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)).

That distinction prevents the input collection from being confused with the filtering benchmark. The
matrix supplies the set of signals; the consensus score is derived during filtering.

Rank and trust are later consensus terms. Rank names the miner-side result after aggregation, and
trust describes how much miner-side support remains after clipping
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

Validator trust is also later than the matrix. It measures validator-side influence after consensus
filtering, while the matrix names the collection of validator weight vectors before those effects
are read
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

This order keeps input and output language plain. The matrix gathers submitted validator vectors;
consensus score, rank, trust, and validator trust describe processing results.

That order also explains why matrix examples need the selected subnet. Without the task and
mechanism, the matrix only shows the input shape, not what the validator signals evaluated.

## Timing and Mechanism Scope

Tempo and Commit Reveal add timing detail around validator weights. Tempo provides the block-based
cadence for subnet epochs, while Commit Reveal can delay visibility of newly set validator weights
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo),
[Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

Those timing concepts do not replace the matrix. They explain when related validator-weight signals
are processed or revealed, while the weight matrix names the structured collection itself.

When a subnet uses multiple incentive mechanisms, each mechanism can have its own weight matrix for
independent consensus calculations. The mechanism identifies which evaluation setting the matrix
belongs to
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)).

Mechanism scope is therefore part of the term. A matrix for one mechanism is still a
validator-weight collection, but another mechanism in the same subnet can have a different matrix.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For weight matrix
on a subnet such as netuid 1, that sequence changes how readers should interpret grouped
validator-vector collections, mechanism-scoped matrix examples, and the tempo window that assembles
them.

In localnet, weight-matrix examples can be tested in an isolated environment. Local validator
vectors reflect local chain configuration rather than production subnet consensus input.

On testnet, a weight matrix can be observed in a shared, non-production network. Testnet matrix
readings on a selected netuid and incentive mechanism are separate from mainnet matrix state
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)).

On mainnet, a weight matrix describes live grouped validator-weight input for Yuma Consensus on the
production Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A weight-matrix example from one environment should not be
read as representing production consensus input on another network.

## Yuma Resolves the Matrix Into Emission Shares

[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) resolves the matrix of
validator rankings into two emissions vectors, one for miners and one for validators. The
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix) names
the grouped validator inputs; those inputs feed the on-chain step that allocates subnet
[emissions](https://docs.learnbittensor.org/resources/glossary#emission).

That keeps input and payout vocabulary separate. The matrix is structured consensus input; miner and
validator emission shares are among the outputs Yuma produces from that input.

## Pre-Consensus Filtering Shapes Matrix Rows

Not every validator row in a submitted matrix survives into Yuma Consensus unchanged. Epoch
documentation lists pre-consensus filtering steps that remove certain weights before stake-weighted
aggregation begins
([Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

Self-assigned weights are removed so validators cannot vote for themselves, with an exception for
the subnet creator when that role exists on the subnet. Weights tied to a neuron before that
neuron's latest [registration](https://docs.learnbittensor.org/resources/glossary#register) are also
dropped so stale assignments do not follow re-registered identities.

Only rows from validators holding a
[validator permit](https://docs.learnbittensor.org/resources/glossary#validator-permit) influence
consensus. Those checks limit which submitted vectors can enter the matrix Yuma actually processes.

## Kappa Clips Matrix Values Before Rank

Before rank and trust are computed, Yuma Consensus compares matrix values against a stake-weighted
[consensus score](https://docs.learnbittensor.org/resources/glossary#consensus-score) benchmark. The
hyperparameter kappa sets how much validator stake must support a weight level before that level
becomes the clipping ceiling for a miner
([Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

When a submitted weight exceeds the clipped benchmark, neither the miner nor the submitting
validator receives emissions for the excess portion. The matrix therefore supplies raw input;
clipping turns that input into the post-filter signal that later drives rank and incentive
allocation.

## Each Mechanism Can Keep Its Own Matrix

When a subnet runs
[multiple incentive mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms),
each mechanism can maintain a separate weight matrix for independent consensus calculations
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

Matrix reading therefore needs mechanism context. Two matrices on the same subnet can share a
cadence but still represent different evaluation settings rather than one combined input set.

## Bond Weights Adjust From Clipped Matrix Values

[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) documentation describes bond
weights as a blend of a validator's submitted weight and the consensus benchmark when a submission
exceeds that benchmark. Those adjusted values update validator-miner
[bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds) that later factor
into validator dividend allocation.

That step sits between raw matrix input and validator-side payout. The matrix gathers submitted
weights; clipping and bond adjustment translate those inputs into the bond terms used in validator
emissions.

## Activity Cutoff Limits Which Rows Enter

Subnet [hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) include an
activity cutoff measured in blocks within each epoch. A validator that does not submit weights
within the first activity-cutoff blocks of an epoch cannot participate until the next epoch
([Glossary: Activity Cutoff](https://docs.learnbittensor.org/resources/glossary#activity-cutoff)).

That timing rule affects which validator rows can enter the matrix for the current consensus pass.
Validators who miss the early window sit out the epoch rather than joining that round of aggregation
([Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

## Reader Boundary

Weight matrix names consensus input terminology. It describes the organized validator-vector
collection before Yuma Consensus filtering and aggregation
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The central point is that the matrix organizes validator weight vectors for consensus. Later results
depend on how Yuma Consensus filters and aggregates that organized input.

## Relationship to Weight Setting

A weight matrix and weight setting describe consensus input as collection and source action. Weight
setting is the per-validator `set_weights` action that submits one validator's weight vector, while
the weight matrix is the subnet-level collection that stacks those submitted vectors into the
structured input Yuma Consensus processes, so each weight-setting call contributes one row to the
matrix ([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Glossary: Weight Setting](https://docs.learnbittensor.org/resources/glossary#weight-setting),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Weight setting** — the action each validator uses to submit its weight vector.
- **Weight matrix** — the subnet-level stack of those vectors used as consensus input.
