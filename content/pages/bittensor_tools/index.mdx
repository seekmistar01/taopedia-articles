# Bittensor Tools

Bittensor tools are software interfaces around the network. The official tools overview groups
tooling as a way for developers, miners, validators, and other readers to interact with Bittensor
features without treating the tools as the protocol itself
([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

The central distinction is interface versus mechanism. Tools expose ways to reach Bittensor
features; protocol, subnet, and consensus documentation explain the rules those features follow.

## Interface Layer

Tooling sits around the protocol. A tool can expose subnet information, role-related surfaces, or
network-facing views, but the tool does not define the subnet task or consensus rule
([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

That boundary keeps tool vocabulary separate from mechanism vocabulary. Bittensor tools are access
surfaces around network features, not the rules that make those features meaningful.

## Tool Families

Bittensor tooling includes developer libraries, terminal interfaces, and user-facing interfaces
([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools)).

Those families differ by audience and shape. Libraries support programmatic integration, terminal
interfaces support direct interaction, and user interfaces support guided interaction. The family
distinction helps readers understand what kind of surface is being discussed.

## SDK Context

The SDK belongs to the developer-library side of the tool family. It is a programmatic surface for
software that interacts with Bittensor features
([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools),
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction)).

The SDK should be read as an integration interface. It is not a subnet, a consensus rule, or a
replacement for the mechanism documentation that defines what a network feature means.

## Terminal and User Interfaces

Terminal and user-facing tools can expose many Bittensor features through one interface
([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools)).

That does not make the interface the source of the underlying rule. A terminal surface or user
interface can help reach Bittensor functionality, while the network and subnet mechanisms determine
the meaning of the action or information being shown.

## Role and Subnet Context

Miners, validators, subnet creators, and developers can all use tools, but the tool does not define
the role. The Bittensor introduction frames Bittensor around subnets, miners, validators, and
digital commodities; tools are the software surfaces around that structure
([Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

Subnets still provide the work-and-evaluation context. The subnet model explains what is being
coordinated, while the tools overview explains the interface layer around that model.

## Network Boundary

Bittensor documentation separates mainnet, testnet, and localnet contexts
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

A tooling example from one environment should not be treated as proof about another environment.
Localnet examples can show controlled behavior, testnet examples can show shared non-production
conditions, and mainnet references need separate support from mainnet-specific material.

## What Bittensor Tools Do Not Mean

Bittensor tools do not name shortcuts around protocol rules, guarantees of role readiness, or proof
that the same result applies in every environment
([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Keeping that boundary preserves the term's narrower meaning. Bittensor tools identify interface
surfaces, while subnets, roles, consensus, wallets, and network context describe neighboring parts
of the system.
