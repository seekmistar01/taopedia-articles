# Bittensor Logging Levels

Bittensor logging levels are severity labels for diagnostic messages
([Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

They help readers distinguish detailed diagnostic output from normal status messages, successful
events, warnings, errors, and critical failures.

## Severity Ladder

The logging ladder runs from trace and debug through info, success, warning, error, and critical
([Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

The lower-detail levels help with normal status reading. More verbose levels can expose diagnostic
detail, while warning, error, and critical messages point readers toward conditions that may need
attention.

## Choosing Detail Level

The level also tells readers how much diagnostic detail they are seeing. Trace and debug messages
are useful when the reader needs more detail about what the software is doing, while info and
success messages are closer to normal progress reporting
([Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels)).

This helps keep noisy diagnostic output separate from messages that summarize expected progress or
attention-worthy problems.

## Bittensor-Specific Levels

Bittensor adds trace and success to the standard logging ladder. Trace represents fine-grained
diagnostic output; success marks an event that completed as expected
([Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels),
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction)).

These additions make logs easier to scan without changing the core idea that the level describes how
the software classified the message.

## Reading a Log Message

A log level is a clue about message importance, not a complete account of what happened. The
surrounding message, tool, and network context still matter
([Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

The useful reading order is level, then message, then context. The level says how the software
categorized the message; context explains where the message belongs.

For example, a warning in a local run and a warning while connected to a shared network can deserve
different follow-up. The severity label helps with triage, but the environment and message text tell
readers what they are actually looking at.

## Logs and Error Codes

Logging levels and Subtensor error codes answer different questions. A logging level describes
message severity or verbosity; an error code identifies a rule family or failure category from the
Subtensor layer ([Subtensor Error Codes](https://docs.learnbittensor.org/errors),
[Subtensor Standard Errors](https://docs.learnbittensor.org/errors/subtensor)).

Reading them together can be useful. A log may point attention toward a failure, while the error
code explains the Subtensor rule family involved.

## Network Context

Bittensor logging examples should stay attached to the environment where they occurred. Mainnet,
testnet, and localnet are separate network contexts, and a diagnostic message from one environment
should not be treated as proof of behavior in another
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels)).

This matters for tool output because the same logging level can appear while a tool is connected to
different environments. The level labels severity; the network context tells readers where the
message belongs.

## Practical Limits

Bittensor logging levels should be read as diagnostics vocabulary. They are not a complete chain
record, a role guide, or proof that a submitted action succeeded
([Logging](https://docs.learnbittensor.org/concepts/bt-logging-levels),
[Subtensor Error Codes](https://docs.learnbittensor.org/errors)).

Logging levels label diagnostic message severity and verbosity. When the result of a chain action
matters, readers should pair the log with the relevant error code, event, storage value, or
transaction context rather than relying on the level alone.
