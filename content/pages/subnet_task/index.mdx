# Subnet Task

A subnet task is the work target a Bittensor subnet asks miners to perform. The term sits inside the
subnet's incentive mechanism and helps explain what kind of digital commodity the subnet is trying
to produce ([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

The term belongs to subnet design vocabulary. It names the work being requested, not the protocol
for exchanging work, the scoring model for judging it, or the final consensus result.

## Work Target

The task gives a subnet a concrete work target instead of making every subnet produce the same kind
of output. Subnet documentation describes each subnet as having its own goal and standards, and the
glossary places the subnet task inside the work miners perform
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task)).

For readers, the task answers the basic question of what work is being requested inside that subnet
or mechanism.

## Miner and Validator Context

Miners are the role that performs the subnet task. Validators evaluate miner output against the
subnet's standards
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

This keeps producer and evaluator roles connected without collapsing them. A subnet task identifies
the work target; miner and validator vocabulary identifies who produces and judges work around that
target.

## Protocol Boundary

Subnet task and subnet protocol are related but separate ideas. The task describes the work target,
while the protocol describes the interaction pattern used around that work
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Subnet Protocol](https://docs.learnbittensor.org/resources/glossary#subnet-protocol)).

This boundary keeps "what miners produce" separate from "how validators and miners exchange requests
and responses."

## Scoring Boundary

The subnet scoring model is also adjacent but different. The task says what work is being requested,
while the scoring model describes how returned work is evaluated
([Glossary: Subnet Scoring Model](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

For readers, a task is not a guarantee that every response is useful. The scoring model is the
mechanism-side method for judging responses against the task.

## Incentive-Mechanism Context

A subnet task is one part of a larger incentive mechanism. The incentive design also covers the
protocol around the task and the method validators use to evaluate responses
([Glossary: Incentive Mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

That means task claims should stay scoped. A task names the requested work; the full incentive
mechanism explains how work is requested, scored, and connected to rewards.

## Multiple-Mechanism Boundary

When multiple incentive mechanisms operate inside a subnet, each mechanism can define its own task
context for miners to perform
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task)).

For readers, the work target for one mechanism should not automatically be treated as the task for
every mechanism in the subnet. The mechanism context should stay attached to the claim.

## Reader Boundary

Subnet task should be read as a concept about requested work. It is not a claim that every miner
result is useful, that every subnet uses the same evaluation standard, or that the task alone
explains final emission outcomes
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

The stable article-level point is that a subnet task defines the work miners are asked to produce
inside a subnet or mechanism, while protocol, scoring, and consensus vocabulary describe later parts
of the evaluation path.
