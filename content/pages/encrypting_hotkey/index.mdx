# Encrypting the Hotkey

Encrypting the hotkey is an optional security measure for a Bittensor hotkey. It protects the
locally stored hotkey material so that a copied keyfile is not immediately useful without the
unlocking secret.

The concept matters because a hotkey is intentionally closer to day-to-day network activity than a
coldkey. Bittensor documentation describes hotkeys as operational keys for mining, validation, and
weight commits, while coldkeys hold the financial authority over TAO and stake.

## Hotkey Context

A hotkey identifies and authorizes operational participation in Bittensor. Miners and validators use
hotkeys for routine network work, including subnet participation and weight-related activity. That
is why a hotkey may need to be available on infrastructure that is online more often than a coldkey
should be.

This does not make a hotkey unimportant. The security documentation says hotkeys have a lower direct
financial risk than coldkeys because they do not control direct movement of TAO balances. It also
says a malicious actor with hotkey control can still damage reputation, submit invalid weights as a
validator, or serve malicious responses as a miner.

## Why Encryption Matters

Encryption adds a storage boundary around hotkey material. Without that boundary, anyone who obtains
the local hotkey secret material may be able to reuse it wherever the hotkey can authorize network
activity. With encryption, copying the stored file alone is not the same as having an immediately
usable hotkey.

That protection is especially relevant because hotkey environments are operational environments.
Mining and validation systems can involve long-running services, dependencies, automation, and
remote administration. Those conditions make local key material more exposed than a coldkey kept in
cold custody.

## What Encryption Changes

Encrypting a hotkey changes how the local hotkey secret is stored and accessed. It turns the local
key material into something that requires an unlock step before use, instead of leaving the usable
secret directly available from storage.

This is a local security property. It does not create a new on-chain role, and it does not change
which coldkey owns the hotkey. It also does not change the fact that the hotkey remains the
operational identity used for miner or validator activity.

The useful distinction is between stored key material and live authority. Encryption helps protect
stored material at rest. Once the hotkey is unlocked and available to software, the running
environment still matters.

## What It Does Not Protect

Hotkey encryption is not a full defense against a compromised machine. If malware can observe the
unlock process, read memory, control the running process, or cause signed actions after the hotkey
is available, encryption of the stored file may not stop the abuse.

It also does not make a hotkey equivalent to a coldkey. The coldkey remains the security-sensitive
key for funds, stake, hotkey ownership, and other higher-risk authority. A hotkey should still be
treated as operational material whose compromise can create real subnet and reputation damage even
when direct TAO movement is not exposed.

## Encrypted Storage Versus Unlocked Runtime

Encrypting a hotkey protects the stored key material at rest, but it does not remove runtime
exposure once the hotkey is unlocked for mining, validation, or weight-related work. The
[Glossary: Encrypting the Hotkey](https://docs.learnbittensor.org/resources/glossary#encrypting-the-hotkey)
describes encryption as protecting locally stored material so a copied keyfile is not immediately
useful without the unlocking secret. After that unlock step completes, the running process and host
environment still determine whether signed actions can be abused.

Readers should treat encryption as a storage boundary, not as proof that an online validator or
miner host is safe. Operational systems may need the hotkey available for long periods, which is why
workstation security guidance still treats hotkey compromise as a serious subnet and reputation risk
even when direct TAO movement is not exposed.

## Copied Keyfiles Without the Unlock Secret

A common threat model for operational keys is offline theft of a keyfile from backups, disks, or
misplaced copies. Encryption addresses that case by requiring an unlock secret before the stored
material becomes usable signing authority. Without encryption, obtaining the file can be closer to
obtaining the hotkey itself wherever that hotkey can authorize network activity.

This protection is local and file-scoped. It does not change on-chain hotkey ownership, coldkey
control, or subnet registration state. Readers should still protect backup locations and unlock
secrets with the same care applied to other wallet recovery material described in key documentation.

## Operational Hosts and Higher Exposure

Hotkeys often live on infrastructure that stays online for mining, validation, automation, or remote
administration. Security documentation notes that hotkeys are intentionally closer to day-to-day
network activity than coldkeys and that compromised hotkeys can still damage reputation, submit
invalid weights, or serve malicious responses. Encryption adds a layer around stored hotkey files on
those hosts, but it does not replace host hardening, access control, or coldkey separation.

For readers, the useful boundary is custody location. Coldkeys should remain in higher-trust
custody, while encrypted hotkey files may exist on operational machines that must sign subnet work.
Encryption supports that split by reducing plain-text keyfile exposure without making the hotkey
equivalent to a coldkey. Unlocked services on those hosts still need monitoring because runtime
exposure remains after the storage boundary is crossed.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Encrypting
Hotkey, this sequence gives readers a boundary for interpreting hotkey-protection examples and
environment-specific handling notes.

Localnet examples are isolated and reflect local chain state, so they are useful for controlled
experiments rather than evidence of live Bittensor behavior. Testnet examples add shared
non-production conditions, which can reveal integration behavior without touching mainnet state.

On mainnet, Encrypting Hotkey examples should be read as live production hotkey-protection behavior
on the production Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, so outcomes from one environment should not be treated as
proof of behavior in another.

## Encryption Protects the Keyfile at Rest

Coldkey and hotkey workstation security separates at-rest encryption from runtime, network, and
coldkey concerns. An encrypted hotkey file is useful when the device is offline or the key is not
currently in use
([Coldkey and Hotkey Workstation Security](https://docs.learnbittensor.org/keys/coldkey-hotkey-security),
[Creating/Importing a Bittensor Wallet](https://docs.learnbittensor.org/keys/working-with-keys)).

For readers, at-rest encryption changes offline file exposure risk. It does not remove operational
signing risk once the hotkey is loaded for subnet work.

## Local At-Rest Scope

Encrypting the hotkey should be read as protection for the on-disk keyfile at rest, not as
protection for the hotkey while the key is unlocked at runtime, for the network traffic it signs, or
for the coldkey it pairs with. Coldkey and hotkey workstation security separates at-rest encryption
from runtime, network, and coldkey concerns
([Coldkey and Hotkey Workstation Security](https://docs.learnbittensor.org/keys/coldkey-hotkey-security),
[Creating/Importing a Bittensor Wallet](https://docs.learnbittensor.org/keys/working-with-keys)).

For readers, the relevant boundary is when the encryption helps. An encrypted hotkey file is useful
when the device is offline or the key is not currently in use, while a hotkey that is already loaded
in a running miner has the same operational risk profile whether the keyfile was encrypted or not. A
precise claim should keep the at-rest scope separate from runtime behavior and from the surrounding
coldkey and workstation vocabulary.

## Distinction from Mnemonic

Encrypting the hotkey locks the on-disk keyfile behind a password. A mnemonic is the recovery phrase
that can regenerate the hotkey independent of any particular keyfile
([Glossary: Mnemonic](https://docs.learnbittensor.org/resources/glossary#mnemonic),
[Glossary: Encrypting the Hotkey](https://docs.learnbittensor.org/resources/glossary#encrypting-the-hotkey)).

Encryption secures the stored file; the mnemonic is separate recovery material that can restore the
key even when the file is gone.

## Distinction from Coldkey

A coldkey is the ownership key that controls TAO balances and hotkeys. Encrypting the hotkey
protects only the operational hotkey keyfile at rest
([Glossary: Coldkey](https://docs.learnbittensor.org/resources/glossary#coldkey),
[Coldkey and Hotkey Workstation Security](https://docs.learnbittensor.org/keys/coldkey-hotkey-security)).

Custody vocabulary and hotkey at-rest protection sit on different security layers.

Wallet documentation assigns TAO balances, stake management, and high-authority ownership actions to
the coldkey, so hotkey-file encryption sits on the operational key layer rather than replacing
coldkey custody practice
([Wallets, Coldkeys and Hotkeys](https://docs.learnbittensor.org/keys/wallets),
[Glossary: Hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey)).

## Distinction from Hotkey Swap

Hotkey swap rotates subnet participation to a fresh operational hotkey. Encrypting the hotkey
protects the current hotkey keyfile while it is stored on disk
([Coldkey and Hotkey Workstation Security](https://docs.learnbittensor.org/keys/coldkey-hotkey-security)).

Rotation procedure vocabulary and at-rest file protection answer different security questions.

## Distinction from Seed Phrase Security

Seed phrase security covers practices for protecting recovery phrases. Encrypting the hotkey covers
optional password protection for the hotkey keyfile itself
([Handle your Seed Phrase/Mnemonic Securely](https://docs.learnbittensor.org/keys/handle-seed-phrase),
[Glossary: Encrypting the Hotkey](https://docs.learnbittensor.org/resources/glossary#encrypting-the-hotkey)).

Recovery-material hygiene and keyfile encryption are related but not interchangeable protections.

## Distinction from Hotkeys

A hotkey is the operational key used on online infrastructure to authorize routine work such as
registration and weight submission. Encrypting the hotkey is a password-protection step that secures
the hotkey's keyfile at rest
([Glossary: Hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey),
[Glossary: Encrypting the Hotkey](https://docs.learnbittensor.org/resources/glossary#encrypting-the-hotkey)).

The hotkey names the key and its operational role, while encryption names how that key's stored file
is protected. Encryption does not change what the hotkey can authorize once it is unlocked
([Coldkey and Hotkey Workstation Security](https://docs.learnbittensor.org/keys/coldkey-hotkey-security)).

## Distinction from NaCl Format

NaCl format names an encryption format used around legacy wallet files. Encrypting the hotkey is the
operation that applies encryption to a hotkey's key material
([Glossary: NaCl Format](https://docs.learnbittensor.org/resources/glossary#nacl-format),
[Glossary: Encrypting the Hotkey](https://docs.learnbittensor.org/resources/glossary#encrypting-the-hotkey)).

- **NaCl format** — an encryption format used for stored wallet-key files.
- **Encrypting the hotkey** — the operation that applies encryption to a hotkey.

NaCl format describes how an encrypted file is written, not whether or when a hotkey is encrypted
([Coldkey-Hotkey Security](https://docs.learnbittensor.org/keys/coldkey-hotkey-security)).

## Distinction from Coldkey-Hotkey Pair

The hotkey-coldkey pair names the ownership relationship between operational and custody keys.
Encrypting the hotkey protects only the hotkey keyfile at rest on the operational side
([Glossary: Hotkey-Coldkey Pair](https://docs.learnbittensor.org/resources/glossary#hotkey-coldkey-pair)).

Pair vocabulary names the wallet structure; encryption vocabulary names at-rest protection for one
keyfile in that structure.

An encrypted keyfile still requires the unlock secret at runtime before the hotkey can sign routine
subnet actions on connected infrastructure.

## Distinction from Coldkey Safety

Encrypting the hotkey strengthens at-rest protection for operational key material on online hosts.
Coldkey safety names the separate custody practices that keep the ownership key, balances, and stake
protected away from mining and validation infrastructure
([Coldkey and Hotkey Workstation Security](https://docs.learnbittensor.org/keys/coldkey-hotkey-security),
[Glossary: Coldkey](https://docs.learnbittensor.org/resources/glossary#coldkey),
[Glossary: Encrypting the Hotkey](https://docs.learnbittensor.org/resources/glossary#encrypting-the-hotkey)).

- **Encrypting the hotkey** — optional at-rest protection for the hotkey keyfile.
- **Coldkey safety** — custody practices that protect the ownership key and TAO balances.

## Distinction from Coldkey and Hotkey Workstation Security

Workstation security and encrypting the hotkey protect operational keys at different layers. Coldkey
and hotkey workstation security is the practice of keeping the coldkey off internet-exposed machines
while hotkeys run on operational hosts, while encrypting the hotkey is at-rest protection that locks
the hotkey keyfile behind a password on whatever host holds it
([Coldkey and Hotkey Workstation Security](https://docs.learnbittensor.org/keys/coldkey-hotkey-security),
[Glossary: Encrypting the Hotkey](https://docs.learnbittensor.org/resources/glossary#encrypting-the-hotkey)).

Encryption complements host separation rather than replacing the rule that coldkey material stays
off operational machines.

- **Workstation security** — where key material is allowed to live and run.
- **Encrypting the hotkey** — at-rest password protection for a stored hotkey keyfile.

## Distinction from Yuma Consensus

Encrypting the hotkey protects locally stored hotkey material with a password at rest. That wallet
file-protection step is separate from the on-chain pass that converts validator weight submissions
into emission shares
([Glossary: Encrypting the Hotkey](https://docs.learnbittensor.org/resources/glossary#encrypting-the-hotkey),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

A password-encrypted keyfile affects how the hotkey is stored locally. Yuma still reads submitted
weights from validators, including those signing with encrypted hotkeys, when crediting outcomes at
the epoch boundary ([Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers should keep local key security vocabulary and consensus-settlement vocabulary separate.
Encryption protects files on disk; Yuma settles evaluation signals recorded on chain
([Coldkey and Hotkey Workstation Security](https://docs.learnbittensor.org/keys/coldkey-hotkey-security)).

- **Encrypting the hotkey** — password protection for a stored hotkey keyfile at rest.
- **Yuma Consensus** — on-chain mechanism that turns validator weights into emission shares.

## Distinction from Hotkey Rotation

Encryption is preventive local protection, not a recovery mechanism. If a hotkey is suspected to
have been leaked or used by an attacker, the relevant security question becomes whether the hotkey
should be replaced and the operational environment cleaned up.

The security documentation frames hotkey compromise as something that requires a rapid mitigation
strategy. That framing is important: encryption reduces some at-rest theft risk, but it does not
prove that a suspected exposure was harmless.

## Reader Boundary

Encrypting the hotkey should not be read as coldkey protection, network encryption, or proof that a
running miner is safe while online. It names optional at-rest protection for the hotkey keyfile on
disk
([Glossary: Encrypting the Hotkey](https://docs.learnbittensor.org/resources/glossary#encrypting-the-hotkey),
[Coldkey and Hotkey Workstation Security](https://docs.learnbittensor.org/keys/coldkey-hotkey-security)).
