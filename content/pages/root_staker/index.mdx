# Root Staker

A root staker is a TAO holder who stakes on the Root Subnet through validators. Root Subnet, also
called Subnet Zero, is the subnet-agnostic staking venue in Bittensor
([Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims),
[Glossary: Root Subnet/Subnet Zero](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero)).

The term names the holder role in Root Subnet staking. It should not be confused with a validator
role, an alpha staker role, or a root-claim preference.

In plain language, root staker answers who is supplying TAO through the Root Subnet path. It does
not answer which validator should be selected or how later dividends are handled. The core reading
is holder, Root Subnet, and validator.

## Related Measures

Root proportion is nearby vocabulary because it describes how root-routed TAO appears in a subnet's
dividend context. Root staker is the role term, while root proportion is the subnet-level measure
([Glossary: Root Proportion](https://docs.learnbittensor.org/resources/glossary#root-proportion),
[Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

Keeping those terms separate helps prevent a holder role from being read as a subnet metric,
validator recommendation, or dividend outcome.

A root-proportion statement can describe a subnet's dividend split. A root-staker statement instead
identifies the holder role connected to TAO routed through Root.

## Development Stage Context

Bittensor separates localnet, testnet, and mainnet environments. Root-staker examples from one
environment should not be treated as evidence for another because validators, balances, dividends,
and root staking state are environment-specific
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Introduction to Bittensor: Subnet development](https://docs.learnbittensor.org/learn/introduction#subnet-development)).

Localnet examples can test root staking mechanics in isolation. Testnet examples add shared
non-production staking state. Mainnet root-staker interpretation concerns production Root Subnet
relationships on the active network.

## Claims Can Be Automatic or Manually Triggered

A root staker does not have to actively trigger every claim. Documentation describes automatic
claims that the chain selects each block, landing roughly once every two days per account, alongside
a manual claim path the staker can trigger directly for up to five subnets at a time
([Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

Both paths apply the same configured Keep-or-Swap handling to the claimed dividends. The staker
chooses the handling setting once; automatic and manual claims both follow it rather than requiring
a separate choice each time.

## Distinction from Alpha Staker

A root staker and an alpha staker name two staking roles split by venue and asset. A root staker
stakes TAO on the Root Subnet (Subnet Zero), the only subnet with no alpha currency of its own,
while an alpha staker holds stake denominated in a specific subnet's alpha after TAO converts
through that subnet's pool, so a root staker's position stays in TAO on Subnet Zero while an alpha
staker's position tracks one subnet's alpha
([Glossary: Root Subnet/Subnet Zero](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

- **Root staker** — a TAO holder staking on the Root Subnet (Subnet Zero).
- **Alpha staker** — a holder whose stake is denominated in one subnet's alpha.

The
[Glossary: Root Subnet/Subnet Zero](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero)
describes the root subnet as the only subnet without its own alpha currency. Root staker vocabulary
therefore names TAO staking on Subnet Zero rather than alpha-denominated stake on an alpha subnet
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

When the focus is subnet alpha held through staking, alpha staker is the more precise term.

## Distinction from Delegate

A root staker and a delegate sit on opposite sides of the staking relationship. A root staker is the
TAO holder who supplies stake on the Root Subnet, while a delegate is the validator that receives
that delegated support
([Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims),
[Glossary: Delegate](https://docs.learnbittensor.org/resources/glossary#delegate)).

- **Root staker** — the holder who supplies TAO stake on Root.
- **Delegate** — the validator that receives delegated support.

## Distinction from Root Proportion

A root staker and root proportion name a role and a measure. A root staker is the holder of a
root-routed TAO position, while root proportion is the subnet-level measure of how much of a
subnet's dividend context is attributed to that root-routed TAO
([Glossary: Root Proportion](https://docs.learnbittensor.org/resources/glossary#root-proportion),
[Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

- **Root staker** — the holder role on the root-routed side.
- **Root proportion** — the per-subnet dividend measure for root-routed TAO.

Root Subnet staking lets TAO holders stake through validators in a subnet-agnostic setting. That is
different from subnet-local alpha staking, where staking belongs to a particular subnet's
alpha-token context
([Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation),
[Emission: Distribution](https://docs.learnbittensor.org/learn/emissions#distribution)).

This asset and venue distinction is the center of the term. A root staker supplies TAO through Root,
while alpha-side staking belongs to a specific subnet's alpha context.

That distinction keeps Root Subnet staking separate from alpha-side participation in a specific
subnet. Dividend references can mention the role because Root Subnet staking can feed dividend
outcomes.

Root-routed stake can earn alpha dividends from subnets. Root claims explain how those accumulated
dividends are handled, including Keep and Swap outcomes
([Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims),
[Emission: Distribution](https://docs.learnbittensor.org/learn/emissions#distribution)).

That claim layer comes after the staking role. Root staker names the holder side of the position;
root claims describe how accumulated dividends from that position can later be handled.

This is the concise difference from root claim type: root staker identifies the holder-side staking
relationship, while root claim type identifies the Keep or Swap handling setting for processed
dividends.

## Distinction from TAO Weight

A root staker and tao weight name a holder role and a protocol-level parameter. A root staker is the
TAO holder who stakes on the Root Subnet through validators, while tao weight is the protocol-level
multiplier that sets how much root (TAO) stake counts relative to a subnet's alpha stake when stake
weight is computed
([Glossary: TAO Weight](https://docs.learnbittensor.org/resources/glossary#tao-weight),
[Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

Because tao weight scales root-routed TAO before stake weight merges alpha and TAO influence, a root
staker supplies the TAO-side input without setting the multiplier itself
([Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

- **Root staker** — the holder who stakes TAO on the Root Subnet through validators.
- **TAO weight** — the multiplier scaling how root TAO stake counts against subnet alpha when stake
  weight forms.

## Distinction from Yuma Consensus

A root staker is a TAO holder who stakes on the Root Subnet through validators. That role links
holder stake to root validators for network-wide reward participation
([Glossary: Root Staker](https://docs.learnbittensor.org/resources/glossary#root-staker),
[Glossary: Root Subnet/Subnet Zero](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero),
[Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

Yuma Consensus is the on-chain step that runs each tempo on a subnet. It turns submitted validator
weights into miner incentives and validator dividends
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Root staking is about where TAO is delegated on the Root Subnet. Yuma is about how validator weights
are turned into rewards after miners and validators are evaluated on a selected netuid.

The root staker names the holder role on Subnet Zero, while root claims name how rewards are
received. Yuma names the reward settlement step that follows validator evaluation on a subnet, not
the act of delegating to a root validator
([Glossary: Root Claims](https://docs.learnbittensor.org/resources/glossary#root-claims)).

A root staker supports root validators by delegating TAO, but that support does not replace
validator weight submissions or the consensus pass that credits emissions on a subnet
([Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions)).

Readers should not read root staking as proof of subnet performance or validator weight quality.
Those belong to subnet evaluation and Yuma outcomes rather than to root delegation alone.

- **Root staker** — stake delegated to root validators for network-wide rewards.
- **Yuma Consensus** — turn validator weights into emission shares.

## Distinction from Delegation

A root staker participates by delegating TAO to a validator on the Root Subnet. Staking and
delegation vocabulary keeps the holder and validator roles separate: the holder supplies stake
support, while the validator receives that support
([Staking and delegation overview](https://docs.learnbittensor.org/staking-and-delegation/delegation),
[Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims)).

Root staker is therefore descriptive rather than advisory. It names the TAO holder in the Root
Subnet staking relationship without choosing a validator or explaining every staking action.

Most references only need this division: root staker provides stake, and validator language names
the receiver of delegated support.

## Reader Boundary

Root staker identifies a TAO holder staking through validators on the Root Subnet
([Root Claim Overview](https://docs.learnbittensor.org/staking-and-delegation/root-claims),
[Glossary: Root Subnet/Subnet Zero](https://docs.learnbittensor.org/resources/glossary#root-subnetsubnet-zero)).

Validator language names the receiver of delegated support. Root claim language names dividend
handling. Root staker is the holder-side staking term.
