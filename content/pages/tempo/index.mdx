# Tempo

A tempo is a block-based timing interval used by Bittensor subnets. Rewards accumulate during a
tempo and are distributed at the interval boundary through
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)
([Emission](https://docs.learnbittensor.org/learn/emissions)).

As a subnet hyperparameter, `tempo` gives subnet processes a cadence measured in blocks. It is a
timing concept rather than a definition of what useful subnet work means
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Block-Based Cadence

Tempo names a cadence, not a task. A subnet's incentive mechanism defines the work being evaluated;
tempo defines the block interval over which periodic subnet processes are grouped
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

That distinction keeps timing vocabulary separate from evaluation vocabulary. Tempo helps determine
when subnet work is processed, while the incentive mechanism explains what validators are evaluating
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Block Count Versus Wall Clock

Tempo is counted in blocks, so its wall-clock length depends on the network context producing those
blocks. Bittensor network documentation separates localnet timing modes from testnet and mainnet,
including fast-block and non-fast-block local settings
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)).

For readers, that means a tempo example should keep the block count and the environment label
together. The tempo value explains how many blocks group the subnet process; the surrounding network
context explains how quickly those blocks pass in that example.

## Emission Timing Context

Bittensor emission has an accumulation-and-distribution rhythm. Rewards accumulate during a tempo
and are distributed through consensus at the interval boundary
([Emission](https://docs.learnbittensor.org/learn/emissions)).

Yuma Consensus is the aggregation step that uses validator-submitted signals to determine how those
rewards are allocated inside the subnet. Tempo supplies the cadence for that periodic process
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Hyperparameter Boundary

Tempo is a subnet hyperparameter, so it should not be read as one universal value for every subnet.
The stable concept is block-based cadence; exact values belong with the relevant subnet parameter
source ([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

This also means a tempo claim needs context. A statement about cadence should say whether it is
explaining the concept or referring to a specific subnet setting
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

## Epoch Boundary

Tempo and epoch are related but distinct timing terms. Tempo is the subnet-specific cadence for
epoch transitions, while an epoch is the consensus period produced by that cadence
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

For readers, tempo names the interval; epoch names the period in which consensus work is carried
out. A tempo should not be treated as the full consensus process
([Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch)).

## Commit Reveal Timing

Commit Reveal uses tempo-based timing for delayed weight visibility. Its reveal period is expressed
in tempos, making tempo relevant to when concealed validator weights become usable
([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal)).

This is a timing relationship, not a new definition of tempo. Tempo remains the block-based cadence;
Commit Reveal uses that cadence as part of delayed weight submission and reveal timing
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Multiple Mechanisms Scope

When a subnet has multiple incentive mechanisms, tempo still names the shared subnet cadence. The
mechanisms can have separate weight and consensus paths, but that does not require treating tempo as
a separate schedule for each mechanism
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

Multiple mechanisms clarify how evaluation is organized; tempo clarifies when periodic processing
occurs
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)).

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For tempo on a
subnet such as netuid 1, that sequence changes how readers should interpret block cadence, epoch
boundaries, and reward timing.

In localnet, tempo behavior can be tested in an isolated environment. Local examples reflect local
block production and subnet parameters rather than production timing.

On testnet, tempo can be observed in a shared, non-production network. Testnet interval boundaries
on a selected netuid are separate from mainnet emission and consensus outcomes
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

On mainnet, tempo is the production block cadence that groups subnet processes into periodic reward
and consensus timing on the connected Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A tempo example from one environment should not be read as
representing block cadence or emission timing in another environment.

## Injection Runs Every Block, Distribution Waits for Tempo

[Emission](https://docs.learnbittensor.org/learn/emissions) documentation separates injection from
distribution. Injection adds new liquidity to each subnet's pools every
[block](https://docs.learnbittensor.org/resources/glossary#block), while distribution waits until
the end of each tempo.

The [Glossary: Emission](https://docs.learnbittensor.org/resources/glossary#emission) states the
same rhythm: every block, TAO is injected into each subnet, and every tempo the accumulated value is
extracted by participants including miners, validators, stakers, and subnet creators.

That split explains why tempo reading must keep two speeds in view. Block progression continuously
feeds subnet pools, but participant-facing reward allocation is grouped and settled on the tempo
boundary through [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus).

## Miner Ranks Refresh From the Weight Matrix

The [Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo) defines tempo as the
block interval over which Yuma Consensus calculates
[emissions](https://docs.learnbittensor.org/resources/glossary#emission) from the latest ranking
[weight matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix).

Each miner's [rank](https://docs.learnbittensor.org/resources/glossary#rank) is computed from
validator miner-ratings submitted during that tempo, with outlier weights removed through consensus
clipping first. Tempo therefore marks when consensus refreshes performance scores from gathered
validator signals, not only when rewards are distributed.

## Activity Cutoff Sets the Early Epoch Window

Subnet [hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters) include an
activity cutoff measured in blocks within each
[epoch](https://docs.learnbittensor.org/resources/glossary#epoch). Because an epoch spans the block
count set by tempo, that cutoff applies inside a tempo-sized consensus period.

A [validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator) that does not
submit [validator weights](https://docs.learnbittensor.org/resources/glossary#weight-vector) within
the first activity-cutoff blocks of the epoch cannot participate in the current epoch and must wait
until the next epoch begins
([Subnet Hyperparameters: Activity Cutoff](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#activitycutoff)).

Tempo sets how long each epoch lasts; activity cutoff sets how early in that epoch a validator must
act to be included in the current consensus pass.

## Tempo Is Measured in Blocks, Which Decouples It From Clock Time

Because a tempo is a block count rather than a fixed duration, the same tempo can correspond to
different amounts of wall-clock time if block production varies. Documentation frames the tempo as
the block-based cadence on which subnet reward processing repeats, so reasoning about it in blocks
is more reliable than assuming a fixed number of minutes
([Emission](https://docs.learnbittensor.org/learn/emissions),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

For readers, this is why distribution timing is tied to epoch boundaries rather than a clock. The
validator-side and miner-side outcomes are finalized when the tempo completes in blocks, which is
the unit the protocol actually counts
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

That block framing also explains why the activity-cutoff window is expressed in blocks within the
period. Both the cadence and the early-submission window share the same block-based clock, so a
validator's timing has to be reasoned about in blocks, not seconds
([Subnet Hyperparameters: Activity Cutoff](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#activitycutoff)).

## Distinction from Epoch

Readers often swap epoch and tempo because both appear in subnet timing discussions. Tempo is the
hyperparameter that counts how many blocks make up one consensus cycle; epoch names the consensus
period in which validators submit weights and Yuma settles outcomes at the cycle boundary
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo),
[Glossary: Epoch](https://docs.learnbittensor.org/resources/glossary#epoch),
[Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

Use tempo when the question is the block count for one cycle. Use epoch when the question is the
full window from submission through boundary settlement.

## Distinction from Block

A block is one step of chain progression. Tempo counts how many blocks pass before the next subnet
emission step ([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

Blocks advance continuously; tempo groups them into the interval used for periodic distribution.

## Distinction from Emission

Emission names both injection and distribution. Injection can run every block, while distribution
from accumulated alpha waits for the tempo boundary
([Emission](https://docs.learnbittensor.org/learn/emissions),
[Emission: Injection and distribution](https://docs.learnbittensor.org/learn/emissions#injection-and-distribution-two-stages-of-the-emissions-process)).

If the reader is asking when pooled rewards are credited to participants, tempo is the timing
label—not the whole emission model.

## Distinction from Yuma Consensus

Yuma Consensus turns submitted weights into miner incentives and validator dividends at tempo
boundaries. Tempo only names the block cadence between those runs
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

Timing and aggregation are separate questions. Tempo answers when the boundary arrives; Yuma answers
what happens to weights there.

## Distinction from Weights Rate Limit and Activity Cutoff

These block-count settings are easy to confuse with tempo because all are measured in blocks:

| Term               | What it names                                                               |
| ------------------ | --------------------------------------------------------------------------- |
| Tempo              | Block interval for accumulation and distribution at the boundary            |
| Weights rate limit | Blocks a validator must wait between weight commits                         |
| Activity cutoff    | Early blocks in the epoch when weights must arrive to join the current pass |

([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Subnet Hyperparameters: Activity Cutoff](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#activitycutoff)).

Tempo sets how long the epoch lasts. Activity cutoff sets how early a validator must act inside that
epoch. Weights rate limit spaces repeated commits from the same validator.

## Reader Boundary

Tempo should be read as a block-count cadence for subnet processes. It is not a wall-clock promise,
a subnet quality signal, or a substitute for Yuma Consensus. It tells readers how timing is grouped
for rewards, epochs, and related subnet processes
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

The article-level point is stable even when specific values differ by subnet: tempo is the cadence
that helps turn block progression into periodic subnet consensus and emission timing
([Emission](https://docs.learnbittensor.org/learn/emissions)).

## Relationship to Fast Blocks

Tempo and fast blocks interact because tempo is measured in blocks, not seconds. Tempo is the fixed
number of blocks that defines a subnet's epoch interval, while fast blocks are a local-development
mode that accelerates block production, so under fast blocks the same tempo elapses in far less
wall-clock time, letting epoch-paced behavior be tested quickly
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks),
[Run a Local Bittensor Blockchain Instance](https://docs.learnbittensor.org/local-build/deploy)).

- **Tempo** — the fixed block count that defines a subnet's epoch interval.
- **Fast blocks** — the local mode that speeds block production, shortening that interval's
  wall-clock time.

## Relationship to Rate Limits

Tempo and rate limits are both block-counted timers that pace different things. Tempo is the block
interval that defines a subnet's epoch and paces consensus and emission, while a rate limit is the
block-based cooldown that spaces repeated chain actions from the same key, so tempo gates how often
the subnet settles a round and rate limits gate how often an action may repeat
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo),
[Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Tempo** — the block interval pacing each subnet epoch's consensus and emission.
- **Rate limits** — block-based cooldowns spacing repeated actions from one key.
