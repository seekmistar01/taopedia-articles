# Weight Setting

Weight setting is the validator process of assigning numerical scores to miners in a Bittensor
subnet. Validators observe miner work and submit weights that reflect how well each miner performed
the subnet task ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

Weight setting is process vocabulary for the evaluation side of subnet consensus. It does not name
the emission outcome or the miner incentive that follows from consensus.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For weight setting
on a subnet such as netuid 1, that sequence changes which subnet evaluation context a reader should
assume when interpreting validator score submissions.

In localnet, weight-setting flows can be tested in an isolated environment. Localnet validator
weights do not represent production consensus outcomes.

On testnet, weight setting can be exercised on a shared non-production network. Testnet consensus
results on a selected netuid are separate from mainnet metagraph and subnet state
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

On mainnet, weight setting concerns live validator evaluations on the selected
[netuid](https://docs.learnbittensor.org/resources/glossary#netuid) that feed Yuma Consensus.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A weight-setting example from one environment should not
be read as representing production consensus behavior in another environment.

## Distinction from Weights Version Key

The weights version key is the per-subnet hyperparameter that sets the minimum version a
`set_weights` call must carry to be accepted. Weight setting is the action that submits that vector,
so a call carrying a version below the key is rejected before weights are recorded
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

The version key gates whether a submission is accepted; weight setting is the scoring action that
must satisfy that gate.

## Distinction from Validator-Miner Bonds

Validator-miner bonds are the smoothed, time-accumulated relationships built from submitted weights.
Weight setting is the per-round action by which a validator assigns weights to miners, so each
submission updates the bonds that shape later emission outcomes
([Glossary: Validator-Miner Bonds](https://docs.learnbittensor.org/resources/glossary#validator-miner-bonds),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Weight setting is the recurring submission act; validator-miner bonds are the smoothed state those
submissions build over time.

## Distinction from Subjective Evaluation

Subjective evaluation is the challenge that subnet work often has no single correct answer, so
validators may score the same submission differently. Weight setting is the action each validator
uses to submit its own scores, so consensus over those submitted weights, not an answer key, is what
resolves the subjectivity
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

Weight setting is each validator's submission act; subjective evaluation is the reason consensus
over those submissions is needed.

## Distinction from Subnet Task

Subnet task names the work miners are asked to produce inside a subnet. Weight setting names the
validator process of assigning scores to that work before consensus runs
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Subnet task names the work miners return; weight setting names the validator scoring step applied to
that output ([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets),
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator)).

## Distinction from Subnet Scoring Model

Subnet scoring model names the subnet-defined method a validator uses to judge returned miner work.
Weight setting names the on-chain `set_weights` action that submits the scores that method produces
([Glossary: Subnet Scoring Model](https://docs.learnbittensor.org/resources/glossary#subnet-scoring-model),
[Glossary: Weight Setting](https://docs.learnbittensor.org/resources/glossary#weight-setting),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Subnet scoring model names the evaluation rule; weight setting records those results on-chain.

## Distinction from Subnet Protocol

Subnet protocol names the exchange rules for how validators request work and miners return
responses. Weight setting names the on-chain `set_weights` action that submits evaluation scores
after that returned work is judged
([Glossary: Subnet Protocol](https://docs.learnbittensor.org/resources/glossary#subnet-protocol),
[Glossary: Weight Setting](https://docs.learnbittensor.org/resources/glossary#weight-setting),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Subnet protocol describes how validators and miners interact; weight setting records evaluation
results after that interaction.

## Distinction from Weight Vector

Weight setting is the per-validator action that submits scores. A weight vector is the structured
payload that one submission records on the selected subnet
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Weight setting** — validator submission act.
- **Weight vector** — structured miner scores inside that act.

## Distinction from Weight Matrix

Weight setting produces one validator row per successful submission. A weight matrix is the
assembled collection of those rows before Yuma Consensus processes them on the selected subnet
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Implementation of the Yuma Consensus Epoch](https://docs.learnbittensor.org/navigating-subtensor/epoch)).

- **Weight setting** — per-validator submission act.
- **Weight matrix** — grouped consensus input built from many submissions.

## Distinction from Activity Cutoff

Activity cutoff names the early-epoch window in which a submission can enter the current pass.
Weight setting names the validator scoring action whose timing must satisfy that window
([Subnet Hyperparameters: Activity Cutoff](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#activitycutoff),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Activity cutoff** — eligibility timing window.
- **Weight setting** — evaluation submission that must meet that window.

## Distinction from Validator Permit

Validator permit names authorization to submit weights that can count in consensus on the selected
subnet. Weight setting names the scoring action a permitted validator performs
([Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Validator permit** — authorization to submit weights that can count.
- **Weight setting** — evaluation act a permitted validator performs.

## Distinction from Commit Reveal

Commit reveal names a staged visibility scheme for weight submissions. Weight setting names the
validator evaluation act whether or not that subnet uses commit-reveal timing
([Commit Reveal](https://docs.learnbittensor.org/concepts/commit-reveal),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

- **Commit reveal** — staged visibility for weight submissions.
- **Weight setting** — scoring action that produces a weight vector.

## Distinction from Min Allowed Weights

Min allowed weights and weight setting act on the same submission as constraint and action. Min
allowed weights is the per-subnet hyperparameter that sets the fewest weights a validator must
assign, while weight setting is the `set_weights` action that submits the vector
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

A weight-setting call that includes fewer weights than the minimum is rejected, so the constraint
shapes what a valid submission must contain without being the submission action itself.

## Distinction from Off-Chain Validation Systems

Off-chain validation systems and weight setting connect evaluation to the chain. Off-chain
validation systems are the subnet-specific layers where validators measure miner work outside the
chain, while weight setting is the on-chain `set_weights` action that submits the resulting scores
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Off-chain evaluation produces the rankings; weight setting records them on-chain. One names where
the measuring happens, the other names the submission that puts the result into consensus.

## Distinction from Dividends

Weight setting and dividends name different positions in the validator reward flow. Weight setting
is the on-chain `set_weights` action a validator submits to record validator-to-miner scores for the
epoch, while dividends are the validator-side emission share that Yuma Consensus distributes from
the subnet's emissions after consensus reads those recorded weights. Weight setting records scores;
dividends name the validator-side outcome those scores produce.

The two operate on different sides of the same epoch cycle. A validator's weight-setting submission
feeds the weight matrix that the epoch's consensus reads, and the validator's dividend share is
computed from how that validator's recorded weights survive consensus clipping. Weight setting is
the action; dividends are the validator-side reward the action eventually produces
([Glossary: Weight Setting](https://docs.learnbittensor.org/resources/glossary#weight-setting),
[Glossary: Dividends](https://docs.learnbittensor.org/resources/glossary#dividends),
[Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Distinction from Stake Weight

Weight setting and stake weight name different positions in a validator's consensus influence.
Weight setting is the on-chain `set_weights` action a validator submits to record validator-to-miner
scores for the epoch, while stake weight is the computed total stake value that determines a
validator's consensus power and emissions in a subnet. Weight setting records scores; stake weight
names the validator's standing in the consensus arithmetic that those scores feed into.

The two operate on different sides of the same validator position. A validator's recorded weights
describe how that validator scores miners, while stake weight describes how much influence that
validator carries when Yuma Consensus reads the recorded weights to produce miner incentives and
validator dividends. Weight setting is the action; stake weight is the validator's standing in the
consensus arithmetic that action feeds
([Glossary: Weight Setting](https://docs.learnbittensor.org/resources/glossary#weight-setting),
[Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

## Distinction from Yuma Consensus

Weight setting and Yuma Consensus name different positions in a subnet's consensus flow. Weight
setting is the on-chain `set_weights` action a validator submits to record validator-to-miner scores
for the epoch, while Yuma Consensus is the algorithm that aggregates the recorded weights across
validators, applies consensus clipping, and produces miner incentives and validator dividends.
Weight setting is the per-validator input; Yuma Consensus is the subnet-level aggregation that reads
the recorded weights.

The two operate at different stages of the same epoch cycle. A validator's weight-setting submission
feeds the weight matrix that the subnet's epoch reads, and Yuma Consensus runs at the epoch boundary
to clip outlier weights and distribute miner incentives and validator dividends from that matrix.
Weight setting is the action each validator takes; Yuma Consensus is the per-subnet process that
consumes the action's output
([Glossary: Weight Setting](https://docs.learnbittensor.org/resources/glossary#weight-setting),
[Glossary: Yuma Consensus](https://docs.learnbittensor.org/resources/glossary#yuma-consensus),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#validator-weights),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## Reader Boundary

Weight setting is concept vocabulary for the validator evaluation process in Bittensor subnets. The
term should not be read as a miner ranking table, a staking guide, or an emission forecast
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Actual weight values and their emission effects depend on subnet context, validator behavior, and
consensus configuration. The concept travels across subnets; observed outcomes belong to specific
environments ([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).
