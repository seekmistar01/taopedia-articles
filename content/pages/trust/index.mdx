# Trust

Trust is a Bittensor consensus metric for the miner side of evaluation. It describes how much miner
support remains after validator weights are clipped by consensus
([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The term belongs to Yuma Consensus vocabulary because it is read after validator weight signals have
been filtered and aggregated. It is not a standalone miner reputation label.

In plain language, trust answers: how much validator support did this miner retain after consensus
clipping removed outlier weight signals? A miner with high trust received broad support that
survived the consensus filtering step, not just a few validators rating it highly
([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

For a reader, trust differs from raw validator weight totals. Raw weight totals count all support
before filtering; trust reflects the miner-side support that consensus considers reliable after
comparing validator signals across the set
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Trust therefore sits after validator weights and consensus score. Validator weights provide input,
consensus score supports clipping, and trust describes the miner-side support left after that
filtering step
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

## Consensus Clipping

Validators set weight signals after evaluating miner work. Consensus filtering can reduce outlier
support before the final miner-side result is read
([Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

Trust describes the miner-side effect of that filtering. It is therefore about consensus-adjusted
support, not a separate label outside the consensus process.

Trust describes what remains of miner-side support after consensus filtering. Validator weights name
the original input signals, and emissions describe later reward flow
([Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

## Rank and Pre-Rank

Rank and trust are connected but not identical. Rank names the consensus-adjusted miner result,
while trust describes how much support is preserved between pre-rank and final rank
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

Rank says where the miner lands after aggregation. Trust says how much clipping changed the
pre-rank-to-rank relationship.

That relationship keeps the result and the comparison separate. Rank is the adjusted miner-side
outcome, while trust describes the preservation of support between the pre-clipping signal and final
rank ([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

That makes trust a comparison term inside the Yuma process. It is meaningful because pre-rank and
rank can differ after clipping.

## Validator Trust

Trust is read from the evaluated miner's side of the process. Validator trust is read from the
validator side, describing validator influence after clipping
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

Keeping the two terms separate avoids treating trust as a universal quality score for every role in
a subnet.

The distinction is role direction. Trust concerns support for an evaluated miner. Validator trust
concerns validator-side influence reflected by clipped weights
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

## Weights and Scores

Validator weights are the input signals, and consensus score is the benchmark used to clip outlier
weights. Trust is read after those filtering steps affect miner-side support
([Glossary: Validator Weights](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

This makes trust downstream of weight input language. A validator weight can contribute to the
process, but trust describes the miner-side relationship after consensus filtering.

Weight matrix terminology is also earlier than trust. The matrix collects validator weight vectors,
while trust is interpreted after the Yuma process has filtered and aggregated those signals
([Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

This order keeps input and result terms separate. Weights and matrices describe submitted evaluation
material; trust describes a miner-side result after filtering.

## Same-Subnet Comparison

Trust comparisons are most meaningful when the compared values come from the same subnet consensus
setting. Yuma Consensus operates over validator weights inside a subnet, so miner-side trust values
are clearest when read against the process that produced them
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)).

Across different subnets, the task, validator set, incentive mechanism, and weight pattern can
differ. Trust keeps the same definition, but the surrounding consensus setting can change what a
particular value is telling the reader.

A trust value is most informative when the subnet task, validator set, and consensus process remain
part of the comparison ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## Incentive Role

Trust belongs near emissions vocabulary because miner-side support, rank, and clipping all feed into
the broader Yuma Consensus path for incentives
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Trust explains a consensus relationship between pre-rank and rank, while emissions describe the
broader reward-flow setting.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For trust on a
subnet such as netuid 1, that sequence changes how readers should interpret miner-side support after
clipping, the validator set that produced the reading, and the rank outcome it compares against.

In localnet, trust examples can be exercised in an isolated environment. Local pre-rank and final
rank comparisons reflect local chain configuration rather than production consensus history.

On testnet, trust can be observed in a shared, non-production network. Testnet trust readings on a
selected netuid are separate from mainnet miner-evaluation state
([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

On mainnet, trust describes live miner-side support preservation after Yuma Consensus clipping on
the production Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A trust example from one environment should not be read as
representing production clipping behavior on another network.

## Trust Is the Pre-Rank to Rank Ratio

Inside Yuma Consensus, [trust](https://docs.learnbittensor.org/resources/glossary#trust) is the
ratio of final [rank](https://docs.learnbittensor.org/resources/glossary#rank) to pre-rank. It shows
how much of the original validator support remained after consensus clipping rather than naming the
final miner position itself ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank)).

A miner can show strong pre-rank support yet lower trust when outlier weights were removed above the
consensus benchmark. Trust therefore answers how much evaluation signal survived filtering, not
whether the miner performed useful work under the subnet's incentive mechanism.

## Consensus Score Sets the Clipping Benchmark

The [Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score)
names the stake-weighted benchmark Yuma Consensus uses when validator weights are combined into
final rank results. [Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) describes
weights that exceed that benchmark as candidates for clipping before aggregation continues.

Trust is read after that filtering step on the miner side. Consensus score names the agreement
benchmark; trust describes how much miner-side support remains once outlier validator signals have
been reduced ([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

## Excess Weights Lose Emissions for Both Roles

When a submitted weight exceeds the clipped consensus benchmark for a miner, neither the miner nor
the submitting validator receives emissions for the excess portion
([Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping)).

That penalty keeps trust tied to consensus filtering rather than to raw validator enthusiasm. A
miner can retain high raw weight totals yet lower trust when many of those signals were trimmed as
outliers relative to stake-backed agreement
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

## Validator Trust Reads the Validator Side Instead

[Validator trust](https://docs.learnbittensor.org/resources/glossary#validator-trust) sums clipped
weights a validator successfully contributed across neurons, which measures validator-side influence
after filtering rather than miner-side support preservation
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

The two terms share one Yuma pass but point in different directions. Validator trust concerns
validator-side influence after filtering; trust concerns how much miner-side support survived the
same clipping step. A strong trust reading for a miner does not automatically imply a matching
validator trust reading for any individual validator.

## Tempo Boundaries Credit Rank Before Trust Is Read

Each miner's rank is computed from validator miner-ratings submitted during a
[tempo](https://docs.learnbittensor.org/resources/glossary#tempo), with outlier weights removed
through consensus clipping first. Trust compares the pre-clip and post-clip rank relationship inside
that same pass rather than updating on every intermediate weight submission
([Emission](https://docs.learnbittensor.org/learn/emissions),
[Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo)).

## Distinction from Rank

Rank is the miner-side aggregate standing after Yuma Consensus filters and combines validator
weights. Trust is the ratio of final rank to pre-rank that shows how much support survived clipping
([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Rank vocabulary names standing after aggregation. Trust vocabulary names how much of the pre-rank
signal survived filtering.

## Distinction from Consensus Score

Consensus score is the stake-weighted benchmark used to clip outlier validator weights. Trust is
read after that benchmark has shaped miner-side support on the selected subnet
([Glossary: Consensus Score](https://docs.learnbittensor.org/resources/glossary#consensus-score),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Yuma Consensus: Clipping](https://docs.learnbittensor.org/learn/yuma-consensus#clipping)).

Consensus-score vocabulary names the agreement benchmark. Trust vocabulary names clipping impact on
miner-side support.

## Distinction from Validator Trust

Validator trust measures validator-side clipped influence across neurons. Trust measures miner-side
support preservation between pre-rank and final rank
([Glossary: Validator Trust](https://docs.learnbittensor.org/resources/glossary#validator-trust),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

The two terms share one Yuma pass but answer different reader questions on the selected subnet.

## Distinction from Weight Vector

A weight vector is one validator's submitted evaluation signals. Trust is interpreted after the full
validator set is filtered and aggregated into miner-side outcomes
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust)).

Weight-vector vocabulary belongs to input signals. Trust vocabulary belongs to miner-side clipping
impact after those signals are processed.

## Distinction from Incentives

Incentives name miner-side emission outcomes credited after rank is normalized. Trust names the
pre-rank to final-rank ratio that shows clipping impact inside the same pass
([Glossary: Incentives](https://docs.learnbittensor.org/resources/glossary#incentives),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Yuma Consensus: Miner emissions](https://docs.learnbittensor.org/learn/yuma-consensus#miner-emissions)).

Incentive vocabulary belongs to credited results. Trust vocabulary belongs to support preservation
through clipping.

## Distinction from Weight Copying

Weight copying concerns whether a validator derived its submission from independent miner evaluation
or from another validator's visible scores. Trust is a miner-side ratio read after Yuma Consensus
clips and aggregates validator weights into rank
([Weight Copying Problem](https://docs.learnbittensor.org/concepts/weight-copying-in-bittensor),
[Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Weight copying** — reuse of another validator's published scores.
- **Trust** — miner-side measure of how much pre-rank support survived clipping.

## Reader Boundary

Trust is a miner-side consensus metric after clipping. It is not rank itself, validator trust, or a
reward guarantee ([Glossary: Trust](https://docs.learnbittensor.org/resources/glossary#trust),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

Specific trust values belong to the subnet, network, and consensus run that produced them.
