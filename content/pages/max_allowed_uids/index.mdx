# Max Allowed UIDs

Max allowed UIDs is a per-subnet hyperparameter, listed in the documentation as `MaxAllowedUids`,
that sets the maximum number of neurons a subnet can hold. Each registered neuron occupies one
[UID slot](https://docs.learnbittensor.org/resources/glossary#uid-slot), so this value is the
ceiling on how many [UIDs](https://docs.learnbittensor.org/resources/glossary#uid-slot) can exist on
a subnet at once
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## What It Controls

The hyperparameter bounds capacity, not behavior. It does not change how a
[neuron](https://docs.learnbittensor.org/resources/glossary#neuron)
[registers](https://docs.learnbittensor.org/resources/glossary#register), what registration costs,
or how rewards are calculated. It only fixes how many UID slots the subnet exposes. Once the subnet
holds that many neurons, it is full, and further entry depends on displacement rather than on free
space ([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot)).

That makes max allowed UIDs a structural limit. It defines the size of the competitive set on a
subnet such as netuid 1: every miner and validator on that subnet shares the same fixed pool of
slots.

## Documented Type, Default, and Setter

The [Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)
reference lists `MaxAllowedUids` as a `u16` with a default of `256` and the description "Maximum
number of neurons on a subnet." It is changed through the `sudo_trim_to_max_allowed_uids` extrinsic,
exposed in tooling as `btcli sudo set --param sudo_trim_to_max_allowed_uids` and the
`btcli sudo trim` command, and the reference marks the permission required to set it as subnet
owner.

The setter name carries an operational caveat. Because it is a trim operation, lowering the cap
below the current neuron count removes excess UIDs rather than leaving them in place, so a reduction
is not a passive setting change. Raising or lowering the value is governed at the subnet-owner level
rather than freely tuned by individual participants
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Subnet Context

Max allowed UIDs is one of the
[subnet hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters), the
on-chain state variables that configure a single
[subnet](https://docs.learnbittensor.org/resources/glossary#subnet), for example netuid 1. Because
each subnet carries its own hyperparameters, the UID ceiling is defined per subnet and applies only
within that subnet.

The documented default of 256 is a starting point, not a guarantee for any specific subnet. The live
ceiling is chain state and can differ from the default, so the current value should be read for the
relevant netuid rather than assumed.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For max allowed
UIDs on a subnet such as netuid 1, that sequence changes how readers should interpret slot-capacity
and displacement examples.

In localnet, max-allowed-UIDs hyperparameters can be tested in an isolated environment. Local slot
ceilings reflect local chain configuration rather than production subnet participant limits.

On testnet, registration and displacement behavior can be observed in a shared non-production
network. Testnet UID ceilings and metagraph counts are separate from mainnet subnet state
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

On mainnet, max allowed UIDs is a live per-subnet hyperparameter on production subnets. Observed
slot ceilings depend on the selected subnet's on-chain hyperparameter state.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A max-allowed-UIDs example from one environment should not
be read as representing production slot limits in another environment.

## BTCLI Reads the Live Hyperparameter for a netuid

Any user can read subnet hyperparameters for a selected netuid with
`btcli subnet hyperparameters --netuid 1`. The command output includes the live `MaxAllowedUids`
value for that subnet at query time, along with other owner-set and root-set fields
([Subnet Hyperparameters: View hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#view-hyperparameters)).

That read path keeps live ceiling claims tied to a parseable netuid. A statement about max allowed
UIDs on netuid 1 should be checked with the hyperparameter table for netuid 1 rather than inferred
from the documented default alone.

## Metagraph Count Shows How Many Slots Are Occupied

The [Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph) documentation describes
the metagraph as the complete subnet state at a block, including all registered neurons. Reading
`btcli subnets metagraph --netuid 1 --network finney` returns the current participant set for that
subnet, which can be compared against the configured `MaxAllowedUids` ceiling from the
hyperparameter table for the same netuid
([Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

The hyperparameter names the ceiling; the metagraph snapshot names how many slots are occupied at
one block height. When occupied count reaches the ceiling, further registration depends on
displacement rules rather than spare capacity
([Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)).

## Behavior When the Subnet Is Full

When every UID is occupied on a subnet such as netuid 1, a new registration does not expand the
subnet. Instead it displaces the neuron with the lowest
[pruning score](https://docs.learnbittensor.org/miners#miner-deregistration), subject to protection
from the [immunity period](https://docs.learnbittensor.org/resources/glossary#immunity-period),
which shields recently registered neurons from being pruned for a configured window
([Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)).

The displaced neuron loses its slot through
[deregistration](https://docs.learnbittensor.org/resources/glossary#deregistration), and the
incoming registrant takes the freed UID. Max allowed UIDs names the ceiling that makes this
competition necessary, while the pruning score and immunity period decide which occupant is removed
when the ceiling is reached
([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot)).

## Relationship to UID Trimming

[UID trimming](https://docs.learnbittensor.org/subnets/uid-trimming) is a subnet-owner operation
that reduces the live UID set down to a chosen maximum, and max allowed UIDs is the configured
ceiling that trimming and the `sudo_trim_to_max_allowed_uids` extrinsic enforce. The hyperparameter
states the desired capacity, while trimming is how the chain removes low emitters and compacts
survivors when the owner lowers that capacity
([UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming)).

For readers, the distinction is between the number and the operation. Max allowed UIDs is the cap;
trimming is the mechanism that brings an oversized UID set into line with that cap on a subnet such
as netuid 1.

## Relationship to Max Validators

Max allowed UIDs is not the same ceiling as
[max validators](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#maxallowedvalidators).
The hyperparameters reference lists `MaxAllowedValidators` as a separate `u16`, defaulting to `64`
and set at the root level, with the description "Maximum validators on a subnet." Max allowed UIDs
bounds the total neuron population, while max validators bounds how many of those neurons can hold a
[validator permit](https://docs.learnbittensor.org/resources/glossary#validator-permit)
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

The two caps therefore answer different questions on netuid 1. One sets how many participants can
exist on the subnet at all; the other sets how many of them can act as validators within that
population.

## Relationship to Registration Allowed

Max allowed UIDs and
[registration allowed](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#networkregistrationallowed)
sit on the same registration path as ceiling and switch. Registration allowed is the Boolean that
decides whether a subnet accepts new registrations at all, while max allowed UIDs sets how many
slots exist when registration is permitted. A subnet can have registration open yet still be full,
in which case entry depends on displacement rather than spare capacity
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Registration allowed** — whether new entry is currently open.
- **Max allowed UIDs** — how many neuron slots the subnet exposes.

## Capacity Ceiling and Validator Cap Answer Different Questions

Subnet hyperparameter documentation lists `MaxAllowedUids` as the total neuron population ceiling
and `MaxAllowedValidators` as a separate limit on how many neurons may hold validator permits
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit)).

For readers, a subnet can be full of miners and non-validator neurons while still having validator
permit slots unused, or vice versa. Max allowed UIDs bounds metagraph size; max validators bounds
how many of those rows can submit validator weights
([Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph),
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator)).

That distinction keeps registration competition vocabulary separate from validator-authorization
vocabulary when a subnet approaches its occupancy limit
([Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)).

## Full Subnets Shift Entry From Spare Capacity to Displacement

When occupied UID count reaches the configured ceiling, documentation describes further registration
as competing for an existing slot rather than expanding metagraph size. The incoming registrant must
displace a low-scoring incumbent subject to immunity rules, which links max allowed UIDs vocabulary
to pruning-score ranking rather than to a simple open-door model
([Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration),
[Glossary: Deregistration](https://docs.learnbittensor.org/resources/glossary#deregistration)).

For readers, that is why the hyperparameter is a capacity bound and not a performance score. It sets
how many neurons may coexist; pruning score and immunity period decide which occupant leaves when
the bound is already met
([Glossary: Immunity Period](https://docs.learnbittensor.org/resources/glossary#immunity-period),
[Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph)).

UID trimming vocabulary names an owner operation that compacts the live set down toward a lower
ceiling, while max allowed UIDs names the configured target that trimming enforces
([UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming)).

## Registration Allowed and Capacity Ceiling Are Separate Switches

Subnet hyperparameter documentation lists `NetworkRegistrationAllowed` as the Boolean that decides
whether a subnet accepts new registrations at all, while `MaxAllowedUids` sets how many slots exist
when registration is open. A subnet can therefore have registration enabled yet still be full, in
which case entry depends on displacement rather than spare capacity
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register)).

For readers, max allowed UIDs vocabulary names the population ceiling, not the on/off switch for
entry. Registration allowed names whether new attempts are accepted; pruning score and immunity
period decide which incumbent yields a slot when the ceiling is already met
([Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration),
[Glossary: Burn Cost](https://docs.learnbittensor.org/resources/glossary#burn-cost)).

That keeps capacity planning separate from registration economics. A rising burn can make each new
identity expensive even while the subnet still has spare slots, and a full subnet can still reject
displacement when immunity protects the lowest scorer
([Understanding Neurons](https://docs.learnbittensor.org/learn/neurons)).

## Distinction from UID Slot

A UID slot is one participant position inside a subnet. Max allowed UIDs is the hyperparameter
ceiling on how many of those slots may exist at once on the selected subnet
([Glossary: UID Slot](https://docs.learnbittensor.org/resources/glossary#uid-slot),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **UID slot** — one occupied neuron position.
- **Max allowed UIDs** — total slot capacity for the subnet.

## Distinction from UID Trimming

Max allowed UIDs names the configured population ceiling. UID trimming is the subnet-owner operation
that removes low emitters and compresses survivors when the live set must be brought down to a lower
target ([UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

The hyperparameter states the desired capacity. Trimming is how the chain enforces that capacity
when the owner lowers the ceiling.

## Distinction from Pruning Score

Max allowed UIDs sets how many neurons may coexist on a subnet. Pruning score ranks which incumbent
is weakest when a full subnet must displace one participant for a new registration
([Glossary: Pruning Score](https://docs.learnbittensor.org/resources/glossary#pruning-score),
[Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)).

Capacity vocabulary names the ceiling. Pruning-score vocabulary names the ranking input used once
the ceiling is already met.

## Distinction from Max Validators

Max allowed UIDs bounds the total neuron population on a subnet. Max validators bounds how many of
those neurons may hold validator permits and submit weights
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit)).

A subnet can be full of miners while validator-permit slots remain unused, or approach its validator
cap with a smaller total population.

## Distinction from Registration Allowed

Registration allowed is the Boolean switch that decides whether a subnet accepts new registrations.
Max allowed UIDs sets how many slots exist when registration is open
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

A subnet can have registration enabled yet still be full, in which case entry depends on
displacement rather than spare capacity
([Glossary: Register](https://docs.learnbittensor.org/resources/glossary#register),
[Mining: Miner deregistration](https://docs.learnbittensor.org/miners#miner-deregistration)).

## Relationship to Mechanism Count

Max allowed UIDs and mechanism count are linked by a documented capacity constraint. Mechanism count
sets how many incentive mechanisms a subnet runs, and the implementation requires that mechanism
count multiplied by the subnet's maximum UID count must not exceed 256. Raising mechanism count can
therefore require first lowering max allowed UIDs, so the two values are planned together rather
than set independently
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

- **Max allowed UIDs** — the ceiling on registered UID slots in the subnet.
- **Mechanism count** — how many incentive mechanisms run, with their product against the UID
  ceiling capped at 256 or less.

## Reader Boundary

This page defines the concept at a high level. It does not report the live UID ceiling on any
particular subnet or how many slots are currently occupied. Those are on-chain state values that
should be checked for the relevant netuid with `btcli subnet hyperparameters --netuid 1` and the
metagraph read path for that same netuid. The documented default is 256, the type is `u16`, and the
parameter is set by the subnet owner through a trim extrinsic
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

That reading keeps max allowed UIDs separate from registration cost and reward mechanics. It bounds
how many neurons a subnet can hold; it does not predict who will hold those slots or what they will
earn.
