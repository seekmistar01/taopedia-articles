# Inspecting the Chain with Polkadot.js

Inspecting the chain with Polkadot.js means using the Polkadot.js browser app to observe Bittensor
chain data. It is a chain-inspection concept, not a wallet safety shortcut.

The official
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
page frames the app as a way to query chain state, inspect runtime metadata, browse blocks, and
submit extrinsics when signing authority is available.

## Chain Inspection Context

The official [inspection source](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
frames Polkadot.js as a way to query Bittensor chain state. Many questions can therefore be answered
without controlling the account involved in a record. Polkadot.js exposes Subtensor data such as
storage items, runtime constants, blocks, events, and metadata through a general Substrate-family
interface.

That makes the tool useful for inspection, but not identical to Taopedia, BTCLI, or the Bittensor
SDK. Taopedia explains concepts, Bittensor-specific tooling exposes Bittensor workflows, and
Polkadot.js gives a general interface for reading chain state and interacting with runtime calls.

## Network Connection Boundary

The connected network is part of the evidence. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, and the inspection source describes choosing a connected
chain endpoint. A query result only supports claims about the environment being inspected.

This boundary matters because similar storage names, constants, blocks, or events can appear in
different environments. A value observed on one network should not be treated as proof of the same
state on another network.

## Read-Only Surfaces

The inspection source describes several read-oriented surfaces: storage queries, runtime constants,
runtime metadata, runtime calls, and block browsing. These surfaces can show what the selected chain
exposes without asking Subtensor to record a new action.

The [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) reference describes
queryable storage entries, while the
[Subtensor Constants](https://docs.learnbittensor.org/subtensor-api/constants) reference describes
runtime constants. These references explain the surfaces that Polkadot.js can help inspect.

## Extrinsic Boundary

Polkadot.js can also submit extrinsics, but that crosses from inspection into action. The
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) reference lists
submitted runtime methods, and the inspection source distinguishes read-only inspection from
extrinsic submission when authority is available.

This boundary is security-sensitive. Looking at public chain state is not the same as signing a
submitted call. Articles about inspection should keep public observation separate from wallet
authority, transaction submission, and account-operation topics.

## Blocks and Events

Block browsing gives chain data ordering context. A block can show included calls, and events can
show named runtime records emitted during processing. The
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events) reference lists event names
grouped by runtime area.

This makes block and event inspection different from a simple state query. A state query can show a
value exposed at the selected context, while block and event inspection helps identify when activity
appeared and which runtime records accompanied it.

## Evidence Boundary

The
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
source supports claims about public chain inspection. It can corroborate that a record, value,
metadata item, block, or event is visible through the connected chain interface.

That evidence still has limits. Inspection does not by itself explain off-chain intent, wallet
custody practice, governance choices, or every cause behind an observed value. It is strongest when
the article claim is about chain-visible state and weakest when the claim depends on private or
off-chain context.

## Authority Boundary

Inspection should stay separate from authority. Looking at storage, constants, metadata, blocks, or
events does not require control over the account involved in those records. Submitting an extrinsic
is different because it can require signing authority and can change chain state if the runtime
accepts the action.

The [Bittensor CLI permissions guide](https://docs.learnbittensor.org/btcli/btcli-permissions) and
[wallet documentation](https://docs.learnbittensor.org/keys/wallets) provide the source context for
permission and wallet topics. Those topics should not be inferred from ordinary public chain
inspection alone.
