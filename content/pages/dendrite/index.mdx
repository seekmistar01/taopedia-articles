# Dendrite

A dendrite is the client-side part of Bittensor's Axon-dendrite communication model. The official
[Glossary](https://docs.learnbittensor.org/resources/glossary#dendrite) defines it as a client
instance that communicates with axons, and the
[Dendrite reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/dendrite/)
describes Dendrite as a network client that sends requests to axon endpoints and processes responses
from those endpoints.

## Relationship to Axons

Dendrite and axon are paired terms, but they should not be reduced to a one-way sender and receiver.
A dendrite starts the exchange with an axon endpoint and handles the response. The
[Glossary entry for Axon](https://docs.learnbittensor.org/resources/glossary#axon) describes axons
as receiving incoming synapse objects and as the advertised entry point commonly associated with a
subnet miner.

## Subnet Communication Context

The term is useful because subnet validation depends on role interaction. Subnet validators evaluate
miner work, and the Axon-dendrite model gives the communication vocabulary around that
request-and-response path. At Taopedia level, the important distinction is conceptual: dendrite is
the client-side communication component, while axon is the endpoint it contacts.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from local testing to testchain and then mainchain. For a
dendrite, that sequence changes how readers should interpret examples of subnet communication.

In local testing, a dendrite example can show the client-side request path that sends synapse
objects toward axon endpoints inside an isolated environment. That is useful for understanding the
shape of request and response logic, but it does not show public validator participation or live
miner reachability.

On testchain, dendrite behavior can be observed in a shared testing network. This gives stronger
evidence about validator-side requests, miner axon endpoints, synapse messages, and subnet protocol
rules than a private local run, while still keeping the result separate from production mainchain
activity.

On mainchain, a dendrite belongs to live subnet communication context. The
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite) defines it as a
client instance that communicates with axons, so production readings should keep the selected
subnet, validator process, contacted endpoint context, and block timing attached.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A dendrite example in one environment should not be read
as identical evidence about validator behavior or axon response quality in another environment.

This distinction matters because dendrite names the client-side communication component, not the
whole validator role or the final evaluation outcome. The network environment explains what kind of
evidence the example provides, while the subnet protocol explains how the contacted endpoint is
expected to respond.

For readers, this keeps dendrite examples from sounding stronger than their environment supports.
Local, testchain, and mainchain contexts can all contain dendrite language, but they do not carry
the same interpretive weight.

## Synapse Objects

The structured messages that dendrites carry to axon endpoints are called synapse objects. The
official [Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse) describes
synapse objects as the main vehicle for information exchange between subnet validators and subnet
miners. A dendrite carries a synapse object when it contacts an axon endpoint; the axon handles the
incoming synapse.

The [Learn: Neurons](https://docs.learnbittensor.org/learn/neurons) documentation shows that when a
dendrite queries multiple axons, it receives back a list of synapse responses — one synapse response
per axon queried. Synapse objects are therefore the shared message format used on both sides of the
exchange.

## Relationship to Multiple Mechanisms

Subnets can run more than one incentive mechanism in parallel. The
[Glossary](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms) notes
that validators must evaluate miners separately for each mechanism.

A dendrite remains the client-side component that sends requests to miner axon endpoints. When a
subnet uses multiple mechanisms, that communication role stays the same while evaluation may cover
more than one mechanism inside the subnet.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)

## Relationship to Subtensor

Dendrite and Subtensor are related but different parts of Bittensor chain vocabulary. Subtensor
names Bittensor's layer 1 blockchain and system of record, while dendrite names the client-side
component used in off-chain subnet communication before validator evaluation is recorded on that
chain layer. The [Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor)
describes the blockchain layer, and the
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite) defines dendrite
as a client instance that communicates with axons.

For readers, Subtensor names the chain layer, while dendrite names the client-side communication
component used in subnet request-and-response work.

References: [Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite)

## Relationship to Netuid

Dendrite and a netuid are related but different parts of Bittensor vocabulary. A netuid identifies a
subnet on the network, while dendrite names the client-side component validators use to contact
miner axon endpoints inside that subnet context. The
[Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid) places netuid at the
subnet level, and the [Learn: Neurons](https://docs.learnbittensor.org/learn/neurons) documentation
shows dendrites querying axons within subnet communication.

For readers, a netuid selects which subnet is in scope, while dendrite names the client-side
communication role inside that subnet.

References: [Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid),
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite)

## Relationship to Synapse

Dendrite and a synapse object are related but different parts of Bittensor subnet communication
vocabulary. A synapse object names the structured message exchanged between subnet validators and
subnet miners, while dendrite names the client-side component that carries those messages to axon
endpoints. The [Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse)
describes synapse objects as the main vehicle for information exchange, and the
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite) defines dendrite
as a client instance that communicates with axons.

For readers, a synapse object names the message format, while dendrite names the client-side
component that sends it.

References: [Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite)

## Relationship to Subnet Validator

Dendrite and a subnet validator are related but different parts of Bittensor subnet role vocabulary.
A subnet validator evaluates miner work and submits weights on chain, while dendrite names the
client-side component used to contact miner axon endpoints during that evaluation path. The
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator)
describes the validator role, and the
[Learn: Neurons](https://docs.learnbittensor.org/learn/neurons) documentation shows dendrites
querying axons in subnet communication.

For readers, a subnet validator names the evaluation role, while dendrite names the client-side
communication component used along that path.

References:
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator),
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite)

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)

participants than a private local run, while still keeping the result separate from production

[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite) defines dendrite
as a client instance that communicates with axons, so production dendrite context is tied to actual
subnet validator-to-miner communication on the live network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A dendrite example observed in one environment should not
