# Subtensor Storage

Subtensor storage entries are runtime state definitions exposed by the official
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) reference. That source
describes the page as storage query definitions for the Subtensor runtime.

## What Storage Entries Represent

The storage reference lists named entries under runtime areas. For each entry, it gives the entry
name, data type, access path, and a short summary where one is available. This makes the page a map
of where runtime state is exposed, not a full explanation of every mechanism that writes or reads
that state.

Reference: [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Runtime Areas

The official reference is grouped by runtime areas such as admin utilities, alpha assets, balances,
commitments, contracts, drand, Ethereum, EVM, proxy, registry, safe mode, system, transaction
payment, swap, and the Subtensor module. This grouping helps readers locate which part of the
runtime owns a storage entry before interpreting the entry name and summary.

Reference: [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Bittensor-Specific Context

The Subtensor module section contains entries for Bittensor-specific network state. Examples include
subnet limits, subnet ownership, subnet timing, token symbols, validator permits, validator trust,
voting-power tracking, weight commits, weights, and Yuma3 status. Other sections cover broader
runtime state such as account balances, contract state, EVM state, and system events.

Reference: [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Careful Interpretation

A storage entry definition shows that a runtime item exists and how the official reference names it.
It should not be read as a complete behavioral description of the underlying feature. Current
values, feature behavior, and subnet-specific meaning need the surrounding chain context and the
relevant mechanism documentation.

Reference: [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Relationship to Multiple Mechanisms

Subtensor storage exposes runtime entries such as weight commits and weights for Bittensor subnets.
The [Glossary](https://docs.learnbittensor.org/resources/glossary#weight-matrix) notes that each
incentive mechanism has its own weight matrix for independent consensus calculations.

For readers, a storage entry still names where chain state is recorded. Weight-related entries
should be read with per-mechanism matrix context when a subnet runs more than one incentive
mechanism.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)

## Relationship to Subtensor Events

Subtensor storage and Subtensor events address related but different views of runtime activity.
Storage entries expose where chain state is recorded, while events expose named runtime records
emitted during activity. The
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) reference lists storage
query definitions grouped by runtime area, and the
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events) reference lists event names
emitted by the runtime.

For readers, a storage entry answers where state can be queried, while an event answers what runtime
record was emitted when state changed. A weight-related storage entry and a weight-related event
name may describe the same subnet activity from different reference angles.

References: [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)

## Relationship to Extrinsics

Subtensor storage and extrinsics address related but different parts of chain interaction. An
extrinsic is a submitted state-changing runtime request, while a storage entry is a named location
where current runtime state can be read. The
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) reference lists
submitted methods, and the
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) reference lists query
definitions for existing state.

For readers, extrinsics are how software asks the runtime to change state, while storage entries are
how readers locate that state afterward. Chain reads query storage without submitting an extrinsic,
which is why the two references serve different steps in the same workflow.

References: [Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Relationship to Blocks

Subtensor storage and blocks address related but different parts of chain history. The
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block) describes a block as
containing a collection of transactions that update chain state, while storage entries name where
that updated state can be queried after processing. The
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) reference is organized by
runtime area rather than by block height.

For readers, a block records ordering and inclusion of accepted actions, while a storage entry
describes a queryable state item that reflects the outcome of runtime processing. Storage values
change as blocks are processed, but the entry definition itself is a stable reference map.

References: [Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Relationship to Network Context

Subtensor storage is read inside a specific Bittensor network environment. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet as deployment contexts, and a storage query returns state
from whichever network endpoint is being used. The same entry name can exist on each network, but
the values belong to separate chain histories.

For readers, this matters when comparing subnet hyperparameters, balances, or weight entries across
environments. A storage entry read on mainnet describes live production state, while the same entry
on testnet or localnet describes that network's own isolated history.

References: [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Relationship to Subtensor

Subtensor storage and Subtensor are related but different parts of Bittensor chain vocabulary.
Subtensor names Bittensor's layer 1 blockchain and system of record, while Subtensor storage names
the runtime storage entries exposed as query definitions within that chain layer. The
[Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor) describes the
blockchain layer, and the [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)
reference lists storage query definitions grouped by runtime area.

For readers, Subtensor names the chain layer, while Subtensor storage names the reference map of
runtime state locations on that layer.

References: [Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Relationship to Mempool

Subtensor storage and the mempool are related but different stages of Bittensor chain activity. The
mempool names pending transactions before block inclusion, while Subtensor storage names queryable
runtime state after accepted actions are processed into blocks. The
[Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool) describes the
pending-transaction context, and the
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) reference lists storage
query definitions for current runtime state.

For readers, the mempool belongs to the pre-inclusion waiting stage, while Subtensor storage belongs
to the post-processing state reference stage.

References: [Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Relationship to Netuid

Subtensor storage and a netuid are related but different parts of Bittensor vocabulary. A netuid
identifies a subnet on the network, while Subtensor storage includes runtime entries that can hold
subnet-scoped chain state tied to that identifier. The
[Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid) places netuid at the
subnet level, and the [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)
reference lists storage query definitions grouped by runtime area, including the Subtensor module
section.

For readers, a netuid selects which subnet context is in scope, while Subtensor storage names where
subnet-related runtime state can be queried on chain.

References: [Glossary: Netuid](https://docs.learnbittensor.org/resources/glossary#netuid),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Relationship to Rate Limits

Subtensor storage and rate limits are related but different parts of Bittensor chain vocabulary.
Rate limits name block-based cooldown rules that pace repeated successful chain actions, while
Subtensor storage names query definitions for runtime state that can include timing-related entries.
The [Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits) documentation describes
cooldown rules measured in blocks, and the
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) reference lists storage
query definitions grouped by runtime area.

For readers, rate limits belong to the timing rules for repeated actions, while Subtensor storage
belongs to the reference map for reading runtime state that may reflect those rules.

References: [Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)
