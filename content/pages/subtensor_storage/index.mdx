# Subtensor Storage

Subtensor storage entries are runtime state definitions exposed by the official
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) reference. That source
describes the page as storage query definitions for the Subtensor runtime.

## What Storage Entries Represent

The storage reference lists named entries under runtime areas. For each entry, it gives the entry
name, data type, access path, and a short summary where one is available. This makes the page a map
of where runtime state is exposed, not a full explanation of every mechanism that writes or reads
that state.

Reference: [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Runtime Areas

The official reference is grouped by runtime areas such as admin utilities, alpha assets, balances,
commitments, contracts, drand, Ethereum, EVM, proxy, registry, safe mode, system, transaction
payment, swap, and the Subtensor module. This grouping helps readers locate which part of the
runtime owns a storage entry before interpreting the entry name and summary.

Reference: [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Bittensor-Specific Context

The Subtensor module section contains entries for Bittensor-specific network state. Examples include
subnet limits, subnet ownership, subnet timing, token symbols, validator permits, validator trust,
voting-power tracking, weight commits, weights, and Yuma3 status. Other sections cover broader
runtime state such as account balances, contract state, EVM state, and system events.

Reference: [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Careful Interpretation

A storage entry definition shows that a runtime item exists and how the official reference names it.
It should not be read as a complete behavioral description of the underlying feature. Current
values, feature behavior, and subnet-specific meaning need the surrounding chain context and the
relevant mechanism documentation.

Reference: [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)

## Relationship to Multiple Mechanisms

Subtensor storage exposes runtime entries such as weight commits and weights for Bittensor subnets.
The [Glossary](https://docs.learnbittensor.org/resources/glossary#weight-matrix) notes that each
incentive mechanism has its own weight matrix for independent consensus calculations.

For readers, a storage entry still names where chain state is recorded. Weight-related entries
should be read with per-mechanism matrix context when a subnet runs more than one incentive
mechanism.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix)
