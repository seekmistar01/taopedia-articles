# Bittensor EVM Smart Contracts

Bittensor EVM smart contracts are Ethereum Virtual Machine-compatible contracts executed on the
Bittensor blockchain. The official
[Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials) overview describes a
full EVM runtime operating as an application layer on top of Subtensor.

## What the EVM Runtime Provides

The official overview says the EVM runtime lets users deploy most EVM smart contracts on Subtensor,
interact with deployed smart contracts on the Subtensor blockchain, and access standard Ethereum
JSON-RPC methods. At a Taopedia level, the key point is compatibility: the runtime lets
Ethereum-style smart contracts operate in the Bittensor chain context.

Reference: [Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials)

## Chain Boundary

The same source emphasizes that Bittensor EVM smart contracts execute solely on the Bittensor
blockchain, not on the Ethereum blockchain. The EVM label describes the runtime and interface model;
it does not mean that the contract is running on Ethereum mainnet.

Reference: [Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials)

## Address Formats

The official overview distinguishes Bittensor's SS58-style addresses from Ethereum-style H160
addresses. It also notes that converting between address formats does not create the corresponding
signing authority. This makes address format a compatibility topic rather than proof that one
account model has become the other.

Reference: [Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials)

## Relationship to Multiple Mechanisms

Bittensor EVM smart contracts execute on Subtensor, the chain layer that records subnet incentive
markets. The [Glossary](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism)
notes that each subnet has one or more incentive mechanisms.

For readers, EVM contract activity belongs to chain execution context. Subnet mechanism boundaries
are defined separately from the EVM runtime interface.

References:
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Glossary: Incentive Mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism)

## Careful Interpretation

Bittensor EVM support is not the same thing as using Ethereum itself. It is an EVM-compatible
runtime on the Bittensor blockchain. The official documentation includes workflow examples, but the
core concept is simply that EVM-compatible contracts can execute on Subtensor.

Reference: [Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials)

## Relationship to Hotkey Coldkey Pair

Bittensor EVM smart contracts and the hotkey coldkey pair describe different parts of how accounts
work on Bittensor. The hotkey coldkey pair is Bittensor's native signing model, where a coldkey
holds funds and a hotkey performs subnet operations, both using SS58-style addresses. The
[Glossary: Hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey) describes hotkeys as
the signing key used for operational actions on Bittensor, while the EVM runtime operates with
Ethereum-style H160 addresses in a separate address space.

For readers, this distinction matters when reading address claims. An SS58 address belongs to the
Bittensor hotkey coldkey model; an H160 address belongs to the EVM runtime layer on Subtensor. The
[Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials) documentation notes
that converting between address formats does not create the corresponding signing authority, so the
two account models remain separate even when addresses can be mapped.

References: [Glossary: Hotkey](https://docs.learnbittensor.org/resources/glossary#hotkey),
[Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials)
