# Proposal

A proposal is the governance item that the Triumvirate puts forward for Senate review. It is the
object being considered in Bittensor governance, separate from the bodies that create, review, or
execute governance actions
([Glossary: Proposal](https://docs.learnbittensor.org/resources/glossary#proposal),
[Governance Overview](https://docs.learnbittensor.org/governance)).

The term names the item in the governance path. It does not by itself say whether the item has been
approved, rejected, finalized, or applied.

## Governance Role

Bittensor governance separates proposal creation from proposal review. The Triumvirate originates
proposals, while the Senate approves or rejects them before privileged changes can proceed
([Governance Overview](https://docs.learnbittensor.org/governance),
[Glossary: Triumvirate](https://docs.learnbittensor.org/resources/glossary#triumvirate)).

That makes a proposal the item moving between governance roles. The Triumvirate is the origin body;
the Senate is the review body; the proposal is the plan or action placed in front of that review
process.

This is the main governance boundary. Bicameral legislature names the two-body structure,
Triumvirate names the origin body, Senate names the review body, and proposal names the item being
reviewed
([Glossary: Bicameral Legislature](https://docs.learnbittensor.org/resources/glossary#bicameral-legislature),
[Senate](https://docs.learnbittensor.org/governance/senate)).

The distinction also keeps proposal language separate from older sudo vocabulary. Sudo refers to
privileged administrative authority, while a proposal belongs to the governance process that
replaced single-key administration with proposal review
([Glossary: Sudo](https://docs.learnbittensor.org/resources/glossary#sudo),
[Governance Overview](https://docs.learnbittensor.org/governance#security)).

This makes proposal vocabulary useful for privileged-change review, not ordinary subnet activity.
Subnet work and scoring belong to incentive-mechanism context, while proposals belong to governance
context ([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## Bicameral Context

Bittensor governance uses a bicameral structure for proposal handling. The Triumvirate is the
proposal-origin side, while the Senate is the review and approval side
([Glossary: Bicameral Legislature](https://docs.learnbittensor.org/resources/glossary#bicameral-legislature),
[Senate](https://docs.learnbittensor.org/governance/senate)).

That structure gives the proposal a path through governance. It starts as an item created for
review, then moves through Senate voting and later governance stages when the process continues.

## Proposal Hash

A proposal hash is the identifier used to reference a proposal during voting
([Glossary: Proposal Hash](https://docs.learnbittensor.org/resources/glossary#proposal-hash)).

The distinction matters because the hash is not the same as the governance item itself. The proposal
is the item under review, while the hash is the reference used to point to that item in governance
records and voting context.

A hash also does not carry the proposal's status by itself. The same proposal can be referenced
during review, after approval, or after rejection, so the governance context around the hash still
matters ([Glossary: Proposal](https://docs.learnbittensor.org/resources/glossary#proposal),
[Senate](https://docs.learnbittensor.org/governance/senate)).

The pair is straightforward: proposal names the governance item, and proposal hash names a compact
reference to that item.

## Review Flow

The governance overview presents proposals as part of a bicameral process involving the Triumvirate
and Senate ([Governance Overview](https://docs.learnbittensor.org/governance)). A proposal can be
created, reviewed by senators, voted on, and then finalized through the governance process if it has
the required support.

The useful boundary is the stage being discussed. A submitted proposal, an approved proposal, and an
applied change are related states, but they do not mean the same thing. Proposal wording should
therefore stay attached to the governance stage that the surrounding context shows.

This stage boundary is especially important in summaries. Calling something a proposal explains what
item is under governance review; the governance record explains what happened to it.

The same stage boundary keeps approval language precise. Senate approval is a review result, while
finalization or application belongs to later governance context
([Governance Overview](https://docs.learnbittensor.org/governance)).

## Network Scope

Bittensor documentation separates localnet, testnet, and mainnet environments
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)).

Proposal examples should keep that environment boundary attached. A localnet or testnet example can
illustrate governance mechanics, but it is not evidence of a mainnet governance outcome or
production parameter change.

The concept is stable across environments, while the proposal status and impact belong to the
environment where the governance record exists.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For proposal
vocabulary, that sequence changes how readers should interpret governance proposals, Senate review,
and privileged-change examples.

In localnet, proposal examples can be exercised in an isolated environment. Local governance
configuration and proposal lifecycles reflect local chain state rather than production governance
history.

On testnet, proposals can be observed in a shared, non-production network. Testnet Senate review and
proposal outcomes are separate from mainnet governance state
([Governance Overview](https://docs.learnbittensor.org/governance)).

On mainnet, a proposal is part of the live governance path for privileged network changes on the
production Bittensor network
([Glossary: Proposal](https://docs.learnbittensor.org/resources/glossary#proposal)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A proposal example from one environment should not be read
as representing production governance outcomes in another environment.

## Distinction from Triumvirate

A proposal and the Triumvirate name the governance item and the body that originates it. A proposal
is the plan or privileged action placed in front of governance review, while the Triumvirate is the
proposal-origin body that creates it and later closes it for execution, so the Triumvirate is an
actor in the process and the proposal is the item that actor puts forward
([Glossary: Proposal](https://docs.learnbittensor.org/resources/glossary#proposal),
[Glossary: Triumvirate](https://docs.learnbittensor.org/resources/glossary#triumvirate)).

- **Proposal** — the governance item submitted for Senate review.
- **Triumvirate** — the body that originates and closes that item.

## Distinction from Senate

A proposal and the Senate name the reviewed item and the body that reviews it. A proposal is the
governance item put forward for review, while the Senate is the body of senators that votes to
approve or reject it
([Glossary: Proposal](https://docs.learnbittensor.org/resources/glossary#proposal),
[Senate](https://docs.learnbittensor.org/governance/senate)).

- **Proposal** — the governance item under review.
- **Senate** — the body that votes to approve or reject it.

## Distinction from Proposal Hash

A proposal and a proposal hash name a governance item and the reference to it. A proposal is the
item and its requested action, while a proposal hash is the compact identifier used to point at that
item during voting
([Glossary: Proposal](https://docs.learnbittensor.org/resources/glossary#proposal),
[Glossary: Proposal Hash](https://docs.learnbittensor.org/resources/glossary#proposal-hash)).

- **Proposal** — the governance item and its requested action.
- **Proposal hash** — the identifier used to reference it during voting.

## Distinction from Subtensor

Subtensor is Bittensor's layer-one blockchain and system of record. Proposals belong near Subtensor
vocabulary because accepted governance actions can affect privileged Subtensor behavior through the
governance process
([Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Governance Overview](https://docs.learnbittensor.org/governance)).

A proposal is a governance item that may request a privileged change affecting that system, not the
chain layer itself.

- **Proposal** — the governance item submitted for Senate review.
- **Subtensor** — the layer-one blockchain and system of record that proposals may affect.

## Distinction from Bicameral Legislature

Bicameral legislature names the two-body governance structure that pairs proposal creation with
proposal review. A proposal is the governance item the Triumvirate creates and the Senate votes on
([Glossary: Bicameral Legislature](https://docs.learnbittensor.org/resources/glossary#bicameral-legislature),
[Glossary: Proposal](https://docs.learnbittensor.org/resources/glossary#proposal),
[Governance Overview](https://docs.learnbittensor.org/governance)).

- **Bicameral legislature** — structural label for the two-body governance system.
- **Proposal** — the item under review in that system.

Structure vocabulary and item vocabulary answer different governance questions
([Glossary: Triumvirate](https://docs.learnbittensor.org/resources/glossary#triumvirate)).

## Distinction from Yuma Consensus

A proposal is the governance item the Triumvirate submits for Senate review. It names the object
under bicameral review, not the on-chain step that turns validator weight submissions into emission
shares ([Glossary: Proposal](https://docs.learnbittensor.org/resources/glossary#proposal),
[Governance Overview](https://docs.learnbittensor.org/governance)).

Yuma Consensus is the on-chain process that converts submitted validator weights into emission
shares after evaluation completes on a subnet. Governance proposals can authorize privileged network
changes, but Yuma Consensus names subnet emission settlement, not Senate review of a proposal item
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers should keep governance-item vocabulary separate from the settlement step. Proposal answers
what item is under review; Yuma Consensus answers how validator weights become emission shares in a
subnet ([Glossary: Proposal](https://docs.learnbittensor.org/resources/glossary#proposal),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Proposal** — governance item submitted for Senate review.
- **Yuma Consensus** — on-chain weight aggregation that settles emissions.

## Reader Boundary

A proposal is the governance item submitted into review
([Glossary: Proposal](https://docs.learnbittensor.org/resources/glossary#proposal),
[Governance Overview](https://docs.learnbittensor.org/governance)).

When the focus is the voting body, Senate is the more precise term. When the focus is the origin
body, Triumvirate is more precise. When the focus is the identifier used during voting, proposal
hash is more precise.

When the focus is a completed network effect, proposal alone is also too broad. The governance
record must show the relevant review state before a proposal should be described as approved,
rejected, or applied ([Governance Overview](https://docs.learnbittensor.org/governance),
[Senate](https://docs.learnbittensor.org/governance/senate)).

## Senators Cast Approval or Disapproval Votes

The [Senate](https://docs.learnbittensor.org/governance/senate) voting documentation describes
senators choosing approval or disapproval after entering a proposal hash obtained from the proposals
overview. Proposal status reading should therefore name the vote type senators record, not only that
a proposal exists.

Public viewing and senator voting remain separate paths in that reference. Anyone can inspect
proposals currently before the Senate, while senator votes decide whether the item meets the
approval threshold described in governance material.

## Governance Examples Can Request New Subnets

The [Governance Overview](https://docs.learnbittensor.org/governance) example shows a Triumvirate
member creating a proposal whose calldata calls a privileged subnet-creation extrinsic. Proposal
vocabulary therefore covers governance items that can request network-level changes such as adding a
subnet, not only hyperparameter edits.

Readers should pair that example with Senate review context. The proposal names the reviewed item;
the calldata names the privileged action senators are deciding whether to authorize.

## Compromise Requires Triumvirate and Senate Control

Governance security material states that approving privileged actions under the governance protocol
requires compromising a Triumvirate member and controlling a Senate majority, rather than a single
`sudo` key alone ([Governance Overview](https://docs.learnbittensor.org/governance#security)).

That threat model keeps proposal reading tied to bicameral review. A proposal is the reviewed
container; Senate majority and Triumvirate close remain separate conditions before execution
([Glossary: Bicameral Legislature](https://docs.learnbittensor.org/resources/glossary#bicameral-legislature)).
