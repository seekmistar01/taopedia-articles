# Bittensor Tools

Bittensor tools are software interfaces around the network. The
[Bittensor tools](https://docs.learnbittensor.org/concepts/tools) overview groups tooling as a way
for developers, miners, validators, and other readers to interact with Bittensor features without
treating the tools as the protocol itself.

The central distinction is interface versus mechanism. Tools expose ways to reach Bittensor
features; protocol, subnet, and consensus documentation explain the rules those features follow
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## Interface Layer

Tooling sits around the protocol. A tool can expose subnet information, role-related workflows, or
network-facing surfaces, but the tool does not define the subnet task or consensus rule
([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

That boundary keeps tool articles from becoming operating manuals. The useful article-level point is
that Bittensor tools are access surfaces around network features.

## Tool Families

Bittensor tooling includes developer libraries, terminal interfaces, and user-facing interfaces
([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools)). Those families differ by
audience and shape: libraries support programmatic integration, terminal interfaces support direct
operator-facing use, and user interfaces support guided interaction.

The family distinction helps readers choose the right article context. A library article should be
read as software-integration vocabulary, while a terminal or user-interface article should be read
as an interaction-surface topic.

## SDK Context

The SDK belongs to the developer-library side of the tool family. It is a programmatic surface for
software that interacts with Bittensor features
([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools),
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction)).

For readers, the SDK should be read as an integration interface. It is not a subnet, a consensus
rule, or a replacement for the mechanism documentation that defines what a network feature means.

## Terminal Interface Context

Terminal tooling belongs to the direct-interaction side of the tool family
([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools)). It can expose many Bittensor
features through one interface, but those exposed features still belong to their underlying protocol
or subnet context.

That means a terminal surface should not be treated as the rule that validates a feature. It is an
interface for reaching Bittensor functionality; the network and subnet mechanisms determine the
meaning of the underlying action.

## Network Context

Tools can show information from different network environments. Bittensor documentation separates
mainnet, testnet, and localnet, so tool output should be interpreted with the connected environment
attached ([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

This does not make the tool family environment-specific. It means the same kind of interface can
point at different network contexts, and the context affects how the displayed information should be
read.

## Relationship to Roles and Subnets

Miners, validators, subnet creators, and developers can all use tools, but the tool does not define
the role. The [Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction) frames
Bittensor around subnets, miners, validators, and digital commodities; tools are the software
surfaces around that structure.

Subnets still provide the work-and-evaluation context. The
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) reference is
the better source for the subnet model, while the tools article explains the interface layer around
that model.

## Reader Boundary

Bittensor tools should be read as interface vocabulary
([Bittensor Tools](https://docs.learnbittensor.org/concepts/tools)). They are not shortcuts around
protocol rules, guarantees of readiness for a role, or proof that the same result applies in every
environment.

This page defines the interface layer around Bittensor features rather than tool-specific procedures
or troubleshooting transcripts. Detailed usage belongs in the relevant official tool reference.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For Bittensor
Tools, that sequence changes how readers should interpret tooling examples, network selection, and
development workflows.

In localnet, Bittensor Tools examples can be exercised in an isolated environment. Localnet
observations reflect local chain state and local configuration rather than production Bittensor
behavior.

On testnet, Bittensor Tools behavior can be observed in a shared, non-production network. Testnet
examples are useful for checking interactions under shared chain conditions while staying separate
from mainnet state.

On mainnet, Bittensor Tools describes live tool behavior against live chain state on the production
Bittensor network.

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. An example or outcome from one environment should not be
read as evidence of production behavior in another environment.
