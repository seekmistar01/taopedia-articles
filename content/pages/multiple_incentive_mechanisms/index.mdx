# Multiple Incentive Mechanisms

Multiple incentive mechanisms describe the case where one Bittensor subnet contains more than one
separate evaluation mechanism. Each mechanism has its own evaluation system and bond pool for Yuma
Consensus calculations
([Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

In plain language, multiple incentive mechanisms answers: can one subnet run more than one
independent evaluation track? The answer is yes — each track has its own evaluation system and bond
pool, so a miner's performance in one mechanism does not automatically carry into another
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

For a reader, this distinction matters most when reading weight or rating data from a
multi-mechanism subnet. A score from mechanism A is not a score from mechanism B, even though both
mechanisms share the same subnet market and UID space
([Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)).

The term is narrower than incentive mechanism by itself. An incentive mechanism connects work,
evaluation, and rewards; multiple incentive mechanisms name the structure where several such systems
coexist inside one subnet
([Glossary: Incentive Mechanism](https://docs.learnbittensor.org/resources/glossary#incentive-mechanism),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

## Subnet Container

Multiple mechanisms still live inside one subnet. The subnet provides the shared market setting,
while each mechanism provides a separate evaluation frame within that subnet
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

This keeps the term from meaning multiple subnets. The phrase points to several evaluation systems
inside one subnet, not to a collection of independent subnet markets.

That container distinction matters for claims about work quality and rewards. A result belongs to
the mechanism that produced it, while the subnet remains the shared market around those mechanisms
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

That also separates multiple mechanisms from ordinary subnet comparison. The comparison is inside
one subnet, between mechanisms, rather than between independent subnet markets
([Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets)).

## Evaluation Separation

The core idea is separation by mechanism. Documentation describes validators evaluating miners
separately for each mechanism, so an evaluation result belongs to the mechanism that produced it
([Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

That separation matters when reading ratings, weights, or work quality. A result from one mechanism
does not automatically describe another mechanism just because both mechanisms belong to the same
subnet.

## Bond Pool Context

The glossary ties multiple mechanisms to separate bond pools. Yuma Consensus uses validator weight
signals when calculating miner incentives and validator dividends, and the separate bond-pool frame
keeps those calculations divided by mechanism
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)).

The bond-pool detail is structural. It explains why the mechanism split can affect how evaluation
signals are interpreted before rewards are assigned.

## Yuma Consensus

Yuma Consensus is the aggregation process that turns validator weight information into miner-side
and validator-side outcomes. Multiple incentive mechanisms do not replace that consensus process;
they separate the inputs and bond-pool frame used for each mechanism
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

The mechanism tells which evaluation frame is in use; Yuma Consensus describes the aggregation that
follows from the signals in that frame.

This keeps the shared consensus step visible without flattening the mechanism-specific inputs.

## Emission Flow

Emission documentation gives the broader reward flow for Bittensor, while the multiple-mechanism
references explain why evaluation and bond-pool frames can be separated inside a subnet
([Emission](https://docs.learnbittensor.org/learn/emissions),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

That makes emission claims more precise. The multiple-mechanism term supports structural claims
about separated evaluation systems, while emission documentation explains how reward flows are
allocated through the wider tokenomics process.

## Validators Maintain Separate Weight Vectors Per Mechanism

The [Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)
notes that when multiple mechanisms are used, validators maintain separate weight vectors for each
one. Validators evaluate miners separately for each mechanism, so an evaluation result belongs to
the mechanism that produced it
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

A score from one mechanism does not automatically describe another mechanism just because both
mechanisms belong to the same subnet.

## Separate Bond Pools Keep Emission Paths Divided

Because the bond pools are separate, the miner-incentive and validator-dividend amounts Yuma
Consensus derives from each mechanism's validator weights stay divided by mechanism rather than
pooled together, which is what keeps one mechanism's emission path distinct from another's inside
the same subnet ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms)).

Emission documentation explains the broader reward flow, while multiple-mechanism vocabulary
explains why evaluation and bond-pool frames can stay separated inside one subnet
([Emission](https://docs.learnbittensor.org/learn/emissions)).

## Parallel Yuma Paths Inside netuid 1

On a subnet such as netuid 1, multiple incentive mechanisms mean validators submit separate
[weight](https://docs.learnbittensor.org/resources/glossary#weights) vectors and bond-pool inputs
for each mechanism rather than one merged evaluation track
([Glossary: Weight Vector](https://docs.learnbittensor.org/resources/glossary#weight-vector)).

[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus) then runs that aggregation
separately per mechanism, producing distinct miner
[incentives](https://docs.learnbittensor.org/resources/glossary#incentives) and validator
[dividends](https://docs.learnbittensor.org/resources/glossary#dividends) for each evaluation frame
inside the same netuid 1 market
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

Multiple incentive mechanisms therefore name subnet-internal structure on netuid 1; Yuma names the
consensus step that turns each mechanism's signals into its own emission outcome rather than one
shared payout pool.

## Validators Must Validate Every Mechanism or Lose Emissions

Running more than one mechanism does not give a validator a choice of which ones to evaluate without
consequence. Official documentation states that a validator's dividends are calculated as a weighted
sum of that validator's performance across all of a subnet's mechanisms, and that a validator who
does not validate on every mechanism receives proportionally reduced emissions rather than simply
forgoing the share tied to the skipped mechanism alone
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

That rule keeps multi-mechanism participation from becoming optional in practice. A validator's
stake weight stays identical across every mechanism on the subnet, so skipping evaluation work on
one mechanism still costs dividends calculated against the full weighted sum, not against a reduced
set of mechanisms the validator chose to cover.

The operational load follows from the same separation. A validator must set weights independently
for each mechanism and apply that mechanism's own evaluation criteria, since the mechanisms are
documented as requiring separate assessment rather than one shared scoring pass
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

## Validators and Miners Participate Automatically, Not by Opt-In

Multiple mechanisms do not create a separate enrollment step for either role. Documentation
describes every validator on a subnet as participating in every one of that subnet's mechanisms with
identical stake weight, and every miner registered on the subnet as eligible to participate in any
of its mechanisms without a separate registration step
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

A miner also keeps the same UID across every mechanism on the subnet rather than receiving a
distinct identity per mechanism. Performance still stays independent per mechanism, so a miner's
standing in one mechanism does not change its standing in another, but the underlying registration
and identity are shared
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

For a staker, this automatic participation means delegated stake applies across all of a validator's
mechanisms equally rather than being split or chosen per mechanism. Stake delegated to a validator
on a multi-mechanism subnet follows that validator's combined performance, not a single mechanism
selected in advance.

## Multiple Mechanisms Redistribute Emissions Rather Than Multiply Them

Adding mechanisms to a subnet does not change how much the subnet receives in total. Official
documentation states that a subnet's total emissions remain unchanged when it runs multiple
mechanisms; the mechanisms only redistribute that same total internally among participating miners
and validators
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

That framing keeps multiple incentive mechanisms from being read as an emissions-increasing feature.
The subnet-level emission amount is set by the broader emission process described elsewhere; running
several mechanisms changes how that fixed amount is divided among evaluation tracks, not its size.

## Mechanism Count Is Bounded by Available UID Slots

A subnet cannot add mechanisms without limit. The documented constraint is that the number of
mechanisms multiplied by the subnet's maximum UID count must stay below a fixed ceiling, since each
mechanism needs its own slot accounting within that shared UID space
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets),
[UID Trimming](https://docs.learnbittensor.org/subnets/uid-trimming)).

That constraint ties mechanism count directly to UID capacity rather than treating the two as
unrelated settings. Raising the number of mechanisms a subnet runs requires enough UID headroom to
satisfy the same inequality, which is why mechanism count and UID trimming belong to the same
capacity-planning picture rather than separate topics.

## Weights Become a Two-Dimensional Structure

Weight setting itself changes shape once a subnet has more than one mechanism. Each validator sets
weights for each miner on each mechanism separately, so the weight data for a multi-mechanism subnet
is two-dimensional: one axis for miners and one axis for mechanisms, rather than the single weight
vector a one-mechanism subnet uses
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

The metagraph reflects that same structure. Documentation describes the metagraph gaining separate
columns for mechanism weights and mechanism-level incentives once a subnet runs more than one
mechanism, so a reader inspecting metagraph data for such a subnet should expect per-mechanism
columns rather than one combined weight and incentive column.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For multiple
mechanisms on a subnet such as netuid 1, that sequence changes how readers should weigh separated
evaluation and bond-pool examples
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

In localnet, multiple mechanisms on netuid 1 can be demonstrated in isolation. Local mechanism
separation reflects local configuration rather than production subnet structure.

On testnet, multiple-mechanism behavior on netuid 1 can be observed in a shared non-production
network. Testnet mechanism weights and bond pools are separate from mainnet subnet state
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

On mainnet, multiple incentive mechanisms on netuid 1 are live subnet-internal evaluation structure
on the production Bittensor network, with Yuma settling each mechanism's incentives separately.

Examples of multiple mechanisms keep the environment attached because local testing, testnet
behavior, and mainnet behavior provide different kinds of evidence.

## Reader Boundary

Multiple incentive mechanisms are subnet-internal evaluation structure. The term does not by itself
say how many mechanisms any particular subnet has, which mechanism is preferable, or how rewards
were distributed in a specific case
([Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

The stable reference point is separation: several evaluation mechanisms can exist inside one subnet,
with their own evaluation systems and bond pools.

## Relationship to Subnet Task

Multiple incentive mechanisms and subnet task are related but different subnet-design terms.
Multiple incentive mechanisms names the case where one subnet runs parallel evaluation tracks with
separate weight matrices and bond pools, while subnet task names the work target inside a subnet
([Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms),
[Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task)).

On a multi-mechanism subnet, each mechanism identifies its own evaluation track and work target, so
a single subnet-task statement does not describe every mechanism; the mechanism label specifies
which track and task are meant
([Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

## Relationship to Rank

Multiple incentive mechanisms and rank connect a subnet's split evaluation to separate standings.
Multiple incentive mechanisms is the case where one subnet runs more than one evaluation mechanism,
each with its own bond pool, while rank is the consensus standing produced per mechanism, so a
subnet with several mechanisms computes a separate rank within each rather than one combined
ordering ([Glossary: Rank](https://docs.learnbittensor.org/resources/glossary#rank),
[Glossary: Multiple Incentive Mechanisms](https://docs.learnbittensor.org/resources/glossary#multiple-incentive-mechanisms),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Multiple incentive mechanisms** — several evaluation mechanisms inside one subnet.
- **Rank** — the consensus standing computed separately within each mechanism.
