# Bittensor EVM Smart Contracts

Bittensor EVM smart contracts are Ethereum Virtual Machine-compatible contracts that execute on
Subtensor. The [Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials)
material places the EVM runtime as an application layer for Bittensor rather than as Ethereum
mainnet execution.

## Runtime Compatibility

The EVM runtime gives Bittensor an Ethereum-compatible contract environment. Compatibility means
many Ethereum-style contract patterns can be represented in the runtime, while execution still
belongs to Bittensor's Subtensor context. The
[EVM smart-contract overview](https://docs.learnbittensor.org/evm-tutorials) frames this as EVM
support on top of Subtensor.

## Subtensor Boundary

Subtensor remains the base layer for the activity. The
[Subtensor glossary entry](https://docs.learnbittensor.org/resources/glossary#subtensor) names
Subtensor as Bittensor's layer 1 blockchain and system of record, and Bittensor EVM contract
execution sits within that Bittensor environment. The EVM label names the execution model, not a
move onto Ethereum itself.

## Interface Context

Ethereum JSON-RPC compatibility gives familiar interface expectations for EVM-style interaction.
That interface layer should be read separately from the protocol boundary: using an Ethereum-shaped
interface does not make a contract an Ethereum-mainnet contract. The
[EVM overview](https://docs.learnbittensor.org/evm-tutorials) places Ethereum JSON-RPC access inside
Bittensor's EVM support.

## Address Format Context

Bittensor EVM material distinguishes SS58-style Bittensor addresses from Ethereum-style H160
addresses. The
[H160 address glossary entry](https://docs.learnbittensor.org/resources/glossary#h160-address) keeps
the Ethereum-format address concept separate from Bittensor address context, and the EVM overview
treats conversion between address formats as a compatibility concern rather than proof of shared
signing authority.

## Smart Contracts and Subnets

EVM smart contracts are not the same thing as subnet incentive mechanisms. Subnets define markets
for useful work, while EVM contracts are application-layer programs running in the Bittensor
environment. The
[Understanding Subnets](https://docs.learnbittensor.org/subnets/understanding-subnets) material
gives the miner-validator market frame that remains separate from EVM contract execution.

## Reading the Term

The term is best read as EVM-compatible execution on Bittensor. It does not mean Ethereum custody,
Ethereum mainnet settlement, or a subnet scoring model. The
[Bittensor EVM Smart Contracts](https://docs.learnbittensor.org/evm-tutorials) overview is the
anchor for runtime and interface claims, while Subtensor and subnet references explain the
surrounding Bittensor context.
