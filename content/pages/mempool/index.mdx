# Mempool

In Bittensor context, a mempool is the temporary holding area for pending transactions before block
inclusion ([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool),
[Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

The term belongs to pre-inclusion transaction vocabulary. It should be read separately from final
chain history, emitted events, and storage-state evidence.

## Pending Transaction Context

The official glossary places the mempool between transaction submission and block inclusion
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)).

That means mempool evidence is temporary. It can support a claim that a transaction was pending in a
selected context, but it does not prove final inclusion or the later state produced by that
transaction.

## Block Inclusion Boundary

A block is a chain data unit that contains transactions and has a block hash
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block)).

This makes mempool and block evidence complementary. The mempool describes the waiting area before
inclusion, while a block record gives inclusion and ordering in chain history.

## Extrinsics and Fees

Submitted extrinsics can pass through the mempool before block inclusion. The Subtensor extrinsics
reference lists submitted runtime methods, while the fees documentation explains fee accounting for
state-changing actions
([Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics),
[Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees)).

The mempool does not define the extrinsic method or the fee model. It names the pending-transaction
stage before the chain records the transaction in a block.

## Events and State Evidence

After inclusion, related evidence can come from emitted events and storage state. The Subtensor
events reference lists runtime event names, and the storage reference lists queryable state entries
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Those surfaces answer different questions. Mempool evidence concerns pending status; events and
storage help interpret what happened after processing.

## Network Context

Bittensor documentation separates mainnet, testnet, and localnet contexts
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

A mempool observation should therefore keep the selected network attached. A pending transaction in
one environment is not evidence about pending or included transactions in another environment.

## Visibility and Shielding

Mempool visibility concerns what can be observed while a transaction is pending. The MEV Shield
documentation describes a protection mechanism related to sensitive transaction details before block
inclusion ([MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield/)).

That makes visibility and shielding related to mempool context, but they are not the same term. The
mempool names the pending area; shielding describes a protection feature around pre-inclusion
visibility.

## Reader Boundary

The mempool glossary entry supports a narrow definition: a temporary holding area for pending
transactions before block inclusion
([Glossary: Mempool](https://docs.learnbittensor.org/resources/glossary#mempool)).

That boundary keeps the article focused. Mempool language should not be used as proof of final chain
history, broad network behavior, or the full cause of a transaction outcome.

## Pending-State Boundary

Mempool language should be kept separate from finalized chain state. Transaction and block
references place submitted activity into chain processing, while MEV Shield documentation focuses on
protecting sensitive pending transaction details before inclusion
([Transaction Fees in Bittensor](https://docs.learnbittensor.org/learn/fees),
[MEV Shield](https://docs.learnbittensor.org/concepts/mev-shield/)).

For readers, a pending transaction is not the same as a completed on-chain result. Claims about
mempool visibility should avoid implying final execution, final fee outcome, or finalized state
until block inclusion is established.
