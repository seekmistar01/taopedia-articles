# Proposal Hash

A proposal hash is the compact identifier used to refer to a proposal in Bittensor governance. It
gives Senate review and voting a stable reference for the proposal under consideration
([Glossary: Proposal Hash](https://docs.learnbittensor.org/resources/glossary#proposal-hash),
[Governance Overview](https://docs.learnbittensor.org/governance)).

The hash is useful because governance discussions often need to point at the same item across
review, voting, and later references. The proposal carries the requested action; the hash keeps
references to that proposal compact.

## Senate Review

Bittensor governance separates proposal creation from Senate approval. Proposal review and voting
sit inside Senate review, so proposal hashes belong naturally beside Senate review terms
([Governance Overview](https://docs.learnbittensor.org/governance),
[Senate](https://docs.learnbittensor.org/governance/senate)).

During review, the same proposal can appear in proposal descriptions, Senate discussions, and later
governance references. The hash keeps those references aligned with the same item without restating
the whole proposal each time.

Senate review still supplies the governance action. A proposal hash identifies the item under
review, while the Senate source explains the approval process and vote result
([Senate](https://docs.learnbittensor.org/governance/senate)).

## Voting Records

Senate voting centers on the proposal being reviewed. A proposal hash identifies the proposal being
referenced, while the surrounding Senate record explains the vote, review state, and later outcome
([Senate](https://docs.learnbittensor.org/governance/senate),
[Governance Overview](https://docs.learnbittensor.org/governance)).

That separation matters in governance summaries. The hash can tell readers which proposal is in
view, but the source using the hash must supply the decision, review state, or applied-change
details.

When a hash appears with little surrounding explanation, the narrow reading is usually best: it
identifies which proposal is in view, and the neighboring governance material explains why that
proposal matters.

## Governance Relationships

Proposal hash belongs near proposal, Senate, senator, and Triumvirate language. Proposal names the
governance item, Triumvirate names the proposal-origin body, Senate names the review body, and
senators are the delegates who vote during Senate review
([Governance Overview](https://docs.learnbittensor.org/governance),
[Senate](https://docs.learnbittensor.org/governance/senate)).

The hash is not a governance actor. It is the reference attached to the item moving through those
relationships
([Glossary: Proposal Hash](https://docs.learnbittensor.org/resources/glossary#proposal-hash)).

## Distinction from Proposal

A proposal hash and a proposal name different parts of governance records. A proposal is the
governance item under review, while a proposal hash is the compact identifier senators use when
looking up and voting on that item
([Glossary: Proposal Hash](https://docs.learnbittensor.org/resources/glossary#proposal-hash),
[Glossary: Proposal](https://docs.learnbittensor.org/resources/glossary#proposal)).

- **Proposal** — the governance change item being reviewed.
- **Proposal hash** — the identifier used to reference that item in voting.

A proposal hash points to the governance item; it does not contain the full proposal text, vote
reasoning, or later outcome
([Glossary: Proposal Hash](https://docs.learnbittensor.org/resources/glossary#proposal-hash),
[Governance Overview](https://docs.learnbittensor.org/governance)).

## Distinction from Senate

A proposal hash and the Senate answer different governance questions. The Senate is the review body
that approves or disapproves proposals, while a proposal hash is the lookup key senators supply when
casting a vote on a specific item
([Glossary: Proposal Hash](https://docs.learnbittensor.org/resources/glossary#proposal-hash),
[Glossary: Senate](https://docs.learnbittensor.org/resources/glossary#senate),
[Senate](https://docs.learnbittensor.org/governance/senate)).

- **Senate** — the governance body that records approval or disapproval.
- **Proposal hash** — the reference senators use to select which proposal they vote on.

## Distinction from Nominator

A proposal hash and a nominator name different parts of Bittensor. A proposal hash is the compact
identifier used to refer to a proposal during Senate review and voting, while a nominator is an
account that stakes TAO on a validator's hotkey to earn dividends while retaining control of its
tokens ([Glossary: Nominator](https://docs.learnbittensor.org/resources/glossary#nominator)).

- **Proposal hash** — the governance reference used to look up and vote on a proposal.
- **Nominator** — the account that stakes TAO on a validator's hotkey to earn dividends.

## Distinction from Coldkey

A proposal hash and a coldkey identify different things in Bittensor. A proposal hash is the
governance reference that points to a specific proposal under Senate review, while a coldkey is the
wallet component responsible for securely storing funds and performing high-risk operations such as
transfers and staking
([Glossary: Coldkey](https://docs.learnbittensor.org/resources/glossary#coldkey)).

- **Proposal hash** — the identifier that points to a governance proposal during review and voting.
- **Coldkey** — the wallet component that securely stores funds and authorizes high-risk operations.

## Distinction from Yuma Consensus

A proposal hash is the identifier senators use when voting on a Triumvirate-originated governance
item. It names governance reference vocabulary, not the on-chain step that turns validator weight
submissions into emission shares
([Glossary: Proposal Hash](https://docs.learnbittensor.org/resources/glossary#proposal-hash),
[Governance Overview](https://docs.learnbittensor.org/governance)).

Yuma Consensus is the on-chain process that converts submitted validator weights into emission
shares after evaluation completes on a subnet. Governance votes reference proposal hashes, but Yuma
Consensus names emission settlement rather than governance identifiers
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers should keep proposal-identifier vocabulary separate from the settlement step. Proposal hash
answers which governance item is under vote; Yuma Consensus answers how validator weights become
emission shares in a subnet
([Glossary: Proposal Hash](https://docs.learnbittensor.org/resources/glossary#proposal-hash),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Proposal hash** — identifier senators use when voting on a proposal.
- **Yuma Consensus** — on-chain weight aggregation that settles emissions.

## Reader Boundary

Proposal hash is proposal-identifier terminology. It is the compact governance reference used during
proposal review and voting, while the proposal and governance record carry the requested change,
review state, and outcome
([Glossary: Proposal Hash](https://docs.learnbittensor.org/resources/glossary#proposal-hash),
[Glossary: Proposal](https://docs.learnbittensor.org/resources/glossary#proposal),
[Senate](https://docs.learnbittensor.org/governance/senate)).

## Senators Look Up the Hash Before Casting a Vote

The [Senate](https://docs.learnbittensor.org/governance/senate) voting process has senators look up
a proposal's hash from the proposals overview, then supply that hash to cast an approval or
disapproval. The vote is counted once it is included in the next block.

That sequence makes the hash an active step in voting, not only a passive label. A senator cannot
cast a vote without first identifying the correct proposal hash to vote on
([Senate](https://docs.learnbittensor.org/governance/senate)).

## Triumvirate Originates, Senate Reviews

Because Bittensor governance separates proposal creation from approval, the Triumvirate originates a
proposal and the Senate then reviews and votes on it. The proposal hash is the single reference that
points to that same item across both stages, so origination-side and review-side governance records
can be tied to one proposal ([Governance Overview](https://docs.learnbittensor.org/governance),
[Glossary: Triumvirate](https://docs.learnbittensor.org/resources/glossary#triumvirate)).

## Development Stage Context

Bittensor separates mainnet, testnet, and localnet environments. Governance examples from one
environment belong to that environment because proposal records and vote outcomes can differ
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks),
[Introduction to Bittensor: Subnet development](https://docs.learnbittensor.org/learn/introduction#subnet-development)).
