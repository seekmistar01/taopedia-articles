# Synapse

A synapse is the structured message object used in Bittensor subnet communication
([Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[Synapse reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/synapse/)).

The term belongs to communication vocabulary. It is separate from the subnet role, subnet task,
scoring rule, and reward outcome that may interpret information carried by a message.

## Message Object Role

Bittensor documentation describes synapse objects as the main vehicle for information exchange
between subnet validators and subnet miners
([Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[Synapse reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/synapse/)).

That makes synapse a message-object term. It names what travels through the subnet communication
path, not the entire miner role, validator role, scoring model, or emission result.

## Axon and Dendrite Path

Synapse sits between axon and dendrite vocabulary. Axon is the server-side endpoint that receives
incoming synapse objects, while dendrite is the client-side component that sends requests to axon
endpoints and processes responses
([Axon reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/),
[Dendrite reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/dendrite/)).

In that path, dendrite contacts an axon and the synapse is the structured object exchanged through
the request and response flow. The three terms describe neighboring parts of communication rather
than a complete account of subnet behavior.

## Miner and Validator Exchange

Synapse language often appears near subnet miners and subnet validators because those roles exchange
task-related information. Miner and validator remain role concepts, while synapse names the object
that can carry material between them
([Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner),
[Glossary: Subnet Validator](https://docs.learnbittensor.org/resources/glossary#subnet-validator)).

The synapse itself does not decide whether a response is useful. A subnet's incentive mechanism
explains how exchanged information is evaluated.

## Task-Specific Content

The synapse concept is shared communication vocabulary, but the useful content inside a message
depends on the subnet task. Bittensor documentation describes subnet tasks as the work miners
perform for a subnet, and multiple-mechanism documentation notes that different mechanisms can
define different task contexts inside a subnet
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Multiple Incentive Mechanisms Within Subnets](https://docs.learnbittensor.org/subnets/understanding-multiple-mech-subnets)).

That boundary prevents synapse from becoming a universal description of every subnet payload. The
term identifies the communication object; task-specific message content belongs to the relevant
subnet API reference.

## Incentive Mechanism Handoff

Synapse exchange can feed a subnet's evaluation pipeline. The incentive-mechanism documentation
places miner work, validator evaluation, and emission outcomes inside subnet-specific mechanisms
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

That does not make synapse a reward rule. Synapse names the object carrying task information before
the subnet's own evaluation logic turns responses into weights, scores, or other outcome signals.

## Subnet Creators Define Synapse Fields

Incentive-mechanism documentation states that a
[subnet creator](https://docs.learnbittensor.org/resources/glossary#subnet-creator) must define how
validators query miners and how miners respond. Those query rules are built using the Axon-Dendrite
client-server model and [Synapse](https://docs.learnbittensor.org/resources/glossary#synapse) data
objects
([Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

The creator-defined query rules therefore decide what belongs inside the synapse object for that
subnet. Synapse naming identifies the shared message type, while the subnet task specifies which
challenge fields a validator sends and which response fields a miner is expected to fill.

## Challenge and Response Share One Object

[Understanding Neurons](https://docs.learnbittensor.org/learn/neurons#neuron-to-neuron-communication)
walks through a prompting example in plain terms. A validator initializes a synapse with fixed
challenge fields, the miner updates a mutable response field, and the validator later reads that
filled field back. The round trip stays inside one structured object rather than an open-ended
message format.

That pattern keeps synapse vocabulary tied to a single exchange cycle. The validator-side dendrite
client sends the initialized object, the miner-side axon server receives it, and the returned
synapse carries the miner's additions back to the validator for evaluation.

## Miner Serving Completes the Query Cycle

Miner serving vocabulary describes how a miner exposes an axon endpoint that accepts synapse traffic
from validator dendrites. The synapse carries the task payload; the axon endpoint is the serving
surface that receives it
([Glossary: Subnet Miner](https://docs.learnbittensor.org/resources/glossary#subnet-miner),
[Axon reference](https://docs.learnbittensor.org/python-api/html/autoapi/bittensor/core/axon/)).

Within the incentive query pipeline, a validator dendrite sends a challenge synapse, the miner axon
processes it, and the returned synapse supplies material for validator-side scoring. Serving and
synapse vocabulary sit on the communication layer before weights or emissions are decided.

## On-Chain Weights Are Separate From Synapse Exchange

Synapse exchange happens through subnet communication endpoints, while
[validator weights](https://docs.learnbittensor.org/resources/glossary#weight-vector) are submitted
to the chain as consensus inputs after evaluation
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

A synapse can carry task output that a validator later scores, but the synapse itself is not a
weight vector, an extrinsic, or a persistent chain record
([Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events)). Weight setting and
synapse transport therefore sit on different layers: communication objects move work between roles,
while on-chain weight submission feeds the subnet consensus path.

## Synapse and Dendrite Layers

Synapse and dendrite are related but different subnet communication terms. Synapse names the
structured message object passed between validators and miners, while dendrite names the client
component that sends and receives those messages during a subnet task
([Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite)).

For readers, synapse is the message-object term and dendrite is the communication-component term.
They often appear together in one exchange example, but one describes what is carried and the other
describes which client part carries it
([Understanding Neurons](https://docs.learnbittensor.org/learn/neurons#neuron-to-neuron-communication)).

A synapse definition says which challenge and response fields belong in the object. A dendrite
reference describes how that object is sent toward a miner axon and how the reply returns to the
validator for evaluation.

Readers should not treat a synapse type name as proof of dendrite behavior, or a dendrite mention as
a substitute for the synapse schema that defines the message contents.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For synapse on a
subnet such as netuid 1, that sequence changes how readers should interpret validator-to-miner
exchange examples.

In localnet, synapse wiring can be tested in isolation on a subnet such as netuid 1. Local examples
show message shape but do not represent live public endpoint behavior.

On testnet, synapse exchange can be observed in a shared non-production network. Testnet
validator-to-miner communication is separate from mainnet endpoint behavior even when the same
netuid number appears in both environments
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

On mainnet, synapse behavior concerns production validator-to-miner communication on the active
network.

A synapse example from one environment should not be treated as proof about endpoint behavior in
another.

## Distinction from Axon

Synapse names the structured message object exchanged in subnet communication. Axon names the
server-side endpoint that receives incoming synapse objects
([Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[Glossary: Axon](https://docs.learnbittensor.org/resources/glossary#axon)).

Message-object vocabulary and endpoint vocabulary answer different questions on the same exchange.

## Distinction from Dendrite

Synapse names what is carried through subnet communication. Dendrite names the client-side component
that sends and receives those objects during the exchange
([Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[Glossary: Dendrite](https://docs.learnbittensor.org/resources/glossary#dendrite),
[Understanding Neurons](https://docs.learnbittensor.org/learn/neurons#neuron-to-neuron-communication)).

Payload vocabulary and client-component vocabulary describe neighboring layers of the same path.

## Distinction from Neuron

A neuron is the UID-identified subnet participant. A synapse is the structured message object those
participants exchange across the axon-dendrite path
([Glossary: Neuron](https://docs.learnbittensor.org/resources/glossary#neuron),
[Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse)).

Participant identity and message-object vocabulary are separate labels in the same exchange.

## Distinction from Subnet Task

Subnet task names the work a mechanism is meant to evaluate. Synapse names the structured object
that can carry challenge and response fields for that exchange
([Glossary: Subnet Task](https://docs.learnbittensor.org/resources/glossary#subnet-task),
[Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Task vocabulary describes what work means; synapse vocabulary describes the message shape used to
move that work between roles.

## Distinction from Subnet Protocol

Subnet protocol names the rule set for how validators and miners exchange work inside a subnet.
Synapse names the structured message object that carries that exchange in Bittensor client code
([Glossary: Subnet Protocol](https://docs.learnbittensor.org/resources/glossary#subnet-protocol),
[Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[Understanding Incentive Mechanisms](https://docs.learnbittensor.org/learn/anatomy-of-incentive-mechanism)).

Exchange-rule vocabulary and message-object vocabulary answer different questions on the same
miner-validator path.

## Distinction from Replay Attack

Synapse is the structured message object carried through dendrite-to-axon communication, while a
replay attack is the reuse of a valid signed message when freshness checks fail
([Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[Understanding Neurons](https://docs.learnbittensor.org/learn/neurons#neuron-to-neuron-communication)).

Message-object vocabulary and message-reuse threat vocabulary answer different questions. Synapse
names the structured payload exchanged between neurons; replay attack names reuse of a previously
valid signed message.

- **Synapse** — the structured message object in dendrite-to-axon communication.
- **Replay attack** — reuse of a valid signed message when freshness checks fail.

## Reader Boundary

Synapse does not name a miner role, a validator decision, a persistent chain record, a wallet
operation, or a complete subnet task design
([Glossary: Synapse](https://docs.learnbittensor.org/resources/glossary#synapse),
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

Synapse identifies what is exchanged, while axon, dendrite, subnet task, and incentive mechanism
references describe neighboring parts of the system.
