# Subtensor Storage

Subtensor storage entries are runtime state definitions exposed by the official
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) reference. That source
describes the page as storage query definitions for the Subtensor runtime.

## What Storage Entries Represent

The [storage reference](https://docs.learnbittensor.org/subtensor-api/storage) lists named entries
under runtime areas. Each entry can include a name, type, access path, and short summary. Together,
those fields make the page a map of where runtime state is exposed, rather than a walkthrough of
every mechanism that changes that state.

## Runtime Areas

The official [Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage) page groups
entries by runtime area. That organization matters because the same kind of state label can carry
different meaning depending on the module that owns it. A storage item should first be read through
its owning area, then through its entry name and type.

## Bittensor-Specific Context

The [Subtensor module section](https://docs.learnbittensor.org/subtensor-api/storage) contains many
Bittensor-specific entries. These include records related to subnets, validators, weights, permits,
timing, ownership, token symbols, and incentive-mechanism state. Other runtime areas expose broader
chain state, so the Subtensor module is the part of the reference most directly tied to Bittensor
network behavior.

## Reading Storage Entry Definitions

The [entry definitions](https://docs.learnbittensor.org/subtensor-api/storage) should not be read as
standalone phrases. The runtime area identifies ownership, the entry name identifies the state item,
and the type describes the shape of the value exposed by the runtime. The access path then shows
where the item sits in the reference structure.

This structure helps avoid over-reading a storage name. A short entry summary may identify the state
being exposed, but it does not necessarily explain the full business rule, incentive rule, or chain
history that produced a value.

The storage reference is therefore strongest as a naming and location source. If a page needs the
official runtime name for a state item, the storage reference can provide the owning area, entry
name, type, and access path
([Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)). If a page needs to
explain why the value changes, it needs the mechanism or concept documentation that governs that
state.

This distinction is especially useful for Bittensor-specific entries. A subnet-related storage name
can sit near staking, weights, permits, or incentive vocabulary without replacing the article that
explains those concepts. The storage entry identifies where state is exposed; the concept article
explains what the state means for readers.

## Entry Definitions and Queried Values

The [storage reference](https://docs.learnbittensor.org/subtensor-api/storage) describes where state
is exposed. It is not the same thing as a snapshot of every value on every network. A definition can
show that a runtime item exists and how the reference names it, while a query result belongs to the
chain state being inspected.

The distinction is important when using storage as evidence. The reference gives the map; the
queried value gives one observed result within a network and block context. The Bittensor
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain)
guide provides the surrounding context for chain-state inspection.

A queried value also needs a selected network and inspection moment. The
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet, while the inspection guide describes reading chain state
from a connected endpoint. The same entry name can be used across environments, but an observed
value belongs to the environment and chain history where it was read.

For Taopedia, this keeps storage evidence from becoming too broad. Storage can support a claim that
a value exists in runtime state, or that a value was observed in a stated context. It cannot by
itself prove every action, motive, or off-chain process that produced that state.

## Block and Network Scope

Storage values are scoped by chain history. The
[Inspecting the Chain with Polkadot.js](https://docs.learnbittensor.org/concepts/inspecting-the-chain),
documentation describes inspecting chain state, while the
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) documentation
distinguishes mainnet, testnet, and local network environments. A storage value should therefore be
read with both block context and network context when a claim depends on an observed state.

The same storage entry name can be meaningful across environments, but returned values belong to
separate chain histories. Shared vocabulary does not make values from different networks
interchangeable.

## Storage as Evidence

Subtensor storage can support claims about chain-visible state, but it does not explain every cause
behind that state. A storage read may show that a runtime item is queryable or that a value is
recorded at a given point in chain history. It does not by itself prove off-chain intent, design
motivation, custody practice, or every action that led to the value.

Events and submitted calls provide different evidence than storage. The
[Subtensor Events](https://docs.learnbittensor.org/subtensor-api/events) reference lists emitted
runtime records, and the
[Subtensor extrinsics](https://docs.learnbittensor.org/subtensor-api/extrinsics) reference lists
state-changing calls. Storage is strongest as evidence for the state exposed by the runtime, while
events and extrinsics help explain how runtime activity was recorded.

That separation gives storage a clear role in article evidence. Storage is useful when the claim is
about a value the runtime exposes for inspection. Events are useful when the claim is about a named
record emitted during processing. Extrinsics are useful when the claim is about a callable action
submitted to the runtime. These references can describe the same feature from different angles, but
they should not be collapsed into a single evidence type.

For example, a feature may have a storage entry that records state, an event that announces a
processed change, and an extrinsic that can request a state change. The storage entry alone does not
show the full lifecycle. It shows the state surface that can be queried after the relevant runtime
activity has occurred.
