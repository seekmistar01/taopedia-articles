# Max Validators

Max validators is a per-subnet hyperparameter that sets the maximum number of validators a subnet
allows. Documentation describes it as the maximum validators on a subnet, making it a cap on how
many neurons can act as validators there at once
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## What It Controls

The hyperparameter bounds the size of a subnet's validator set. A subnet can hold many registered
neurons, but only up to this maximum may serve as validators at the same time. It does not decide
which specific neurons validate; it only sets the ceiling on how many can
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

## Default and Setting

The documentation lists a default of 64 and marks the parameter as set at the root level rather than
by the subnet owner. The value actually in force is per-subnet chain state and can differ from the
documented default, so the validator cap on netuid 1 can differ from another subnet's ceiling
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

Validator status on a subnet is granted through validator permits awarded to higher-stake neurons.
The max-validators cap bounds how many permits can exist at once
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit)).

## Registration Count Can Exceed the Validator Cap

A subnet can register more neurons than its max-validators setting allows. The hyperparameter caps
how many of those registered participants may act as validators at the same time, not how many UIDs
the subnet can hold overall
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

That split keeps slot limits separate from validation rights. Many neurons may remain registered
while only the permitted subset submits validator weights under the cap.

## Metagraph Exposes validator_permit Flags

The [Subnet Metagraph](https://docs.learnbittensor.org/subnets/metagraph) documentation lists
`validator_permit` as a boolean array showing whether each neuron has validator permits to set
weights and participate in consensus. Reading permit state for netuid 1 therefore uses metagraph
snapshot fields rather than the hyperparameter name alone.

The boolean array describes chain-recorded permit status at one block height. The max-validators
hyperparameter describes the configured ceiling those flags cannot exceed.

## Distinction from Validator Permit

Max validators caps how many validator permits a subnet can issue at once. Validator permit is the
on-chain flag that records whether a UID may submit non-self weights that count in consensus
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit)).

- **Max validators** — configured ceiling on validator permits for a subnet.
- **Validator permit** — on-chain flag recording whether a UID may submit consensus weights.

## Distinction from Max Allowed UIDs

Max validators bounds how many neurons may hold validator permits, while max allowed UIDs bounds the
total neuron population a subnet exposes
([Subnet Hyperparameters: MaxAllowedValidators](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#maxallowedvalidators),
[Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

- **Max allowed UIDs** — ceiling on how many neurons the subnet can hold.
- **Max validators** — ceiling on how many of those neurons may act as validators.

## Distinction from Stake Weight

Max validators is the hyperparameter ceiling on how many validator permits a subnet can issue. Stake
weight is the computed influence measure used to rank validators for those permit slots
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight)).

- **Max validators** — configured cap on validator permits.
- **Stake weight** — computed influence that ranks who receives permits.

Validator permits are awarded to the top K neurons by stake weight, and max validators names that K
ceiling: stake-weight ranking fills the available permit slots up to the configured maximum rather
than picking specific UIDs
([Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit),
[Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight)).

## Chain Reads for netuid 1

Readers can verify live hyperparameter values for the documented example netuid with
`btcli subnet hyperparameters --netuid 1 --network finney`
([Subnet Hyperparameters: View hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters#view-hyperparameters)).

That read path keeps live hyperparameter claims tied to a parseable netuid.

## Per-Subnet Live Value Boundary

Max validators is per-subnet chain state. The validator cap configured on one netuid bounds how many
permit flags can be true on that subnet, but the live ceiling can differ from reference defaults and
from the cap on another netuid
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).

This article's infobox uses netuid 1 as an example label when reading one subnet's hyperparameter
table on Finney mainnet. That example helps readers verify one row; it is not proof that every
subnet exposes the same live validator cap
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Max validators governs validator-permit ceiling; Yuma Consensus still converts permitted validator
weights into emission shares each epoch on the active subnet
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Max validators** — validator-permit ceiling on a subnet.
- **Per-subnet boundary** — live value is chain state on one netuid.

## Distinction from Yuma Consensus

Max validators is the per-subnet hyperparameter that caps how many neurons may hold validator
permits and submit weights that can enter consensus. It names a participation ceiling, not the
mechanism that credits emission shares at the epoch boundary
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters),
[Glossary: Max Validators](https://docs.learnbittensor.org/resources/glossary#max-validators)).

Yuma Consensus is the on-chain mechanism that reads the eligible weight matrix from permitted
validators, applies clipping and bonding, and then credits miner incentives and validator dividends
on the selected netuid ([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Glossary: Weight Matrix](https://docs.learnbittensor.org/resources/glossary#weight-matrix),
[Emission: Distribution](https://docs.learnbittensor.org/learn/emissions#distribution)).

The stake-weight section above covers how permit slots are filled up to that cap. Max-validators
vocabulary names the ceiling; Yuma vocabulary names settlement from permitted submissions
([Glossary: Validator Permit](https://docs.learnbittensor.org/resources/glossary#validator-permit),
[Glossary: Stake Weight](https://docs.learnbittensor.org/resources/glossary#stake-weight)).

Readers should not read max validators as a payout quote or rank label. It limits who may submit
counting weights; outcomes still depend on the matrix Yuma processes on the relevant netuid
([Emission](https://docs.learnbittensor.org/learn/emissions)).

- **Max validators** — configured ceiling on validator permits for a subnet.
- **Yuma Consensus** — on-chain mechanism that uses validator weights to determine emission shares
  ([Yuma Consensus: Validator emissions](https://docs.learnbittensor.org/learn/yuma-consensus#validator-emissions)).

## Reader Boundary

This page defines the concept at a high level. It does not report the max-validators value set on
any particular subnet or which neurons currently hold validator permits. Those are live chain state
and should be checked for the relevant netuid. The default of 64 is the documented value, and the
parameter is governed at the root level
([Subnet Hyperparameters](https://docs.learnbittensor.org/subnets/subnet-hyperparameters)).
