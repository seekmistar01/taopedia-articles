# Fast Blocks

Fast blocks are a local-development configuration that accelerates block production for Bittensor
testing ([Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks),
[Run a Local Bittensor Blockchain Instance](https://docs.learnbittensor.org/local-build/deploy)).

The term belongs to local-chain timing. It names a local development mode, not a property of
mainnet, testnet, or the general Bittensor block cadence.

## Local Timing Role

Fast blocks sit inside local development. A local blockchain is a private chain used for development
and testing, and localnet is the local Bittensor development environment
([Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks),
[Glossary: Local Blockchain](https://docs.learnbittensor.org/resources/glossary#local-blockchain),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Fast blocks therefore describe a timing choice for an isolated development environment. They are not
a separate subnet role, public network, or consensus rule.

## Network Scope

Bittensor separates local testing, testchain, and mainchain stages for subnet development
([Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)).

Bittensor network material also separates mainnet, testnet, and localnet contexts
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Fast blocks belong to the local-testing side of those distinctions. A result observed under
accelerated local timing belongs to that local setting, not to shared testing or production network
history.

## Development Stage Context

The
[Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)
describes subnet development as moving from localnet to testnet and then mainnet. For fast blocks,
that sequence changes how readers should interpret block-timing examples and tempo comparisons
across network environments.

In localnet, fast-block behavior can be tested in an isolated environment. Local block intervals
reflect local chain configuration rather than production block timing.

On testnet, block timing follows the shared non-production cadence rather than accelerated local
fast-block mode. Testnet block intervals are separate from mainnet chain history
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

On mainnet, block production follows the standard 12-second interval documented for production
networks, not the accelerated local fast-block cadence described above
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

The [Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) reference
separates mainnet, testnet, and localnet. A block-timing example from one environment should not be
read as representing production tempo or block history in another environment.

## Documented Timing Contrasts Across Networks

[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks) documentation
states that localnet can produce one block every 0.25 seconds in fast-block mode or one block every
12 seconds in non-fast-block mode, while mainnet and testnet use one block every 12 seconds. The
fast-blocks glossary entry describes the accelerated local mode as 250 millisecond intervals.

That table keeps fast blocks inside local development vocabulary. The 250 millisecond cadence is a
local testing convenience; it is not the production block interval readers should expect on mainnet
or testnet ([Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks),
[Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)).

## Distinction from Block

Fast blocks and a block name a timing configuration and the unit it schedules. A block is the chain
unit in which transactions are included and chain state advances, while fast blocks are a
local-development mode that produces those blocks at an accelerated 250 millisecond interval, so
fast blocks change how often blocks are produced on a local chain rather than what a block is
([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)).

- **Block** — the chain unit in which transactions are included and state advances.
- **Fast blocks** — the local mode that produces blocks at an accelerated 250 ms interval.

The glossary defines fast blocks as accelerated local block production at 250 millisecond intervals
([Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)).

That number is useful as a local timing label. It tells readers how quickly blocks are produced in
the local fast-block mode, not how quickly public-network blocks are expected.

Localnet has the same timing contrast in network terms. It can produce one block every 0.25 seconds
in fast-block mode or one block every 12 seconds in non-fast-block mode
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Mainnet and testnet use one block every 12 seconds
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)). That makes fast
blocks a local acceleration mode rather than a different public-network cadence.

The 250 millisecond value belongs with the localnet label. Without that label, a fast-block example
can make a local timing result sound broader than its environment supports.

Localnet is also the development-and-testing environment
([Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)). That purpose
fits the fast-block term: the faster cadence is useful for local testing, not for describing mainnet
or testnet behavior.

The local-chain setup documentation frames this as running a local Bittensor blockchain instance, so
fast-block timing should be read as part of that private instance rather than as a hosted-network
selection
([Run a Local Bittensor Blockchain Instance](https://docs.learnbittensor.org/local-build/deploy)).

Fast blocks matter because many Bittensor examples involve block-based behavior. Faster local block
production can shorten the time needed to observe local effects from submitted actions, storage
changes, or other block-driven processes
([Run a Local Bittensor Blockchain Instance](https://docs.learnbittensor.org/local-build/deploy),
[Subtensor Storage](https://docs.learnbittensor.org/subtensor-api/storage)).

The term clarifies timing in a local development chain. It does not explain the economic or
operational meaning of a public-network event.

This is especially important for block-counted examples. Rate-limit documentation describes
cooldowns in blocks, so a local fast-block setting can shorten the wall-clock wait for a local
block-counted test without changing the block count itself
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

The same point applies to any local observation that depends on blocks passing. Fast blocks make the
local chain advance more quickly in wall-clock time, while the block-counted behavior remains tied
to the local environment.

## Distinction from Subtensor

Fast blocks and Subtensor name a timing configuration and the blockchain it runs on. Subtensor is
the Bittensor blockchain that records state and processes transactions, while fast blocks are a
local-development mode that runs a Subtensor instance with accelerated 250 millisecond block
production, so fast blocks are a setting applied to a local Subtensor chain rather than a separate
chain ([Glossary: Subtensor](https://docs.learnbittensor.org/resources/glossary#subtensor),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)).

- **Subtensor** — the Bittensor blockchain that records state and processes transactions.
- **Fast blocks** — a local mode that runs Subtensor with accelerated block production.

## Distinction from Rate Limit

Fast blocks and a rate limit both involve blocks but act on different quantities. A rate limit is a
cooldown measured in a fixed number of blocks before an action can be repeated, while fast blocks
shorten the wall-clock time each block takes on a local chain, so fast blocks reduce how long a
block-counted rate limit takes to elapse without changing the block count itself
([Rate Limits](https://docs.learnbittensor.org/learn/chain-rate-limits),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)).

- **Rate limit** — a cooldown counted in a fixed number of blocks.
- **Fast blocks** — the local mode that shortens the wall-clock time those blocks take.

## Distinction from Tempo

Fast blocks and tempo interact because tempo is measured in blocks, not seconds. Tempo is the fixed
number of blocks that defines a subnet's epoch interval, while fast blocks are a local-development
mode that accelerates block production, so under fast blocks the same tempo elapses in far less
wall-clock time, letting epoch-paced behavior be tested quickly
([Glossary: Tempo](https://docs.learnbittensor.org/resources/glossary#tempo),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks),
[Run a Local Bittensor Blockchain Instance](https://docs.learnbittensor.org/local-build/deploy)).

- **Tempo** — the fixed block count that defines a subnet's epoch interval.
- **Fast blocks** — the local mode that speeds block production, shortening that interval's
  wall-clock time.

## Distinction from Non-fast Blocks

Non-fast blocks and fast blocks name opposite sides of one local timing distinction. Fast blocks
speed up local block production, while non-fast blocks keep the ordinary twelve-second local cadence
([Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

- **Non-fast blocks** — ordinary local block timing (~12 seconds).
- **Fast blocks** — accelerated local block timing for quicker feedback.

Fast blocks are clearest when read beside non-fast blocks. The non-fast-blocks glossary entry
describes local blocks that follow Subtensor's default 12-second interval
([Glossary: Non-fast blocks](https://docs.learnbittensor.org/resources/glossary#non-fast-blocks)).

Both terms belong to local blockchain context. Fast blocks shorten local feedback loops, while
non-fast blocks keep local testing closer to default timing.

The shared term is still block. A block is the chain unit in which transactions are included and
chain state advances ([Glossary: Block](https://docs.learnbittensor.org/resources/glossary#block),
[Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks)).

## Distinction from Yuma Consensus

Fast blocks are a local-development configuration that accelerates block production for Bittensor
testing. They name local-chain timing vocabulary, not the on-chain step that turns validator weight
submissions into emission shares
([Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks),
[Run a Local Bittensor Blockchain Instance](https://docs.learnbittensor.org/local-build/deploy)).

Yuma Consensus is the on-chain process that converts submitted validator weights into emission
shares after evaluation completes. Faster local blocks can shorten development epochs, but Yuma
Consensus settles scoring outcomes on subnets rather than configuring local block cadence
([Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus),
[Emission](https://docs.learnbittensor.org/learn/emissions)).

Readers should keep local-timing vocabulary separate from the settlement step. Fast blocks answer
how quickly a local chain produces blocks; Yuma Consensus answers how validator weights become
emission shares in a subnet
([Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks),
[Yuma Consensus](https://docs.learnbittensor.org/learn/yuma-consensus)).

- **Fast blocks** — accelerated block timing for local development testing.
- **Yuma Consensus** — on-chain weight aggregation that settles emissions.

## Reader Boundary

Fast blocks should not be read as a separate chain category, a subnet task, a production performance
statement, or a substitute for network-specific documentation. They name a local timing
configuration that shortens wall-clock block intervals during development testing
([Glossary: Fast blocks](https://docs.learnbittensor.org/resources/glossary#fast-blocks),
[Bittensor Networks](https://docs.learnbittensor.org/concepts/bittensor-networks)).

Fast-block language belongs with the network and chain setting where block timing is being
discussed. A timing observation from one environment does not automatically generalize to another
([Introduction to Bittensor](https://docs.learnbittensor.org/learn/introduction#subnet-development)).
